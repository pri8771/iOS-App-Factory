**Codex (QA & Risk · Pragmatist) — Round 1**

BatchBridge is already constrained enough that the wrong tech-spec move here is over-architecting for flexibility and then missing the trust-critical behaviors. The smallest architecture that still ships cleanly is a single FastAPI backend process with an in-process job runner, a React/Vite frontend, SQLite in WAL mode, and local filesystem storage for PDFs, rendered page images, OCR artifacts, exports, and run payloads. No queue broker, no worker fleet, no auth layer, no event bus. The concurrency boundary should be explicit: HTTP requests stay short and transactional; document processing runs as background jobs owned by a `DocumentProcessingService`; SQLite remains the single source of truth for document status. That matters because the app's main failure modes are not theoretical AI mistakes, they are stale UI state, half-written processing runs, and documents stuck forever in `processing`.

The architecture overview I would lock is this: frontend talks only to backend REST endpoints; backend persists coarse entities in SQLite and stores large blobs on disk with paths recorded in the DB; processing is append-only by run, never in-place mutation. A document row represents lifecycle state. A page row represents page artifacts. OCR and extraction runs are immutable records. Corrections are stored as explicit corrected snapshots plus per-field/line-item edits, not "overwrite the extraction JSON." Approval freezes a corrected snapshot. Export reads only from the approved corrected snapshot. Reprocess creates a new extraction run and marks prior corrections potentially stale; it does not destroy them. If repair retry fails twice, the document transitions to `extraction_failed` and the UI still has enough artifacts to support manual completion. That is the core trust model, and if tech specs blur it, the app degrades into a generic extractor demo.

**Claude (Frontend Engineer · User Advocate) — Round 1**

BatchBridge's tech spec has to do two things at once: lock down a backend data/pipeline architecture solid enough that WAL-mode SQLite and serialized processing actually hold under a real batch upload, and lock down a frontend architecture where every screen decision from the design handoff (server-authoritative status, no silent autosave, keyboard-first review) is expressed as an actual type contract, not a vibe. I'm going to commit to specifics rather than leave anything as "TBD in build," because the acceptance check for this phase is literally whether parallel build lanes can work without inventing incompatible types.

Backend layout: `backend/app/main.py` wires the FastAPI app; `db.py` owns the SQLAlchemy engine with `PRAGMA journal_mode=WAL` and `PRAGMA busy_timeout=5000` set on every connection, plus a session-per-request dependency. `models/` holds one SQLAlchemy model file per table named in the contract — `document.py`, `page.py`, `ocr_result.py`, `extraction_result.py`, `line_item.py`, `corrected_result.py`, `ship_to_record.py`, `prompt_config.py`, `audit_log_entry.py`. `schemas/` holds Pydantic request/response shapes, kept separate from ORM models so API contracts don't leak SQLAlchemy internals. `services/pipeline/` holds `convert.py` (PDF→image+text), `ocr_provider.py` (a `Protocol` plus a `MockOCRProvider` and a real provider adapter), `extraction_provider.py` (same pattern for the LLM call, including the repair-prompt retry), and `orchestrator.py` which runs the full per-document pipeline end to end. `services/validation/needs_review.py` implements the three-layer computation exactly as locked in `detailed_discussion` — confidence threshold, missing/mismatched source_text override, and business-rule reconciliation — as three pure functions that combine, not one blob. `services/matching/ship_to_matcher.py` does fuzzy matching and produces the deterministic reason string. `services/export/` does JSON/CSV generation. `worker/queue.py` is a single in-process `asyncio.Queue` with exactly one consumer task — I want to be blunt about why: the ios_architecture_review round already decided processing must be serialized against SQLite, and the prompt_contract's non-goal list explicitly excludes "queue workers" as in external broker infrastructure (Celery/Redis) — an in-process asyncio consumer is not that, it's just disciplined concurrency, and conflating the two would wrongly block a decision we already need. `worker/stuck_reset.py` is a periodic sweep (every 60s) that resets any document stuck in `processing` past a timeout back to a retryable state with an audit entry, satisfying the must-have from `app_features`.

Frontend layout mirrors the five locked routes: `src/routes/Dashboard.tsx`, `Documents.tsx`, `Review.tsx`, `PromptConfig.tsx`, `ShipToCatalog.tsx`. `src/api/` holds one typed client file per resource (`documents.ts`, `dashboard.ts`, `shipTo.ts`, `prompts.ts`, `audit.ts`) — every function returns a type from `src/types/` that's a hand-written mirror of the backend Pydantic schema, not an inferred `any`. `src/hooks/useDocumentStatusPolling.ts` is the one place the 2-3s polling loop lives, feeding `PipelineStatusStrip` — I want to be the skeptic on this from my own earlier instinct: this hook must poll only while at least one document is in `uploaded`/`processing` state and stop entirely otherwise, or we've built a background tab that never idles, which is exactly the kind of thing that makes a "local-first" desktop tool feel cheap. `src/hooks/useReviewDraft.ts` owns the unsaved-edit draft state for the Review form, reconciled against the TanStack Query cache on load — this is the hook that enforces "no silent autosave": it exposes `isDirty` and the route only navigates away after either a save or an explicit discard confirmation.

On data models: every extracted field — header or line item — is stored as a JSON envelope column, not five separate scalar columns, because there are roughly 14 header fields and 8 line-item fields and hand-rolling five columns each would make every migration and every API response absurd. Concretely: `{"value": ..., "confidence": 0.0-1.0, "source_text": string|null, "page_number": int|null, "needs_review": bool, "review_reason": string|null}`. `extraction_results` is append-only — every reprocess inserts a new row with an incrementing `run_number`, never an update — and it stores `raw_model_output_1`/`raw_model_output_2` so a double-failure has both attempts visible per the `extraction_failed` spec. `corrected_results` is a separate table holding the operator's edited snapshot (header_fields JSON, line_items JSON array, ship_to_resolution JSON), tied to the `extraction_result_id` it was based on — and `is_stale` isn't a stored column, it's computed at read time by comparing `corrected_results.extraction_result_id` against the document's current latest `extraction_result_id`; if they don't match, the API returns a `stale: true` flag and the Review screen surfaces it near the top, exactly as locked. `documents.status` is the single enum the entire frontend keys off — I'm restating this because it's the one rule I'd fight hardest to keep intact: no derived/cached status anywhere in the client.

For the review screen's actual data shape, the `GET /api/documents/{id}` response bundles document metadata, the latest extraction_result's fields, the corrected_results overlay (if any) with the stale flag, ship_to resolution state, and page count — one request, not five waterfalled ones, because a coordinator with 40 documents a day cannot tolerate a review screen that spinner-chains its way to usable.

On the fuzzy Ship To matcher: it has to return not just a score but the actual matched attributes so the reason string is deterministic and inspectable — `matched_on: ["alias", "postal_code"]` as structured data, with the human-readable reason string generated from that list, not the other way around, so tests can assert on the structured field instead of parsing prose.

Testing: pytest fixtures under `backend/tests/fixtures/` include a clean sample PDF, a messy/scanned one, a totals-mismatch one (subtotal+tax≠total, used for the named blocking test), and a fixture that forces the mocked LLM provider to fail JSON validation exactly twice (for the extraction_failed blocking test) — both providers are `Protocol`-typed so the mock and the real adapter are interchangeable without touching the orchestrator. Frontend: Vitest+RTL component tests render `Review.tsx` off four mocked document-status fixtures and assert four distinct outputs; one Playwright spec drives the full upload→flag→correct→resolve→approve→export loop against the mock providers so the app is demoable and testable with zero API keys configured, which also satisfies the "app must remain demoable without keys" assumption from `detailed_discussion`.

The biggest risk in what I'm proposing is the JSON-envelope-column approach for fields — it's the right call for velocity and matches "every field carries value/confidence/source_text/page_number/needs_review" literally, but it means the database can't enforce field-level constraints or do efficient SQL-level querying/filtering on individual field values (e.g., "find all documents where PO number confidence < 0.5" requires either a JSON query or an app-level scan). For an MVP with realistic per-operator document volumes this is fine — I would not add per-field relational columns to fix a query pattern nobody's asked for yet.

## Final Output

**Architecture overview:** FastAPI backend, React+Vite frontend, SQLite (WAL mode, `busy_timeout` set on every connection), single-writer discipline enforced via one in-process `asyncio.Queue` consumer task (not an external broker — that stays correctly out of scope). Docker/docker-compose with named volumes for PDFs, rendered page images, the SQLite file, and exports, verified to survive a `down`/`up` restart cycle. Server-authoritative `status` enum drives all frontend rendering; the frontend never infers or caches a derived mode. Provider integrations (OCR, LLM extraction) are `Protocol`-typed so mock and real adapters are interchangeable, keeping the app fully demoable and testable with zero API keys configured.

**File/module layout:**
Backend — `app/main.py` (FastAPI wiring), `app/db.py` (engine + WAL pragmas + session-per-request), `app/models/` (one file per table: document, page, ocr_result, extraction_result, line_item, corrected_result, ship_to_record, prompt_config, audit_log_entry), `app/schemas/` (Pydantic request/response shapes, kept separate from ORM models), `app/services/pipeline/` (convert.py, ocr_provider.py, extraction_provider.py, orchestrator.py), `app/services/validation/needs_review.py` (three pure, independently-testable layers: confidence threshold, missing/mismatched source_text override, business-rule reconciliation), `app/services/matching/ship_to_matcher.py`, `app/services/export/`, `app/worker/queue.py` (single-consumer processing queue), `app/worker/stuck_reset.py` (60s sweep resetting timed-out `processing` documents with an audit event).
Frontend — `src/routes/{Dashboard,Documents,Review,PromptConfig,ShipToCatalog}.tsx` (the five locked routes), `src/api/` (one typed client file per resource), `src/types/` (hand-written TS mirrors of backend schemas), `src/hooks/useDocumentStatusPolling.ts` (polls only while ≥1 document is uploaded/processing, otherwise idle), `src/hooks/useReviewDraft.ts` (owns unsaved-edit state, exposes `isDirty`, enforces the no-silent-autosave rule from `design_handoff`).

**Data models:** Every header/line-item field is a `FieldValue`/`ExtractedField` envelope (`value`, `confidence`, `needs_review`, `reasons: FieldReason[]`, `evidence: {source_text, page_number, bbox}`) — bbox nullable per field, never per document, since some fields land from context without a literal span. `documents.status` is the single source-of-truth enum. `extraction_results` is append-only (`run_number` increments, both raw model outputs stored for the double-failure case). `corrected_results` is separate, tied to the `extraction_result_id` it was based on, with staleness computed at read time (not stored) by comparing against the document's current extraction result. `ship_to_resolution` carries a `matched_on` string list that the human-readable reason is generated from, plus alternates in the same shape. `audit_log` has a nullable document FK and a denormalized snapshot label so entries outlive a delete cascade.

**Persistence strategy:** WAL-mode SQLite with short transactions and a single background writer (the asyncio queue consumer), avoiding "database is locked" errors under concurrent batch upload + status polling. JSON-envelope columns for field data traded deliberately against SQL-level per-field querying — acceptable at MVP volume, not something to over-engineer around. Docker named/bind-mounted volumes for all artifacts, with an explicit restart test as a build-verification gate, not an assumption from the compose file looking correct.

**Testing strategy:** Backend fixture tests (blocking): subtotal+tax≠total forces `needs_review` on `total` despite high self-reported confidence; a mocked provider failing JSON validation exactly twice lands the document in `extraction_failed` with both raw outputs preserved and never a third retry; Ship To matching against known-alias fixtures asserts both the match and the specific `matched_on`/reason output. Frontend tests: four mocked statuses render four distinct Review outputs; unsaved edits block navigation without explicit confirm; stale-correction banner appears near the top of Review; Documents defaults to attention-priority sort. One Playwright E2E happy path (upload → flag → correct → resolve Ship To → approve → export) and one forced-failure path (double invalid JSON), both against mock providers.

**Architecture overview:** FastAPI backend, React+Vite frontend, SQLite (WAL mode, `busy_timeout` set on every connection), single-writer discipline enforced via one in-process `asyncio.Queue` consumer task (not an external broker — that stays correctly out of scope). Provider integrations (OCR, LLM extraction) are `Protocol`-typed so mock and real adapters are interchangeable, keeping the app fully demoable and testable with zero API keys configured.

**File/module layout:**
Backend — `app/main.py` (FastAPI wiring), `app/db.py` (engine + WAL pragmas + session-per-request), `app/models/` (one file per table: document, page, ocr_result, extraction_result, line_item, corrected_result, ship_to_record, prompt_config, audit_log_entry), `app/schemas/` (Pydantic request/response shapes, kept separate from ORM models), `app/services/pipeline/` (convert.py, ocr_provider.py, extraction_provider.py, orchestrator.py), `app/services/validation/needs_review.py` (three pure, independently-testable layers: confidence threshold, missing/mismatched source_text override, business-rule reconciliation), `app/services/matching/ship_to_matcher.py`, `app/services/export/`, `app/worker/queue.py` (single-consumer processing queue), `app/worker/stuck_reset.py` (60s sweep resetting timed-out `processing` documents with an audit event).

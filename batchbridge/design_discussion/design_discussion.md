**Codex (Product Strategist · Closer) — Round 1**

BatchBridge should be designed as a three-surface product with two support surfaces, and that constraint needs to hold hard or the app will sprawl into admin clutter. The real product is the review workspace. The document queue exists to get the operator into the next risky document fast. The dashboard exists to orient, not to analyze. The support surfaces are prompt configuration and Ship To catalog management, both reachable from the main shell but clearly subordinate to review work. If the design phase produces more than these five screens plus lightweight dialogs/drawers, it is probably inventing product that the spec did not ask for.

The right navigation model is a left rail or top-level shell with Dashboard, Documents, Settings, but the operator should spend most of her day in Documents and then inside a single Review screen. From the document list, “open next needing review” should be a first-class action, not a convenience buried in a menu. Batch upload starts either from an empty state on Documents or a persistent import affordance in the shell. After upload, the user lands back in the queue, watching rows progress through uploaded, processing, extraction_failed, needs_review, approved, or exported. Clicking any row opens the Review workspace. Approving a document should offer an immediate “next document needing attention” path so the operator can stay in flow. Export should be available per document in Review and as a batch action from the Documents screen for approved rows.

The screen inventory should be concrete enough that a React developer can build routes and state without guessing. The Dashboard is a compact status view: counts for uploaded, processing, needing review, approved/exported, average confidence, success rate, recent failures, and a short “attention needed” list that deep-links into documents. It needs empty state for a brand-new install, loading state while stats are queried, success state with live counts, and error state if metrics cannot load. The Documents screen is a queue/worklist with batch upload, search/filter, row status, duplicate hinting if available, and obvious retry/reopen/export affordances based on state. Its empty state is “drop PDFs here”; loading state is initial fetch and per-row progress; success state is a mixed queue; error state covers failed list fetch and row-level action failures. The Review workspace is the main screen: PDF viewer on one side, extracted content on the other, with flagged fields surfaced first, editable header sections, editable line-item table, Ship To resolution card, and secondary tabs for raw OCR and raw JSON. It needs explicit variants for needs_review, extraction_failed, approved/read-only, and reopened-after-export. The Prompt Config screen should expose the global prompt, per-customer overrides, prompt version history enough to understand what ran, and rerun actions tied back to documents. The Ship To Catalog screen should support list/search/edit/create records and aliases, but also the same create/edit form should appear inline from Review as a sheet or drawer so a no-match never ejects the operator into settings.

The primary user flow should be fixed now. First-run user opens Documents and sees an empty state with a short privacy note that files are stored locally and OCR/LLM calls may send content to configured providers. She uploads a batch of PDFs. Rows appear immediately as uploaded, then processing. When processing completes, the queue sorts or highlights documents needing attention first. She opens the first needs_review document and lands in Review with the riskiest fields already visible, not at the top of an arbitrary form. She inspects inline evidence only where needed, edits header fields and line items, resolves Ship To from suggested match or manual create/select, then approves. On approval she can export now or continue to the next document. If extraction fails twice, she still opens the same Review route, but in an extraction_failed variant with blank editable fields, visible OCR, both failed model outputs, and a rerun-with-prompt-edit path. At end of session she batch-exports approved documents from Documents or exports individually from Review. That is the whole product loop and it should be frictionless enough to demo in a few minutes.

The state model per screen is where this can quietly go wrong, so I would lock it down explicitly. Dashboard has no-document empty, loading, populated, refresh-in-progress, and error. Documents has first-run empty, upload-in-progress, populated list, filtered-empty, and list error; each row has its own state and row-level actions keyed off the backend enum, not inferred in the client. Review has loading, not-found, needs_review editable, extraction_failed editable-manual, approved read-only with reopen action, reopened editable, export-in-progress, and save/reprocess error substates. Prompt Config has loading, empty-no-overrides-yet, editing-draft, saving, saved, rerun-in-progress, and error. Ship To Catalog has loading, empty catalog, populated list, inline create/edit sheet, save success, and validation/error. The key product rule is that no state should leave the operator wondering whether work was saved, overwritten, or merely displayed optimistically; stale-correction warnings after reprocess must be visible in Review, not hidden in audit logs.

The visual direction should feel like a premium operations tool, not a consumer AI toy and not a dead enterprise gray slab. I would push for a restrained, evidence-first interface: warm off-white background, charcoal text, one deep accent color for trusted actions, one amber/red system for attention states, and a distinct treatment for extracted evidence chips and review flags. The PDF should feel like a document under inspection, with page thumbnails or page jump affordances, while the data side should feel like a working panel, not a long form. Confidence should not be rendered as tiny percentages everywhere; the default language should be action-oriented states like Ready, Review, Failed, Approved, with confidence exposed where it helps explain risk. Raw OCR and raw JSON need to be present but visually demoted. The product should look calm and serious, because the user is processing orders, not exploring AI output.

Accessibility has to be built into the design now because the review surface will otherwise become a maze. Keyboard-first operation matters for this persona, so tab order, field focus, approve-and-next shortcuts, and table editing need deliberate support. Color cannot be the only review signal; flagged fields need iconography, labels, and text contrast. Evidence snippets and status badges must meet contrast standards. The PDF/data split should survive zoom and narrow widths without becoming unusable; on smaller screens the layout can stack, but the operator still needs fast switching between source and extracted fields. Screen reader labels need to distinguish original extracted value, corrected value, confidence, source page, and review status so the provenance model is not invisible to assistive tech. Error states must be plain-language and actionable: “Password-protected PDF could not be processed” is acceptable; “conversion error” is not.

The biggest risk in this direction is making the Review workspace so information-dense that it becomes slower than manual entry. The fix is not hiding evidence deeper; it is structuring the workspace around triage. Flagged fields first, safe fields collapsed or deemphasized, Ship To as its own focused decision block, line-item anomalies called out inline, and raw/debug material kept secondary. If the default review view shows every provenance attribute for every field equally, the product will fail its own speed promise.

The detail the final phase output must preserve is that Review is not a generic form screen with a PDF next to it. It is a stateful decision workspace built around risk ranking, provenance, and safe progression to the next document. If that core idea gets watered down into “dashboard plus upload plus editor,” BatchBridge loses the one thing that differentiates it from every other extraction demo.

**Claude (Design Lead · Visionary) — Round 1**

The version of this I want to fight for is one where the review workspace doesn't just show evidence when asked — it makes the PDF itself the source of truth and treats every extracted field as a claim you can click back to. Concretely: if the OCR provider gives us word or line-level bounding boxes (many do — Textract, Google Document AI, Azure Document Intelligence all support this), clicking a flagged field doesn't just jump to a page number, it draws a highlight directly on the rendered PDF page image over the exact text the model read. That's the difference between "trust me, I read page 2" and "here is the literal ink I extracted this from." If the configured provider doesn't return coordinates, we degrade gracefully to page-jump-plus-snippet — I don't want to overpromise this as a hard requirement, but I want it fought for as the flagship interaction, because it's the single most convincing thing we could put in a 30-second demo video and it's directly in service of the trust story every prior phase has centered on.

On screens: I count seven real surfaces, not more. A dashboard (status/counts, entry point). A document inbox/list that's also the upload surface — the drop zone should be a persistent pinned strip at the top of the list, not a separate "upload" screen, so uploading and triaging live in the same glance. The review workspace, which is really three visual modes of one screen depending on document state: the needs_review mode (flagged-first, split PDF/fields), the extraction_failed mode (blank hand-entry form plus both raw failed model outputs sitting side by side so the operator can literally see what went wrong twice), and the clean/no-flags mode (fields visible but approve is one click away, no forced tour through evidence nobody asked for). A Ship To resolution surface that lives inline inside review, not a modal or separate page — it should render like a comparison scorecard: suggested match at top with the matched attributes visually checked off (postal ✓, city ✓, name partial match), alternates below as lighter cards, and a "create new from this address" affordance sitting right there when nothing matches, so a brand-new customer never dead-ends the flow. The state model that matters most is on the review workspace, because that's where the product lives or dies. Loading isn't a spinner — it's a short narrated pipeline strip ("converting pages… OCR page 2 of 4… extracting fields…") that turns the wait itself into a trust-building moment instead of dead time, and it should persist as a compact bar even after the operator navigates back to the inbox, since batch upload means multiple documents are cooking at once. Needs_review mode defaults to showing only flagged fields and the line-item table with a clearly reachable "show all fields" toggle — this is the one place I'd push back hardest against any temptation to show everything at once, because that's the exact failure mode called out already: a wall of metadata makes review slower than retyping. Extraction_failed is visually distinct at the state-badge level with a genuinely different color and icon, not a shade of the same red as a generic error, because a human scanning a list of forty documents needs to tell "needs my judgment" from "the pipeline actually broke" at a glance. Reopened documents need their own banner treatment — a clear "this was previously exported, you're editing a locked record" strip — so nobody edits a reopened doc thinking it's still pre-export.

Visually, I want to steer away from generic SaaS-blue-gradient-dashboard territory entirely. Think evidence desk, not marketing site: warm neutral background, a restrained accent palette where amber/gold means needs_review, a distinct violet or teal means "matched with explanation," green means approved, and red is reserved exclusively for extraction_failed and hard errors so it never gets diluted into background noise. Typography should do real work: a humanist sans for UI chrome and labels, but a monospace face for source-text snippets, OCR output, and raw JSON — that typographic shift is itself a signal to the operator that "this is the actual machine output, not our copy," which reinforces the whole "prove it" wedge without adding a single extra UI element. The PDF pane should support real zoom and should feel like the primary object on screen, with the fields panel as a resizable companion rather than the PDF being squeezed into an afterthought thumbnail, because operators are visually cross-checking against the actual document constantly.

Accessibility can't be an afterthought bolted onto a keyboard-optional design, because the product brief already committed to a keyboard-driven approve-and-advance loop for daily throughput — so keyboard support isn't a nice-to-have, it's the same feature as the speed story. Every status and confidence signal needs an icon-plus-text pairing, never color alone. Focus needs to be managed explicitly when an evidence popover opens and closes so keyboard users don't get stranded. The raw OCR text tab effectively serves double duty as the accessible text alternative to an image-only PDF render, which is worth stating outright rather than leaving implicit, since screen-reader users otherwise have no way to consume a page image at all.

The biggest risk in what I'm pushing is that the bounding-box highlight is a genuinely nice differentiator that depends on OCR provider capability we don't control, and if it gets treated as a hard requirement rather than a progressive enhancement, it'll either block the build or quietly get faked. I want it specified as tiered: page-jump-plus-snippet is the baseline that always works, coordinate-level highlighting is the enhancement when the provider supports it. What I don't want lost from this phase, no matter how debate goes: flagged-first progressive disclosure, the three visually distinct document-state treatments, the Ship To match as a scorecard rather than a score, and the keyboard-first approve/advance loop are not polish — they're the literal mechanism by which this app is faster and more trustworthy than retyping a PO by hand, and if any of them get traded away for a cleaner-looking generic dashboard, the app stops being the thing six phases of discussion already agreed it has to be.

**Visual direction:** An "evidence desk," not a SaaS dashboard or an AI-toy interface — warm off-white background, charcoal text, one deep accent for trusted actions, amber/gold for needs_review, a distinct violet/teal for "matched with explanation," green for approved, red reserved exclusively for extraction_failed/hard errors so it's never diluted. Humanist sans for UI chrome; monospace specifically for source-text snippets, OCR output, and raw JSON, so the typography itself signals "this is literal machine output" versus app copy. PDF pane is the primary object on screen with real zoom and page navigation, fields panel as a resizable companion, not a squeezed afterthought. Confidence is expressed primarily through action-oriented state language (Ready/Review/Failed/Approved) with raw numbers available but not the default surface.

**Accessibility notes:** Keyboard-first is a core feature, not a nice-to-have, because the product brief already committed to a keyboard-driven approve-and-advance loop — deliberate tab order, field focus, and explicit focus management when evidence popovers open/close. No status or confidence signal relies on color alone; every one pairs an icon with text. Raw OCR text doubles as the accessible text alternative to an image-only PDF render (screen-reader users otherwise have no way to consume a page image). Error states use plain, actionable language ("Password-protected PDF could not be processed"), never generic technical phrasing.

**What must not get diluted going into architecture/build:** the Review workspace is a stateful decision workspace built around risk-ranking and provenance, not a generic form-next-to-a-PDF; the three distinct document-state visual treatments, the Ship To scorecard, flagged-first progressive disclosure, and the keyboard-first approve/advance loop are load-bearing product mechanics, not polish. The bounding-box PDF highlight is explicitly a tiered progressive enhancement (page-jump+snippet is the baseline that must always work) — it should be pursued but never allowed to block the build or get faked if the OCR provider doesn't support coordinates.

### Round 1


**Codex (Design Lead · Systems Thinker) — Round 1**

The feature set needs to stay brutally aligned to the product promise: get a messy PO from uploaded PDF to trustworthy approved export with the fewest possible operator decisions, while making risk visible instead of pretending extraction is magic. If we let this phase drift into “nice admin features,” the build phase will drown. I’d treat the review workspace, the document queue, and the processing pipeline as the only true product surfaces. Everything else exists to support those.

My must-have set is compact but complete. First, batch PDF intake with per-document status and durable local persistence. The user story is: “I can drop in a stack of PO PDFs and see each file move through processing without losing work on refresh or restart.” Acceptance criteria: multiple PDFs can be uploaded in one action; each document gets a visible state (`uploaded`, `processing`, `extraction_failed`, `needs_review`, `approved`, `exported`); documents persist across restart; invalid/encrypted PDFs fail with a visible error state, not a crash.

Second, the end-to-end extraction pipeline with one repair retry and a reviewable failure state. The user story is: “I can run OCR and structured extraction automatically, and if extraction fails, I still have a usable manual recovery path.” Acceptance criteria: every page is OCR’d; extraction returns strict validated JSON or retries once with a repair prompt; if both attempts fail, the document lands in `extraction_failed`; the operator can still open that document, view PDF/OCR/raw outputs, and hand-enter data.

Third, server-computed review prioritization. The user story is: “The app tells me what needs attention instead of making me inspect every field manually.” Acceptance criteria: every required field and line-item field includes `value`, `confidence`, `source_text`, `page_number`, and `needs_review`; `needs_review` is computed from thresholding plus evidence presence plus business-rule checks; missing required fields, unreconciled totals, and unresolved Ship To cases are visibly flagged.

Fourth, the side-by-side review workspace. This is the actual product. The user story is: “I can review, correct, and approve a PO faster than retyping it.” Acceptance criteria: PDF and extracted data are visible together; flagged fields are easy to identify at a glance; header fields are editable; line items are editable in a table; source evidence is visible inline or one click away, not buried; raw OCR and raw JSON are available as secondary tabs; approving is blocked until required issues are resolved or explicitly accepted.

Fifth, Ship To matching with explanation and manual fallback. The user story is: “I can trust or override the suggested Ship To without leaving the workflow.” Acceptance criteria: the system proposes a best match plus alternates; each suggestion includes confidence and a plain reason; no-match and empty-catalog states are handled in-flow; the user can manually select a catalog record or create a new one inline from the document.

Sixth, correction history and safe reprocessing. The user story is: “I can re-run extraction without losing my work, and I can trust the audit trail.” Acceptance criteria: each extraction run is stored separately; corrected values are stored separately from raw extraction; reprocessing never silently overwrites corrections; stale corrections relative to a new extraction are surfaced; exported/approved docs require explicit reopen before further edits.

Seventh, approval and export. The user story is: “Once I trust the document, I can lock it and export usable outputs.” Acceptance criteria: only approved documents export through the normal path; JSON and CSV exports are generated from corrected data; original and corrected versions remain available; export is recorded in audit history.

Eighth, minimal operator settings for prompts and catalog management. The user story is: “I can improve extraction for repeat customers without changing code.” Acceptance criteria: global prompt config exists; per-customer override exists; editing either can trigger reprocessing; Ship To catalog records and aliases are manageable from the UI.

The should-have list is short. Keyboard-first review actions and next-document navigation should be in if build time allows, because they reinforce the speed story. Duplicate-document signaling should be in if easy, since double-exporting the same PO is a realistic failure. A small dashboard with counts, average confidence, and success rate should exist because it’s required, but it should stay lightweight and not become an analytics product. Batch export of all approved documents is also worth including if it doesn’t destabilize the core flow.

Could-have features are only things that polish the core loop without expanding scope: saved filters on the document list, richer diff views between extraction and corrected values, and lightweight sample/demo mode messaging when no API keys are configured. These are only worth doing after the must-have flow is real.

Won’t-build needs to be explicit or the build will sprawl: no auth, no multi-user workflow, no ERP push, no email ingestion, no invoice or non-PO document support, no multi-document PDF auto-splitting, no cloud sync, no storage cleanup subsystem, no model-training studio, no advanced analytics, no regression detection for prompt quality, no multilingual optimization beyond graceful degradation.

The information architecture should preserve one principle: the document list answers “what needs my attention,” the review workspace answers “what is wrong and why,” and settings answer “how do I make repeat work faster.” If any must-have feature doesn’t strengthen one of those three jobs, it probably shouldn’t be in scope.

The biggest risk in my own cut is that even this “ruthless” list can still become too broad if the team treats every acceptance criterion like a mini-product. The mitigation is to keep the UI simple and progressive: one queue, one review surface, one settings area, and very few detours. The detail the final phase output absolutely must preserve is that BatchBridge is not a generic extraction dashboard. It is a correction-first workbench where the system actively guides the operator to risk, preserves evidence, and never makes them choose between speed and trust.

   Acceptance: each reprocess creates a new `extraction_results` row; `corrected_results` persist separately and are never silently overwritten; the UI surfaces when a correction may be stale relative to a fresh extraction; once a document is exported, its corrected snapshot is immutable — any PATCH attempt without an explicit, audit-logged "reopen" is rejected.

7. **Approval and export.**
   Story: "Once I trust the document, I lock it and get usable structured output."
   Acceptance: only `approved` documents can export; JSON and CSV are generated from corrected data; original and corrected versions both remain retrievable; export is recorded as a boundary event in `audit_log`.

8. **Prompt configuration and Ship To catalog management from the UI.**
   Story: "I can make extraction better for repeat customers without touching code."
   Acceptance: global prompt + per-customer override, both editable in-UI; editing and hitting rerun produces a new tied `extraction_results` row; Ship To catalog records and aliases are manageable from the UI, not just seeded via script.

9. **Dashboard (uploaded/processed/needing-review counts, average confidence, success rate).**
   Story: "I can see at a glance how the queue is doing."
   Acceptance: loading the dashboard against a seeded, mixed-state document set shows counts that match the database exactly — not a cached or estimated figure. Kept lightweight; this is a status view, not an analytics product.

10. **Stuck-`processing` recovery (elevated from should- to must-have).**
    Story: "A backend crash mid-pipeline doesn't leave a permanent zombie document."
    Acceptance: a document stuck in `processing` past a timeout is either auto-reset or exposes a manual retry action — never a permanently stuck row with no recovery path.

**Should-have:** keyboard-driven approve-and-advance-to-next-document; visual flagging of likely duplicate uploads; a tunable confidence threshold exposed in settings rather than hardcoded; batch end-of-day export of all approved documents in one action.

**Could-have:** inline highlighting of line-item math mismatches beyond just triggering `needs_review`; a simple original-vs-corrected diff view; saved filters on the document list; demo-mode messaging when no API keys are configured.

**Won't-build:** ERP integration, SSO, multi-user auth/permissions, queue-based async processing, vendor portals, human-assignment workflows, mobile clients, cloud sync, multi-document-PDF auto-splitting, disk-retention/cleanup automation, multi-language extraction quality, any document type beyond POs, and automated detection of "this prompt override made things worse" (the per-customer editor + rerun is the mitigation, not a regression-detection system).

**Bar for "done," not just a checklist:** the build must demonstrably pass two specific tests — a subtotal+tax≠total sample forcing `needs_review` on `total` despite high model confidence, and a forced double-failure (mocked bad-JSON twice) landing cleanly in a usable `extraction_failed` state rather than a hang or crash. Without both, the app has built a generic extraction demo, not the trusted-correction workbench this whole program has been converging on.

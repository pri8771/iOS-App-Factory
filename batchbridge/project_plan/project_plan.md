If mock-provider work slides to "whenever someone needs it for a test," it'll get built inconsistently three times by three build lanes instead of once, correctly, up front.

I want to flag a sequencing trap here specifically: it would be easy to build the pipeline orchestrator and the needs_review logic as one entangled unit inside `run_pipeline`, since that's the path of least resistance, but the three-layer needs_review rule (confidence threshold, missing/mismatched source_text, business-rule reconciliation) is supposed to be independently unit-testable against the subtotal+tax≠total fixture.

## Final Output

**Screen inventory (5 real surfaces, no more):**
1. **Dashboard** — status/orientation only (counts, average confidence, success rate, recent failures, "attention needed" deep-links). Not an analytics product.
2. **Documents** (inbox/queue) — the upload surface (persistent drop-zone strip pinned at top, not a separate screen) plus the worklist with per-row state, search/filter, duplicate hinting, and state-appropriate row actions (retry/reopen/export). "Open next needing review" is a first-class action here.
3. **Review workspace** — the actual product. One screen, three visual modes driven by document state:
   - *needs_review*: flagged-fields-first, split PDF/fields, Ship To resolution card inline (scorecard style: suggested match with matched attributes checked off, alternates as lighter cards, "create new from this address" affordance), secondary raw-OCR/raw-JSON tabs, "show all fields" toggle to escape the default flagged-only view.
   - *extraction_failed*: visually distinct state badge (different color/icon from needs_review, not a shade of the same red), blank hand-enterable fields, both raw failed model outputs shown side by side, reprocess-with-edited-prompt action.
   - *approved/read-only* and *reopened*: read-only with an explicit reopen action; reopened documents get their own banner ("previously exported, editing a locked record").
4. **Prompt Config** — global prompt editor, per-customer override list with clear override indicators, rerun action explicitly tied to the extraction_results row it produces.
5. **Ship To Catalog** — list/search/edit/create records and aliases; the same create/edit form also appears as an inline sheet/drawer from Review so a no-match never ejects the operator out of the review flow.

A first-run/demo-mode condition (banner on Dashboard or Documents when no API keys are configured, pointing at sample PDFs) is a state, not a sixth screen.

**Primary user flow:** Land on Documents (first-run shows empty state + local-storage/privacy note) → drop a batch of PDFs → rows appear as uploaded → processing, with a narrated pipeline strip ("converting… OCR page 2 of 4… extracting…") rather than a dead spinner, visible even after navigating away since multiple documents cook concurrently → queue surfaces documents needing attention first → open one, land in Review pre-flagged → inspect evidence inline (page-jump+snippet always; coordinate-level PDF highlighting as a progressive enhancement when the OCR provider returns bounding boxes) → correct/accept fields → resolve Ship To via the scorecard or manual/inline-create → approve → immediate "next document needing attention" path or export → end-of-session batch export of approved documents from Documents. Extraction_failed documents follow the same Review route in their distinct mode, with a rerun-with-edited-prompt escape hatch.

**State model per screen:**
- *Dashboard*: no-document empty, loading, populated, refresh-in-progress, error.
- *Documents*: first-run empty, upload-in-progress, populated list, filtered-empty, list error; each row's state and actions are keyed strictly off the backend enum, never inferred client-side.
- *Review*: loading, not-found, needs_review-editable, extraction_failed-editable-manual, approved-read-only-with-reopen, reopened-editable, export-in-progress, save/reprocess-error. No state should ever leave the operator unsure whether work was saved, overwritten, or merely displayed optimistically — stale-correction warnings after reprocess must be visible in Review itself.
- *Prompt Config*: loading, empty-no-overrides-yet, editing-draft, saving, saved, rerun-in-progress, error.
- *Ship To Catalog*: loading, empty catalog, populated list, inline create/edit sheet, save success, validation/error.

**Visual direction:** An "evidence desk," not a SaaS dashboard or an AI-toy interface — warm off-white background, charcoal text, one deep accent for trusted actions, amber/gold for needs_review, a distinct violet/teal for "matched with explanation," green for approved, red reserved exclusively for extraction_failed/hard errors so it's never diluted. Humanist sans for UI chrome; monospace specifically for source-text snippets, OCR output, and raw JSON, so the typography itself signals "this is literal machine output" versus app copy. PDF pane is the primary object on screen with real zoom and page navigation, fields panel as a resizable companion, not a squeezed afterthought. Confidence is expressed primarily through action-oriented state language (Ready/Review/Failed/Approved) with raw numbers available but not the default surface.

**Accessibility notes:** Keyboard-first is a core feature, not a nice-to-have, because the product brief already committed to a keyboard-driven approve-and-advance loop — deliberate tab order, field focus, and explicit focus management when evidence popovers open/close. No status or confidence signal relies on color alone; every one pairs an icon with text. Raw OCR text doubles as the accessible text alternative to an image-only PDF render (screen-reader users otherwise have no way to consume a page image). Error states use plain, actionable language ("Password-protected PDF could not be processed"), never generic technical phrasing.

**What must not get diluted going into architecture/build:** the Review workspace is a stateful decision workspace built around risk-ranking and provenance, not a generic form-next-to-a-PDF; the three distinct document-state visual treatments, the Ship To scorecard, flagged-first progressive disclosure, and the keyboard-first approve/advance loop are load-bearing product mechanics, not polish. The bounding-box PDF highlight is explicitly a tiered progressive enhancement (page-jump+snippet is the baseline that must always work) — it should be pursued but never allowed to block the build or get faked if the OCR provider doesn't support coordinates.

## Final Output

**Screen inventory & routes:** Five routes — `/dashboard`, `/documents`, `/documents/:id/review`, `/settings/prompts`, `/settings/ship-to`. Review is one route with rendering driven strictly by the server-returned `status` enum (`needs_review`, `extraction_failed`, `approved`, `reopened`) — refetched on every route entry and after every mutation, never inferred or cached client-side. This is a hard architectural rule, not a preference: it exists to prevent the exact bug where a completed reprocess flips a document's real status while the UI still shows a stale mode.

**Per-screen design (layout, hierarchy, components, states):**

- **Dashboard** — orientation only, not analytics. Top row: four metric cards (uploaded, needs review, approved, extraction failed). Second row: average confidence + success rate. Right-side "attention now" panel deep-linking into flagged documents (this is the number that drives behavior, so it leads visually). Terse recent-activity feed below. Empty state explains local-first storage + the honest provider-call carve-out, offers "Load sample documents" and "Upload PDFs." Skeleton-card loading, not spinners. Error state keeps shell chrome with an inline retryable panel. States: no-document empty, loading, populated, refresh-in-progress, error.

- **Documents** — a queue, not a file browser. Pinned intake strip (drag-drop + rules text about password-protected/invalid PDFs failing visibly) always present, including in the empty state — never a separate upload page. Filter bar: search, state pills, customer filter, duplicate toggle. Default sort is **attention-priority** (needs_review/extraction_failed floated first), with an explicit toggle to switch to recency/filename — this default was an open gap in the design round and is now resolved as a deliberate product decision, not left to whoever builds the sort. "Open next needing review" is a first-class button and keyboard shortcut. Row status badges are state-keyed with state-specific actions (retry/reopen/export/continue). Processing rows show live pipeline text via lightweight polling (2–3s) against a status endpoint from a single top-level provider — not a persistent WebSocket, which is unjustified complexity for a single-user local tool. If that polling narration slips, the acceptable fallback is a static "processing" badge refreshed on the normal list interval; this was explicitly named a should-have, not a spec regression. Duplicate detection is a soft warning chip, never blocking. States: first-run empty, upload-in-progress, populated, filtered-empty, list error; every row's state/actions come from the backend enum.

- **Review workspace** — the product. Two-pane layout: PDF pane (left, co-primary not secondary — real zoom, thumbnails, click-to-jump, resizable/remembered split) and a structured decision pane (right) led by a document state bar (status, customer, PO#, last run, actions: reprocess/approve/export/next). Right pane default view is flagged-fields-first (each row: label, editable value, confidence treatment, source snippet, page ref, plain-language reason — low confidence / missing evidence / totals mismatch / unknown Ship To / stale-after-reprocess), then the Ship To scorecard, then the line-item table, then a collapsed "all other fields" section — never a full form dumped by default. Evidence: bbox overlay on the PDF when the OCR provider returns coordinates, else page-jump + text snippet — resolved **per field**, not per document, since some fields land from context without a literal span; never faked, never blocking. Ship To resolution is a scorecard, not a dropdown: dominant best-match card with checked-off matched attributes and a deterministic reason string, lighter alternate cards below, and when nothing matches the card flips its primary CTA to "create new from extracted address" with manual-pick as secondary — this keeps a brand-new customer from ejecting the operator out of review. Line items behave like a correction-tuned spreadsheet: frozen line-no/material-code columns, editable cells, row-level anomaly markers, and a totals-reconciliation block comparing extracted vs. computed subtotal/tax/total beneath the table, stating plainly when business-rule checks forced review. Raw OCR/raw JSON are secondary tabs (OCR keeps page grouping + search; JSON is monospaced, copyable, tagged with run metadata) — except in `extraction_failed` mode, where both raw failed outputs sit prominently beside a blank hand-entry form so the operator can recover without leaving the workspace. Approved/reopened get distinct calm/locked treatments — reopened carries a persistent "previously exported, editing a locked record" banner. Stale-correction warnings after reprocess surface near the top of the pane, never buried in audit history. States: loading, not-found, needs_review-editable, extraction_failed-editable-manual, approved-read-only-with-reopen, reopened-editable, export-in-progress, save/reprocess-error.

- **Prompt Config** — global editor (left) + per-customer override list (right) showing customer, last-used date, doc count affected, supplements-vs-replaces indicator. Opening an override shows the full override prompt alongside the global baseline for comparison and recent rerun outcomes. Primary interaction is explicitly "save and rerun on this document" vs. "save for future documents," not a bare save button. States: loading, empty-no-overrides-yet, editing-draft, saving, saved, rerun-in-progress, error.

- **Ship To Catalog** — searchable table (Ship To ID, customer, normalized address, country, alias count, last matched date). Edit opens a side sheet, never a page nav. Alias chips prominent (they're the moat). The inline create/edit sheet used from Review is the *same component*, prefilled from the extracted address, with non-blocking duplicate warnings. States: loading, empty catalog, populated, inline create/edit sheet, save success, validation/error.

**State management architecture (load-bearing, not implementation detail):** Server state (documents, extraction/correction results, catalog, prompts, dashboard counts) lives in a query-cache layer (e.g., TanStack Query) with explicit invalidation on every mutation — never hand-rolled global state that can drift from the backend. Local UI state (focus, toggle states, in-progress draft text) stays component-local and never leaks into the server-state cache. Review-form edits are drafts reconciled against server state on load, with an explicit unsaved-changes indicator; navigating away with unsaved edits prompts confirmation — **no silent autosave-on-blur**, since that quietly violates "operator must never wonder if work was saved."

**Design tokens:** Background `#FAF8F3`, surface `#FFFFFF`, text primary `#2B2620`, text secondary `#6B6459`. Trusted-action accent `#2B4C5C` (deep teal-blue). Needs-review `#B8860B` on `#FDF6E3`. Matched/explained (Ship To) `#5B4B8A` on `#F2EEFA`. Approved `#2F6B4F` on `#E8F3ED`. Extraction-failed/hard-error `#B23A2E` on `#FBEAE8` — reserved exclusively for that state, never reused as generic warning. Humanist sans (Inter/system-ui) for chrome at 14–16px; monospace (JetBrains Mono/ui-monospace) at 13px on a faint gray-tinted background for source snippets/OCR/JSON, so the type shift itself signals "literal machine output." 4px spacing scale (4/8/12/16/24/32). Radius 6px chips/buttons, 10px cards. Minimal elevation — one subtle level for the Ship To card and inline sheets only; nothing floats over the PDF pane.

**Component inventory:** StatusBadge (7 states, icon+text always paired), ConfidenceChip (secondary to StatusBadge), EvidencePopover (snippet + page-jump + optional bbox trigger, Escape-dismissible, returns focus to trigger), FieldRow, LineItemTable (ARIA grid roles, deliberate arrow-key/tab cell navigation — not a table-library default), ShipToScorecard, PipelineStatusStrip (polling-driven, degrades to static badge), PromptEditor, UploadDropzone (persistent-strip + full-empty-state variants), DocumentRow, DashboardStatCard, ConfirmDialog (reopen, discard-unsaved-edits, delete).

**Motion:** Restrained and causal, never decorative. Row insertion on upload gets a soft settle; state-badge changes crossfade color/icon over ~150ms; field→PDF evidence jump animates the page transition and briefly pulses the highlighted region; flagged-only↔show-all toggles animate height/opacity under 200ms; Ship To candidate changes crossfade emphasis; reprocess preserves scroll position and surfaces stale-correction markers in context; approve-and-advance shows a ~150–200ms confirming pulse on the button before route transition so rapid keyboard users trust the first press registered.

**Accessibility:** Keyboard-first is load-bearing, not optional — full tab order across PDF pane → flagged fields → line-item table → Ship To card → approve action, plus a global approve-and-next shortcut (Cmd/Ctrl+Enter) that's disabled while any field has an unsaved draft. Focus moves deterministically to the first flagged field (or the Ship To card if that's the only open item) on Review load. No status/confidence signal is color-only — every one pairs icon + text. Raw OCR text is the accessible fallback for an image-based PDF render, stated explicitly. Error copy is plain and actionable ("Password-protected PDF could not be processed," never "conversion error"). Two-pane layout degrades to stacked only at genuinely narrow widths, preserving usability under zoom/high-contrast/text-resize.

**Progressive-enhancement guardrails (explicit, not implicit):** Both the live pipeline-narration strip and coordinate-level PDF evidence highlighting are should-have enhancements with mandatory working fallbacks (static badge; page-jump+snippet) checked in from day one. Neither may block the build or ship half-wired.

**Claude Design decision: no pause for external design.** The token system, component specs, and state models above are concrete enough for direct implementation by a frontend builder without a round-trip to an external design tool — this is a single-operator internal tool where getting the state machine and keyboard flow right has far higher ROI than a visual-novelty pass. ## Ios Architecture Review

CONSENSUS: YES

Both Codex and I landed in the same place, from two different angles that don't actually conflict. Codex tackled the phase's literal question head-on — "could this be a native iOS app" — and concluded no: BatchBridge's actual product (side-by-side PDF inspection, dense line-item correction, raw OCR/JSON tabs, keyboard-driven batch throughput) is a desktop-web shape, and forcing it onto iPhone would just be a worse version of the same tool. If someone insists on an Apple-platform answer anyway, Codex's fallback is concrete and narrow: iPad-first SwiftUI only, scoped down to a review/approval station rather than the full intake desk, with boring first-party choices throughout (PDFKit, Vision only as an optional on-device OCR fallback, StoreKit 2 for functional tier limits, SQLite via a thin store layer instead of SwiftData given the audit-heavy versioned data model, zero third-party dependencies by default).

I took the practical view that since this app was locked as a FastAPI/React/Vite/SQLite/Docker product six phases ago with no dissent, the useful thing this phase can do is re-map its actual required outputs (architecture risk, persistence, privacy, dependencies, testability) onto the real stack instead of manufacturing a SwiftUI answer nobody asked for. That surfaced three concrete risks nobody had stress-tested yet: SQLite write-lock contention once multiple documents are processing concurrently while the frontend polls status every 2-3 seconds (fix: WAL mode + short transactions, likely serialize per-document processing), Docker volume/bind-mount correctness so PDFs/images/SQLite actually survive a `docker-compose down`/`up` cycle (needs an explicit restart test, not just a compose file that looks right), and a real network-exposure question (bind to loopback by default rather than `0.0.0.0` unless documented otherwise, since there's no auth layer). I also turned the "done" bar from app_features into an actual test plan: named fixture-based backend tests for the three-layer needs_review logic, the exactly-one-repair-retry-then-extraction_failed path, and Ship To match+reason-string assertions — plus frontend tests asserting four distinct renders off the four server statuses, and one real Playwright end-to-end run of the full upload→correct→approve→export loop.

Neither of us found any dependency that would violate the permissive-license policy (pdf.js is Apache-2.0 and is the one to pin deliberately), and neither of us found any ML/LLM/AR use that isn't already justified by user value from prior phases (OCR/LLM are the core pipeline, not a bolt-on). Gemini still couldn't join. No real disagreement surfaced — this is genuine convergence on "don't force native iOS, and here's the substantive risk review for the stack this app actually is."

## Final Output

**Platform verdict:** BatchBridge stays a local-first web app (FastAPI + React/Vite + SQLite + Docker) — this phase does not re-scope it to native iOS. The product's core mechanics (side-by-side PDF + dense editable data, raw OCR/JSON tabs, batch intake, keyboard-driven throughput) are a desktop-web shape; forcing them onto iPhone would degrade the product, and even iPad only works if scoped down to a narrower "review station," which is not what six phases of consensus already designed.

**If a native Apple client is ever pursued (explicit fallback answer, not a build instruction):**
- iPad-first SwiftUI only, scoped to document review/approval, not full intake — no serious ambition for iPhone.
- Architecture: `NavigationSplitView` for queue/review/settings, `@Observable` `@MainActor` view models, repository/service layer so the UI is agnostic to whether data came from OCR, manual entry, or a mock provider.
- Persistence: plain SQLite behind a thin internal store (not SwiftData) — the data model (documents, pages, OCR runs, extraction runs, corrected snapshots, line items, Ship To records, prompt configs, audit events) needs explicit IDs and append-only history that's easier to reason about outside SwiftData's migration model.
- Frameworks: PDFKit for rendering/navigation, Vision only as an optional on-device OCR fallback (not the primary pipeline), StoreKit 2 for functional-limit tiers (document volume, customer-prompt-profile count, Ship To alias limits, audit retention depth) — never a cloud-sync-based monetization model, since that undercuts the local-trust pitch. FileImporter/FileDocument + UniformTypeIdentifiers for strict PDF-only intake.
- Privacy: no camera/photos/contacts/location permissions needed at all — just document-picker access. Must disclose in-product and in App Privacy that document content is transmitted to whichever OCR/LLM provider is configured, unless a genuine on-device-only mode exists (Vision + deterministic parsing heuristics, lower quality, zero egress).
- Offline: review, correction, Ship To resolution, prompt editing, audit history, browsing, and export all work fully offline against local artifacts; new OCR/extraction jobs queue visibly as "awaiting processing" when the configured provider is remote and unreachable, never a silent failure.
- Dependencies: zero third-party packages by default; the only one worth even considering is a SQLite wrapper (e.g., GRDB) and only if it's clearly justified and permissively licensed.

**Real-stack risk review (what actually matters for the app being built):**
1. **SQLite concurrency.** Enable WAL mode, keep write transactions short, and treat document processing as effectively single-writer (serialize per-document background work) rather than assuming free concurrency — otherwise a real batch upload plus the frontend's status polling will produce "database is locked" errors that never show up with a single test PDF.
2. **Docker persistence.** PDFs, rendered page images, and the SQLite file must live on named/bind-mounted volumes, not the container's writable layer. This must be verified with an actual `docker-compose down && up` restart test, not just eyeballing the compose file — losing data on restart is the worst possible failure for a tool sold on local-first trust.
3. **Network/privacy honesty.** The externally-transmitted-content disclosure already decided in `detailed_discussion` must actually surface in-product, not just the README. Bind the API to loopback by default (or explicitly document why it's exposed) since there's no auth layer. API keys via env vars only, with `.env.example` committed and `.env` gitignored.
4. **Dependency/license check.** No violations found — FastAPI, React, TanStack Query, pdf.js (Apache-2.0, the deliberate PDF-rendering choice over anything GPL), SQLAlchemy are all permissive. Nothing here needs gating in tech_specs.
5. **Testability, concretely scoped:** backend fixture tests for the three-layer `needs_review` computation (with the subtotal+tax≠total case as a named fixture, not an incidental assertion), the exactly-one-repair-retry-then-`extraction_failed` path (mocked LLM client failing validation exactly twice), and Ship To matching (asserting both the match and the specific reason string against known-alias fixtures) are the three tests treated as blocking for calling this app "tested." Frontend needs component tests asserting four distinct Review renders off four mocked server statuses (since server-authoritative status is a hard architectural rule), plus at least one Playwright end-to-end test driving the full upload → flagged review → correction → Ship To resolution → approve → export loop, since that's the only way to catch integration seams unit tests miss (e.g., a status refetch not firing after a mutation).

**What must not get lost going into tech_specs:** this app's trust story depends on the pipeline degrading gracefully *and* the data surviving restarts under real concurrent use — not just passing the two demo-level acceptance tests from `app_features` in isolation. WAL-mode SQLite (or an equivalent concurrency-safe pattern) and a verified volume-persistence-across-restart test are both non-negotiable before this can be called production-ready, alongside the already-locked two tests (forced totals-mismatch review, forced double-failure into `extraction_failed`).

## Tech Specs

Both Codex and I spent this round doing the same job from two different altitudes, and the result is a real, close convergence rather than a rubber-stamp — but there's one genuine wrinkle worth naming and resolving rather than glossing over.

Where we agree completely: FastAPI + SQLite in WAL mode with a single in-process asyncio queue serializing document processing (satisfying the concurrency risk flagged in `ios_architecture_review` without reaching for external broker infrastructure, which stays correctly excluded per the non-goals list); every extracted field stored as a JSON envelope (value/confidence/source_text/page_number/needs_review/reason) rather than dozens of scalar columns; `extraction_results` as strictly append-only with incrementing run numbers and both raw failed outputs preserved for the double-failure case; `corrected_results` as a separate table with staleness computed at read time by comparing extraction-result IDs, never stored as a stale boolean that could drift; Ship To matching returning a structured `matched_on` list that a reason string is generated from (so tests assert structure, not prose); a periodic sweep resetting zombie `processing` documents; `audit_log` with a nullable document FK plus a denormalized label so entries survive deletion cascades; and a bundled single-request payload for the review screen so the operator's screen doesn't waterfall five fetches. Both of us also independently produced nearly identical API surfaces (upload, list, detail, reprocess, correction patch, approve, reopen, export, dashboard, prompts, ship-to CRUD) and the same three-blocking-test bar from the prior phase.

The one place we genuinely diverged rather than just used different words: Codex modeled the review-screen response (`ReviewDocumentPayload`) as one already-merged, UI-ready shape with a flat `header_fields: Record<string, ExtractedField>` and reasons as an array of a `FieldReason` enum; I modeled the backend's internal persistence layer in more granular relational terms (separate `Document`, `ExtractionResult`, `CorrectedResult`, `LineItem` rows) with a single loose `review_reason: string | null` per field. These aren't actually incompatible — they're different layers of the same system — but if left as two competing "canonical" contracts, parallel build lanes would genuinely invent conflicting types, which is the exact failure this phase's acceptance check exists to catch. I'm resolving it by keeping Codex's flatter `ReviewDocumentPayload`/`ExtractedField`/`FieldReason[]` shape as the one API-facing contract the frontend codes against, while keeping my more detailed internal DB models (`Document`, `Page`, `OcrResult`, `ExtractionResult`, `CorrectedResult`, `LineItem`, `ProcessingQueue`, provider protocols) as the backend-internal layer that the API layer assembles into that payload. That's not a compromise that waters anything down — it's the normal shape of a service layer that has richer internal models than its external contract, and it's consistent with the JSON-envelope field decision we both made independently.

## Final Output

**Architecture overview:** FastAPI backend, React+Vite frontend, SQLite (WAL mode, `busy_timeout` set on every connection), single-writer discipline enforced via one in-process `asyncio.Queue` consumer task (not an external broker — that stays correctly out of scope). Docker/docker-compose with named volumes for PDFs, rendered page images, the SQLite file, and exports, verified to survive a `down`/`up` restart cycle. Server-authoritative `status` enum drives all frontend rendering; the frontend never infers or caches a derived mode. Provider integrations (OCR, LLM extraction) are `Protocol`-typed so mock and real adapters are interchangeable, keeping the app fully demoable and testable with zero API keys configured.

**File/module layout:**
Backend — `app/main.py` (FastAPI wiring), `app/db.py` (engine + WAL pragmas + session-per-request), `app/models/` (one file per table: document, page, ocr_result, extraction_result, line_item, corrected_result, ship_to_record, prompt_config, audit_log_entry), `app/schemas/` (Pydantic request/response shapes, kept separate from ORM models), `app/services/pipeline/` (convert.py, ocr_provider.py, extraction_provider.py, orchestrator.py), `app/services/validation/needs_review.py` (three pure, independently-testable layers: confidence threshold, missing/mismatched source_text override, business-rule reconciliation), `app/services/matching/ship_to_matcher.py`, `app/services/export/`, `app/worker/queue.py` (single-consumer processing queue), `app/worker/stuck_reset.py` (60s sweep resetting timed-out `processing` documents with an audit event).
Frontend — `src/routes/{Dashboard,Documents,Review,PromptConfig,ShipToCatalog}.tsx` (the five locked routes), `src/api/` (one typed client file per resource), `src/types/` (hand-written TS mirrors of backend schemas), `src/hooks/useDocumentStatusPolling.ts` (polls only while ≥1 document is uploaded/processing, otherwise idle), `src/hooks/useReviewDraft.ts` (owns unsaved-edit state, exposes `isDirty`, enforces the no-silent-autosave rule from `design_handoff`).

**Data models:** Every header/line-item field is a `FieldValue`/`ExtractedField` envelope (`value`, `confidence`, `needs_review`, `reasons: FieldReason[]`, `evidence: {source_text, page_number, bbox}`) — bbox nullable per field, never per document, since some fields land from context without a literal span. `documents.status` is the single source-of-truth enum. `extraction_results` is append-only (`run_number` increments, both raw model outputs stored for the double-failure case). `corrected_results` is separate, tied to the `extraction_result_id` it was based on, with staleness computed at read time (not stored) by comparing against the document's current extraction result. `ship_to_resolution` carries a `matched_on` string list that the human-readable reason is generated from, plus alternates in the same shape. `audit_log` has a nullable document FK and a denormalized snapshot label so entries outlive a delete cascade.

**Persistence strategy:** WAL-mode SQLite with short transactions and a single background writer (the asyncio queue consumer), avoiding "database is locked" errors under concurrent batch upload + status polling. JSON-envelope columns for field data traded deliberately against SQL-level per-field querying — acceptable at MVP volume, not something to over-engineer around. Docker named/bind-mounted volumes for all artifacts, with an explicit restart test as a build-verification gate, not an assumption from the compose file looking correct.

**Testing strategy:** Backend fixture tests (blocking): subtotal+tax≠total forces `needs_review` on `total` despite high self-reported confidence; a mocked provider failing JSON validation exactly twice lands the document in `extraction_failed` with both raw outputs preserved and never a third retry; Ship To matching against known-alias fixtures asserts both the match and the specific `matched_on`/reason output. Additional backend/API tests: reprocess preserves prior corrections and flags them stale; reopen creates a new editable snapshot while the prior approved snapshot survives; startup reconciliation clears stale `processing` rows; delete cascades documents/pages/results but preserves audit rows; non-PDF/password-protected upload fails visibly; export only allowed from `approved`. Frontend tests: four mocked statuses render four distinct Review outputs; unsaved edits block navigation without explicit confirm; stale-correction banner appears near the top of Review; Documents defaults to attention-priority sort. One Playwright E2E happy path (upload → flag → correct → resolve Ship To → approve → export) and one forced-failure path (double invalid JSON), both against mock providers.

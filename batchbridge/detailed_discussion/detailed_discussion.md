**Codex (Product Strategist · User Advocate) — Round 1**

The biggest thing to settle in this phase is that BatchBridge is not just “a PO extractor with review.” It is a local exception-handling desk for a person whose day is dominated by ambiguity. The user I would keep in the room is someone like an order-entry lead at a distributor processing 40 ugly PDFs before end of day, where the cost of a wrong field is not abstract accuracy loss but a wrong shipment, a customer call, or a pricing dispute. That means any requirement that makes her stop, interpret system jargon, or wonder whether she can trust a value is a product bug, not a polish issue.

So I would lock a few requirements down now. Batch upload is required, but review remains one document at a time. The default landing point after processing should be the next document that needs attention, not a generic list. “Needs review” cannot be driven only by model confidence; it also needs deterministic rules for structurally risky cases like missing required fields, totals that do not reconcile, suspiciously empty line items, duplicate PO numbers, Ship To ambiguity, and OCR text quality below threshold. If the app only trusts the model to self-report uncertainty, it will miss the most dangerous failures. I would also require an explicit document state model that the UI and backend both use: uploaded, processing, extraction_failed, needs_review, approved, exported. Anything fuzzier will create edge-case chaos.

The edge cases worth designing around before the feature set expands are very concrete. Multi-page PDFs where line items continue across pages. PDFs with selectable text plus image scans mixed together. Rotated pages. Duplicate uploads of the same PO with a slightly different filename. A PO with header totals but no parseable line table. A line table whose math does not add up to subtotal or total. A Ship To nickname that could plausibly match two catalog entries. A customer with no prompt override configured. A prompt override that makes extraction worse than the global prompt. OCR succeeds but LLM extraction fails twice. Extraction succeeds but omits one required field. A user edits values, then reprocesses the document: you need a decision on whether manual edits are discarded, preserved separately, or compared side by side. That last one matters a lot; I would require an explicit warning and versioning behavior rather than letting reprocess silently blow away corrections.

Privacy and data handling should stay simple but strict. PDFs, rendered page images, OCR text, extracted JSON, corrected values, prompt configs, Ship To catalog data, and exports are all sensitive business records and should remain local on disk by default. The app should make it obvious which parts leave the machine: only the configured OCR/LLM provider calls, never hidden telemetry. Audit logs should record value changes, who/what changed them, and when, but in this MVP “who” can just be the local operator identity or system action. Export should be treated as a boundary event in the audit log. Deleting a document should not silently orphan derived files; either soft-delete it with audit history intact or remove the full asset bundle predictably. I would also treat local storage exhaustion as a real product concern, not an ops footnote, because scanned PDFs and rendered images accumulate fast.

The main risk register is pretty clear. First, confidently wrong extraction is a bigger product risk than visible uncertainty; mitigation is rule-based review triggers, provenance everywhere, and reconciliation checks that can override model confidence. Second, the review screen can become slower than manual entry if too much evidence is always visible; mitigation is progressive disclosure with a strong flagged-fields-first flow. Third, Ship To matching can become untrustworthy if the “reason” is vague; mitigation is deterministic explanation output that cites matched alias, customer, postal code, city, and rank against alternates. Fourth, reprocessing can destroy user trust if it overwrites manual corrections; mitigation is immutable extraction runs plus a separate corrected layer and explicit compare/reset behavior. Fifth, local-first can degrade into local clutter and brittle file references; mitigation is treating each document as a managed asset set with integrity checks and cleanup rules. Sixth, the app could drift into fake completeness if sample PDFs are too clean; mitigation is requiring at least one ugly, one ambiguous, and one failure-mode sample in-repo.

My final assumptions would be these unless someone can make a strong case otherwise. Single-user only. Desktop browser usage, not mobile. Batch upload yes, batch review no. Batch export yes. No auth, no cloud sync, no ERP push. OCR and LLM providers are configurable adapters, but the product must still be demoable locally with mocked/sample processing paths if keys are absent. The system stores every extraction attempt and every correction version rather than mutating one record in place. “Approve” means the operator has accepted responsibility for the corrected document state; only approved documents can export in the normal flow. If extraction fails twice, the document moves to a visible failed-reviewable state rather than disappearing into an error toast.

The biggest objection to my own direction is that all of this can harden the app so much around risk and auditability that it becomes heavy and joyless. That is a real danger. The mitigation is not removing safeguards; it is making the default operator path ruthless and fast: open next flagged doc, see only what matters first, fix it, approve, continue. The detail I would insist the final output preserve is this: BatchBridge only deserves to exist if it helps a busy operator catch the dangerous mistakes faster than manual entry, not merely produce impressive-looking JSON from a PDF.

**Claude (QA & Risk · Systems Thinker) — Round 1**

Coming at this from what will actually break when someone points this at a real stack of ugly supplier POs, not the demo PDF.

The thing I want nailed down before design starts is what "needs_review" actually means mechanically, because right now it's ambiguous whether it's the LLM's own self-assessment or something the system computes. It cannot just be the LLM's self-reported confidence passed through — that's exactly the "confidently wrong" failure mode flagged last phase, and if we don't resolve it now, architecture will build a pipeline that trusts a number the model itself can't be held to. My resolution: needs_review should be computed server-side by a validation layer that combines the LLM's stated confidence against a threshold, a hard rule that any field with missing, empty, or non-matching source_text gets forced into needs_review regardless of stated confidence (if the model can't point to where it read the value, that's automatically suspect), and a set of cross-field business-rule checks — line items summing to something other than the stated subtotal, subtotal+tax not equaling total, a requested delivery date in the past, a PO date after today's date, a Ship To ID present in the document that doesn't exist in the catalog at all. Those arithmetic/business checks are a different validation layer than JSON-schema validation and I don't want them conflated: schema validation asks "did the model return parseable, structurally valid JSON with the required keys," and it's what triggers the one repair retry; business-rule validation asks "is this plausible," and it should always run afterward regardless of whether schema validation passed on the first or second try, feeding needs_review independent of the repair mechanism.

On the double-failure terminal state — we said last phase there has to be one, but we didn't define it, and it's a real gap. My assumption for it: if the repair retry also fails to produce valid JSON, the document moves to an explicit "extraction_failed" status, distinct from "needs_review," visible in the document list with its own visual treatment, and the operator can still open it — landing on a review workspace with all fields blank/unpopulated but the PDF, OCR text, and raw model output (both failed attempts) fully visible, so they can hand-enter the fields from scratch rather than the document just vanishing or the UI presenting a broken/empty screen. That also gives us a natural place to expose a manual "reprocess with edited prompt" action, which matters because a customer-specific prompt tweak is the realistic fix for a document that's failing extraction repeatedly.

Edge cases on the ingestion side that need explicit answers rather than being discovered during build: password-protected or encrypted PDFs (should fail cleanly at the convert step with a specific error, not crash the pipeline); non-PDF files renamed with a .pdf extension; image-only scanned PDFs with zero embedded text layer versus digital PDFs with a text layer (does the pipeline always OCR every page regardless, or try to shortcut using embedded text — I'd say always OCR for consistency, since a hybrid path is exactly the kind of two-code-path complexity that causes subtle bugs); a single PDF file that actually contains multiple distinct POs concatenated together (common when someone scans a stack) — I think this is legitimately out of scope for MVP and should be a named exclusion rather than something design quietly tries to handle, because auto-splitting multi-document PDFs is a whole different feature; extremely long PDFs where line items span many pages and a table row gets split across a page boundary; rotated or mixed-orientation pages; non-English PO text, which should degrade to low-confidence/needs_review rather than silently failing, but full multi-language support is out of scope.

On Ship To matching, the case I want made explicit: what happens when there is no match above any reasonable threshold, or the catalog is empty (fresh install, no data loaded yet)? The UI needs a real "no match found" state that lets the operator either pick manually from the full catalog or, more importantly, create a new Ship To catalog entry inline from the document's extracted address — otherwise every new customer becomes a dead end that dumps the operator into a separate settings screen mid-review, which breaks the core loop we just spent two phases protecting. Also distinguish "customer itself is unrecognized" from "customer known, ship-to address doesn't resolve" — those are different problems with different remedies and the review UI shouldn't conflate them into one generic low-confidence flag.

The interaction I'm most worried about getting quietly wrong is reprocessing versus manual corrections. If an operator has already hand-corrected three fields and then hits reprocess (either because they edited the prompt, or just to retry), does that silently overwrite their corrections with fresh model output? That's a real data-loss edge case and it needs a resolved rule now: reprocess should generate a new extraction_results row without destroying the existing corrected_results, and the UI must make it obvious which corrections might now be stale relative to the new extraction rather than silently reapplying old corrections over new values or vice versa. Related: once a document is approved and exported, I'd treat that corrected_results snapshot as immutable — reprocessing or further edits after export should require an explicit "reopen" action that's visibly logged in audit_log, not a silent unlock, because the whole competitive wedge here is a trustworthy audit trail, and an audit trail that can be quietly rewritten after the fact isn't one.

On privacy: local-first is true for storage, but it has an asterisk that should be stated plainly rather than glossed over — page images and/or OCR text get sent to whatever external OCR and LLM provider is configured via API key, meaning PO content (pricing, account numbers, customer addresses) does leave the machine for that one call even though everything else stays local. That's fine and consistent with what was already decided, but the README and probably the UI itself should say this outright, since the sales pitch to an ops manager depends on that trust boundary being precisely and honestly drawn, not oversold as "nothing ever leaves your machine."

Other things worth flagging as real risks with named mitigations or explicit limitations rather than silent gaps: a backend restart or crash mid-pipeline can leave a document stuck in a "processing" status forever with no way to tell it's actually dead — needs either a timeout-based reset or at minimum a manual "retry" action visible on stuck documents, otherwise the document list fills with zombies after the first crash. Deletion needs a defined cascade (document delete should clean up pages, ocr_results, extraction_results, corrected_results, line_items, but audit_log entries probably should survive deletion for accountability — that's a real tension worth deciding explicitly rather than leaving to whoever implements it first). Disk usage from storing original PDFs, page images, and OCR text with no retention policy will grow unbounded; for MVP I'd just name this as a known limitation rather than build a cleanup feature, since it's explicitly not something the paid-tier story depends on. And duplicate uploads (same PDF dropped in twice) — I wouldn't block this, since false-positive dedup is worse than a harmless duplicate, but the document list should probably make it visually obvious when two documents look identical so the operator doesn't approve/export the same PO twice by mistake.

The risk to my own instinct here: if every one of these edge cases becomes a built feature, this MVP balloons way past "smaller finished app over larger half-built app." Most of these should resolve as explicit assumptions or named exclusions this phase, not scope for architecture to implement exhaustively — multi-document PDFs, disk retention, and multi-language support in particular I'd push firmly into "known limitation, not handled," while the needs_review computation logic, the double-failure terminal state, the reprocess-vs-correction interaction, and the Ship To no-match/empty-catalog state are the ones that actually have to be built correctly, because getting any of those wrong directly undermines the trust story the whole product is betting on.

- Ship To resolution distinguishes "customer unrecognized" from "customer known, address doesn't resolve," and defines a real no-match/empty-catalog state: the operator can pick manually from the full catalog or create a new catalog entry inline from the extracted address, without leaving the review flow.
- Ingestion behavior is fixed as: always OCR every page regardless of an embedded text layer (no dual code path); password-protected/encrypted PDFs and non-PDF files renamed with a `.pdf` extension fail cleanly at the convert step with a specific, visible error rather than crashing the pipeline; a stuck `processing` document (e.g., after a backend crash) gets a timeout-based reset or a manual retry action rather than becoming a permanent zombie.
- Deletion cascades documents → pages, ocr_results, extraction_results, corrected_results, line_items, but `audit_log` entries survive document deletion for accountability.

**Edge cases (explicitly answered):**
- Multi-page line items spanning a page break; rotated/mixed-orientation pages; PDFs mixing embedded text and image-only pages — all handled by the always-OCR rule above.
- Duplicate uploads of the same PO (same content, different filename): not blocked, but the document list should make visual duplicates obvious so an operator doesn't double-approve/export the same PO.
- A prompt override that makes extraction worse than the global default: no automatic detection required for MVP; the per-customer prompt editor and rerun capability is the mitigation path, not an automated regression check.
- Non-English PO text: degrades to low-confidence/`needs_review`, not a silent failure; full multi-language support is out of scope.
- **Named exclusions (not handled in MVP, stated explicitly rather than discovered later):** a single PDF containing multiple distinct concatenated POs (auto-splitting is a different feature); any disk-retention/cleanup policy for accumulating PDFs/images/OCR text (named as a known limitation); multi-language extraction quality.

**Data and privacy implications:**
- All primary artifacts (original PDFs, rendered page images, OCR text, extraction results, corrections, prompt configs, Ship To catalog, audit log, exports) remain local on disk by default.
- The one honest exception: page images/OCR text are sent to whichever external OCR/LLM provider is configured via API key for that specific call — meaning PO content (pricing, account numbers, addresses) does leave the machine for that call. This must be stated plainly in the README and ideally surfaced in-product, not oversold as "nothing ever leaves your machine."
- No hidden telemetry; export is treated as a boundary event and recorded in `audit_log`.
- Local storage growth (PDFs + rendered images + OCR text with no retention policy) is named as a known MVP limitation, not solved with a cleanup feature.

**Risk register:**
1. *Confidently-wrong extraction slips past a naive confidence check* → mitigated by the three-layer `needs_review` computation (threshold + missing-evidence rule + business-rule reconciliation), not model self-report alone.
2. *Review screen becomes slower than manual entry from too much always-visible evidence* → mitigated by progressive disclosure with a flagged-fields-first flow (carried from prior phases, reaffirmed here).
3. *Ship To "reason" strings feel vague, eroding trust in matching* → mitigated by requiring deterministic explanation output citing matched alias/customer/postal/city and rank vs. alternates (already decided; reaffirmed).
4. *Reprocessing silently destroys manual corrections* → mitigated by immutable extraction runs plus a separate corrected layer, explicit stale-correction surfacing, and reopen-required-after-export.
5. *Local-first degrades into file clutter / broken references* → mitigated by treating each document as a managed asset bundle with a defined delete cascade; unbounded growth is named as a limitation, not solved.
6. *Sample PDFs are all clean, masking failure-mode gaps* → mitigated by requiring at least one ugly/scanned, one ambiguous, and one deliberate-failure sample PDF in-repo (reaffirms prior phase's success criterion #3).
7. *Backend crash mid-pipeline leaves zombie "processing" documents* → mitigated by timeout-based reset or manual retry, named explicitly rather than left implicit.
8. *Scope creep from building every edge case as a full feature* → mitigated by explicitly naming multi-document-PDF splitting, disk retention, and multi-language support as out-of-scope limitations rather than partially-built features.

**Final assumptions:**
- Single-user, single-tenant, desktop-browser usage; no auth, no cloud sync, no ERP push.
- Batch upload yes; batch review no; batch export yes.
- OCR/LLM providers are configurable adapters; the app must remain demoable via sample/mocked processing paths if no API keys are configured.
- Every extraction attempt and every correction version is stored (not mutated in place) — the audit trail is additive by construction.
- "Approve" signifies the operator has accepted responsibility for the corrected document state; only approved documents export through the normal flow.
- Multi-document-PDF splitting, storage-retention/cleanup automation, and multi-language extraction are explicitly out of scope for this MVP.

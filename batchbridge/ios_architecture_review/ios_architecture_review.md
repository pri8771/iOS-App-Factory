Before getting into specifics, I want to flag the obvious mismatch up front: this phase's playbook is written for a native iOS/SwiftUI review — StoreKit, HealthKit-style permissions, ARKit feasibility, entitlements — and none of that applies here. On architecture: the FastAPI + React/Vite split with TanStack Query for server state (already locked in design_handoff) is sound, but the thing I'd stress-test before tech_specs locks anything is SQLite concurrency. This app has a background pipeline (convert → OCR → extract → repair-retry) running per document, a polling status endpoint the frontend hits every 2-3 seconds, and potentially several documents processing at once from a batch upload. If the backend uses plain synchronous SQLAlchemy against SQLite without WAL mode, "database is locked" errors under that read/write mix are close to guaranteed the first time someone uploads a real batch of 10+ PDFs while polling status. This needs to be decided now, not discovered during build: enable WAL mode, keep write transactions short, and seriously consider whether background processing should be a single worker thread/process per document rather than fully concurrent, given SQLite's single-writer model. This is boring but it's exactly the kind of thing that turns a demo that works with one PDF into a broken product with five.

On persistence more broadly: Docker volumes need to be named/bind-mounted correctly so that PDFs, rendered page images, and the SQLite file all survive a `docker-compose down`/`up` cycle — if the Dockerfile or compose file puts any of that inside the container's writable layer instead of a mounted volume, "production-ready local-first" quietly becomes "your data vanishes on restart," which is the single worst thing a local-first tool marketed on trust can do. I'd want this explicitly verified with a restart test, not assumed from the compose file looking right.

On the "permissions/privacy" analogue: there's no iOS-style permission system here, but there is a real privacy story that was already named in earlier phases and needs to survive into implementation — PDFs and OCR text get sent to whichever external OCR/LLM provider is configured, and that has to be stated plainly in-product, not just in the README, exactly as the detailed_discussion phase already decided. The practical risk I'd add: if the backend binds to `0.0.0.0` inside the Docker container and the compose file publishes that port, anyone else on the same local network can hit the API with zero auth — that's fine for a genuinely single-user local tool, but it needs to be a deliberate, documented choice (bind to loopback by default, or document the exposure) rather than an accidental side effect of a default Docker network config. API keys must come from env vars with a `.env.example` committed and `.env` gitignored — this is basic but it's exactly the kind of thing that gets forgotten under deadline pressure.

On dependencies: permissive-license policy is easy to satisfy here (MIT/BSD/Apache-2.0 across FastAPI, React, TanStack Query, pdf.js, SQLAlchemy) but the one library worth naming explicitly and checking is whatever PDF rendering library is used client-side — pdf.js is Apache-2.0 and is clearly the right choice over anything GPL-licensed, and it needs to be pinned deliberately rather than whatever a scaffolding tool defaults to.

On testability, this is where I want to be the most concrete, because "done" for this app was already defined two phases ago as passing two specific tests, and I don't think that's been translated into an actual test-execution plan yet. **Real-stack risk review (what actually matters for the app being built):**
1. **SQLite concurrency.** Enable WAL mode, keep write transactions short, and treat document processing as effectively single-writer (serialize per-document background work) rather than assuming free concurrency — otherwise a real batch upload plus the frontend's status polling will produce "database is locked" errors that never show up with a single test PDF.
2. **Docker persistence.** PDFs, rendered page images, and the SQLite file must live on named/bind-mounted volumes, not the container's writable layer. This must be verified with an actual `docker-compose down && up` restart test, not just eyeballing the compose file — losing data on restart is the worst possible failure for a tool sold on local-first trust.
3. **Network/privacy honesty.** The externally-transmitted-content disclosure already decided in `detailed_discussion` must actually surface in-product, not just the README. Bind the API to loopback by default (or explicitly document why it's exposed) since there's no auth layer. API keys via env vars only, with `.env.example` committed and `.env` gitignored.
4. **Dependency/license check.** No violations found — FastAPI, React, TanStack Query, pdf.js (Apache-2.0, the deliberate PDF-rendering choice over anything GPL), SQLAlchemy are all permissive. Nothing here needs gating in tech_specs.
5. **Testability, concretely scoped:** backend fixture tests for the three-layer `needs_review` computation (with the subtotal+tax≠total case as a named fixture, not an incidental assertion), the exactly-one-repair-retry-then-`extraction_failed` path (mocked LLM client failing validation exactly twice), and Ship To matching (asserting both the match and the specific reason string against known-alias fixtures) are the three tests treated as blocking for calling this app "tested." WAL-mode SQLite (or an equivalent concurrency-safe pattern) and a verified volume-persistence-across-restart test are both non-negotiable before this can be called production-ready, alongside the already-locked two tests (forced totals-mismatch review, forced double-failure into `extraction_failed`).

## Design Discussion

Round 2 closes the gap the quality gate flagged, and closes two more gaps in the same breath before anyone else had to catch them — that's good instinct, not scope creep, since all three are the same category of miss (empty states) the gate was already pointing at.

The fix on the actual flagged gap is well-reasoned: the empty Piece Library gets a bespoke, full-bleed treatment, not the reusable status card, because this screen is genuinely different in kind from the seven forgettable non-happy-path states — it's the first thing a brand-new user ever sees, and it doubles as the app's entire onboarding. The reasoning chains correctly off something already locked: since mic permission is checked live at session start (not app launch) and notification permission is requested contextually rather than up front, the same logic means mic permission shouldn't be requested at launch either — so the empty library really is the first and only thing standing between install and first action, with zero permission modals stacked in front of it. Concretely: warm illustration matching the app's visual language, one line of copy naming the actual value prop (not generic "no items" filler), and exactly one action — "Add Your First Piece." No disabled "Start Session" button sitting there as a tease; it structurally doesn't exist yet because there's nothing to attach a session to.

Two more gaps get closed alongside it, both good catches: a piece that exists but has zero sessions yet (right after creation, before the first Start Session tap) gets the status-card treatment, not bespoke — "no sessions yet" plus reliance on the screen's already-pinned Start Session button, so there's no redundant second CTA. And the widget's pre-first-session state gets a single static line ("Open Practice Loop to start your streak") in the same timeline-placeholder slot — deliberately minimal, consistent with the widget's locked "must be small" scope.

The rationale for keeping two of these on the status-card component while giving only the library a bespoke design is sound and consistent with the "don't over-build seven states" concern raised in round 1 — it's a real distinction (high-stakes/first-impression vs. low-stakes/transient), not an arbitrary line.

Nothing here contradicts anything locked upstream, and there's no competing voice to create real disagreement — this reads as a clean, complete close to the phase.

Carrying forward everything locked in round 1 (screen inventory, primary user flow, the seven-state status-card model, free/paid screen split, visual direction, accessibility notes) unchanged, this round adds the required empty-state coverage:

**Piece Library — empty state (new, bespoke, not the status-card component):** At true first launch, before any piece exists and before any permission has been requested, the list area of Piece Library is replaced entirely by a full-bleed onboarding-equivalent layout: a warm illustration in the same visual language as the rest of the app (e.g., an empty stave rendered in the same soft organic waveform stroke used elsewhere, echoing the idle-waveform motif on populated library cards), a value-prop-specific line of copy (e.g., "Add the piece you're working on, and Practice Loop remembers how each take goes") rather than generic "no items yet" filler, and exactly one action: "Add Your First Piece," opening the same create-piece sheet used elsewhere. No "Start Session" affordance appears in any form (not even disabled) — it's structurally absent, not greyed out, since a session cannot exist without a piece. This is a genuine layout difference from the populated library (whole content area replaced vs. piece-card list + toolbar add button), justified because this is a high-stakes, first-impression, one-time screen rather than a forgettable transient state.

**Piece Detail — zero-sessions state (new, uses the existing status-card component):** A piece that exists but has never had a session run shows a "no sessions yet" status card filling the space above where the trend chart will eventually render, relying on the screen's already-pinned "Start Session" button rather than introducing a second competing CTA. **Widget — pre-first-session placeholder (new):** Before any session has ever completed, the streak/last-session widget shows a single static line ("Open Practice Loop to start your streak") in the timeline-provider slot that would otherwise hold the last-session date — kept deliberately minimal, consistent with the widget's already-locked small scope.

With these three additions, every reachable screen in this phase's inventory now has an explicit empty, loading, success, and error/edge state, closing out the design-handoff prompt's requirement and the global "every screen needs empty/loading/success/error states" rule in full.

## Ios Architecture Review

Round 2 closes both gaps the gate flagged, cleanly and specifically.

The privacy-manifest question gets a real, checkable answer: exactly one Info.plist usage-description key (`NSMicrophoneUsageDescription`) — with an explicit, deliberate list of what's *not* needed (camera, tracking, location, contacts, photos) so nobody cargo-cults extra keys in later, and a correction that notification permission needs no plist string at all since it's purely a runtime `UNUserNotificationCenter` call. The `PrivacyInfo.xcprivacy` piece gets treated as the build-blocking requirement it actually is, not paperwork: file-timestamp required-reason APIs (triggered by the persistence writes) and the `UserDefaults(suiteName:)` App Group handoff both get named with their required declared-reason categories, and `NSPrivacyCollectedDataTypes`/`NSPrivacyTracking` get locked to empty/false — explicitly conditioned on "no analytics or crash-reporting SDK ever gets added later without revisiting this," which is a good instinct: naming the thing that would silently invalidate the clean story.

The backgrounding/Live Activity reconciliation also lands with real mechanics, not just a restated intention: backgrounding (scenePhase leaving `.active`) or an audio session interruption both trigger an immediate, non-resumable stop — not a pause — reusing the already-designed "too-short-to-analyze" state for anything caught early, firing an immediate Live Activity end from that same code path rather than leaning on the next-launch reconciliation sweep (which stays reserved for the actual crash/force-quit case it was built for). That's a clean two-mechanism split for two different triggers, stated plainly enough that nobody merges them into one lazy always-eventually-consistent path. And it closes with a clean, testable claim: zero `UIBackgroundModes` entries anywhere in the app.

Nothing here walks back anything from round 1 or any earlier phase — it's a direct repair of the two named gaps. No competing voice to disagree with, and the repairs are concrete enough that there's nothing left dangling.

## Final Output

**SwiftUI architecture:** MVVM using the iOS 17 `@Observable` macro, one view model per screen, no third-party architecture framework — the app is small enough that the repository-protocol boundary already locked upstream gives the testability benefit a heavier framework would otherwise justify. Dependency injection via initializers and the SwiftUI environment (no singleton service locator), so fake repositories and a fake audio provider can be swapped in for previews/tests. `AVAudioEngine` sits behind an `AudioSessionProviding` protocol for the same reason.

**Apple framework choices:** `AVAudioEngine` + `Accelerate`/vDSP for on-device audio capture and pitch/tempo DSP (autocorrelation/YIN — deliberately classical, not Core ML, so accuracy stays explainable against the invalidation criteria); SwiftData for persistence via a `ModelActor`-based fixed-cadence background write pattern; WidgetKit reading a denormalized App Group snapshot (not a live cross-process SwiftData query); ActivityKit using `Text(timerInterval:)` for the Live Activity; StoreKit 2 with `StoreKitTest`/`SKTestSession` wired into the automated test target; `UNUserNotificationCenter` for reminders, requested contextually.

**Persistence and local data plan:** SwiftData for Piece/Session behind the repository protocol (kept explicitly as a fallback point if SwiftData's background-write behavior proves flaky mid-build — swappable without a rewrite). Raw audio lives only in a transient in-memory buffer/temp file during capture and analysis, positively deleted right after derived data (pitch series, tempo points, confidence, compressed waveform outline) is extracted — enforced by a test asserting the temp directory is empty after Processing, including on a crash-during-processing path. Widget state is a small App-Group-shared snapshot (streak, last session), refreshed via `WidgetCenter.reloadTimelines` after each session.

**Permissions/privacy plan:** One Info.plist usage-description key total — `NSMicrophoneUsageDescription` — with camera/tracking/location/contacts/photos explicitly confirmed as not needed. Notification permission requested purely at runtime, contextually, with no plist string required. `PrivacyInfo.xcprivacy` declares required-reason API usage for file-timestamp access (from persistence writes) and `UserDefaults(suiteName:)` (App Group widget handoff), each with the "app's own storage" reason code; `NSPrivacyCollectedDataTypes` is empty and `NSPrivacyTracking` is false, conditioned on never silently adding an analytics or crash-reporting SDK later without revisiting this. Recording is foreground-only: backgrounding (scenePhase leaving `.active`) or an audio-session interruption triggers an immediate, non-resumable stop (not a pause) — whatever was captured becomes the session and flows through Processing normally, reusing the existing "too-short-to-analyze" state if it's brief. This stop synchronously ends the Live Activity in the same code path; the next-launch reconciliation sweep remains reserved solely for the force-quit/crash case. Zero `UIBackgroundModes` entries anywhere in the app, testable via an Info.plist assertion.

**Dependency/license decision:** Zero third-party dependencies — no architecture framework, no DSP library, no analytics/crash-reporting SDK — trivially satisfying the permissive-license policy and keeping the privacy-manifest story clean.

**Testing implications:** A pure, UIKit/AVFoundation-free Swift DSP module (samples + sample rate → pitch series/tempo points/confidence) unit-tested against a checked-in fixture corpus (clean in-tune, clean out-of-tune, ambient noise, silence, a double-stop) — the automated version of the manual accuracy/confidence criteria locked earlier. `StoreKitTest` exercises the paywall gate (block/unlock/lapse-relock) on every CI run. A temp-directory-empty assertion enforces the no-raw-audio-storage privacy claim. An Info.plist assertion checks exactly one usage-description key and no `UIBackgroundModes`. Live Activity and widget-timeline behavior aren't meaningfully unit-testable and get a standing manual QA checklist instead (kill app mid-session → confirm Live Activity ends; complete a session → confirm the widget refreshes within a reasonable window).

My fix: one `SummaryView` backed by a `SummaryContent` enum (`.free(duration:, note:)` / `.paid(pitchScore:, tempoResult:, confidence:, annotation:)`), sharing one layout shell (header, content region, CTA region) so the chrome can't silently diverge even though the content genuinely differs. Same logic applies to the reusable status-card component from the design-discussion phase — if it's built as "pass in whatever strings you want per call site," every one of the seven-plus locked states (mic denied, mic busy, low-signal, too-short, multi-pitch, first-session, locked-piece, now plus the two empty states) will get bespoke ad hoc content wired in by hand at the point of use, and someone will eventually add an eighth state and forget a case. I want this locked as an enum, `StatusCardKind`, exhaustively switched over, so the compiler — not a reviewer — catches a missing state.

The Recording screen is the one place I'd push hardest, because it's the one screen doing three jobs at once: owning a live AVAudioEngine session, rendering at frame rate, and incrementally persisting. On persistence — "incremental writes so force-quit doesn't lose the session" was locked upstream, but if that's implemented as a SwiftData write on every audio callback or every UI tick, it will either thrash the persistent store or, worse, get attempted from a background audio thread against a main-thread-bound ModelContext and crash. Concretely: writes should happen on a fixed cadence (a few seconds), off a background `ModelActor`/context, not the callback thread, with a final consolidating write at Stop — and this cadence, not "constantly," is the thing the handoff needs to state explicitly so nobody picks a naive default under time pressure. Same skepticism on the live waveform itself: feeding raw per-sample audio buffer data straight into a `@State` array and re-rendering SwiftUI on every tap callback will visibly stutter. It needs a decimated ring buffer (a few hundred amplitude points, updated at something like 20–30Hz off a timer, not every audio frame) drawn with `Canvas`, not per-sample `Path` points. And the haptic channel — already correctly restricted to haptic-not-speaker-audio to avoid mic contamination — needs its own rate limit and hysteresis band, or a live pitch signal that hovers right at the in-tune/out-of-tune boundary will fire a haptic buzz on every frame, which reads as a malfunctioning phone, not a coaching cue. I'd lock a minimum inter-pulse interval and a debounce band around the threshold now, because that's exactly the kind of detail that gets "figured out during build" and then feels bad in the demo.

I found a real gap in the trend/annotation logic nobody's caught yet: a piece can have its target tempo added or removed after it already has session history — nothing in any locked phase forbids editing a piece's target tempo later. If that happens, the piece's earlier sessions were scored in self-consistency mode and its later ones in target-tempo mode, and those two numbers are not on the same scale. The "best score yet" annotation and the trend line as currently specified would silently compare across two different measurement systems and produce a comparison that's technically computed but semantically meaningless — precisely the confidently-wrong failure mode this whole project has otherwise been careful about.

Now the skeptic part, because this is where I think the plan is most likely to quietly fail later.

First, "on-device audio analysis (pitch tracking, tempo-stability detection)" is being treated in the spec as a solved bullet point, but it's actually the single highest-risk piece of engineering in the whole app, and the invalidation criteria the user wrote ("if real instrumentalists find feedback inaccurate, rework before the paid tier leans on it") is a tacit admission of that. My objection to shipping this as currently scoped: a from-scratch pitch-detection and tempo-stability algorithm, built inside an MVP build cycle, for polyphonic or noisy real-world instrument audio, is a research problem, not an implementation task. Monophonic instruments (voice, single-line winds/brass/strings) with a YIN or autocorrelation-based pitch tracker are tractable on-device with AVAudioEngine taps and vDSP/Accelerate. Anything polyphonic (piano, guitar chords) is not tractable to do well in an MVP timeframe. So a decision rule I want locked now, not discovered during build: v1 must explicitly pick monophonic instrument families only (voice/winds/brass/strings — one note at a time), and the app must say so on the tin, because "works for one or two instrument families at launch" in the spec is doing a lot of quiet load-bearing work that a later phase could easily gloss over by just picking "guitar and piano" without realizing that's the hard case, not the easy one.

Second, the ambient noise risk the spec names is real and I don't think "graceful handling" is enough as a requirement — it needs a concrete, testable definition of what graceful degradation looks like: a confidence/signal-quality indicator that visibly downgrades feedback when SNR is poor, rather than silently producing wrong pitch numbers that look authoritative. A wrong-but-confident pitch readout is worse than no readout, because it's the exact failure mode the invalidation criteria warns about (users trusting inaccurate feedback). Production-readiness for this app has to include: the app must be able to detect and communicate "I can't tell" as a first-class state, not just success/error in the generic sense.

Third, tempo-stability detection needs a reference to compare against — is there a target tempo/rhythm grid the user defines per piece (e.g., set a BPM, maybe tap in a rhythm), or is the app inferring "stability" purely from the played audio's own self-consistency with no ground truth? The spec doesn't say, and this materially changes both the UX (does the user set up a piece with a tempo before playing) and the algorithm (self-consistency variance detection vs. deviation-from-reference detection). This needs to be decided explicitly in an early build phase, not left ambiguous, because it changes what "per-piece" setup requires.

Fourth, on the monetization/paywall boundary: StoreKit 2 in an MVP that will very likely be sideloaded/simulator-tested rather than shipped to the App Store this session — the paywall needs to be real code (real StoreKit 2 product fetch/purchase/entitlement flow) but the actual App Store Connect product configuration won't exist, so testing has to rely on StoreKit configuration files/local testing, not a live sandbox purchase. Production-readiness definition for this piece specifically means: the code path is correct and would work with real product IDs, verified via StoreKit's local testing tools, not "it compiles."

Fifth, Live Activities plus background audio recording is a real interaction risk — Live Activities have strict update-budget and lifecycle rules, and if the design assumes a Live Activity that live-updates pitch/rhythm feedback continuously during a session, that will likely blow through ActivityKit's update frequency limits. Decision rule: the Live Activity should show coarse session state (elapsed time, piece name, maybe streak), not live per-note feedback — live per-note feedback belongs in the foreground UI only. If a later phase designs a Live Activity that promises real-time pitch accuracy on the lock screen, that's a claim I'd flag as very likely to break against real ActivityKit constraints.

For production-readiness as a definition I'd hold every later phase to: it means the four core workflows run end-to-end on a real device or simulator with real microphone input (not mocked audio), every screen in those workflows has all four states, the pitch/tempo analysis has a stated, honest accuracy scope (which instrument families, what confidence signaling, what it explicitly does not handle), the paywall gate is functionally real even if App Store Connect products aren't live, data persists locally across launches, and there's been an explicit build-verification pass plus a final review that agree on what works versus what's still limited — per the global rule, nobody gets to declare this done just because a spec was written and code compiles.

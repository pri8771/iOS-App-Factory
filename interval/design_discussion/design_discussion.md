**Codex (Product Strategist · Closer) — Round 1**

The design needs to make one thing feel true within five seconds of launch: this is not a generic timer with analytics attached, it is a timer that will eventually earn the right to tell you a better next interval than you would have picked yourself. That means the home screen has to carry almost all of the product weight. I would center the app on three tabs, not four: Home, History, Settings. Insights should not be its own destination in v1 because the product promise is immediate recommendation trust, not retrospective browsing. If we later add a paid insights surface, it can grow out of Home or History without changing the base architecture.

Home is the primary screen and it needs four clearly different states. In default state, it shows the fixed 25/5 recommendation, a calm “learning hasn’t started yet” explanation, and a visible countdown like “5 completed sessions until adaptation starts.” In learning state, it still keeps the recommendation fixed at default, but now shows lightweight pattern signals without claiming adaptation, like “You complete shorter afternoon sessions more often.” In adapted idle state, it shows the current recommended focus and break length, a one-line why, and a clear override affordance before start. In active session state, the screen collapses into a large timer, progress ring, current planned duration, stop/end controls, and a smaller secondary line reminding the user whether this session matches or overrides the recommendation. Break state is the same screen, visually calmer and clearly distinct so users never wonder whether they are in focus or recovery. Error or degraded states also live here: notification permission denied, unresolved session recovered on relaunch, store reset after corruption. Those cannot be buried elsewhere because they directly affect trust in the timer.

The start flow should be one sheet, not a separate screen. Tap the recommendation card, and a bottom sheet opens with focus duration stepper, break preview, a short explanation of whether this matches the engine recommendation, and one primary start button. That keeps override lightweight instead of turning configuration into a mini settings workflow. After a session ends, the outcome capture should also be a sheet, not a route change: completed, stopped early, interrupted, or canceled if it was effectively abandoned, plus optional energy rating. If the app auto-reconciles a session on relaunch, it should not dump the user into a modal wall. It should show a non-blocking banner at the top of Home saying the last session was auto-logged and can be adjusted, and tapping that banner opens the same outcome sheet prefilled with the provisional state. That keeps the recovery path honest without stalling the next session.

History should be a real verification screen, not a decorative feed. Each row needs planned duration, actual elapsed time, outcome tag, whether it matched or overrode the recommendation, and time bucket context in plain language. Empty state should explain what will appear here after the first session. Success state is the session list plus a simple filter or segmented control for all/completed/early stop/interrupted. ## Final Output

**Screen inventory:** Three tabs — Home, History, Settings. No separate Insights tab in v1 (it lives as a "This week" strip at the top of History). Two modal/sheet surfaces off Home: the start-override sheet and the outcome-capture sheet. One full-screen active-session view (focus and break are the same screen, distinct accent/color treatment).

**Primary user flow:** Launch → Home shows current recommendation (or default) with learning-state chip and, once adapted, a one-line "why" directly under the number → tap the recommendation to open the override sheet (stepper + break preview + start button) or tap Start directly → full-screen session view (large monospace countdown, plain ring, confirm-to-stop control) → natural end or manual stop → two-tap outcome sheet (five-value taxonomy + optional energy rating) → auto-transition into break screen (skippable via subdued text link, not a competing button) → back to Home, updated. A first-run-only dismissible explainer card sits above the fold on first launch and never reappears.

**State model per screen:**
- *Home:* default (fixed 25/5, zero personalization language), learning (chip shows progress toward the 5-session threshold, number frozen), adapted (number can move, why-line populated, optional adapted-state history strip under the number), reconciliation-pending (non-blocking banner overlaid on whichever state above, expands into prefilled outcome sheet on tap), degraded-permission (persistent but calm notice, not a nag), store-corrupted (honest full-screen reset explanation, not a crash or silent wipe).
- *Active session:* running (focus), running (break, distinct accent), notification-permission-degraded sub-state (small persistent in-session note).
- *History:* empty (explains what will appear and why there's nothing yet), populated (list with planned/actual duration, outcome tag, match-vs-override flag, time bucket; optional "recommendation moved because…" card pinned at top once adapted), store-error state.
- *Settings:* normal, subscription-active, subscription-lapsed (locked rows with lock glyph + explanation, data preserved per the soft-lock decision), notification-permission status shown plainly.

**Visual direction:** Precision-instrument aesthetic, not wellness-pastel or gamified-neon — near-monochrome base, one accent hue used to encode confidence/state (cool neutral in default, gradually warming through learning into adapted) and always backed by redundant text/shape, never color-only. Tabular/monospaced numerals for the countdown so digits don't jitter. Duration is the visual hero; the "why" line is treated as important copy, not a footnote. Minimal, meaningful motion only (state transitions, ring completion, banner slide) — fully disabled under Reduce Motion.

**Accessibility notes:** Milestone-based VoiceOver announcements only (start/halfway/one-minute/end), never per-second. Outcome buttons are plain-language labeled and reflow to a vertical stack at large Dynamic Type rather than truncating. Any optional history-strip visualization gets a single summarized accessible label, never per-segment enumeration. Confirm-to-stop control has a real 44pt target with deliberate spacing from the break-skip link. Color is never the sole carrier of learning-state or outcome-state information.

No open disagreement to resolve — the phase's required outputs (screen inventory, primary flow, state model, visual direction, accessibility notes) are all covered with specific, buildable detail, and the one added idea (session-history ring) is explicitly scoped so it can be cut without touching the non-negotiable plain-language "why."

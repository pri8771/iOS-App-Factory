## Final Output

**Screen inventory:** Home, active-session (focus + break, same screen different accent), start-adjustment sheet, outcome-capture sheet, History, Settings, paywall sheet, delete-all-data confirmation sheet. Three tabs only: Home, History, Settings.

**State ownership (architecture, load-bearing for the design to be buildable correctly):** A single `@MainActor @Observable SessionCoordinator` is the sole writer of a `SessionPhase` enum (`idle(Recommendation)`, `running(SessionRecord)`, `onBreak(SessionRecord)`, `awaitingOutcome(SessionRecord)`, `reconciliationPending(SessionRecord, provisional: Outcome)`). Every screen renders as a pure function of this one phase — no independent local state that can drift out of sync. The active-session view is a `fullScreenCover(item:)` bound to this phase, never a `NavigationStack` push (push implies a back-swipe out of a running timer, which must not exist). Timer *truth* is always `plannedEnd = startTimestamp + plannedDuration` recomputed from `Date()`, never a paused in-memory counter, so background/lock/relaunch all agree. **Reconciliation fix:** the moment a session transitions running → ended (natural completion or manual stop), a provisional outcome is written immediately using the same elapsed-vs-planned heuristic already agreed for backgrounded sessions — before the outcome sheet appears. The sheet becomes confirm-or-correct, not the only write path, which is also what makes it safe to be swipe-dismissible without breaking the no-null-outcome guarantee.

**Home states (seven, one stable vertical layout):** default (25/5, `Default` chip, "0 of 5 completed sessions toward adaptation"), learning (`Learning` chip, progress count, numbers still frozen, optional soft pattern-signal note), adapted (hero number can move, `Adapted` chip, mandatory one-line why directly under the number, optional evidence strip), plus three system-honesty overlays that share one slot and never stack — store-reset (highest priority) → reconciliation banner → notification-degraded notice (lowest). Evidence strip: a compact row of 5–8 session marks (solid/tapered/broken by outcome), rendered with `Canvas` (not stacked `Shape` views, to avoid render/hit-testing cost for a non-interactive glanceable element), only in adapted state, with one summarized VoiceOver label rather than per-mark enumeration.

**Active session / break:** large `.monospacedDigit()` countdown, plain unadorned progress ring (deliberately simpler than the Home evidence strip — this moment needs calm, not proof), a small "Recommended" vs "Override" + planned-duration label, a one-line trust note when overridden, confirm-required Stop control, and a persistent (non-alarming) note when notifications are denied. Break reuses the layout with a shifted accent and a low-emphasis "Skip break" text action, never a competing button.

**Sheets:** Start-adjustment — recommendation shown above an editable "Starting: X min" value, tactile ruler-style picker, explicit match/override copy. Outcome capture — large outcome buttons (Completed/Stopped early/Interrupted primary; "Mark as accidental start" secondary/floor-gated), optional 5-level energy row, pre-filled and swipe-dismissible when a provisional outcome already exists. Paywall — default/purchasing/success/failed-purchase/restore states, product-specific unlock language, live preview generated from the user's own adapted-state data where available, restore as a secondary text action. Delete-all-data — dedicated confirmation sheet stating exactly what's erased and that subscription status is untouched, destructive-styled confirm.

**History:** real "This week" strip (best window, trend) usable by free tier, one locked teaser row below it opening the paywall, session rows anchored on planned duration with outcome badge and match/override flag, segmented All/Completed/Early-stop/Interrupted filter, value-reinforcing empty state, explicit store-error state.

**Settings:** Upgrade row, notification status, precise local-storage privacy copy, delete-all-data row, subscription status including lapsed (locked rows, data preserved, resubscribe-to-unlock explanation).

**Design tokens:**
- *Type:* Dynamic Type text styles throughout; `.monospacedDigit()` mandatory on all timing numerals; `.title2`-class for Home's hero duration; `.body` for the why-line with ≥3-line layout headroom reserved to prevent clipping/relayout jumps.
- *Color:* semantic tokens only, light/dark/high-contrast variants — `background.primary`, `background.elevated`, `text.primary`, `text.secondary`, and exactly three fixed named accent stops (`accent.default`, `accent.learning`, `accent.adapted` — cool slate-blue → warming → confident teal/amber-green) with a single cross-fade on state transition, not a continuous interpolated formula (rejected as untestable and hostile to Increase Contrast/Reduce Transparency). Five semantic `outcome.*` colors shared identically between History badges and the evidence strip.
- *Spacing/shape:* 4/8/12/16/24/32 spacing scale; two-value corner radius (12 cards, 20 sheet top corners).
- *Motion:* named duration constants (`motion.stateChange` ~0.25s, `motion.sheetTransition` system default, `motion.ringComplete` ~0.4s), all routed through one shared Reduce-Motion-aware helper rather than checked ad hoc per call site.

**Component inventory:** RecommendationCard, LearningStateChip, WhyLine, SegmentedHistoryStrip (Canvas-rendered), CountdownDisplay, SessionRing, OutcomeButtonGroup, ReconciliationBanner, DegradedModeNotice, HistoryRow, LockedInsightRow, PaywallPlanCard, DestructiveConfirmSheet, PurchaseCTA, RestoreTextAction, shared EmptyStateView/ErrorStateView pair (reused across History, Settings, and store-corrupted screen rather than bespoke per screen). Each with default/pressed/disabled/loading/accessibility-focused variants where applicable.

**Accessibility:** color never the sole state signal (chips carry text always); VoiceOver milestone-only announcements (start/halfway/one-minute/complete); outcome buttons reflow vertically at large Dynamic Type; evidence strip gets one summarized accessible label; Stop control has a large hit target with deliberate spacing from Skip-break to prevent mistaps that would poison adaptation data; Reduce Motion replaces all state-change animation with instant transitions via the shared helper.

**External design decision:** No Claude Design / external mockup pass for the interactive build — build directly from this handoff, since the visual system is already pinned to concrete tokens and SwiftUI-specific component behavior that a static design tool can't faithfully carry back (Dynamic Type, safe-area sheets, Canvas rendering, Reduce Motion routing). If a visual mockup pass is wanted later purely for App Store listing/marketing screenshots, use this narrower prompt (not the full interactive system):

> Design static hero screenshots for an iPhone app called Interval, a premium local-first focus timer that adapts session length and break length based on real session history. The UI should feel like a precision instrument, not a wellness app or gamified Pomodoro clone — graphite-and-bone base palette, one restrained accent family shifting from cool (default) to confident teal/amber-green (adapted), tabular/monospaced numerals for all timing. Produce marketing-quality static shots of: Home in its adapted state (hero duration, one-line "why," evidence strip), the active focus session screen, and the paywall sheet showing a live preview built from real user data. Avoid purple gradients, neon productivity cues, and illustration-heavy styling. These are static assets for store listing purposes only, not an interactive design system — the working app is already built from a separate SwiftUI-native token and component spec.

## Final Output

**SwiftUI architecture recommendation:** One app target. A single `@MainActor @Observable SessionCoordinator` is the sole owner/writer of session lifecycle state (idle/running/break/awaiting-confirmation/reconciliation-pending), with focused services behind it: `RecommendationEngine` (pure, deterministic, framework-light — Foundation only, no SwiftUI/StoreKit/SwiftData imports, so it stays unit-testable and reusable behind a future sync layer), `NotificationScheduler`, `SubscriptionStore` (StoreKit 2 entitlement state), and `SessionRepository` (a protocol in front of SwiftData, not because multiple backends are needed in v1, but because corrupted-store recovery and a future CloudKit sync layer both need that seam to exist now rather than be retrofitted). Timer truth is always timestamp-derived (`plannedEnd = start + plannedDuration`, recomputed from `Date()`), never a live in-memory counter — this is the one invariant both agents said must not get weakened anywhere downstream.

**Apple framework choices:** SwiftUI, Observation, SwiftData, UserNotifications (`UNUserNotificationCenter`), StoreKit 2, `Canvas` for the evidence-strip rendering, XCTest/Swift Testing. Nothing else — no HealthKit, Calendar, microphone, Focus status, Screen Time APIs, or Background Modes entitlement, since none are load-bearing for the product promise and each adds review/privacy surface for no user-facing return.

**Persistence and local data plan:** SwiftData as the store, behind a `SessionRepository` protocol. Minimal schema: `Profile`, `SessionRecord` (start timestamp, planned focus/break duration, actual end timestamp if known, actual elapsed seconds, outcome, match-vs-override flag, recommendation inputs snapshot, optional energy rating, coarse time bucket, profile ID, and a provenance field — `userConfirmed` / `autoReconciled` / `autoReconciledThenEdited` — so reconciliation behavior is auditable and testable), optionally `RecommendationSnapshot`, and a tiny `AppSettings` record for one-time banner/permission-prompt flags. No task names, notes, or behavioral-surveillance-adjacent fields. Corrupted-store recovery has a named mechanism: catch the `ModelContainer` init failure at app launch, delete the underlying store file, reinitialize with a fresh empty container, and set a flag the first screen reads to show the honest "started fresh" message — tested by deliberately corrupting the SQLite file on disk. One schema-migration test (`VersionedSchema`/`SchemaMigrationPlan` round-trip) is cheap insurance against SwiftData's early-iOS-17 migration rough edges, even though the v1 schema itself is simple.

**Permissions/privacy plan:** Only local notification permission, requested contextually at first "Start" tap (not cold launch), with the request call site guarded against re-prompting after denial. No Background Modes capability — stated as a deliberate decision, not an emergent property, since timestamp-based reconciliation on relaunch/foreground already delivers the "survives backgrounding" promise without it. Notifications are cancelled by a stable per-session UUID identifier, never a blanket cancel-all, to avoid accidentally nuking a still-valid pending notification during edge-case transitions (e.g., early-stop landing right as break would auto-start). Privacy copy stays precise: data stays on-device and in the user's own device backups, not an absolute "never leaves your phone" claim.

**Dependency/license decision:** Zero third-party packages. Every must-have is fully covered by first-party frameworks; nothing here (evidence strip, paywall, persistence, notifications) needs an external dependency, and each one avoided is one less license-review and build-fragility risk for a portfolio-scale build.

**ML/AR/LLM justification:** None used, and that's the correct call — the adaptation engine is a deterministic, explainable heuristic over local history, which both beats a trained model on this product's actual differentiator (inspectability: "show the signal behind the number") and works from session 1, unlike an ML model that would need training data a brand-new user doesn't have yet.

**Testing implications:** Heaviest investment on the `RecommendationEngine` (pure function over `[SessionRecord]`, tested against every clamp/step-cap/degenerate-floor boundary with synthetic histories) and the reconciliation matrix (the existing 10/50/95/100%+ elapsed-time cases, plus the outcome-sheet-open-when-killed case flagged in the design-handoff phase, verified as its own explicit test rather than assumed). Second tier: repository/coordinator integration tests (corrupted-store force-recovery, entitlement-lapse-preserves-data), StoreKit flows via `StoreKitTest`/`SKTestSession` for repeatable CI coverage of purchase/restore/failure-types/lapse (real sandbox reserved for a final pre-submission pass only), and a mechanical/structural check that no networking types are reachable from the timer/adaptation module boundary. UI/sheet-state coverage is explicitly acknowledged as manual/spot-check given MVP scope — named honestly now rather than implied as fully automated later.

## Tech Specs

Looking at round 1, both Codex and Claude are proposing the same underlying architecture — a `SessionCoordinator` owning `SessionPhase`, a pure `RecommendationEngine`, `SessionRepository` in front of SwiftData, notification/subscription services — and neither is fighting the other on strategy. Claude's round did what tech_specs is supposed to do: it turned a few of Codex's "this should probably work like X" statements into concrete, testable mechanisms.

Two real refinements stand out. First, Claude split the engine's input type cleanly: rather than feeding the recommendation engine anything SwiftData-flavored, there's now a plain `SessionSnapshot` struct that's the *only* thing `RecommendationEngine` ever sees, with `SessionRecord` promoted to being the actual SwiftData `@Model` and the repository doing the translation at the boundary. Codex had already separated `Domain/SessionRecord.swift` from `Data/SwiftDataModels.swift` in the file layout, so this isn't a contradiction — it's Claude naming the same split Codex already implied, just more concretely (and matching the "zero SwiftData imports in the engine" invariant from `ios_architecture_review` more literally). Second, Claude caught a real gap in the provisional-outcome-write rule: applying the same elapsed-vs-planned heuristic to a manual stop as to a relaunch reconciliation would mislabel someone who just tapped Stop as "expired while away," which is exactly the kind of small dishonesty this app can't afford. The fix — a `SessionEndTrigger`-aware `ReconciliationPolicy` that treats manual stop, natural completion, and relaunch reconciliation as three distinct cases sharing the same constants — closes that without touching the "auto-write before the sheet" guarantee everyone already agreed to.

Everything else stacks: dropping the optional `RecommendationSnapshot` entity in favor of folding snapshot fields onto `SessionRecord` directly (smaller finished schema, consistent with Codex's own "persist facts, not denormalized summaries" instinct), the documented time-bucket-computed-once-at-write rule, no `@Query` splashed into views so History goes through the repository/coordinator like everything else, and the `IntervalCore` local-package idea for making "framework-light" a compiler-enforced boundary rather than a convention. Gemini didn't participate again.

No open disagreement worth another round — this reads as one spec built collaboratively, not two competing ones.

## Final Output

**Architecture overview:** One app target plus one local Swift package, `IntervalCore` (zero external dependencies, no `import SwiftUI`/`StoreKit` allowed), holding the domain enums/structs, the SwiftData `@Model` types, `SessionRepository`, and `RecommendationEngine`. This makes the "engine and persistence stay framework-light" invariant a build error if violated, not a review-time hope. The app target holds `SessionCoordinator` (`@MainActor @Observable`, sole writer of `SessionPhase`), `NotificationScheduler`, `SubscriptionStore`, `HistoryViewModel`, and all SwiftUI views. Timer truth is always `startTimestamp + plannedDuration` recomputed from `Date()`; the moment a session transitions running → ended (manual stop, natural completion, or relaunch reconciliation), a provisional outcome is written immediately via `ReconciliationPolicy`, before any confirm/correct UI appears — and that policy is now trigger-aware (`SessionEndTrigger`: `naturalCompletion` / `manualStop` / `appRelaunchReconciliation`) rather than one blind elapsed-fraction formula for all three cases.

**File/module layout:** `IntervalCore` package — `Domain/` (`SessionOutcome`, `LearningState`, `OutcomeProvenance`, `TimeBucket`, `SubscriptionTier`, `SessionEndTrigger`, `SessionSnapshot`, `EvidenceMark`, `Recommendation`, `RecommendationEngine`, `ReconciliationPolicy`), `Data/` (`SessionRecord`, `Profile`, `AppSettingsRecord` as SwiftData `@Model`s, `RepositoryError`, `SessionRepository` protocol + SwiftData implementation, `AppLaunchBootstrap`). App target — `Services/` (`SessionCoordinator`, `NotificationScheduler`, `SubscriptionStore`, `Clock`), `UI/Home`, `UI/Session`, `UI/History` (`HistoryViewModel`, `HistoryFilter`), `UI/Settings`, `UI/Shared` (`EmptyStateView`, `ErrorStateView`, `ReduceMotionAnimator`). Tests split into `RecommendationEngineTests`/`ReconciliationPolicyTests` (pure, no SwiftData setup), `SessionRepositoryTests` (incl. forced-corruption recovery), `SchemaMigrationTests`, `NotificationSchedulerTests`, `SubscriptionStoreTests` (via `StoreKitTest`), and one thin `UITests` target.

**Data models:** `SessionRecord` (SwiftData model) carries the full audit trail — planned/actual timing, the recommendation snapshot at start time (folded in directly, no separate `RecommendationSnapshot` entity), outcome, provenance, override flag, energy rating, time bucket computed once from `Calendar.current` at write time and never retroactively recomputed. `SessionSnapshot` is the plain `Codable` mirror that's the *only* type `RecommendationEngine` ever accepts. `Profile` exists from day one (single auto-created instance free tier, multi-profile paid-gated at the view layer only). `Recommendation` carries focus/break duration, learning state, explanation, evidence marks, and adaptation progress — computed once per session write and cached on the coordinator, never recomputed inside a view `body`.

**Persistence strategy:** SwiftData behind `SessionRepository`. `AppLaunchBootstrap` catches `ModelContainer` init failure, deletes the store file, reinitializes empty, and flags `AppSettingsRecord.storeWasReset` for a one-time honest "started fresh" message. No `@Query` used directly against `SessionRecord` anywhere in the view layer — `HistoryViewModel` and the coordinator are the only read/write paths, so the repository seam stays real (needed for both corrupted-store recovery and a future CloudKit layer).

**Testing strategy:** Heaviest investment in `IntervalCore`'s own pure tests (engine clamp/threshold/step-cap boundaries and the full reconciliation matrix, including the trigger-differentiated cases, against synthetic `[SessionSnapshot]` arrays — no `ModelContext` needed). Second tier: repository/bootstrap tests (forced corruption, delete-all-data across all profiles), one `VersionedSchema` migration round-trip, notification cancel-by-UUID and no-reprompt-after-denial tests, and `StoreKitTest`/`SKTestSession` coverage of purchase/restore/lapse/failure-types. UI/sheet coverage stays honestly manual/spot-check given MVP scope, as already agreed in prior phases.

## Final Output

**Milestones**

1. **Foundation & domain contracts** — Scaffold the Xcode project and the `IntervalCore` local package with its zero-import rule enforced structurally (no `SwiftUI`/`StoreKit`/`UserNotifications` reachable from that target). Land the domain enums, `SessionSnapshot`, `Recommendation`, and — critically — a single constants file holding every tunable (10–90 min focus clamp, 3–20 min break clamp, 10%/5-min step cap, 60s degenerate floor, 5-session adapted threshold, 20-session/unbounded rolling windows) so no consuming site can hand-copy its own drifting copy. In parallel, land the SwiftData models (`SessionRecord`, `Profile`, `AppSettingsRecord`), the `SessionRepository` protocol + implementation, and `AppLaunchBootstrap` — the corrupted-store recovery must delete the store's main file *and* its WAL/SHM companions as one atomic set, not just the primary file, or a "fresh" reinit can inherit orphaned WAL corruption. *Exit:* app launches to Home showing the default 25/5 recommendation with a seeded default profile; a deliberate on-disk corruption test proves clean recovery; a structural test proves zero networking types are reachable from the core module.

2. **Trustworthy session runtime** — Two tracks converge here. Track A: `RecommendationEngine` + `ReconciliationPolicy` as pure logic, unit-tested against synthetic `SessionSnapshot` arrays (no SwiftData involved). Track B: the `SessionRecord` → `SessionSnapshot` translation function, with its own explicit integration test (build a real SwiftData record, translate it, run it through the engine, assert the output) — this is the one seam where a persistence mistake would silently masquerade as a wrong recommendation, so it gets verified in isolation before the coordinator, notifications, and UI all start depending on it. Concurrently — because `SessionCoordinator` only needs `NotificationScheduler`/`SubscriptionStore` as protocols, not concrete implementations — build the real `UNUserNotificationCenter` scheduler (per-session-UUID cancellation, no-reprompt-after-denial guard) and the StoreKit skeleton in parallel with the coordinator itself, wiring them in once both sides are ready. The coordinator implements timestamp-truth timing, trigger-aware provisional-outcome writes before any sheet appears, and relaunch reconciliation. Elapsed-time math everywhere (engine, reconciliation) must be raw `Date`/`TimeInterval` arithmetic, never calendar-component subtraction — a DST-spanning session computed via naive hour subtraction would be wrong in a way no demo would ever catch. *Exit:* a scripted scenario matrix passes **on a real device, not just simulator** (start→lock→elapse-past-end→relaunch; start→stop-at-40-seconds; outcome-sheet-open→force-quit→relaunch; break transition; notification fires and cancels by the correct UUID).

3. **Adaptation made visible** — Wire the engine into the coordinator (recomputed once per session write, cached, never recomputed in a view body), connect learning-state UI, evidence-strip data, and explanation strings. Build one seeded debug dataset that visibly moves Home's recommendation and one-line "why." *Exit:* full synthetic-history clamp/threshold/step-cap test suite passes, and the seeded dataset visibly changes the on-screen recommendation — closing the specific risk that "the math works in tests but the UI still looks static."

4. **Product surfaces, in priority order** — Home, the active-session `fullScreenCover`, the start-adjustment sheet, and the outcome sheet first, since they're the only screens that exercise the coordinator's write paths — verify these manually on-device before moving on. Then History (filters, rows, "This week" strip), Settings, and every empty/error state, plus accessibility work scoped to this app specifically (VoiceOver milestone-only announcements, Dynamic Type headroom on the why-line, one summarized accessible label for the evidence strip) rather than generic box-checking. *Exit:* every reachable state on every screen is enumerated and reachable; accessibility checks pass as explicit test cases, not spot checks; notification-denied degraded mode is honest and non-repeating.

5. **Monetization** — Wire the StoreKit skeleton (already scaffolded in milestone 2) into the real paywall sheet (default/purchasing/success/failed-by-type/restore states), the locked History teaser row, and entitlement-gated multi-profile/insights views. Subscription checks stay strictly at the feature-view layer — never inside session start. Delete-all-data wipes session history, profile data, and learning state across all profiles, and leaves `AppSettingsRecord`'s onboarding/permission-prompt flags untouched, so clearing history never silently re-triggers the first-run explainer or re-prompts for notifications. *Exit:* `StoreKitTest`/`SKTestSession` automated coverage of purchase, restore, restore-with-nothing-to-restore, and every documented failure type; lapse verified to soft-lock (preserve, don't delete) paid data; one real sandbox pass reserved for the pre-submission window only.

6. **Integration hardening & final verification** — Not generic polish: a dedicated repair pass across every boundary crossed above (coordinator/repository race conditions, store-reset messaging, accessibility regressions introduced by final UI tweaks), the mechanical zero-network audit across the timer/logging/adaptation path, and the full scripted QA matrix run end-to-end on-device with realistic interruption patterns. *Exit:* a go/no-go review against the production-readiness bar set in `prompt_contract` — not "it compiles," but "engine tests pass, corruption recovery is exercised, background/lock/kill reconciliation is proven on-device, and the core loop works fully offline."

**Dependencies**
- Domain types + the single constants file are the true root — nothing else can freeze until these do.
- `RecommendationEngine`/`ReconciliationPolicy` (pure) and SwiftData persistence/`AppLaunchBootstrap` can build in genuine parallel; they converge only at the `SessionRecord`→`SessionSnapshot` translation, which gets its own integration test before anything downstream depends on it.
- `SessionCoordinator` depends on domain types, persistence, and the *protocols* for notifications/subscriptions — not their concrete implementations. This means real `UNUserNotificationCenter` work and the StoreKit skeleton can be built in parallel with the coordinator, not gated behind it or gating it.
- UI cannot do meaningful work until `SessionPhase` and the coordinator API are frozen (freeze point 1). Within UI, Home/Session/Outcome must stabilize before History/Settings/paywall, since the latter mostly consume state the former already had to get right.
- The paywall and multi-profile UI depend on `Profile` schema and Settings navigation existing, but StoreKit logic itself stays isolated from the session/timer path throughout.
- Recommendation explanation UI and the paywall's live-data preview depend on the `Recommendation` struct and `RecommendationEngine` signature being frozen (freeze point 2) — changing that later invalidates tests and copy simultaneously.

**Risk mitigations**
- Data trust / null-outcome risk → provisional writes before any confirmation UI, on every terminal transition, as a hard invariant.
- Interruption/backgrounding risk → timestamp-derived state only, never a live counter; verified on real devices, not simulator, since simulator can't honestly represent kill/background behavior.
- Cold-start churn risk → default/learning states built and verified as a complete, polished experience in their own right, not a crippled prelude.
- SwiftData fragility → small schema, one migration round-trip test, and a corrupted-store recovery mechanism that explicitly deletes WAL/SHM companion files alongside the primary store file, verified by an actual on-disk corruption test.
- DST/timezone correctness → all elapsed-time math uses raw `Date`/`TimeInterval` arithmetic, never calendar-component subtraction.
- Subscription lapse risk → entitlement checks live only at the feature-view layer, never inside session start; lapsed data is soft-locked, never deleted.
- Delete-all-data surprise risk → scope explicitly pinned to session/profile/learning data only, excluding onboarding/permission-prompt flags, so users aren't unexpectedly re-onboarded.
- UX/scope-drift risk → the "why" stays on Home, no feature ships that doesn't strengthen the adaptive loop.
- Schedule risk from constants duplication → clamps/thresholds/windows defined exactly once, with a test that would fail if any consuming site hardcodes its own copy.

**Verification plan**
Each milestone above ends on an explicit go/no-go gate, not a status update: corrupted-store recovery and no-network structural test (M1... wait, M0/foundation); the full on-device interruption scenario matrix (session runtime); synthetic-history clamp/threshold tests plus a seeded dataset showing visible recommendation movement (adaptation); full per-screen state enumeration plus explicit accessibility test cases (product surfaces); `StoreKitTest`-driven purchase/restore/lapse/failure-type coverage plus one real sandbox pass reserved for pre-submission (monetization); and a final cross-boundary repair pass plus scripted end-to-end regression on-device (hardening). Integration and device-based testing are treated as first-class milestone work throughout — never a single "testing" line item deferred to the end.

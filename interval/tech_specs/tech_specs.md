**Codex (QA & Risk · Pragmatist) — Round 1**

I would keep the tech spec narrower than the design might tempt us to make it. This app is a timer with adaptive recommendations, not a productivity platform. The smallest architecture that actually ships is one app target with four layers that stay boring and explicit: domain types and recommendation logic, a SwiftData-backed repository boundary, a `@MainActor` session/application coordinator, and SwiftUI screens that render coordinator state. The risk to manage is not “can we draw the UI,” it is “can we avoid corrupting truth when the app is backgrounded, killed, denied notifications, or left mid-flow.” So I would spec every cross-lane boundary around persisted timestamps and provisional writes, not around timer animations.

The build should split cleanly into lanes without incompatible types if we freeze a small shared contract now. `data_domain` owns pure value types and the heuristic engine. `services_utilities` owns persistence, notifications, subscriptions, and app boot/recovery. `primary_ui` owns views and presentation-only state derived from coordinator outputs. `polish_resilience` owns accessibility helpers, motion wrappers, corruption recovery messaging, and test fixtures/scenarios. The one place I would not allow duplication is session lifecycle. A single `@MainActor @Observable SessionCoordinator` must be the only writer of live session state and the only caller allowed to start/stop/reconcile sessions. If multiple views or services can mutate session outcome independently, this app will eventually log contradictory histories.

The file layout should reflect that separation so parallel work can proceed without tripping over each other. I would use `App/IntervalApp.swift` and `App/AppBootstrap.swift` for launch/container recovery. I would not store too much derived data. History is small. But I would keep the repository itself main-actor-friendly in v1; this dataset is tiny and the complexity of building a custom actor-backed store layer is not justified unless profiling says otherwise.

The session runner needs one hard rule: the app never trusts an in-memory countdown as source of truth. On start, the coordinator writes a draft `SessionRecord`, schedules the notification by session UUID, and flips phase to running. On manual stop or natural completion, it writes a provisional terminal outcome immediately before presenting the outcome sheet. On relaunch, it loads any unresolved session and reconciles from `startedAt + plannedFocusSeconds` against current time. That reconciliation should produce one of the agreed taxonomy values plus provenance, and the user can correct it later. This is the smallest system that survives kill/background cases without leaving null outcomes behind.

I would make the recommendation engine pure and test-first. Input is recent session records for a profile plus “now” context if needed for coarse time bucket logic. Output is `Recommendation`. The important thing is not the exact math sophistication; it is that the function is deterministic, explainable, and easy to snapshot-test.

Concurrency should stay conservative. `SessionCoordinator`, `SubscriptionStore`, and UI-facing permission state are `@MainActor`. `RecommendationEngine` is plain synchronous Foundation logic. Everything else should be direct. No extra modules, no generic abstractions, no “future-proofing” around analytics, sync, or alternate stores.

**Claude (Frontend Engineer · User Advocate) — Round 1**

I'm picking up tech_specs as the person who actually has to open Xcode and start typing, with a specific user in my head: someone who taps Start at 2:47pm, locks their phone, and doesn't look at it again until their meeting ends at 3:35 — the app has to get that right without me hand-waving "reconciliation" as a concept, it has to be one function I can point at.

The architecture from ios_architecture_review holds up and I'm not reopening it — one `@MainActor @Observable SessionCoordinator` owning `SessionPhase`, a pure `RecommendationEngine`, `SessionRepository` in front of SwiftData, `NotificationScheduler`, `SubscriptionStore`. What tech_specs actually needs to add is the thing that turns "framework-light" from a nice sentence into something the compiler enforces. My concrete proposal: pull the data_domain layer into a local Swift package, `IntervalCore` — zero external dependencies, just a first-party module boundary inside the same repo — containing the SwiftData `@Model` types, the plain domain enums/structs, `SessionRepository`, and `RecommendationEngine`. The app target depends on it; it depends on nothing SwiftUI or StoreKit. This isn't modularization for its own sake — three phases back everyone agreed "if this invariant gets weakened anywhere downstream, the app will still look polished in demos and still fail its core promise," and a convention that lives only in a code review comment is exactly the kind of thing that gets quietly violated at hour 40 of a build under time pressure. A package boundary makes `import SwiftUI` inside `RecommendationEngine.swift` a build error, not a hopeful norm. I'd stop at one package, though — services and UI genuinely need UIKit/StoreKit/SwiftUI, so a hard boundary there wouldn't buy anything; this is one seam exactly where it matters, not four seams for symmetry.

Inside that package I want to flag something I think the design phase glossed over: `RecommendationEngine` cannot take `SessionRecord` (the SwiftData `@Model` class) as input, and not just for the "no framework imports" reason — SwiftData model instances are awkward to construct standalone for tests, they generally want a live `ModelContext` behind them. But the app_features phase's own acceptance criterion is "synthetic histories engineered to want a bigger jump... covered by an automated unit test suite" — that only works cleanly against plain `Codable` structs. So I'm introducing `SessionSnapshot`, a plain Foundation struct that's the actual engine input, and the repository is responsible for the `SessionRecord` → `[SessionSnapshot]` translation at the boundary. This is a small addition but it's the difference between "the tests are annoying to write so people skip the boundary cases" and "the tests are trivial to write so people actually write forty of them."

Second thing I want to sharpen, and I'll say directly this is stress-testing a decision two phases already made rather than deferring to it: the design handoff says the provisional-outcome write uses "the same elapsed-vs-planned heuristic already agreed for backgrounded sessions" for every running→ended transition. I don't think that's quite right once you sit down to implement it, and I'd rather flag it now than have a builder discover the mismatch mid-sprint. A user who manually taps Stop at second 40 of a 25-minute session didn't "expire while away" — they're standing right there. Collapsing that into the same ≥90%-elapsed/<90%-elapsed split as relaunch reconciliation would provisionally tag an obvious manual stop as `expiredWhileAway`, which is a wrong-sounding label even before the user gets to the confirm sheet, and it's exactly the kind of small dishonesty this app has built its entire identity against. So I want one function with a trigger parameter instead of implicitly assuming relaunch semantics everywhere: natural completion while foregrounded provisionally resolves to `completed`; manual stop while foregrounded provisionally resolves to `stoppedEarly` (or `canceledBeforeMeaningfulProgress` if under the 60-second floor, foreground or not); relaunch/background reconciliation is the only path that actually needs the elapsed-fraction split against the 90% threshold. Same shared floor and threshold constants, same "auto-written before the sheet, sheet corrects rather than originates" guarantee, just not a single blind formula pretending three different situations are one situation.

On persistence, I'm dropping the "optionally `RecommendationSnapshot`" entity from the architecture round rather than carrying it forward as a maybe — folding `recommendedFocusDuration` / `recommendedBreakDuration` / `explanationText` directly onto `SessionRecord` as the snapshot-at-start-time gives us the audit trail the why-line needs without a second entity and a second set of migration/query concerns for something this small. Smaller finished schema over a technically-more-normalized one nobody will query separately. Time-bucket needs an actual documented rule since it's flagged twice now as needing one and never pinned: bucket is computed from `Calendar.current.component(.hour, from: startTimestamp)` at write time and stored as the enum value directly (not recomputed from the raw timestamp later), so a timezone change or midnight crossover can never retroactively reclassify a session that already happened — history is what it says it is at the moment it was written, which matters because the whole "why" narrative depends on time-bucket counts staying stable once shown to the user.

For the view layer, my one hard rule as the frontend/user-advocate voice: no `@Query` splashed directly into `HistoryView` or anywhere else. It's tempting because it's idiomatic SwiftUI, but it would give History its own private read path into SwiftData that bypasses the repository, which is exactly the seam we need intact for corrupted-store recovery and the eventual CloudKit layer — and it makes the segmented All/Completed/Early-stop/Interrupted filter, which needs dynamic predicate composition, awkward to express cleanly anyway. Instead, `HistoryViewModel` is an `@Observable` that asks `SessionRepository` for data explicitly and gets nudged to refetch whenever `SessionCoordinator` finishes a write — one writer, and every read path goes through the same door. Views bind to `SessionCoordinator` via `.environment(_:)` (Observation-framework style, not a singleton or notification-center broadcast), injected once at the app root after `IntervalApp` finishes the `ModelContainer` bootstrap — which is also where the corrupted-store recovery lives concretely: catch the throwing initializer, delete the store file, reinit empty, set `AppSettingsRecord.storeWasReset = true`, and `RootView` reads that flag on first appearance to show the honest message once, then clears it.

Navigation stays exactly what design_handoff specified — `TabView` with three tabs, `fullScreenCover(item:)` for the active session bound to `SessionPhase`, sheets for start-adjustment/outcome/paywall/delete-confirm — I have nothing to add there, it's already unambiguous. What I'd add for offline/performance correctness: `RecommendationEngine.recommend(...)` gets called exactly once per session write inside the coordinator, result cached on the coordinator as a plain property, never recomputed inside a view's `body` — this was flagged in design_handoff and I want it preserved as an actual signature constraint, not just a caution, which is why `recommend` takes an explicit `now:` and returns a value type rather than being something a view could be tempted to call reactively.

Testing strategy: heaviest weight on `RecommendationEngineTests` and `ReconciliationPolicyTests` in `IntervalCore`'s own test target, run against synthetic `[SessionSnapshot]` arrays with zero SwiftData/ModelContext setup needed — that's the payoff of the snapshot boundary. Second tier is `SessionRepositoryTests` (including a deliberately-corrupted-file test asserting clean recovery) and `SchemaMigrationTests` (one `VersionedSchema` round trip). Third tier is `NotificationSchedulerTests` (per-UUID cancellation, no-reprompt-after-denial) and `StoreKitFlowTests` via `SKTestSession`. UI/sheet coverage stays honestly manual/spot-check, as already agreed — I'm not pretending that changes here.

## Final Output

**Architecture overview:** One app target plus one local Swift package, `IntervalCore` (zero external dependencies, no `import SwiftUI`/`StoreKit` allowed), holding the domain enums/structs, the SwiftData `@Model` types, `SessionRepository`, and `RecommendationEngine`. This makes the "engine and persistence stay framework-light" invariant a build error if violated, not a review-time hope. The app target holds `SessionCoordinator` (`@MainActor @Observable`, sole writer of `SessionPhase`), `NotificationScheduler`, `SubscriptionStore`, `HistoryViewModel`, and all SwiftUI views. Timer truth is always `startTimestamp + plannedDuration` recomputed from `Date()`; the moment a session transitions running → ended (manual stop, natural completion, or relaunch reconciliation), a provisional outcome is written immediately via `ReconciliationPolicy`, before any confirm/correct UI appears — and that policy is now trigger-aware (`SessionEndTrigger`: `naturalCompletion` / `manualStop` / `appRelaunchReconciliation`) rather than one blind elapsed-fraction formula for all three cases.

**File/module layout:** `IntervalCore` package — `Domain/` (`SessionOutcome`, `LearningState`, `OutcomeProvenance`, `TimeBucket`, `SubscriptionTier`, `SessionEndTrigger`, `SessionSnapshot`, `EvidenceMark`, `Recommendation`, `RecommendationEngine`, `ReconciliationPolicy`), `Data/` (`SessionRecord`, `Profile`, `AppSettingsRecord` as SwiftData `@Model`s, `RepositoryError`, `SessionRepository` protocol + SwiftData implementation, `AppLaunchBootstrap`). App target — `Services/` (`SessionCoordinator`, `NotificationScheduler`, `SubscriptionStore`, `Clock`), `UI/Home`, `UI/Session`, `UI/History` (`HistoryViewModel`, `HistoryFilter`), `UI/Settings`, `UI/Shared` (`EmptyStateView`, `ErrorStateView`, `ReduceMotionAnimator`). Tests split into `RecommendationEngineTests`/`ReconciliationPolicyTests` (pure, no SwiftData setup), `SessionRepositoryTests` (incl. forced-corruption recovery), `SchemaMigrationTests`, `NotificationSchedulerTests`, `SubscriptionStoreTests` (via `StoreKitTest`), and one thin `UITests` target.

**Data models:** `SessionRecord` (SwiftData model) carries the full audit trail — planned/actual timing, the recommendation snapshot at start time (folded in directly, no separate `RecommendationSnapshot` entity), outcome, provenance, override flag, energy rating, time bucket computed once from `Calendar.current` at write time and never retroactively recomputed. `SessionSnapshot` is the plain `Codable` mirror that's the *only* type `RecommendationEngine` ever accepts. `Profile` exists from day one (single auto-created instance free tier, multi-profile paid-gated at the view layer only). `Recommendation` carries focus/break duration, learning state, explanation, evidence marks, and adaptation progress — computed once per session write and cached on the coordinator, never recomputed inside a view `body`.

No `@Query` used directly against `SessionRecord` anywhere in the view layer — `HistoryViewModel` and the coordinator are the only read/write paths, so the repository seam stays real (needed for both corrupted-store recovery and a future CloudKit layer).

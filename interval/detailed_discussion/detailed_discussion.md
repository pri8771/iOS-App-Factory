**Codex (Product Strategist · User Advocate) — Round 1**

The biggest thing this phase needs to lock down is what the app does when reality is messy, because messy is the normal case here. The user this app is for is not someone running perfect, clean Pomodoro cycles at a desk all day. It is someone who starts a session, gets interrupted, sometimes stops early, sometimes powers through, sometimes overrides the recommendation because they know they only have 18 minutes before a meeting, and then expects the app not to get confused or dishonest. If we don't resolve those cases now, later phases will quietly design for the ideal path and call the rest "edge cases," which would be wrong for this product.

My preferred direction is to make the adaptive system intentionally conservative, legible, and forgiving. The app should only change recommendations when it has enough evidence, and it should bias toward small recommendation moves rather than dramatic swings. A user should feel, "this is noticing my pattern," not "this thing is jerking my timer around." That means we should explicitly decide that recommendations change in bounded increments, that manual overrides are logged as useful signal but do not instantly retrain the system on their own, and that interruptions count differently from deliberate early stops. Those distinctions matter. A stopped-early session often means the recommended duration overshot the user's capacity. An interruption may say more about context than stamina. If the app treats both the same, the adaptation becomes noisy and starts making false claims.

Cold start still needs one more layer of specificity beyond "use defaults until threshold." The app needs a visible learning state with three phases: default, learning, adapted. Default is zero meaningful data. Learning is enough data to show pattern signals but not enough to move recommendations aggressively. Adapted is when the system can honestly say the recommendation is based on your behavior. That avoids the binary trap where the app acts generic for several sessions and then suddenly pretends it knows the user. It also gives design and copy clear states to build around.

A few requirements should be resolved now rather than left vague. Session outcome taxonomy needs to be fixed: completed, stopped early, interrupted, expired while away, and canceled before meaningful progress. Those are not the same thing. We should also decide what "completed" means if the user comes back after the timer elapsed while the phone was locked. I think it should be treated as completed if the elapsed time was actually reached, but the app should still ask for a quick confirmation on return so the history stays trustworthy. Similarly, force-quit mid-session cannot just become data loss. On relaunch, the app should reconcile from stored start time plus current clock and offer a simple recovery state: "That session should have ended 12 minutes ago. Mark it completed, interrupted, or discard?" That is exactly the kind of behavior a real user will judge harshly if it is missing.

Manual override needs a clean rule too. Users should be allowed to override the recommended duration before a session starts, because not allowing that is paternalistic and unrealistic. But the app should treat overrides as context, not blind truth. If someone repeatedly overrides upward and then completes those longer sessions, the system should eventually learn from that. If someone overrides because they only have a narrow time window, that should not immediately drag the baseline down or up. So the final output should preserve that override behavior is permitted, logged, and used carefully rather than either ignored entirely or over-weighted.

Privacy and permissions are straightforward but still need explicit framing. Notification permission is essential to the product's reliability, but denial cannot break the app. If notifications are off, the app still works locally, but the UI needs a clear degraded-mode explanation: timers remain visible in-app, but lock-screen/end-of-session alerts won't fire. No scare tactics, no repeated nagging. On privacy, the app should store only what it needs to adapt: timestamps, planned duration, actual elapsed duration, outcome, optional energy rating, and perhaps coarse time-of-day bucket. It should not drift into collecting task names, notes, or personal journaling unless later phases can justify it directly against the adaptive loop. More data is not automatically better here; unnecessary intimacy weakens trust.

The biggest product risk in my own direction is that a conservative learning system may feel too subtle. If the app waits too long, changes too little, or explains itself too cautiously, users may never feel the promised intelligence. But the opposite risk is worse: overreactive fake intelligence that changes recommendations constantly and loses credibility. The mitigation is not making the engine more dramatic; it's making the feedback more visible. Even small changes need to be surfaced clearly: "Recommended focus moved from 30 to 28 minutes because you stopped early in 3 of your last 5 afternoon sessions." That gives the user evidence that something is happening without the system needing to overfit.

The risk register should explicitly include notification denial, force-quit during session, device clock changes, midnight crossover, very sparse history, highly irregular schedules, and users who mostly override recommendations. Those are not obscure QA footnotes; they are common enough that the architecture and copy need a position on them now. Clock changes especially matter because this app relies on elapsed time. We should assume wall-clock anomalies happen and derive session truth from stored start/end timestamps carefully rather than trusting a continuously running in-memory timer.

The final output needs to preserve one hard product principle above all: the app is allowed to be narrow, but it is not allowed to be vague. Every adaptive claim must map to a specific user-visible rule, every messy session outcome must have a defined interpretation, and every state recovery path must protect trust more than cleverness. If later phases keep that, the app stays honest and valuable. If they lose it, this turns into another pretty timer with "AI" language wrapped around undefined behavior.

**Claude (QA & Risk · Systems Thinker) — Round 1**

Picking up where the product brief left off, my job this round is to stress-test the loop we've already agreed on and find where it actually breaks in practice, not in theory.

The reconciliation problem is the one I'd put first because everything else depends on getting it right. We've agreed the timer has to survive lock/background via local notifications, and that early-stop/interrupted is the main case, not the edge case. But we haven't actually defined what "interrupted" means mechanically. Is it purely self-reported at the two-tap outcome screen, or does the app try to detect it (audio session interruption from a phone call, app backgrounding duration, etc.)? I'd push for self-reported as the primary signal because passive detection adds a lot of surface area for false classification, but that creates a gap: what happens when the user never comes back to log the outcome at all — session ends, phone stays locked, they don't reopen the app for six hours, or ever? We need an explicit rule now: a session with no logged outcome within some window should auto-resolve based on elapsed time versus scheduled duration (ran to completion with no interaction = treat as completed-but-unconfirmed; check-in prompt appears next launch but doesn't block starting a new session), rather than sitting in limbo or silently vanishing from history. If we don't decide this now, whoever builds it will improvise something, and "improvised state machine for the one thing that has to be bulletproof" is exactly the failure mode this app can't afford.

Adjacent to that: whatever schedules the "session complete" notification also has to cancel it cleanly when the user stops early from inside the app, or they get a stale notification firing later claiming a session ended that they already know they abandoned. That's a small thing that's easy to miss and immediately reads as buggy to a user, undermining trust in the one feature (notification reliability) the whole architecture leans on.

On the adaptation math itself, I want to flag a couple of things that aren't algorithm design (that's next phase) but are edge-case constraints the algorithm has to respect. First, clamping — sparse or noisy data (a user who does three sessions and two are outliers) shouldn't be able to push the recommendation to something absurd like a 3-minute or 4-hour session; there needs to be a sane min/max bound regardless of what the heuristic computes. Second, data poisoning from degenerate input: if someone taps start and immediately stops (fat-fingered launch, or testing the app), that shouldn't count as a real "early stop at 25 minutes" data point pulling future recommendations down. I'd propose a floor — sessions under some small threshold (say under a minute) are logged for history/transparency but excluded from the adaptation calculation — otherwise a few accidental taps can visibly warp the "why" explanation in a way that makes the app look erratic rather than smart. Third, and this one's a real open question rather than something I can resolve alone: does old session history ever decay in weight, or is everything equal-weighted forever? If someone's focus pattern genuinely changes (new job, semester ends), an app that never lets go of six-month-old data will feel stuck exactly when it needs to feel adaptive. I think recency weighting or a rolling window (which the brief already gestures at for the paid tier's "longer window") needs to be a v1 decision, not deferred, because it directly determines whether the "genuinely different recommendations over time" success criterion stays true after week one or only proves itself once and then plateaus.

Permissions are a real risk I don't think we've fully priced in. The whole backgrounding-survival requirement is contingent on the user granting notification permission — and if they deny it (which a meaningful fraction of users do, especially if asked at first launch before they've seen any value), the core promise silently degrades. That needs an explicit fallback story: the app should still function via in-app timer state while foregrounded, and should honestly communicate reduced reliability if permission is denied, rather than pretending it's fine or nagging aggressively. I'd also flag that asking for notification permission at cold launch, before the user has started a single session, is a well-known way to tank opt-in rates — better to ask contextually, right when they first tap start, when the value of "tell me when this ends" is obvious.

On persistence and data loss: local-first means the local store is the only copy, so store corruption or a failed migration between app versions isn't a minor bug, it's total data loss of the thing the entire product is built on. We need a decision now that a corrupted or unreadable local store fails safe — reset to a fresh empty history with a clear, honest message to the user about what happened — rather than crashing on launch or silently losing history without telling anyone. Related smaller point: SwiftData-backed local stores live under Application Support/Documents, which by default get swept into the user's standard device backup (iCloud or iTunes). That's the user's own backup, not us phoning home, so I don't think it violates the "no network calls in the core path" rule, but it's worth deciding explicitly rather than discovering later that "local-first, never leaves your phone" in App Store copy is technically imprecise once backups are considered.

A couple of smaller but real ones: time-of-day bucketing needs a defined rule for sessions that cross midnight or happen right at a bucket boundary, since we're using time-of-day as an explanatory signal even in v1. Accessibility for a live countdown needs actual thought, not just VoiceOver labels bolted on — a screen reader announcing every second of a countdown timer is unusable, so milestone-based or on-demand announcements are a requirement, not a nice-to-have, and Dynamic Type needs to be checked against the "why" explanation text specifically since that's often the line most likely to get clipped in layout. And on monetization: what happens to a user's second/third session profile if they let their subscription lapse — do we hard-lock the data, make it read-only, or delete it? This is both a UX decision and something App Store review actually checks for (can't just destroy purchased-tier user data on downgrade), so it needs an answer now rather than being improvised at paywall-implementation time.

The biggest risk to my own framing here is that I'm adding a lot of "decide this now" items to a phase whose job is supposedly to convert open questions into assumptions — I'm aware the temptation is to over-specify edge cases that could reasonably be resolved as documented assumptions rather than fully designed behavior. I think that's fine as long as we're explicit about which of these get a real design (reconciliation state machine, notification cancellation, clamping bounds) versus which just get named as an assumption to revisit (recency weighting exact formula, exact grace-window duration, backup inclusion). The detail I most want preserved into architecture is the reconciliation logic for unresolved sessions on relaunch — that's the one piece of state management that, if under-designed, breaks the single hardest-to-fake requirement in the whole product: proving the timer is trustworthy through backgrounding, lock, and force-quit.

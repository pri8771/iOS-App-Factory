**Claude (Frontend Engineer · Skeptic) — Round 1**

I want to take this phase from "we agreed on tabs and sheets" to "a SwiftUI engineer could open Xcode and start typing," and my job this round is to poke at the places where the last two phases' decisions are elegant in prose but ambiguous in code — because ambiguity here doesn't become a design nitpick, it becomes a state bug that corrupts the adaptation history the whole product is built on.

Start with the thing I think is genuinely unresolved: what owns session truth, and where does it live relative to the view hierarchy. I'd model this as a single `@MainActor @Observable` `SessionCoordinator` that is the sole writer of a `SessionPhase` enum — `idle(Recommendation)`, `awaitingOutcome(SessionRecord)`, `running(SessionRecord)`, `onBreak(SessionRecord)`, `reconciliationPending(SessionRecord, provisional: Outcome)` — and every screen (Home, the full-screen session cover, the outcome sheet) is a pure function of that one enum, never independent local state that could drift out of sync with it. The active-session screen should be a `fullScreenCover(item:)` bound to that phase, not a `NavigationStack` push, because push implies you can back-swipe out of an active timer, which we don't want and which would also leave the notification scheduled against a screen the user thinks they left. Timer *display* can use a lightweight repeating `Timer` or `.onReceive` tick for the countdown label, but timer *truth* has to be `plannedEnd = startTimestamp + plannedDuration`, recomputed from `Date()` every time the view appears — never a paused/resumed in-memory counter — because that's the only way background/lock/relaunch all produce the same number instead of three slightly different ones depending on how the OS scheduled the app.

Here's a real gap I don't think either design round closed: what happens if the user is sitting in the outcome-capture sheet — session already ended, taxonomy not yet chosen — and the app gets backgrounded or killed. The relaunch-reconciliation logic as specified covers "session was still running when we went away," using elapsed-vs-planned time to guess completed-vs-expired. It does not cover "session already ended, outcome sheet was open, nothing was tapped." That's a distinct state and needs its own rule, or the five-value-taxonomy acceptance criterion (every terminal session gets exactly one value, no nulls) quietly breaks in exactly the moment that's hardest to test manually. My proposal: the moment a session transitions from running to ended (whether by timer completion or manual stop), it's immediately persisted with a provisional outcome using the same elapsed-vs-planned heuristic already agreed for backgrounded sessions, before the outcome sheet even appears — the sheet is now "confirm or correct," not "the only place the outcome gets written." That also means the outcome sheet is safe to be swipe-dismissible without violating the no-null-outcome guarantee, which matters because a genuinely non-dismissible sheet during a moment when the user might be frustrated or in a hurry is a bad interaction, not a good one.

On the paywall's live-preview-from-real-data idea — I like it, but it has a real architectural implication that needs to be locked now, not discovered mid-build: the insights/adaptation computation must be a pure function over `[SessionRecord]` that has no idea whether the user is entitled to see its output. Entitlement can only be a thin gate at the view layer (`if hasEntitlement { InsightsView(data) } else { LockedRow() }`), never baked into the query or the computation itself — because the moment someone "simplifies" this by making the paid-only data inaccessible at the persistence layer, the paywall preview becomes impossible to build honestly and someone will quietly swap in fake sample data instead, which is exactly the trust failure this whole design language is trying to avoid. Same reasoning applies to multi-profile: even though it's should-have and paid-gated, the free-tier schema needs a `Profile` entity from day one with exactly one instance auto-created — retrofitting a profile concept onto a schema that assumed a single global history later is a real migration, not a toggle.

I'll push back a little on the accent-color-warms-with-learning-state idea from the design round — not to kill it, it's a good instinct, but "gradually warms" as a continuous interpolation is something nobody can actually QA, and it interacts badly with Dark Mode, Increase Contrast, and Reduce Transparency, each of which needs its own defined endpoint anyway. I'd scope it as three or four fixed, named accent tokens tied to discrete learning-state transitions (`accent.default`, `accent.learning`, `accent.adapted`) with a single cross-fade animation between them on state change, not a formula. That's buildable, testable, and still delivers the "felt" quality without inventing an infinite space of untested intermediate colors.

On tokens concretely, since that's a required output and "premium and precise" isn't a token: type scale should ride Dynamic Type text styles, not fixed point sizes — `largeTitle`/rounded-monospaced-digit variant for the countdown number itself (`.monospacedDigit()` is non-negotiable, already flagged and correct, it's what stops the digits from jittering horizontally), `.title2` for the recommendation duration on Home, `.body` for the why-line with a minimum of 3 lines of layout headroom reserved so it doesn't get clipped or cause a relayout jump when it populates. Color as semantic tokens, not raw hex, defined in an asset catalog with light/dark/high-contrast variants: `background.primary`, `background.elevated` (for sheets/cards), `text.primary`, `text.secondary` (why-line, timestamps), `accent.*` (the three fixed stops above), `outcome.completed`/`outcome.stoppedEarly`/`outcome.interrupted`/`outcome.expired`/`outcome.canceled` as distinct semantic colors so History and the session-ring both draw from one source of truth instead of each screen inventing its own palette for the same five outcomes. Spacing on an 4/8/12/16/24/32 scale, corner radius on a two-value scale (12 for cards, 20 for sheets' top corners) so nothing looks like it was eyeballed per-screen.

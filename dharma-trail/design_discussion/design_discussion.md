That last part is the detail I'd fight to keep: never let "we couldn't check" collapse to "treat as unsubscribed," because that's the one design choice that would make a real StoreKit hiccup feel like the app punishing a paying customer.

On the completion-save-failure gap, the gate is pointing at something I think is actually the single scariest edge case in this entire app, more than the DST/timezone stuff that's already gotten a lot of attention, because it's silent by nature — a failed write doesn't announce itself the way a crash does. Here's how I'd design it: Session Complete isn't allowed to be a pure celebratory transition that assumes the SwiftData save succeeded before it ever renders. The actual sequence has to be attempt-save-first, then branch on the result — if the save succeeds, you get the exhale/transition I described, trail advances, streak ticks. If the save fails, the user does NOT see the trail advance or the streak tick, full stop, because showing that and then having it not persist is worse than not showing it at all. Instead they see an explicit, calm "your session didn't save — let's try again" state with a retry action that re-attempts the write against the same completed-session data (which should already be sitting fully formed in memory at that point, so retry is just a persistence retry, not redoing the whole session), and only escalates to something more serious like "check your device storage" if retry fails repeatedly. This state needs the same non-punitive tone as everything else in the app — it's a save-retry moment, not a failure the user did anything wrong to cause, and the copy should make that distinction clear rather than reading like a generic iOS error dialog. I'd also note this connects to the existing "never silently count as complete" principle from the force-quit/abandon states — same family of bug, just triggered by a different cause (persistence failure vs. user action), so a developer building this should treat all of "force-quit," "abandoned," and "save failed" as siblings under one rule: no path exists where the UI shows advancement that isn't backed by a confirmed write.

Both of these are additive, like the gate said, and I don't think they touch the visual direction, the primary flow, or the accessibility notes at all — those stood on their own. But I want to flag one more thing while we're in the neighborhood of "silent failure that erodes trust," because I think it's the natural extension of the same principle and worth deciding now rather than letting it surface later: what happens if a *retry* on the save-failure state also fails, repeatedly? I'd resolve that as: after a small number of retry attempts, the state changes from "let's try again" to something that explicitly tells the user their completion is being held safely in a pending/unsaved state and will be retried automatically on next launch — never just an infinite "try again" button that a frustrated user eventually gives up on and closes, assuming the app "figured it out." That's a small addition but it's the same trust logic carried all the way through, and I think it's worth locking in now so it doesn't get improvised during build.

Both additions are treated as extensions of principles already locked in earlier phases (re-check-live-status-don't-cache from the notification work; never-silently-count-as-complete from the force-quit/abandon states) rather than new mechanisms, so nothing else in the design needs to change to accommodate them. This closes the design_discussion phase for dharma-trail.

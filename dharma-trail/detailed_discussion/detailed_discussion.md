Picking up from the product brief, the thing that jumps out at me as unresolved is the multi-trail streak model — the brief commits to running Foundations plus a second trail concurrently without losing progress, but never says what a "streak" *means* once two trails exist. Is it one unified daily-practice streak that ticks up if you complete a session in either trail, or does each trail carry its own streak? If it's per-trail, a user doing both simultaneously has to mentally juggle two counters, which fights the "gentle structure" promise. My resolution: one unified daily streak (did the user complete at least one session today, in any active trail) plus separate per-trail completion progress (day 6 of 14, day 2 of 10). That's simpler to build, simpler to explain in one sentence of UI copy, and avoids the awkward question of what happens to your "real" streak when you finish a shorter trail first.

The next thing I don't think is actually settled yet, despite being flagged twice already, is timer vs. TTS for guided sessions — and I want to push on why that decision has teeth beyond "pick one." A silent timer with haptic/chime cues means the entire guided-session experience has to work with the phone locked in a pocket, which is realistic for meditation but useless for a mantra-counting session where the user needs to know when to start and stop without looking. If we go with AVSpeechSynthesizer, we inherit real audio-session lifecycle problems: what happens when a phone call interrupts mid-session, what happens if another app is already playing audio, what happens if the phone is on silent (does TTS respect the ringer switch or not — for spoken content it typically does play through media volume, which could surprise a user in a quiet space). I'd resolve this as: default to silent timer plus haptic/chime cues for meditation and reflection waypoints, and use on-device TTS only for the mantra counter's optional guidance (announcing the count on request), gated behind an explicit user toggle rather than defaulting on — that sidesteps most of the interruption-handling burden for two of the three waypoint types while still delivering the VoiceOver-count-announcement requirement, which is a different, accessibility-driven code path from a general TTS narrator anyway.

On the mantra counter accessibility requirement specifically — "each tap announces the running count" — the real risk is a user tapping fast (japa counting is often rapid, rhythmic tapping), which will either flood VoiceOver with overlapping announcements or force a debounce that makes the counter feel laggy and unreliable for counting. This needs an explicit interaction test: rapid sequential taps must still produce a correct final count even if not every intermediate count is spoken aloud, and the accessibility announcement should coalesce rather than queue indefinitely. I'd also flag that haptic feedback and VoiceOver together can conflict with a user's system-level Reduce Motion or haptics-off setting — the counter's correctness cannot depend on haptics firing.

Session-interruption handling deserves more precision than "don't lose state." A timed session's elapsed time must be computed from wall-clock start/end timestamps, not from a running in-memory Timer, because backgrounding suspends timers but not wall-clock time — otherwise a 10-minute meditation backgrounded for 3 minutes either finishes early or drifts. Here's the concrete resolution: notification permission is requested contextually, not at cold-launch onboarding — specifically, right after the user completes their *first* session, when they're in the best possible emotional state to say yes to a reminder ("want a nudge to keep this going?"). Asking during onboarding before they've done anything is a proven way to tank grant rates, and it also front-loads a permission dialog before the user has any reason to trust the app. If they deny it, that's final for the system prompt — iOS does not allow a second native re-ask, full stop, so whatever the design phase does with a "please enable reminders" nudge later has to be an in-app affordance with a deep link via `openSettingsURLString`, not an attempt to re-trigger the system dialog. I'd cap that in-app nudge at appearing once, ever, dismissible permanently, probably surfaced after a few missed days rather than nagging on every cold launch — because a "turn on notifications" banner that reappears every session is its own kind of anti-pattern that actively works against the calm, non-punitive tone we've spent this whole phase protecting.

The part I actually think matters more than the permission-request choreography is the thing the gate implicitly asked for: proving the core loop is *completely* unaffected by permission state. Concretely, that means a real test case — fresh install, deny notifications outright, never grant them, complete an entire 14-day trail — and the app must never show a blocking permission screen, never gray out the "start session" button pending a permission decision, never silently fail to log a completed session because a scheduling call errored. Notifications here are purely an outbound reminder mechanism; they must have zero read path back into the core loop. The way I'd guard against this regressing during build is to make sure the streak/session-completion code path literally has no reference to `UNUserNotificationCenter` authorization status at all — if a "did we get permission" check ever ends up gating anything other than "should we schedule a reminder," that's a bug by construction, not just a bad outcome.

One more wrinkle worth naming now so it doesn't surprise anyone later: authorization status isn't a one-time fact captured at onboarding, it can change any time the user goes into Settings and flips it off (or, less commonly, an OS update resets it). So whatever schedules the daily reminder needs to check current authorization status at the time it schedules — probably each time the app goes to background, or on a daily basis — rather than caching "granted" from the original prompt and assuming it still holds weeks later. Otherwise you get the silent failure mode where the app thinks it's reminding someone who quietly turned reminders off three weeks ago, which reads to us as "working" and to the user as "the app stopped caring," which is a bad look for an app whose whole pitch is gentle persistence. I think that's the whole gap closed — permission-request timing, the one-shot nature of the native prompt, the in-app Settings-redirect fallback, and the hard guarantee that the practice loop itself never depends on any of it.

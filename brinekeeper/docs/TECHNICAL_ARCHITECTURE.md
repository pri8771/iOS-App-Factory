## Final Output

**Screen inventory:**
- **Tank Setup** (first-run, blocking): tank name (optional), `netSystemVolume` as its own field with inline unit picker (gal/L) and explicit "actual water volume, not display-tank size" guidance. No wizard, no progress dots — one screen, save disabled until a positive volume is entered.
- **Dashboard** (tab 1): status-first home. Per-parameter cards in three visual states — nominal (teal glow, last value + days since tested), flagged (coral glow, becomes primary tap target into that parameter's dose recommendation), and never-tested (dim/outlined, visible not hidden, violet "no data yet" treatment). Surfaces next-due reminder and permission status inline.
- **Tests** (tab 2): parameter picker (all five visible; NO3/PO4 tagged "log only"), value entry with inline unit picker for Alk, date/time defaulting to now. History list with sparkline/chart per parameter. Saving a test pushes directly into:
  - **Dose Recommendation** (pushed, not a tab): trace expanded by default (current → target → delta → product/concentration math → computed volume), product/unit editable inline rather than pre-configured during onboarding. Card is built as a component holding zero-to-many structural "advisory rows" — cap rationale, same-day-already-applied reconciliation, multi-parameter stagger warning, NO3/PO4 "not modeled" note — never ad hoc banners. Re-opening a past test's row re-enters this same screen showing its frozen snapshot, rather than a separate "workbench" you navigate to independently.
  - **Apply Dose** (pushed sheet from the recommendation): explicit "Log as Applied" action, prefilled/editable amount, product, timestamp — never automatic or inferred.
- **Livestock** (tab 3): roster of current occupants as glass cards with a persistent bioload capacity meter at the top. Add/Check Candidate pushes a result screen with risk tier up top and each triggered mechanism as its own row (aggression, allelopathy, bioload), including an explicit "no conflicts detected" state when clean.
- **Settings** (tab 4, single sectioned screen): Tank(s) [including the add-tank paywall gate as an upgrade card], Reminders [notification toggle with plain-language explanation], Data [export/import via share sheet], Subscription [entitlement status]. No separate Reminder screen, no separate Compliance/Lifecycle screen.
- **Delete confirmations**: sheets, not full screens, always naming exact dependent counts (e.g., "This will also delete 2 applied-dose records and 1 dose recommendation").

**Primary user flow:**
First run: create tank (volume + unit, blocking) → log first test → land directly on the dose recommendation with full trace (product/unit editable in place, no separate setup step). Day-to-day: Dashboard surfaces the one thing that needs attention → tap into Tests to log a value → save pushes straight into the dose screen, which accounts for any dose already applied today against the same target → optional "Log as Applied." Separately: Livestock tab → tap a candidate → live compatibility result with named mechanisms. Every other screen (applied-dose log, compatibility result, delete confirmation) exists only as a direct consequence of an action just taken — nothing floats as an independently-reachable destination outside the four tabs.

**State model per screen:** every included screen (Tank Setup, Dashboard, Tests/history, Dose Recommendation, Apply Dose, Livestock roster, Candidate check, Settings, delete sheets) implements distinct empty, loading, success, and error states — not shared/generic ones. Notably: Dashboard's empty/nominal/flagged are three different visual treatments, not one conditional string; Tests' first-ever-test state is an explicit sentence ("dosing against target only, no trend yet"), never a blank chart; Dose Recommendation's "already applied today" state is a different card treatment, not just different text, specifically to prevent double-dose blindness; validation errors on test entry suggest the likely correction (unit/decimal mistake) rather than a generic "invalid input."

**Visual direction:** near-black (deep navy-charcoal, not pure black) base with glass cards — subtle blur, hairline border catching accent glow, low-saturation fill so text stays legible. Status color system locked: teal = nominal/safe, coral = flagged/warning/high-risk, violet = informational/no-data-yet/log-only. Glow is reserved for critical values and trace/chart lines, never a blanket background wash — the chemistry numbers are the content, the glass is the frame. Motion: short (180–260ms) purposeful springs tied to real state changes only (save, apply, warning escalation), brief deliberate compute animation on the dose screen rather than an instant pop, haptics on apply/log/critical-warning only.

**Accessibility notes:** WCAG AA contrast must be verified specifically on body-text-sized elements sitting on glow gradients (trace rows, mechanism explanations) — this is flagged as the likeliest silent failure point, not the large numerals. Dynamic Type tested explicitly against the dose-trace layout (the most number-dense screen). VoiceOver order on the trace and compatibility mechanism list must read as one sequential reasoning chain (current → target → delta → dose → cap → remaining-after-applied → action), which argues for building these as genuinely sequential accessibility elements rather than overlapping decorative glow layers. Explicit reduced-motion fallback disables shimmer/pulse effects. 44pt minimum touch targets, larger for primary actions (compute, apply, add species). Delete and other destructive actions must be reachable via standard focus order, not hidden-gesture-dependent.

## Final Output

### Screen-by-screen specification

**App root / launch gate.** No modal, no sheet. Root view is a hard switch: if no tank exists yet, render `TankSetupView` full-screen with no tab bar mounted underneath. Once a tank is persisted, root swaps permanently to the four-tab shell. Nothing about first-run is dismissible by gesture.

**Tank Setup** (blocking, first-run only). Exactly two inputs: tank name (optional, defaults to "My Tank"), and `netSystemVolume` as its own labeled field with an inline gal/L segmented picker and one line of caption text ("Enter your actual water volume, not the size printed on the tank — sump and rock/sand displacement matter"). No target-default fields, no cadence controls, no product wizard. Save/Continue is disabled (visibly, not hidden) until volume is a valid positive number. States: empty (initial render), loading (brief skeleton while checking persisted state), error (inline, non-positive/blank volume — styled with system semantic red, not coral), success (Continue enabled, routes straight into Tests). Target values fall back to standard reef-range constants owned by the solver's config and become editable later inline on the parameter card or dose screen — never collected before first value.

**Dashboard** (tab 1). Fixed-order rows for all five parameters (Alk, Ca, Mg, NO3, PO4), always present. Each card in one of three states — nominal (teal, calm glow, last value + days since tested), flagged (coral, becomes the primary tap target straight into that parameter's dose recommendation), never-tested (violet outline, dim, explicitly visible, not hidden). Card status comes from a dedicated, cheap, pure function (`ParameterStatus.evaluate(latestValue:target:) -> nominal | flagged | noData`) that only compares the last value against the target band — it must never call into the dose-recommendation/solver path, which separately carries cap logic and applied-dose reconciliation and has no business running on every Dashboard render. Below the parameter stack: a persistent, non-dismissible status strip showing next-test-due and reminder permission state in plain language (e.g. "Reminders are off — tap to enable in Settings" rather than a system nag). States: empty (no tests logged anywhere — single centered card with CTA to log first test), loading (store hydrate), success, error (local read failure with retry).

**Tests** (tab 2). Parameter picker showing all five, NO3/PO4 visually tagged "log only" via an inline violet badge (not a separate section). Value entry field, Alk-only inline unit picker (dKH/meq-L/ppm-CaCO3), date/time defaulting to now. Below: history list scoped to the selected parameter with a trend chart on top. First-ever-test state replaces the chart with an explicit sentence ("No trend yet — dosing uses target only"), never a blank axes frame. Validation errors are corrective, not generic ("82 dKH looks like a decimal error — did you mean 8.2? Typical reef range is 6–12 dKH") and use system semantic red, never coral. Saving a test pushes directly into Dose Recommendation using the just-saved context.

**Dose Recommendation** (pushed, not a tab — reached only by saving a test or reopening a historical test row). Renders a fully-formed `DoseTrace` value object; the view never invokes the solver itself. A `RecommendationStatusChip` (nominal/limited/capped/blocking) sits at the top as a one-glance summary before the full trace. Trace expanded by default as discrete sequential rows (current → target → delta → product/concentration math → computed volume), monospaced digits throughout, each row a real distinct accessibility element. Below the trace: zero-to-many `AdvisoryRow`s from a shared component family — cap-applied (with plain-language rationale), same-day-already-applied (its own distinct card treatment/banner, not just different text), multi-parameter stagger warning, NO3/PO4 not-modeled note. Two data paths feed the same presentation component: "just computed" builds a fresh `DoseTrace` from live inputs and pushes it; "reopened historical" loads the persisted frozen snapshot and renders it read-only except for the apply action — the view itself has no branch that decides whether to recompute. Primary action "Log as Applied" pinned at the bottom, always explicit, never inferred.

**Apply Dose** (sheet, pushed from the recommendation). Amount (prefilled/editable), product, timestamp (defaults now), optional note. Confirm writes the applied-dose record only on explicit save; cancel dismisses without side effects.

**Livestock** (tab 3). Persistent capacity meter at the top (numeric value shown as text, never color-only). Roster list below as glass cards (species, aggression tier icon, bioload points). Add/Check Candidate pushes a result screen: risk tier badge (icon + text + color, never color alone) at top, then each triggered `MechanismRow` (aggression, allelopathy, bioload) as its own labeled block, with an explicit "no conflicts detected" state when clean. Live checks always recompute against current roster state on render (via `@Query`-driven computation, never cached into stale `@State`); a separate `CompatibilityDecision` audit record is persisted at add-time, storing species **name strings** (not live foreign keys) so historical entries render coherently even after roster changes — this is the deliberate mirror image of the dosing trace: dosing is evidence and freezes, compatibility is live policy that only freezes at the moment of an explicit audit write.

**Settings** (tab 4, single sectioned screen). Four sections: Tank(s) (including add-tank paywall gate as an upgrade card, not a dead-end alert), Reminders (notification toggle + plain-language explanation of denied state), Data (export/import via share sheet), Subscription (entitlement status). Settings intentionally uses plain grouped-list styling rather than glass cards — glass is reserved for data-bearing screens. Delete actions live here and elsewhere as sheets (not full screens), always naming exact dependent counts (e.g., "This will also delete 2 applied-dose records and 1 dose recommendation").

### Design tokens

- **Base background:** deep navy-charcoal, ~`#0A0D14` — never pure black.
- **Status colors, two-tier per hue** (glow variant for shadow/blur layers only, capped opacity ~30–40%; solid variant at full opacity for text/icons/borders where contrast matters): teal ~`#3ECFC0` (nominal/safe), coral ~`#FF6B5B` (flagged/warning/high-risk — chemistry and compatibility status only), violet ~`#9D8CFF` (informational/no-data-yet/log-only).
- **Form validation errors and destructive actions** use system semantic red — deliberately outside the teal/coral/violet system, so a data-entry mistake never visually reads as a chemistry warning and delete actions stay unambiguous.
- **Surfaces:** `.ultraThinMaterial`/`.regularMaterial` for glass cards, not hand-rolled blur — this automatically respects Reduce Transparency and Increase Contrast.
- **Typography:** system font throughout (no custom font — Dynamic Type risk on a numbers-dense app), monospaced-digit style applied globally to all numeric readouts via one shared `Text` style so trace/chart numbers never jitter row height.
- **Spacing:** 8pt base scale (8/16/24/32/48).
- **Radii:** 20pt cards, 28pt sheet top corners, 14pt buttons.
- **Motion:** 180–260ms springs on real state changes only (save, apply, warning escalation); a brief, honest compute transition on the dose screen tied to actual state progression, never a faked delay. Reduced Motion strips shimmer/pulse/chart-glow shadows entirely, falling back to flat high-contrast strokes and static labels.
- **Haptics:** light on successful save/apply, medium/heavy reserved for cap-triggered or high-risk-compatibility moments — haptic weight itself communicates severity.

### Component inventory

`NavigationCoordinator` (shared `@Observable`, owns per-tab `NavigationPath`s so Dashboard's flagged card can atomically switch tab + push into the right Tests-flow destination — a cross-tab action that per-tab-local `@State` paths cannot support), `GlassCard`, `ParameterCard`, `StatusChip`/`StatusBadge`, `RecommendationStatusChip`, `TraceRow`, `AdvisoryRow` (base + cap/applied-today/stagger/not-modeled subtypes), `AppliedTodayBanner`, `CapacityMeter`, `MechanismRow`, `RiskTierBadge`, `TrendChart` (Swift Charts wrapper with built-in glow-shadow + Reduce Motion fallback), `EmptyStateView`, `DestructiveConfirmationSheet` (title + dependency-count list), `UnitSegmentedPicker`, `ValidationCorrectionBanner`, `PrimaryButton`/`SecondaryButton` with shared disabled-state styling.

### State and interaction notes

Every included screen implements distinct empty, loading, success, and error states — never shared/generic ones. Two structural data-flow rules govern correctness: (1) `DoseRecommendationView` renders a pre-built `DoseTrace` value type — "just saved" and "reopened historical" are two separate call sites feeding one presentation component, and the view itself never decides whether to recompute; (2) compatibility works the opposite way — live checks always recompute from current roster state, while only the `CompatibilityDecision` audit record freezes a name-string snapshot at add-time. Same-day-already-applied and cap-applied states get materially different card treatments, not just different copy, specifically to prevent double-dose blindness. Dashboard status coloring is computed via a lightweight pure function, entirely decoupled from the full solver path.

### Accessibility requirements

WCAG AA contrast verified on body-text-sized elements sitting on glow gradients (trace rows, mechanism explanations) at both standard and largest Dynamic Type sizes — this is the likeliest silent failure point, not large numerals. VoiceOver order on trace and mechanism lists reads as one sequential reasoning chain (current → target → delta → dose → cap → remaining-after-applied → action), built as genuinely separate accessibility elements. Capacity meter and risk tier badges never rely on color alone — always icon + text + color. 44pt minimum tap targets with full tappable area matching visual bounds, larger for primary actions. Reduced Motion and Increased Contrast/Reduce Transparency are tested as first-class states, not afterthoughts — enabled by using system Materials rather than custom blur. Destructive and all primary actions reachable via standard focus order, no hidden-gesture dependencies.

### Claude Design prompt / external-design decision

**No external Claude Design handoff for this build.** This specification is implementation-ready in SwiftUI as written — the navigation coordinator, the frozen-vs-live `DoseTrace` type boundary, the inverse live/frozen rule for compatibility, Material-based glass surfaces, and the two-tier glow/solid token system are interaction and data-flow contracts, not visual adjectives, and would be lost or require reverse-engineering if routed through a static external design tool. Implementation proceeds directly from this document. An optional, non-gating informal visual spot-check (Dashboard nominal/flagged states + the expanded Dose Recommendation trace) is permitted after initial implementation to sanity-check that the glow/glass palette reads as premium rather than muddy, but this is QA, not a required deliverable, and nothing in this phase is blocked on it.

## Final Output

**SwiftUI architecture:** Pure, side-effect-free domain services in plain `struct`/`actor` types (`DosingSolver`, `CompatibilityEngine`) fully separated from `@Observable` `@MainActor` UI coordinators — no domain logic lives in views. A shared `NavigationCoordinator` (from the design phase) owns cross-tab navigation state. A `ModelContainerManager` wraps SwiftData container creation with explicit failure handling — never `try!` — degrading gracefully to a recovery screen (export-what's-recoverable or start-fresh) on corruption or schema mismatch, since a launch crash is a worse trust failure for this app than almost any other bug class. A `SnapshotService`, fed by repository write callbacks, is the only thing that writes the widget's derived data, and it fires only at real commit points (test saved, dose logged, tank changed, entitlement change) — never on every keystroke or chart redraw.

**Apple framework choices:** SwiftUI, SwiftData, Swift Charts, UserNotifications, BackgroundTasks (housekeeping only), WidgetKit, StoreKit 2, Foundation/OSLog. ActivityKit is explicitly excluded — Live Activity is cut completely for this build, zero ActivityKit code or scaffolding, not deferred-as-optional. AppIntents only if it earns clear value later; no AVFoundation/Vision in v1 (matches the no-photo-strip-reading scope).

**Persistence and local data plan:** SwiftData models keyed by UUID from day one, but uniqueness is enforced at the write/insert-update service layer, not via `@Attribute(.unique)` — SwiftData's CloudKit-sync mode (the app's own future roadmap path) doesn't support unique constraints or non-optional relationships, so building on the constraint now would force a painful migration later. The primary SwiftData store stays a normal local store — explicitly not relocated into an App Group in v1. The widget/extension never touches the live database; it only reads a small derived `WidgetSnapshot` DTO (absolute `lastTestDate`, `nextDoseDate`, `riskState`, `activeTankId`, `nextReminderDate` — not precomputed "days ago" strings) written to App Group storage by `SnapshotService`. This keeps the widget's own `TimelineProvider` computing freshness itself at render time, keeps BGTaskScheduler a pure opportunistic optimization rather than a correctness dependency, and caps the blast radius of any App Group/extension misconfiguration at "widget looks stale," never at "main app can't read its own data." Deletes always follow: query dependent counts (applied-dose logs, dose snapshots, water tests, compatibility decisions) → render exact counts in the confirmation sheet → delete only on explicit confirm, with SwiftData's `.deny` rule kept solely as a defensive backstop whose triggering would itself indicate a bug in the counting logic.

**Permissions/privacy plan:** Notification authorization requested only when the user toggles reminders in Settings, never during onboarding; the full test→dose→apply→compatibility loop must work end-to-end with notifications denied, with an explicit test case for it. `PrivacyInfo.xcprivacy` ships despite zero network calls and zero third-party SDKs — a submission requirement, not optional polish. Zero telemetry, zero analytics, zero crash-reporting SDKs, zero network calls anywhere in the app.

**Dependency/license decision:** No third-party packages. Everything needed is covered by first-party Apple frameworks.

**Testing implications:** Deterministic reference-vector tests for the dosing solver (all Alk units, cap application, same-day applied-dose reconciliation) and the compatibility engine (distinct outputs per pairing, bioload-only overload despite clean pairwise checks); SwiftData tests against in-memory `ModelConfiguration`; a store-corruption/recovery integration test proving the app degrades gracefully instead of crashing on a bad container open; permission-denial tests confirming the core loop has zero dead ends with notifications off; an automated WCAG contrast regression test computing real ratios for the teal/coral/violet solid tokens against `#0A0D14` at normal and largest Dynamic Type sizes, failing CI if it drops below 4.5:1; and an `EntitlementStore`/`MockEntitlementStore` pair (plus a local `.storekit` config for Simulator testing) verifying every gated surface behaves correctly with entitlement flipped both directions, including that a lapsed subscription locks (never deletes) a second tank or the expanded species library.

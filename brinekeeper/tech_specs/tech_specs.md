I'm deliberately keeping the repository layer thin — four protocols, not a generic `Repository<T>` abstraction. The only reason they exist at all is that three things must happen together on every write and I don't want that logic duplicated across five feature modules: enforce UUID/business-key uniqueness at insert (never `@Attribute(.unique)`, per the architecture phase's CloudKit-migration concern), fire `SnapshotService` only at real commit points, and compute `DeletePlan` dependency counts before any destructive action is even offered. If a build lane is tempted to reach for `@Query` directly inside a view for something other than read-only display, that's a signal the write should go through a repository instead.

On concurrency: I'm rejecting background `ModelActor` complexity for v1. This app's data volume (five parameters, weekly cadence, one hobbyist) is trivially small — years of history is still a few thousand rows. `ModelContainerManager` vends a single MainActor-bound `ModelContext`, all repositories are `@MainActor`, and the only place I'd consider offloading is JSON import/export parsing (pure `Decodable` work, doesn't need SwiftData at all until the parsed structs are ready to insert). Reaching for actor isolation here would be complexity theater — the honest architectural flex is using the type system to make bad states impossible, not distributing simple work across threads it doesn't need to leave.

The one place I want to be opinionated about performance: Dashboard must query only the latest `WaterTest` per parameter (`fetchLimit: 1`, sorted descending) to feed `ParameterStatus.evaluate`, never a full history fetch. This is a `@Query`-per-parameter or a single predicate query grouped client-side — either is fine, but it must not scale with history length, because paid users get unlimited trend history and a Dashboard that silently gets slower as someone's data grows for two years is exactly the kind of thing that never shows up in a demo and always shows up in a review.

```swift
    case notModeled(parameter: WaterParameter)   // NO3/PO4 always land here
}
```

`DoseTrace` is the thing every view renders — the "just computed" call site builds one fresh, the "reopen historical" call site decodes one out of a `DoseRecommendationSnapshot`. `DoseRecommendationSnapshot` (the SwiftData `@Model`) stores the encoded `DoseTrace` plus `tankId`/`testId` foreign keys for querying and delete-counting — it's a dumb container, `DoseRecommendationSnapshot` itself has no logic. That split is what makes "the view never decides whether to recompute" actually true at compile time rather than by convention: `DoseRecommendationView` takes a `DoseTrace`, full stop, and has no solver dependency at all.

Compatibility is the mirror image, and I want that asymmetry preserved as literally two different function shapes so nobody can accidentally give compatibility a "frozen" mode:

```swift
enum RiskTier: String, Codable { case clean, caution, highRisk }
enum CompatibilityMechanism: String, Codable { case aggression, allelopathy, bioload }

struct MechanismFinding: Codable, Equatable {
    let mechanism: CompatibilityMechanism
    let triggered: Bool
    let detail: String
    let involvedSpeciesNames: [String]
}

struct CompatibilityResult: Codable, Equatable {
    let riskTier: RiskTier
    let mechanismFindings: [MechanismFinding]   // always all three, triggered or not
}
```

`CompatibilityEngine.evaluate(...)` is a pure function with no persistence dependency at all — it always recomputes from whatever roster you hand it. `CompatibilityDecision` (the `@Model`) is only ever written once, at add-time, from the `CompatibilityResult` that was just computed, storing name strings not foreign keys. There's no "load historical CompatibilityResult and render it read-only" code path — the historical screen (if we ever build one) renders the stored strings directly, it never round-trips back through the engine.

**Persistence strategy**

`ModelContainerManager.makeContainer()` never throws into a crash — it returns an enum:

```swift
enum ContainerOutcome { case ready(ModelContainer), recoveryNeeded(underlying: Error) }
```

`RootView` switches on this before it even considers the tank-exists gate: `.recoveryNeeded` renders a dedicated recovery screen (export-what's-readable / start-fresh), `.ready` proceeds to the existing tank-exists-or-not gate from the design phase. Two independent root-level branches, not nested ad hoc.

Deletes always go: `repository.deletePlan(for:) -> DeletePlan` (proactive count query) → render counts in `DestructiveConfirmationSheet` → `repository.delete(...)` only on explicit confirm. `DeletePlan` is a plain struct, not an error type, precisely so the UI can render it before any destructive call is made.

**Testing strategy**

Pure-function tests dominate: `DosingSolver` against known reference vectors per Alk unit, cap-triggered vectors, same-day-applied-dose reconciliation vectors, and NO3/PO4 always resolving to `.notModeled`. `CompatibilityEngine` tests proving distinct `RiskTier`/mechanism output per candidate swap, plus the specific pairwise-clean-but-bioload-overloaded case. `ContrastValidator` gets its own regression test computing real ratios for the shipped hex values against the background at both text sizes, wired into the normal test target so it fails CI, not a manual checklist item. `ModelContainerManager` gets a corrupted-store-simulation test proving `.recoveryNeeded` is reachable without a crash. Repository tests run against `ModelConfiguration(isStoredInMemoryOnly: true)`. `EntitlementStore` tests flip `MockEntitlementStore` both directions across every gated surface (tank count, species library size, history window).

The one thing I'd flag as a real risk in my own proposal: four repository protocols is more ceremony than an MVP strictly needs, and if a build lane isn't careful they could balloon into a generic data-access-object pattern that fights SwiftData's own `@Query` ergonomics instead of complementing them. I want it stated plainly that repositories exist for exactly three jobs — uniqueness enforcement, snapshot-commit hooks, delete-plan counting — and everything else (read-only lists, chart data) goes straight through `@Query` in the view/view-model layer with no repository indirection at all.

### Round 2


**Codex (QA & Risk · Closer) — Round 2**

Codex, I’m still pushing for zero ambiguity here: if we leave any of the previous forked names in play, we’ll just split the implementation and re-merge bugs in the first pull. Claude is right to force this now because the highest failure modes in this build aren’t chemistry math—they’re silent drift and deletion/data-loss edges. My strongest position is to lock the interface surface to one canonical vocabulary and one canonical lifecycle contract, even if it means carrying two extra names in your head during coding.

On the compatibility axis: we should keep the split you both already intended but failed to finalize in clean terms. `CompatibilityResult` is the pure in-memory evaluator return (`CompatibilityEngine.evaluate`), and `CompatibilityDecision` is only the persisted audit snapshot (timestamp + candidate + roster snapshot strings + mechanism summaries + risk tier). That is not stylistic, it is exactly how we keep the trust boundary honest. I’m explicitly rejecting any future “single compatibility type for both purposes” approach.

On persistence/bootstrap: we stick with the `ContainerOutcome`-first API for container bootstrap because it makes “no force-try launch crashes” structurally impossible to forget. If I only saw a `throws`, we’d be one missed `try/catch` away from an app-killer in the field. The app’s very first branch must be container outcome -> setup/recovery -> normal root. `NotificationScheduler` is still okay as a protocol with async request/auth and schedule/cancel methods, and it must only be invoked from Settings, never from onboarding. `ModelContainer` remains local-only in app sandbox and widget reads only a tiny `WidgetSnapshot` payload from App Group (`Date`-based, absolute fields only) at render time.

I’m also closing one more risky point while we’re here: `DoseRecommendationSnapshot` should remain denormalization-light and faithful. Don’t add extra computed fields unless they are strictly derived and documented. Keep `encodedTrace` as source of truth, plus ids and timestamp. If someone wants list previews, decode the snapshot in presenter scope; don’t maintain parallel mutable summary columns that can diverge and make “did this recompute?” bugs inevitable.

Finally, the delete path must be explicitly proactive in both model and UI contracts. `TankRepository.deletePlan(for:)` and `WaterTestRepository.deletePlan(for:)` have to return exact dependency counts, and the confirmation sheet must render those counts before `.delete()` can be triggered. `.deny` remains as an integrity backstop only, and if it ever fires in the wild, we should treat it as an inconsistency bug with telemetry-free user-visible guidance for recovery. If we codify that now, we avoid the “vague couldn’t-delete” failure mode forever.

One more hard QA requirement I want in the same contract: the “no startup crash” and contrast checks are not test suggestions, they are product gates. We already agreed on `ModelContainerManager` recovery and token contrast test; include these as hard acceptance around root boot and snapshot rendering states. That means the interface contract should include a lightweight `LoadState` and error-bearing bootstrap path that the root UI actually branches on.

Codex, on the `CompatibilityDecision` collision — I went back and actually checked, and this isn't a coin-flip naming choice, it's a case where one of us already deviated from something the *previous* phase locked. `ios_architecture_review`'s own final output literally lists `CompatibilityDecision snapshot` as the audit record in its data model plan, and `design_handoff` calls it out explicitly as "a separate, small `CompatibilityDecision` audit record (timestamp, input snapshot, resulting tier(s), mechanism string(s))." That's not up for a fresh vote — it's a name two prior consensus phases already assigned to the persisted `@Model`. So I'd ask you to rename your ephemeral live-evaluation struct to something else — `CompatibilityResult` works fine, or `CompatibilityCheck` if you want a verb-flavored name — and keep `CompatibilityDecision` for the thing that gets written once at add-time. I also want to push on something else in your draft while we're here: you've actually got *three* types doing this job — the live `CompatibilityDecision` struct, plus a separate `@Model CompatibilityDecisionRecord` that stores `snapshot: String`. That's one too many, and the serialized-string-blob shape on top of it is worse than what it replaces: the whole reason design_handoff wanted "resulting tier(s), mechanism string(s)" as named fields instead of an opaque payload was so old compatibility logs render coherently without a decode step. If `mechanismSummaries` is just `[String]` and `riskTier` is an enum, there's no reason to wrap them in a JSON blob — store them as real SwiftData properties. Two types total: an ephemeral `CompatibilityResult` that `CompatibilityEngine.evaluate` returns and that never touches disk, and one `@Model CompatibilityDecision` with flat, directly-queryable fields written exactly once. I'll die on this hill because it's the literal mechanism that makes "dosing freezes, compatibility is live policy that freezes only at the audit write" true in code, and three overlapping types is exactly how that asymmetry gets muddled during implementation.

On `ModelContainerManager` — you went `async throws -> ModelContainer`, I went a non-throwing `ContainerOutcome` enum. I want to actually make the case harder than I did last round instead of just re-asserting it: SwiftData's `ModelContainer(for:configurations:)` initializer isn't async at all, so the `async` in your signature doesn't buy anything real — it just adds an await at every call site for no functional reason. **Codex (QA & Risk · Closer) — Round 3**

I’m going to close this on the hard-architecture side and not leave ambiguity for a later phase to discover: we should freeze to a synchronous, non-throwing container bootstrap with explicit recovery branching at the App root, keep Widget communication to a tiny DTO only, and cut all optional-but-easy scope creep (ActivityKit) that doesn’t earn a concrete user value in v1. **Codex (QA & Risk · Closer) — Round 4**

This is where we lock it down and stop looping. Codex, Claude, your convergence is real, but I’m pushing one more hard line before we call this phase done: the architecture is stable only if we make failure states explicit and machine-checked, not just “we’ll handle later.” The biggest trust breakers for this app remain: 1) silent correctness drift in recommendation rendering, 2) delete ambiguity, 3) crash-on-launch fallback quality, and 4) accessibility regressions hidden behind eye candy. We’ve already resolved most of that, but I want to tighten two concrete implementation details that are too easy to regress:

First, don’t over-constrain model types with `Sendable` on `@Model` classes. That is technically unnecessary here and can become compiler friction because SwiftData model classes aren’t always safely shared across actors the same way plain structs are. Keep concurrency boundaries by putting all model mutation through `@MainActor` coordinators/repositories; keep pure helpers pure and value-typed. Second, the `WidgetSnapshot` risk state should represent dashboard-facing “health aggregate” as a typed token, not just one `ParameterStatus` intended for a single parameter, because the widget is supposed to show “days since last test / next dose due” at the tank level. A simple aggregate enum (`widgetRisk: .ok/.warning/.blocked`) alongside the two absolute dates is safer and future-proof than reusing a per-parameter enum there. Everything else can stay as in the current contract.

I’m still strongly in the “no ActivityKit” camp with zero scaffolding, and I want this repeated as a non-negotiable release gate: if a feature isn’t in `computeAndPersistRecommendation`, `checkCandidate`, `deletePlan`, `export/import`, or `notification toggle`, it doesn’t belong in v1 even if it looks cool in one-off demos. The same principle applies to compatibility and dosing contracts: keep `DoseRecommendationView` as a pure renderer of a prebuilt frozen `DoseTrace`, and keep compatibility live/eager-to-recompute. That asymmetry is the only thing that prevents accidental stale math and user mistrust. On delete safety, keep the sequence immutable: `deletePlan(...)` first, UI text from that plan, then explicit confirm; `.deny` only as integrity tripwire afterward. For launch safety, `ModelContainerManager.makeContainer()` + `RecoveryView` on `recoveryNeeded` must be executable deterministically and tested against an induced-corruption container. For accessibility, keep the current automated WCAG check on exact tokens vs `#0A0D14` (normal + largest text) and fail CI when ratios drop below 4.5:1.

### Round 8


**Codex (QA & Risk · Closer) — Round 8**

I’m going to push this to the hard finish line and not leave anything dangling. Codex and Claude are aligned on the main stack, but the next thing that can quietly sink implementation is cross-actor leakage: you can still end up passing `@Model` references through non-main paths and reintroducing races, and nobody notices until a random “Publishing changes from background threads” crash in field. I want a concrete lock: `Tank`, `WaterTest`, `DoseRecommendationSnapshot`, `AppliedDose`, `LivestockSpecies`, etc. stay main-actor owned entities and never appear in `Sendable` signatures. Services can be pure and testable; repositories may be `@MainActor` and return immutable DTOs (`DoseTrace`, `CompatibilityResult`, `ImportSummary`, `DeletePlan`) for any cross-layer handoff. That is the single architectural decision that prevents a lot of later “it crashes only when saving while rendering the chart” bugs.

There’s also one more thing I want explicitly non-negotiable in code, not in docs: `DosingSolver` and `CompatibilityEngine` cannot be called from view render paths (`body`, `onAppear`, chart bodies). The body is not just a function call site; it is a re-run engine. We already say this in prose, but we should codify it as an internal API shape. Put one command-path facade (`DoseCommandService.computeAndPersistRecommendation(...)`) that is the only entrypoint used by Tests→Save and history-reopen, and only that path updates snapshots, writes app-group widget state, and emits advisory rows. Similarly, compatibility “live check” must come from explicit memoized query inputs (roster + rule cache), not from stale state in mutable `@State`.

On persistence and failure mode hardening, I’d like to keep current choices and sharpen the behavior around recovery: container corruption triggers recovery immediately and deterministically (show `RecoveryView` with export/restart), and import failures are schema-first. But even with schema versioning, partial-import failure must remain explainable and non-destructive: import a corrupt record, continue with valid subset only if policy says recoverable; otherwise reject whole payload with machine-readable `ImportExportError.partialFailure` so no one assumes a “mostly imported” lie happened silently. That plus immutable `DoseTrace` snapshots gives auditability all the way through.

My strongest remaining nudge: make a small “dependency plan” contract non-optional everywhere destructive actions surface. Do not build delete off view counts that can drift; compute `DeletePlan` from repository query methods, render exact counts, require explicit confirm, then execute. Keep `.deny` as an invariant check only. Also keep subscription downgrades locked, not destructive, and don’t hide owned historical data from the free user—freeze actions and surface why.

The concurrency + actor rule plus this command-path discipline are the two biggest reliability gates for this MVP. Everything else is already decided and can be executed as-is: root-gated setup (not modal), no Live Activity, no App Group store move for SwiftData, widget snapshot DTO only, absolute dates in snapshot, no force-try container open, contrast-automation in CI, and zero analytics/telemetry. I’m fine shipping with these as the tech-spec contract.

**Claude (Frontend Engineer · Systems Thinker) — Round 1**

The navigation reconciliation from the design_discussion phase is solid, but nobody's yet worked through what it costs in actual SwiftUI state architecture, so let me start there before getting into the visual system, because a couple of the locked decisions have real second-order consequences for how this gets built.

Four tabs, each independently navigable, means four separate `NavigationStack`s living inside a `TabView`, which is the correct default — it's what gives you native back-stack behavior per tab and matches how iOS users expect tab switching to preserve where they were. But the Dashboard's flagged-parameter card is specified to jump straight into that parameter's Dose Recommendation screen, and that screen only conceptually exists inside the Tests tab's stack. That's a cross-tab push, and it's the first place a naive implementation will quietly break: if each tab's `NavigationPath` is purely local `@State`, there's no way for a Dashboard button to push onto a stack it doesn't own. The fix is to lift a shared `NavigationPath` (or a per-tab dictionary of paths keyed by tab) into an `@Observable` coordinator that both Dashboard and Tests read from, so "tap flagged Alk card" becomes "set selected tab to Tests, then append `.doseRecommendation(testID:)` to Tests' path" as one atomic action, not two separate state writes racing each other across a tab change. I want this called out explicitly in the handoff so whoever builds it doesn't discover this the hard way three screens in — same pattern applies to notification taps eventually (a reminder notification tapped from the lock screen needs the exact same cross-tab deep-link capability, so build the coordinator generically now even though notifications are must-have and deep-linking from them isn't separately scoped).

Second thing worth locking now because it's a data-flow decision, not a visual one: the Dose Recommendation screen has two fundamentally different data sources depending on how you arrive at it, and the view needs to model that as a real type distinction, not an if-branch on optionals. Arriving via "just saved a test" means computing a fresh trace against live tank/product state. Arriving via "tapped a historical test row" means loading a frozen, previously-persisted trace snapshot and rendering it read-only-except-for-the-apply-action. If this is built as one view model with a "maybe recompute, maybe don't" flag, someone will eventually wire the wrong path and violate the frozen-snapshot guarantee that three prior phases spent real effort locking in. I'd specify this as two distinct call sites feeding one shared presentation component: `DoseRecommendationView(trace: DoseTrace)` where `DoseTrace` is always a fully-formed, already-computed value struct — the "compute new" path builds one via the solver and pushes it; the "view historical" path fetches one from SwiftData and pushes it. The view itself never calls the solver. That's the boundary that keeps this correct.

On the compatibility side there's an inverse issue worth flagging: the roster check is specified as always-live (never frozen), which means the Livestock tab's compatibility result view has to re-run against current SwiftData state every time it's shown, including if the user backgrounds the app mid-check and a roster edit happens elsewhere (unlikely in a single-user local app, but the pattern still matters for view lifecycle) — so this should be built as a computed value driven by `@Query` on the roster plus the candidate selection, recalculated in the view's body or a lightweight `.task(id:)`, not cached into `@State` that could go stale. The one thing that must be frozen here is the `CompatibilityDecision` audit record written at add-time — so there are two rendering paths again: the live check screen (always recomputes) and a historical decision log entry (renders the stored snapshot). Same pattern as dosing, opposite default.

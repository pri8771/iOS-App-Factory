Reef aquarium chemistry & livestock management

## One-Sentence Promise

Turns a water test into an exact dosing plan and warns you before a new fish or coral wrecks your tank.

## Target User

Serious saltwater reef aquarium hobbyists who test water parameters weekly and actively manage coral/fish stocking.

## Painful Moment Solved

After a water test you have raw numbers (Ca, Mg, Alk, NO3, PO4) but have to hand-calculate reef chemistry to know exactly how much of which product to dose without overshooting and crashing the tank, and you're guessing whether a new species will fight or chemically poison something already in there.

A real deterministic dosing-optimization solver (tank volume + current + target parameters + product concentration → exact dose) and a livestock compatibility graph that reasons about bioload, aggression, and allelopathy instead of a static compatibility list; bioluminescent glass-morphism visual language with glowing coral-accent gradients, distinct from any tracker template and from existing single-purpose dosing calculators.

## Why Users Would Pay

Reef hobbyists already spend heavily on test kits and equipment; subscription unlocks multi-tank management, parameter-trend analytics over time, and unlimited livestock graph size.

## Subscription Model

Free tier: one tank, basic dosing calculator, manual logging. Paid: multi-tank support, trend analytics, unlimited livestock compatibility graph, data export.

## Local-First MVP Scope

SwiftData models for tanks, water tests, dosing products, livestock, and compatibility rules; dosing solver and compatibility engine run fully on-device; local notification/BGTask reminders for test/dose schedule.

## Future Backend Roadmap

CloudKit sync across a hobbyist's devices; optional shared tank profiles for clubs — no rewrite required since the solver and data model stay local-first.

## AR / Local ML / Local LLM Fit

On-device Vision for optional test-strip colorimetric reading; no LLM use — chemistry and compatibility logic must stay deterministic and auditable.

## Design Direction

Bioluminescent depths: near-black abyssal backgrounds, glowing coral-reef accent gradients (teal/violet/coral), glass-morphism data cards, luminous line graphs for parameter trends.

**Codex (Design Lead · Skeptic) — Round 1**

If this is Brinekeeper-only, the fastest path to a shippable first build is to treat it as a precision safety tool for a small, experienced cohort, not a broad hobbyist lifestyle app. The likely audience is advanced reef keepers who already run weekly water tests, own dosing hardware, and value reducing uncertainty; they’re used to charts, units, and dosing math and will tolerate a more technical UI. Assumption: these users are willing to trade extra taps for transparency, especially if each recommendation shows the formula, units, and safety rationale.

Comparable patterns are strongest in three buckets: aquarium diaries that optimize for journaling first, dosing calculators that optimize for one-off formulas, and rule-based checklists for compatibility advice. The gap is connecting all three in a deterministic, auditable workflow where “what to dose” and “what stock is risky to add next” are produced from shared tank state and clear guardrails. Assumption: competitors currently silo these functions and often leave safety framing implicit or absent.

On iOS, native behavior should feel mandatory, not decorative: Home Screen widget for “days since last test / next dose” provides immediate value as a glance surface, BGTask reminders is a must for weekly maintenance rhythm, and Live Activity can add tangible UX value by replacing sticky “start/pause/complete dose” modal friction during a mixing session. If those features feel optional in MVP, the value proposition weakens; users need to see the iOS-first advantage quickly.

The biggest platform-specific design opportunity is reducing “math anxiety” with high-density, instrument-like visuals: parameter cards, trend sparklines, and staged dose cards where each dose card expands to a trace (“from x to target x, at product concentration y, applying max-safe limiter z”). This is a strong differentiator because it matches how serious reef hobbyists reason, but it’s also a risk: if the math feels hand-wavy, trust collapses. Assumption: users will not forgive a “wizard” feel.

The largest unknown is data quality. We can implement deterministic compatibility logic, but whether it feels credible depends on seeded relational data quality. Assumption: seed species/rule coverage can be enough for v1 without pretending to be exhaustive; if not, we must hit the fallback and ship no-complexity compatibility rather than fake depth. Related unknown: correct stoichiometry constants (e.g., conversion factors and delta-per-dose equations) and acceptable safe-step caps; we need explicit, conservative bounds, and transparent “this is capped because…” messaging.

A second unknown is permission and adoption risk: BGTask and notifications are often denied or delayed, so the app must degrade into passive scheduling without dead-ends. Assumption: a strong in-app explanation of required permissions plus manual “test/dose planned” actions can maintain perceived value even with full denial.

A third unknown is accessibility under the requested glow-heavy style. Assumption: if we don’t validate contrast and dynamic type against actual surfaces, this becomes a premium-looking but unusable product. The “bioluminescent” requirement therefore becomes a design system constraint, not a decoration layer: high-contrast text on dark surfaces, strong VoiceOver labels, and 44pt targets must be built-in decisions from day one.

From a first-build perspective, the implications are: focus on the six workflows in a tight loop where each outcome is immediately actionable; avoid abstract screens; keep onboarding to tank setup + one test + one dose + one livestock check. Add explicit empty states (“Add your first water test to unlock dose planner”) and explainable empty-to-success transitions. Include deterministic engine tests from the start, especially for unit mix math and compatibility warnings across different bioload/aggression/allelopathy combinations, so the interface claim and mathematical claim stay aligned. The first version should feel smaller but complete rather than broad and shallow, with a hard stop on fake sophistication.

**Claude (QA & Risk · Pragmatist) — Round 1**

Reef hobbyists who test weekly are a specific, well-documented subculture — they hang out on Reef2Reef and r/ReefTank, follow Bulk Reef Supply's "52 Weeks of Reefing" content, and already own Hanna checkers or Red Sea/Salifert kits plus a dosing pump (Kamoer, GHL, or a Neptune Apex). That matters because the comparable products aren't generic habit trackers — they're Neptune's Apex Fusion app (which is controller-tied and shows live sensor data but doesn't do a real dosing solver), GHL's ProfiLux app, BRS's static web dosing calculators (one-shot forms, no persistence, no trend history), and simple logging apps like Reef Companion or "My Reef" that record numbers but don't compute anything meaningfully. None of those combine a persisted trend history with an actual solver and a real compatibility engine in one native app with this level of design intent — that's a genuine gap, not just a design refresh of an existing category.

On platform fit: the home screen widget (days since last test / next dose due) is the single most native-feeling piece of this spec, because the actual user behavior is "I test on Sundays and forget mid-week" — a glanceable widget solves a real recurring friction point for a weekly-cadence habit. I'm less convinced about the Live Activity for "an in-progress dosing session" as specified — dosing a two-part solution is typically a two-minute action (measure, pour, log), which is thin justification for a Dynamic Island / Lock Screen activity. I'd flag this as an assumption to pressure-test in the design phase: either "dosing session" actually means a longer multi-step maintenance routine (water change + multiple product doses + retest), in which case a Live Activity earns its keep, or it's a stretch feature that shouldn't get build priority over the six core workflows. I don't want us building a Live Activity nobody asked to use in practice while the trend charts are half-done.

There's also a technical mismatch worth naming now so architecture doesn't inherit it: the spec says "BGTaskScheduler for test and dosing reminders," but BGTaskScheduler is for periodic background compute (recomputing overdue status, refreshing widget data), not for firing user-facing reminders at a specific time — that's UNUserNotificationCenter with calendar/time-interval triggers. If we don't separate these in the architecture phase, we'll end up with reminders that fire unreliably because iOS doesn't guarantee BGTask timing the way local notifications guarantee scheduled delivery.

Three concrete risks I'd want on record before we get into product debate: first, the dosing chemistry itself is not actually invented risk — Randy Holmes-Farley's reef chemistry formulas (raise-per-dose for Ca/Mg/Alk against volume and product concentration) are long-published and public domain, so "deterministic and explainable" is achievable if we implement known stoichiometry with cited formulas rather than reinventing chemistry, and we should have known input/output reference vectors to test against, not just eyeballed math. Second, the compatibility graph's realistic scope is the actual open question — getting real sourced bioload/aggression/allelopathy data for even 30-50 common species (clownfish vs. damsels, Galaxea's allelopathy toward neighboring corals, urchins vs. certain fish) is a research task with a real time cost, and if we shortcut it we get exactly the "static list in a trenchcoat" failure mode called out in the invalidation criteria — better to ship fewer, well-sourced species relationships with an in-app "why this warning" explanation than a padded shallow list. Third, every iOS-native feature here (notifications, BGTask, widget, Live Activity) is permission-gated, and the core value — log a test, get a dose, see the warning — has to work completely for a user who denies every permission prompt, since that's a common and testable failure path, not an edge case.

The implication I'd carry into the build: treat water-test logging + the dosing solver + trend charts as the must-nail spine, since that alone is usable and valuable even single-tank/free-tier; treat the compatibility engine as the second pillar but scope its seed data deliberately small-and-real rather than large-and-thin; and treat Live Activity specifically as the one iOS-native item that should flex or drop first if time runs short, since its user-value premise is the shakiest of the four native features listed.

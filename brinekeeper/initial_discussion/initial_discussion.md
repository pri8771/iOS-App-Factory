Brinekeeper’s promise is: it turns raw reef water tests into a safe, auditable dosing plan and turns livestock add-to-cage decisions into deterministic compatibility risk alerts.

The core loop is: define a tank profile → log test values → set target and preferred products → app computes constrained dose with visible formula trace and safety caps → log dosing action → trend update across time → add/edit livestock → compatibility engine returns risk tier + mechanism + mitigation, then schedule and execute reminders.

Hard scope boundaries are strict: deterministic chemistry and compatibility logic only, entirely on-device, no remote chemistry APIs, no LLM, no photo/Vision strip capture in v1, no cloud sync, no backend-dependent paywall, and no pseudo “AI” or fake intelligence—if data can’t support true graph reasoning, the product must scale back to a single-tank deterministic calculator with explicit warning and not pretend there is meaningful compatibility depth.

The big success bar is trust, not bells and whistles: Dana must be able to use the app safely without spreadsheets. Success is measurable if (1) a complete test→recommendation cycle works in one flow on every path (missing data, out-of-range values, duplicate tests handled without crash), (2) dose math can be traced from inputs to ppm/°dKH deltas and dose amount with conservative caps that block dangerous single-step changes, (3) compatibility checks produce different outcomes for different species-bioload-aggression-allelopathy combinations, (4) core reminders and chart/history states are usable even with notifications denied, and (5) all six v1 workflows are reachable with explicit empty/loading/error states and no placeholder screens.

The strongest objection I see is that this can drift into a “fancy dashboard” while still not being safer than a notebook. **Claude (Design Lead · Systems Thinker) — Round 1**

Let me look at this as someone thinking through actual screens, actual navigation, and where the seams will show under real use.

The one-sentence product promise I'd commit to: "Log your water test, get the exact dose to fix it, and know before you buy whether a new animal will hurt what you already have." Anything the app does that doesn't serve one of those two sentences is decoration.

On information architecture — I think the natural shape is a single tank-scoped hub with four destinations hanging off it: Dashboard (the "how am I doing right now" glance — parameter status chips, days-since-last-test, next-dose-due), Tests (log + trend history), Dosing (recommendation + trace, and this has to be reachable both from "I just logged a test" and standalone as "just tell me what to dose today"), and Livestock (roster + add-new-species compatibility check). Tank switching, if multi-tank is unlocked, should be a top-level control, not buried in settings, because for a paid user the whole app reparents around "which tank am I looking at right now" — get that wrong and every screen underneath has to be rebuilt to be tank-aware later.

The core loop I want to nail as an actual sequence of screens: user opens app → dashboard shows either "everything nominal" or a flagged parameter → tap flagged parameter → water test log screen prefilled with last-test values as a diff baseline → save → immediately routes into the dosing recommendation for whatever drifted, not a dead-end "test saved" toast. That transition — test log to dose recommendation — is the single most important navigational connection in the app, and I'd treat a version where saving a test doesn't proactively offer a dose calc as a failed design, because it re-creates exactly the two-app-context-switch problem (test kit app + separate calculator) the product promise says it's fixing.

The screen I think is highest-risk and needs the most design thought is the dosing trace itself. It's not one number — it's a small structured explanation: current value → target value → delta needed → product + concentration → computed volume → any cap applied and why. I'd design this as an expandable card, collapsed by default showing just the answer ("Add 42 mL of Alk Buffer today"), tap to expand into the full trace. If it's always-expanded, it's noisy for a user who's done this fifty times; if it's never expandable, it fails the "auditable, not black box" requirement. Same logic applies to compatibility warnings — collapsed severity badge, expandable mechanism explanation ("Torch and Hammer are both Euphyllia — sweeper tentacle overlap risk at your rack spacing"), not a modal that interrupts the add-livestock flow.

Second-order consequence I want flagged now, before architecture locks the data model: if dosing recommendations are generated from a snapshot of "current parameters," what happens when a user logs a new test the same day after already dosing once? The recommendation has to know about doses already administered today against the same target, or it will recommend a second full dose on top of one already in the tank and actually cause the overshoot the whole app exists to prevent. This isn't a UI nicety, it's a correctness requirement on the solver's inputs — the dosing screen needs a visible "already dosed today: X mL of Y" state, and the empty/success/error state set for the dosing screen must include "you already hit today's safe cap, here's why we're not letting you add more," not just refuse silently.

Boundary I want explicit: the livestock roster and compatibility check should not try to be a full stocking-plan simulator (no "here's your ideal future roster" recommendations) — it answers one question, "if I add this now, what happens," against what's currently in the tank. Trying to do more turns it into a different, much bigger product.

Measurable success criteria I'd hold this to: a user can go from opening the app to seeing an exact dose number in three taps or fewer from a cold dashboard; a user can get a compatibility answer for a candidate species in two taps from the roster screen; every one of the four destination screens has a real, non-generic empty state that tells the user the one next action that unlocks it (not "no data yet"); and the free-tier single-tank experience must be a fully complete, non-crippled version of this loop — the paywall gates *more tanks and more history*, not a worse version of the core loop, otherwise the free tier reads as a demo rather than a real product and undermines trust before the sell.

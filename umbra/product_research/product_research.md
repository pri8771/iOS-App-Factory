That second segment is what makes the designer/share mode more than a checkbox — it's the app's only realistic growth loop, since the contract explicitly rules out any in-app social/publishing layer.

**Comparable apps or patterns to learn from** (pattern-matches from general knowledge, not fresh research — labeled as assumptions): Shadowmatic-style silhouette-fit intuition for the core mechanic; Monument Valley as the benchmark for premium spatial clarity and onboarding discipline, not mechanics; level-code-sharing conventions from editor-driven puzzle/community-level games, with the standard failure mode being that the export format quietly drifts from the play format as the game gets patched, breaking old shared codes.

**Platform-specific opportunities:** Haptic snap-detents when an occluder or light source aligns meaningfully (edge, angle) — cheap to build, reads as premium. Native Share Sheet for exporting/importing the level code, satisfying "share mode" without any custom social infrastructure. VoiceOver treated as a real design problem, not a labeling checkbox — a spatially-driven puzzle needs an actual strategy for how a screen-reader user builds a mental map of light/occluder/target positions. iPad as a legitimately better first-class canvas for the editor, worth designing for deliberately rather than just "scales up." Dynamic Type and high-contrast support as core requirements, since alignment judgment is the entire mechanic.

**Major assumptions and risks:**
1. Geometry determinism at the edges — zero-area shadows, edge-only overlaps, near-collinear/rotation floating-point precision — is where a silent wrong-answer bug lives, and it's more dangerous than a crash because nobody notices.
2. The round-trip encode/decode/diff test proves no data loss today but says nothing about forward compatibility once new occluder types get added later; a schema version field should exist from day one even though nothing needs multiple versions yet.
3. Randomized fuzz testing only catches the dangerous cases if it's biased toward near-degenerate geometry — uniformly random placement will pass every test while a real bug ships.
4. The editor must not be able to save a level the solver can't correctly grade (overlapping occluders, unachievable target overlap, light source inside an occluder) — the fix is running the same solve-checker as a save-time validator, not a separate "valid level" rule set.
5. Authoring five levels that are genuinely solvable, fun, and free of unintended shortcut solutions is likely the hardest problem in the build, harder than any of the engineering.
6. Onboarding has a tight window — if a player can't discover the grab/rotate/translate affordances in 30-45 seconds, level quality won't save the first impression.
7. A corrupted or truncated share code is the single point of failure for the app's only growth mechanism, since the recipient didn't cause the corruption — the import failure state needs to be genuinely useful (explain what went wrong), not a generic error, or the share loop dies at the exact moment it was supposed to work.

**Implications for the first build:** Structure the build around three flows that map directly to the one-sentence promise — Play, Edit, Share — with Edit built as a strict superset of Play's controls (drag, rotate, duplicate/delete, reset) so there's never a "the rules are different in edit mode" trust break. Build and test the geometry/solve engine and the canonical level format before UI polish, since editor, share format, and accessibility cues are all downstream of that data model being right. Nail the share-artifact format early (with a version field) since play, edit, export, and import all depend on its shape. Auto-save should be invisible until it matters, then visibly reassuring on relaunch. Monetization shape (free starter vault, paid challenge packs, pro editor tier) should be visible in the product framing from the first build, even if the purchase flow itself isn't wired up yet.

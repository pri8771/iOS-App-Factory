## Final Output

**SwiftUI architecture recommendation:** Split the geometry/solve engine and the canonical level schema into a local Swift Package, UmbraCore, importing only Foundation — no SwiftUI, no UIKit, and Observation kept out of the pure-geometry types. UmbraCore owns the Level/Occluder/LightSource/Target structs, encode/decode, the solve-checker, and the save-time validator. The app target imports UmbraCore and wraps it in the already-locked `@Observable LevelSessionModel` and `VaultStore`. This boundary is what lets the 50-run determinism replay and the fuzz suite run fast and reproducibly via `swift test` with no simulator boot — a slow or flaky suite is one nobody runs before merging, and this is the one architectural risk most likely to quietly undermine the project's core "no silent geometry drift" guarantee if it's skipped.

**Apple framework choices:** SwiftUI for all UI; Observation for the two model types; Foundation for Codable/FileManager; Core Graphics via SwiftUI's `Path`/`Canvas` for shadow rendering, with `CastShadowRenderer` doing closed-form polygon clipping (not sampling/ray-marching) so it stays cheap; CoreHaptics or `UIImpactFeedbackGenerator`/`UINotificationFeedbackGenerator` for the solve haptic; `ShareLink`/`UIActivityViewController` for export; `UIPasteboard.general.string` read only on an explicit user-tapped "paste from clipboard" action, never auto-read on appearance; StoreKit 2 (`Product`, `Transaction`, `Transaction.currentEntitlements`) as a defined-but-unimplemented seam for future paid tiers.

**Persistence and local data plan:** No SwiftData, no CoreData — plain Codable structs, `JSONEncoder`/`JSONDecoder` with an explicit key strategy and a `schemaVersion` field, written to files in Application Support (one vault-manifest file, one file per level, a separate small file for in-progress edit-draft state so a mid-edit crash can't corrupt the vault manifest). All writes are atomic — write to a temp file, then `FileManager.replaceItemAt` — specifically because forced-quit-mid-save is a locked requirement and a half-written file is a real corrupted-save scenario to survive, not just plan around. `UserDefaults` is used only for small flags like the onboarding-seen boolean, never for anything level-shaped.

**Permissions/privacy plan:** Zero runtime permission prompts — no camera, location, contacts, microphone, notifications, or network access, fully offline by construction (no ATS exception needed since there's no networking code at all). Default App Sandbox entitlement only — no App Groups, no iCloud container, no push, no background modes — stated explicitly rather than left as whatever Xcode's template generates. A `PrivacyInfo.xcprivacy` privacy manifest is still required even with zero data collection, called out now as a concrete build task so it doesn't get missed at App Store submission; the actual privacy story is "no data collected," full stop.

**Dependency/license decision:** Zero third-party dependencies. Every capability above is first-party. This is stated as a hard decision, not a default, because a convenience package for JSON diffing, geometry, or a share-sheet wrapper would introduce license/supply-chain risk for a problem the app's own geometry code already has to get right itself.

**Testing implications:** The geometry/solve-checker suite lives in the UmbraCore package and uses a custom, explicitly-seeded PRNG (not `SystemRandomNumberGenerator`) so any fuzz failure is deterministically replayable and can be locked in as a new fixture. Serialized-snapshot comparisons use a custom equality check, not synthesized `Equatable` on raw `Double`s, matching the standing two-tier model — epsilon (~1e-8) for live/replay geometry, exact structural equality for canonical serialized snapshots. UI testing deprioritizes XCUITest simulated drags in favor of calling `LevelSessionModel`'s mutation functions directly (the same functions gestures and the discrete VoiceOver controls both route through), plus a thin smoke-level XCUITest pass (launch, open the starter level, confirm the solve sheet appears). ## Final Output

**Architecture overview:** A local Swift Package, `UmbraCore` (Foundation-only, no SwiftUI/UIKit/Observation), owns all pure geometry, the canonical `Level` schema, the solve engine, and the save-time validator. The app target wraps it in a thin `@Observable` layer. Every level mutation — drag, rotate, duplicate, delete, reset, nudge-arrow, rotation-stepper — is a case of one closed enum, `LevelMutation`, and the only way to change level state is `LevelSessionModel.apply(_ mutation: LevelMutation)`. That function is synchronous: it computes a new `Level` value from the current one via pure functions and then assigns it — no `Task`, no `async`, no in-place array mutation — because gesture input, the discrete VoiceOver control cluster, and the 50-run determinism replay/fuzz tests all need to be different callers of the exact same code path, not three things that could quietly drift apart. `SolveEngine` is pure and stateless; overlap between the target polygon and a possibly-non-convex union of rectangle-occluder shadows is computed via a fixed-resolution deterministic scanline quadrature (2048 horizontal strips across the canvas bounding box, exact analytic per-strip trapezoid math over crossing edges) — this is the one hard computational-geometry risk in the project, and the decision is to solve it with first-party closed-form arithmetic rather than attempt general polygon boolean ops from scratch or reach for a geometry library, which the standing zero-dependency policy already rules out. A single `LevelStore` protocol is the cloud-ready persistence seam locked in architecture_review, with `FileLevelStore` as its only conformance for this build. There is no networking anywhere in this app — no `URLSession`, no network-shaped protocol seam, not even an unused one — matching the ios_architecture_review decision that the entire privacy/permissions story is "fully offline by construction."

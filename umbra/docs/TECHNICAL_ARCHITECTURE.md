**State model per screen:** Vault — empty/pre-seeded-first-run (ships with a starter level so true empty is rare), loading (skeleton grid on cold launch), success, and a distinct corrupted-save error state with honest copy and a recovery action (never a silent blank grid). Dynamic Type at the largest accessibility sizes must not let the control cluster overlap the puzzle geometry — needs testing now, not discovered later.

**Screen-by-screen specification**

Edit mode is visually identical to Play with one difference: the target silhouette becomes editable, plus a small "Editing" label/border treatment — never a screen navigation. - **Solve confirmation:** Not a screen — a state Play transitions into in place. Canvas holds still, outline locks, a soft highlight-sweep plays (with a straight cross-fade fallback under Reduce Motion — the cue itself is mandatory, only its motion style adapts), haptic fires, and a bottom sheet slides up showing the actual overlap number large (not just "Solved!") plus two actions: Remix or Back to Vault. The sheet never auto-dismisses or times out.

- **Share:** Triggered from Edit's save sheet or a level's Vault detail, via the system Share Sheet only — no custom share UI.

- **Import:** One dedicated screen off Vault. States: empty (paste field, explicit "paste from clipboard" button, never silent auto-read), loading (parsing), a preview-before-commit success state (shows the incoming thumbnail, explicit "add to vault" confirm — never auto-added), and three separately worded, separately iconed error states (truncated/malformed, unsupported schema version, geometrically-invalid), each with a one-line plain-language explanation plus technical detail behind a disclosure.

- **Onboarding:** Two to three coach-marks anchored to real live canvas elements (via `PreferenceKey`-reported frames, not a screenshot), gated by a persisted "seen" flag, dismissed by performing the taught gesture — never a separate tutorial mode with different rules than the real game.

**Design tokens / visual system**

Near-black canvas background, one or two slightly elevated surface tones for chrome (toolbars, sheets); a warm accent color reserved specifically for the light source; desaturated neutral tones for occluders so they don't compete with the shadow. Two distinct, non-interchangeable shadow tokens: a small "object elevation" drop-shadow style (occluders, cards) versus the cast puzzle-shadow, which is a live geometry render from the solve engine, never a decorative effect — conflating the two in implementation is flagged as a specific risk. Spacing on a 4/8/16/24/32 scale. Generous corner radii on cards/sheets. Typography: system Dynamic Type styles only, no custom font; monospaced-digit variant specifically for the overlap readout (in-canvas and on the solve sheet) so the number doesn't jitter the layout; headline weight for the solve-sheet number. Semantic states (close-to-solved, solved, error) expressed via shape/icon/weight, color as reinforcement only.

**Component inventory**

`LevelCardView`, `VaultGridView`, `PuzzleCanvasView` (mode-parameterized: `.play` / `.edit`), `LightSourceHandle`, `OccluderHandle`, `TargetSilhouetteView`, `CastShadowRenderer` (pure geometry-to-path, no gesture logic), `SelectionControlCluster` (floating and docked layout variants), `OverlapReadoutBadge`, `SolveConfirmationSheet`, `EditSaveShareSheet`, `ImportScreen` (with its four state subviews), `CoachMarkOverlay`, `CorruptedSaveErrorView`.

**State and interaction notes**

One shared `LevelSessionModel` (`@Observable`) owns light source, occluders, target, and selection; every mutation — drag, rotate, duplicate, delete, reset, nudge-arrow, rotation-stepper — goes through the same functions on that model regardless of mode or input source. `VaultStore` holds the persisted level list plus the currently-open session. Drag/rotate gestures mutate a transient in-flight transform and commit only on gesture end; an interrupted gesture (e.g. app backgrounded mid-drag) rolls back to the last committed state, never persisting a partial frame. Live shadow recompute is driven directly by committed+in-flight transform on each gesture callback, not by SwiftUI body re-evaluation, to keep it cheap and isolated from unrelated state changes. Control-cluster layout: anchors to the selected object's bounding box with a flip order (below → right → above → left) to avoid canvas-edge/silhouette collisions at standard type sizes; at AX1+ Dynamic Type or below a canvas-size threshold, it collapses to a fixed bottom dock with the same buttons/functions — same underlying model, two container layouts, tested explicitly at the transition boundary.

**Accessibility requirements**

The nudge/rotation-stepper control cluster is a visible, always-on affordance for everyone (not VoiceOver-only), appearing near the selected object. Every draggable object gets a plain-language accessibility label including approximate position/rotation. The overlap readout announces via accessibility notification on meaningful change (throttled — at minimum on crossing the close-match/solved thresholds and on request), not on every frame. Dynamic Type tested at every step, not just the extremes, since the floating→docked control-cluster transition point is the likeliest place for a layout bug to hide. Reduce Motion gets a cross-fade substitute for the solve-confirmation sweep, never a dropped cue.

**Claude Design decision**

No external Claude Design pass for this build — the interaction structure (shared mutation model, anchor/dock control-cluster logic, shadow-token distinction) is architecture-bearing, and a visual-only design pass done separately risks producing screens that look right but weren't built with those constraints in mind. If a later, purely cosmetic reskin is wanted, the scoped prompt to use is: *"Premium, restrained dark-canvas puzzle app called Umbra. Core screen is a near-black canvas showing a warm point-light source, a small number of rectangular occluders each with a subtle physical drop-shadow, and a soft-edged cast shadow blending toward a target silhouette outline. A persistent monospaced overlap-percentage readout. A bottom sheet on solve showing that same number large, with two text actions. A vault screen as an adaptive grid of dark cards with silhouette thumbnails. Style reference: Monument Valley's spatial calm and restrained chrome, not its color saturation — keep this darker and quieter. Do not alter interaction structure, control placement logic, or add decorative elements that would compete with the shadow itself."*

## Final Output

**SwiftUI architecture recommendation:** Split the geometry/solve engine and the canonical level schema into a local Swift Package, UmbraCore, importing only Foundation — no SwiftUI, no UIKit, and Observation kept out of the pure-geometry types. UmbraCore owns the Level/Occluder/LightSource/Target structs, encode/decode, the solve-checker, and the save-time validator. The app target imports UmbraCore and wraps it in the already-locked `@Observable LevelSessionModel` and `VaultStore`. This boundary is what lets the 50-run determinism replay and the fuzz suite run fast and reproducibly via `swift test` with no simulator boot — a slow or flaky suite is one nobody runs before merging, and this is the one architectural risk most likely to quietly undermine the project's core "no silent geometry drift" guarantee if it's skipped.

**Apple framework choices:** SwiftUI for all UI; Observation for the two model types; Foundation for Codable/FileManager; Core Graphics via SwiftUI's `Path`/`Canvas` for shadow rendering, with `CastShadowRenderer` doing closed-form polygon clipping (not sampling/ray-marching) so it stays cheap; CoreHaptics or `UIImpactFeedbackGenerator`/`UINotificationFeedbackGenerator` for the solve haptic; `ShareLink`/`UIActivityViewController` for export; `UIPasteboard.general.string` read only on an explicit user-tapped "paste from clipboard" action, never auto-read on appearance; StoreKit 2 (`Product`, `Transaction`, `Transaction.currentEntitlements`) as a defined-but-unimplemented seam for future paid tiers.

**Persistence and local data plan:** No SwiftData, no CoreData — plain Codable structs, `JSONEncoder`/`JSONDecoder` with an explicit key strategy and a `schemaVersion` field, written to files in Application Support (one vault-manifest file, one file per level, a separate small file for in-progress edit-draft state so a mid-edit crash can't corrupt the vault manifest). All writes are atomic — write to a temp file, then `FileManager.replaceItemAt` — specifically because forced-quit-mid-save is a locked requirement and a half-written file is a real corrupted-save scenario to survive, not just plan around. `UserDefaults` is used only for small flags like the onboarding-seen boolean, never for anything level-shaped.

**Permissions/privacy plan:** Zero runtime permission prompts — no camera, location, contacts, microphone, notifications, or network access, fully offline by construction (no ATS exception needed since there's no networking code at all). Default App Sandbox entitlement only — no App Groups, no iCloud container, no push, no background modes — stated explicitly rather than left as whatever Xcode's template generates. A `PrivacyInfo.xcprivacy` privacy manifest is still required even with zero data collection, called out now as a concrete build task so it doesn't get missed at App Store submission; the actual privacy story is "no data collected," full stop.

**Dependency/license decision:** Zero third-party dependencies. Every capability above is first-party. This is stated as a hard decision, not a default, because a convenience package for JSON diffing, geometry, or a share-sheet wrapper would introduce license/supply-chain risk for a problem the app's own geometry code already has to get right itself.

**Testing implications:** The geometry/solve-checker suite lives in the UmbraCore package and uses a custom, explicitly-seeded PRNG (not `SystemRandomNumberGenerator`) so any fuzz failure is deterministically replayable and can be locked in as a new fixture. Serialized-snapshot comparisons use a custom equality check, not synthesized `Equatable` on raw `Double`s, matching the standing two-tier model — epsilon (~1e-8) for live/replay geometry, exact structural equality for canonical serialized snapshots. UI testing deprioritizes XCUITest simulated drags in favor of calling `LevelSessionModel`'s mutation functions directly (the same functions gestures and the discrete VoiceOver controls both route through), plus a thin smoke-level XCUITest pass (launch, open the starter level, confirm the solve sheet appears). New performance acceptance bar, closing an item that's been carried forward unresolved since detailed_discussion: live per-frame shadow recompute must run at 60fps on the oldest iOS-17-supported device at a realistic occluder count, verified with an actual Instruments Time Profiler pass during build — if it's not cheap enough, the fallback is throttling visual recompute to a fixed interval during drag (e.g. 30Hz) while still committing final geometry on gesture-end, never a third-party geometry library. StoreKit testing for the future paid tiers uses a local StoreKit Configuration file (simulated transactions in Xcode), so purchase-flow testing stays no-network-dependency consistent with the rest of the app once that work happens.

## Final Output

**Architecture overview:** A local Swift Package, `UmbraCore` (Foundation-only, no SwiftUI/UIKit/Observation), owns all pure geometry, the canonical `Level` schema, the solve engine, and the save-time validator. The app target wraps it in a thin `@Observable` layer. Every level mutation — drag, rotate, duplicate, delete, reset, nudge-arrow, rotation-stepper — is a case of one closed enum, `LevelMutation`, and the only way to change level state is `LevelSessionModel.apply(_ mutation: LevelMutation)`. That function is synchronous: it computes a new `Level` value from the current one via pure functions and then assigns it — no `Task`, no `async`, no in-place array mutation — because gesture input, the discrete VoiceOver control cluster, and the 50-run determinism replay/fuzz tests all need to be different callers of the exact same code path, not three things that could quietly drift apart. `SolveEngine` is pure and stateless; overlap between the target polygon and a possibly-non-convex union of rectangle-occluder shadows is computed via a fixed-resolution deterministic scanline quadrature (2048 horizontal strips across the canvas bounding box, exact analytic per-strip trapezoid math over crossing edges) — this is the one hard computational-geometry risk in the project, and the decision is to solve it with first-party closed-form arithmetic rather than attempt general polygon boolean ops from scratch or reach for a geometry library, which the standing zero-dependency policy already rules out. A single `LevelStore` protocol is the cloud-ready persistence seam locked in architecture_review, with `FileLevelStore` as its only conformance for this build. There is no networking anywhere in this app — no `URLSession`, no network-shaped protocol seam, not even an unused one — matching the ios_architecture_review decision that the entire privacy/permissions story is "fully offline by construction."

**Data models:** `Point2D`/`Size2D`/`Polygon` are the geometric primitives — `Polygon` is always a vector point ring, never a raster image, since the exact-equality/epsilon determinism model and the VoiceOver "approximate position/rotation" accessibility requirement both depend on comparable, describable vertices. `Occluder` (rectangle primitive today, shape stored as an enum case so future primitives don't force a schema rewrite), `LightSource` (single point, matching the locked single-point-light determinism decision), `TargetSilhouette` (a `Polygon`), and `Level` (id, `schemaVersion`, title, light source, occluders, target, timestamps) form the one canonical schema shared by Play, Edit, export, import, and the solve-checker — no parallel editor-draft shape. `LevelMutation` is the sole state-change surface. `SolveResult`/`MatchState` are always derived, never independently stored (solved status can't drift from geometry). `LevelArtifactEnvelope` wraps a `Level` with its schema version for the share string; `ImportFailure` has exactly the three locked cases (truncated/malformed, unsupported schema version, geometrically invalid).

**Persistence strategy:** No SwiftData, no CoreData — plain Codable structs, atomic file writes (temp file + `FileManager.replaceItemAt`) in Application Support: one vault-manifest file, one file per level, a separate draft file so a mid-edit crash can't corrupt the manifest. `LevelStore` is a synchronous, throwing protocol — no async networking layer anywhere in the persistence path. `importAsNewEntry` always inserts a new entry, never overwrites on ID collision. `UserDefaults` only for small flags (onboarding-seen).

**Testing strategy:** `UmbraCoreTests` runs via `swift test` with no simulator boot — the five internal geometry fixtures, a seeded (explicitly, replayably) PRNG fuzz suite, round-trip encode/decode/diff tests, and the 50-run determinism replay, which is literally a recorded `[LevelMutation]` script fed through a fresh `LevelSessionModel` fifty times with canonical-snapshot diffing — the same production code path, not a bespoke test-only mutation surface. Two-tier comparison: epsilon (~1e-8) for live/replay geometry, exact structural equality for serialized snapshots. App-level logic tests call `LevelSessionModel.apply(_:)` directly rather than simulating XCUITest drags. One smoke-level XCUITest covers launch → open starter level → solve → confirmation sheet. New performance acceptance bar: live shadow recompute verified at 60fps on the oldest iOS-17-supported device via Instruments Time Profiler, with a 30Hz-during-drag throttle as the named fallback if needed — never a third-party geometry library.

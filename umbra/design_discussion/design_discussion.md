I want to design around a single object the player never leaves: one canvas that shows a light source, occluders, and a target silhouette, and every screen in this app is either "you're looking at the vault of these canvases" or "you're inside one." So here's the screen inventory as I see it. Play is the canvas screen: light source, occluders, live shadow, a persistent but unobtrusive overlap readout, and a bottom control cluster (duplicate/delete/reset are visible but disabled/greyed until something's selected). Solve confirmation isn't a separate screen, it's a state that Play transitions into in place — the canvas holds still, a distinct visual settle (outline locks, a soft highlight sweep) plus haptic plays, and a small sheet slides up from the bottom with two choices: "Remix this level" and "Back to Vault." Edit is visually the identical Play canvas with one difference — the target silhouette is now draggable/editable too instead of fixed, and the bottom sheet options change to Save/Share instead of "you solved it." Share is a system Share Sheet triggered from Edit or from a level's detail state in Vault, never a custom share UI we build ourselves. Import is a single small screen reachable from Vault ("Import a level") with one text field for pasting a code plus a Share Sheet extension entry point, and it's also where all three of the failure states have to live front and center, not buried in a toast. Onboarding is not a separate screen stack — it's two or three short coach-mark overlays layered on top of the first real Play session, dismissible by just doing the thing being taught, because a tutorial that isn't the real game is exactly the kind of "different rules" trust break we already ruled out for editor mode.

The flow I'd defend hardest: launch always lands in Vault, never a splash screen with no function. Empty Vault (first ever launch, before any level exists — though we ship with at least one preloaded, so this state should really only be reachable if a corrupt save wipes everything) shows the starter level pre-seeded with an explicit "start here" affordance, not a blank grid with a button, because a blank grid is where new users bounce. Tapping a level card goes straight into Play with the canvas already interactive, no loading spinner unless we're actually decoding something nontrivial, and if there's a real decode delay it's a skeleton of the canvas shape, not a generic spinner, so the shape of what's coming is already legible. Solving transitions in place as I said above — I feel strongly this should not navigate to a new screen, because part of what sells "this is trustworthy" is that the exact geometry you were just looking at is still right there, unchanged, when you get the confirmation; if solving takes you somewhere else, you can't as easily verify the connection between what you did and what happened. Remixing takes you into Edit on that same object, not a fresh blank canvas — blank canvas as a first editor experience is intimidating and also contradicts "the puzzle you're editing is the one you just played," which is the whole trust story. Export happens from inside Edit's save sheet — you save, and saving offers "share this" as a next action, not a separate hunt through a menu. Import is the one place I'd put a real standalone screen, because it's an entry point that has to work even when someone has literally nothing else in their vault yet, arriving cold from a friend's message.

State model, per screen, since that's explicitly required and where I think this app lives or dies on care: Vault has empty (first-run, pre-seeded so effectively never truly empty unless save is corrupted — that corrupted-empty state needs its own honest copy, "we couldn't read your saved levels" with a recovery action, not a silent blank grid pretending nothing's wrong), loading (only on cold launch while decoding the local store, should be near-instant but needs a real skeleton state anyway since we can't promise device speed), success (populated grid), and error (corrupt local save — distinct from empty, must say what happened and offer either reset-to-starter-only or a "try again" if it looks like a transient read failure). Play has loading (decoding a specific level's saved state, same skeleton-canvas idea), success (interactive), an implicit "close" success sub-state which is the solve confirmation, and error only in the case a resumed level's geometry is itself invalid somehow — which given the save-time validator shouldn't happen for anything the app itself saved, but a defensively-coded "this level's data looks damaged, reset to last known-good" beats a crash if it ever does. Edit has the same loading/success shape as Play plus a distinct "invalid to save" inline state — the save action is disabled or explicitly explains why with the specific unsolvable/degenerate reason, not just a greyed-out button with no explanation, because a mysteriously disabled button is its own trust failure. Import has empty (nothing pasted yet, field with placeholder text and a "paste from clipboard" affordance rather than auto-reading clipboard silently), loading (parsing — should be instant but again, real state not assumed), success (a preview of the incoming level's thumbnail plus an explicit "add to vault" confirm step, not an auto-add — I want the user to see what they're about to get before it lands in their library), and three distinct, separately worded error states for truncated/malformed, version-mismatch, and geometrically-invalid, each with its own icon or visual treatment so a user glancing at a screenshot in a bug report can tell which failure it was without reading text.

Visual direction: I want this to look like an object lit from one real source, not flat vector shapes on a background — a warm, focused light point, occluders rendered with enough dimensionality (a soft drop-shadow of their own, separate from the cast puzzle-shadow) that they read as physical things you're placing, and the cast shadow itself rendered slightly soft-edged like real shadow, not a hard-edged silhouette, specifically because that softness is where we can encode the overlap percentage visually (crisper/darker as it approaches solved) without relying on color alone. Dark canvas background, because shadow-on-shadow contrast is the entire point and a light background fights that. Typography and chrome around the canvas should stay quiet — this isn't a chatty app, the canvas is the product.

Accessibility notes, and this is the part I'll push hardest on: the discrete non-gesture control path we locked isn't a hidden mode you dig for in settings, it needs a visible, always-present "select and adjust" affordance even for sighted users — tap an object to select it, and a small control cluster (nudge arrows, coarse/fine rotation stepper) appears near it. Every numeric overlap readout needs to be announced on change for VoiceOver, not just visually updated. Dynamic Type needs to not break the canvas control cluster — test at the largest accessibility sizes now, not after the fact, since a control cluster that overlaps the puzzle geometry at XXL type is a real failure, not a nitpick.

My biggest worry about this direction is the solve-confirmation-in-place idea — if the sheet that slides up from the bottom on solve doesn't feel like a big enough moment, "solving" might feel anticlimactic compared to a full-screen celebration, and this app's whole retention story depends on solving feeling genuinely satisfying. I think it's the right tradeoff for trust and clarity, but it's a real risk and I'd want it to be one of the first things tested against an actual person's reaction, not just assumed to work from a wireframe. The thing I most want preserved into the final output no matter how this debate goes: Play and Edit must remain visually and structurally the same screen with different affordances active, never two screens that happen to look similar, because that's the only way "remix" stays honest, and the three import-failure states must stay visually distinguishable from each other at a glance, not just differently worded.

## Final Output

**Screen inventory:** Vault (home — grid/list of level cards, thumbnail of target silhouette, solved/unsolved/starred indicator, tap-to-open only); Play (the canvas — light source, occluders, live shadow, persistent overlap readout, bottom control cluster); Solve confirmation (not a separate screen — an in-place state Play transitions into: canvas holds still, outline locks, soft highlight sweep, haptic, and a bottom sheet slides up); Edit (visually identical Play canvas, target silhouette now editable, bottom sheet swaps to Save/Share); Share (triggered from Edit or a level's Vault detail via the system Share Sheet only — no custom share UI); Import (one small dedicated screen off Vault — paste field, "paste from clipboard" affordance, decode preview before commit, three visually distinct failure states); Onboarding (two-to-three short coach-mark overlays layered on the first real Play session, dismissed by doing the taught action — never a separate tutorial-only mode, since a tutorial with different rules than the real game would break the same trust story that keeps Edit as a strict superset of Play).

**Primary user flow:** Launch → Vault (always, never a splash screen) → tap a level → Play (canvas interactive immediately, no spinner unless a real decode delay exists, in which case a skeleton-canvas shape rather than a generic spinner) → solve → in-place confirmation sheet showing the actual overlap number (not just "Solved!") with two choices, Remix or Back to Vault, and the sheet never auto-dismisses or times out — the player leaves it on their own terms → optional Remix, which is a mode change on the same canvas (subtle border/label treatment, not a screen navigation) rather than a push to a new screen → Save/Share from Edit's own sheet, offering "share this" as the natural next step after saving, not a separate menu hunt. Import is the one flow that has to work standalone, cold, arriving from outside the app with no other vault content yet.

**State model per screen:** Vault — empty/pre-seeded-first-run (ships with a starter level so true empty is rare), loading (skeleton grid on cold launch), success, and a distinct corrupted-save error state with honest copy and a recovery action (never a silent blank grid). Play — loading (skeleton canvas), success/interactive, the in-place solve-confirmation sub-state, and a defensive "this level's data looks damaged" fallback if a resumed level somehow fails validation. Edit — same loading/success shape as Play, plus an inline "why save is disabled" state that names the specific reason (unreachable target, degenerate geometry) instead of a mysteriously greyed-out button. Import — empty (nothing pasted, placeholder + paste-from-clipboard affordance, never silent auto-read of clipboard), loading, a preview-before-commit success state, and three separately worded, separately iconed error states for truncated/malformed, version-mismatch, and geometrically-invalid.

**Visual direction:** Dark canvas so shadow-on-shadow contrast reads clearly; occluders rendered with their own physical drop-shadow distinct from the puzzle-cast shadow, so they read as objects being placed, not flat vector shapes; the cast shadow itself soft-edged rather than hard-silhouette, so overlap percentage can be felt visually (crisper as it approaches solved) without depending on color; quiet typography/chrome around the canvas since the canvas is the product; the solve-confirmation sheet foregrounds the actual overlap number as the app's core "we show you the real number" personality trait, with no confetti/timer theatrics that would fight the restrained premium language.

**Accessibility notes:** The discrete nudge/rotation-stepper control cluster is a visible, always-on affordance for every user (tap to select an object, control cluster appears near it) — not a hidden VoiceOver-only mode — both because motor-precision users benefit directly and because building it as first-class means it's exercised by everyone, which is what actually guarantees it stays wired to the same mutation functions as drag instead of quietly forking into a parallel path. Numeric overlap readouts announce on change for VoiceOver, not just update visually. Dynamic Type at the largest accessibility sizes must not let the control cluster overlap the puzzle geometry — needs testing now, not discovered later.

**Named open question, unresolved by this discussion:** whether an on-canvas, always-visible control cluster near the selected object causes real layout collisions (canvas edges, other occluders, the target silhouette itself, especially on smaller iPhone sizes) is a legitimate implementation risk that never got a systems-level answer — nobody with the engineering seat engaged this round. Carried forward as a specific question for the architecture phase to resolve, not assumed safe.

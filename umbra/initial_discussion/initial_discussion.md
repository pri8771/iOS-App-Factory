Grounding this in what's already locked rather than re-opening it: Umbra is a local-first SwiftUI shadow-casting puzzle where a player positions a light source and occluders to cast a shadow matching a target silhouette, with a designer/share mode that's a real extension of the same primitives, not a bolted-on toy. My job this round is to think about how that actually becomes screens, flows, and states — because a lot of what's been debated so far is abstract ("real designer mode," "production-ready") and none of it yet has a concrete shape a developer could start wireframing from.

One-sentence promise, stated as I'd actually build toward: Umbra lets you solve — and design — physical light-and-shadow puzzles on your phone, then hand a friend the exact puzzle you made with one tap. The "hand a friend" clause matters more than it looks like it should, because it's the only part of this promise that isn't purely solo, and I think the whole app's information architecture should be organized around that seam rather than around "game" vs "editor" as separate destinations.

Primary user and scenario: someone who already likes short, deterministic spatial puzzles — Monument Valley/The Witness taste — opens the app in a spare five minutes, plays two or three levels from the starter vault, and occasionally, when they hit a level they found clever or maddening, either replays it or opens the editor to riff on it and send a variant to a friend. That's the core scenario I'd design every screen around: short sessions, high polish, low ceremony to get from "open app" to "manipulating geometry."

Core loop, concretely, as a sequence of screens rather than a phrase: Vault (level list, grouped by pack, showing solved/unsolved/star state) → Play (light source + occluders on a canvas, target silhouette overlay, live shadow preview, a match-confidence readout that isn't just color) → Solve confirmation (a distinct success state, not just the same screen with a checkmark — this is where the "was that satisfying" feeling either lands or doesn't) → optional branch into Edit (same canvas, same drag/rotate/duplicate gestures, but now the target silhouette is something you're drawing or adjusting instead of matching) → Share (export a code, System Share Sheet) or Save-to-vault as a personal level. I want to flag one structural decision now rather than let it get invented ad hoc later: Edit should not be a separate top-level tab from Play. If a user can start from any solved (or unsolved) level and hit "remix" to drop straight into edit mode on that level's exact geometry, the editor stops being a scary blank-canvas feature and becomes a natural continuation of something they already understand. A blank "new level" entry point should still exist, but it shouldn't be the primary way people discover editing.

The screen/state matrix is where I think the real risk lives, and it's bigger than the five-states-per-flow rule implies once you actually enumerate flows: Vault has empty (first launch, zero solved), loading (should basically never be visible since this is local, but a corrupt-save recovery path needs a real "reconstructing your vault" state), populated, and an error state for a corrupted save file that has to say something more useful than "error" — something like "we couldn't read your saved progress; here's what we could recover" if partial recovery is possible, or a clean reset path if it isn't. Play has the obvious success/failure states plus one people forget: an "impossible from here" state isn't really a thing in this puzzle type since occluders are always repositionable, but a "you've moved everything and lost track of the target" recoverable state (a reset-to-start affordance) matters a lot for a spatial puzzle where frustration compounds fast. Edit has empty canvas, populated, an invalid-save-attempt state (this is the save-time-validator idea from product research — if you try to save a level the solver can't grade, that has to be a real, specific error: "the light source is inside an occluder" not "invalid level"), and its own success state on save. Share/Import is the one I'd worry about most from a systems-thinking angle: export success, import success, and then a whole family of import failure states that aren't interchangeable — truncated code, code from a newer schema version than this build understands, code that's syntactically valid but describes a geometrically invalid level. Those three are different problems with different honest messages, and collapsing them into one generic "couldn't import this code" error would quietly break the app's only growth loop at the exact moment someone's excited about a friend's puzzle.

The second-order consequence I want on record now, before design goes further: if Edit is a strict superset of Play's controls (as already agreed), then every future change to Play's interaction model — say, adding a second occluder shape, or changing how rotation snapping feels — automatically becomes a change to Edit too, and by extension to what the share format has to be able to express. That's a good constraint, not a bad one, but it means the canvas/gesture layer and the level data model need to be built as one shared component from day one, not as parallel implementations that happen to look similar. If those diverge even slightly — say, Play snaps rotation to 15-degree increments for feel but Edit allows freeform rotation for precision — then a level built in Edit could look subtly different in Play, which breaks the same trust promise the group has already flagged around editor/play parity.

My biggest objection to my own direction: folding Edit into Play via a "remix" affordance instead of a separate tab is a smaller, cleaner IA, but it does mean first-time discovery of "you can build your own levels" is less obvious than a big tab labeled "Create" would be — that's a real onboarding tradeoff, not a free win. I'd rather solve that with a one-time contextual nudge after someone's first solve ("try adjusting this level") than with a permanent extra tab, but I want that named as a deliberate tradeoff rather than something that just happens to be how it's structured.

The thing I most want preserved verbatim in whatever this phase locks: the core loop is Vault → Play → Solve → (Remix into Edit) → Share/Save, Edit is reached primarily through an existing level rather than a blank-canvas entry point, and the three import-failure states (truncated, version-mismatch, geometrically-invalid) are treated as three distinct, honestly-worded error states rather than one generic import error.

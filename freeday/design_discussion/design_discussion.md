Freemium, $4.99/mo or $29.99/yr — positioned as cheaper than one week of the habit itself, stated explicitly in the paywall copy.

SwiftData store for habits, relapse/reset events, and craving-session logs; savings math computed on-device; all guided craving-support content bundled locally, no network dependency.

None; explicitly not using LLM chat-based coaching — the craving-support flow is a fixed, well-designed guided sequence that must work instantly and offline.

Let me put myself in the room as the person who has to defend a specific user against a confusing or joyless screen, and lay out the actual screens this app needs.

The primary flow I care about most isn't setup, it's the two loops layered on top of each other. Loop one, the ambient one: open app, home screen renders from cached SwiftData state before any spinner could even show, user sees their number, closes the app. That's it — if this ever shows a loading spinner on a warm launch, that's a bug, not a state. Loop two, the acute one: home screen, one tap on the craving button, and the guided flow needs to already be visually "arriving" within a frame or two — I'd push back hard on any transition here fancier than a fast, springy full-screen cover, because a slow modal presentation animation is friction at the exact moment friction is most dangerous. The guided flow runs its script, and when it completes, I do not want a "Congratulations!" screen with confetti and a tap-to-continue button — that's an extra screen standing between someone and being done, and it also risks feeling gamified-aggressive, which the spec explicitly said to avoid. Instead the timer's final frame settles, a haptic fires, and the view auto-dismisses back to Home half a second later with a small transient reinforcement (a soft highlight or checkmark that fades on the craving button itself, not a new screen). One tap in, zero taps out.

Reset is the flow I'd protect most jealously in this whole app, and I want to describe its actual mechanical states because "non-punitive" is a slogan until it's a specific state machine. Tap reset: no confirm sheet, no alert. Home screen instantly re-renders as if day 1 already happened — day count flips to "Day 1," lifetime-total (a smaller, quieter stat that was already on screen, not one that appears for the first time now) stays exactly where it was. Simultaneously a snackbar slides up from the bottom edge, low-contrast, reading "Day 1 · Undo," auto-dismissing in 5 seconds. If Undo is tapped, the home screen reverts and nothing was ever persisted — visually this needs to look like a value flicker, not a screen change, or it'll read as "undoing a mistake" rather than "changing your mind," which are different emotional registers. My one real worry about my own design here: if the reset affordance is too subtle, someone in a moment of self-directed frustration might not be able to find it, or might tap it repeatedly looking for a confirmation that never comes and wonder if it worked — I think the fix is that the day-count number itself needs a very brief, satisfying transition (a soft cross-fade/count change, not a jarring hard-cut) so the UI visibly acknowledges the tap without needing a dialog to do it.

State model, screen by screen, the parts that actually determine what gets built: Home has empty (first-run, no habit — this isn't really "empty," it should redirect straight into setup, there's no reason to ever show a blank home screen), loading (should basically never be visible given local SwiftData, but exists for the first frame after a fresh install before the store initializes), and populated — and populated itself branches on whether cost is set (streak+savings vs. streak-only, and I want to be explicit that streak-only is not a degraded state visually, it's a fully intentional card layout, same visual weight, just one fewer row). Habit Setup has the two-tier form state (name-only vs. name-with-cost-expanded) and a validation error state limited to the cost sanity ceiling (an inline warning, not a blocking alert, when someone fat-fingers $5000). Craving Timer has running, paused-by-interruption (phone call, backgrounding — resumes on return using elapsed monotonic time, doesn't restart), and abandoned (user force-exits or force-quits — logged honestly, and if they reopen the app there's no lingering "resume your session?" prompt hanging over Home, that would just be a guilt reminder wearing a helpful hat). Milestone Card has rendering (should be near-instant since it's local composition, but needs a skeleton state for the split second before the image is composed) and a rendered-with-no-cost variant that swaps the dollar line for something equally confident, not a blank gap — I'd suggest the card leads with the day count large regardless of cost, so the no-cost version doesn't read as "the cheap version of the real card." Paywall has default (feature comparison, pricing) and a purchase-in-flight state (native StoreKit sheet takes over, nothing custom to build there). Settings has the plain static disclaimer text, always visible, never a modal to dismiss.

On visual direction: I want the mint-to-sky-blue gradient system doing real work, not just sitting as a header decoration — I'd use it as the background field on Home and the Craving Timer specifically, because those are the two screens where the emotional tone matters most, and I'd keep Habit Setup and Settings on a plain, calmer neutral background so the gradient reads as special rather than wallpaper everywhere. The sunrise/counter motif is the natural fit for the day-count number itself — I'd treat the number as the actual sunrise, maybe a subtle radial glow behind the digit that very slowly brightens as the streak grows, rather than a literal illustrated sun icon competing for attention next to the number. Iconography rounded and soft throughout, and I want to explicitly rule out any color-coded red/orange "danger" treatment anywhere near the reset control — if reset gets colored like a destructive action anywhere in this app, we've silently reintroduced the shame framing through color alone, independent of what the copy says.

Accessibility: the reset VoiceOver announcement needs to say "Day 1" exactly like a normal daily VoiceOver read of the home screen would, per what detailed_discussion already locked — I want that same discipline applied to the craving-complete transition too, so VoiceOver announces something like "Day continues" or just re-reads the day count, not "session complete" in a tone that implies a test was passed or failed. Every tappable target — craving button, reset, share, skip — needs to hit 44pt minimum, and I'd flag the reset affordance specifically as a place designers will be tempted to shrink it down out of "let's not make it prominent," which is the right instinct visually but can't come at the cost of it being a legitimate, reachable tap target. Dynamic Type needs real testing on the Home screen specifically, since the day-count number wants to be huge for impact, and huge-number-plus-largest-accessibility-text-size is exactly the combination that breaks layouts — I'd rather the number scale down gracefully than the screen require horizontal scrolling.

The single biggest risk to my own plan: collapsing habit setup into one progressive-disclosure screen instead of three separate screens saves taps, which I think is right for the under-60-seconds goal, but if it's built sloppily it can look like one cluttered form rather than an airy, calm first impression — which would undercut the entire "clean, encouraging, light" design direction before the user even gets to day 1. Worth a second look once real spacing is in front of us rather than something I'd insist on defending to the death right now.

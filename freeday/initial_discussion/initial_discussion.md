Freemium, $4.99/mo or $29.99/yr — positioned as cheaper than one week of the habit itself, stated explicitly in the paywall copy.

SwiftData store for habits, relapse/reset events, and craving-session logs; savings math computed on-device; all guided craving-support content bundled locally, no network dependency.

Repository boundary allows CloudKit sync later; an opt-in anonymous community layer is a plausible v2 but explicitly out of scope for v1 to avoid backend/moderation overhead.

None; explicitly not using LLM chat-based coaching — the craving-support flow is a fixed, well-designed guided sequence that must work instantly and offline.

Thinking about this as the person who has to actually draw the screens: the product promise I'd lock in is "Freeday lets you name anything you're trying to quit, then turns each clean day into a visible, growing win — a streak and a running savings number — with a 90-second guided moment to lean on when a craving hits." Primary user and scenario: I'd anchor on two concrete moments rather than one persona, because the IA has to serve both without feeling like two apps stapled together. Moment one is the calm morning/evening check-in — low stakes, almost ambient, "still going," takes three seconds. Moment two is the acute craving at 9pm on the couch, or worse, standing in a gas station parking lot — high stakes, one-handed, needs to open to the exact right screen with zero navigation required. If the app's home screen makes craving-support something you have to hunt for (buried in a tab, behind a menu), it's failed the moment it was actually designed for. My take: the craving entry point needs to be a persistent, always-visible affordance from the home screen — not equal-weighted with "view streak," but the dominant visual action when you open the app, because nobody opens a craving-support app to admire their dashboard, they open it because they're about to cave.

Core loop, the way I'd sequence it: open app → see today's status (day count + optional savings, framed positively) → either do nothing (passive win, app closes) or tap the craving button → 90-second guided ride-it-out → land back on "you got through it" with the streak reinforced → periodically (weekly, milestone days) get offered a shareable card. The reset/relapse action is not really part of this loop, it's an escape hatch that has to exist without dominating the IA — I'd put it as a small, low-visual-weight "reset" affordance, not a big red button, precisely because prompt_contract already flagged that this is the make-or-break screen. My specific addition here: the reset flow should not be a single modal that vanishes after tapping — it should transition into the *same* home screen state the user sees every other day, just with day count at 1 and lifetime total still sitting right there, so there's no dead-end "you failed" screen at all. The failure mode I'd watch for is a designer instinct to add a supportive message ("that's okay, tomorrow's a new day!") on a dedicated interstitial — that instinct is right in spirit but wrong in execution, because any screen you land on specifically because you reset is emotionally coded as a consequence screen no matter how warm the copy is. Better to make reset boring — it just quietly updates the same screen you always see.

The IA fork I want flagged now, because it's a decision that's expensive to reverse later: free tier is one habit, paid tier is multiple. That means the home screen has two fundamentally different shapes — single-habit free users get one hero screen with no list/switcher chrome at all, while paid multi-habit users need some kind of habit switcher (tabs, a carousel, a list-then-detail push). If we build the free experience as "detail screen with an invisible list of one" from day one, upgrading to multi-habit is just unlocking a switcher that was already latent in the architecture. If instead we build free as a bespoke single-screen app and bolt on a list later, that's a rewrite disguised as a feature unlock. I'd insist the data layer and navigation stack support N habits from day one even though the UI only exposes 1 in the free tier — this is a systems-level decision, not a paywall-copy decision.

Scope boundaries I'd hold hard: no LLM chat coaching (the craving flow is fixed content, which actually helps here — it means it can be pre-scripted and tested for tone precisely, no risk of a model generating something clinical or preachy under pressure); no Screen Time/DeviceActivity auto-tracking; no cloud sync or account system in v1; no community/social feed, only outbound sharing via ShareLink. I'd also explicitly scope out any in-app "why did you relapse" analytics/journaling prompt for v1 — it's tempting to add since it'd be genuinely useful data, but it adds a whole optional-text-entry-during-an-emotionally-loaded-moment UX problem that isn't in the locked v1 build scope, and half-building it is worse than not having it.

Measurable success criteria, the ones I'd actually hold this app to: (1) a user can go from zero to a running habit — name entered, optional cost entered, streak showing day 1 — in under 60 seconds and three or fewer screens; (2) the craving button is reachable and the guided flow begins in one tap from the home screen, no intermediate confirmation screen, because friction here is the opposite of the app's job; (3) directly testing the invalidation criteria from prompt_contract — a user who taps reset lands back on a normal, non-terminal home screen and nothing about that screen visually reads as a dead end or scoreboard-wipe; (4) every screen that can show a savings number has a defined non-broken state when cost is nil, verified by literally setting up a habit with cost skipped and walking every surface (home, milestone card, widget) to confirm none of them show $0.00 or a blank hole where a number should be.

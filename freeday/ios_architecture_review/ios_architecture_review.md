Cross-process SwiftData access via a shared App Group container has a real history of lock contention and silent corruption; the shared snapshot needs to be a flat, trivially-serializable thing (a small Codable struct written to an App Group `UserDefaults(suiteName:)` or a tiny JSON file), not a second `ModelContainer` pointed at the same store from the widget extension. If anyone on the build tries to "just share the ModelContainer across targets" as a shortcut, that's the failure mode that produces a stale or crashing widget that's only catchable by actually installing it on a home screen — which was already flagged as a risk, but I want to close the loophole explicitly: the widget extension target should have zero SwiftData import at all. Read-only snapshot, full stop.

On the `HabitStore`/`ModelContext` ownership: this needs to be strictly main-actor. SwiftData's `ModelContext` is not safe to hand across actors or background queues, and craving-session writes plus reset writes need to happen synchronously on the main actor in the same call that updates the App Group snapshot and fires `reloadAllTimelines()` — not dispatched to a background task "for performance," because the actual risk here isn't performance, it's a race where the widget reloads before the snapshot write lands and shows the previous day's number for one refresh cycle, which is exactly the "stale widget contradicts reset" bug everyone already said is worse than no widget.

Schema migration is the thing nobody has said out loud yet and it's a real risk: even for an MVP, this app will ship a 1.1 with at least one more field (a second guided-flow variant, maybe a habit-type tag). If the SwiftData schema isn't versioned from day one (`VersionedSchema` + a `SchemaMigrationPlan`, even a trivial one-version plan), the first update that adds a field either force-migrates users off their existing streak data or crashes on launch. I want this decided now: ship v1 with an explicit versioned schema, not a bare `@Model` class with no migration story, because retrofitting migration after users have real streak data in the store is a much worse day than doing it up front.

Craving session lifecycle has a real gap: a `scenePhase` hook that writes "abandoned" on backgrounding handles the polite case (phone call, app switch), but a genuine force-quit means that hook never runs — the process is just gone. So the abandoned-state requirement needs two mechanisms, not one: an immediate write attempt on `scenePhase == .background`/`.inactive`, plus a defensive sweep at next launch that checks for any session still in `.started` state whose `startedAt` is older than its expected duration plus a grace window, and retroactively marks it abandoned. Without the launch-time sweep, a hard force-quit mid-session leaves a permanently "started" row that never resolves, which is a real data-integrity bug, not an edge case we can wave off.

On Live Activity specifically: I'd add one thing to what's already locked — the Activity needs to be explicitly `.end()`ed when the craving session completes or is abandoned, in the same code path as the SwiftData write. An Activity that's merely left to expire on its own is a lingering "session still running" tile on someone's lock screen at 2am, which is the opposite of calming.

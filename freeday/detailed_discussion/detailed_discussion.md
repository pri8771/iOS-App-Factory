**Claude (QA & Risk · Skeptic) — Round 1**

Coming at this round as the person trying to break the plan before it's built, not polish it.

The biggest gap I don't think has actually been resolved yet, despite four prior phases touching it: "cost-per-use" alone cannot produce a savings number. Product_research flagged this as needing "a coherent unit model decided at setup time" but never actually picked one, and I don't think it's safe to leave that to the design phase to improvise. A vaper who enters "$8 per pod" and a pack-a-day smoker who enters "$0.35 per cigarette" produce wildly different daily savings depending on how often they used — and the spec's own example ("47 days, $310 saved," about $6.60/day) implies some frequency assumption is baked in, not just a raw per-use price. If setup only asks for cost-per-use and silently assumes "once per day," a pack-a-day smoker's savings will be understated by roughly 20x, which is the kind of wrong number that actively undermines the app's one visible differentiator. I'd resolve this now: habit setup asks for cost-per-use AND typical uses-per-day as a linked optional pair — both skippable together, but if you set one you're prompted for the other, and if either is left blank the app shows a streak-only card with no savings figure at all rather than guessing a frequency. This keeps setup at the same screen count (one extra optional field) and avoids a confidently-wrong number, which is worse than an honestly-absent one.

Day-boundary logic needs one more layer of precision beyond "use calendar-day comparison": what does day 1 mean the moment you finish setup at 11:58pm — is that already day 1, and does it become day 2 two minutes later at midnight, or does day 1 need a full elapsed calendar day to complete? I'd resolve this as: the day you set up the habit is day 1 by convention (same-day framing, matching how "day 1 of quitting" is colloquially used), and the day count increments at the next local-midnight rollover. This same rule needs to apply identically to a reset — tapping reset makes *today* day 1 again, not day 0 counting up to day 1 tomorrow, since the whole point of prior phases' work was that reset must look like every other day-1 experience.

On the craving timer specifically: it must be driven by a monotonic elapsed-time measurement (uptime-based), not wall-clock Date math, or else a DST transition or a user manually adjusting the clock mid-session could make a 90-second timer jump to instantly "complete" or hang indefinitely — a bad failure in the single moment this app can't afford to glitch in. I'd also want a defined behavior for a craving session that gets abandoned mid-flow: phone call interrupts, app gets backgrounded and evicted by the OS under memory pressure, user force-quits. The craving-session log needs a real state machine (started/completed/abandoned-under-threshold) rather than only ever writing a "completed" row, both so the data model doesn't lie and so relaunching mid-session resumes sensibly (or at minimum doesn't crash or show a stuck timer at 0:00 forever).

The "brief undo window" on reset has been mentioned three separate times across phases without ever getting a number or a persistence model — I'd pin it down here: a 5-second snackbar-style "Day 1 · Undo" affordance, and the reset isn't actually committed to SwiftData (no RelapseEvent row written, no lifetime-total mutation) until the window closes without being tapped. If Undo is tapped, the UI reverts instantly and nothing was ever persisted — this avoids a write-then-delete inconsistency and means a fat-fingered tap genuinely costs nothing. I'd also flag that VoiceOver's spoken announcement on reset needs its own copy check — an unglossed accessibility announcement like "streak reset" can reintroduce exactly the shame framing the visual design worked to avoid, so it should announce "Day 1" the same way a fresh launch would, not narrate the reset event itself.

A real edge case nobody's named yet: what happens on a Freeday+ downgrade when the user has three active habits? Prior phases correctly decided data isn't deleted and only *adding new* habits/premium tools gets gated — but the free UI is architecturally single-habit, so which one habit does downgraded free UI show? I'd resolve this as: whichever habit was most recently interacted with (checked in, or had a craving session logged) becomes the visible one, and the others remain fully intact in storage, re-exposed instantly if they resubscribe — no picker UI needed for v1, since that's an edge case affecting a small population of lapsed subscribers, not a path worth new screens for.

On the widget and Live Activity, both need to react to a reset immediately, not on their next natural refresh cycle — a widget still showing "Day 47" for hours after an in-app reset is arguably a worse experience than no widget at all, since it directly contradicts the "reset looks like every other day" design goal by showing a *stale, wrong* boring screen instead of the *current, correct* one. `WidgetCenter.shared.reloadAllTimelines()` needs to fire synchronously with the reset write, and the Live Activity (if one happens to be running during a reset, unlikely but possible) needs the same explicit update call. I'd also want it confirmed that `NSSupportsLiveActivities` is set and that the code checks `ActivityAuthorizationInfo().areActivitiesEnabled` before ever attempting to start one — not just at first launch, since a user can disable Live Activities for the app at any time in Settings, and a silent failed `Activity.request` shouldn't visibly break anything in the standalone in-app timer.

Two smaller things worth locking down as explicit exclusions rather than silent gaps: currency should be locale-derived (`NumberFormatter` with `.currency` style and the device locale), not a hardcoded "$" — this is a five-minute fix now and a real bug for any non-US App Store listing later. And I'd put a sanity ceiling on cost-per-use input (say, reject or double-confirm anything above some absurd threshold like $500) purely to avoid a fat-fingered decimal point producing a milestone card claiming "$4.7 million saved," which would be actively embarrassing to have someone screenshot and share.

On privacy: this data is more sensitive than a typical tracker — "quitting alcohol" or "cutting back on drinking" sitting in an unencrypted-by-default local store is something a user might not want visible if someone else picks up their phone. Given local-only + zero network + zero third-party SDKs is already locked, the actual residual risk is physical device access, not data exfiltration. I don't think an app-lock/Face ID gate belongs in this MVP slice — it's new scope this late in a time-constrained build — but I'd name it explicitly as a known limitation for this version rather than let it go unstated, since it's a legitimate and foreseeable user concern. Separately, I'd want a one-line disclaimer living somewhere low-friction (an About/Settings screen, not a modal anyone has to dismiss) stating this is a personal tracking tool and not medical or addiction-treatment advice — this operationalizes the key-risk item from prompt_contract as an actual concrete artifact rather than just a copywriting guideline that could get missed screen-by-screen.

Last one: the guided flow includes a "text someone" variant per the product brief — that needs to be scoped now as literally opening the system Messages composer to a contact the user picks (via `MessageUI` or a contact picker), with zero built-in crisis-line numbers, hotline integration, or "are you in danger" branching logic. The moment this app suggests it has any crisis-intervention capability, the health-claims risk gets much sharper, and that's explicitly not what a fixed, offline, non-clinical script should be doing.

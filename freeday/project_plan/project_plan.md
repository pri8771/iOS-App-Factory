Coming at this from the backend/systems side, and I want to open by pressure-testing a few things nobody has actually nailed down yet before I lay out sequencing, because a project plan built on top of unexamined assumptions is worse than no plan.

If it's a mutable field that gets incremented somewhere, that "somewhere" is exactly the kind of place a bug hides for months — miss one code path that should bump it (say, a habit edit that doesn't go through `resetStreak`) and the lifetime stat quietly desyncs from reality, which is catastrophic for this app specifically since the lifetime-total stat is the entire mechanism keeping reset non-punitive. I'd rather this be derived, not stored-and-maintained: `lifetimeCleanDays = habit.lifetimeCleanDaysAtLastReset + currentStreakDays`, recomputed by `StreakCalculator` on every read, with the stored field only ever written once, at the moment a reset actually commits (`lifetimeCleanDaysAtLastReset += streakLengthAtReset`). That turns a "must remember to update this everywhere" bug class into a "compute it correctly in one place" property. This needs to be nailed down in the plan's first milestone, not discovered mid-build.

Second, the 5-second undo window interacts with backgrounding in a way nobody's resolved. If a user taps reset and immediately backgrounds the app (call comes in, they swipe to another app) before the 5 seconds elapse, does the pending reset commit, or does it silently vanish because the in-memory timer never fires? Silently vanishing is the worse failure — the user believes they reset, comes back tomorrow, and sees yesterday's streak count still climbing, which reads as the app not working.

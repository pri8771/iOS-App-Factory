The biggest gap I see coming out of design_discussion is navigation plumbing. My answer: TabView with exactly two tabs, Home and Library, each owning its own NavigationStack and its own NavigationPath. A single lightweight router object owns a "pending deep link" (a place ID) that gets set when a notification is tapped or a widget/share-extension deep link fires, and gets consumed on scene-became-active by force-switching to the Library tab and pushing that place's detail onto Library's stack — never presenting Detail as an orphaned modal floating outside the tab structure. That way the back button always does something coherent ("back to Library"), VoiceOver's screen hierarchy stays sane, and there's exactly one code path for "open place X," whether the user tapped a row or a notification. Settings is not a tab — it's a sheet presented from the icon on Home, and it owns its own internal NavigationStack for Diagnostics and Subscription so it can push/pop without touching the tab-level stacks underneath it. Paywall is always a `.sheet`, never pushed, since it needs to be dismissible from anywhere (cap-hit moment in Library, Settings entry point) without polluting either tab's navigation history.

On state management, I want to push back gently on treating "snoozed," "paused," and the day-scale soft-pause decay as stored enum values that some background job flips at the right moment — that's a trap. If "snoozed-until" or "no response for ~24h → soft-pause" is a value some timer has to actively update, then the UI can show stale state any time the app wasn't running when the timer should have fired, which is exactly the unfalsifiable-bug problem four phases of discussion were trying to kill. The fix is to store raw facts — capturedAt, lastNotifiedAt, snoozedUntil, lastUserActionAt — and compute the displayed state as a pure function of those fields plus the current time, evaluated fresh every time the UI reads it, not cached and flipped by a scheduled task. A BGAppRefreshTask can opportunistically re-evaluate and re-register regions when it gets CPU time, and CLLocationManager's own region-entry callbacks can also trigger a re-evaluation pass, but neither should be load-bearing for correctness — the moment the user opens the app, or an event fires, the state is recomputed truthfully regardless of whether any background job ran in the interim. ### Navigation architecture
`TabView` with exactly two tabs (Home, Library), each owning its own `NavigationStack`/`NavigationPath`. Settings and Paywall are always sheets with their own internal navigation, never pushed onto a tab stack. ### State model (core rule)
Every place's displayed state (active/queued/paused+subreason/snoozed-until/dismissed/visited) is a **pure function of persisted facts** (capturedAt, lastNotifiedAt, snoozedUntil, lastUserActionAt) evaluated against the current time at read time — never a cached enum a background timer is responsible for flipping. Background tasks and CoreLocation callbacks may opportunistically trigger re-evaluation, but correctness never depends on one having run. On every launch, reconcile the persisted "active" set against `CLLocationManager.monitoredRegions` (the OS's actual truth) so the diagnostic view stays honest rather than aspirational.

### Navigation architecture
`TabView` with exactly two tabs (Home, Library), each owning its own `NavigationStack`/`NavigationPath`. Settings and Paywall are always sheets with their own internal navigation, never pushed onto a tab stack. ### State model (core rule)
Every place's displayed state (active/queued/paused+subreason/snoozed-until/dismissed/visited) is a **pure function of persisted facts** (capturedAt, lastNotifiedAt, snoozedUntil, lastUserActionAt) evaluated against the current time at read time — never a cached enum a background timer is responsible for flipping. Background tasks and CoreLocation callbacks may opportunistically trigger re-evaluation, but correctness never depends on one having run. On every launch, reconcile the persisted "active" set against `CLLocationManager.monitoredRegions` (the OS's actual truth) so the diagnostic view stays honest rather than aspirational.

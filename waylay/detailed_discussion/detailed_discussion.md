**Codex (Product Strategist · Visionary) — Round 1**

The biggest thing to resolve in this phase is that Waylay is not actually a “saved places” app. It is a proximity-recall engine with a capture shell around it. So I would lock one hard product rule now: every saved place must always be in an explicit, user-legible state, and every state must imply concrete notification behavior. If a user cannot answer “will this place alert me, and if not, why not?” in one glance, the app will feel untrustworthy no matter how polished it is.

That means the state model needs to be resolved more aggressively than earlier phases did. I would make it: active, queued, snoozed-until, visited, dismissed, and paused. “Paused” matters because permission denial, force-quit, or globally disabled notifications are not the same as queued; they are system-blocked states and should be shown as such. “Dismissed” should be terminal and hidden from default active browsing, but still recoverable from a history/archive view so the user does not fear permanent loss. “Visited” should suppress future notifications by default, but not erase the place; the user may want to re-arm it later. “Queued” should never mean “maybe broken”; it should always carry a reason like “not one of your 8 active places right now” or “recently far away.” That one detail is the difference between a premium memory product and something that feels random.

The rotation engine also needs product-level rules now, not just architecture later. It cannot simply pick the geographically nearest places or most recent saves. That would be technically tidy and behaviorally wrong. **Claude (QA & Risk · Skeptic) — Round 1**

Before this phase can call anything "resolved," there's a permissions fact that nobody in four prior phases has said out loud: `startMonitoring(for:)` requires **Always** authorization, not When-In-Use. Every prior phase has talked about "graceful permission handling" in the abstract, but the concrete reality is that iOS makes the Always upgrade a two-step, deliberately-friction-heavy flow — you get When-In-Use first, and the system either auto-prompts for the upgrade later or you have to send the user to Settings, and Apple's own reviewers scrutinize apps that ask for Always aggressively on first launch. This isn't a minor implementation detail, it's the single permission this entire product's core promise depends on, and if a meaningful fraction of users say yes to "while using" and no (or never get asked, or ignore) the Always upgrade, the resurfacing loop is just dead for them with the app still technically "working." I want this phase to resolve, explicitly: we request When-In-Use at first capture (so capture itself never needs Always), then request the Always upgrade at a moment that's earned — probably right after the first successful save, framed as "let Waylay watch for this place even when you're not in the app," not on cold launch. And we need a real degraded state in the UI for "When-In-Use only" users — not an error, but an honest one-line explanation that resurfacing won't work until they upgrade, with a Settings deep link. Second thing I don't think is settled yet: the share-sheet capture path (already locked as v1 scope) writes into a data store, but who actually runs the rotation engine on that new place? - *Notification/cooldown model (locked, replacing the earlier "4+ hour" rule):* one notification per place per unresolved episode. After it fires, the place goes silent by default — no repeat nudging on subsequent region re-entries — until the user takes an explicit action (snooze, dismiss, visited) or a long, day-scale reset condition is met. If a fired notification gets no user response for roughly a day, the place auto-transitions to a soft-paused "we noticed this one didn't seem useful, tap to bring it back" state rather than mechanically re-arming on a timer forever. This directly targets the daily-commute spam case named as a real failure mode.

- *Rotation/priority recipe:* deterministic and visible, not a black box — inputs include proximity likelihood, recency of capture, recent resurfacing history, snooze status, manual promotion, and demonstrated pattern near the area. A place captured within the last 24–48 hours provisionally contests for an active slot rather than queuing behind older saves indefinitely, so "just captured" and "actually being watched" don't quietly become different claims. Free tier caps monitored places (~8), never total saved places — nothing the user saves is ever at risk of being lost or downgraded in status, only in monitoring priority.

- *Duplicate/near-duplicate handling:* silent auto-merge by default when a new capture lands within a small radius of an existing unresolved/active place (with loose name-similarity if a name exists), surfaced after the fact as a soft, dismissible note ("we think this is the same as [place] — tap to separate them if we're wrong"). No merge decision is forced into the capture moment itself — consistent with the standing "no forced decisions at capture" rule from earlier phases.

- *Location accuracy:* precise-location detection is a named requirement. If the user granted reduced/approximate location, the app says so plainly ("proximity reminders may be fuzzy") and offers the precise-location upgrade path, rather than silently degrading.

**Edge cases (explicitly decided, not left to "the logic will handle it")**

- Saving a place while already standing inside its geofence → no immediate notification; that would be nonsense.
- Driving quickly through a monitored region → resurfacing should account for dwell/speed context rather than firing indiscriminately, to avoid becoming car spam.

- This is a sensitive behavioral trail (where the user noticed things, when they passed them) even though it's fully local — treated with that seriousness, not waved off as "no backend, so it's fine."

**Risk register**

## Final Output

**Screen inventory**
1. **Onboarding** (one screen only) — states the promise ("tap once when something catches your eye, we'll remind you when you're near it again"), one CTA, drops straight into Home. No permission ask, no cap explanation, no premium pitch here.
2. **Home** (tab 1) — capture-first surface.
3. **Library** (tab 2) — state-grouped management surface.
4. **Place Detail** — pushed from Library or opened from a notification tap; one shell with six state-driven variants.
5. **Settings** (reached via small icon on Home, not a tab) — contains Diagnostics/Status and Subscription management.
6. **Paywall** — sheet, triggered contextually (cap-hit moment in Library, or from Settings).
7. **Notification** — its own first-class designed surface with attached actions.
8. **Widget** (miniature Home: status line + recent strip, tap opens Home) and **Share Extension** (minimal capture receipt: save-then-toast) — reuse Home's exact interaction model, no separate design language.

**Primary user flow**
Onboarding screen → Home. First capture: tap the save button — the tap *is* the save, no confirm gate — a receipt toast slides up (place name, undo, rename), auto-dismisses. Immediately after that first successful save, the Always-location upgrade sheet appears, earned-context framing, with a real declined path that surfaces honestly everywhere downstream (never a silent dead loop). Later: proximity triggers a notification with snooze/dismiss/mark-visited actions attached; tapping the body opens that place's Detail screen directly (not Library root). Trust-check flow: Home's one-line status → Library row (state pill + reason, always visible, never behind a filter) → Place Detail if the user wants the full explanation. Recovery flow: a degraded Home state or a paused Library row leads to a one-tap fix (Settings deep link). Management flow: Library → tap a place → Detail → snooze/dismiss/mark-visited/manual-promote.

**State model per screen**
- **Home:** first-run-pre-save (empty, warm, CTA only) · healthy/watchful ("watching 6 places") · degraded-but-save-still-works ("3 saved, resurfacing paused — tap to fix") · location-temporarily-unavailable. Same dominant capture button in all four; only the one-line status + optional recovery action + recent-captures strip change. No proximity-sorted content ever.
- **Library:** grouped sections always visible (never filter-gated) — Active & Queued at top, Snoozed (with countdown), Paused (mandatory sub-reason inline in the row, never a bare badge), then Visited/Dismissed collapsed by default but present and expandable. Empty state for zero saves; populated state exercises all groups.
- **Place Detail:** six variants — active, queued, paused (one of the four locked sub-reasons: permission-revoked / notifications-disabled / force-quit-awaiting-reopen / device-rebooted-reregistering), snoozed-with-countdown, visited, dismissed — each showing only the actions valid for that state, plus manual-promote on active/queued.
- **Diagnostics:** three-tier summary card up top (all systems go / some capability reduced / monitoring unavailable) for calm at-a-glance reading, with full four-way sub-reason precision per place underneath — compression at the summary, precision at the place level, never one undifferentiated bucket.
- **Paywall:** normal pitch / purchase-in-progress / restored / failed-restore, with Restore Purchases always visible, never a footer afterthought.
- **Notification:** delivered state with snooze/dismiss/mark-visited actions; tapping body routes to Detail, not Library root.

**Visual direction:** warm, tactile, analog — cream/paper/terracotta/amber/moss tones, rounded card shapes, "found object" feel over map-pin/dashboard iconography. State communicated via icon+text pairing everywhere, never color alone. Notification and reason copy stays in specific, warm plain language ("watching this because you saved it yesterday and you're often nearby"), never system-alert or engineering tone — except inside Diagnostics, which is allowed to read more technical since its audience there is actively troubleshooting.

**Accessibility notes:** every state pill/badge pairs an icon with text, never color-only. VoiceOver announces place name → state → reason → available action, in that order. Dynamic Type must not break compact state chips or grouped library rows — they need to wrap, not truncate silently. Reduced Motion: delight comes from copy/spacing/material, never from an animation the app depends on to communicate state. Notification action buttons and permission-recovery links use plain verbs and human language, not authorization-class jargon. Large tap targets on all notification actions.

## Final Output

### Screen-by-screen design specification

**Onboarding** — single full-screen view, shown once (gated by a UserDefaults flag), never returned to. One line of promise copy ("tap once when something catches your eye, we'll remind you when you're near it again"), one illustration, one CTA that flips the flag and drops straight into Home. No permission ask, no cap explanation, no premium pitch, no back button.

**Home (tab 1)** — fixed vertical hierarchy: small settings icon in the nav bar, a one-line status sentence, the dominant capture button (centered, thumb-reachable, occupies the most visual weight on screen), then a compact recent-captures strip (last 2–3 saves, most-recent-first, horizontal small cards, tap opens rename/detail). Nothing else. No proximity-sorted content ever — that idea was explicitly killed in design_discussion and stays dead.
- *States:* first-run-pre-save (warm, spare, CTA only) · healthy/watchful ("watching 6 places") · degraded-but-save-still-works ("3 saved, resurfacing paused — tap to fix," one inline recovery link) · location-temporarily-unavailable (capture still works, uses cached-but-fresh-enough location rather than blocking).
- *Capture button internal state machine:* idle → pressed (haptic + scale-down) → resolving (in-button pulse/spinner, or a static "locating…" label under Reduced Motion) → committed (brief checkmark glyph, reverts to idle). The tap is the save — no confirm gate, ever.
- *Receipt toast:* slides up on commit with place name, undo, rename; auto-dismisses on a timer **only when VoiceOver/Switch Control is not active** — when accessibility focus is engaged, it holds until explicit dismissal or resets its timer on focus, since undo/rename are live actions in the exact moment those users need them most.

**Library (tab 2)** — grouped sections, always visible, never filter-gated: Active & Queued at top, Snoozed (with live countdown), Paused (mandatory sub-reason inline in the row subtitle — never a bare badge), then Visited/Dismissed as collapsed-by-default `DisclosureGroup`s that are always present even at zero count. Every row shares one anatomy: leading state icon, title, one line of reason/countdown, trailing chevron.
- *States:* empty (zero saves) and populated (all sections exercised).

**Place Detail** — one shell parameterized by state, not six divergent views. Title, location snippet, capture source/date, a current-state card, the full reason sentence, only the actions valid for that state, an activity section (last notification, snooze expiry), destructive/archive actions lower down.
- *Six state variants:* active, queued, paused (one of four locked sub-reasons: permission-revoked / notifications-disabled / force-quit-awaiting-reopen / device-rebooted-reregistering), snoozed-with-countdown, visited, dismissed. Manual-promote action available on active/queued.

**Settings** (sheet from Home's icon, not a tab) — own internal NavigationStack, holding Diagnostics/Status and Subscription.
- *Diagnostics:* three-tier summary card up top (all systems go / some capability reduced / monitoring unavailable), then capability rows (location permission, notification permission, precise location, app-running status, monitored slots used, last rotation pass), then a per-place list with full sub-reason, last-notified timestamp, and whether it currently occupies one of the 20 system region slots. Allowed to read more technical than the rest of the app.

**Paywall** — always a `.sheet`, triggered contextually (cap-hit moment in Library, or from Settings). States: pitch, purchasing, restored, failed-restore/offline. Restore Purchases rendered as a plain, always-visible text button above the fold, never a footer afterthought.

**Notification** — first-class designed surface. Warm, specific copy ("oh hey, [place] is right around the corner," never system-alert tone). Snooze/dismiss/mark-visited actions attached directly. Tapping the body routes into Place Detail via the router (see below), never Home or Library root.

**Widget & Share Extension** — miniature Home: status line + recent-captures strip in small form, tap opens Home/relevant detail. Share extension is save-then-toast only, no mini-form. Widget timeline reloads are coalesced/debounced (batched, plus one reload on foreground/background transition) rather than fired per mutation, since WidgetKit's daily reload budget is a hard ceiling and a silently-stale "3 places nearby" count is the same trust bug as a bad notification, just relocated to the home screen.

### Navigation architecture
`TabView` with exactly two tabs (Home, Library), each owning its own `NavigationStack`/`NavigationPath`. Settings and Paywall are always sheets with their own internal navigation, never pushed onto a tab stack. A single router object holds a pending deep-link place ID, set when a notification/widget/share-extension link fires; on scene-active it switches to the Library tab and **appends** the target place's detail onto Library's existing NavigationPath (`path.append`, not a stack reset) — so a notification always wins the moment ("land on the right place now"), but never destroys whatever the user was mid-editing; backing out resumes exactly where they left off.

### State model (core rule)
Every place's displayed state (active/queued/paused+subreason/snoozed-until/dismissed/visited) is a **pure function of persisted facts** (capturedAt, lastNotifiedAt, snoozedUntil, lastUserActionAt) evaluated against the current time at read time — never a cached enum a background timer is responsible for flipping. Background tasks and CoreLocation callbacks may opportunistically trigger re-evaluation, but correctness never depends on one having run. On every launch, reconcile the persisted "active" set against `CLLocationManager.monitoredRegions` (the OS's actual truth) so the diagnostic view stays honest rather than aspirational.

### Design tokens
- **Color:** `bg/base` pale cream paper · `bg/elevated` slightly warmer card tone · `ink/primary` soft charcoal · `ink/secondary` muted brown-gray · `accent/primary` terracotta (capture button, active-state badge — nothing else competes with it) · `accent/secondary` amber (highlights, receipt moments) · `state/watchful` moss/olive, reserved exclusively for resolved-positive (visited) · `state/quiet` muted rust-gray stone (queued/dismissed/structural chrome) · `state/destructive` restrained desaturated brick red, used sparingly (dismiss/permanent actions only).
- **Spacing:** 8pt scale — 8/12/16/24/32.
- **Radius:** 16pt for cards/rows, a larger dedicated radius token for the capture button so it reads as an object, not a stretched pill.
- **Type:** SF Pro / Dynamic Type everywhere by default. System serif confined to exactly two moments — the onboarding promise line and Home's first-run empty-state headline — both pure emotional copy with nothing functional attached. Everywhere else (status line, badges, diagnostics, library rows) stays SF so it can reflow and never silently truncates.

### Component inventory
`CaptureButton` (four-phase internal state) · `ReceiptToast` (accessibility-aware dismissal) · `StatusLine` · `RecentCaptureCard` · `StateBadge` (icon+text required, never color-only) · `StateReasonBlock` (single shared "state + one-line why" component reused across Library rows, Detail, and Diagnostics — not rephrased three times) · `LibraryRow` · `LibrarySectionHeader` · `PlaceDetailActionBar` (dynamic action set per state) · `RecoveryLink` (shared "explain + deep-link to Settings" affordance) · `DiagnosticsSummaryCard` · `DiagnosticsRow` · `PaywallPlanCard` · `RestoreLink` · `PermissionUpgradeSheet`.

### Motion
Purposeful, never decorative. Receipt toast rises and settles, fades/slides away (accessibility-gated dismissal as above). Capture button compresses on tap, success haptic on commit. Library section expands/state transitions cross-fade. Recovery banners slide/fade in. Under Reduced Motion: every fact conveyed by motion must still be readable as static text — pulsing becomes a "locating…" label, nothing depends on animation to communicate state.

### Accessibility requirements
Every state indicator pairs icon + text, never color alone. VoiceOver announces place name → state → reason → available action, in that order, with explicit accessibility labels (not inferred from SF Symbol names). Capture button's accessibility label changes with its internal phase ("Save this place" / "Saving…" / "Saved"). Dynamic Type must wrap, never truncate, in library rows, state pills, and reason text — recent-captures strip collapses to vertical past a Dynamic Type threshold rather than clipping. Receipt toast never auto-dismisses out from under VoiceOver/Switch Control users. Permission-recovery links use plain verbs ("Open Settings to allow reminders"), never authorization-class jargon. ## Final Output

**SwiftUI architecture:** Thin SwiftUI views over a small set of long-lived domain services injected via environment — `PlaceRepository`, `MonitoringEngine`, `LocationService`, `NotificationService`, `EntitlementService`, `DiagnosticsService`. Strict separation of three concerns: persisted facts (what actually happened), derived state (active/queued/paused+subreason/snoozed/visited/dismissed — always computed fresh from stored facts + current time, never a cached enum flipped by a background timer), and side-effect orchestration (CoreLocation registration, notification scheduling, widget refresh, StoreKit entitlement refresh), each owned by exactly one service. `MonitoringEngine` is a pure function: given places, `AppCapabilitySnapshot`, `effectiveMonitoringCap` (from `EntitlementService`, not StoreKit calls inline), recent actions, and "now" → ranked active/queued/paused decisions with reason codes. A separate reconciler diffs that desired set against `CLLocationManager.monitoredRegions` (OS ground truth) and applies the minimal change set, with a hard-coded defensive clamp that never calls `startMonitoring` past 20 regions regardless of what ranking computes. `AppCapabilitySnapshot` (location authorization level, notification authorization, precision authorization, force-quit/reboot state, last sync time) is computed by one service and read everywhere state is shown (Home, Library, Diagnostics) — never independently inferred per-screen. `locationManagerDidChangeAuthorization` and any mid-session permission revocation immediately triggers a capability recompute and engine re-evaluation, not something deferred to next launch.

**Apple framework choices (all first-party):** SwiftUI, Core Data (see persistence), CoreLocation (region monitoring/authorization), UserNotifications (local notification scheduling + custom category with snooze/dismiss/mark-visited actions), BackgroundTasks/BGAppRefreshTask (opportunistic reconciliation only, never correctness-critical), WidgetKit (Timeline provider reading the shared store, coalesced/debounced reloads per design_handoff), a real Share Extension target (distinct from App Intents), AppIntents (Siri shortcuts, separate surface), StoreKit 2 (subscriptions/restore), MapKit (static location snippet + manual pin only), PhotosUI/AVFoundation only if photo/voice notes survive as Pro features, UINotificationFeedbackGenerator for capture haptics. No Core ML, Vision, ARKit, or LLM integration — OCR is reaffirmed cut not just as low-priority but as structurally incompatible with the tap-is-the-save rule, since any OCR failure mode naturally invites a confirm/correct step back into capture.

**Persistence and local data plan:** Core Data (`NSPersistentContainer`, not the CloudKit-backed variant) in an App Group container from the first commit. Schema: `Place` (stable UUID, title, coordinates, radius, capture source, capturedAt, Pro media metadata, lifecycle facts — dismissedAt, visitedAt, snoozedUntil, manuallyPromotedUntil, lastNotificationAt), `UserActionEvent` and `NotificationEpisode` as append-only history, `MonitoringDecision` as a persisted audit trail (timestamp, ranking inputs, disposition, human-readable reason string) so "why is this watched right now" is answerable from real history, not a guess, and `AppCapabilitySnapshot` as the single derived-state cache read by all UI. Stable UUIDs and lightweight last-modified fields from day one so a future CloudKit sync layer doesn't require a schema rewrite, without shipping any cloud dependency in v1. A stub extension target reading the same store is a required integration test, not a manual spot-check.

**Permissions/privacy plan:** Staged capability ladder — When-In-Use requested at first capture, Always + notification permission bundled into one earned-context ask immediately after first successful save (not split across sessions). Always authorization plus `UIBackgroundModes: location` is the load-bearing prerequisite for any background resurfacing at all; a When-In-Use-only user gets zero background loop, which is why permission-revoked is one of the four locked paused sub-reasons rather than a soft degradation. Force-quit specifically blocks background region relaunch until manual reopen; plain reboot does not (once Always is intact) — these are real, distinguishable iOS behaviors and must stay visually distinct in diagnostics. Data minimization is architectural: only user-saved coordinates persist, no passive location trail, notification/diagnostic history stays local and user-deletable, zero analytics/telemetry in v1. Always-usage-description string must explicitly name the background reminder benefit in plain language (a vague string is a documented App Review rejection reason), and a `PrivacyInfo.xcprivacy` manifest is a required submission artifact to plan into the build now, not a release-week scramble. No accounts, no push, no ATT prompt — a genuine simplification this architecture gets for free by staying local-first.

**Dependency/license decision:** Zero third-party dependencies. Every required capability is covered by first-party Apple frameworks; the bar for introducing a package later is "materially reduces risk without licensing/privacy-manifest/maintenance drag" — nothing in current scope clears that bar (RevenueCat named and explicitly rejected in favor of StoreKit 2, which is sufficient for a single-platform local-first app).

**StoreKit/subscription architecture:** StoreKit 2 only, long-lived `Transaction.updates` observer from app launch, entitlement state always live-computed via `Transaction.currentEntitlements` (never a launch-time cached bool) so a lapsed/refunded subscription correctly demotes the monitored cap on next evaluation. `EntitlementService` produces `effectiveMonitoringCap` (entitlement tier × real 20-region OS ceiling) as a plain input to `MonitoringEngine`, keeping rotation logic fully testable at both free (~8) and Pro cap without mocking StoreKit inside domain code. Free tier never blocks saving places, only monitoring priority. Restore Purchases wired end-to-end via `AppStore.sync()` + entitlement rescan, tested with the `.storekit` local configuration across active/expired/revoked/grace-period states.

**Testing implications:** Pure unit tests (no CoreLocation/StoreKit dependency) for rotation/ranking, cooldown/state-derivation, duplicate-merge heuristics, entitlement gating at multiple caps, and diagnostics reason-string generation. A `LocationProviding` protocol wraps `CLLocationManager` with a fake implementation that can simulate region entry/exit and authorization-change callbacks, making the reconciler and force-quit/permission-revoked/reboot distinctions assertable in CI rather than hoped-for on device. Integration tests cover Core Data + App Group cross-target reads (via a stub extension target) and StoreKit entitlement transitions. Real-device (or Xcode GPX-route-simulated) walking tests are a named, mandatory release gate for the actual geofencing product claim — explicitly not something CI or unit-test coverage can be considered a substitute for.

**Acceptance checks satisfied:** Architecture has no vague gaps — storage engine, framework list, permission sequencing, and entitlement flow are all concretely decided, not deferred. No ML/LLM/AR is used anywhere in v1 (OCR explicitly and permanently cut, with reasoning). No dependency of any kind is introduced, so the permissive-license policy is trivially satisfied.

## Final Output

**Architecture overview.** Thin SwiftUI views (using `@Observable`, never `ObservableObject`, anywhere in the codebase) sit over a small set of long-lived domain services — `PlaceRepository`, `MonitoringEngine`, `LocationService`, `NotificationService`, `EntitlementService`, `DiagnosticsService` — injected via `.environment(_:)`. Three concerns stay strictly separated: **persisted facts** (what actually happened — captures, actions, notification episodes, monitoring decisions), **derived state** (active/queued/paused+subreason/snoozed/visited/dismissed, always computed fresh from facts + "now" by `PlaceStateResolver`, never a cached enum a timer flips), and **side-effect orchestration** (CoreLocation registration, notification scheduling, widget refresh, StoreKit entitlement refresh), each owned by exactly one service. `MonitoringEngine` is pure — given places, capability snapshot, effective cap, recent actions, and notification episodes, it returns ranked `MonitoringDecisionRecord`s — and a separate `MonitoringReconciler` diffs that desired set against `CLLocationManager.monitoredRegions` (OS ground truth), applying a hard-coded clamp that never calls `startMonitoring` past 20 regions regardless of what ranking computes. Domain code (`data_domain`) never imports CoreLocation, UserNotifications, or StoreKit directly — `LocationAuthLevel`/`NotificationAuthLevel` are domain-owned enums mapped from Apple's raw types at the `LocationService`/`NotificationService` boundary, and `EntitlementSnapshot` is StoreKit's `Transaction.currentEntitlements` normalized before it ever reaches domain logic.

**File/module layout.** A local Swift Package, `WaylayCore`, is linked by all three targets (`WaylayApp`, `WaylayShare`, `WaylayWidget`) and owns the Core Data stack, all domain value types, `MonitoringEngine`, `PlaceStateResolver`, `EntitlementService`, `DiagnosticsService`, and the `LocationProviding` abstraction. `WaylayCore` never imports SwiftUI. `NSManagedObject` subclasses never leave the persistence files inside `WaylayCore` — every read out of the store is mapped to a plain `Sendable` struct (`PlaceRecord`, `UserActionEvent`, `NotificationEpisode`, `MonitoringDecisionRecord`) before touching the engine or any UI, which is what keeps `MonitoringEngine`/`PlaceStateResolver` genuinely pure and fixture-testable. `WaylayApp` holds SwiftUI views, `AppRouter`, and the four owning-lane view models; `WaylayShare` is a thin facade calling the same `capturePlace` use case as Home; `WaylayWidget` reads cached derived state, it never recomputes `MonitoringEngine` itself (widgets are time/memory constrained — they read, they don't decide).

**Data models.** `Place` (stable `PlaceID`, title, coordinates, radius, capture source, capturedAt/updatedAt, optional Pro media fields, lifecycle facts — dismissedAt/visitedAt/snoozedUntil/manuallyPromotedUntil/lastNotificationAt), `UserActionEvent` and `NotificationEpisode` as append-only history, `MonitoringDecisionRecord` as a persisted audit trail (now including `requiresManualReactivation: Bool` so soft-pause-after-no-response is a typed fact on the decision record, not a parsed reason string), and `AppCapabilitySnapshot` as the single derived-capability cache read everywhere state is shown. `PlaceDerivedState` stays at exactly the six locked Place Detail variants (active/queued/paused+subreason/snoozed/visited/dismissed); `PausedSubReason` stays at exactly the four locked capability blockers; `MonitoringDisposition` stays at exactly three cases (monitor/queue/pause); `LibrarySection` stays at exactly five always-visible groups. Soft-pause renders as `queued(reason:)` with distinct copy, driven entirely by the `requiresManualReactivation` flag on the latest decision — no new enum case, no new screen, no new Library section anywhere in the contract.

Third — and this is where I'd focus actual adversarial energy — the entire product concept lives or dies on two failure modes the spec itself calls out, and I don't think "we implemented CoreLocation region monitoring" is sufficient proof either one is solved: (1) the 20-region cap and rotation logic. Region monitoring on iOS is notoriously unreliable in practice — CLLocationManager region entry/exit events can be delayed by minutes, suppressed by battery/motion state, silently dropped if the app is force-quit by the user (Apple's documented "if the user force-quits, region monitoring stops until relaunch" behavior), and monitoring silently fails past the real cap with no loud error, just event non-delivery. A rotation engine that reshuffles which places are "active" based on distance-to-user needs a testable, inspectable decision log, or nobody will ever be able to tell whether a missed notification was "correctly rotated out" versus "just broken." I want an explicit success metric here: given N saved places > 20, can we produce a deterministic, logged answer for which places are currently monitored and why, and can we simulate entering a monitored region in the simulator/test harness and see a notification fire. If that can't be demonstrated, "resurfacing" is vaporware wearing a UI.

(2) Sub-3-second capture is a measurable claim, not a vibe — it should mean something concrete like: from share-sheet invocation or a home-screen widget/App Intent tap to a persisted place, no required typing, no required screen transition beyond one confirmation. If the capture flow requires picking a category, writing a note, or waiting on geocoding to complete before it saves, that's a spec violation even if the UI looks nice. I'd want this treated as a hard, testable requirement, not a nice-to-have — the spec's own invalidation criteria says kill the product if capture isn't reliably instant.

Fourth, notification quality has no defined ceiling in the spec — "proactive resurfacing" and "not spammy" are subjective. Someone needs to decide now, not later, what counts as spam: e.g. do we suppress re-notifying about a place already dismissed/snoozed within some window, do we rate-limit total notifications per day, do we suppress notification when the app is foregrounded and the user is already looking at the library. Without a decision rule here, "delightful nudge vs annoying" becomes a purely aesthetic argument in the design phase with no way to adjudicate it, and QA has nothing to test against.

Fifth on non-goals — I want it explicit and written down, not implied, that the following are out of scope for this build: CloudKit/multi-device sync, collaborative/shared lists, on-device OCR/Vision, any backend service, any App Intents/Siri work beyond what's trivially achievable with local data (if Siri shortcuts turn into a rabbit hole, MVP wins — cut it, the spec itself says cut things that add friction), and building a second app. Any of these creeping back in mid-build should be treated as a scope violation requiring an explicit call-out, not a silent "while I was in there" addition.

**Claude (Frontend Engineer · Visionary) — Round 1**

I'm looking at this from the client side: what actually has to be true in the Swift code for "sub-3-second capture" and "state is a pure function of facts + now" to survive contact with three separate processes (main app, widget extension, share extension) all touching the same store.

The module boundary I want locked is a local Swift Package, `WaylayCore`, embedded in the Xcode project and linked by all three targets (`WaylayApp`, `WaylayShare`, `WaylayWidget`). It owns the Core Data stack, the domain value types, `MonitoringEngine`, `EntitlementService`, `DiagnosticsService`, and `LocationProviding`. Nothing in that package imports SwiftUI. This isn't just tidiness — it's what lets the "parallel build lanes can't invent incompatible types" acceptance check actually hold: whoever builds the widget and whoever builds the paywall are both compiling against the exact same `PlaceRecord` struct instead of three slightly-different reimplementations drifting apart. Inside `WaylayCore`, Core Data's `NSManagedObject` subclasses never leak past the persistence layer — every read out of the store gets mapped to a plain `Sendable` struct (`PlaceRecord`, `ActionEvent`, `NotificationEpisode`, `MonitoringDecisionRecord`) before it touches `MonitoringEngine` or any UI. That's the only way `MonitoringEngine` stays the pure, deterministic, easily-fixture-tested function the architecture phase promised — if it took `NSManagedObject` directly, "pure function of facts + now" becomes a lie the moment someone accesses a Core Data property off its owning context's queue and crashes a test in CI three weeks from now.

On state management: this app should commit to `@Observable` (not legacy `ObservableObject`/`@EnvironmentObject`) since the deployment target is iOS 17+. `PlaceRepository`, `MonitoringEngine`'s host service, `EntitlementService`, `AppCapabilitySnapshot`'s owner, and the new `AppRouter` are all `@Observable` classes injected via `.environment(_:)`. This matters more than it sounds like for this specific app because Home's status line, a Library row's badge, and Diagnostics all need to react instantly to the same underlying fact changing (a permission revocation, an entitlement change) — with `@Observable`'s fine-grained field tracking, a Library row only re-renders when its own place's derived state actually changes, not on every mutation to the array, which is the difference between this app feeling snappy with 60 saved places and feeling janky.

The concurrency design I want to nail down precisely, because it's where "instant capture" and "boring, trustworthy persistence" actually collide: `NSPersistentContainer` gets a `viewContext` (main queue, UI reads only, never written to directly) and one dedicated background context for writes. The capture write itself — insert one `Place` row, save — happens via `performAndWait` on that background context, called synchronously from the capture action. That's a single-row insert; it commits in low single-digit milliseconds, so blocking briefly is imperceptible and it means the receipt toast is never shown before the row is actually durable — no optimistic-UI lie. Everything downstream of that — reverse geocoding, `MonitoringEngine` evaluation, region reconciliation — dispatches asynchronously afterward and updates the UI when it resolves; capture success and "is this now monitored" are explicitly decoupled events. Because three separate processes (app, share extension, widget) each instantiate their own `NSPersistentContainer` against the same App Group SQLite file, I want persistent history tracking and remote-change notifications turned on in the store options (`NSPersistentHistoryTrackingKey`, `NSPersistentStoreRemoteChangeNotificationPostOptionKey`, both true) and each process observing `.NSPersistentStoreRemoteChange` to merge transactions from the others. Without that, a share-extension save can sit invisible to the main app's `viewContext` until an unrelated fetch happens to refresh it — which would make "share-captured place gets an immediate state decision" (already locked as a hard requirement) unreliable in exactly the case that matters most.

One real risk I want flagged now rather than discovered in the widget target: extension processes are memory- and time-constrained (widgets have a strict per-timeline-entry budget, share extensions get a short wall-clock window before the OS kills them), and `WaylayCore` pulling in the full Core Data stack on every widget refresh needs to stay lightweight — small schema, indexed lookups only, no forcing a full fetch-and-recompute of `MonitoringEngine` from the widget when a cached last-known-derived-state read is sufficient for a status line and recent-captures strip. The widget should read, not decide.

Here's the boldest thing I want to argue for, and I think it's a genuine upgrade rather than scope creep: the Diagnostics `MonitoringDecision`/`NotificationEpisode`/`UserActionEvent` audit trail was locked as a legibility requirement, but nothing says it has to read like a database dump. I want a derived `PlaceTimelineEvent` view that merges those three append-only logs into one chronological, plain-language feed per place — "captured Tuesday from a shared link," "considered for a watch slot Wednesday, prioritized because it was just saved," "reminded you Friday morning," "you snoozed it 3 days." That's the exact same data the architecture phase already requires us to persist, just rendered as a small story instead of a table — and it's the cheapest possible way to make the one screen that's explicitly "allowed to be more technical" also feel like this app's whole "little discoveries, not managing a system" promise instead of a debug menu bolted onto a warm app. It doesn't replace the plain per-place state+reason row everywhere else — that stays terse and always-visible — this is additive, and it's the first thing I'd cut under time pressure, but I think it's worth naming as the difference between "adequate diagnostics" and "diagnostics that's actually part of the product's charm."

The risk in my own direction: pushing everything shared into one Swift package plus cross-process Core Data merge notifications is more upfront wiring than three independent targets each doing their own thing, and if someone gets the remote-change observation wrong, bugs there are exactly the "why didn't this notify me" class of unfalsifiable problem this whole project has been trying to design out. I think it's worth it because the alternative — skipping persistent history tracking and hoping fetches happen to be fresh — is a landmine for the share-extension-to-main-app handoff specifically.

On testing, I want one addition beyond what's already locked: an automated performance gate, not just a vibe-check, for the sub-3-second capture claim — an `XCTest` on-device measurement asserting the Core Data commit path itself completes in well under a second (it should be near-instant), so "capture feels instant" has a number attached to it in CI rather than living purely in manual QA notes.

**Coordinator (Claude) — decision after round 1**

Round 1 of tech_specs is really two people independently building the same house, and it mostly lines up — but there's real, worth-naming daylight in a couple of spots that round 2 needs to close explicitly rather than let slide.

Codex laid out the systems/backend skeleton: strict synchronous-commit-before-anything-else capture semantics, share extension writing straight into the shared store with an explicit "pending-evaluation" state if it can't fully decide yet (never silent limbo), Core Data writes going through serialized background contexts rather than ad-hoc mutation from views/extensions, a clean rule that only a persistence-write failure blocks save (location-unavailable, monitoring-can't-arm, and cap-full all still let the save succeed with an honest reason attached), and a release-contract-grade testing plan (unit → integration → mandatory real-device GPX/walking tests, named as non-negotiable).

Claude took the same architecture and pushed it into concrete Swift/Xcode mechanics: a shared local Swift Package (`WaylayCore`) linked by all three targets so app/widget/share-extension can't drift into incompatible type definitions, Core Data's `NSManagedObject` never leaking past the persistence layer (everything crosses as plain `Sendable` structs), `@Observable` for the UI-facing services (fine-grained re-render control matters a lot once Library has 60 rows), a precise concurrency recipe (main `viewContext` for reads only, one background context for writes, capture commits via `performAndWait` before the toast ever shows), and — the piece nobody had said out loud yet — turning on Core Data's persistent history tracking and remote-change notifications so the three separate processes touching the same App Group SQLite file actually see each other's writes promptly, which is exactly what makes "share-captured place gets an immediate state decision" true in practice rather than in theory. She also floated an additive, first-to-cut "plain-language timeline" view for Diagnostics, and a hard CI performance assertion on the capture-commit path.

Nobody contradicted the other's core model — persisted facts vs. derived state vs. side-effect orchestration, `MonitoringEngine` staying pure, the reconciler owning the hard 20-region clamp, `EntitlementService` feeding a plain `effectiveMonitoringCap` — all of that is one shared design, not two competing ones.

But this wasn't a response-and-rebuttal round, it was two parallel first passes, and their interfaces-json blocks disagree in a few concrete, needs-a-decision ways:
- **State management pattern**: Codex's `AppRouterProtocol` is written against `ObservableObject`; Claude's `AppRouter` is `@Observable`. Since the target is iOS 17+, `@Observable` is almost certainly right, but nobody's explicitly said "yes, `@Observable` everywhere, `ObservableObject` nowhere" — that needs a clean, stated yes/no in round 2.
- **Paused sub-reasons count**: design_discussion and design_handoff locked exactly **four** paused sub-reasons (permission-revoked, notifications-disabled, force-quit-awaiting-reopen, device-rebooted-reregistering). Codex's `PausedReason` enum has seven, adding `preciseLocationReduced`, `noFreshLocation`, and `softPausedAfterNoResponse`. Claude's `PausedSubReason` sticks to the locked four. This isn't a style difference — it's a real question of whether those three extra conditions are (a) actually new paused sub-reasons that expand a previously-locked list, (b) capability notices that don't belong in the `paused` bucket at all, or (c) in the case of "soft-paused after no response," a distinct state that detailed_discussion already described separately from "paused" (the day-scale auto-soft-pause after a fired notification gets no response). That needs to be resolved by name in round 2, not silently merged either way.
- Minor shape drift on the same concepts (Codex's richer `PlaceRecord`/`UserActionEvent`/`MonitoringDecision` vs Claude's leaner `PlaceRecord`/`ActionEvent`/`MonitoringDecisionRecord`) needs to collapse into one canonical name and field list so parallel build lanes aren't coding against two slightly different structs.

Also open: Codex hasn't yet reacted to Claude's three additions (the `WaylayCore` shared-package boundary, persistent-history-tracking cross-process merge, and the `@Observable` pattern) — nothing suggests he'd object, but the acceptance check here is "parallel lanes can't invent incompatible types," so it needs an actual sign-off, not an assumption of agreement.

### Round 2


**Codex (QA & Risk · Closer) — Round 2**

Claude’s additions are the right ones to harden this from “clean architecture on paper” into something that won’t lie under pressure, and I don’t think we should leave any of them as soft preferences. I’m with her on `@Observable` over `ObservableObject`, full stop. This app is iOS 17+, we already want thin SwiftUI surfaces over long-lived services, and the failure mode to avoid is broad, noisy invalidation where a permission change or one row update causes half the Library to redraw because we hid everything behind an `ObservableObject` with coarse-grained publishing. `@Observable` is the cleaner fit for a two-tab SwiftUI app with a lot of derived row state. I also think the shared `WaylayCore` package boundary is no longer optional if we’re serious about “parallel lanes can work without inventing incompatible types.” The moment the share extension, widget, and app each define their own slightly different `PlaceRecord` or reason-code enum, the whole “why is this watched right now” promise becomes translation glue and drift. One package, one type system, `NSManagedObject` never leaves persistence files, plain `Sendable` structs everywhere else. That is not overengineering here; it is the minimum needed to stop three targets from quietly disagreeing.

The place I want to be stricter than the current merged draft is the paused-state creep. We already locked exactly four paused sub-reasons in the design phase, and I do not want us quietly reopening that because it feels technically expressive. Claude is right to keep `PausedSubReason` at four. `preciseLocationReduced` is a capability warning, not a paused reason. `noFreshLocation` is a transient capture/runtime condition, not a lifecycle state for a saved place. And `softPausedAfterNoResponse` is definitely not a fifth paused sub-reason because detailed_discussion already described it as its own product behavior: after an unresolved notification sits for about a day, the place moves into a soft-paused state so we stop nagging. If we jam that into the paused enum, we lose the distinction between “the OS/app capability prevented monitoring” and “the product intentionally backed off because this resurfacing didn’t seem useful.” Those are not the same truth and the user should not see them rendered as the same bucket. So I want one concrete decision: keep `PausedSubReason` at exactly the locked four, and add a separate derived state for `softPaused` if we need to represent the day-scale backoff explicitly in UI and diagnostics. If we don’t do that, we’ll end up with the exact kind of muddy state model that makes every bug report sound the same.

I’m also yes on Claude’s Core Data persistent history tracking and remote-change notifications. Without that, “share-captured place gets an immediate state decision” is aspirational. With three processes touching the same App Group store, stale reads are not a theoretical edge case; they are the default if we do nothing. Turn on `NSPersistentHistoryTrackingKey` and remote change notifications from day one, wire the main app and extensions to merge deterministically, and stop pretending polling/fetch-on-foreground is good enough. Same answer on the CI capture-latency assertion: yes, keep it. The product contract says sub-3-second capture, and the persistence portion of that path should be nowhere near the ceiling. If the Core Data commit path itself regresses above a few hundred milliseconds on device, that is an engineering failure we should catch automatically, not discover in a demo. Manual QA still matters for the whole flow, but the synchronous commit step is measurable and should have a guardrail.

So my push is: close this now with one canonical model, not two drafts half-merged. `@Observable` app-wide for UI-facing state holders. `WaylayCore` shared package required. **Claude (Frontend Engineer · Visionary) — Round 2**

Okay, Codex, let's actually resolve the three things the coordinator flagged instead of dancing around them, because I think all three have clean answers if we're precise about what each one actually is.

On `@Observable` vs `ObservableObject` — your `AppRouterProtocol` is written as `protocol AppRouterProtocol: ObservableObject`. I get why that's the reflexive choice, it's the pattern everyone's muscle memory defaults to, but there's no reason to pay for the old Combine-backed publisher machinery on a 17+ minimum deployment target, and for this app specifically it's not just a style tax. Library can realistically hold 60+ rows once someone's past the free cap testing scenario we're required to seed, and every one of those rows needs its own badge to re-render the instant its derived state changes — permission revoked, entitlement lapsed, snooze window elapsed — without re-rendering the other 59. `ObservableObject`'s `objectWillChange` fires for the whole object on any `@Published` mutation; `@Observable`'s compiler-synthesized access tracking means a `ForEach` row only invalidates when the specific field it actually reads changes. That's not a nice-to-have for this app, it's the difference between "diagnostics feels instant" and "diagnostics feels laggy the moment you have a real library." I want this locked flatly: `@Observable` everywhere in `WaylayApp`, `ObservableObject`/`@Published` banned from the codebase entirely, so nobody building a lane later has to guess which pattern a given file uses.

On the paused sub-reasons — I went back and actually checked what design_discussion and design_handoff locked, and it's unambiguous: Place Detail has exactly six state variants, and the `paused` variant has exactly four sub-reasons, named explicitly. Your three extras aren't wrong observations, Codex, they're real conditions that need to go *somewhere*, but two of them don't belong in `PausedSubReason` at all and the third needs to be folded in without adding a seventh case to an enum the design phase counted on staying at four. `noFreshLocation` is a capture-time condition — that's `HomeViewState.locationTemporarilyUnavailable`, already in the contract, nothing to add. `preciseLocationReduced` doesn't stop monitoring from working at all — a `CLCircularRegion` still fires entry/exit with reduced accuracy, just fuzzier — so a place with reduced precision should still read as `active`, with the caveat surfaced once at the capability level (`AppCapabilitySnapshot.isPreciseLocation == false` driving `DiagnosticsSummaryTier.capabilityReduced` and the Home degraded copy), not repeated as a per-place pause reason. And `softPausedAfterNoResponse` — that one's real and detailed_discussion actually names it, but notice it calls it "soft-paused," not "paused," and its whole point is that it's recoverable with a single tap and driven by engagement, not permissions. I think the clean fit is inside `queued`, not `paused` — `MonitoringDisposition.queue` already carries a `reason: String`, so "we noticed this one didn't seem useful, tap to bring it back" is just another queued reason string, no new enum case, no seventh Place Detail variant, and it still renders in the Paused-or-queued grouping in Library with the required plain-English explanation. I also want to flag something in your own draft that contradicts a rule you stated yourself, Codex — you wrote "`data_domain` should never know CoreLocation or StoreKit APIs," which I fully agree with, but then your `AppCapabilitySnapshot` used raw strings for auth status (fine) while my version from round 1 used `CLAuthorizationStatus`/`UNAuthorizationStatus` directly, which is the exact leak you correctly warned against — if `data_domain` imports CoreLocation just to hold an enum, every unit test for `PlaceStateResolver` now has to link CoreLocation for no real reason, and worse, it tempts someone building rotation logic to accidentally call a CoreLocation API from "pure" code because the type's already sitting right there. ### Round 3


**Codex (QA & Risk · Closer) — Round 3**

Claude, I think your round-2 instincts were right, and the fastest way to get this phase over the line is to stop inventing semantic distinctions the UI never promised. `softPausedAfterNoResponse` should not become a seventh top-level `PlaceDerivedState`. The design handoff locked exactly six Place Detail variants, and this is not a case where architecture gets to quietly overrule product because the state machine feels cleaner on paper. A place that got one notification, no response, and is now intentionally deprioritized is still not “paused” in the same sense as permission-revoked / notifications-disabled / force-quit / reboot-reregistering. Those four are capability blockers. No-response backoff is ranking behavior. So I want it folded into `queued(reason:)` exactly as you argued: same place, still eligible in principle, just not currently spending a slot because the app learned “this didn’t seem useful last time.” That keeps the user-facing model honest and prevents Diagnostics from ballooning into seven subtly different badges nobody can explain.

On the other two loose ends, I don’t think they’re actually loose if we take the app’s own surfaces seriously. `widget` should come out of `CaptureSource`. A widget tap routes into the app; it does not originate a durable capture the way Home, share extension, camera, or Siri can. If we keep it, somebody will eventually branch analytics, copy, or receipt behavior on a source that cannot really exist, and that’s exactly the kind of fake state that makes debugging impossible later. And `saveCapture` should return `CaptureResult`, not `PlaceRecord`, full stop. The whole product contract is “tap, persist, immediately know what happened.” The capture call site needs the persisted ID, the title it actually ended up with, the derived display state, and the confirmation copy for the toast without forcing Home and the share extension to each re-fetch and re-resolve on their own. Returning raw `PlaceRecord` would just leak orchestration work back into UI lanes and create race-y duplication around “what state do we show right after save?” I’m also fully with Claude on the domain enums: `LocationAuthLevel` and `NotificationAuthLevel` need to replace raw Apple types in `AppCapabilitySnapshot`. If the contract says data-domain code stays decoupled from CoreLocation/UserNotifications, then we should obey it instead of hand-waving one exception into the middle of the model layer.

The only extra thing I want locked before we call this done is one explicit failure rule around capture: if `saveCapture` has durably committed the row but monitoring evaluation or notification category registration fails afterward, the function still returns success with a degraded `displayState` and a truthful confirmation message, not an error that makes the user think the save itself failed. That distinction matters because “I saved it” and “the app is currently capable of resurfacing it” are not the same thing, and collapsing them is how trust gets lost. With that, I think the contract is coherent: six real user-facing states, no ghost capture source, one capture API that returns the exact artifact the UI needs, and domain models that don’t import framework types they shouldn’t know about.

### Round 4


**Codex (QA & Risk · Closer) — Round 4**

Claude’s reversal on `softPaused` is the right one, and I don’t want us to get cute and leave a stringly-typed trap in the one part of the app that most needs to stay mechanically honest. If a place is genuinely queued because it lost the cap contest, that place should automatically re-contest the next time the engine runs and a slot frees up. If a place is soft-paused because we already fired a notification, got no response for roughly a day, and deliberately backed off, that place must not silently drift back into monitoring just because the engine reran. Those are different semantics, not different copy. If we collapse both into `.queued(reason: ...)`, the only thing preventing a bug is “remember what this reason string meant,” which is exactly the class of mistake we’ve already spent this whole phase eliminating. So yes: `softPaused` should be a real `PlaceDerivedState` case, and yes: `MonitoringDisposition` should get a real `.softPause` case. Same for `MonitoringEngineProtocol` taking `notificationEpisodes` explicitly; the unresolved-episode age is part of the decision input, and pretending recent user actions alone are enough is just hiding state off to the side.

Where I do want to push back is the extra `LibrarySection.softPaused`. That’s the part that feels like architecture escaping into UX. Design locked a specific management surface: Active & Queued, Snoozed, Paused, then Visited/Dismissed. Soft-paused is not a new user-facing category on the same level as “permissions are broken” paused; it’s a behavioral subtype of “not being actively watched right now because we backed off after a weak signal.” Claude’s own point was that it should reuse the queued visual treatment and not become a seventh distinct screen concept. I think that has to carry through to Library grouping too. Put soft-paused rows inside `activeAndQueued`, but render them with their own `PlaceDerivedState.softPaused` badge/reason and only the reactivation action in detail. That keeps engine correctness and testability where they belong, without inventing a new top-level library bucket the user now has to learn. So the clean close is: distinct data/model case, distinct monitoring disposition, explicit `notificationEpisodes` engine input, but no distinct `LibrarySection`. That gives us the semantics we need without bloating the information architecture. I’m also fully with Claude on the router detail: consume pending deep links via `.onChange(of: pendingDeepLinkPlaceID)` rather than scene-phase gymnastics. That’s the safer edge-case answer and it doesn’t cost us anything.

Three concerns stay strictly separated: **persisted facts** (what actually happened — captures, actions, notification episodes, monitoring decisions), **derived state** (active/queued/paused+subreason/snoozed/visited/dismissed, always computed fresh from facts + "now" by `PlaceStateResolver`, never a cached enum a timer flips), and **side-effect orchestration** (CoreLocation registration, notification scheduling, widget refresh, StoreKit entitlement refresh), each owned by exactly one service. `MonitoringEngine` is pure — given places, capability snapshot, effective cap, recent actions, and notification episodes, it returns ranked `MonitoringDecisionRecord`s — and a separate `MonitoringReconciler` diffs that desired set against `CLLocationManager.monitoredRegions` (OS ground truth), applying a hard-coded clamp that never calls `startMonitoring` past 20 regions regardless of what ranking computes. Domain code (`data_domain`) never imports CoreLocation, UserNotifications, or StoreKit directly — `LocationAuthLevel`/`NotificationAuthLevel` are domain-owned enums mapped from Apple's raw types at the `LocationService`/`NotificationService` boundary, and `EntitlementSnapshot` is StoreKit's `Transaction.currentEntitlements` normalized before it ever reaches domain logic.

**File/module layout.** A local Swift Package, `WaylayCore`, is linked by all three targets (`WaylayApp`, `WaylayShare`, `WaylayWidget`) and owns the Core Data stack, all domain value types, `MonitoringEngine`, `PlaceStateResolver`, `EntitlementService`, `DiagnosticsService`, and the `LocationProviding` abstraction. `WaylayCore` never imports SwiftUI. `NSManagedObject` subclasses never leave the persistence files inside `WaylayCore` — every read out of the store is mapped to a plain `Sendable` struct (`PlaceRecord`, `UserActionEvent`, `NotificationEpisode`, `MonitoringDecisionRecord`) before touching the engine or any UI, which is what keeps `MonitoringEngine`/`PlaceStateResolver` genuinely pure and fixture-testable. `WaylayApp` holds SwiftUI views, `AppRouter`, and the four owning-lane view models; `WaylayShare` is a thin facade calling the same `capturePlace` use case as Home; `WaylayWidget` reads cached derived state, it never recomputes `MonitoringEngine` itself (widgets are time/memory constrained — they read, they don't decide).

**Data models.** `Place` (stable `PlaceID`, title, coordinates, radius, capture source, capturedAt/updatedAt, optional Pro media fields, lifecycle facts — dismissedAt/visitedAt/snoozedUntil/manuallyPromotedUntil/lastNotificationAt), `UserActionEvent` and `NotificationEpisode` as append-only history, `MonitoringDecisionRecord` as a persisted audit trail (now including `requiresManualReactivation: Bool` so soft-pause-after-no-response is a typed fact on the decision record, not a parsed reason string), and `AppCapabilitySnapshot` as the single derived-capability cache read everywhere state is shown. `PlaceDerivedState` stays at exactly the six locked Place Detail variants (active/queued/paused+subreason/snoozed/visited/dismissed); `PausedSubReason` stays at exactly the four locked capability blockers; `MonitoringDisposition` stays at exactly three cases (monitor/queue/pause); `LibrarySection` stays at exactly five always-visible groups. Soft-pause renders as `queued(reason:)` with distinct copy, driven entirely by the `requiresManualReactivation` flag on the latest decision — no new enum case, no new screen, no new Library section anywhere in the contract.

**Persistence strategy.** Core Data (`NSPersistentContainer`, not the CloudKit-backed variant) in an App Group container from the first commit, shared by all three processes. `NSPersistentHistoryTrackingKey` and `NSPersistentStoreRemoteChangeNotificationPostOptionKey` are both enabled so app/widget/share-extension merge each other's writes via `.NSPersistentStoreRemoteChange` — required so a share-extension capture is immediately visible to the main app, not just eventually. `viewContext` is main-queue, read-only for UI; writes go through a dedicated background context via `performAndWait`, called synchronously from the capture path so the receipt toast never shows before the row is durable. Stable UUID-based `PlaceID`s and `updatedAt` fields exist from day one so a future CloudKit sync layer doesn't require a schema rewrite, without shipping any cloud dependency in v1.

**Testing strategy.** Pure unit tests (no CoreLocation/StoreKit dependency) for `PlaceStateResolver`, `MonitoringEngine` ranking/cooldown/soft-pause logic, duplicate-merge heuristics, entitlement gating at multiple caps, and diagnostics reason-string generation — including fixture-driven tests seeding >8 places to exercise the cap/rotation logic under real contention. A `LocationProviding` fake simulates region entry/exit/authorization-change deterministically so the reconciler and the four paused sub-reasons are assertable in CI. Integration tests cover Core Data + App Group cross-target reads (via a stub extension target) and StoreKit entitlement transitions across active/expired/revoked/grace-period states using the `.storekit` local config. An on-device `XCTest` performance assertion guards the capture-commit path specifically, giving "sub-3-second capture" a number in CI rather than a vibe. Real-device (or Xcode GPX-route-simulated) walking tests remain a named, mandatory release gate for the actual geofencing product claim — never something CI coverage substitutes for.

## Final Output

**Architecture overview.** Thin SwiftUI views (using `@Observable`, never `ObservableObject`, anywhere in the codebase) sit over a small set of long-lived domain services — `PlaceRepository`, `MonitoringEngine`, `LocationService`, `NotificationService`, `EntitlementService`, `DiagnosticsService` — injected via `.environment(_:)`. Three concerns stay strictly separated: **persisted facts** (what actually happened — captures, actions, notification episodes, monitoring decisions), **derived state** (active/queued/paused+subreason/snoozed/visited/dismissed, always computed fresh from facts + "now" by `PlaceStateResolver`, never a cached enum a timer flips), and **side-effect orchestration** (CoreLocation registration, notification scheduling, widget refresh, StoreKit entitlement refresh), each owned by exactly one service. `MonitoringEngine` is pure — given places, capability snapshot, effective cap, recent actions, and notification episodes, it returns ranked `MonitoringDecisionRecord`s — and a separate `MonitoringReconciler` diffs that desired set against `CLLocationManager.monitoredRegions` (OS ground truth), applying a hard-coded clamp that never calls `startMonitoring` past 20 regions regardless of what ranking computes. Domain code (`data_domain`) never imports CoreLocation, UserNotifications, or StoreKit directly — `LocationAuthLevel`/`NotificationAuthLevel` are domain-owned enums mapped from Apple's raw types at the `LocationService`/`NotificationService` boundary, and `EntitlementSnapshot` is StoreKit's `Transaction.currentEntitlements` normalized before it ever reaches domain logic.

**File/module layout.** A local Swift Package, `WaylayCore`, is linked by all three targets (`WaylayApp`, `WaylayShare`, `WaylayWidget`) and owns the Core Data stack, all domain value types, `MonitoringEngine`, `PlaceStateResolver`, `EntitlementService`, `DiagnosticsService`, and the `LocationProviding` abstraction. `WaylayCore` never imports SwiftUI. `NSManagedObject` subclasses never leave the persistence files inside `WaylayCore` — every read out of the store is mapped to a plain `Sendable` struct (`PlaceRecord`, `UserActionEvent`, `NotificationEpisode`, `MonitoringDecisionRecord`) before touching the engine or any UI, which is what keeps `MonitoringEngine`/`PlaceStateResolver` genuinely pure and fixture-testable. `WaylayApp` holds SwiftUI views, `AppRouter`, and the four owning-lane view models; `WaylayShare` is a thin facade calling the same `capturePlace` use case as Home; `WaylayWidget` reads cached derived state, it never recomputes `MonitoringEngine` itself (widgets are time/memory constrained — they read, they don't decide).

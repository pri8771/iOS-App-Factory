**Codex (Product Strategist · Closer) — Round 1**

The design needs to make one thing unmistakable: this is not a music browser with note fields attached. It is a gift-making flow with three distinct emotional modes that should feel different on screen: starting the gift, building the sequence, and receiving the keepsake. If those modes blur together into one generic list-detail app, the product loses its reason to exist.

I would make the first screen a warm, low-friction library of projects, but not a utility dashboard. The top half should feel like a shelf of keepsakes: large cassette-card previews for existing projects, one obvious “Make a mixtape” action, and a small free-tier status chip that only becomes loud when the user is at the limit. Empty state here is important: when there are no projects, the whole screen should explain the promise in one glance with a single sample cassette card and one sentence about pairing songs with voice or text liner notes. Error state is mostly import failures or corrupted project files, surfaced as a non-destructive banner with “Couldn’t open project.” Loading state is minimal because this is local-first, but initial library hydration can show skeleton cassette cards instead of a spinner.

Tapping create should open a short occasion setup flow that feels ceremonial, not form-heavy. One screen is enough if it is structured well: title, recipient, occasion tag, optional short dedication, and theme selection. Theme choice should be visual and fast: maybe three or four premium cassette/booklet styles shown as tactile swatches, not a scrolling gallery. The primary action should read like “Start side A,” not “Next,” because the app needs a little authored character without becoming cute. Validation states matter: missing title or recipient should show inline guidance, not alerts. If the free tier blocks a second project, that block belongs right here at the commit action with a clear explanation that the user can keep existing projects but needs paid to start another.

The project authoring screen is the core of the app and should do most of the work. I would make it a vertically scrolling “liner notes booklet” view with a large hero cover at the top, recipient/occasion metadata under it, a compact progress summary, and then the track sequence as a stack of rich cards. Each track card needs to show artwork, title, artist, contributor, note status, and a strong “Add note” or “Play note” affordance. This screen should not make the user drill into a separate editor just to understand what is incomplete. The state model here has to be explicit: an empty project state with a prominent “Add your first song” button; a partial state where some tracks have notes and others do not; a fully authored state where the primary action shifts to preview/share; a read-only over-limit state if entitlement lapses; and a degraded playback state when songs are unavailable but notes remain playable. Reordering should happen here, and because drag handles are inaccessible by default, every card also needs an overflow menu with “Move earlier” and “Move later.”

Adding tracks should be its own modal flow because MusicKit search is a distinct mental mode. I would not bury search inside the project screen. The add-track screen should open with three clear states: permission-needed, browse/search, and no-results. Permission denied cannot dead-end the user; it should explain why music access matters and let them back out cleanly. Search results should be visually lighter than the project view because this is a sourcing tool, not the emotional center of the app. Once a track is selected, it should snap back into the authoring screen immediately and visibly land in the sequence. If the user hits the five-track free limit, the paywall belongs at the add action, not later.

The note capture flow should be a focused full-screen sheet for one track at a time. This is where I would spend the design budget. The top of the screen shows the track artwork and why this moment matters; the center is a large record surface with live waveform and countdown; the lower section has a tab-like switch between voice and text, but text must not look secondary or apologetic. Voice mode states are very specific: ready, recording, interrupted draft, recorded preview, and re-record confirm. Text mode states are draft, saved, and empty prompt. The app should be able to move linearly from one track to the next with a “Next song” action after saving a note, because users are likely in a momentum state while authoring. The screen must also support backing out without fear: autosave drafts, obvious saved status, and no hidden loss. If mic permission is denied, the sheet should simply default into text mode with a clear note that voice can be enabled later in Settings.

The playback screen for a finished mixtape should feel materially different from authoring. This is the keepsake screen, not an editor with controls added. I would make it more immersive and less scannable: enlarged cover, warm background gradient pulled from the theme, current track centered, liner note content foregrounded, sequence tucked below like a booklet track list. The important product truth is that the note is always primary and locally playable, while song playback is conditional. So the UI needs two separate playback layers: a guaranteed liner-note player and a MusicKit song player when available. If a song is unavailable, the screen should stay graceful: the note still plays, the track still holds its place in the sequence, and the song affordance becomes “Open in Apple Music” instead of failing in place. Empty and error states here are really “no tracks yet” and “this track can’t play here,” not generic loading spinners.

Sharing should split cleanly into two paths from the same share screen: “Send as gift” and “Invite a contributor.” Those are different promises and need different copy, artwork, and payload labels even if they use the same export machinery underneath. The share screen should preview exactly what is being sent: cover, occasion, track count, whether voice notes are included, and a blunt rights-safe line explaining that songs are referenced, not bundled. For contributor invites, the screen should also say that editing is one-at-a-time and returned as a project file, because the app must never imply live collaboration. Importing a received project should have a dedicated arrival screen, not dump straight into the editor. That arrival screen needs two versions: a recipient-friendly keepsake intro that explains the occasion and creator before showing tracks, and a contributor-friendly handoff intro that says what contribution is being requested.

Minimal widget and Live Activity support should hang off the above structure, not invent new concepts. The widget should surface either “in progress: 3 of 5 songs annotated” or “recent keepsake for Maya,” and deep-link to the relevant project. The Live Activity should be tied only to mixtape playback, showing current track and note progress, not raw Apple Music controls alone. If these surfaces do anything more than that, they are probably stealing energy from the core flow.

Visually, the direction should be warm, tactile, and a little imperfect in a premium way. Not skeuomorphic novelty, but clear physical references: paper grain, softened shadows, cassette-label typography, slightly aged color edges, booklet spacing. I would avoid flat neon or glossy streaming-app aesthetics. The palette should stay in warm ambers, faded reds, tobacco browns, cream paper, and deep charcoal for contrast. Typography should feel editorial for notes and compact/mechanical for track metadata so the story layer and music layer read differently at a glance. Motion should be small but intentional: track cards settling into order, waveform breathing while recording, a subtle tape-transport feel in playback transitions. No fake tape gimmicks that slow the app down.

Accessibility has to be designed in, especially because the most custom parts are the riskiest. Every track card needs explicit note status labels like “Voice note recorded” or “No note yet,” not just icons. Reordering must have non-drag actions. Recording must expose time remaining, recording state, and save state to VoiceOver. Theme contrast has to stay readable even when the app leans warm and nostalgic. Dynamic Type should reflow the booklet layout instead of truncating the emotional text. The keepsake playback screen cannot rely on artwork alone to communicate context. If a recipient cannot play any referenced songs, they still need a complete, understandable path through the gift using the note audio and text alone.

The biggest risk in this direction is that it asks the authoring and playback modes to feel premium and emotionally distinct, which can tempt the build into expensive surface work before the hard flows are solid. The correction is simple: the emotional differentiation should come from layout hierarchy and copy, not from overbuilt animation or custom rendering. If engineering time gets tight, cut decorative motion before you flatten the three-mode structure.

The detail the final phase output must preserve is this: every required workflow needs a screen path where the note stays inseparable from the song and often more important than the song. If any design choice makes the liner note feel hidden, optional, or fallback content, the app has drifted off-spec even if all the buttons technically exist.

**Claude (Design Lead · Visionary) — Round 1**

Let me put down my own strongest take on how this should actually look and feel as a shipped set of screens, before anyone else weighs in.

I'd start the whole app from a shelf, not a list. The home screen is a physical-feeling case rack — each mixtape rendered as a cassette spine/case with hand-set title, recipient, and a sliver of the cover art peeking through, not a generic SwiftUI List row with a thumbnail and a chevron. First-launch empty state is one cassette case sitting alone on the shelf with a soft prompt to make the first one — no onboarding carousel, no feature tour, just "make one" front and center, because the whole pitch collapses if the first thing a user sees is a form. Tapping "new" doesn't push a fresh screen, it's a full-screen modal ceremony: pick who this is for, pick the occasion (birthday, goodbye, closure, new baby, just because — a short fixed list, not a free-text field pretending to be meaningful), then pick a cassette shell color/theme. That theme choice is doing real work later, since it becomes the accent color and cover treatment threaded through the whole project, not a cosmetic dead end.

The project/authoring screen is the one I'd protect most, and I want to push back on treating it as a standard track list with a note icon bolted onto each row. I think the right model is a liner-notes booklet spread: each track is a page-like card showing the art, title, artist, and — right there, not hidden behind a tap — either the waveform of the recorded note or the first couple lines of the typed note. The note is visually part of the track's row, same size hierarchy as the song title, not a small subordinate icon. If a track has no note yet, that card should look visibly unfinished — a dashed outline or an empty tape-spool glyph — so the booklet itself communicates "this page isn't done" without needing separate progress UI. Reordering happens in an edit mode on this same screen: drag handles for sighted users, and a genuinely separate, equally first-class VoiceOver action set (move up / move down) exposed as custom accessibility actions on each card, not a hidden gesture-only path.

Recording a note deserves its own full-screen moment, not a sheet that feels like a form field. Tap a track's empty card and it transitions into a record screen where the art is still visible, the two-minute countdown ring wraps around the record button, and the amplitude visualization is a spooling tape reel rather than a generic bar waveform — turning while you talk, slowing when you stop. Retake is one tap and visually replaces the reel's fill, and I'd make the text-note entry point equally sized and equally reachable from the very first screen state, not a smaller "or type instead" link underneath — if it reads as secondary even slightly, people will default to skipping recording under social awkwardness, which is the one failure mode every phase before this has flagged as fatal.

For playback, I want to push the boldest idea I've got: split the finished mixtape into Side A and Side B, like an actual cassette. This isn't nostalgia for its own sake — it gives makers a real sequencing tool (build the arc, then flip for the resolution) and gives the playback screen a natural, delightful transition: reels visibly rotating, a tactile "flip" animation between sides instead of an arbitrary scroll position. Playback screen shows one track at a time full-bleed with its note surfaced in sync — voice notes play inline, text notes appear as it plays, and if the song itself can't resolve through MusicKit, that same card just swaps its "now playing" indicator for "open in Apple Music," never a spinner, because the note and art still carry the moment either way.

Handoff needs its own explicit fork in the UI, not a single ambiguous "share" button: one path is "send the finished gift," the other is "invite someone to add a track." They should look different enough at the button level that a recipient never has to guess which one they got — and on the receiving side, opening an imported file should land on a short intro card (who it's from, the occasion, "you've been invited to add a track" or "here's your finished mixtape") before ever showing a track list. Paywall moments are contextual sheets triggered exactly at the three real gates — second project, sixth track, adding a collaborator — framed around the occasion ("make one for someone else too") rather than a generic feature-comparison table, since the upsell logic here is emotional, not feature-based.

State-wise, every one of these screens needs the obvious four handled explicitly: the shelf has an empty/first-run state, a populated state, and needs a distinct "processing import" transient state for files just received. The track picker needs loading (MusicKit authorizing), denied-permission (with a real retry and an explanation of why access matters), empty search results, and populated. The record screen needs idle/recording/recorded/re-recording/interrupted-recoverable as distinct states, and interrupted-recoverable is the one I'd test hardest since it's the single scenario every prior phase flagged as the highest-consequence failure to get wrong. The playback screen needs a state for "track resolves and plays," a state for "track unavailable, deep-link only," and has to look intentional in both, not degraded in the second.

The biggest risk in the direction I'm pushing is that the cassette/tape metaphor is exactly the kind of thing that's fun to design and easy to overbuild — spinning reels, flip transitions, and spool visualizations are all custom-drawn, all need reduced-motion fallbacks, and none of them can be allowed to slow down the actual record-a-note interaction, which has to stay instant regardless of how much visual richness sits around it. If I had to name the one thing this phase's output must preserve above everything else, it's that the note and the song are rendered as one fused object at every step — same card, same size, same moment — because the instant they're visually separated into "song info" and "attached comment," the entire ceremonial premise flattens into a playlist app with voice memos stapled on, which is precisely the thing three phases of prior discussion have already said this product must not become.

5. **Record/Note Capture (full-screen per track)** — art context up top, large record surface with live waveform/spool visualization and 2-minute countdown ring, equally-weighted voice/text toggle. States: idle/ready, recording, interrupted-recoverable (highest-priority state to get right), recorded-preview, re-record-confirm (old file deleted immediately), text draft/saved/empty-prompt, mic-denied (defaults cleanly to text with an explanation). "Next song" action for linear momentum through the sequence.
6. **Playback (finished keepsake)** — immersive, visually distinct from authoring: full-bleed current track, warm gradient from theme, note foregrounded and synced (voice plays inline / text appears in sync), sequence tucked below. Optional Side A/Side B split as a sequencing and transition device. States: track resolves & plays, track unavailable (graceful "Open in Apple Music" swap, never a spinner), no-tracks-yet empty state.
7. **Share/Handoff** — explicit fork: "Send as gift" vs "Invite a contributor," visually distinct buttons/copy, preview of exactly what's included (cover, occasion, track count, notes included, rights-safe "songs are referenced, not bundled" line).
8. **Import Arrival** — dedicated intro screen before any track list, two variants: recipient keepsake intro (who/occasion) and contributor handoff intro (what's being asked), plus fail-closed error state for malformed/corrupt files.
9. **Widget** — in-progress project status ("3 of 5 annotated") or recent keepsake, deep-links into the app.
10. **Live Activity** — tied to mixtape playback only: current track + note progress.

**Primary user flow**
Shelf → New Occasion (title/recipient/occasion/theme) → Authoring booklet → Add Track modal (MusicKit) → returns to booklet with track landed in sequence → tap track card → Record/Note Capture (voice or text) → Next song loop until sequence feels done → reorder as needed (drag or VoiceOver actions) → Playback preview (Side A/B) → Share/Handoff (gift vs invite) → recipient/collaborator lands on Import Arrival → either plays the keepsake (Playback screen) or contributes a track+note and hands it onward.

**State model per screen**
Every screen above lists its explicit empty/loading/populated/error states; the two states called out as highest-consequence across both agents: (a) interrupted-recoverable during recording — must never silently lose a note, and (b) track-unavailable during playback — must degrade gracefully to a deep link while the note keeps playing, never a stuck spinner. Paywall states (2nd project, 6th track, collaborator invite) are contextual sheets triggered at the exact action point, not a separate settings-style paywall screen.

**Visual direction**
Warm, tactile, premium-nostalgic: paper grain, softened shadows, cassette-label typography for track/song metadata (compact/mechanical) contrasted against editorial typography for liner-note text, amber/faded-red/tobacco-brown/cream/charcoal palette. Small intentional motion only — cards settling into order, waveform "breathing" while recording, a tape-transport-style transition between playback sides — with reduced-motion fallbacks, and explicitly no decorative motion allowed to slow down the record-button's <300ms start time. The cassette/tape metaphor (spooling reel, flip transitions) is approved as the visual language but flagged as the top over-build risk — cut decorative richness first if schedule pressure hits, never the note/song fusion or recording responsiveness.

**Accessibility notes**
Non-drag reorder via explicit custom VoiceOver actions ("Move earlier/Move later") on every track card. Every track card exposes note status as a spoken label ("Voice note recorded" / "No note yet"), not icon-only. Recording screen exposes time-remaining, recording state, and save state to VoiceOver. Dynamic Type must reflow the booklet layout rather than truncate liner-note text. Theme contrast must stay readable within the warm/nostalgic palette. Keepsake playback must remain fully understandable via note audio/text alone when zero referenced songs resolve for a recipient — artwork alone is never sufficient context.

### Round 1


**Codex (Design Lead · Skeptic) — Round 1**

Linernotes should be defined this tightly: a local-first iPhone app for making a giftable mixtape project where each song from your library carries a short spoken or written explanation of why it belongs, and the finished result is experienced as a story-rich listening keepsake rather than a plain playlist.

The primary user is someone making a one-off but emotionally important gift for a specific person or moment: birthday, breakup, anniversary, new baby, long-distance goodbye. That matters because the UX should not feel like daily playlist management. It should feel like assembling something deliberate, almost ceremonial. The app starts with the occasion and recipient, not with a music search box. If the first screen feels like “make a playlist,” the concept is already diluted.

The core loop should stay brutally simple: create a mixtape project for an occasion, pick tracks from your library, attach a liner note to each track by voice or text, optionally hand the project file to one collaborator to add their contribution, then play or share the finished keepsake. The playback experience is not just audio controls. It needs to foreground sequence, cover art, and the note for the current track so the “why” is inseparable from the song order.

The hard scope boundaries need to be explicit because this concept can sprawl fast. It is not a social network, not a feed, not group chat, not open-ended collaborative editing, not cloud sync, not a generic music journal, and not a tool for exporting copyrighted songs into a video or audio file. Collaboration in v1 should be linear handoff or forked copies through share sheet exchange, nothing smarter. If two people edit separately, the app should not pretend to merge. Widgets and Live Activities should exist only in minimal form if they do not threaten the main loop. Speech-to-text is accessibility support, not a pillar.

The biggest product risk is that the gifting payoff collapses if the share outcome is vague or disappointing. “Share” cannot imply “I sent you a playable mixtape file with the songs inside,” because that will run straight into rights and expectation problems. The app has to teach the real artifact clearly: you are sharing a beautifully packaged mixtape project containing the story layer, track references, artwork, and voice notes, while actual song playback stays tied to MusicKit and the recipient’s access. If that is not clear in the product promise and in the share flow copy, users will feel tricked.

Measurable success for the MVP should be concrete. A new user can create a project, add up to five tracks on free tier, record or type a note for each, preview the sequence in a finished playback screen, and export or share a project artifact without needing any server. Permission-denied states for microphone and music access are handled without dead ends. The unpaid gate actually blocks a second active project, a sixth track, and collaborator adds. A recipient opening a shared project can understand the occasion, the track order, and the meaning behind each song in under a minute. That last point is the detail the final output must preserve: the product only works if the emotional narrative is instantly legible, not hidden behind utility-app structure.

**Claude (QA & Risk · Pragmatist) — Round 1**

Here's where I'd land on this, thinking about what actually has to be true for someone to call this "done" and ship it.

The product promise, stated as tight as I can make it: Linernotes lets you build a mixtape for one specific person and occasion by picking songs from your own library and recording a short voice or text note on why each song matters, then hand the finished thing to them as a playable keepsake. I'd resist any temptation to soften that into "a music sharing app" or "a playlist app with comments" — the occasion-bound, one-recipient framing is what makes the scope tractable and what makes the free/paid split make sense (1 project free is stingy for a general playlist tool, but completely reasonable for "your one gift right now").

Primary user and scenario: someone with an existing Apple Music library who has a specific person and a specific moment in mind — a birthday, a breakup, a new baby — sitting down once, over maybe 20-40 minutes, to build something with intention. Not a habitual user opening the app daily. That has a direct consequence I want on the record now: retention metrics like DAU/WAU are the wrong yardstick for this app, and the invalidation criteria already baked into the spec (one mixtape, no return within 90 days, no organic invites) is honestly a plausible outcome even for a well-built version of this app. The build should not be judged as failing if usage looks "bursty" — that's what correct usage looks like for a gifting tool. Where I'd actually get worried is if the core loop itself has friction that makes people abandon mid-build, which is a build-quality problem, not a retention problem.

Core loop, concretely: create occasion → pick tracks from library via MusicKit → record a note (voice or text) per track → arrange/sequence → play back locally to check it → hand off via share sheet (either as a collaborator invite pre-completion, or as a finished keepsake to the recipient). Five steps, and every one of them is a place a real build can silently fail if we're not careful: MusicKit auth denied or no subscription: what does step 2 look like with an empty or restricted library? Recording denied mic permission or interrupted by a call: does the user lose a note they just spent two minutes recording? Share sheet handoff where the recipient doesn't have the same track available in their region/catalog — do we fail silently or do we make it obvious?  I don't think we solve all of these elegantly, but I do think each one needs a defined, non-crashing behavior before I'd call this loop "real."

Hard scope boundaries, the ones I'd fight to keep out even under time pressure: no server, no account system, no real-time multi-user editing, no literal audio baked into an exported file (this one isn't negotiable — it's a platform/legal wall, not a design choice), no merge resolution for concurrent collaborator edits. I'd also explicitly keep speech-to-text captioning out of the v1 build entirely rather than half-building it — it's labeled optional/accessibility in the spec, and a half-working transcription feature is worse than no transcription feature.

Measurable success criteria I'd actually hold this MVP to: a first-time user can go from "no project" to "playable finished mixtape with at least one recorded voice note" without hitting a crash, a silent data-loss moment, or an unhandled permission-denial dead end. The free-tier ceiling (1 project, 5 tracks, no collaborators) has to visibly and actually block the user, not just show a paywall that doesn't gate anything. And critically: if I kill the app mid-recording, mid-arranging, or right after adding a track and before saving, the project needs to still be there when I reopen — that's the single test I'd run first on any build claiming to be "done," because local-first persistence is the whole premise and it's the easiest thing to get subtly wrong (autosave timing, recording temp files not committed, etc).

The single biggest risk to the whole effort, if I had to name one: the emotional pitch ("why this song matters, fused with the music") is strong on paper but rides entirely on the recording-and-playback experience feeling good, not janky. If voice note recording is clunky — bad latency starting/stopping, no re-record option, no visual feedback that it saved — the whole premium/keepsake positioning collapses no matter how nice the cassette UI looks. I'd rather see one track's record-and-playback flow be flawless than five tracks with a shaky one.

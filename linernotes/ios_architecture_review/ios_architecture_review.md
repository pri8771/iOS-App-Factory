### Round 1


**Codex (Backend / Systems Engineer · Skeptic) — Round 1**

I would lock this app to a very boring, explicit architecture, because the product surface is sentimental but the failure modes are not. The wrong move here is inventing a clever reactive graph, overfitting for future CloudKit, or pretending StoreKit and extensions are minor add-ons. The safest shape is a SwiftUI app on iOS 17+ with Observation, one app-level library store, one per-project editor, and a small set of isolated services behind protocols so the product code is testable without MusicKit, AVFoundation, or StoreKit being present.

Concretely, I’d use `@Observable` models marked `@MainActor` for UI-facing state, but I would not let those types own raw file I/O, recording sessions, or MusicKit calls directly. The app should have a `ProjectLibrary` that indexes project summaries and routes open/import/delete actions, a `ProjectEditor` for one loaded project, and separate services for persistence, package import/export, artwork snapshotting, audio recording/playback, MusicKit lookup/playback, entitlements, widget/live-activity summaries, and permissions. That keeps the view models small and makes the hard parts replaceable in tests. The data model needs stable IDs everywhere now, even though there is no cloud yet: `Project`, `TrackEntry`, `LinerNote`, `Contributor`, `Theme`, `EntitlementSnapshot`, and `ProjectPackageManifest`. If IDs are sloppy in v1, CloudKit later becomes a rewrite, not an extension.

For persistence, I would not use SwiftData for the core project store. It is tempting, but this app’s real requirements are atomic writes, schema-versioned import/export, explicit asset lifecycle, easy corruption handling, and deterministic offline behavior. A file-backed store is the cleaner fit. Each project should live in its own folder containing one versioned JSON manifest plus subfolders for artwork snapshots and recorded note files. The library root keeps a lightweight index file for fast shelf loading and migration bookkeeping. Writes should be atomic write-then-rename, debounced during editing, and force-flushed on background/inactive transitions and interruption paths. Deleting a track or project should synchronously delete owned assets first or mark them for immediate cleanup with a journal so a crash mid-delete cannot leave private audio lying around forever. Importantly, imported packages should never be “opened in place” as live project files; they should decode into memory, validate schema/version/content limits, then be copied into the app’s own storage or rejected.

The package format should be treated as an untrusted input surface from day one. I would make it a package directory or zipped container with a manifest JSON, explicit schema version, declared package kind (`keepsake` vs `invite`), note/audio/artwork file references, and checksums or at least length/type validation. No package should be able to overwrite an existing project by path trickery, inject absurdly large files, or crash decode because the app assumes its own output is the only input. That means import has to be fail-closed and size-bounded. This matters more here than in a normal local-first app because share sheet exchange is effectively the app’s external API.

Apple framework choices are straightforward if we stay disciplined. SwiftUI plus Observation for UI. MusicKit for library search and playback. AVFoundation for audio recording and local note playback. UniformTypeIdentifiers plus file/document handling for package import/export. ActivityKit and WidgetKit for the required extensions, but only against a tiny shared summary model in an App Group, never full project state. StoreKit 2 for local paywall state and future subscription wiring, but for MVP I would still architect an entitlement provider abstraction so the app can run with a local debug/mock provider and the free-tier enforcement logic is identical in production later. If the paid tier is part of product truth, the gate belongs in the domain layer, not in view conditionals.

On StoreKit specifically, I would be skeptical of any plan that says “we’ll wire it later” without defining the contract now. The app needs an `EntitlementService` that answers very small questions deterministically: can create project, can add track, can add collaborator, can export keepsake, and why not. Free vs paid gating should be evaluated from a cached entitlement snapshot, not from live purchase checks at the moment of action. If StoreKit is unavailable, offline, or the receipt is stale, the user should not get random behavior. Existing over-limit content must remain readable/playable based on the last known entitlement policy, with creation blocked only where the contract says so. If that rule is not explicit in the architecture, somebody will accidentally make access depend on network-ish purchase refresh behavior and break the “permanent keepsake” trust promise.

Permissions and privacy need to be treated as core architecture, not screen copy. Music access and microphone are the only truly necessary permissions in v1. There is no reason to ask for notifications, photo library write access, speech recognition, contacts, or anything else. Text notes must be first-class specifically so microphone denial does not break the product. Music permission denial should still allow project creation and import/playback of note-only keepsakes. The privacy model should be simple: all intimate content remains on-device unless the user explicitly exports or shares a package; no analytics dependency should exist for the core app. If analytics are ever added later, voice-note metadata and recipient names should be considered sensitive by default and excluded unless there is a compelling reason. I would also store note audio with complete file protection where practical, because these are private spoken messages, not generic cache files.

Offline behavior is where naive MusicKit assumptions quietly break the product. The app is local-first, but MusicKit is not an offline music source in the general sense. So the architecture must separate “track identity and story object” from “track playback capability.” When a track is added, the app snapshots display metadata and artwork bytes locally so the project is always renderable offline. Playback state for the licensed song should be computed at runtime and allowed to fail independently of the note experience. If the user or recipient has no connectivity, no subscription, a different region, or catalog drift, the project still opens cleanly and the note still works. That means the playback coordinator cannot be the owner of the sequencing truth; the project model owns sequence, and playback is an optional overlay on top.

I would explicitly reject any ML, LLM, or AR work in v1. Speech-to-text is the only remotely plausible one, and even that is unjustified here. It adds another permission domain, more state, more storage, more edge cases, and does not strengthen the core gifting loop enough to earn the complexity. AR is pure distraction. The right architecture move is to keep `LinerNote` extensible enough to carry optional transcript metadata later, but do not build the feature now.

Dependency policy should be nearly zero third-party. I do not see a justified need for any external package if the target is iOS 17+. Apple gives enough here. The only reason I would allow a dependency is if there were a severe gap in waveform rendering or zip/package handling that materially reduced risk, but I don’t think that threshold is met. Given the permissive-license constraint and the sensitivity of audio/files/import, first-party frameworks are the safer call both legally and operationally.

Testability is where I’d be hardest on the architecture. If MusicKit, AVAudioSession, file I/O, and StoreKit are all called directly from views or monolithic observable objects, this app will be miserable to verify and regression-prone around the exact flows that matter. I would require protocol-backed services and a package of fixture project files. Unit tests should cover project creation, add/remove/reorder tracks, note attachment lifecycle, entitlement gate enforcement, schema migration, import rejection of malformed packages, atomic persistence recovery, and orphaned asset cleanup. Integration tests should run against temporary directories, not mocks, for persistence and package export/import. UI tests should cover permission-denied flows, free-tier blocks, recording interruption recovery with a mocked recorder, import arrival variants, and the unresolved-track playback fallback. The most important tests are not visual; they are around “did we lose or leak private data” and “did we corrupt a gift.”

The biggest risk in my own direction is that file-backed persistence plus explicit import/export/package validation is more engineering work up front than dropping everything into SwiftData and shipping faster. That’s true. But this app’s trust boundary is the package, the note file, and the autosave guarantee. SwiftData helps less with those than people want to believe, and it introduces another layer of opaque behavior right where deterministic recovery matters. If the team chooses convenience here, the likely failures are silent data loss, messy asset cleanup, and import brittleness.

The detail the final phase output must preserve is this: Linernotes is not just “local-first.” It is a file-owning, package-importing, privacy-sensitive app whose primary artifact is intimate user-generated audio attached to licensed music references it does not control. The architecture must separate local truth from external capabilities at every seam: project state from MusicKit playback, entitlement enforcement from live StoreKit calls, package validation from in-app trust, and UI state from persistence side effects. If that separation gets blurred, the app will feel fine in happy-path demos and break exactly where real users most care.

**Claude (QA & Risk · Pragmatist) — Round 1**

I'm going to hammer on the parts of this architecture that are easy to get wrong specifically because they're invisible in a demo and only surface under real device conditions, real interruptions, or CI — since that's exactly the gap between "looks done" and "is done" for this app.

On SwiftUI architecture: the `@Observable ProjectLibrary` + per-project `@Observable ProjectEditor` split locked in the design handoff is right, but I want to add a hard requirement the prior phases didn't spell out — every external dependency (music catalog access, audio recording, file storage, entitlement state) needs to sit behind a narrow protocol from day one: `MusicLibraryProviding`, `AudioRecording`, `ProjectFileStore`, `EntitlementProviding`. FileManager plus `UniformTypeIdentifiers` for the project package format and the document-opener capability. WidgetKit and ActivityKit for the widget/Live Activity, sharing state through an App Group container as already decided. StoreKit 2 for entitlement, but I want to flag precisely how much of that is in scope: build against `Product`/`Transaction` types and a `.storekit` local configuration file in the Xcode scheme so the paywall and entitlement gate can be exercised in simulator without App Store Connect, but the actual purchase flow itself does not need to be wired to a real product ID per the production-readiness contract — the entitlement gate's *logic* is must-have, live commerce is not.

Given the explicit requirements — atomic multi-file writes, binary audio/image assets living alongside metadata, an inspectable export format, and an explicit schema version for fail-closed import — I'd use a plain FileManager-based store: one directory per project, a versioned Codable JSON manifest at its root, subfolders for artwork and voice-note audio files, written with the temp-file-then-rename pattern already agreed. SwiftData would add an ORM layer fighting against exactly the properties this app needs (a project needs to *be* a self-contained folder you can literally hand to the share sheet). On the storage-growth risk flagged in an earlier phase: I'd lock the audio format now rather than let it drift — mono AAC at a modest bitrate (32-48kbps) via `AVAudioRecorder` settings, which puts a 2-minute note at well under 1MB. That's a five-minute decision now versus a "why is this project 40MB" bug report later.

Permissions and privacy: `NSAppleMusicUsageDescription` and `NSMicrophoneUsageDescription` with the specific, plain-language copy already decided, not boilerplate. Entitlements needed: MusicKit capability, App Groups (widget + Live Activity), and a declared exported UTType for the project package plus Info.plist document-type registration so "Open in Linernotes" actually works from Mail/Files/Messages — I'll flag this as a real risk because it's the kind of thing that's trivial to configure and trivial to forget, and if it's missing the entire import/receiving side of the product — half the core loop — silently doesn't work despite all the Swift code being correct. I do not think this app needs a Background Modes audio capability, and I'd actively argue against adding one: if the user backgrounds mid-recording, the pragmatic and lower-risk behavior is to treat that identically to any other interruption — stop, save the draft, surface "interrupted-recoverable" on return — rather than trying to keep recording alive in the background, which adds App Review scrutiny and real audio-session complexity for a scenario (dictating a liner note while backgrounded) nobody asked for.

On testability specifically, since that's my seat this round: I want three tiers named explicitly so nobody assumes UI tests alone prove correctness. Unit tests cover the stuff that's actually likely to break silently — Codable round-trip of the versioned project schema, the entitlement gate's boundary conditions (exactly one project allowed, exactly five tracks, sixth blocked, collaborator blocked) tested as pure logic independent of any UI, the reorder mutation function, the orphan-file-sweep logic, and the atomic-write-then-rename path deliberately tested by simulating a crash mid-write and confirming the app can still open its own file afterward. Protocol-mocked tests cover the state machines that depend on hardware — a fake `AudioRecording` that can synthesize an interruption notification lets you actually assert the record screen reaches `interruptedRecoverable` without touching a real microphone. One XCUITest smoke test covers the golden path end-to-end (create → add a fake-injected track → record via a fake audio engine → reorder → preview → share sheet appears), gated behind a launch-argument test seam that swaps in fake MusicKit/audio providers — without that seam this test literally cannot run in CI. Everything else — real permission-denial prompts, a real phone call interrupting a real recording, sending a real package to a second physical device and reopening it — has to be named plainly as a manual, device-only test checklist, not something the team can pretend is covered by automation.

The biggest risk to my own recommendation here is scope: protocol-seaming four subsystems before any UI exists is real up-front engineering cost, and on a build with this many "must-have" acceptance criteria already, there's a genuine temptation to build straight against MusicKit/AVAudioRecorder to move faster and add the seams later. I think that's a trap specifically for this app, because so much of what's being graded is failure-mode behavior that can't be exercised without fakes. The detail I want the final output to preserve above everything else: nothing in this app should call MusicKit, AVAudioRecorder, or FileManager directly from a View or a screen-level view model — every one of those calls goes through a protocol with a real implementation and a fake implementation from the start, because that's the only way "interrupted-recoverable never loses data" and "malformed import fails closed" end up being things we've actually verified rather than things we've merely built and hoped about.

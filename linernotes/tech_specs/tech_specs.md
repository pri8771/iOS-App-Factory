### Round 1


**Codex (QA & Risk · Pragmatist) — Round 1**

I’d keep the tech spec narrower than the design language. This app will look emotionally rich, but the implementation needs to be aggressively boring: one local project store, one project editor at a time, explicit file ownership for every artwork and voice-note asset, and hard seams around anything the OS or Apple services can make flaky. If this gets built as “SwiftUI screens talking straight to MusicKit, AVAudioRecorder, StoreKit, and FileManager,” it will demo fine and fail exactly where the spec is strictest: interrupted recording, malformed imports, offline rendering, and entitlement enforcement.

The smallest architecture that actually ships is a four-lane split. `data_domain` owns pure models, validation, migrations, package manifests, and entitlement rules. `services_utilities` owns MusicKit lookup/playback, recording, file persistence, package import/export, artwork snapshotting, and widget/live-activity summary writing. `primary_ui` owns the shelf, authoring, recording, playback, share, and import screens plus their `@Observable @MainActor` editors. `polish_resilience` owns interruption handling, orphan cleanup, accessibility helpers, reduced-motion fallbacks, and test fixtures/harness seams. That lets parallel work proceed without lane-local type invention. The build should target iOS 17+, SwiftUI + Observation, first-party frameworks only, and no SwiftData. The project itself is the export artifact: a versioned `.linernotes` file package with a manifest JSON plus `Artwork/` and `VoiceNotes/` folders. That choice solves persistence, import/export honesty, and local-first behavior in one move.

The module layout should follow that reality instead of trying to look “clean” in an abstract MVVM sense. I’d use something like `App/` for entry, routing, app-group config, and scene-phase hooks; `Domain/` for `Project`, `TrackEntry`, `LinerNote`, `ContributorIdentity`, `ThemeToken`, `EntitlementSnapshot`, validation, and migration code; `Services/ProjectStore`, `Services/Packages`, `Services/Music`, `Services/Audio`, `Services/Entitlements`, `Services/Extensions`; `Features/Shelf`, `Features/NewOccasion`, `Features/Authoring`, `Features/AddTrack`, `Features/RecordNote`, `Features/Playback`, `Features/Share`, `Features/ImportArrival`; `UI/Components` for shared cards/buttons/indicators; and `Tests/Fixtures` for sample manifests, corrupt packages, and fake services. I would not bury package decoding or cleanup logic inside a view model. Those need to be standalone and testable with temp directories.

The data model needs to be strict about what is local truth versus external capability. `Project` is the owned artifact with stable `projectID`, title, recipient, occasion, optional dedication, theme, created/updated timestamps, contributor mode, track order, package kind, and a schema version. `TrackEntry` stores stable `entryID`, sequence index, contributor identity, music reference, local snapshot metadata, and note pointer. The music reference should separate external identifiers from renderable snapshot fields: `appleMusicSongID` is optional capability, while `snapshottedTitle`, `snapshottedArtist`, `snapshottedArtworkFilename`, and duration are local truth. `LinerNote` should be an enum-backed payload: text, voice, or both, with voice storing filename, duration, bitrate preset, and created timestamp. `ContributorIdentity` can stay minimal in v1: display name and role. `EntitlementSnapshot` should not model billing complexity; it just answers whether the app is free or paid and when that was last resolved. A separate pure `ProjectLimits` rule type should answer “can create project,” “can add track,” “can invite collaborator,” and “can export,” so gating is deterministic and unit-testable.

Persistence should optimize for survival, not elegance. Every project lives in `/Projects/<projectID>.linernotes/` with `manifest.json`, `Artwork/`, and `VoiceNotes/`. Writes are debounced during active editing but must force-flush on `scenePhase` changes to `.inactive` and `.background`, and on audio interruption callbacks. All writes are temp-file then atomic rename. The manifest should be small and self-sufficient enough that the shelf can load summaries without touching audio blobs. Imported packages are never trusted in place. The flow is decode to a staging directory, validate schema version, file count, total size, declared filenames, and manifest coherence, then copy into owned app storage. Any mismatch fails closed with a clear import error and no mutation to the existing library. For privacy, deleting a track should delete its owned voice asset immediately; deleting a project should delete the whole package directory; app launch can run a cheap orphan sweep as a safety net, but cleanup-on-delete is the primary correctness path.

Concurrency boundaries need to stay simple. UI-facing editors are `@MainActor`; file I/O, image decoding/downsampling, import validation, and package copy run off-main in async services. `Task {}` from UI should only call service methods and assign resulting state back on the main actor. The recording service should expose a narrow async/stateful API rather than leak `AVAudioRecorder` details upward. The Music service should do two jobs only: search/select tracks and attempt playback/open-in-Apple-Music resolution. It should not own sequence, note state, or import logic. The playback screen must be able to function even if the Music service says every track is unavailable.

Error handling needs named, stable failures rather than ad hoc strings. I’d use `ProjectImportError`, `ProjectPersistenceError`, `AudioRecordingError`, `MusicLookupError`, and `EntitlementGateReason`. The important thing is not exhaustive taxonomy; it’s that the UI can distinguish retryable from terminal and private-content-safe from user-visible. For example, “track unavailable in this region/account” is a normal playback state, not an error banner. “Malformed package” is a terminal import state. “Could not save recording draft before interruption” is a critical failure and should surface as such because it violates the trust boundary.

Testing has to be designed before implementation or it won’t happen. I’d require pure unit tests for manifest encode/decode, schema migration, entitlement boundaries, reorder mutation, track deletion asset cleanup, and orphan sweep. Then temp-directory integration tests for project create/save/load, atomic overwrite recovery, import validation, export contents, and corrupted package rejection. Then fake-service state-machine tests for record interruption, mic denied, music permission denied, sixth-track gate, second-project gate, collaborator gate, and unavailable-track playback fallback. One XCUITest smoke path can run with injected fake music/audio services. Real device manual checks still matter here: actual microphone permission flow, actual MusicKit account weirdness, actual background interruption while recording, actual widget/live-activity update timing, and actual package handoff between devices. If those aren’t explicitly called out, people will pretend the simulator covered them.

The biggest objection to my own direction is obvious: file-package persistence plus staged import validation plus protocol seams is more work than throwing models into SwiftData and wiring views directly to Apple frameworks. That’s true. I still think it’s the smallest thing that actually ships because this app’s “done” definition is mostly about not losing intimate audio, not lying about exports, and not corrupting shared project files. The cheap-looking architecture is the risky one here.

The detail the final tech spec must preserve is this: the project package is the product boundary. Everything important flows from that. Local persistence, export/import, privacy, cleanup, offline rendering, collaborator handoff, and malformed-input defense all get simpler if the app treats a mixtape as an owned versioned package with snapshotted metadata and locally owned note assets, while treating MusicKit playback and StoreKit entitlement resolution as optional external capabilities layered on top.

**Claude (Frontend Engineer · User Advocate) — Round 1**

I'm looking at this from the seat of the person who actually has to wire up the screens and keep them fast and legible on a real phone, and the person who's picturing an actual user — let's say Priya, making a goodbye mixtape for her college roommate the night before a flight, on a five-year-old iPhone with mediocre signal at the airport — sitting through every one of these interactions.

The first thing I want locked down before any parallel lane starts writing code is the module boundary, because "protocols behind everything" only prevents incompatible types if the protocols and their real signatures exist before four people start guessing at them independently. I'd structure this as one local Swift package, `LinernotesKit`, imported by three targets: the app, the widget+Live-Activity extension (WidgetKit and ActivityKit widgets live in the same extension bundle in iOS 17+, so that's one target, not two), and the test target. `LinernotesKit` holds `Models/` (Project, TrackEntry, LinerNote, Contributor, Theme, EntitlementSnapshot, ProjectPackageManifest, LibraryIndex, SharedProjectSummary — all Codable, Sendable, stable-UUID-keyed per the architecture phase), `Services/` split into protocols plus two implementation folders (`Live/` and `Fake/`), and `ViewModels/`. The app target itself is just `Features/<ScreenName>/View.swift` files that hold no business logic, only bind to the view models. Critically, the widget extension should only ever import the tiny `SharedProjectSummary` type and a read-only `WidgetSummaryReading` protocol — never the full `Project` model — so a future schema change to notes or tracks can't silently break the widget build or force a cross-target migration.

The place I want to be most prescriptive, because it's the single biggest source of "lane A and lane B built incompatible things," is the exact public API surface of `ProjectLibrary`, `ProjectEditor`, `RecordCaptureViewModel`, and `PlaybackViewModel`, plus the explicit state enums per screen — `AddTrackState`, `RecordCaptureState`, `PlaybackState`, `ImportArrivalState`. I've written these out as full signatures in the JSON block below rather than just naming the classes, because "ProjectEditor handles track mutations" is not something two engineers will implement identically without the actual method list.

On performance and the user experience specifically: the booklet screen is a scrolling stack of cards that each carry artwork, a title/artist block, and either a waveform sliver or note-text preview — this only stays smooth if each `TrackCard` reads only its own `TrackEntry` slice (matching the state-ownership decision from design handoff) and if artwork loads through `UIImage.byPreparingThumbnail` off the main thread rather than a synchronous decode in body, per the standard SwiftUI pitfall of blocking the main thread with image work. `ForEach` over tracks must key on `TrackEntry.id`, never array index, because reorder and delete both shuffle indices and an index-keyed `ForEach` will visibly glitch or misattribute state during a drag. I want to push one concrete addition into the persistence contract that prior phases described narratively but didn't nail as an API: the debounced-write-plus-forced-flush behavior needs to live in one small utility, not be reimplemented per view model. ## Final Output

**Architecture overview.** SwiftUI + Observation, `@MainActor @Observable` classes only (`ProjectLibrary` for the shelf, one `ProjectEditor` per open project, `RecordCaptureViewModel`/`PlaybackViewModel` per active screen). Every external capability — MusicKit, audio recording/playback, file persistence, entitlements, widget summary — sits behind a protocol with a real (`Live`) and fake implementation, selected through one `AppEnvironment` composition root at launch. Nothing in a View or screen-level view model calls MusicKit, AVAudioRecorder, or FileManager directly.

**File/module layout.** One local Swift package, `LinernotesKit`, imported by three targets: the app, the widget+Live-Activity extension (one target, iOS 17+ lets WidgetKit and ActivityKit share it), and the test target. Inside the package: `Models/` (all Codable/Sendable, UUID-keyed), `Services/Protocols` + `Services/Live` + `Services/Fake`, and `ViewModels/`. The app target holds only `Features/<Screen>/View.swift` files binding to view models — no business logic in views. The app target holds only `Features/<Screen>/View.swift` files binding to view models — no business logic in views. **Data models.** `Project` (root manifest, one per project folder) owns `[TrackEntry]` and `[Contributor]`; `TrackEntry` carries snapshotted `snapshotTitle`/`snapshotArtist`/`artworkFilename` (relative path to bytes on disk, never a remote URL — satisfies the airplane-mode test) plus an optional `LinerNote`; `LinerNoteContent` is a two-case enum (`voice`/`text`) so text is a real content variant, never a fallback flag. `LibraryIndex`/`LibraryIndexEntry` is the cheap shelf-load projection so the app never decodes every full project just to render the shelf. `SharedProjectSummary` is the only type the widget ever sees. `ProjectPackageManifest` wraps a `Project` with `schemaVersion`, `packageKind` (`.keepsake`/`.invite`), `exportedAt`, and `appBuildVersion` — this exact struct backs both the Share preview and the Import Arrival read so they can never drift apart. `EntitlementSnapshot` is a cached tier + timestamp; gate decisions never make a live purchase check at the moment of action.

**Persistence strategy.** One folder per project holding the versioned JSON manifest plus artwork/audio subfolders; all writes go through `FileManagerProjectStore` (an actor) using atomic write-then-rename; `DebouncedSaveController` centralizes the ~300ms debounce plus a forced flush wired to one `scenePhase` observer at the app root and to interruption/termination paths, so the flush-on-background guarantee can't be silently skipped on one screen. Packages are never opened in place — `ProjectPackageImporting.inspect()` decodes and validates (schema version, size/file-count caps, structural checks) before `importProject()` copies anything into app storage; malformed/oversized/schema-mismatched input throws a `ProjectImportError` case and fails closed. Audio is locked to mono AAC 32–48kbps.

**Testing strategy.** Three tiers, unchanged from architecture and now given concrete seams to hang off: unit tests over pure logic (manifest Codable round-trip, `EntitlementProviding`'s boundary functions, reorder/move mutations, orphan sweep, crash-mid-write recovery); protocol-mocked tests via `FakeServices` (a fake `AudioRecording` can push `.interrupted` through its `events()` stream to drive `RecordCaptureState.interruptedRecoverable` without touching real hardware; a fake `MusicLibraryProviding` returns fixture arrays since real MusicKit is unreliable in Simulator/CI); one XCUITest golden-path smoke test gated behind a `-linernotesUITestMode` launch argument plus a fixture-name environment variable that swaps `AppEnvironment.live()` for `AppEnvironment.fake(fixture:)`.

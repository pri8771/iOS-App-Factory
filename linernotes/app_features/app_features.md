Build a mixtape gift for someone, pairing your music with short voice or text notes on why each track matters.

Real-time collaborative editing via CloudKit sharing once a backend exists.

Optional on-device speech-to-text captioning of liner notes for accessibility; not core to the pitch.

Nostalgic mixtape/cassette-inspired but premium — tactile album art, warm gradients, a liner-notes-booklet track list.

The mixtape, reinvented — music with the story behind each song, made to give.

The feature set needs to stay ruthless or this app turns into a sentimental concept deck instead of a shippable MVP. The product promise only survives if a user can move from “I want to make this for one person” to “I have a finished keepsake with real notes attached to real songs” in one sitting, without getting trapped in setup, rights confusion, or overbuilt collaboration mechanics.

My must-have set is small and all of it is directly on the promise path.

- A project creation flow that starts with occasion, recipient, and cover theme.
  User story: as a maker, I want to start with who this is for and why, so the mixtape feels like a gift instead of a playlist draft.
  Acceptance criteria: I can create a new mixtape with title, recipient name, occasion, and one selected visual theme; it persists immediately; if I relaunch, the project is still there; free users are blocked from creating a second active project with a clear upgrade explanation.

- Music selection from Apple Music/MusicKit with persistent track cards.
  User story: as a maker, I want to add songs from my library/catalog so each note attaches to a real track, not a manual text stub.
  Acceptance criteria: I can authorize music access, browse/search, add tracks, and see saved title, artist, and artwork in the project; added tracks remain visible after relaunch even if live playback later fails; free users are blocked from adding a sixth track with a clear upgrade explanation; denied/failed music access shows a real fallback state, not a dead end.

- Per-track liner notes with text and voice, where text is first-class.
  User story: as a maker, I want to explain why each song matters in the fastest comfortable format, so I don’t skip the emotional part.
  Acceptance criteria: for every track I can add either a text note or a voice note; voice recording starts quickly, shows obvious in-progress feedback, can be replayed, replaced, and saved; if mic permission is denied, text note flow remains fully usable; if recording is interrupted, I get a recoverable state and do not silently lose the rest of the project.

- Track ordering and booklet-style project review.
  User story: as a maker, I want the sequence to carry meaning, so the finished mixtape reads like a deliberate arc.
  Acceptance criteria: I can reorder tracks, review them in sequence, and see each track paired with its note in a readable project view; reorder survives relaunch; accessibility includes a non-drag reorder path.

- Finished keepsake playback mode.
  User story: as a maker or recipient, I want to experience the mixtape as a story, not a data entry form.
  Acceptance criteria: there is a distinct playback/presentation mode showing cover, sequence, current track context, and liner note content; voice notes always play locally if present; song playback attempts through MusicKit when available; unavailable songs show explicit unavailability and an Apple Music deep-link path instead of spinner failure.

- Share-sheet export and import for keepsakes and collaborator handoff.
  User story: as a maker, I want to send either a finished mixtape or a contribution invite without a server.
  Acceptance criteria: I can export a project package through the share sheet; another device/app instance can import it; the package includes metadata, artwork snapshot, track references, order, text notes, and voice notes, but not copyrighted song audio; the imported experience clearly distinguishes finished gift vs collaborator invite; malformed imports fail safely.

- Local entitlement gating for solo-only free tier.
  User story: as the business, I need the free limit to be real so paid value is not imaginary.
  Acceptance criteria: free users cannot create project 2, add track 6, or send a collaborator invite; existing over-limit content remains viewable/playable if entitlement changes downward; all blocks happen at the action point with understandable copy.

- Minimal but real widget and Live Activity support.
  User story: as a user, I want lightweight iPhone-native presence while a project is in progress or playback is active.
  Acceptance criteria: at least one working widget surfaces an in-progress mixtape and opens back into the app; at least one working Live Activity reflects current mixtape playback state; both can be minimal, but they must run as real product surfaces, not mocked screens.

The should-have set is where the app starts to feel premium rather than merely complete.

- Voice-note trimming after recording.
- A stronger cover-art/theme picker with a few distinct cassette/booklet looks.
- A recipient-first intro screen on import that explains the occasion before dumping into tracks.
- Read-only archive treatment for finished projects so the library feels curated, not like raw drafts.
- Better permission education screens before the OS prompt.
- Small playback niceties like auto-advance through the sequence and visible contributor attribution.

The could-have set is useful, but I would cut these first if they threaten completion.

- Home screen widgets in more than one size.
- Multiple premium themes beyond the minimum viable visual set.
- Progress celebration moments like “mixtape complete” transitions.
- Basic project duplication for remixing an older gift.
- Lightweight storage indicators for voice-note size.
- Trim polish like waveform scrubbing beyond simple replay/replace.

The won’t-build list needs to be explicit because these are exactly the traps.

- No cloud sync or CloudKit sharing.
- No concurrent multi-user editing or merge logic.
- No exported audio/video file containing licensed music.
- No social feed, profiles, comments, likes, or discovery surfaces.
- No speech-to-text in v1.
- No open-ended reminder/notification system.
- No support for non-Apple-Music sources in v1.
- No advanced audio editing suite.

The best direction is to design the whole app around three emotional states: making the gift, handing off the gift, and receiving the gift. If those states feel different in the UI, the app feels intentional. If everything is just one generic list/detail CRUD shell with a record button bolted on, the concept collapses even if every function technically exists.

The biggest risk in my own direction is that “minimal but real” widgets and Live Activities still consume disproportionate engineering time relative to user value. They are required by the spec, but they are also the easiest place to accidentally burn build budget while the core note-recording and import/export flows remain shaky. They should be treated as thin extensions hanging off a stable core, never as parallel product bets.

- Voice liner notes are hard-capped at 2 minutes with a visible countdown during recording — turns the spec's vague "short voice notes" into an enforceable, testable limit.
- An un-surfaced soft technical ceiling (~50 tracks/project, a few hundred projects total) protects list/scroll/persistence performance for "unlimited" paid users; it is never shown to the user as a limit, purely an engineering guardrail against pathological accounts breaking during testing or use.

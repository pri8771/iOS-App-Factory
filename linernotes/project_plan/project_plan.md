The plan should be dependency-first, not screen-first. If we let people sprint on views before the package format, local store, recorder seam, and entitlement rules are real, we will get a pretty app that loses notes, lies about sharing, or has gates that only exist in UI copy. Widgets and Live Activity should come late but not last, because they depend on the shared summary writer and can expose architectural drift early. The two places I would explicitly budget repair time are recording interruption recovery and malformed import handling, because both will take one pass to build and another to harden.

A realistic milestone set for this app is seven chunks. First, project scaffolding and configuration: targets, capabilities, UTType/document registration, App Group, local `.storekit`, privacy strings, launch seam for fake services. Second, core models and persistence: Codable schema, library index, file-store actor, atomic save path, debounced save controller, orphan cleanup, schema/version validation. Third, live and fake services: MusicKit provider, AV audio recorder/player, entitlement provider, package exporter/importer, widget summary writer, plus fakes good enough for unit tests and one smoke UI test. Seventh, native surfaces and finish: widget, Live Activity, accessibility passes, manual device verification, bug repair, and limitation review against the contract.

The ownership split should follow those dependencies. One lane owns domain plus persistence plus package validation, because the package is the trust boundary and local truth. One lane owns live/fake services, especially audio and MusicKit, because they need protocol parity for tests. One lane owns shelf/creation/authoring UI once the editor contract is stable. **Claude (Backend / Systems Engineer · Systems Thinker) — Round 1**

Coming at this as the person who has to worry about what happens when two pieces of this system touch each other under stress, my read is that the plan's shape should be driven by which lane is genuinely load-bearing for everything else, not by which lane feels most "product." The domain layer — `Project`, `TrackEntry`, `LinerNote`, the enums, `ProjectPackageManifest`, the entitlement types, and the pure `ProjectLimits`/boundary-check functions — has zero framework dependencies and should be the very first thing built and merged, with its Codable round-trip and schema-migration and entitlement-boundary unit tests green before anything else starts. Critically, UI work doesn't need to wait for either of these tracks to finish their *live* implementation — it only needs the protocol signatures, which are already frozen from tech_specs, plus working fakes, which are cheap to stand up first. I want to flag two things that didn't get named explicitly in the tech-spec phase and that I think belong in the risk register now rather than being discovered during implementation. First: the import validator checks schema version, file count, and total size, but nothing in the prior phases explicitly calls out path-traversal defense on the filenames declared inside an imported package manifest — since a `.linernotes` package is a directory with a JSON manifest naming artwork/audio filenames, a malformed or adversarial package could declare a relative path like `../../../SomewhereElse/file` and if the importer naively joins that against the app's storage root, that's a sandbox-adjacent write outside the intended project folder. `validateImportedPackage` needs an explicit "reject any declared filename containing path separators or `..` components" check, and that needs its own fuzzed test case alongside the schema-mismatch and oversized-file cases, not just an implicit assumption that "file count and size limits" cover it — they don't, since a small file at a bad path passes both checks. Second: the orphan sweep and the crash-recovery path can race each other at launch — if the sweep runs against a project whose save was interrupted mid-write and hasn't finished being reconciled yet, it could delete an asset file that a recovering project still legitimately needs, right before the manifest catches up to reference it. The sequencing needs to be explicit: full library load and any per-project recovery/reconciliation must complete before that project's orphan sweep runs, and the sweep should probably be scoped and triggered per-project after a clean save, not as one global sweep racing arbitrary app-launch timing.

The single biggest objection to this sequencing is that pulling M4 (entitlement + widget/Live-Activity plumbing) forward, ahead of most UI screens existing, means building infrastructure for features that don't have much to show yet — it'll feel like slow, unrewarding work early on when there's schedule pressure to show a working record-a-note flow. I still think it's right because the alternative — discovering App Group misconfiguration or StoreKit local-config gaps after the UI is mostly built — creates exactly the kind of late, cross-cutting rework this phase is supposed to prevent. The detail I most want preserved into task assignment: every milestone needs its own exit criteria stated as passing tests or a specific manual check, not a vague "should mostly work," because the phase rule against treating testing as end-of-project polish only holds if verification is attached milestone-by-milestone instead of batched into one late phase.

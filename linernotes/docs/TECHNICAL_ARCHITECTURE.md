# linernotes — Technical Architecture

_Deterministic render of the design/spec/planning phases' final decisions. Non-AI; missing phases are N/A, never invented._

## Design Discussion

Both agents delivered full screen-by-screen designs this round and they're describing the same app almost card-for-card, not two competing visions. Codex framed it as three emotional modes (making, handing off, receiving) that need visibly different screen treatments; Claude pushed further into concrete visual metaphor (a cassette-case shelf, Side A/B split playback, a spooling-tape record UI) and both land on identical structural decisions: shelf-style project list rather than a generic table, a ceremonial one-screen occasion-setup flow, a "liner-notes booklet" authoring screen where the note is rendered at equal visual weight to the track (not a subordinate icon), a full-screen record moment with waveform/countdown and text as an equally-sized entry point (not a smaller fallback link), reorder with both drag and explicit VoiceOver move-up/move-down actions, a distinct immersive playback mode where song-unavailable gracefully degrades to "open in Apple Music" while the note always plays, an explicit two-path share flow ("send the gift" vs "invite a contributor") with a dedicated arrival/intro screen on import, and minimal widget/Live Activity surfaces that deep-link back in without inventing new concepts. Both name the same top risk from opposite angles — Codex warns against burning build budget on decorative motion, Claude warns specifically that the cassette/tape metaphor (reels, flip transitions) is exactly the kind of thing that's fun to overbuild — and both resolve it the same way: cut decorative richness before ever letting the note/song fusion or the core recording speed degrade. Gemini was unavailable again. There's no actual disagreement to hash out; Claude's contribution sharpens Codex's structure into a specific, buildable visual language rather than proposing something different.

## Final Output

**Screen inventory**
1. **Shelf (Home)** — cassette-case-style grid/list of projects, "Make a mixtape" primary action, free-tier status chip. States: first-run empty (single sample case + one-line pitch), populated, processing-import (transient), corrupted-import error banner (non-destructive).
2. **New Occasion (creation ceremony)** — one full-screen modal: title, recipient, occasion (fixed short list, not free text), theme/shell color swatch picker. States: empty/validation (inline guidance, no alerts), free-tier block on 2nd project (contextual, occasion-framed paywall at the commit action).
3. **Authoring (liner-notes booklet)** — hero cover + occasion metadata + progress summary + stacked track cards, each showing art/title/artist/note-status at equal visual weight (waveform sliver or note-text preview inline, not an icon). Edit mode has drag handles plus custom VoiceOver "Move earlier/Move later" actions per card. States: empty (prompt to add first song), partial (mixed finished/unfinished cards, unfinished visually marked e.g. dashed outline), fully authored (primary action shifts to preview/share), read-only over-limit (lapsed entitlement), degraded-playback-available (some tracks unresolved but notes intact).
4. **Add Track (MusicKit modal)** — separate modal, not inline. States: permission-needed (explain + retry, non-dead-end), browse/search, no-results, populated results, free-tier block on 6th track at the add action.
5. **Record/Note Capture (full-screen per track)** — art context up top, large record surface with live waveform/spool visualization and 2-minute countdown ring, equally-weighted voice/text toggle. States: idle/ready, recording, interrupted-recoverable (highest-priority state to get right), recorded-preview, re-record-confirm (old file deleted immediately), text draft/saved/empty-prompt, mic-denied (defaults cleanly to text with an explanation). "Next song" action for linear momentum through the sequence.
6. **Playback (finished keepsake)** — immersive, visually distinct from authoring: full-bleed current track, warm gradient from theme, note foregrounded and synced (voice plays inline / text appears in sync), sequence tucked below. Optional Side A/Side B split as a sequencing and transition device. States: track resolves & plays, track unavailable (graceful "Open in Apple Music" swap, never a spinner), no-tracks-yet empty state.
7. **Share/Handoff** — explicit fork: "Send as gift" vs "Invite a contributor," visually distinct buttons/copy, preview of exactly what's included (cover, occasion, track count, notes included, rights-safe "songs are referenced, not bundled" line).
8. **Import Arrival** — dedicated intro screen before any track list, two variants: recipient keepsake intro (who/occasion) and contributor handoff intro (what's being asked), plus fail-closed error state for malformed/corrupt files.
9. **Widget** — in-progress project status ("3 of 5 annotated") or recent keepsake, deep-links into the app.
10. **Live Activity** — tied to mixtape playback only: current track + note progress.

**Primary user flow**
Shelf → New Occasion (title/recipient/occasion/theme) → Authoring booklet → Add Track modal (MusicKit) → returns to booklet with track landed in sequence → tap track card → Record/Note Capture (voice or text) → Next song loop until sequence feels done → reorder as needed (drag or VoiceOver actions) → Playback preview (Side A/B) → Share/Handoff (gift vs invite) → recipient/collaborator lands on Import Arrival → either plays the keepsake (Playback screen) or contributes a track+note and hands it onward.

**State model per screen**
Every screen above lists its explicit empty/loading/populated/error states; the two states called out as highest-consequence across both agents: (a) interrupted-recoverable during recording — must never silently lose a note, and (b) track-unavailable during playback — must degrade gracefully to a deep link while the note keeps playing, never a stuck spinner. Paywall states (2nd project, 6th track, collaborator invite) are contextual sheets triggered at the exact action point, not a separate settings-style paywall screen.

**Visual direction**
Warm, tactile, premium-nostalgic: paper grain, softened shadows, cassette-label typography for track/song metadata (compact/mechanical) contrasted against editorial typography for liner-note text, amber/faded-red/tobacco-brown/cream/charcoal palette. Small intentional motion only — cards settling into order, waveform "breathing" while recording, a tape-transport-style transition between playback sides — with reduced-motion fallbacks, and explicitly no decorative motion allowed to slow down the record-button's <300ms start time. The cassette/tape metaphor (spooling reel, flip transitions) is approved as the visual language but flagged as the top over-build risk — cut decorative richness first if schedule pressure hits, never the note/song fusion or recording responsiveness.

**Accessibility notes**
Non-drag reorder via explicit custom VoiceOver actions ("Move earlier/Move later") on every track card. Every track card exposes note status as a spoken label ("Voice note recorded" / "No note yet"), not icon-only. Recording screen exposes time-remaining, recording state, and save state to VoiceOver. Dynamic Type must reflow the booklet layout rather than truncate liner-note text. Theme contrast must stay readable within the warm/nostalgic palette. Keepsake playback must remain fully understandable via note audio/text alone when zero referenced songs resolve for a recipient — artwork alone is never sufficient context.

## Design Handoff

Both of them landed on the same app design almost exactly — Codex wrote the full screen-by-screen spec with tokens, components, motion, and accessibility, and Claude (playing skeptical frontend engineer this round) didn't contest any of the visual/IA decisions but instead pressure-tested how they'd actually get built: treat the spooling-reel/waveform as a cheap amplitude-meter skin (not the baseline), scope state management so editing one track card doesn't re-render the whole booklet, make "snapshot artwork" mean literal bytes on disk (not a cached URL, which would silently fail the airplane-mode test), nail down debounced-autosave-plus-forced-flush-on-background as the real mechanism behind "autosave," and flag that Widgets/Live Activities need an App Group decided now, not bolted on later. That's worth resolving now rather than kicking to another round, since it's a scope call, not a contradiction — I'm resolving it below in favor of the narrower ask, with Codex's fuller prompt kept as an optional fallback if broader visual exploration is wanted later.

## Final Output

### Screen-by-screen specification

**1. Shelf (Home)** — root of a single `NavigationStack`, no tab bar. Vertically scrolling shelf of cassette-case cards (2-up on standard widths, 1-up if Dynamic Type forces it). Top bar: wordmark, small entitlement chip, one dominant "Make a mixtape" CTA. Each card: cover art, title, recipient, occasion, count like "3 of 5 annotated" or "Finished." States: first-run empty (single sample cassette + one-line pitch), populated, transient import-processing banner/card, non-destructive corrupted-import error card with retry/dismiss. Motion: subtle lift/settle on tap only.

**2. New Occasion** — presented as a sheet, not a push. Order: title → recipient → occasion (fixed chip set: Birthday, Goodbye, Anniversary, New Baby, Just Because) → optional dedication → theme swatch picker (tactile cassette-shell colors, instant preview). Primary CTA reads "Start Side A." Inline validation only, no alerts. Free-tier block on 2nd project appears as a contextual paywall sheet triggered at submit, framed around "make another gift."

**3. Authoring (liner-notes booklet)** — pushed from Shelf. Hero cover block, recipient/occasion metadata, progress summary, then a `LazyVStack` of track cards for cell-reuse efficiency. Each TrackCard is one component with three interchangeable content states in its note-status region: compact waveform sliver, 2-line note-text preview, or dashed "no note yet" placeholder — same visual weight as the song metadata, never a small icon. Edit mode: drag handles plus explicit "Move earlier/Move later" actions calling the same underlying reorder function. States: empty, mixed complete/incomplete, fully authored (primary action shifts to preview/share), read-only over-limit (lapsed entitlement), degraded-playback-available (notes intact, songs unresolved).

**4. Add Track** — sheet, MusicKit-only source. Local state machine: `.authorizing → .denied | .ready`, then `.idle → .searching → .results | .empty`. Denied state explains and offers retry, never a dead end. Selecting a track snaps back to the booklet with a visible insertion animation. Free-tier block fires exactly on the 6th track, at the add action.

**5. Record/Note Capture** — `fullScreenCover`, one track at a time. Art/title/artist/recipient crumb up top. Center: record surface with a live amplitude indicator and a 2-minute countdown ring — built first as a cheap `AVAudioRecorder.updateMeters`-driven bar/ring at ~10–15Hz; the "spooling reel" look is a visual skin over that same data feed, swappable out with zero rewrite if it costs recording responsiveness. Equal-sized Voice/Text toggle, text never secondary. Explicit states: `idle, recording, interruptedRecoverable, recordedPreview, reRecordConfirm, textDraft, textSaved, micDenied`. "Next Song" action after save for linear momentum. Re-record deletes the old file from disk immediately, not just unlinks it.

**6. Playback (finished keepsake)** — visually distinct from authoring: full-bleed current-track art, warm theme-keyed gradient, note foregrounded and synced (voice inline / text in sync with position), sequence tucked below. Side A/B presented as a simple paged transition (`TabView(.page)` or lightweight custom swipe) — not a physics-heavy flip unless proven cheap in practice. Per-track resolved/unavailable state: unavailable swaps cleanly to "Open in Apple Music," never a spinner. No-tracks-yet empty state.

**7. Share/Handoff** — sheet, explicit two-button fork: "Send the Gift" vs. "Invite a Contributor," visually distinct copy and treatment. Manifest preview (cover, occasion, track count, notes included, rights-safe "songs are referenced, not bundled" line) is rendered from the same package-building code that produces the actual export, so preview can never drift from reality. Contributor path states one-at-a-time editing plainly.

**8. Import Arrival** — reached via a declared UTType/document-opener capability (real Info.plist/entitlement work, not just a screen). Two content variants keyed on a `packageKind: .keepsake | .invite` field set at export time; fail-closed error state on any decode/schema-version mismatch, calm copy, no crash.

**9. Widget** — reads a small separate shared-summary file (title, recipient, annotated/total count, theme color) via App Group, not full project JSON. Shows in-progress status or a recent finished keepsake; deep-links in.

**10. Live Activity** — tied only to mixtape playback: current track + note progress, sourced from the same shared summary mechanism.

### Design tokens
Semantic color roles (never raw hexes in component code): `surfacePaper` (cream base), `surfaceCharcoal` (playback contrast), `accentAmber` / `accentFadedRed` / `accentTobacco` (theme choices), `accentBrass`, `success` (mossed green), `error` (rust red), `recordActive` (reserved solely for recording state, never reused decoratively elsewhere). Typography: `textEditorial` (literary, for liner-note copy) vs. `textMechanical` (compact, label-like, for track metadata) as distinct styles even if same underlying font family at different weight/tracking. Spacing/radius: fixed scale 4/8/12/16/24; corners medium/soft ("booklet," not pill-shaped, which reads as generic-app). Motion budget: small and intentional only (card settle, waveform breathing, side transition), with an explicit Reduced Motion fallback for every one of them, and a hard rule that no decorative motion may add latency to record-start.

### Component inventory
TrackCard (single component, three note-status content states), RecordButton (idle/recording/disabled + countdown ring), AmplitudeIndicator (shared between live recording and any played-back preview), ThemeSwatchPicker, PaywallSheet (one component parameterized by which of the three gates fired), PackagePreviewManifest (shared, read-only, between Share and Import Arrival), EmptyStateCard (reused across Shelf/Authoring/Add-Track), OccasionChip, ContributorBadge, RightsSafeInfoRow, ImportStatusBanner, SideMarker, CompactPlaybackStrip.

### State and interaction notes
Navigation: single `NavigationStack` rooted at Shelf; Add Track and Record present as sheet/fullScreenCover respectively — genuine modal moments, not stack pushes. Persistence: debounced writes (~250–400ms after last mutation) plus a forced synchronous flush on `scenePhase` → `.background`/`.inactive` and on interruption/termination paths, using atomic write-then-rename so a crash mid-write never produces a file the app can't reopen itself. Artwork/metadata snapshot means fetching and writing actual image bytes to the project's local asset folder at add-time (with its own loading state), never a cached remote URL. State ownership: one `@Observable ProjectLibrary` for the Shelf; each opened project gets its own `@Observable ProjectEditor`; individual track cards read only their own slice of state so editing/re-recording one card doesn't force a diff pass across the whole booklet. Accessibility actions ("Move earlier/Move later") call the identical function the drag gesture calls — one mutation path, two input methods.

### Accessibility requirements
Every TrackCard exposes one combined accessibility label (song, artist, note status spoken as "Voice note recorded"/"Text note: [preview]"/"No note yet", contributor, available actions) rather than child elements. Non-drag reorder via custom VoiceOver actions is mandatory, not optional. Record screen posts live `accessibilityValue`/announcements for elapsed/remaining time and state changes (recording started/stopped/interrupted), since the countdown ring alone is invisible to VoiceOver. Dynamic Type must reflow the booklet and playback screens — truncating liner-note text is not acceptable. Reduced Motion disables the spool/flip flourishes via `@Environment(\.accessibilityReduceMotion)` while preserving underlying structure. Playback must remain fully legible on note audio/text alone when zero songs resolve for a recipient.

### External design decision
Scope any external Claude Design pass narrowly rather than commissioning a full 10-screen mockup set: the hard problems here (interrupted recording, reorder accessibility, degraded playback) are interaction/state-machine problems that get validated by building real SwiftUI, not by static comps. If external design is used, this is the prompt:

> "Design 3–4 cassette-shell cover themes (amber, faded red, tobacco brown, plus one dark/charcoal variant) as flat color-and-texture swatches usable as a SwiftUI gradient+grain background, and one full-bleed immersive now-playing screen mockup showing album art, a synced liner-note text block, and a Side A/B indicator — warm paper-grain premium aesthetic, no skeuomorphic tape gimmicks."

Codex's fuller 10-screen prompt (covering Shelf, New Occasion, Authoring, Add Track, Record Capture, Playback, Share/Handoff, Import Arrival, Widget, and Live Activity concepts, in the same warm/tactile/premium direction) is preserved as an optional fallback if broader upfront visual exploration is wanted later — but it is not the recommended default, since it risks producing polished screens disconnected from the real interaction model.

**Top risk, named once more so it survives into build:** the cassette/tape metaphor (spooling reel, Side A/B flip) is real visual language, approved, but every instance of it must be built as a skin over a plain, cheap, accessible baseline — never the baseline itself — so cutting it under schedule pressure is a one-line config change, not a rewrite. The non-negotiable invariant across every screen: the note and the song are rendered as one fused object, equal size, equal moment — the instant they visually separate into "song info" plus "attached comment," the product flattens into a playlist app with voice memos stapled on.

## Final Output

Both Codex and Claude landed on the same architecture, with Claude (this round playing pragmatist/QA) mostly turning Codex's structural recommendations into concrete, nameable implementation choices rather than disagreeing with anything. Gemini was unavailable again.

**SwiftUI architecture:** `@Observable`, `@MainActor` view models — one `ProjectLibrary` for the shelf, one `ProjectEditor` per open project, individual track cards reading only their own state slice (carried forward from design handoff). The one hard addition this phase locks in: every external dependency (MusicKit, audio recording, file storage, entitlements) sits behind a narrow protocol from day one — `MusicLibraryProviding`, `AudioRecording`, `ProjectFileStore`, `EntitlementProviding` — with real + fake implementations from the start. Nothing calls MusicKit, AVAudioRecorder, or FileManager directly from a View or screen-level view model. Both agents agree this isn't purism — it's the only way to actually test interruption-recovery and malformed-import handling, which is most of what this app is graded on.

**Apple framework choices:** SwiftUI + Observation; MusicKit + `MusicAuthorization` for catalog/library; AVFoundation (`AVAudioSession`, `AVAudioRecorder`, `AVAudioPlayer`) for note capture/playback, with `AVAudioSession.interruptionNotification` as the actual mechanism behind "interrupted-recoverable" (not custom polling); FileManager + UniformTypeIdentifiers for the project package and document-opener capability; WidgetKit + ActivityKit against a tiny shared App-Group summary file, never full project state; StoreKit 2 for entitlement, built against a local `.storekit` config so the gate is fully testable in-simulator — live purchase wiring to a real product ID is explicitly out of scope per the production-readiness contract, only the gating logic itself is must-have.

**Persistence:** Rejected SwiftData in favor of a plain FileManager-based store — one folder per project holding a versioned Codable JSON manifest plus artwork/audio subfolders, atomic write-then-rename, debounced writes with forced flush on background/interruption. The shareable package is just that same project folder wrapped as a declared `.linernotes` UTType — no custom zip/archive code, no third-party dependency needed. Imported packages are never opened in place: decode into memory, validate schema version/size/type, then copy into app storage or reject. New concrete decision: audio locked to mono AAC at 32-48kbps now, to keep a 2-minute note under ~1MB and avoid a later storage-bloat surprise.

**Permissions/privacy:** Only two real permissions — Apple Music and microphone — with specific plain-language usage strings already decided in earlier phases. No Background Modes audio capability: backgrounding mid-recording is treated as just another interruption (stop, save draft, recover on return), deliberately avoiding the App Review/complexity cost of trying to keep recording alive in the background. Entitlements needed: MusicKit capability, App Groups, and the exported UTType/Info.plist document-type registration so share-sheet import actually works — flagged as an easy-to-forget config step that would silently break the entire receiving half of the core loop if skipped.

**Dependency/license decision:** Effectively zero third-party dependencies; both agents agree Apple's first-party frameworks cover everything needed given the file-package approach to export, and any dependency would add legal/audit risk disproportionate to the gap it closes.

**Testing implications:** Three explicit tiers — (1) unit tests on pure logic: Codable round-trip of the versioned schema, entitlement boundary conditions, reorder mutation, orphan-sweep, crash-mid-write recovery; (2) protocol-mocked tests using fake `AudioRecording`/`MusicLibraryProviding` to synthesize interruptions and catalog responses without real hardware or a signed-in Apple Music account (real MusicKit is unreliable in Simulator/CI); (3) one XCUITest golden-path smoke test gated behind a launch-argument seam that swaps in fake providers. Real permission-denial prompts, an actual phone call interrupting an actual recording, and cross-device package handoff are named explicitly as manual/device-only checks, not something to pretend is covered by automation.

No open disagreement to carry into another round — this phase is ready to close.

## Final Output

**Architecture overview.** SwiftUI + Observation, `@MainActor @Observable` classes only (`ProjectLibrary` for the shelf, one `ProjectEditor` per open project, `RecordCaptureViewModel`/`PlaybackViewModel` per active screen). Every external capability — MusicKit, audio recording/playback, file persistence, entitlements, widget summary — sits behind a protocol with a real (`Live`) and fake implementation, selected through one `AppEnvironment` composition root at launch. Nothing in a View or screen-level view model calls MusicKit, AVAudioRecorder, or FileManager directly.

**File/module layout.** One local Swift package, `LinernotesKit`, imported by three targets: the app, the widget+Live-Activity extension (one target, iOS 17+ lets WidgetKit and ActivityKit share it), and the test target. Inside the package: `Models/` (all Codable/Sendable, UUID-keyed), `Services/Protocols` + `Services/Live` + `Services/Fake`, and `ViewModels/`. The app target holds only `Features/<Screen>/View.swift` files binding to view models — no business logic in views. The widget extension imports only `SharedProjectSummary` and a read-only summary-reading protocol, never the full `Project` model, so a notes/track schema change can't break the widget build.

**Data models.** `Project` (root manifest, one per project folder) owns `[TrackEntry]` and `[Contributor]`; `TrackEntry` carries snapshotted `snapshotTitle`/`snapshotArtist`/`artworkFilename` (relative path to bytes on disk, never a remote URL — satisfies the airplane-mode test) plus an optional `LinerNote`; `LinerNoteContent` is a two-case enum (`voice`/`text`) so text is a real content variant, never a fallback flag. `LibraryIndex`/`LibraryIndexEntry` is the cheap shelf-load projection so the app never decodes every full project just to render the shelf. `SharedProjectSummary` is the only type the widget ever sees. `ProjectPackageManifest` wraps a `Project` with `schemaVersion`, `packageKind` (`.keepsake`/`.invite`), `exportedAt`, and `appBuildVersion` — this exact struct backs both the Share preview and the Import Arrival read so they can never drift apart. `EntitlementSnapshot` is a cached tier + timestamp; gate decisions never make a live purchase check at the moment of action.

**Persistence strategy.** One folder per project holding the versioned JSON manifest plus artwork/audio subfolders; all writes go through `FileManagerProjectStore` (an actor) using atomic write-then-rename; `DebouncedSaveController` centralizes the ~300ms debounce plus a forced flush wired to one `scenePhase` observer at the app root and to interruption/termination paths, so the flush-on-background guarantee can't be silently skipped on one screen. Packages are never opened in place — `ProjectPackageImporting.inspect()` decodes and validates (schema version, size/file-count caps, structural checks) before `importProject()` copies anything into app storage; malformed/oversized/schema-mismatched input throws a `ProjectImportError` case and fails closed. Audio is locked to mono AAC 32–48kbps.

**Testing strategy.** Three tiers, unchanged from architecture and now given concrete seams to hang off: unit tests over pure logic (manifest Codable round-trip, `EntitlementProviding`'s boundary functions, reorder/move mutations, orphan sweep, crash-mid-write recovery); protocol-mocked tests via `FakeServices` (a fake `AudioRecording` can push `.interrupted` through its `events()` stream to drive `RecordCaptureState.interruptedRecoverable` without touching real hardware; a fake `MusicLibraryProviding` returns fixture arrays since real MusicKit is unreliable in Simulator/CI); one XCUITest golden-path smoke test gated behind a `-linernotesUITestMode` launch argument plus a fixture-name environment variable that swaps `AppEnvironment.live()` for `AppEnvironment.fake(fixture:)`.

**Claude (QA & Risk · Systems Thinker) — Round 1**

The thing I keep coming back to is that this app's entire premise is "trust this number," and the fastest way to break that trust is a math edge case nobody thought about until a real user hit it. So let me walk through where I think the actual landmines are, beyond what's already been decided.

Start with arithmetic, because this is an accounting app and the math has to be bulletproof. If a batch produces zero units — user typos "0" or leaves the yield field blank before the app validates — the per-unit cost calculation divides by zero. That has to be blocked at input time, not caught downstream as a crash or a silent "$inf" render. Similarly the suggested-price formula, cost ÷ (1 − target margin), blows up at exactly 100% margin and goes negative above it — if the UI lets someone drag a margin slider to 100%, the formula produces infinity or a negative "minimum price," which for a product whose entire value proposition is a trustworthy number is a much worse failure than a generic app bug. The margin input needs a hard ceiling (something like 0–95%) with the boundary enforced, not just visually suggested.

Second, currency math itself: this needs to be `Decimal`, not `Double`, for every cost field. The success criteria from the earlier phase explicitly demands "identical inputs always produce identical outputs, no rounding drift across library → recipe → batch → price" — that's precisely the scenario where floating-point binary representation error shows up, especially once you're summing many small material costs (spec itself names "many small inputs" as a UX risk, and small values summed in Double are exactly where drift accumulates). This should be locked as a technical requirement now, not left for architecture to discover the hard way.

Third — and I think this is a real gap in what's been decided so far — the labor rate itself has an ownership question nobody's resolved: is it one global rate the user sets once, or a per-product or per-batch override? If it's global and mutable, then when it changes, does that retroactively affect anything, or does every batch snapshot the rate that was active at the time it was logged? I'd assume the latter (consistent with the snapshot decision already made for material costs), but if that's not written down explicitly, someone will "helpfully" wire it up as a live lookup and quietly corrupt every historical batch's cost the next time the user updates their hourly rate.

Fourth, the material library creates a subtle two-clock problem. A recipe references materials by definition (quantities), and the library holds current unit costs. When a user logs a new batch, do they see the material's *current* library cost, or the cost as of when the recipe was authored? I'd say current cost at batch-log time, snapshotted into that batch permanently — but that means editing a material's library price should visibly ripple forward (new batches use new price) while never touching old batches. If a material gets deleted from the library entirely, existing recipes and past batches must not break or lose their historical numbers — which means recipes/batches can't hold a live foreign-key-only reference, they need to copy the material's name/cost into the recipe or batch snapshot at time of use, with the library entry itself being just a "starting point" convenience, not the source of truth for anything already committed.

Fifth, real data-loss risk that I don't think has been named yet: this is a fully local, no-backend app, meaning if a user gets a new phone, deletes the app, or their device is lost, every product, recipe, batch, and profit history is gone — permanently, with no recovery path, even for a paying subscriber. StoreKit subscriptions restore fine on a new device, but the underlying business data does not, because there's no export/backup mechanism in v1's scope. That's a serious mismatch: someone can pay $44/year for a year, switch phones, and lose everything except the ability to keep paying. I don't think this blocks v1, but it needs to be an explicit, named limitation, and I'd push hard for standard iOS device/iCloud backup inclusion (i.e., don't exclude the SwiftData store from backup) as the bare-minimum mitigation, since that's free and already-decided default behavior, not a new feature.

Sixth, the widget and the main app are different processes reading what has to be a shared App Group container — that's a real architectural constraint, not a detail: SwiftData store location, or a lightweight shared snapshot file the widget reads, has to be decided, and the widget needs its own timeline-reload trigger whenever a batch is logged (WidgetCenter.reloadTimelines()), otherwise it'll show stale margins indefinitely, which is its own trust failure for an app about accurate numbers. And the widget needs its own compressed version of the labor-not-included flag — in a small widget you can't show a paragraph disclaimer, so there needs to be a defined compact treatment (an icon or a distinct visual style) for "this margin doesn't include labor," not just an omission of the warning because there's no room.

Seventh, share-sheet export: "render a priced product card" implies a concrete visual artifact (image, most likely), and that rendering path has to independently handle Dynamic Type extremes — someone at accessibility XXL text size shouldn't get a shareable card with text overflowing its bounds, since that's a fixed-layout export, not a scrolling view. This needs its own layout contract separate from the live in-app screen.

Eighth, cascading deletes: deleting a product needs an explicit rule for what happens to its recipes, batches, and historical trend data — cascade-delete, or soft-archive. Given the profit-trend and "why paid users see business intelligence across products" framing, I'd lean toward cascade-delete being fine for a deleted product (it's gone, that's the user's call) but this has to be a deliberate SwiftData relationship rule, not whatever the default happens to be, since an accidental orphaned-batch-with-no-parent-product state would corrupt the trend chart.

Ninth, on the already-flagged-but-still-open free-tier downgrade question: I want to push toward a concrete answer rather than leaving it open again — data retained, read-only, for every product beyond the first, with a clear "resubscribe to edit/add" state, because silently deleting a small business owner's cost data on a lapsed payment is the kind of thing that generates one-star reviews and support emails, and it's also just unnecessary; storage is cheap, and read-only lockout preserves trust better than deletion.

Tenth, smaller but real: a recipe with zero materials (labor-only) should trigger the same kind of visible "materials not included" flag that the labor-zero case already gets — the honesty principle needs to be symmetric, not just protective of the labor half of the equation, or a materials-only cost gets treated as suspect while a labor-only cost sails through unflagged.

On privacy: since the actual pitch to the user is "your margins live on your phone, nobody else sees them," that has to be true in the boring literal sense too — no analytics SDK, no crash reporter that transmits cost/pricing data even in aggregate, nothing that quietly contradicts the trust story the product brief leans on. If crash reporting exists at all, it must be scoped to exclude business data entirely.

Final assumptions I'd bake in rather than leave open: Decimal for all currency math; labor rate and material costs both snapshot into the batch at creation time and never retroactively recompute; margin input capped below 100% with the boundary enforced in the UI, not just documented; deleted products cascade-delete their batches/recipes; free-tier downgrade results in read-only lockout of extra products, not deletion; standard device backup is not excluded (that's the only backstop against total data loss in a no-cloud v1); and the widget requires its own explicit reload trigger and its own compact "incomplete cost" indicator design, not a shrunk copy of the in-app one.

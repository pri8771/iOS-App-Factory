Since 1 tracked product is free, the price you calculate for that one product still has to be trustworthy enough on its own to convince a skeptical maker to pay $5.99/mo just to add a second one — so the whole business model rides on the free tier's single calculation being airtight, not on the paid tier's extra features.

The direction I'd push: the smallest thing that actually ships here is one product, one recipe, N materials, batches with labor time, a suggested price, and a trend chart with at least 2-3 data points — and every one of those pieces has to work correctly on day one, because there's no "core feature" to fall back on if the pricing math is wrong. This isn't an app where you can ship a rough v1 and patch the calculation later; if the number is wrong even once, the user's mental model becomes "this app lies to me" and they uninstall, because the entire promise is trustworthiness of a number, not feature breadth.

The biggest risk I see, and it's not really about scope, it's about definition: the prompt asks for a "one-sentence product promise" and "measurable success criteria," but this product's core promise only works if we define what "true cost" precisely means in numeric terms before writing a line of UI code. Concretely: true unit cost = (sum of material costs allocated to the batch ÷ units produced) + (labor hours × hourly rate ÷ units produced). That's simple. But "measurable success criteria" for an app like this can't just be "user completes the workflow" — it has to be "the calculated cost is internally consistent and reproducible," meaning if I enter the same batch data twice I get the same number, and if I change one input (say, hourly rate) only the labor term moves, nothing else drifts. That sounds trivially obvious but it's exactly the kind of thing that breaks silently when someone snapshot-vs-live-recomputes carelessly, or rounds intermediate values instead of the final result, or double-counts a material shared across two products. I'd want "no silent rounding drift across the material library → recipe → batch → suggested price chain" written down as an actual acceptance check, not left implicit.

The other failure mode I'd flag for this phase specifically (not architecture, but scope-definition): the target user is described as someone who "prices intuitively" and has "never calculated true per-unit cost" — that's a user who is going to be intimidated by a spreadsheet-like entry form. If the core loop requires them to pre-populate a material cost library with unit-price and unit-conversion data before they can log their first batch, most of them will bounce before reaching the "aha" moment (the suggested price). So the core loop for this phase needs to explicitly sequence: user should be able to get from "open app" to "see a suggested price" in under maybe 5 inputs for their very first product, even if that first pass is rough (single unit, no conversion, no library reuse yet), with the reusable-material-library value being something they discover on their second product, not a prerequisite for their first. If we don't write that ordering down now, design will default to "build the data model first, screens follow the model," which produces a form-heavy onboarding that kills exactly the user this promise targets.

For hard scope boundaries at this phase-of-discussion level (product promise, not architecture): I'd explicitly exclude any notion of "categories" or "tags" for materials/products, exclude any batch-editing-after-the-fact question (a batch's snapshot is either locked or it isn't — don't let "can I fix a typo in a past batch" become a mutable-vs-immutable debate that spills into every downstream calculation), and exclude the profit-trend chart from needing more than a bare-minimum "cost per unit over time" line — no need for multi-metric dashboards, that's feature creep dressed up as "reporting."

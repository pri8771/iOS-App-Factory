**Claude (QA & Risk · Closer) — Round 1**

I want to close out the architecture-level gaps before this goes to tech_specs, because a few things that read as "obviously fine" in the design_handoff phase are actually concrete iOS platform decisions nobody has pinned down yet, and I'd rather lock them now than have tech_specs discover them mid-implementation.

Start with the SwiftUI architecture recommendation, which I think is basically already right and I want to ratify rather than reopen: MV, not MVVM — `@Query` directly in views backed by the pure `RunwayEngine.project(...)` function already locked in design_handoff, no view-model layer in between. That's the correct call for an app this size, and adding an MVVM layer here would be complexity for its own sake. One concrete addition though: the `@Query` filters for Bill and Paycheck need to use the `#Predicate` macro to exclude soft-deleted rows at the fetch layer — `@Query(filter: #Predicate<Bill> { $0.deletedAt == nil })` — not fetched-then-filtered-in-Swift, because that's both simpler and means a stale soft-deleted row can never leak into the projection engine by accident.

The single most dangerous unstated engineering decision in this whole app is date arithmetic, and I want to make it explicit and non-negotiable: all cadence math — biweekly, semimonthly, monthly generation, "days until" calculations — has to go through `Calendar.date(byAdding:to:)` using the device's autoupdating current calendar, never raw `TimeInterval`/epoch-second arithmetic. Adding `86400 * 14` seconds to compute "two weeks from now" silently breaks the day a DST transition sits inside that window — the user gets a paycheck that's off by an hour's worth of a day, which sounds trivial until it's the difference between a bill landing before or after a payday in the projection. This has been named abstractly as a risk since the prompt contract, but nobody has said "use Calendar, not TimeInterval" as an actual engineering rule, so I'm saying it now: that's the rule, and it needs to show up as a code-review-blocking constraint in tech_specs, not just a test case.

On Apple framework choices: SwiftData for persistence, Observation for state, LocalAuthentication only if the should-have Face ID/passcode lock gets built, and that's it — no WidgetKit target, no StoreKit imports anywhere in this codebase, no CloudKit. I want to be blunt about StoreKit specifically because I think there's a real temptation trap in the Premium stub: someone building this is going to be tempted to wire up a `.storekit` configuration file and a real (if sandboxed) purchase button "since it's easy," and that is scope creep past what next_steps_small and app_features already decided. The Premium screen is static text, zero StoreKit import, full stop — if that changes, it's a new decision requiring explicit sign-off, not something that drifts in because it was convenient to add.

Persistence plan, concretely: `ModelConfiguration` targeting the App Group container (`group.<bundle-id>.shared`, decide the literal string now so it's not improvised at build time), `cloudKitDatabase: .none` since there's no sync this pass. The one thing I want to add on top of what's already locked: define a `VersionedSchema` and an (even if trivial, single-version) `SchemaMigrationPlan` from day one, not just a bare `@Model` schema. This is the same argument that already won the App Group debate — cheap now, and it means the first time this app's data model actually changes (adding the widget, adding what-if scenarios for Premium, adding CloudKit), there's already a migration plan shape to extend instead of a bare schema that's never been versioned and needs to be retrofitted under pressure with real user data on the line. I'd also want the `ModelContainer` construction to live in one shared, static factory function (not inlined in the App struct) specifically so a future widget extension target can call the identical factory and guarantee it's pointed at the same container with the same schema — that's the actual mechanism that makes "widget-ready" true rather than aspirational.

Permissions and privacy: this app requests exactly one permission, conditionally — Face ID, only if the should-have lock ships, with an `NSFaceIDUsageDescription` string that's honest and calm ("Used to keep your financial information private"), and a graceful path when biometrics aren't enrolled on the device — the toggle should just not be offered rather than erroring, and Settings shouldn't dead-end into a broken state. No camera, no location, no contacts, no push notification permission, no network permission dialog because there's no network entitlement at all. One concrete deliverable I don't think anyone's named yet: Apple now requires a `PrivacyInfo.xcprivacy` manifest at submission time, and this app's is genuinely easy to get right — zero collected data types, and if UserDefaults is used anywhere (onboarding-complete flag, appearance override), the required-reason API declaration for UserDefaults access needs to be in that manifest or the build gets flagged in App Store Connect. I want this named now as a build-phase checklist item, not discovered at submission time.

Dependency and license decision: zero third-party dependencies, Swift Package Manager unused entirely for this pass. I'll flag the one place this could be quietly violated: someone reaching for a snapshot-testing library (like swift-snapshot-testing) to verify the five home states visually. I don't think that's needed — Xcode's built-in `XCTAttachment`-based screenshot capture inside XCUITest, plus SwiftUI Previews for manual state verification, covers this without adding a dependency, and I'd close the door on it explicitly rather than let it sneak in as "just a test-only dependency."

Testing implications, and this is where I want to be a closer rather than gesture at "unit tests required": the `RunwayEngine` test suite needs to explicitly enumerate, as separate test cases, every edge case already named in detailed_discussion — semimonthly cadence with the 15th on a weekend, semimonthly with the 1st falling on Feb 29/30/31, biweekly generation hitting the two-year horizon cap and stopping, a bill dated the 31st clamped into a 30-day month, a paycheck dated the 31st clamped the same way, the same-day-bill-already-subtracted rule, negative/zero rejection at the model layer, and the bills-only-no-income state producing the correct honest projection. If tech_specs doesn't carry this exact list forward as named test cases, it'll get built as "some tests" and that's not good enough for the one part of this app that isn't allowed to be wrong. Alongside that, SwiftData persistence itself needs an in-memory `ModelConfiguration` test harness (`isStoredInMemoryOnly: true`) so persistence and migration logic are testable without touching disk. And I want to flag a real testability gotcha for whoever builds the Face ID should-have: LocalAuthentication doesn't have enrolled biometrics on a fresh simulator by default, and testing it requires enabling Face ID/Touch ID simulation through the simulator's Features menu — that's not obvious, and a builder who doesn't know it will think the feature is broken when it's actually a simulator configuration step.

Last thing, stated as an explicit decision so the acceptance check ("any ML, LLM, or AR use is justified by user value") isn't left ambiguous: this app uses none of those, and that's correct, not a gap. Nothing about a calm, single-number cash-flow answer benefits from ML or AR, and forcing one in would violate the "complexity must serve real user value" principle from the very top of this whole project. Same for background work — no `BGTaskScheduler`, no background refresh, no push registration — there is nothing that needs to run when the app isn't open, since there are no notifications and no sync in this pass.

The one place I'd push back on my own instinct here: the versioned-schema-and-migration-plan requirement might look like premature engineering for an MVP that's explicitly supposed to be small. I don't think it is, for the same reason the App Group decision already isn't — a schema migration on a device holding someone's real financial data, done for the first time under a future feature-launch deadline, is a much worse time to learn the migration APIs than now, before any user data exists. If this phase is going to draw one hard line, that's the one I'd draw: App Group container, versioned schema, and Calendar-based (never epoch-based) date math are non-negotiable architecture requirements for tech_specs to carry forward as-is, not something to be simplified away if the build phase feels time pressure.

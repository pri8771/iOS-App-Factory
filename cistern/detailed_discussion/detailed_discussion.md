**Claude (QA & Risk · Skeptic) — Round 1**

Picking up where the prior phases left off — the cash-flow math correctness bar and staleness problem have been named again and again, but nobody has actually enumerated the specific scenarios that will break a naive implementation, and nobody has touched privacy at the OS-surface level at all. Those are the two things I want to push on hardest this round.

On the math: "handle month-end and DST" is not a spec, it's a wish. Let me name the actual cases someone has to write logic (and ideally tests) for. Semimonthly cadence (1st and 15th) needs a decision for what happens when the 15th falls on a weekend or the 1st falls on Feb 29/30/31 — does payday shift, or stay calendar-fixed with the user expected to know their bank posts early? Biweekly cadence drifts across a year (26 vs 27 pay periods), so a naive "add 14 days forever" loop needs a hard cap or a sane generation horizon, or it'll happily compute a runway 40 years out the first time someone fat-fingers a start date. A bill due on the 31st in a 30-day or 28/29-day month was already flagged, but the same clamp logic needs to apply to paycheck dates too, not just bills — nobody's said that explicitly. Then there's the "today" boundary: if a bill is due today, is it already subtracted from the runway or not yet? This is a classic off-by-one, and I think the correct call for this app specifically is to bias conservative — treat a same-day bill as already gone from the number — because the entire tonal contract here is "never tell this user they're fine when they're not," and an optimistic off-by-one is exactly the kind of bug that produces a confidently wrong "you're okay" on the one day they're not. Also: what does the runway say when the user has bills but zero paychecks scheduled? That's not really covered by the "empty state" language used so far — that's a partial-data state, distinct from zero-data, and the honest answer is the projection should run bills-only against the anchor with no assumed future income, and say so, rather than silently pretending a paycheck is coming.

Negative and malformed input is another gap: nothing has said what happens if someone enters a negative bill amount, a $0 paycheck, or a wildly implausible number (a $50,000 bill, probably a typo for $500). I'd disallow negative/zero amounts outright at the input layer rather than defend against them downstream, and I'd think about a soft "that's a big number, just checking" confirmation for outlier magnitudes — not a hard block, because this app's whole ethos is not treating the user like they need supervision, but a fat-finger of a decimal point is a realistic, damaging failure mode for exactly this financial situation.

Now the thing I think is a genuine blind spot across every phase so far: privacy at the OS level, not just "no bank linking." This app's entire home screen is a sentence like "you'll be tight starting Tuesday" sitting in plain text. Nobody has addressed the iOS app-switcher snapshot — by default, that sentence is what shows up as the thumbnail when someone swipes through open apps, which is a real disclosure risk for a user who is, by definition, anxious about their financial situation being seen. I think this needs an explicit decision: obscure or blur sensitive content in the app-switcher snapshot (a cover view swapped in via scenePhase, or `.privacySensitive()` where applicable), and I'd also put on the table — as an optional, not required, free-tier toggle — a Face ID/passcode gate before the app opens. That's cheap to build, respects the exact anxiety this app is designed around, and costs nothing to keep in the free tier since it's a trust feature, not a value-add feature.

Related, and this one's an actual architecture decision hiding as a privacy footnote: should the SwiftData store be excluded from iCloud device backup? My instinct is no — leave it included in the default device backup, because for a local-first, no-sync app, the device backup is the only thing standing between this user and total data loss if their phone breaks or gets replaced, and deliberately excluding a financial runway from backup in the name of "privacy" would actively hurt the exact user this app serves. That's worth stating as a decision, not an accident of whatever the default happens to be.

One more architecture-adjacent risk I want flagged now rather than discovered later: the plan is to keep the data layer "widget-ready" without building the widget this pass. If the persistent store isn't placed in an App Group container from day one, adding the widget later means migrating an existing user's data into a shared container — a real migration with real data-loss risk, for existing users, the first time someone does build the widget. If "widget-ready" is going to mean anything, it should mean the store already lives in an App Group container now, even with zero widget target present. That's cheap to do at the start and expensive to retrofit.

On destructive actions: nothing has decided what happens when a user deletes a recurring bill or paycheck. A blocking "Are you sure?" alert is the kind of small tonal violation that undercuts everything else this app is trying to be — I'd push for a soft-delete-with-undo-toast pattern instead, since it's just as safe against mistakes and doesn't inject a moment of alarm into an app whose whole design goal is calm.

Last thing, named as an accepted limitation rather than something to build defenses for: device clock manipulation (user sets their clock backward/forward) can make the freshness indicator or the runway itself read nonsensically. I don't think this deserves engineering effort in an MVP — it's a genuine edge case but a rare and low-stakes one, and defending against adversarial or broken system clocks is disproportionate here. I'd rather name it explicitly as a known, accepted limitation than have someone quietly try to harden against it and burn time that should go to the actual cadence math.

For final assumptions I'd lock: same-day bills are treated as already subtracted (conservative bias); bills-only projections (no scheduled income) are shown honestly rather than assuming a paycheck; negative/zero amounts are rejected at entry; app-switcher content gets obscured and Face ID lock is offered free; device backup includes the data store by default; the SwiftData store lives in an App Group container from the start even without a widget target; deletions are undo-toast, not confirm-alert; and device-clock tampering is an out-of-scope, named limitation, not a defended-against case.

**Claude (Product Strategist · Visionary) — Round 1**

Here's how I'd push this: the smallest slice that actually proves Cistern isn't just "a form with math behind it" is the one where a user adds a bill, adds a paycheck, gets a runway answer, then changes one of those inputs and watches the answer recompute instantly in front of them. That instant-recompute moment is the whole product in miniature — it's the thing that makes someone feel "oh, this is alive, this isn't a spreadsheet." If we build the data model and the math but the UI only recomputes on a manual refresh or a save button, we've built the boring version of this app and thrown away the one emotional beat that justifies its existence. So the MVP slice I'd commit to: onboarding that never gates on a scary balance field (add income/bills first, balance-anchor offered last and optional) → home screen with the plain-language runway sentence and a freshness indicator → quick-add sheets for paycheck, bill, and balance update, each reachable in the taps we already promised → live recompute → persistence across relaunch. That's it. That's the whole app for this pass, and I'd rather it be gorgeous and correct than have five more screens bolted on that don't get the same care.

Must-have interactions, ranked by what actually breaks the product if missing: the balance-anchor quick-update has to be reachable from literally anywhere in one tap, not buried in settings, because it's the number that decays fastest and the whole promise is "correcting the record should feel as easy as checking it." Quick-add for a recurring bill and quick-add for a paycheck both need sane cadence defaults (weekly/biweekly/semimonthly/monthly) so a user isn't doing date math themselves — if they have to compute "when's my next payday" by hand, we've failed the core thesis. The freshness indicator needs real visual weight, not a gray timestamp — I'd want it to visibly change character (not just say "9 days ago" but change how the runway answer itself is framed, e.g., "as of what you told me 9 days ago") once staleness crosses a threshold, because a technically-present-but-ignorable freshness label is functionally the same as not having one. And the zero-data empty state needs its own considered design, not a placeholder — "add your first paycheck to get a real answer" has to feel like an invitation, not a chore, since this is the very first thing a scared user sees.

What I'd cut without hesitation, and I want this cut list to be the thing that prevents scope creep later: no CSV import in this pass — it's a parsing/edge-case swamp that adds zero emotional payoff to the core loop. No saved/comparable what-if scenarios yet — that's real premium-tier work per the product brief and doesn't belong in the smallest slice; a stub "Premium" affordance that explains what it'll unlock is enough. No variable-income averaging logic — fixed cadence only for this pass, with the app honest about that limitation rather than pretending to handle it. No widget — architect the data layer so it's not blocked, but don't build the widget extension itself; that's real Xcode-target work that isn't free. No notifications, no multi-account, no transaction ledger — all already decided upstream and I'm not reopening them. And I'd explicitly cut any settings screen beyond the bare minimum (currency/locale is basically automatic from device locale, so there may be almost nothing to put there yet) — a settings screen that exists just to look complete is exactly the kind of "half-built extra feature" that dilutes focus from the one workflow that matters.

The biggest risk to my own proposal: it's tempting to read "smallest slice" as license to make the math simplistic — just naive date-add arithmetic — and defer the actually-hard edge cases (month-end bill dates, DST, past-dated paychecks, no-income-scheduled) to a later pass. I think that's the wrong cut, and it's the one place I'd push back hardest on myself. Those edge cases aren't "more features," they're "does the one feature we're building tell the truth." A slice that's small in screen count but rigorously correct in its one calculation is the right kind of small; a slice that's small because the math is sloppy is a liability wearing an MVP costume. So the cut list should protect scope (fewer screens, fewer entry types, no premium plumbing) while explicitly not touching correctness of the cash-flow projection itself.

Validation criteria, concrete enough that build verification can just check them off: from first launch, a user reaches a runway sentence within two minutes with no forced dead-end field; adding a bill or paycheck takes three taps or fewer; updating the balance anchor takes one tap plus a number entry from the home screen; every one of those actions produces a runway answer that updates on-screen with no save-and-wait, no spinner, no stale value lingering after a change; force-quitting and relaunching preserves every entry and the balance anchor exactly; the zero-data state never shows a $0-anchored number as if it were a real answer; a bill due on the 31st placed in a 30-day month lands correctly on the last day of that month, not silently in the next month or dropped; and a paycheck entered with a date in the past is either rejected with a clear reason or correctly folded into the projection, never silently ignored. If any of those nine things fail, this MVP isn't done, no matter how good the visual design looks on top of it.

**Claude (Frontend Engineer · Systems Thinker) — Round 1**

Let me come at this from the implementation side, because I think there are real engineering consequences hiding inside decisions that read as purely visual in the last phase, and if this handoff doesn't pin them down now, a builder will make different guesses than a designer would.

Start with navigation, because "one home screen, not a dashboard" is actually a navigation-architecture claim, not just a layout claim. I'd structure this as a single NavigationStack rooted at Home, with everything else layered on top of it rather than pushed alongside it: onboarding presents as a full-screen cover at first launch (not a pushed stack, so there's no back-swipe that leaves the user in a half-onboarded limbo state), the bills-and-paychecks list is a genuine NavigationLink push from the "what's coming next" row so the system back gesture works and VoiceOver's implicit "back" affordance is free, and all three quick-add/edit flows are `.sheet` presentations with a medium detent, never full-screen — full-screen would visually promote them to the same weight as Home, which contradicts the whole "home is the app" premise. Settings and the Premium stub live behind a small, deliberately unemphasized icon in the top corner of Home — not a tab bar, not a hamburger menu, just a single glyph, because the moment Home grows chrome that looks navigable, it starts reading as a dashboard again even if the content underneath obeys the one-hero rule.

On state management: I want a single `@Observable` `RunwayEngine` that owns a pure function — `func project(bills: [Bill], paychecks: [Paycheck], anchor: BalanceAnchor?, now: Date) -> RunwayProjection` — completely decoupled from SwiftData and completely decoupled from SwiftUI. That purity is what makes the mandated unit-test coverage for month-end clamping, same-day bias, and the generation-horizon cap actually testable without spinning up a persistence stack. The view layer wraps this with `@Query` on Bill/Paycheck/BalanceAnchor so recompute happens for free whenever SwiftData's context changes — no manual "recompute now" call, no spinner state to accidentally build. But I'd flag a real risk here: `@Query` observing the full Bill/Paycheck collections means any edit anywhere invalidates Home's view identity, which is fine for correctness but means the springy recompute transition needs to be driven by `.animation(_:value: projection)` keyed on the derived projection value, not on the raw query results — otherwise unrelated field touches (like a soft-delete's `deletedAt` timestamp ticking) will retrigger the whole hero-text animation even when the runway sentence itself didn't change wording.

Pay-period projection needs a memoization boundary of its own. Generating occurrences out to whatever horizon cap gets picked (I'd say 2 years, comfortably past any "few months" reasonable planning window, hard-stop regardless of cadence) is cheap once, but if it's recomputed on every SwiftUI body evaluation — which can happen many times a second — that's real, avoidable cost for no reason. I'd cache generated occurrences per entity keyed on a hash of (start date, cadence, horizon) and only regenerate when that tuple changes, not on every render pass.

The privacy cover view needs to sit one level higher than the design discussion placed it. It was scoped as swapping over "home's real content" on backgrounding, but that's not sufficient — if the user backgrounds mid-edit inside a quick-add sheet, that sheet is showing a live dollar amount too, and it's still on screen underneath the cover if the cover only targets Home. The cover has to be applied at the window/root level, above the entire presentation stack including whatever sheet or the list screen is active, triggered off `scenePhase` at the App or root-Scene level rather than inside HomeView. This is the kind of thing that's invisible in a static mock and only shows up once you think through what's actually on screen when someone backgrounds the app mid-flow.

Layout for Home under Dynamic Type: "the sentence must reflow, never truncate or clip" is a real constraint that pushes a real decision — Home can't be a fixed-height VStack pinned between safe areas, because at accessibility text sizes the hero sentence plus the balance pill plus the summary row plus two quick-add buttons will not fit in one un-scrolled screen on smaller devices. So Home is a ScrollView with the fill gradient as a full-bleed background behind the scroll content, and the content itself — hero, pill, summary row, quick-add buttons — stacked in a VStack inside it. At default text sizes it'll look and feel like a fixed hero screen because everything fits without scrolling; at max accessibility sizes it becomes genuinely scrollable rather than clipped, which is the only way to honor both "one hero screen" and "never truncate" at the same time.

Soft-delete needs a concrete mechanism, not just a UX pattern: Bill and Paycheck get a `deletedAt: Date?` field, the projection engine filters it out, the undo toast simply nils it back out within its visible window, and only after the toast expires does an actual SwiftData delete happen (or it just stays soft-deleted forever, which is honestly simpler and cheaper than a cleanup pass for an MVP this size — I'd default to that unless storage becomes a real concern, which it won't at this data scale).

Component inventory, concretely: `HeroRunwayText` (one AttributedString-backed view, single accessibility element, never composed of multiple Text views so VoiceOver doesn't fragment it), `CisternFillView` (the ambient gradient, driven by a 0–1 "fullness" value plus a "temperature" value derived from runway state, animated via `.animation(.easeInOut, value:)`, and must respect `accessibilityReduceMotion` by disabling the transition animation and just cross-fading), `BalancePill` (capsule, tappable, opens the balance quick-add sheet), `ComingNextRow` (single line, tap target the full row height even though the text is short, pushes to the list screen scrolled/highlighted to the relevant items), `QuickAddButton` ×2, a single parameterized `QuickAddSheet` that takes an `EntryKind` (bill/paycheck/balance) and doubles as the edit sheet when initialized with an existing entity, `UndoToast`, `EmptyStateView`, `BillsOnlyBanner` (or inline sentence variant), `DataLoadErrorView`, `PrivacyCoverView`, and `PremiumFeatureRow` for the stub list.

Tokens, so nobody has to guess a hex value mid-build: background base in dark mode around a deep ink (#12161C), light mode around a soft warm-white (#F7F3EC) rather than pure white, since pure white is exactly the sterile-spreadsheet feeling this app is trying to avoid. The fill gradient runs between two token pairs — a warm end (amber-to-coral, roughly #E8A65C to #E0785E) for "okay/full" and a cool end (slate-to-teal, roughly #5C7A8A to #3F5C68) for "tight/low" — always interpolated continuously by the fullness value, never swapped as discrete states. Hero typography: SF Pro Rounded (or SF Pro if Rounded reads too playful for a finance app — I'd actually lean plain SF Pro for credibility, Rounded for the fill-motif accents only), sized around 34pt at default Dynamic Type scaling normally up through accessibility sizes, with a max-width-driven line-wrap rather than a font-size ceiling so it keeps growing per Dynamic Type but wraps instead of overflowing. Body/supporting numerals sit at 17pt regular weight, captions at 13pt. Spacing on a 4pt base grid: 4/8/12/16/24/32/48. Corner radii: capsule (pill, buttons), 20pt for the toast and cards, system default for sheets. Motion: recompute transition is a spring at response ~0.45, damping ~0.8, applied to the hero text and fill value together but not to the whole screen's layout (so it feels like the number breathing, not the screen jumping); the undo toast slides up from the bottom with a fade, standard duration; sheet presentation/dismissal use the system default — no custom sheet transition, that's not where the design budget should go.

Accessibility specifics I'd add on top of what design discussion already named: after a quick-add sheet dismisses, VoiceOver focus should return to the control that opened it (the quick-add button or the list row that was being edited), and the recompute accessibility announcement should be posted after that focus change settles, not simultaneously, or the two will race and VoiceOver users will miss one of them. The skip/continue button pair in onboarding needs a consistent reading order across every step (I'd put continue first in VoiceOver order even though skip may render first or leftmost visually, since defaulting to the primary action in swipe order is what most users expect, and consistency step-to-step matters more than which specific order is chosen).

On the external-design question: I don't think this app should go through a Claude Design/Figma handoff step at all. The entire visual identity here is state-driven — the gradient is a function of live projection data, the hero text has to reflow correctly at every Dynamic Type size, the recompute motion is tied to a spring on a derived value — none of that survives translation through a static comp without the implementing engineer just reverse-engineering it back into SwiftUI anyway, which is wasted round-tripping for an app this size. I'd make the explicit call: no external design tool, build directly in SwiftUI from the token system and component inventory specified here. If someone wants App Store marketing screenshots later, that's a separate, much smaller ask and not part of this handoff.

The thing I'd most want preserved if this gets compressed later: the privacy cover has to be root-level, not Home-level, and the projection engine has to be a pure, SwiftData-free function — those are the two places where "obviously fine" instincts produce a real leak or a real untestable mess if a builder just follows the screen-by-screen description literally without the architectural note attached.

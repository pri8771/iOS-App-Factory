**Claude (Backend / Systems Engineer · Skeptic) — Round 1**

Coming at this from the data/systems side, and I want to open by pushing on something that's been quietly inconsistent for two phases now rather than let it ride into the build undetected: `RunwayEngine` is described as "pure, SwiftData-free, SwiftUI-free" and "compiles and tests standalone," but its actual locked signature is `project(bills: [Bill], paychecks: [Paycheck], anchor: BalanceAnchor?, ...)` — and `Bill`, `Paycheck`, `BalanceAnchor` are `@Model` classes, which means the file that declares them imports SwiftData, and anything that takes them as parameters transitively depends on SwiftData whether or not the word "SwiftData" appears in the Engine file itself. That's not pedantry — it means the "Engine has zero SwiftData imports and can be tested without a persistence stack" claim is only true in the narrow sense that it doesn't call `ModelContext` or perform a fetch. In practice, every engine unit test has to construct real `@Model` instances to call `project(...)`, and while that works fine for property-only models with no relationships (which is what we have here, so this isn't a blocker), the project plan needs to say this out loud and pick one of two paths rather than let a builder discover the ambiguity mid-sprint: either (a) accept that the "pure engine" boundary is a testability/isolation property, not a literal zero-SwiftData-import property, and test by constructing bare `@Model` instances outside any container — which works, is simple, and is what I'd actually recommend — or (b) add a genuinely separate plain-struct DTO layer (`BillInput`, `PaycheckInput`) that the engine actually consumes, decoupling it for real. Given this is an MVP and the models have no relationships to complicate free-standing construction, I'd lock (a) now as a plan decision rather than let someone "discover" this contradiction while writing the first engine test and improvise a fix under time pressure.

On sequencing: the tech_specs phase pre-approved a fallback from the sweep-merge algorithm to a boring day-by-day loop "if test coverage is at risk," but a pre-approved escape hatch with no trigger date is exactly the kind of thing that either never gets invoked (because nobody wants to admit they're behind) or gets invoked in a panic the night before the build is due. The plan needs an actual checkpoint: attempt the sweep-merge as part of the data_domain milestone, but set a hard internal checkpoint at the end of that milestone — if the full named edge-case suite (semimonthly-weekend, Feb 29/30/31, horizon-cap, 31st-clamping for both entity types, same-day-bias, negative/zero rejection, bills-only honesty, plus the sweep-specific empty-stream and same-date-tie-break cases) isn't green at that checkpoint, the fallback triggers immediately, not after more time is sunk trying to make the clever version work. I'd rather see the naive loop ship with 100% of that test list passing than the sweep-merge ship with 80% of it passing and a promise to finish later.

More broadly, I think this plan needs to be organized around the four owning_lanes that tech_specs already established — data_domain, services_utilities, primary_ui, polish_resilience — with a strict dependency order: data_domain and services_utilities (models, engine, validation, persistence factory, App Group config) have to be substantially done and tested before primary_ui starts wiring `@Query` against them, because otherwise UI work is guessing at an interface that might still change shape. Polish_resilience (privacy cover, undo toast, empty/error states, accessibility) is explicitly last, not because it's unimportant but because every one of those components wraps or reacts to states the earlier two lanes produce — you can't build a meaningful `DataLoadErrorView` until you know what "ModelContainer construction fails" actually looks like in practice, and you can't build the privacy cover correctly until the sheet-presentation stack it has to sit above actually exists.

I want to flag one concrete, easy-to-skip milestone that has nothing to do with Swift code correctness and everything to do with whether this project builds at all: the App Group entitlement, the literal group identifier in the entitlements file, and target membership/signing configuration have to be real, verified deliverables — not just a string constant referenced in Swift. The pitfalls reference is blunt about this and it's exactly the failure mode I'd expect here: a perfectly correct `ModelContainerFactory.makeContainer()` call will throw at runtime if the `.entitlements` file, provisioning, and bundle identifier don't actually line up, and that failure will masquerade as the "data load failure" edge case (routing to `DataLoadErrorView`) when it's actually a build misconfiguration. I'd make "app launches and successfully creates the App Group–backed ModelContainer on a clean simulator install" an explicit, checked milestone gate before any UI work is trusted, not something assumed to just work because the factory function is written correctly.

Related skepticism about `DataLoadErrorView` itself: its only recovery action is "retry," but for a local on-device SwiftData store, the only real ways `ModelContainer` construction throws are structural (bad entitlement/schema mismatch, disk corruption) — categories where hitting retry again does nothing, because the same misconfiguration is still there. I'm not proposing new scope here, just flagging it plainly in the plan so it doesn't get treated as a fully "solved" edge case: retry mainly covers a genuinely rare disk-level hiccup, and the verification plan should include actually forcing this state (e.g., temporarily pointing the factory at an invalid configuration in a debug harness) rather than assuming the view "must work" because it exists in the file tree.

On the versioned schema and migration plan: I'd explicitly note in the plan that there is nothing to migration-test here, on purpose — one schema version, empty `stages` array, and I don't want a milestone that invents a fake migration scenario just to have "migration test coverage" on a checklist. The value of `VersionedSchema`/`SchemaMigrationPlan` right now is entirely structural (it exists so a real migration later has a shape to slot into), and testing effort should go where it's real: the engine edge cases and the `#Predicate` soft-delete exclusion actually being enforced at the fetch layer, which does need a concrete in-memory-store test, not just code review.

Last thing I'd insist gets carried forward rather than re-derived: the verification plan for this whole app is already fully specified two phases back — the eight-point checklist from `next_steps_small` (2-minute first-run, ≤3-tap add, 1-tap balance update, instant recompute with no spinner, force-quit persistence, no fake $0 empty state, 31st-in-a-30-day-month correctness, past-dated-paycheck handling) plus the named edge-case list from `tech_specs`/`ios_architecture_review`.

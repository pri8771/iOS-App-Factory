The location isn't actually in question — every prior phase already locked it: the whole app lives at `/Users/pchordia/Documents/iOS-App-Factory/kitbag`, one self-contained Xcode project, not a subfolder of this `.agent_cwd`. I'm treating that as settled and moving straight to how four lanes work in that one folder without stepping on each other.

Folder shape I want locked before anyone writes a line of feature code: `Kitbag.xcodeproj` at the root, `Kitbag/App/` for the tiny shared entry point, and then one top-level source folder per lane — `Kitbag/Sources/DataDomain`, `Kitbag/Sources/ServicesUtilities`, `Kitbag/Sources/PrimaryUI`, `Kitbag/Sources/PolishResilience` — plus `Kitbag/Resources/ClimateData` for the bundled JSON, `Kitbag/Kitbag.storekit`, `KitbagWidget/` as its own extension target, and matching test folders (`KitbagTests/DataDomainTests`, `KitbagTests/ServicesUtilitiesTests`, `KitbagTests/PrimaryUITests`, `KitbagUITests/`). This isn't cosmetic — Xcode 16+ synchronized folder groups mean if each lane's folder is registered once against the right target, every new file a lane adds under its own folder is automatically part of the target with zero `project.pbxproj` editing after that. That's the single biggest conflict-prevention lever available to us, because a hand-edited `.pbxproj` merge conflict is exactly the kind of silent, invisible-in-code-review failure the architecture phase already warned about. So: `project.pbxproj` itself, the `.entitlements` files, the scheme, and target definitions are a forbidden edit zone for every lane except the one scaffolding task that creates them. After that task lands, nobody hand-edits the project file — you add files under your own lane folder and the sync group picks it up. If a lane genuinely needs a new target or a new top-level folder, that's a stop-and-ask, not a silent edit.

On ownership: data_domain owns everything with zero SwiftUI/StoreKit imports — the SwiftData models and `VersionedSchema`, the bundled climate JSON plus `ClimateCatalog`, the pure derivation functions, the `GenerationEngine`, `LearningHistoryReader`, and `applyGenerationResult`. This lane can be entirely finished and fully unit-tested before any other lane needs it to exist, which is exactly why I want it to go first and fast. services_utilities owns the repositories, `PackingCoordinator`, `EntitlementStore`, the widget snapshot/publisher, notification scheduler, the App Intent, export, and the deep-link router — everything that's a real seam between SwiftData and the outside world, including the one dual-target-membership file (`WidgetSnapshot.swift`, referenced by both the main app and the widget extension, edited only by this lane). primary_ui owns every screen, every reusable component, the navigation shell, and `ColorStory` — nothing in this lane ever touches `ModelContext` or `StoreKit` directly, only the protocols services_utilities exposes. polish_resilience owns `AppError`, the accessibility collapse adapter and `HeaderRenderStyle`, `PersistenceBootstrap`, `BuildVerificationChecklist`, and is also the lane that does the project scaffolding itself, since target/signing/App-Group setup is systems work, not feature work, and it's the highest-risk, most invisible-when-wrong part of this whole build.

The dependency order isn't a suggestion, it's how we avoid wasted work: nobody in primary_ui or services_utilities should start real implementation until data_domain's generation engine has passed its own go/no-go test (review trip A demotes an item with a named reason, minimum-evidence-of-two respected) — that test is the actual product, and if it's wrong everything built on top of a broken coordinator gets thrown away. So the practical sequencing is: scaffolding first and alone, then data_domain races ahead on models/climate/engine while polish_resilience works on error/persistence-bootstrap scaffolding in parallel (neither depends on the other), then services_utilities's repositories and coordinator come online once the engine's proven, and only then do primary_ui's screens start consuming real data instead of guessing at shapes. Trying to build screens against an unproven coordinator is how you end up rewriting UI twice.

Since there's no git remote here and this isn't currently a repo, I want `git init` done as part of the scaffolding task, with one branch per lane (`lane/data-domain`, `lane/services-utilities`, `lane/primary-ui`, `lane/polish-resilience`) cut from `main` immediately after scaffolding lands, and `main` kept always-buildable. Each lane commits to its own branch and merges back to `main` at defined checkpoints — after the engine go/no-go test passes, after the zero-history core loop is wired end-to-end, after review-to-learning is demonstrable, and once at the end — rather than everyone merging continuously and fighting over half-finished interfaces. Whoever runs build verification at each checkpoint is the de facto integrator: they rebase lane branches onto `main` (not merge commits, keep it linear so `git bisect` stays useful if something breaks), and if two lanes touched the same file outside the explicitly shared zones, that's treated as a process violation to flag and fix, never a merge silently taken from one side.

On knowing it works: each lane owns the tests for its own files, but the tests that prove a cross-lane seam is correct belong to whichever lane owns that seam — data_domain owns the persistence integration tests and the learning go/no-go test, services_utilities owns the coordinator integration tests (create-trip, regenerate-with-diff, submit-review, rebuild-derived-state), primary_ui owns the two XCUITest critical paths (create-trip-to-non-empty-list, and the 3-tap-or-fewer review flow — this is the number the entire product's invalidation criterion hinges on, so it needs an actual measured test, not an eyeballed guess), and polish_resilience is the final gatekeeper that runs the full suite, sweeps for accessibility/error-state gaps, and fills in `BuildVerificationChecklist` with what's actually proven versus what's flagged as unverifiable in this headless environment (widget/Live Activity/Siri on-device rendering, live StoreKit purchases). Nobody declares a milestone done on their own say-so — the checklist is the shared source of truth and it only records what a test actually ran and passed.

**Coordinator (Claude) — decision after round 1**

Both engineers converged on the same shape for this phase, but at different zoom levels — Codex sketched a coarse 9-task backlog split roughly by lane and milestone, and Claude took that same structure and drove it all the way down into 28 concrete, immediately-startable tasks with exact file paths, explicit forbidden-edit zones, and a real conflict-prevention mechanism. Nothing in Claude's version contradicts Codex's — it's the same dependency chain from the locked project_plan (scaffolding → data domain → generation engine → coordinator → UI → review-to-learning → monetization/insights → native surfaces → hardening), just made buildable.

The load-bearing decision everyone lands on: `project.pbxproj`, entitlements, scheme, and target definitions are a forbidden edit zone for every lane except the one scaffolding task (T-001) that creates them — after that, lanes only add files under their own folder and Xcode's synchronized folder groups pick them up automatically, so there's never a second hand-edit to the project file. Git is set up with one branch per lane cut from `main` right after scaffolding, with rebase-not-merge integration at defined checkpoints (engine go/no-go, zero-history loop, review-to-learning, final) rather than continuous merging. Ownership splits by volatility exactly as decided in project_plan: data_domain (models, climate data, pure generation engine — zero SwiftUI/StoreKit), services_utilities (repositories, coordinator, entitlements, widgets, intents — the only real seam to the outside world, including the one dual-target `WidgetSnapshot.swift` file), primary_ui (every screen/component, never touches `ModelContext`/StoreKit directly), and polish_resilience (scaffolding itself, error handling, accessibility, and final verification sweep). Nobody starts real UI work until the generation engine's go/no-go test (review demotes an item with a named reason, respecting the 2-data-point evidence rule) actually passes — that's the one gate everything else waits on.

No open disagreement — Codex's list is a strict subset of what Claude's expands into, so I'm carrying Claude's 28-task backlog forward as the final one since it's what lets build workers start immediately, per this phase's acceptance bar.

You'd need real cross-process locking discipline, and I don't think that's a good use of a priority-4 budget when the actual product requirement is razor-thin: a widget needs a countdown number and a percentage, nothing else, and per the standing privacy rule it's not even allowed to show anything more specific than that.

So my case is: don't give widgets/Live Activities read access to the domain layer at all. Have the main app own one small, explicit "publish" step — whenever a trip's date or packing progress changes, write a tiny value (days-until-trip, percent-complete, maybe a stable trip identifier for the Live Activity's own lifecycle tracking) into App Group `UserDefaults` or a small flat file, and call `WidgetCenter.shared.reloadTimelines` right after. The widget extension never touches SwiftData, never touches the real trip/item graph, and structurally cannot leak destination text even if someone fat-fingers a future UI change, because the data that could leak it was never written to the shared container in the first place. That's not just simpler, it's a stronger privacy guarantee by construction rather than by convention — I'd rather the architecture make the mistake impossible than rely on every future contributor remembering "don't put destination names in the widget view." It also means the App Group surface is trivially testable: you can unit-test "given this trip state, the published snapshot equals this struct" with zero SwiftData or extension-target ceremony at all, versus trying to integration-test actual shared-store consistency across process boundaries, which is a much worse time-to-confidence ratio for the same assurance. I'd call this the resolved answer unless you've got a concrete reason the countdown-only snapshot is insufficient for something Live Activities specifically need — and if it turns out ActivityKit's content-state update mechanism wants slightly richer data to drive the packing-progress bar live, that's still just a slightly bigger denormalized struct, not a reason to open the real store to the extension.

One more thing I want to flag before this closes out, because it hasn't come up yet and it's a real gap: nobody's said what happens when `WidgetCenter.reloadTimelines` gets called a lot in a short burst — say, someone checking off ten items in thirty seconds while packing. Widget reloads are budget-limited by the system; hammering it risks silent throttling where the widget just goes stale and nobody notices because there's no error surfaced anywhere. I'd want the publish step debounced (coalesce updates over a second or two of inactivity) rather than fired on every single checkbox tap, both to respect the budget and because it's one less thing to explain when QA notices the widget "sometimes" lags behind reality.

**SwiftUI architecture:** SwiftUI + Observation (`@Observable`) on iOS 17+, no Combine/MVVM ceremony. Feature services sit behind protocols; views never talk to persistence or StoreKit directly. The core contract: trip context, prior reviewed outcomes, entitlement state, and climate inputs get snapshotted into plain value types (structs, not `@Model` objects), a single pure/synchronous generation-and-weighting function runs over those structs, and the result is written back through `ModelContext` on the main actor as one immutable generated-list artifact (reason strings + changed flags frozen at that moment). Reviews never mutate a generated list in place — they update the learning history, and only the *next* generation reads that history. This is the one architectural rule the whole product's trust claim depends on.

**Apple framework choices:** SwiftData (persistence), StoreKit 2 (entitlements), WidgetKit + ActivityKit (countdown widget, packing-session Live Activity), AppIntents (Siri add-item), UserNotifications (review nudges), os.Logger (structured logging). No Core ML, no NaturalLanguage, no on-device LLM/Foundation Models anywhere in v1 — considered and explicitly rejected, not deferred behind a flag, because any phrasing layer creates a second, untrusted explanation path for the exact reason-strings the app's credibility depends on, for zero product value the spec doesn't already mark as optional/roadmap.

**Persistence and local data plan:** SwiftData with a `VersionedSchema` defined from the first commit (not retrofitted after a migration breaks). Cascade-delete rules explicit on trip deletion; "didn't happen" is a soft-exclude flag, never a delete, and needs its own test proving it's invisible to the learning engine but still visible (clearly labeled) in the trip list. Bundled climate data is static JSON loaded once into an in-memory read-only cache — no database needed for 30-40 regions × 12 months — with a real test that loads the actual bundled resource and asserts every region has all 12 months present with plausible values, so a data-entry mistake fails CI instead of shipping silently.

**Widgets/Live Activities data plan (now resolved):** The main app is the sole owner of the real SwiftData store. Extensions never read or write it directly. Whenever relevant trip state changes, the main app publishes a tiny denormalized snapshot (days-until-trip, percent-complete, a stable trip identifier for Live Activity lifecycle tracking — nothing else, no destination text, per the standing lock-screen privacy rule) into App Group storage, debounced over a second or two of inactivity rather than fired per-checkbox-tap, then calls `WidgetCenter.shared.reloadTimelines`. Widgets and Live Activities read only that snapshot. App Intents that mutate data call back into the main app's domain services rather than standing up a second persistence path.

**Permissions/privacy plan:** No location, camera, contacts, photo library, or health permissions anywhere in the app — a real product asset, stated as a feature in Settings, not buried in policy text. Only notification permission (for post-trip review nudges) and implicit Siri usage description for the App Intent, both requested contextually (first real trip with a pending review / first intent use), never at cold launch. Lock-screen-visible surfaces (widgets, notifications) never carry destination text, enforced structurally by never writing that data into the shared container in the first place rather than relying on convention.

**Dependency/license decision:** Zero third-party dependencies. First-party Apple frameworks only, consistent with the permissive-dependency policy and the "boring at the systems layer" philosophy both engineers converged on.

**Testing implications:** Swift Testing for all pure logic (climate lookup, month-majority resolution, color-story/motif derivation, the generation/weighting engine, entitlement-boundary math) with zero SwiftData/SwiftUI dependency — fast, deterministic, CI-friendly. Data-snapshot assertions pin exact reason strings and changed-flags for known review histories, not just "a string exists." XCTest UI tests limited to two critical paths: create-trip-to-non-empty-list, and the post-trip-review 3-tap-or-fewer flow — the number the entire product's invalidation criteria hinges on. Persistence integration tests cover relaunch survival of checked state/custom items/reviews, cascade-delete correctness, and "didn't happen" exclusion. StoreKit is tested against a local `.storekit` config via Xcode's StoreKit Testing framework (simulated purchase, renewal failure, expiration) — stated plainly as the ceiling, never implied to be a verified live sandbox purchase. Widgets/Live Activities/Siri get their internal logic unit-tested (timeline provider output, intent `perform()` results, published-snapshot correctness) but actual on-device rendering is flagged explicitly as unverifiable in a headless environment, not asserted as working.

**Entitlement/offline behavior:** One long-lived `EntitlementStore` listening continuously to `Transaction.updates` (never just a launch-time check), with an offline-tolerant cache that has an explicit grace window and biases toward preserving already-unlocked access rather than flapping a paying user's history locked/unlocked due to a spotty connection. Trip creation and core packing flow never block on network or entitlement checks. Notifications, widget reloads, and Live Activity state are all best-effort/opportunistic/disposable — none of them are load-bearing for the core loop, consistent with every earlier phase's standing rule that native conveniences never gate a core workflow step.

**Claude (Frontend Engineer · Skeptic) — Round 1**

Let me get into this concretely, because the prior phases produced a genuinely good product/UX contract, but there are architecture-level gaps in it that will bite whoever builds this if they're not nailed down now — and my job this round is to find those before we hand this off.

Start with navigation, because "two tabs plus a persistent toolbar plus" sounds settled but isn't fully specified as an architecture. Two tabs means two independent `NavigationStack`s (Ledger, Calibration), each with its own path. The thing nobody has addressed yet: a notification tap, a widget tap, and a Siri/App Intent completion all need to land on the Revisit/Resolve screen for a specific decision ID regardless of which tab is currently selected, regardless of what's currently on screen, and regardless of whether the Capture sheet happens to be open at that exact moment. That last case is not hypothetical — someone can absolutely be mid-capture on a new decision when an old one's revisit notification fires. If we don't decide now, whoever builds this will improvise, and the improvised answer is usually "queue it and hope," which silently drops the deep link. I want a named rule: a single `AppRouter` observable object owns "pending deep-link target ID," is fed by `onOpenURL`/`UNUserNotificationCenterDelegate`/App Intent perform(), and on receipt it force-switches the tab selection to Ledger, dismisses any open sheet (Capture included — losing an unsaved three-field draft is an acceptable trade since re-entering it takes seconds, but that has to be a stated decision, not an accident), and pushes the Revisit screen onto the Ledger stack. Cold-launch (app was terminated) has to hit the same code path as warm foreground — that's usually where this breaks, because people test the warm case and never the terminated one.

Persistence: this needs to be SwiftData (or Core Data if we want more control over migrations, but SwiftData is the default-correct call for a v1 with this document rightly framing schema stability as a forward-looking constraint) with UUID-keyed models from day one — DecisionRecord and OutcomeRecord as separable entities, exactly as the product brief demanded, so a later CloudKit-backed sync container is an additive change, not a rewrite. But the part that actually threatens correctness is that this app isn't single-process. The widget extension and the App Intent extension are separate processes that need to read and write the same store. That means an App Group container is non-negotiable infrastructure, not a nice-to-have, and the "one shared validation/cap-enforcement layer" that every prior phase insisted on has to be a plain Swift package or shared framework target linked into all three targets — app, widget extension, intent extension — operating against the App Group's SwiftData store. If that shared layer doesn't exist as a real compiled boundary, "the same cap check everywhere" is just a design aspiration that three different engineers will implement three slightly different ways. I'd also flag that widget timelines don't auto-refresh when the main app writes — every mutation (resolve, reschedule, new capture) needs an explicit `WidgetCenter.shared.reloadTimelines(ofKind:)` call, or the "upcoming revisits" widget will show stale counts, which is a believable, embarrassing bug that won't show up in a demo but will show up in real use within a day.

On the rolling-window notification scheduling — everyone agreed on the concept, nobody specified the number. I'd pin it: schedule real `UNNotificationRequest`s for the nearest 30 upcoming active decisions (well under the ~64 ceiling, leaving headroom for the system and any other local notification use), and every time a decision is created, resolved, or rescheduled, recompute which N are inside the window and diff against currently-scheduled requests — cancel what fell out, add what fell in. This recompute has to run on app launch too, not just on the triggering mutation, because a resolve that happens while the app is backgrounded and killed by the OS won't get a chance to run the recompute unless it's also idempotently checked at next launch.

On the visual system, I want to push back gently on the "monospaced/tabular figures for structural data" language from the design phase, because taken literally — "use a monospaced typeface for dates and numbers" — that means introducing a second typeface family (something like SF Mono) alongside the serif, and pairing a full monospace font with New York serif reads cheap, not premium, and monospace fonts have historically weaker Dynamic Type behavior. What actually delivers "tabular, ledger-like numerals" without a second typeface is the `.monospacedDigit()` modifier on the system font — same font family, fixed-width digits, scales properly with Dynamic Type, no second font to license or maintain. I'd lock that as the actual implementation instead of a literal second typeface.

Nobody in any prior phase has said one word about Dark Mode, and I don't think that's an oversight we get to leave unresolved — Apple's App Store review expectations and basic user expectation are that iOS apps support both appearances unless there's a hard justification, and "we only have a light paper aesthetic" is not that justification. I'd rather decide now than have it discovered as a gap during build: support Dark Mode with an ink-on-dark-parchment analog — deep charcoal-brown background instead of pure black, warm off-white ink text instead of pure white — keeping the same restrained, non-neon palette rather than inverting into some generic dark-mode blue-gray. This preserves the "considered, premium" feeling in both appearances instead of looking like the designer only thought about one.

The stamped/postmark treatment on the Before block is a nice idea but it's exactly the kind of decorative element that breaks quietly. If it's implemented as any kind of fixed-size image/texture asset, it needs `accessibilityHidden(true)` with the actual date exposed as a separate, plain accessible text element — a screen reader has no business trying to describe an inked stamp graphic. And it needs to gracefully degrade at the top two accessibility Dynamic Type sizes: either the stamp visual shrinks to a small corner mark while the date renders as normal large text next to it, or it's dropped entirely above a certain content size category, because a rotated/imperfect stamp graphic sized for standard text will clip or overlap once the type doubles in size. This has to be tested at the largest accessibility size before anyone calls the visual design final, exactly as the design phase itself said about Dynamic Type generally — I just want to make sure this specific decorative element doesn't get a pass because it's "just flavor."

On the widget itself: I'd scope it to small and medium home-screen widget families only for v1 (StandBy and Lock Screen variants are explicitly a could-have per the features phase, and I'd hold that line) and its deep link mechanism should be `widgetURL` on non-interactive content pointing to a custom URL scheme the app's `onOpenURL` parses into a router target, since widgets can't push arbitrary in-process navigation. The widget's own entitlement awareness (does this user have Plus) has to come from a cached flag written to the App Group by the main app on every StoreKit transaction update, since the widget process can't cheaply do its own live entitlement check — that's a real subtlety I don't want lost, because paid features that determine widget content (if any exist — right now I don't think they should; the widget should stay identical for free and paid, showing only counts) need to not require the widget process itself to talk to StoreKit.

On motion: given the explicit "no bouncy, no celebratory animation" rule, I'd specify concretely rather than leave it to whoever's implementing to pick a curve — sheet presentations use the system default sheet transition (no custom spring), the calibration number update on resolve uses a plain opacity cross-fade around 150–200ms with no scale or bounce, and the only motion that should feel remotely alive is the ledger row removal when a decision resolves — a simple removal transition (`.opacity.combined(with: .move(edge: .leading))`), not a springy collapse. Anything with `.spring()` or `.bouncy` anywhere in this app is off-brand and should be treated as a review-blocking issue, not a style nit.

For accessibility on the ledger row specifically, I'd be explicit that the row needs `.accessibilityElement(children: .combine)` with a single custom `accessibilityLabel` string assembled from entry number, decision text, due date, and status — the design phase already said this in principle, but the engineering trap is that SwiftUI's default behavior fragments a row's Text views into separate VoiceOver stops unless you explicitly combine them, and it's easy to ship the fragmented default and think it's fine because it's technically readable, just clunky.

The entire visual system here is typographic and componentized rather than illustration-heavy, and the actual risk in this app is behavioral (Dynamic Type reflow, cross-process widget state, Dark Mode parity, deep-link routing) — things a static mockup tool won't surface and could actually paper over by looking fine at one fixed text size in one fixed appearance. I'd rather this handoff go straight into SwiftUI with real accessibility settings toggled during build than get rendered as a pretty static comp first. ### 2. Navigation & routing architecture
- Two independent tab `NavigationStack`s (Ledger, Calibration). Capture is never a tab — it's a sheet reachable from a toolbar `+` present on both tabs.
- **One `AppRouter`** (observable, app-level) is the single owner of "pending deep-link target ID." On receipt it: dismisses any open sheet (an open Capture draft is intentionally discarded — re-entry takes seconds, and this is a stated tradeoff, not an accident), force-switches tab selection to Ledger, and pushes Revisit for the target decision.
- Cold launch (terminated app) and warm foreground must resolve through the identical router path — this is the case that silently breaks if only the warm path is tested.
- A stale notification pointing at an already-resolved decision opens Decision Detail's resolved/read-only mode, never a broken or duplicate outcome form.

### 3. Persistence & cross-process integrity
- SwiftData, UUID-keyed `DecisionRecord` / `OutcomeRecord` as separable entities (cloud-sync-ready later, no rewrite implied).
- **App Group container is required infrastructure** — app, widget extension, and Siri/App Intent extension all read/write the same store.
- **One compiled shared framework/package** (not a design convention) implements validation and the 5-active-decision cap check. All three targets call into it — nothing reimplements the rule locally.
- Every mutation (new capture, resolve, reschedule) triggers `WidgetCenter.shared.reloadTimelines(ofKind:)` explicitly — widgets do not auto-refresh from an external write.
- Widget entitlement awareness (if ever needed) comes from a cached flag the main app writes to the App Group on StoreKit transaction updates — the widget process never talks to StoreKit directly. ### 4. Notification scheduling policy
- Rolling window: real `UNNotificationRequest`s scheduled for the nearest **30** upcoming active decisions (comfortable margin under the ~64 OS ceiling).
- Recompute-and-diff (cancel what fell out of the window, add what fell in) runs on every create/resolve/reschedule mutation **and** on every app launch — the launch-time recompute is the safety net for a resolve that happened while the app was backgrounded and then killed by the OS before it could recompute.
- Revisit dates use calendar-day-plus-fixed-local-time semantics, not an absolute instant.

### 2. Navigation & routing architecture
- Two independent tab `NavigationStack`s (Ledger, Calibration). Capture is never a tab — it's a sheet reachable from a toolbar `+` present on both tabs.
- **One `AppRouter`** (observable, app-level) is the single owner of "pending deep-link target ID." It is fed by `onOpenURL`, the `UNUserNotificationCenterDelegate` callback, and App Intent `perform()`. On receipt it: dismisses any open sheet (an open Capture draft is intentionally discarded — re-entry takes seconds, and this is a stated tradeoff, not an accident), force-switches tab selection to Ledger, and pushes Revisit for the target decision.
- Cold launch (terminated app) and warm foreground must resolve through the identical router path — this is the case that silently breaks if only the warm path is tested.
- A stale notification pointing at an already-resolved decision opens Decision Detail's resolved/read-only mode, never a broken or duplicate outcome form.

### 3. Persistence & cross-process integrity
- SwiftData, UUID-keyed `DecisionRecord` / `OutcomeRecord` as separable entities (cloud-sync-ready later, no rewrite implied).
- **App Group container is required infrastructure** — app, widget extension, and Siri/App Intent extension all read/write the same store.
- **One compiled shared framework/package** (not a design convention) implements validation and the 5-active-decision cap check. All three targets call into it — nothing reimplements the rule locally.
- Every mutation (new capture, resolve, reschedule) triggers `WidgetCenter.shared.reloadTimelines(ofKind:)` explicitly — widgets do not auto-refresh from an external write.
- Widget entitlement awareness (if ever needed) comes from a cached flag the main app writes to the App Group on StoreKit transaction updates — the widget process never talks to StoreKit directly. (Note: current design has no paid-vs-free content difference in the widget itself — it shows the same count/due-ness copy regardless of tier.)

### 4. Notification scheduling policy
- Rolling window: real `UNNotificationRequest`s scheduled for the nearest **30** upcoming active decisions (comfortable margin under the ~64 OS ceiling).
- Recompute-and-diff (cancel what fell out of the window, add what fell in) runs on every create/resolve/reschedule mutation **and** on every app launch — the launch-time recompute is the safety net for a resolve that happened while the app was backgrounded and then killed by the OS before it could recompute.
- Revisit dates use calendar-day-plus-fixed-local-time semantics, not an absolute instant.
- Active `authorizationStatus` recheck on every foreground surfaces silent permission revocation in-app; denied-at-launch has an explicit fallback state.

### 5. Design tokens

**Codex (QA & Risk · Pragmatist) — Round 1**

The smallest architecture that still ships honestly is one shared core module, one app target, one widget target, and one App Intent target, all writing through the same App Group-backed store and all calling the same rule engine. If that shared rule engine is not a compiled boundary, the free-tier cap, active-vs-resolved semantics, outcome handling, and notification scheduling will drift across surfaces. This app is unusually easy to fake in demos and unusually easy to get wrong in production, because the main screens are simple while the real failure modes sit in cross-process persistence, stale notifications, permission revocation, and tier gating.

I would keep the module split plain. `ChoicepointCore` owns data models, repository protocols, policy/rule evaluation, stats computation, and notification-plan generation as pure data. The app target owns SwiftUI screens, navigation, paywall presentation, onboarding, and the `AppRouter`. A services/utilities layer owns concrete persistence, notification scheduling, entitlement snapshot caching, export, and widget refresh hooks. The widget and App Intent extensions stay thin and call into core plus services; they should not reimplement business logic locally. Parallel build lanes can work if the contract is strict: data/domain defines canonical models and use-case protocols, services/utilities provides implementations against SwiftData-or-Core-Data and system frameworks, primary UI consumes only those interfaces, and polish/resilience owns edge-state handling, permission/status banners, analytics phrasing, and QA hooks.

The data model should stay source-of-truth only. `DecisionRecord` needs stable UUID, created/updated timestamps, required capture fields (decision text, prediction, revisit date), optional fields (context, options, confidence, what-would-change-my-mind), status (`active` or `resolved`), and an optional linked `OutcomeRecord`. `OutcomeRecord` should be separate so resolution is an event, not a destructive overwrite: outcome classification (`held`, `missed`, `mixed`), narrative, resolved-at timestamp. A separate `RevisitSchedule` value should capture calendar-day intent plus preferred local reminder time, because storing an absolute fire timestamp will break DST/timezone semantics. Stats should be derived, never persisted as mutable counters. Notification requests should also be derived, never treated as durable business state.

The persistence strategy should be conditional, not assumed. Use SwiftData only if a short App Group multi-process spike survives concurrent writes and cross-process reads cleanly; otherwise drop to Core Data immediately. That spike is a real gate, not a nice-to-have, because app + widget + intent shared persistence is the most likely place this MVP quietly fails. The core package should expose repository-style APIs so the fallback is implementation churn, not type-system fallout across the app. All writes should happen through a serialized mutation boundary in the concrete repository/service layer so notification recompute, widget refresh, and entitlement/cap rechecks happen atomically from the product’s perspective. If a write saves a decision, the notification plan must be recomputed in the same operation; if a decision is resolved, stale pending notifications must be canceled in that same path.

Concurrency boundaries should be blunt. UI-facing view models are `@MainActor`. Persistence, export generation, and notification-plan building can run off-main, but results hop back to main before mutating UI state. The repository should be the only lane allowed to mutate stored records. The policy service should be pure and injectable with `Calendar` and `TimeZone`; otherwise the promised DST/timezone tests won’t actually be possible. The notification scheduler should accept a plain `NotificationPlan` from core and be responsible only for diffing against `UNUserNotificationCenter`. That separation is what makes the scheduling logic testable rather than simulator theater.

Error handling needs to be explicit because this app has real degraded modes. Notification permission denied or revoked is not an alert-only concern; it is an app state with visible fallback copy and a Settings deep link. At-cap is not a generic save failure; it is a first-class domain error surfaced identically in app, widget, and App Intent. Export failure, persistence save failure, stale/deleted deep-link target, and unavailable entitlement snapshot all need typed errors so UI can show specific states instead of “something went wrong.” A stale notification that points to a resolved decision should open resolved detail. A stale notification that points to a deleted/missing decision should open a human-readable fallback state, not silently dump the user on home.

The biggest risk in this direction is the shared store across three processes, not the screen work. If that foundation is flaky, everything else becomes unreliable in subtle ways: Siri can bypass the cap, widgets can lag behind resolution, notifications can reopen already-resolved items, and stats can diverge from what the ledger shows. The final artifact has to preserve one detail above all others: the integrity of the product lives in a single shared rules layer plus a single shared persistence boundary, not in “matching implementations” across targets.

Verification should be split honestly between automated and manual. Unit tests should cover cap enforcement, downgrade behavior, active/resolved counting, `notYet` reschedule semantics, mixed outcomes, low-sample stats thresholds, notification rolling-window selection, stale-target routing decisions, and timezone/DST scheduling with injected calendar/timezone. **Claude (Frontend Engineer · User Advocate) — Round 1**

Let me start by giving my own strongest take on the technical architecture before anyone else weighs in, focused specifically on the client side — navigation, state, and the failure modes a real user actually hits.

The architecture we inherited from the ios_architecture_review is sound and I'm not relitigating it: one `ChoicepointCore` Swift package compiled into the app, widget extension, and intents extension, SwiftData-pending-spike over an App Group container, a single `AppRouter` owning deep-link state. What I want to nail down now is the part that phase left underspecified because it wasn't that phase's job: the actual view/view-model wiring and where this breaks for a real person tapping through it.

Here's my concrete layout. One Xcode project, four targets: `Choicepoint` (app), `ChoicepointWidget`, `ChoicepointIntents`, and a local SPM package `ChoicepointCore` embedded in all three. Inside `ChoicepointCore`: `Models/` (DecisionRecord, OutcomeRecord, the enums), `Policy/` (DecisionPolicy, cap enforcement, notification-window selection — all pure functions over injected Calendar/TimeZone, zero SwiftData imports), `Persistence/` (a `DecisionRepository` protocol plus a `SwiftDataDecisionRepository` and a `CoreDataDecisionRepository` behind it, so the spike's fallback is a swap, not a rewrite), `Entitlements/` (EntitlementSnapshot + a small store that reads/writes the App Group cache), `Notifications/` (the scheduler that talks to `UNUserNotificationCenter`). Inside the app target: `App/` holds `AppRouter` and the root TabView; `Features/` has one folder per screen (Ledger, Capture, DecisionDetail, Revisit, Calibration, Onboarding, Paywall, Settings), each with a thin View and an `@Observable @MainActor` view model that talks only to `DecisionRepository` and `DecisionPolicy` — never directly to SwiftData types, which is what keeps the persistence spike's fallback cheap and keeps view models unit-testable without spinning up a model container.

The one frontend decision I want to make explicitly rather than let it be assumed: Capture does not bind directly to a `DecisionRecord`. It builds a local `DecisionDraft` value type in view-local `@State`, and only calls `repository.create(draft:)` on Save. This matters for a real user because if we bind a live SwiftData-backed object to text fields and the user backgrounds the app mid-typing or force-quits, we either get a half-written phantom record haunting the ledger or we're building manual rollback logic to prevent it. A plain draft struct means "cancel" is just "discard local state," full stop, no cleanup required. This also gives me a clean synchronous place to run cap-enforcement: check `activeCount` when the sheet opens (so the banner is visible before the user even starts typing, not sprung on them at the moment of tapping Save) and re-check atomically inside the repository's create call so a race between two capture surfaces can't both slip through.

My biggest actual worry, and the thing I think this phase has to pin down concretely because I don't trust it to work by accident: the interaction between `AppRouter`'s deep-link push and SwiftUI's `NavigationPath`/sheet-dismissal timing. The locked rule is "dismiss any open sheet, force-switch to Ledger tab, push Revisit for the target decision," and that sounds like three lines of code, but in practice, mutating a `NavigationPath` in the same state update as dismissing a sheet or switching a `TabView` selection is a known source of SwiftUI silently dropping the second mutation — the path push either no-ops or fires before the tab switch has actually taken effect and lands on the wrong stack. I want the router built as an explicit small state machine, not three independent property sets: `pendingDestination` is set first, a single root-level `.onChange`/`.task` observes it and sequences the steps (clear sheet → next runloop tick or `.onDisappear` completion → set tab → push path → clear pending), and this exact sequencing gets tested on the cold-launch path specifically, because cold launch is where nobody happens to have a sheet open and the bug hides until a real user is mid-capture when a two-week-old revisit notification fires. That's not a hypothetical for this app — it's a named scenario in earlier phases — and I don't want it to be the thing that ships untested because it "worked in the simulator" on a warm foreground.

On persistence I'll just make it concrete rather than re-argue the spike: `DecisionRepository` is a protocol so the app, widget, and intent all code against the same interface regardless of which side of the spike we land on. Revisit dates are stored as `DateComponents` (year/month/day only), never a `Date` instant — `DecisionPolicy.revisitFireDate(for:calendar:)` derives the actual moment at read/schedule time. Entry numbers are sequential and assigned by the repository at creation (`max(existing) + 1`), not client-generated, so two capture surfaces creating decisions near-simultaneously can't collide.

For testing, the highest-value target is exactly what the architecture phase already flagged: `DecisionPolicy` and `DecisionRepository`'s cap/resolve/reschedule logic, unit-tested adversarially across all three write paths hitting the cap. On the UI side, I want XCUITest coverage of exactly one thing that matters most to a real user: empty ledger to a saved three-field decision in a fixed small number of taps, with the Save button's enabled state asserted at each intermediate fill state. Deep-link routing (cold launch and warm) needs an XCUITest using launch arguments to simulate a notification-response callback, asserting the Revisit screen for the correct decision ID is what's on screen — this is exactly the class of bug that "looks done" in a demo and breaks in the field. Real force-quit notification delivery, permission-revoked-while-backgrounded, and Dynamic Type at max accessibility size in both appearances stay on the manual pre-release checklist, as already agreed — I'm not duplicating that fight, just making sure the UI-test suite doesn't quietly try to fake coverage of things it can't actually prove.

### Round 2


**Codex (QA & Risk · Pragmatist) — Round 2**

Claude’s draft/local-state point is the one I’d keep and make stricter, because it closes a real class of bugs instead of just looking architecturally tidy. If Capture binds directly to a persisted model, we create failure modes that are completely out of proportion to the feature: half-saved phantom decisions from backgrounding, cap slots getting occupied by drafts that never became real decisions, widget/intent surfaces reading junk the user never meant to commit. A `DecisionDraft` that stays in view-local state until a single `create(draft:)` call is the right boundary. I’d apply the same discipline on the revisit side too: the narrative text for outcome logging should stay local until `resolve(...)` succeeds, not mutate the stored decision incrementally while the user is typing. That sounds obvious, but it matters here because the whole product promise is evidentiary integrity. We do not want “partially edited but not actually resolved” state drifting into the canonical store because a sheet was dismissed or the app was interrupted.

On the repository-versus-mutation-service split, I think Codex’s earlier instinct to separate “raw persistence” from “writes with side effects” was better than folding everything into one fat repository interface, but I’d simplify it one notch from where he had it. The smallest thing that ships cleanly is: repository owns canonical CRUD and transactional semantics; a `DecisionCommandService` owns the three mutations that matter to product integrity — create, resolve, reschedule — and is the only place allowed to trigger notification recompute and widget timeline reload. That gives us one place to test “a resolve always recomputes the 30-window and reloads widgets” instead of trusting every caller to remember. If those side effects live directly on repository methods, people will eventually call the wrong method from one surface and skip the recompute. If we go the other way and invent too many layers, we waste time. One command service is the pragmatic middle. Same answer on `DecisionSurface`: I would keep it, but only on command-service write calls, not threaded through every read and every low-level repository method. It’s useful for diagnostics and adversarial tests of cap enforcement across app/widget/intent. It’s not useful enough to let it infect the entire API surface.

I also agree with Claude’s SwiftUI routing concern, and I don’t think that one should stay at the level of “be careful sequencing it.” This is exactly the kind of thing that passes happy-path testing and then flakes when a real notification arrives while Capture is up. So I want the router contract to name the sequencing behavior as a real implementation rule: deep links are queued as intent, not executed as immediate simultaneous mutations. Dismiss sheet first, wait for the presentation state to clear, then switch tab, then push onto the Ledger stack. And I’d go one step more defensive than Claude did: if another deep link arrives while one is in flight, last write wins, because stacking two revisit pushes is nonsense for this product. We only ever want one pending destination. That’s the kind of ugly edge condition no one thinks about until Siri capture completes at the same time a revisit notification is tapped, and then you learn whether your router is a real state machine or just three `@State` writes pretending to be one.

The other place I want to sharpen the contract is around persistence concurrency, because the earlier architecture review correctly said “SwiftData if the spike passes, Core Data if it doesn’t,” but the interface contract here still assumes synchronous-looking safety that the implementation may not have. Entry-number assignment is the obvious example. Claude is right that it must be repository-assigned, not client-generated, but `max + 1` is only safe if it happens inside one serialized write boundary. If app and intent both ask for `nextEntryNumber()` and then both create, they can collide unless that read-plus-write is atomic. So I’d remove `nextEntryNumber()` from the public worker-facing interface entirely. Nobody outside the write boundary should ask for one. `create(draft:...)` returns the fully assigned `DecisionRecord`, full stop. Same philosophy for stats: nobody stores them, nobody mutates them, nobody “updates counters.” They’re computed views over source records every time. That’s not elegance, that’s how we avoid cross-process drift.

On notification planning, I’d side with Codex’s explicit `NotificationPlan` shape over Claude’s “scheduler recomputes internally” version, because it gives us a smaller, more testable truth surface. We care about proving the policy chose the right nearest 30 decisions, the right day-based fire dates, and privacy-safe content. A plain-data plan lets us unit test that without hitting `UNUserNotificationCenter`. Then the scheduler’s job becomes boring diff/apply plumbing. That’s exactly where we want boring. On the DecisionRepository-vs-DecisionMutationService split — Codex, you're right and I'm updating my position. In my round 1 pass I put create/resolve/reschedule directly on `DecisionRepository`, but sitting with your argument that "if a write saves a decision, the notification plan must be recomputed in the same operation," I don't think that's safe to leave as a convention my view models are supposed to remember to follow. Concretely: `CaptureViewModel.save()`, `RevisitViewModel.resolve()`, `RevisitViewModel.notYet()`, and `CaptureDecisionIntent.perform()` are four separate call sites that all need to trigger repository write + notification recompute + `WidgetCenter.reloadTimelines`. If that's three separate calls each engineer has to remember to chain, it's exactly the kind of thing that gets forgotten in the App Intent path six months from now when someone's fixing an unrelated bug, and then Siri capture silently stops updating the widget. So I'm splitting it the way you proposed: `DecisionRepository` becomes pure storage (fetch/persist/delete, no policy, no side effects), and `DecisionMutationService` is the single actor-isolated write boundary that composes repository-write + `DecisionPolicy` validation + `NotificationScheduling.recomputeAndApply` + widget reload as one atomic-feeling operation. Every view model and the intent calls the mutation service, never the repository directly, for writes. Reads still go straight through `DecisionRepository` since there's no side effect to orchestrate there.

On `NotificationPlan` as explicit data versus a bare recompute call — also agreeing with Codex here, and honestly this was already settled back in ios_architecture_review, which explicitly said "separate notification request generation from the actual UNUserNotificationCenter side effects so it can be snapshot-tested as plain data." A recompute() call that does everything internally can't be unit tested without touching the real notification center; a `NotificationPlan`/`NotificationPlanItem` struct means I can write a test that says "given these 35 active decisions and this instant, the plan contains exactly these 30 decisionIDs with these fire-date-components" with zero UNUserNotificationCenter involvement. Keeping this as data.

Small thing, real value.

On `AppRouter` — here's where I'll actually push back, on Codex specifically. Making it a protocol doesn't buy testability we don't already have — the concrete `@Observable` class is plain Swift state, I can construct one in a test and assert `pendingDestination`/`selectedTab`/`ledgerPath` transitions with zero mocking required. What it does cost is a layer of indirection for something that's supposed to be the single boring, unambiguous nerve center of navigation. I'd rather keep it a concrete final class exactly as I had it, and I do want to explicitly confirm the sequencing concern nobody's contested yet: `routeToDecision` has to be an ordered state machine — clear any presented sheet first, then switch tab, then push path — not three property sets fired in the same SwiftUI update cycle, because that's the exact pattern that silently drops a `NavigationPath` mutation. I want one shared function pair in `ChoicepointCore` — construct on the widget side, parse on the app side — so there's structurally no way for them to diverge.

### Round 3


**Codex (QA & Risk · Pragmatist) — Round 3**

Claude’s right on the AppRouter shape, and I don’t think this is the place to get abstract for abstraction’s sake. A protocol here buys us basically nothing and creates one more seam around the single most timing-sensitive object in the app. There is one router, one navigation graph, one set of ordered transitions that have to happen exactly the same way every time. Making that a protocol with alternate implementations sounds “clean” until somebody tests against a fake that doesn’t reproduce SwiftUI’s actual path-mutation behavior, and then the real app is the only place the race shows up. For this app, the safer move is a concrete `@MainActor @Observable final class AppRouter` with plain mutable state and one explicit method that sequences the transition in order: dismiss sheet, switch tab, then push route. Keep the logic trivially testable by asserting state after each step, but don’t pretend dependency inversion is reducing risk here. It isn’t. Same story on the new `WidgetDeepLink` helper Claude proposed: yes, absolutely put the URL encode/decode in one shared implementation inside `ChoicepointCore`. Widget URL construction and app parsing drifting apart is exactly the kind of stupid breakage that survives code review because both sides “look fine” in isolation. One helper, one format, no duplicate string assembly.

I also want to explicitly ratify Claude’s `refreshOnForeground()` point for Ledger and Calibration, because that’s not nice-to-have polish, it’s how cross-process reality becomes visible. A Siri/App Intent capture while the app is backgrounded, or a widget-triggered deep link after a resolve on another surface, does not magically invalidate the main app’s in-memory view state. If those screens only load once, we ship a product where the data is correct on disk and wrong on screen until the user manually jogs it. That’s the kind of bug users describe as “the app feels flaky,” and they’re right. So yes: both view models need explicit foreground refresh paths driven off `scenePhase`, same as notification permission status and entitlement refresh. The throughline across all of this is pretty simple: anywhere state can change outside the current process or outside the current screen, the spec needs a named resync path, not a hope. With that, I think the contract is finally coherent: pure policy layer, pure repository layer, one serialized mutation service for writes, one concrete router, one shared deep-link codec, and foreground refresh on the read-heavy screens so the app stays honest after backgrounded mutations.

**Claude (Frontend Engineer · User Advocate) — Round 3**

Since Codex hasn't weighed in yet on the router question this round, let me sharpen the case rather than just repeat it, because I think there's a purely technical argument that settles this, not just a taste preference.

`@Observable` is a macro, and it only applies to concrete types — you cannot slap `@Observable` on a protocol. That means if `AppRouter` is a protocol the way Codex wrote it (`@MainActor protocol AppRouter: Sendable { var selectedTab: AppTab { get set }; ... }`), somewhere there still has to be exactly one concrete class implementing it that actually carries the `@Observable` machinery SwiftUI needs to track `selectedTab` for the `TabView` binding and `ledgerPath` for the `NavigationStack`. So the protocol doesn't remove the concrete class — it just wraps it in an extra layer that every SwiftUI call site now has to see through, and worse, `TabView(selection: $router.selectedTab)` and `NavigationStack(path: $router.ledgerPath)` need a `Bindable` wrapper around a concrete `@Observable` type; if the property in the view is typed as the protocol, you either need an existential-safe generic (real complexity, for one implementation that will only ever exist once) or you give up `@Observable`'s ergonomics entirely and hand-roll something Combine-based, which is exactly the kind of unnecessary machinery the ios_architecture_review already told us to avoid ("no Combine unless something narrow in StoreKit forces a bridge"). So this isn't "protocols are bad," it's "this specific protocol can't actually deliver on its own promise without a concrete class underneath doing all the real work anyway." I'd like Codex to either show me a way around that `@Observable`-on-protocol limitation I'm missing, or just concede the point, because right now the protocol version reads like abstraction for its own sake on the one object in this app that's supposed to be the least abstract, most boring thing we build.

I also want to flag something concrete about Codex's `enqueue(_:)` / `processPendingRoute()` split as two separate protocol methods: that's a footgun by construction. If a caller (say, the App Intent's completion handler, written by whoever's on that lane and maybe not staring at the router code that day) calls `enqueue()` and never calls `processPendingRoute()`, the deep link just... sits there, forever, silently. Nothing crashes, nothing errors, the notification just doesn't open anything and the user thinks the app is broken. A single `routeToDecision(_:)` call that does the full dismiss-then-switch-then-push sequence internally, atomically, from the caller's perspective, removes an entire class of "someone forgot the second call" bugs. I'm keeping it as one method, and I am adopting Codex's "last write wins" addition explicitly — if a second `DeepLinkTarget` arrives while the first hasn't finished resolving (genuinely plausible: Siri capture completes right as a stale revisit notification is tapped), the router just overwrites `pendingDestination` and only the latest one actually gets processed. Stacking two pushes onto one `NavigationStack` for this product would be nonsense, agreed.

On the two things I raised last round that are still sitting unconfirmed — `refreshOnForeground()` on `LedgerViewModel`/`CalibrationViewModel`, and the shared `WidgetDeepLink` encode/decode helper — I'll just restate why I think these aren't optional polish. Think about the actual user again: someone logs a decision via Siri while driving, phone locked in their pocket, app fully backgrounded the whole time. They get home, unlock their phone, tap into Choicepoint from the home screen icon (not from a notification, just normal app-open), and land on Ledger. If `LedgerViewModel.load()` only ever runs once at first-appear and this is a warm reopen of an already-alive scene, they will not see the decision they just created, because nothing told the view model anything changed. That's not an edge case, that's the App Intent feature actively lying to the person who just used it. Wiring `refreshOnForeground()` to `scenePhase` at the root is maybe four lines of code and it closes a real trust gap. Same logic, smaller stakes, for the `WidgetDeepLink` helper — it's cheap insurance against the widget extension and the app's `onOpenURL` parser silently disagreeing about URL format, which is exactly the kind of thing that works in every manual test because both sides get rebuilt from the same Xcode run and then breaks for a real user after an update where only one side's cache got evicted.

One thing I want to close off before it becomes a debate nobody needs to have: pagination. Both `fetchActive()` and `fetchResolved()` return plain arrays with no cursor or limit. I don't think that's a gap — this is a personal decision journal, not a firehose; even a very heavy user logging a few decisions a week for years is in the low hundreds of records, and SwiftData/Core Data both handle an in-memory fetch of that size instantly. Adding pagination now would be solving a problem this product doesn't have at the cost of real complexity in the repository contract. I'd rather name that as a deliberate non-goal now than have someone "future-proof" it later into something the widget's tight timeline budget then has to account for.

Testing splits the same way it always has in this phase: `DecisionPolicy` and `DecisionMutationService` get real adversarial unit tests (cap enforcement across all three `DecisionSurface` values, active/resolved counting, DST-safe scheduling via injected calendar, low-sample stats thresholds), `AppRouter`'s state-machine sequencing gets a unit test asserting the exact dismiss→switch→push order plus the last-write-wins behavior under a simulated double-deep-link, XCUITest covers fast-capture-to-save and cold-launch deep-link routing, and force-quit notification delivery / permission-revoked-while-backgrounded / downgrade-above-cap stay on the manual release checklist because no automated test can actually prove those.

## Final Output

**Architecture overview.** One Xcode project, four targets: `Choicepoint` (app), `ChoicepointWidget`, `ChoicepointIntents`, all linking a local Swift package `ChoicepointCore` that owns every cross-process rule. Inside `ChoicepointCore`: a pure `DecisionPolicy` (validation, entry numbering, resolve/reschedule transforms, notification-window selection, stats computation — all with injected `Calendar`/`TimeZone`, never `.current`, so DST/timezone behavior is deterministically testable); a pure-storage `DecisionRepository` (fetch/persist/delete only, backed by SwiftData or Core Data depending on the earlier multi-process spike, no business logic); and a single serialized write boundary, `DecisionMutationService`, that is the *only* caller of repository writes and the only place that composes policy validation + persistence + notification recompute + widget-timeline reload into one atomic-feeling operation per mutation. Every view model and the Siri `CaptureDecisionIntent` call `DecisionMutationService` for writes and `DecisionRepository` for reads — nothing calls SwiftData/Core Data types directly outside the concrete repository implementation, which is what makes the SwiftData→Core Data fallback (if the earlier spike required it) a swap, not a rewrite.

Navigation is a single concrete `@MainActor @Observable final class AppRouter` (never a protocol — `@Observable` can't attach to one, and `TabView`/`NavigationStack` bindings need a concrete `Bindable` type anyway), fed identically by `onOpenURL`, the notification delegate, and App Intent completion. Its one public method, `routeToDecision(_:)`, internally sequences dismiss-sheet → switch-tab → push-path as ordered, non-simultaneous state changes (this exact ordering, not three independent property writes in one update cycle, is what prevents SwiftUI from silently dropping the path push), and only the latest pending destination survives if two deep links arrive close together. A shared `WidgetDeepLink` helper in `ChoicepointCore` is the single place that encodes/decodes the custom URL scheme, so the widget and the app can never disagree about its format.

**File/module layout.** `ChoicepointCore/{Models, Policy, Persistence, Entitlements, Notifications}` submodules; app target has `App/` (AppRouter + root TabView) and `Features/{Ledger, Capture, DecisionDetail, Revisit, Calibration, Onboarding, Paywall, Settings}`, each a thin View plus an `@Observable` view model.

**Data models.** `DecisionRecord`/`OutcomeRecord` as separable entities; `RevisitSchedule` stores calendar-day components plus local reminder time (never an absolute `Date`, to avoid DST/cross-process drift); `DecisionDraft`/`OutcomeDraft` are local, unpersisted working state so cancel or force-quit during capture/resolve never leaves a phantom or partially-mutated record; stats and notification plans are always computed fresh, never stored as counters.

**Persistence strategy.** Conditional per the prior architecture-review spike (SwiftData if it survives concurrent multi-process writes, Core Data otherwise), hidden entirely behind `DecisionRepository`. No pagination in v1 — a named, deliberate non-goal given realistic record volumes for this product.

**Testing strategy.** `DecisionPolicy`/`DecisionMutationService` get adversarial unit tests (cap enforcement parametrized across all `DecisionSurface` values, active/resolved counting, DST-safe scheduling, low-sample stats thresholds); `AppRouter` gets a unit test asserting the exact dismiss→switch→push order and last-write-wins under a simulated double deep-link; XCUITest covers fast-capture-to-save and cold-launch deep-link routing; real force-quit notification delivery, permission-revoked-while-backgrounded, and downgrade-above-cap stay on the manual pre-release checklist since no automated test can prove them.

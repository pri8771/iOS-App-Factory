## Final Output

**Screen inventory**
1. **Ledger** (tab 1, home) — time-grouped active decision list (Overdue, Due Soon, Upcoming) with Resolved reachable via a secondary filter/segment, not a separate tab.
2. **Calibration** (tab 2) — calm stats view built only from resolved decisions, with a locked confidence-analytics teaser for free users.
3. **Capture sheet** — reached via a persistent "+" in the toolbar on both tabs (not a third tab, not floating-only-on-Ledger). Two-speed form: decision, prediction, revisit date above the fold; context, options, confidence, "what would change my mind" behind an "add detail" disclosure.
4. **Decision Detail** — dual-mode: active review (original record + edit/revisit-now/reschedule actions) and resolved archive (read-only original record + outcome record). Also the landing target for a stale notification pointing at an already-resolved decision.
5. **Revisit/Resolve screen** — dedicated screen, entered from a notification deep link or from Detail's "revisit now." Opens directly to the Before block (original decision, prediction, reasoning, entry number, stamped/postmarked capture date) fully visible and dominant at the top, no gate, no reveal tap. Below it: narrative outcome field plus held/missed/mixed actions, with "not yet" as a clearly separated, quieter secondary action beneath those three.
6. **Onboarding** — one honest card explaining the revisit mechanic and why notification permission matters, then the system permission prompt. No swipeable tour.
7. **Paywall sheet** — surfaced contextually from cap-pressure (Ledger/Capture) or from the locked-analytics teaser (Calibration), never a generic standalone screen reached out of context.
8. **Settings** — reached via toolbar from Ledger; shows notification permission status (including revoked-since-grant), subscription state, export, and privacy copy.
9. **Widget** — upcoming/overdue revisit counts and nearest due date only, no decision content, deep-links into the same Revisit/Resolve screen.
10. **Siri/App Intent capture** — same three-field minimum, same shared validation/cap-enforcement path as the in-app Capture sheet.

**Primary user flow**
Empty Ledger → tap "+" → three-field capture (decision, prediction, revisit date) → save in seconds → sheet dismisses, entry appears in Ledger under Upcoming with entry number and due date. Weeks later, a discreet notification ("1 revisit due") arrives; tapping it deep-links straight into that decision's Revisit screen, which opens directly onto the stamped Before block — no gate, nothing to unlock. User writes what happened and taps held/missed/mixed (resolving, freeing a cap slot, updating Calibration live) or taps the secondary "not yet" (only pushes the date forward, decision stays active). Once at least one decision is resolved, Calibration shows real counts instead of an empty state; confidence-bucketed analytics stay behind a locked, honestly-labeled teaser until upgrade. Hitting the 5-active free cap surfaces inline in Capture before save is attempted, routing the user either toward resolving an existing overdue item or toward the contextual paywall.

**State model per screen**
- **Ledger**: first-run empty / no-active empty / populated active / filtered resolved / permission-warning visible / at-cap banner / persistence-error.
- **Capture**: pristine / partially entered / valid-savable / saving / cap-blocked (shown before save attempt) / validation error / save failure / save success.
- **Decision Detail**: loading-by-id / active-loaded / resolved-loaded / stale-or-deleted error.
- **Revisit/Resolve**: loading / due / overdue-with-day-count / ready (Before block shown, outcome actions available) / submitting outcome / resolved-success / reschedule-success (not-yet) / already-resolved-elsewhere fallback (read-only) / submission failure.
- **Calibration**: empty (no resolved data) / low-sample (raw fraction only, no headline percentage below a minimum n) / full (percentage + fraction + held/missed/mixed tallies) / locked-confidence-teaser (free) / confidence-analytics visible (paid) / export unavailable/ready/failure.
- **Onboarding**: pre-permission explanation / granted / denied / skipped.
- **Settings**: normal / notifications off-or-revoked / paid-active / paid-lapsed-over-cap / export running/failed.
- **Widgets/Siri**: success / cap-blocked / validation failure / open-app fallback.

**Visual direction**
Paper-and-ink ledger aesthetic: warm off-white background, near-black ink text, hairline rules between rows instead of shadowed cards, zero gradients or bouncy/celebratory animation. Serif typeface for anything the user actually wrote (decision, prediction, narrative outcome) so it reads as their voice; tabular/monospace figures for structural data (dates, entry numbers, percentages, day-counts). The Revisit screen's Before block gets a literal stamped/postmark treatment on the entry number and original capture date — inked, slightly imperfect, unmistakably "this happened once and can't be re-stamped" — delivering the emotional weight of a sealed record through static typography rather than any interaction gate. One sparse accent color (deep brass or oxblood), used only for the compose action and overdue emphasis; overdue is signaled primarily through weight/placement/text, never color alone.

**Accessibility notes**
Dynamic Type must reflow every ledger row and the Before block without truncating decision text, prediction, or date at accessibility sizes — tested at largest text size before visual design is considered final, given how much of the design leans on serif/tabular typographic contrast. VoiceOver reads each ledger row as one coherent sentence (entry number, decision, revisit date, status), not fragmented elements. Held/missed/mixed and overdue status never rely on hue alone — always paired with explicit text or a glyph. Widget summaries are plain accessible sentences ("2 decisions overdue this week"). Permission, cap, and paywall messaging is plain-language, never buried in decorative copy.

10. **Siri/App Intent capture** — same three required fields, same shared validation/cap path as in-app Capture.

### 2. Navigation & routing architecture
- Two independent tab `NavigationStack`s (Ledger, Calibration). Capture is never a tab — it's a sheet reachable from a toolbar `+` present on both tabs.
- **One `AppRouter`** (observable, app-level) is the single owner of "pending deep-link target ID." It is fed by `onOpenURL`, the `UNUserNotificationCenterDelegate` callback, and App Intent `perform()`. On receipt it: dismisses any open sheet (an open Capture draft is intentionally discarded — re-entry takes seconds, and this is a stated tradeoff, not an accident), force-switches tab selection to Ledger, and pushes Revisit for the target decision.
- Cold launch (terminated app) and warm foreground must resolve through the identical router path — this is the case that silently breaks if only the warm path is tested.
- A stale notification pointing at an already-resolved decision opens Decision Detail's resolved/read-only mode, never a broken or duplicate outcome form.

### 3. Persistence & cross-process integrity
- SwiftData, UUID-keyed `DecisionRecord` / `OutcomeRecord` as separable entities (cloud-sync-ready later, no rewrite implied).
- **App Group container is required infrastructure** — app, widget extension, and Siri/App Intent extension all read/write the same store.
- **One compiled shared framework/package** (not a design convention) implements validation and the 5-active-decision cap check. All three targets call into it — nothing reimplements the rule locally.
- Every mutation (new capture, resolve, reschedule) triggers `WidgetCenter.shared.reloadTimelines(ofKind:)` explicitly — widgets do not auto-refresh from an external write.
- Widget entitlement awareness (if ever needed) comes from a cached flag the main app writes to the App Group on StoreKit transaction updates — the widget process never talks to StoreKit directly. (Note: current design has no paid-vs-free content difference in the widget itself — it shows the same count/due-ness copy regardless of tier.)

### 4. Notification scheduling policy
- Rolling window: real `UNNotificationRequest`s scheduled for the nearest **30** upcoming active decisions (comfortable margin under the ~64 OS ceiling).
- Recompute-and-diff (cancel what fell out of the window, add what fell in) runs on every create/resolve/reschedule mutation **and** on every app launch — the launch-time recompute is the safety net for a resolve that happened while the app was backgrounded and then killed by the OS before it could recompute.
- Revisit dates use calendar-day-plus-fixed-local-time semantics, not an absolute instant.
- Active `authorizationStatus` recheck on every foreground surfaces silent permission revocation in-app; denied-at-launch has an explicit fallback state.

### 5. Design tokens
- **Light**: warm off-white paper background; near-black ink primary text; weathered-graphite secondary text; hairline ink-colored dividers (no cards, no shadows, no gradients).
- **Dark**: deep charcoal-brown "paper," warm off-white ink text, hairline rules lifted just enough for separation — an ink-on-dark-parchment analogue, not an inverted/neon dark mode.
- **Accent** (both modes, used sparingly — compose action + overdue emphasis only): deep brass or oxblood.
- **Typography**: serious serif for anything user-authored (decision, prediction, narrative outcome); system text with **`.monospacedDigit()`** for dates, entry numbers, percentages, counts — no second monospace font family.
- **Status language**: overdue/held/missed/mixed communicated via weight, placement, and explicit text/glyph — never color alone, in either appearance.
- **Motion**: system-default sheet transitions only, no custom springs anywhere; save confirmation is a plain opacity fade (~150–200ms), no scale/bounce; resolved-row removal from the ledger uses a simple opacity+move-edge transition, not a springy collapse; Revisit's Before block gets at most a slight content-load fade, never a theatrical reveal animation.

### 6. Component inventory
Ledger row (single combined VoiceOver element) · section header with count · inline status/permission/cap banner · capture field group · date preset chip row · "Add detail" disclosure · record header with entry number + stamped date · outcome action group (Held/Missed/Mixed) · secondary "Not yet" action · locked-analytics teaser card · empty-state block · paywall value card · widget summary block (single tap target).

### 7. Accessibility requirements
- Dynamic Type must reflow every ledger row and the Before block at accessibility sizes without truncating decision text, prediction, or dates — tested at largest accessibility size before visual design is considered final.
- Ledger rows use `.accessibilityElement(children: .combine)` with one assembled label ("Entry 14. Should we hire the contractor. Due in 6 days.") — not SwiftUI's fragmented per-Text default.

## Final Output

**SwiftUI architecture:** Boring on purpose. `@Observable` view models per screen (main-actor, thin views), a single `AppRouter` observable object as the sole owner of pending deep-link state, fed by `onOpenURL`, the notification delegate, and App Intent `perform()` — routing notification taps, widget taps, and Siri completions through one identical path regardless of entry point. Two independent tab `NavigationStack`s (Ledger, Calibration); Capture is a sheet, never a tab. No third-party state management, no coordinator/redux pattern — the nav graph is small enough that this is the right amount of architecture.

**Apple framework choices:** SwiftUI + Observation, WidgetKit (small/medium families only), AppIntents (not legacy SiriKit), UserNotifications, StoreKit 2 (`Transaction.currentEntitlements` as the source of truth), OSLog with redaction discipline for any diagnostics. No BackgroundTasks (nothing needs periodic background fetch — the notification rolling-window recompute is synchronous, triggered by mutation/launch/foreground, not a timer). No Combine unless something narrow in StoreKit forces a bridge.

**Persistence and local data plan:** A single compiled shared package (`ChoicepointCore`) owns the data models, the cap/validation rule, and notification-plan generation as plain data — linked into the main app, widget extension, and Siri/App Intent extension, all reading/writing one App Group container. **Persistence engine is conditional, not assumed:** run a half-day spike first — three trivial targets sharing one `ModelContainer` against the App Group URL, hammered with concurrent alternating writes (~50 rapid writes across two processes, plus read-after-write-across-processes and delete/resolve/reschedule churn mirroring the real notification recompute path). If SwiftData comes through clean, it's the persistence engine. If it shows any crash, corruption, or stale read, drop immediately to Core Data with `NSPersistentStoreCoordinator` — no debugging spiral. Because the core package exposes repository-style operations rather than leaking SwiftData types into feature code, this fallback is annoying, not catastrophic. Calibration stats are computed from resolved records, never stored as denormalized counters. Notification fire times are derived from calendar-day-plus-local-time semantics at read/schedule time, never stored as an absolute instant — both rules exist specifically to prevent cross-process drift. The `DecisionPolicy` service takes `Calendar`/`TimeZone` as injected dependencies (never `.current`), which is what actually makes DST/timezone unit tests possible rather than theoretical.

**Permissions/privacy plan:** Notification authorization is the only permission the core product needs, requested after the single onboarding card, with an active `authorizationStatus` recheck on every foreground (covering the real-world case of grant-then-revoke-while-backgrounded, not just first-launch denial). No mic, speech, contacts, calendar, or photos permissions added in v1 — a future on-device voice-note feature gets its own permissions surface only if and when it's actually built, never added preemptively. Decision content never appears in notifications or widgets (counts/due-ness only), with no v1 toggle to weaken that. No analytics SDK, no crash reporter that could capture decision content, no background sync, no accounts. Export is user-initiated and local only. AR is explicitly out — no spatial or camera interaction anywhere in this product.

**Dependency/license decision:** Zero third-party packages, no exceptions — including for testing tooling (no snapshot-testing library; Xcode's preview canvas at accessibility sizes plus manual device QA covers it). First-party Apple frameworks are sufficient for the entire v1 scope.

**StoreKit/offline behavior:** Extensions never call StoreKit directly — the main app writes a cached, normalized entitlement snapshot (free/paid state, not just a bare boolean) into the App Group on every transaction update, refreshed on every foreground. No custom grace-period or TTL logic layered on top — trust StoreKit 2's own JWS-verifiable transaction state as sufficient; building a parallel staleness system would just be a new source of bugs. Downgrade only ever blocks new active-decision creation above the cap; it never hides, deletes, or force-resolves existing data.

**Testing implications:** The shared core package's policy logic (cap enforcement across all three write surfaces, active/resolved counting, downgrade behavior, held/missed/mixed aggregation, low-sample stats thresholds, "not yet" preserving active state, rolling notification window selection, stale-notification fallback routing, DST/timezone scheduling via injected calendar) is the highest-value and fully deterministic unit-test target. Real force-quit notification delivery, permission-revoked-while-backgrounded, and downgrade-with-decisions-above-cap are explicitly named as things automated tests cannot prove — these are a manual pre-release QA checklist, not something that rides silently on green CI.

## Final Output

**Architecture overview.** One Xcode project, four targets: `Choicepoint` (app), `ChoicepointWidget`, `ChoicepointIntents`, all linking a local Swift package `ChoicepointCore` that owns every cross-process rule. Inside `ChoicepointCore`: a pure `DecisionPolicy` (validation, entry numbering, resolve/reschedule transforms, notification-window selection, stats computation — all with injected `Calendar`/`TimeZone`, never `.current`, so DST/timezone behavior is deterministically testable); a pure-storage `DecisionRepository` (fetch/persist/delete only, backed by SwiftData or Core Data depending on the earlier multi-process spike, no business logic); and a single serialized write boundary, `DecisionMutationService`, that is the *only* caller of repository writes and the only place that composes policy validation + persistence + notification recompute + widget-timeline reload into one atomic-feeling operation per mutation. Every view model and the Siri `CaptureDecisionIntent` call `DecisionMutationService` for writes and `DecisionRepository` for reads — nothing calls SwiftData/Core Data types directly outside the concrete repository implementation, which is what makes the SwiftData→Core Data fallback (if the earlier spike required it) a swap, not a rewrite.

Navigation is a single concrete `@MainActor @Observable final class AppRouter` (never a protocol — `@Observable` can't attach to one, and `TabView`/`NavigationStack` bindings need a concrete `Bindable` type anyway), fed identically by `onOpenURL`, the notification delegate, and App Intent completion. Its one public method, `routeToDecision(_:)`, internally sequences dismiss-sheet → switch-tab → push-path as ordered, non-simultaneous state changes (this exact ordering, not three independent property writes in one update cycle, is what prevents SwiftUI from silently dropping the path push), and only the latest pending destination survives if two deep links arrive close together. A shared `WidgetDeepLink` helper in `ChoicepointCore` is the single place that encodes/decodes the custom URL scheme, so the widget and the app can never disagree about its format.

**File/module layout.** `ChoicepointCore/{Models, Policy, Persistence, Entitlements, Notifications}` submodules; app target has `App/` (AppRouter + root TabView) and `Features/{Ledger, Capture, DecisionDetail, Revisit, Calibration, Onboarding, Paywall, Settings}`, each a thin View plus an `@Observable` view model.

**Data models.** `DecisionRecord`/`OutcomeRecord` as separable entities; `RevisitSchedule` stores calendar-day components plus local reminder time (never an absolute `Date`, to avoid DST/cross-process drift); `DecisionDraft`/`OutcomeDraft` are local, unpersisted working state so cancel or force-quit during capture/resolve never leaves a phantom or partially-mutated record; stats and notification plans are always computed fresh, never stored as counters.

**Persistence strategy.** Conditional per the prior architecture-review spike (SwiftData if it survives concurrent multi-process writes, Core Data otherwise), hidden entirely behind `DecisionRepository`. No pagination in v1 — a named, deliberate non-goal given realistic record volumes for this product.

**Testing strategy.** `DecisionPolicy`/`DecisionMutationService` get adversarial unit tests (cap enforcement parametrized across all `DecisionSurface` values, active/resolved counting, DST-safe scheduling, low-sample stats thresholds); `AppRouter` gets a unit test asserting the exact dismiss→switch→push order and last-write-wins under a simulated double deep-link; XCUITest covers fast-capture-to-save and cold-launch deep-link routing; real force-quit notification delivery, permission-revoked-while-backgrounded, and downgrade-above-cap stay on the manual pre-release checklist since no automated test can prove them.

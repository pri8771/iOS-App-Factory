Picking up where the product brief left off, my job this round is to stress-test the loop for every way it can actually break in a room full of phones, and to convert the open questions the last three phases punted forward into either firm assumptions or explicit exclusions — not leave them open a fourth time.

Start with the thing nobody has actually answered yet even though every phase has flagged it: what happens when the host's phone goes away mid-round. Every design decision so far — session creation, roster lock, fragment dealing, the live shared board, the countdown, the majority-confirm solution flow — implicitly assumes one device is the source of truth for round state. If that device locks, gets a phone call, force-quits, or its battery dies, the whole session's state either has to live somewhere else too, or the session is just over. I don't think we can leave that ambiguous going into architecture. My proposed resolution: no host migration in v1. The host device is authoritative and if it's lost mid-round, the round is unrecoverable — every peer detects the disconnect (MC session delegate fires), shows an explicit "host disconnected, mission ended" state (not a spinner that hangs forever), and offers "return to lobby" so the app never appears frozen. That's a real limitation, but it's honest, and it's a vastly smaller build than host-election/state-replication across peers, which I don't think this pipeline can build and verify credibly anyway given the earlier phase's own admission that we can't test at real 10-12-device scale. I'd rather ship "host loss ends the round gracefully" than silently promise host resilience we can't prove.

Related edge case that's easy to miss: iOS auto-locks the screen by default after a short idle period, and once a screen locks, MultipeerConnectivity's networking gets suspended for that device almost immediately. A silent, read-and-type game has long stretches where nobody is actively touching the screen — which is exactly when auto-lock kicks in and quietly drops a player from the session without them doing anything wrong. This isn't a rare failure, it's the expected failure if we don't handle it: the app needs to disable the idle timer for the duration of an active session and restore it afterward, and that has to be a stated requirement now, not something discovered when someone's phone dies out of the game five minutes in.

Second big edge case cluster: mid-round departures that aren't host-related. A regular player can still background the app (notification banner, incoming call), lock their phone, or just quit. Because every player holds a unique fragment that's dealt once, a player dropping mid-round doesn't just remove a participant — it can make the puzzle literally unsolvable if their fragment was load-bearing for the solution. That needs an explicit answer, not a hope. I'd propose: reconnect-with-grace-period is the only recovery path (peer has some window, say 60-90 seconds, to rejoin and resume with their same dealt fragment intact — the fragment isn't reshuffled or lost), and if the grace period expires, the round transitions to an explicit "mission compromised" loss state naming which fragment went dark, rather than the group sitting there forever failing to converge on a solution nobody tells them is now impossible. That state needs to exist in the state machine from day one, not get bolted on later — "a player left" is not a rare edge case in a 10-12-person party setting with drinks and phones, it's a near-certainty over enough sessions.

Third: late joins and partial groups, which the prompt contract already called out as required but nobody's specified the actual behavior. I think the clean rule is that joining is only possible in the lobby, before the host locks the roster and starts the mission — once fragments are dealt, the roster is frozen, because there's no coherent way to hand a late arrival a fragment in a puzzle that was already balanced for a fixed player count. A late arrival after start sees an explicit "mission in progress, wait for the next one" state and can join the lobby for the next scenario. That's a real scope boundary worth locking now so design doesn't quietly try to build live re-dealing.

Fourth, and I think underrated: the "clues are structurally true by construction" mechanic is the single mechanical claim the whole product's uniqueness argument rests on, and it's also the easiest thing to accidentally violate at the content layer rather than the code layer. If clues are built from constrained tokens tied to a player's dealt fragment data, that's only as true as the scenario authoring is internally consistent — a hand-authored scenario pack with a data-entry mistake (two fragments that contradict each other, or a clue-token vocabulary that can't actually express the fact needed to solve the puzzle) breaks the "no deception, ever" promise at the exact place we're claiming it's guaranteed. That's not a hypothetical: hand-authored structured content with cross-references is exactly the kind of thing that silently drifts inconsistent. I'd want a scenario-validation step — schema validation plus a solvability check that confirms the puzzle's answer is actually derivable from the union of all dealt fragments using only the allowed clue vocabulary — treated as a build requirement, not a nice-to-have, because otherwise "no deception" is a marketing claim we can't actually back with anything mechanical.

Fifth, permissions and platform friction that will bite during App Store review, not just at runtime: MultipeerConnectivity requires `NSLocalNetworkUsageDescription` and `NSBonjourServices` entries, and as of recent iOS versions the local network permission prompt is a real, user-visible, sometimes-denied gate — if a guest denies it, MC advertising/browsing fails silently unless the app explicitly detects and surfaces that state. Given the core promise is "zero friction join," a first-time guest hitting an opaque "no sessions found" screen because they reflexively tapped "Don't Allow" on a system permission dialog is a first-impression killer. That needs a named error state that detects the permission denial specifically (not just a generic empty-state) and links to Settings.

Sixth, a privacy-adjacent point worth stating plainly even though the stakes are low: there are no accounts, but players do need some kind of display name to be attributable on the shared board and in the debrief reveal, and MultipeerConnectivity's peer advertising broadcasts a display name in the clear over local Bluetooth/Wi-Fi to any nearby listening device, not just intended session members. That's a mild but real local-network exposure (someone's chosen in-game name, not real identity data, assuming we don't default to device name) — the mitigation is simple, just never default the MC display name to the device's actual name (which often leaks a real first name like "Sam's iPhone" today), always use the in-app-entered game name instead, and don't collect or persist anything beyond that name on-device. Seventh, a smaller collision case: duplicate display names in one session (two people who both typed "Sam") create real ambiguity in a game where clue attribution matters — the board needs to show who said what, and "Sam: I have a key" is useless if there are two Sams. Cheap fix, needs to be a stated requirement: enforce uniqueness at join time, auto-suffix collisions.

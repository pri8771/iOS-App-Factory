Coming at this from what will actually break in review or at submission time, not from what sounds architecturally elegant: the biggest thing I want to lock now is the transport abstraction as a real protocol boundary, not a promise. I'd define a `GameTransport` protocol (advertise/browse, send/receive as an async event stream, connection-state callbacks) with exactly two conformances for this build — `MultipeerGameTransport` wrapping `MCSession`/`MCNearbyServiceAdvertiser`/`MCNearbyServiceBrowser`, and `MockGameTransport` for tests that can simulate N peers, injected latency, drops, and reconnects on command. Nothing above the transport — not `MissionViewModel`, not any view — should import `MultipeerConnectivity` directly. That's what makes the "cloud relay can be added later without a rewrite" requirement real instead of aspirational, and it's also the only way the locked acceptance bar ("rules engine unit-tested independent of transport, mocked latency/drop/reconnect") is actually achievable — you cannot get deterministic tests out of real MCSession instances on a CI runner, so the mock isn't a nice-to-have, it's the only way this ships with real coverage at all. I'd resist the temptation to build this as a fully generic pluggable-backend system with runtime configuration — one protocol, two conformances, constructor-injected. Anything fancier than that is gold-plating an MVP that's only ever going to ship one transport.

On GameKit versus MultipeerConnectivity, since the parent prompt named both: I'd explicitly rule GameKit out and say why, because otherwise someone downstream might "helpfully" suggest GKMatchmaker later. GameKit's real-time matchmaking requires `GKLocalPlayer` authentication — a Game Center sign-in — which directly conflicts with the locked "no accounts, zero friction" product promise for guests. MultipeerConnectivity is the correct choice and I don't think this needs more debate; I just want the reasoning written down so it doesn't get re-litigated.

For app architecture above the transport: single `@Observable @MainActor MissionViewModel` per device as already locked in the handoff, owned by a lightweight app-root coordinator that also owns the pre-session view models (name entry, discovery, pack picker, lobby) — those don't need to be one mega view model, they're simple and short-lived, so I'd keep them as separate small `@Observable` types rather than cramming everything into one god object. Navigation should be a plain `NavigationStack` with an enum-driven path for the pre-session flow, and then a hard cutover to the single persistent Mission screen once the mission starts — no `NavigationStack` push/pop during an active round, since that's exactly the "players navigate away from the board" failure mode design already ruled out. One thing I want called out explicitly because it's an easy mistake: MCSessionDelegate callbacks arrive on an arbitrary background queue, not main — every single one of them needs to hop via `Task { @MainActor in ... }` before touching `MissionViewModel`, and I'd want this enforced structurally (the `MultipeerGameTransport` wrapper does the hop internally and only ever emits values through a `@MainActor`-isolated async stream) rather than trusted to whoever writes the delegate methods, because that's precisely the kind of bug that passes every single-peer simulator test and then corrupts state intermittently with real multi-peer traffic — which we've already admitted we can't test at real scale.

Persistence: three genuinely different things need three different mechanisms, and conflating them is a common mistake. Player display name and settings (RF-tip dismissed, etc.) go in `UserDefaults` — trivial, fine. Scenario pack content ships as bundled JSON in the app bundle, versioned with a schema-version field so a future pack format change doesn't silently break old bundled data. The in-progress round state — the thing that has to survive force-quit/relaunch within the grace period — needs an actual on-disk snapshot, not just in-memory state: I'd write a `Codable` snapshot of the round (roster, board, own fragment, phase, remaining time) to the app's Application Support directory on every state-mutating event, using an atomic write (`Data.write(to:options:.atomic)`) so a crash mid-write can't corrupt the file, and on launch check for a live snapshot before showing onboarding. I'd explicitly avoid SwiftData here — it's built for structured, queryable, long-lived data, and this is a single ephemeral blob that gets deleted the moment a session ends; SwiftData would add a migration-and-schema surface area for zero benefit. Debounce the writes (e.g., don't write on every countdown tick, only on actual game-state changes) so this doesn't become a battery/IO problem during a live round.

Permissions and privacy: the only system permission this app needs is local network access, triggered the first time `MCNearbyServiceAdvertiser`/`Browser` runs — that requires both `NSLocalNetworkUsageDescription` and an `NSBonjourServices` array in Info.plist declaring the exact Bonjour service type string the app uses. If `NSBonjourServices` is missing or the string doesn't match what the code advertises/browses with, discovery silently fails and looks indistinguishable from the "no sessions found" empty state already designed for — that's worth flagging as a specific, easy-to-get-wrong build step, not just a permission checkbox. No camera, microphone, location, contacts, or photo library access is needed anywhere in this app, and I'd keep it that way — every additional permission is one more App Store review friction point and one more privacy-policy line to write for zero feature gain. One thing that's easy to miss and will get flagged at App Store submission if skipped: Apple now requires a `PrivacyInfo.xcprivacy` manifest declaring "required reason" API usage — `UserDefaults` is on that list — so this file needs to exist and be accurate, not bolted on at the last minute. On the App Store Connect privacy questionnaire, I'd actually push to answer "data not collected" for the whole app: the only personal-ish data point is a self-chosen display name, and it's broadcast peer-to-peer over local Bluetooth/Wi-Fi, never sent to a developer server, so under Apple's model that's not "collection" at all — that's a genuine, defensible "no data collected" privacy badge, which is real marketing value for a party game where people are naturally a little wary of "who's on my network," and I don't want that badge lost by someone reflexively checking boxes on the questionnaire out of caution.

On backgrounding, I want to be precise about a distinction I think earlier phases have slightly blurred: disabling `UIApplication.shared.isIdleTimerDisabled` only prevents the screen from auto-locking due to inactivity — it does nothing to stop a player from manually pressing the home button, swiping away, or getting a phone call, all of which background the app immediately and, since there's no supported background mode that fits a text game (no audio/VoIP/location justification), the OS gives roughly 30 seconds before likely suspension. That's fully consistent with the already-locked "mid-round departure triggers grace-period reconnect" design — I just want it stated plainly in the architecture doc that "idle-timer suspension" and "can't background" are two different guarantees, so nobody downstream assumes disabling auto-lock somehow protects against a player switching apps.

StoreKit: since a real purchase flow is should-have not must-have for MVP, I'd still architect the paywall placeholder against StoreKit 2's async APIs (`Product.products(for:)`, `Transaction.currentEntitlements`) from day one rather than stub it with fake local state, because retrofitting StoreKit into a screen that was built assuming a boolean flag is genuinely painful later — cheap to do right now, expensive to redo. No server-side receipt validation is needed given there's no backend; `Transaction.currentEntitlements` is sufficient for a local-only entitlement check, and restore-purchases can be a simple button that calls `AppStore.sync()`. Flagging clearly: this makes a live network call to Apple's servers, so it's one of literally two things in this entire local-first app that touch the internet at all (the other being nothing else) — that's fine since it's a deliberate user action, not something gameplay depends on.

Testing implications, since this is my hat: the rules engine (fragment dealing, clue-token validation, timer, solution confirm/decline state machine, solvability check) needs to be a plain Swift package/module with zero import of `MultipeerConnectivity` or SwiftUI, tested with XCTest (or the newer Swift Testing framework, either is fine, I'd just pick one and be consistent) purely against the mock transport and in-memory fixtures — that's the only way to get fast, deterministic, CI-runnable tests given we've already conceded real 10-12-device testing isn't executable in this pipeline. The scenario-pack validation gate should be an actual test, not a manual step: a test that loads the bundled JSON and runs schema + solvability validation for every player count 6 through 12, so a broken pack literally fails the build rather than shipping and being discovered by a host mid-party. Beyond that, I'd keep expectations honest rather than inflated: XCUITest can cover the single-device flows (name entry, discovery empty/permission-denied states, pack picker, settings) but cannot meaningfully exercise a live multi-peer round, so multi-device behavior stays a manual, small-scale (2-3 device) verification pass as already agreed — I don't want the final review implying automated coverage exists for something that structurally can't be automated here.

Dependency decision: zero third-party dependencies, full stop — everything above is achievable with Foundation, SwiftUI, MultipeerConnectivity, StoreKit 2, and XCTest/Swift Testing, all first-party and license-irrelevant. I don't see anything in this app's scope (no networking library, no JSON schema library needed since `Codable` plus a small hand-written validator covers it, no analytics SDK since there's deliberately no telemetry) that would justify pulling in an external package and its license/maintenance risk. On the ML/AR bonus points named in the parent prompt: I'd decline both for this app. There's no point in this design where either adds real user value — forcing an AR fragment-reveal gimmick or an LLM-generated scenario would directly undercut the locked "no procedural content, hand-authored and solvability-validated" requirement and the "silent, calm, structurally-true" tone the whole design system is built around. That's a deliberate no, not an oversight.

The single biggest risk I'd flag for tech_specs to inherit: the on-disk round-snapshot mechanism for relaunch-restore is new in this phase — nothing in the design_handoff actually specified how "force-quit and relaunch within grace period restores state" gets implemented, it was stated as a requirement without a mechanism.

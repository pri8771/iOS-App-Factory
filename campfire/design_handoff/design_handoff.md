**Claude (Frontend Engineer · Systems Thinker) — Round 1**

Coming into this phase with the frontend-engineer hat on, my job is to take everything the design_discussion phase already locked — six screens/states, the handoff→ready→revealed round machine, delayed target reveal, the recap card layout — and turn it into something a SwiftUI builder can actually implement without inventing anything mid-build. Let me lay out where I'd land.

On navigation architecture: I don't think this should be a `NavigationStack` with pushed routes. A pushed stack gives you an interactive back-swipe gesture for free, and that's actively dangerous here — someone swiping back mid-round would put the app in a state nobody designed for (round in progress, but nav says you're on the previous screen). I'd model this as a single root enum (`AppRoute: .entry, .setup, .session, .recap`) driving a `switch` in the root view, with the round screen itself owning its own internal `RoundPhase` enum (`.handoff, .ready, .revealed`) as `@State`/persisted model state, not as separate navigation destinations. That was already the design decision from the last phase; I'm just flagging the implementation consequence — whoever builds this needs to explicitly disable interactive pop everywhere in the session flow, not just "not add" back buttons, because SwiftUI's edge-swipe-to-go-back is enabled by default even with no visible back button.

On persistence and the state machine, this is where I want to be the most concrete, because "checkpoint at every transition" is easy to say and easy to half-implement. I'd model this with SwiftData: a `Session` model holding an ordered `players: [Player]`, `currentRoundIndex`, `activePlayerIndex`, `roundPhase` (the handoff/ready/revealed enum, persisted, not just in-memory `@State` — this is the detail that actually makes termination-survival real, because if `roundPhase` lives only in a SwiftUI `@State` var, relaunching after a force-quit will always drop you back to some default phase instead of the exact one you were in), a `targetHistory` array (needed for the least-recently-targeted rule to survive relaunch), and a `tally`. Every transition — handoff shown, reveal tapped, round advanced — needs an explicit synchronous `try? modelContext.save()` right after the state mutation, not a reliance on SwiftData's autosave, because autosave timing is not guaranteed to run before a kill signal. I'd also flag: on launch, the app needs a real "is there an in-progress session" check that routes straight into the session at the exact persisted phase, and that path needs its own test — it's a distinct code path from "starting fresh," and it's exactly the path that's easiest to leave unbuilt because the happy-path demo never exercises it.

The double-tap guard needs to be a first-class piece of state, not an afterthought — I'd add an `isTransitioning: Bool` (or make the phase transition itself atomic — set phase, save, then re-enable input) and have the reveal/advance button's action check-and-set that flag before doing anything, with the button itself also disabled while true so there's a visual affordance, not just a silent no-op tap. Same guard needs to cover the confetti/haptic trigger specifically, since that's the thing that's most visible if it fires twice.

For `isIdleTimerDisabled`, I'd scope it with `.onAppear`/`.onDisappear` on the round-container view rather than scattering toggles across child views, and explicitly reset it to `false` on every exit path from that container — including the early-exit-on-empty-pack path, which is easy to miss since it's not the "normal" exit.

Typography: SF Rounded (`.fontDesign(.rounded)`), heavy/black weight reserved for state words and names (PASS TO SAM, YOU HOLD), a calmer regular-weight system font for prompt body text, and an explicit truncation/line-limit rule for the two-column header — I'd say `minimumScaleFactor` plus a defined two-line max with tail truncation for the header specifically, tested against a long all-caps name at the largest accessibility Dynamic Type size, since "ALEXANDRIA" at AX5 in a two-column layout will not fit at any font size that's still legible without a defined fallback.

Component inventory I'd want locked: `GradientBackground(context:)` (drives the contextual warm/hot shift), `TwoColumnStateHeader` (handles the hide/reveal of the target name and the truncation rule above), `HandoffBanner` (auto-advance timer + tap-anywhere-to-skip, needs a `Task`-based timer that's cancelled if the user taps early — another place a stray uncancelled `Task` could leak state if not handled via `.task` scoped to the phase), `RevealButton` (owns the double-tap guard), `ConfettiOverlay` (I'd build this with `Canvas` + `TimelineView(.animation)` rather than pulling in a third-party confetti package, consistent with the "no third-party dependency unless it materially reduces risk" rule — a burst of 30-40 particles with random velocity/rotation is genuinely simple to hand-roll and keeps this fully first-party), `RecapCardView`, `ShareLinkButton`, and a single reusable `ErrorStateView(message:, retryAction:)` used for the empty-pack and render-failure cases so error handling doesn't get bespoke-implemented three different times.

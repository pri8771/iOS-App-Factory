Primary user and scenario: I want to anchor on the single hardest scenario, not the easiest one, because if it works there it works everywhere else — a group of 6-8 people at a slightly awkward gathering (family holiday, new roommates' first week, a work happy hour) where one person has downloaded the app moments earlier, has never seen the UI before, and has to confidently explain to a room full of strangers-to-the-app what's about to happen within about 10 seconds of tapping "start." That means the mode-select screen and the player-setup screen are not just entry points, they're the pitch — if the copy on those two screens doesn't make the rules obvious without narration, the "opens it and just plays" promise is already broken before round one starts.

Core loop, screen by screen, because I think this needs to be nailed down precisely now rather than left loose: launch → mode picker (3 modes, each with a one-line "what happens" description and a visible free/locked pack indicator) → player setup (count stepper + name entry, minimum 2, with a sensible soft-cap like 12-15 before the UI starts to strain) → round sequence (one full-bleed prompt card per turn, an unambiguous "who's up" indicator that's readable from arm's length since the phone is being angled toward whoever's turn it is, then an input/vote/reveal step depending on mode) → per-round tally moment (visible score or vote state update, small haptic-confetti beat) → next round or end-of-session trigger → recap card screen (stats + shareable image) → either replay/new session or exit. That's one linear spine, no side-quests, which matches the "one level deep" IA decision from product research and I want to keep it that way — the temptation during build will be to add a settings gear, a history tab, a pack browser as a separate top-level destination, and each of those is fine as long as none of them break the linear spine of an in-progress session. A player mid-round should never be able to accidentally navigate somewhere that loses round state.

The systems-thinking risk I want on record before anyone starts building: session state has to survive being backgrounded, because in the actual physical scenario this is built for, someone will get a text, or the phone will get bumped to the home screen, or a call will come in mid-pass. If the round/turn/tally state isn't durably persisted (not just in-memory) at every transition, the first real-world use of this app produces a broken session and the group just picks up their real phones instead — which is exactly the failure mode this app exists to prevent. I'd treat "backgrounding mid-round doesn't lose state" as a testable success criterion, not an implementation detail.

Second systems risk, tied to the player-count range: the reveal/confession and voting/ranking mechanics both imply some notion of "who is this prompt about," and that logic has to behave sanely at both extremes — 2 players (where "someone other than the current player" is forced) and a 12-15 player group (where naive random selection will visibly repeat the same 2-3 people and the room will notice within a few rounds). I don't think this phase needs to design the algorithm, but I want "no visibly repeated target player within a reasonable window, provably true at both boundary counts" written down now as a scope boundary that the build phase can't quietly skip.

Hard scope boundaries I'd draw: exactly the three locked mechanics (voting/ranking, reveal/confession, rapid binary-choice), no in-session settings screen beyond the essentials (haptics on/off maybe), no account, no network call anywhere in the core loop, no ads, no mode-specific recap card templates for v1 — one shared recap format that pulls the right stats regardless of which modes were played in a session. I'd also explicitly scope out "editing/reordering players mid-session" — if someone needs to add or remove a player, that should mean ending the session and starting a new one, not a mid-flight roster edit; that's a real feature some competitor apps have, but it multiplies the state-machine surface area for a first build and isn't in the original prompt's core workflows.

Measurable success criteria I'd hold this phase's decisions to: a first-time user can go from cold launch to an active round in under 10 seconds without reading instructions; a full 8-round session across all three modes can be played start to finish with the phone backgrounded at least once with zero state corruption; the recap card renders correctly on a physical device (not just simulator) and produces a ShareLink sheet with a completed share action; and at both a 2-player and a 12-player session, no player is ever the ambiguous "current player" and no reveal-mode target repeats twice within a 4-round rolling window. Those are the things I want a later verify phase to actually check, not just infer from the code compiling.

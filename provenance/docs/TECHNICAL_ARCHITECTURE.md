## Design Discussion

CONSENSUS: YES

Round two actually closed both open threads cleanly, and interestingly, both of them ended up switching or refining their own round-one stance rather than digging in.

On value tracking: Codex came down decisively on the side of the original spec — free tier does not get a working value ledger, full stop. The Value card still appears on every item's detail screen for free users (so the feature isn't hidden or a mystery), but it renders as a locked teaser with a one-line pitch and an upgrade action, never an actually-usable empty ledger. Codex's reasoning: letting free users log real dated value entries while only gating the chart creates a mushy tier split and lets people do all the "valuation tracking" work without ever feeling like they got the feature. Claude then closed the one gap Codex's answer left open — what happens on downgrade — by tying it to language the group already locked back in product_research ("value history remains viewable but can't be edited or regenerated without re-subscribing"): a never-subscribed user always sees the locked teaser; a subscriber's Value card comes alive and stays alive while entries exist; a downgraded user keeps their historical entries fully visible read-only, with "add" disabled and a message like "entries preserved — resubscribe to add more." That's not a new argument, it's stitching an already-agreed rule onto a card Codex had already agreed to gate, so there's nothing left in tension here.

On Recently Deleted: Codex reversed his round-one position and agreed it shouldn't live as a second scrollable section on the home screen next to Preserved/Locked, for exactly the reason Claude gave — two "your stuff lives somewhere else" modules on one screen recreates the confusion the group spent three rounds avoiding. Where they landed together: Recently Deleted becomes a quiet, always-available overflow/toolbar affordance right on Catalog Home (Codex said "Settings or an overflow path off the catalog toolbar," Claude specifically proposed the toolbar/overflow-menu version so it needs zero navigation away from home) — plus, Claude added a piece nobody had proposed yet that solves the real emotional moment directly: an inline "Item moved to Recently Deleted — Undo" toast the instant a delete happens, so regret is handled on the spot without anyone needing to visit a destination screen at all. Codex didn't get a chance to react to that specific undo-toast addition, but it doesn't conflict with anything he said — it's a strict improvement on the same agreed placement, not a competing idea.

They also converged in passing on tone: needs-attention/incomplete states should never use red or an alarming visual language — reserve that only for genuinely destructive confirmations — and use a muted, same-typographic-system treatment instead, so the catalog stays one calm archive rather than turning into a to-do list with guilt attached.

## Final Output

**Screen inventory:** App Lock gate (shown on cold launch and on return-from-background when device authentication is available; skipped entirely if no device passcode exists) → Catalog Home (single NavigationStack, no tab bar) → Capture sheet (camera-first, opens instantly, no pre-picker) → Quick Log finishing card (photo + title + six category chips, silent Vision suggestion highlight, Save is the only required action) → Item Detail → Add Condition/Provenance Entry sheet → Add Value Entry sheet (only reachable once Value tracking is unlocked) → Purchase Facts editor → Preserved/Locked read-only list (pushed from its Catalog Home section) → Recently Deleted list (reached via a quiet overflow/toolbar affordance on Catalog Home, not a scrolling home-screen section) → Export Preview (single-item and collection variants share one screen behind a mode toggle) → Paywall → Settings (app-lock toggle, backup/privacy statement, subscription management).

**Primary user flow:** Open app → unlock if applicable → tap Add → capture/pick photo (an Item row is persisted the instant the photo lands) → Quick Log card appears already populated with the photo, an editable "Untitled" title, and six category chips; Vision's suggestion silently highlights a chip if and when it resolves, never blocking; user types a title, optionally taps a category, taps Save → item now exists in the active catalog, which is the moment of relief the product has to deliver in well under a minute. Everything else — purchase facts, condition/provenance entries, value entries, export — is optional enrichment reached later from Item Detail, never forced before the first save.

**Catalog Home layout:** active items as photo-forward cards (cover photo, title, category, and a status affordance only when the computed effective-state model actually has something to say — nothing extra for a normal complete item); directly beneath, the permanently-visible Preserved/Locked section (collapsed, a couple of cover-photo peeks, a count, one tap to the full read-only list) rendered quietly so it doesn't compete for attention when empty; Recently Deleted lives as a small overflow/toolbar affordance on this same screen — always one tap away, but taking no scrolling real estate — and every delete action also surfaces an immediate inline "Item moved to Recently Deleted — Undo" toast so the moment of regret is handled without navigating anywhere.

**Value-tracking gate (resolved this round):** the Value card appears on every item's detail screen regardless of subscription — visibility of the feature's existence is never hidden — but for a user who has never subscribed it renders as a locked teaser (short description + redacted mini-preview + one upgrade action), not a usable empty ledger. Subscribing brings the card to life and lets entries be logged. On downgrade, previously-logged entries stay fully visible read-only ("entries preserved — resubscribe to add more"); locked/preserved items simply inherit the same read-only treatment they already have everywhere else. This is a new decision made in this phase, not an inference — the paywall/detail build should treat it as settled.

**State model per screen:**
- *Catalog Home:* empty (first-launch, calm "Add your first item" prompt), loading (brief local-fetch skeleton, no blank flash), success, error (SwiftData load failure gets a real recovery screen, never silent).
- *Capture/Quick Log:* camera permission denied (manual/library fallback), photo captured, classification pending/unavailable/low-confidence (chip shows no suggestion), photo write failed after metadata save (item marked incomplete, not pretended complete), interrupted/backgrounded/force-quit (draft resumes on relaunch).
- *Item Detail:* complete item, incomplete item (repair-access badge, only when active and not deleted/locked), locked item (read-only, upgrade CTA, never repair flow), deleted item (restore/purge only), each of the three cards with its own distinct empty/populated states (0 condition entries reads differently from 0 value entries; locked Value teaser is its own explicit state, not a generic empty view).
- *Export Preview:* unavailable (free tier, upsell), generating (real progress, off-main-thread), success preview, failed render (retry path), zero-item collection export (calm "nothing to report yet" page, never blank/malformed).
- *Preserved/Locked list:* populated read-only list, attempted-edit shows upgrade prompt.
- *Recently Deleted:* empty, populated, item nearing purge messaging.
- *Paywall:* normal state (real redacted export mock, real layout/typography), StoreKit product-fetch failure (retry, never a silently broken paywall).

**Visual direction:** near-black (not pure-black) neutral background, warm off-white primary text, one sparing accent color reserved for the suggested-category highlight and primary actions only; serif for item titles/section headers/export surfaces, system sans for body, metadata, dates, and numbers so those stay maximally legible; "archival" comes from restraint, whitespace, and quiet borders rather than low-contrast mood lighting — condition-photo timelines and the value chart are explicitly named as the two screens where legibility must win over atmosphere. Needs-attention/incomplete states use a muted, same-typographic-system treatment (e.g., a quiet amber dot or small label) — never red, which is reserved only for genuinely destructive confirmations — so the catalog reads as one calm archive rather than a scolding to-do list.

**Accessibility notes:** category chips announce suggested-vs-manual state distinctly to VoiceOver rather than reading as identical buttons; every effective-state badge (locked, incomplete, deleted) has a spoken label, never a silent icon; cover photos and timeline photos get distinct, dated VoiceOver labels ("condition entry photo from June 3, 2026," not "image"); the value ledger has a fully VoiceOver-navigable list form so understanding value history never depends on interpreting a chart visually; color is never the sole signal separating active/locked/incomplete/deleted; Dynamic Type at the largest accessibility sizes is specifically checked against Item Detail's three-card layout and Export Preview's paginated view, the two layouts most likely to break under long text wrapping.

## Final Output

**External design decision:** No external Claude Design pass. This handoff is builder-ready as written; the drafted Claude Design prompt (below) is archived as an optional, non-blocking reference for future marketing/App Store imagery only — it does not gate the build.

**Navigation model:** One root `NavigationStack` with a typed path enum (`.itemDetail(Item.ID)`, `.preservedList`, `.recentlyDeleted`, `.exportPreview(ExportMode)`, `.settings`) for archive drill-ins, plus `.sheet(item:)`-presented focused tasks (Capture/Quick Log, Add Condition Entry, Add Value Entry, Purchase Facts editor, Paywall). Rule: "drilling into the archive" = push; "a focused task you finish or back out of" = sheet. App Lock is a `.fullScreenCover` driven by a root-level `scenePhase` observer, rendered before the stack's content and paired with a single reusable privacy-overlay modifier applied at the root (not per-screen) so no screen — especially Export Preview — can leak sensitive content in an app-switcher snapshot.

**Screen-by-screen spec:**

- **App Lock gate** — full-screen cover on cold launch and return-from-background when device authentication exists; skipped entirely (never traps the user) if no passcode is set.
- **Catalog Home** — title, trailing Add button, overflow menu (Recently Deleted, Settings). Body: active photo-forward item cards (cover photo, title, category, at most one status affordance), then the quiet collapsed Preserved/Locked shelf (count + 2-3 thumbnail peeks, one tap to full list). No summary strip in the baseline. States: empty (first-launch, serif headline + one line of copy + "Add Your First Item," no illustration), loading (3-4 static muted skeleton cards, no shimmer), success, error (SwiftData load failure → real "Your catalog couldn't be opened" + Retry, never silent).
- **Capture (camera-first, full-screen)** → **Quick Log** finishing card — photo + focused "Untitled" title field + six category chips + Save, nothing else required. An `Item` row persists the instant the photo lands, before the sheet even finishes presenting. States: permission-denied (manual-entry fallback with placeholder photo slot, not a dead end), classification pending (chips render immediately, no spinner), classification resolved (one chip highlights with a soft ~250ms fade + "Suggested" caption, no pop/pulse), low-confidence/unavailable (chips stay neutral), photo-write-failed (item still saves as incomplete; inline non-blocking notice, never blocks Save). Backgrounding/dismissing the sheet never deletes the draft — it closes on top of an already-saved incomplete item, resumable on return.
- **Item Detail** — header (cover photo, title, category/subtype, condition grade, acquisition date, value summary if unlocked/populated), then three data-bearing modules in fixed order — Purchase Facts, Condition & Provenance (grade + dated timeline), Value (ledger first, type-differentiated, chart secondary) — each independently collapsible past ~3 entries with "View all N entries." Below the three modules sits an Export **entry-point CTA** (same card visual language, but a navigation affordance into Export Preview pre-set to single-item mode, not a fourth data track with its own empty-state machinery). Value card renders as a locked teaser (copy + redacted mini-preview + upgrade action) for never-subscribed users, live and editable for subscribers, and read-only-preserved with "entries preserved — resubscribe to add more" after downgrade. Locked/deleted items follow the effective-state precedence: deleted > locked > incomplete-with-repair-access.
- **Preserved/Locked list** — flatter card treatment than active cards to signal read-only, top explanatory line ("Your records are preserved. Upgrade to edit or add beyond your free limit."), attempted edit shows the upgrade prompt.
- **Recently Deleted** — reached via a quiet overflow/toolbar affordance on Catalog Home, never a home-screen scrolling section. Operational, not archival, visual tone: smaller rows, per-item "Deleted N days ago — permanently removed in N days" (never a single global banner), restore/purge actions. Every delete also fires an inline "Item moved to Recently Deleted — Undo" toast at the moment of deletion, independent of this screen.
- **Export Preview** — mode toggle (single item / collection) at top. States: locked/upsell (free tier), generating (real off-main progress, page-placeholder skeleton, toggle/back stay responsive), success (paginated preview + share/save in toolbar), failed (retry), zero-item collection (a real rendered sparse "nothing to report yet" page, never blank/malformed).
- **Paywall** — static single scroll: headline, redacted export mock (real serif headers, real photo placeholders, blurred numeric fields), short feature list, monthly/annual price buttons, restore-purchases link, legal footer. StoreKit fetch failure gets its own retry state, never a silently broken screen.
- **Settings** — app-lock toggle, backup/privacy statement, subscription management, no other surfaces.

**Design tokens (single canonical set):**

**State/interaction notes:** Effective state (deleted / locked / incomplete / normal) is one pure function computed per render from three independent facts (deletedAt, status enum, active-25 membership) feeding one `ItemDisplayState` enum that every card/badge/detail view switches over — never independent per-screen checks. ## Ios Architecture Review

CONSENSUS: YES

Round two closed the loop cleanly. Codex took an explicit, unhedged position on the camera question — system `UIImagePickerController(sourceType: .camera)` for capture, `PHPickerViewController` for the library fallback — for the same reason Claude argued: the product's edge is in what happens after the shutter fires, not in owning a bespoke camera stack, and a custom `AVCaptureSession` trades away Apple's decade of tuned focus/exposure/orientation handling for bugs that are only testable on real hardware anyway (Simulator has no camera either way). Both landed in the same place independently, so this is now a settled, named decision rather than something left for whoever writes the capture view first.

Codex also tightened something that falls directly out of his own "access-layer rules, not storage mutations" principle: Vision's classification result should live in transient capture-workflow state, not get written to the persisted `Item` the instant the classifier returns — otherwise a draft could flicker through multiple stored categories before the user ever taps Save, turning something that should be ephemeral UI state into noisy, hard-to-reason-about persistence churn. Claude explicitly agreed and folded it into the same "one feature model owns the whole capture state machine" discipline already on the table, adding that this state machine (draft created → media pending → suggestion available/absent → save-ready → incomplete) should be a named, diagrammed deliverable in tech_specs, not just an implied consequence of "good architecture."

Claude raised one new risk this round that Codex didn't get a chance to react to, but it doesn't conflict with anything he said — it's a specific implementation call inside territory both already agreed on (off-main PDF rendering, accessible export preview): don't use `PDFKit`'s `PDFView` for the in-app export preview, because its VoiceOver page-position announcement is unreliable, which would quietly violate the "Page 2 of 5" accessibility requirement locked back in design_handoff. The fix — generate the real shareable file with `UIGraphicsPDFRenderer`, but render the in-app preview as a paged native SwiftUI view (e.g. `TabView(.page)` over rasterized page images) where the accessibility announcement is fully controlled — is a refinement of an already-agreed plan, not a new fork.

Everything else — the App Group/SwiftData container plan with per-context thread separation, the repository/service protocol boundary, StoreKit 2's offline-safe entitlement caching with a launch-time listener, zero third-party dependencies, WidgetKit's debounced reload, photos-on-disk with a dedicated settings entity for currency/lock-preference, and the tiered testing strategy — was already aligned and nothing here reopened it.

## Final Output

**SwiftUI architecture recommendation:** MVVM-style, with `@Observable` (Observation framework) feature models marked `@MainActor` owning all UI-facing state — no `ObservableObject`/`@Published`. Views never talk directly to SwiftData, Vision, StoreKit, or the file system; everything routes through a small set of protocol-defined boundaries (`ItemRepository`, `ClassificationService`, `EntitlementProvider`) so every layer above storage/frameworks is testable with fakes. Capture is modeled as one explicit state machine owned by a single feature model — draft created → media processing pending → classification suggestion available/absent → save-ready → incomplete-on-failure — rather than several independent tasks fired from the view; this must be a named, diagrammed deliverable in tech_specs, not an implied side effect. Vision's classification result lives in transient capture-workflow state and is only written to the persisted `Item` if it's still the selected category at save time — it never causes an interim database write-then-flip. Subscription/record-state rules (active/locked/incomplete/deleted) are computed at read time from stored facts; nothing about entitlement or cap status is ever written back into the store as a mutation.

**Apple framework choices:** SwiftUI + Observation, SwiftData (App Group–backed container from day one, not retrofitted for the widget later), `UIImagePickerController(sourceType: .camera)` wrapped in `UIViewControllerRepresentable` for capture (explicitly chosen over a custom `AVCaptureSession` — the product's advantage is in what happens after the photo lands, not in art-directing the live camera chrome, and a custom capture session trades Apple's tuned focus/exposure/orientation handling for a bug surface only verifiable on real hardware), `PHPickerViewController` for library fallback (no photo-library permission needed at all), `VNClassifyImageRequest` for silent, coarse, confidence-floored category suggestion, `UIGraphicsPDFRenderer` for generating the actual shareable/saveable PDF (off-main, with a defined max photo-embed dimension), a native SwiftUI paged view (e.g. `TabView(.page)` over rasterized page images, not embedded `PDFKit`/`PDFView`) for the in-app Export Preview so VoiceOver page-position announcements ("Page 2 of 5") are fully controllable, StoreKit 2's modern `Transaction`/`Product` API with a long-lived `Transaction.updates` listener task started at app launch, WidgetKit with debounced/coalesced reload triggers on local writes, LocalAuthentication for app-lock, and Swift Charts reserved for the (deferred) value chart. No ARKit, no custom/trained Core ML model, no background execution APIs (no `BGTaskScheduler`, no push) — none of this product's scope needs them.

**Persistence and local data plan:** SwiftData store lives in an App Group container from initial setup. One main-actor `ModelContext` handles all UI-driven mutation; background work (PDF export, thumbnail generation) opens its own fresh `ModelContext` from the same `ModelContainer` rather than sharing a context across actor boundaries. No CloudKit-backed SwiftData in v1 — the repository protocol, not a CloudKit-shaped schema, is what protects the future sync roadmap. Data model stays flat: `Item` (identity, timestamps, status enum, deletedAt, cover-photo reference, title/category/subtype, purchase facts, current condition grade) plus child condition/provenance entries and value entries (each with a hard, constrained value-type enum and their own optional photo reference). Photos and thumbnails live on disk with stable item/entry-keyed paths, never blobbed into the database; a dedicated small settings/config entity stores fixed currency and lock preference explicitly at install time rather than re-deriving from `Locale.current` on every render, since re-derivation would make historical value entries ambiguous after a region change.

**Permissions/privacy plan:** `NSCameraUsageDescription` and `NSFaceIDUsageDescription` only — no photo-library usage string needed given the PHPicker choice. No location, contacts, microphone, or network permissions. A Privacy Manifest (`PrivacyInfo.xcprivacy`) declaring required-reason API usage (UserDefaults, file-timestamp APIs) is a concrete, tracked build task, not an afterthought discovered at submission. Root-level privacy overlay on `scenePhase` transitions, temp export files cleaned up after the share flow completes/dismisses, and widget/app-switcher surfaces never show raw values outside the same privacy discipline as the rest of the app.

**Dependency/license decision:** Zero third-party packages. Every requirement in this build is covered by first-party Apple frameworks, which makes the permissive-license check trivial — there's nothing added to review.

**Testing implications:** Pure-function unit tests (hand-built fixtures, no UI) for `effectiveState` precedence and active/locked partitioning, covering every locked edge case (tied timestamps, restore-while-at-cap, deleting a locked item promoting the next-oldest). Repository tests run against real SwiftData with `ModelConfiguration(isStoredInMemoryOnly: true)` plus temp file directories, so photo lifecycle and purge cleanup are verified against real engine behavior, not mocked into fiction. StoreKit purchase/restore/downgrade/offline-startup flows tested via `SKTestSession` against an injectable `EntitlementProvider`. A small set of end-to-end UI tests reserved only for genuine system-integration flows: capture-to-save timing, interrupted/backgrounded capture resuming correctly, PDF export producing a valid file at 0/1/many items, VoiceOver completing the full core loop, and Export Preview's page-position announcement specifically (since that's the one place a plausible-but-wrong implementation choice — `PDFView` — would silently fail an already-locked accessibility requirement).

## Tech Specs

Round 2 closed almost everything. Both Codex and I now explicitly want `Decimal` over minor-unit `Int` for money (Codex's reasoning landed harder than mine — this isn't a payment rail, it's a multi-source valuation ledger, and the real risk isn't float drift, it's one lane silently treating a raw `Int` as cents while another treats it as dollars three files away), and the richer `PhotoAssetRef` struct with pixel dimensions over a bare path string (lets export decide downsample-vs-embed without re-opening the file). ## Final Output

**Architecture overview:** MVVM with `@Observable`, `@MainActor`-bound feature view models owning all UI-facing state — no `ObservableObject`/`@Published`. Views never touch SwiftData, Vision, StoreKit, or the file system directly; everything routes through protocol boundaries (`ItemRepository`, `ClassificationService`, `EntitlementProvider`, `PhotoStore`, `ExportService`, `AppLockService`) so every layer above storage/frameworks is fake-able in tests. Capture is one named, typed state machine (`CaptureState`) owned exclusively by `CaptureViewModel` — Vision's suggestion lives only in transient capture state and is written to the persisted `Item` only if still selected at Save. All subscription/record-state rules (active/locked/incomplete/deleted) are computed at read time from stored facts via pure functions (`computeDisplayState`, `partitionActiveAndLocked`) — nothing about entitlement or cap status is ever written back into storage as a mutation. Swift 6 strict concurrency checking is on from day one; the `Sendable` conformances already threaded through the contract only mean something under strict mode, so this was effectively decided the moment the types were written.

**File/module layout:** `App/` (entry point, root `NavigationStack`, `ModelContainer` bootstrap against the App Group container, scenePhase-driven lock gate + `PrivacyOverlayModifier`) · `Domain/` (all `@Model` types, enums, and every repository/service protocol — zero SwiftUI imports) · `Data/` (concrete SwiftData-backed `ItemRepository`, on-disk `PhotoStore` implementation) · `Services/` (`VisionClassificationService`, `StoreKitEntitlementProvider`, `PDFExportService`, `LocalAuthenticationLockService` — each a concrete conformance to a `Domain` protocol) · `Features/Catalog`, `Features/Capture`, `Features/ItemDetail`, `Features/Export`, `Features/Paywall`, `Features/Settings` (View + ViewModel per folder) · `Widget/` (separate target, read-only App-Group access through the same repository protocol) · `Tests/DomainTests`, `Tests/DataTests`, `Tests/ServiceTests` (Swift Testing, `@Test`/`#expect`), `Tests/UITests` (XCTest/XCUITest, since that's still the underlying automation API). Rule: nothing in `Features/*` redeclares a cross-cutting type — everything shared lives in `Domain/` and gets imported.

**Data models:** `Item` (`@Model`, id, title, `Category`, subtype, `ItemStatus`, `createdAt`, `creationSequence: Int64` as the mandatory active-25 tiebreak, `deletedAt`, `coverPhoto: PhotoAssetRef?`, `conditionGrade: ConditionGrade?`, `purchaseFacts: PurchaseFacts?`, cascade-deleted `[ConditionEntry]`/`[ValueEntry]`); `ConditionEntry` and `ValueEntry` as child records, each with an optional `PhotoAssetRef`; `ValueEntry.amount` and `PurchaseFacts.purchaseAmount` are `Decimal`, never `Double` or minor-unit `Int`; `AppSettings` singleton row stamped once at install (currency, app-lock preference, recovery-window length) rather than re-derived from `Locale.current` on every render. **Persistence strategy:** SwiftData store lives in an App Group container from initial setup, not retrofitted for the widget later. One main-actor `ModelContext` for all UI-driven mutation; background work (PDF export, thumbnail generation) opens its own fresh `ModelContext` from the same `ModelContainer`. No CloudKit-backed SwiftData in v1 — the repository protocol, not a CloudKit-shaped schema, protects the future sync roadmap. Photos and thumbnails live on disk under stable item/entry-keyed relative paths (never blobbed into the database), referenced via `PhotoAssetRef` (path + pixel dimensions) so export can decide downsample-vs-embed without re-opening the file. `ItemRepository.loadActiveAndLocked` is contractually required to fetch all non-deleted items and delegate to `partitionActiveAndLocked` — it may never reimplement that split as an independent query, closing off the exact "two independently-computed truths drift apart" bug class this app has been fighting since detailed_discussion. Same discipline applies to entitlement: `isSubscriptionActive(_:)` is the only legal way to turn `EntitlementState` into the plain bool the domain layer expects.

**Testing strategy:** Swift Testing (`@Test`, `#expect`, parametrized `@Test(arguments:)`) for `DomainTests`/`DataTests`/`ServiceTests`; XCTest/XCUITest reserved for UI automation only. Pure-function tests exhaustively cover `computeDisplayState` precedence (deleted > locked > incomplete > normal) and `partitionActiveAndLocked`, including tied `creationSequence`, restore-at-cap, and delete-of-a-locked-item-promotes-next-oldest.

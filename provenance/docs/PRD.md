## Detailed Discussion

Round 3 actually closed it, and closed it well — both of them converged on essentially the same third option independently, from different angles. Codex proposed it directly: don't interleave locked items into the active list and don't banish them to a separate destination either — put a clearly-labeled "Preserved Items" section right on the main catalog screen, collapsed with a couple of cover-photo peeks and a count, tap through to a full read-only list. Claude arrived at the same place by actually simulating the failure mode: since the tiebreak rule is "oldest stays active" and lists sort newest-first by default, inline-in-place would mean a downgraded user's *most recent* purchases are the ones sitting dimmed at the top of their list, burying the 25 items they can actually still use — that's a concrete, specific reason inline-in-place is worse than either of them gave it credit for, not just a stylistic preference.

Claude also added a genuinely new and useful piece underneath the IA decision: whether an item is "active" or "locked" should be computed at read-time (oldest 25 by creation order, stable tiebreaker) rather than stored as a persisted flag, because a stored flag means every mutation — delete, resubscribe, downgrade, restore — needs its own bookkeeping to keep it correct, and that's exactly the shape of bug that leaves someone with only 24 usable items until they force-quit. Computed-not-stored makes that class of bug structurally impossible. That naturally answered a question nobody had asked yet — yes, a user can delete an item sitting in the locked/preserved section without resubscribing, since deletion is data-minimization, not enrichment, and doing so should just automatically promote the next-oldest locked item into the active set with no special-case code. And the naming discipline matters too: this preserved/locked section is permanent and never auto-purges, explicitly distinct from the soft-delete "recently deleted" bin, so an anxious user post-downgrade can't mistake "frozen but safe forever" for "on a countdown."

## Final Output

**Resolved requirements:**
- *Provenance* means user-entered acquisition/history context (source, acquisition date, seller/location, optional note and photo) plus dated follow-up notes over time — explicitly not document verification or chain-of-custody proof.
- *Condition* is a hybrid model: a current-state grade field on the item plus a dated timeline of condition entries (note, optional photo, optional "condition changed" marker) — neither a grade alone nor a pile of notes alone is sufficient.
- *Value entries* carry amount, date, a constrained valuation-type enum (purchase price / appraisal / dealer offer / market observation / insured value), and a source note. This is locked as a hard constraint for whoever designs the future value chart: entries must be visually distinguished by type, never plotted as one undifferentiated connected line, or the "don't conflate value semantics" discipline gets undone by its own visualization.
- Draft capture: the instant a photo is taken, a real persisted `Item` row is created (placeholder title "Untitled", category "Other"), not held in a separate ephemeral staging mechanism. This is the only way to guarantee zero data loss without inventing a second persistence path, and it's an accepted tradeoff that abandoned drafts become real cataloged rows the user has to notice and clean up.
- Photo failure after metadata save, and never-confirmed drafts, are the same underlying problem (an incomplete record in the store) and share one `status` enum on Item: draft / incomplete / complete. One shared "needs attention, tap to finish" list-card treatment covers both, rather than two independently-invented half-finished-item mechanisms.
- Vision classification shows nothing rather than a low-confidence guess; requires an explicit label-to-category mapping table for the six-category enum decided at build time, not improvised.
- App-lock via Face ID/passcode, with an explicit escape hatch: if the device has no passcode set at all (so `LocalAuthentication` can't fall back), the lock is skipped entirely rather than trapping the user out of their own catalog.
- PDF export renders off-main-thread with a defined max embed dimension per photo, for both responsiveness (loading state must be real, not a frozen UI) and file size.
- Currency is a single fixed device-locale unit, no per-item override. Entry dates reject future dates but allow arbitrary backdating.
- **Free-tier downgrade / over-cap items:** the active 25 (oldest-by-creation-order, with a stable secondary tiebreaker — insertion order or row ID, since raw Date equality isn't fine-grained enough to guarantee a deterministic test) render as a clean, fully-interactive primary list with zero locked clutter. Everything beyond the cap lives in a permanently-visible, clearly-labeled "Preserved/Locked" section on the same main catalog screen (peek thumbnails + count, tap through to a full read-only list) — never interleaved as dimmed rows in the active list, never moved off to a hidden destination, and never described or behaving like the soft-delete "recently deleted" bin (no countdown, no auto-purge, no "empty" action).
- Active-vs-locked status is computed at read time from subscription state + item count, not stored as a persisted flag, so deletes/resubscribes/downgrades/restores can't desync it.
- Item creation is blocked once a user is at-or-above 25 active items, regardless of how they got there (organic growth or downgrade) — no loophole where a new item goes active while older items sit orphaned.
- A user can delete an item sitting in the locked/preserved section without resubscribing; doing so promotes the next-oldest locked item into the active set automatically, with no special-case logic required.
- Privacy: values are hidden from widgets, and that same discipline extends to app-switcher snapshots and notification previews — a privacy overlay swaps in on the background `scenePhase` transition.

**Edge cases:** device with no passcode set (skip lock, don't trap); Vision returns no confident mapping (show nothing); photo write fails after metadata write (mark incomplete, don't pretend whole); app backgrounded/force-quit mid-capture (draft already persisted, resumable); two items with identical creation timestamps (stable secondary sort key); downgraded user at the cap tries to create a new item (blocked, same upsell prompt as organic cap-hit); downgraded user deletes a locked item (promotes next-oldest automatically); PDF export with many high-res photos (downsampled, off-main-thread, must not freeze UI); soft-deleted item still logically counts toward nothing (excluded from the 25-cap math entirely).

**Data and privacy implications:** all core data stays on-device, no cloud account, no remote classification calls, no analytics transmitting item details or photos; app is responsible-user-content-only — no redaction/OCR/content moderation of photos, stated explicitly rather than left implicit; exported temp PDF files must be cleaned up after the share sheet completes or is dismissed; standard encrypted device backup coverage should be explicitly confirmed and communicated to the user (carried from earlier phases); locked/preserved items remain fully present in local storage and in backups — restriction is a UI/access-layer concept, never a storage-layer deletion.

**Risk register:** Vision misclassification on niche items — mitigated by confidence floor + silent no-suggestion fallback, never blocking or misleading. Manual value-entry fatigue leading to sparse/stale value history — named as a real limitation, not solved in this build; the chart's near-empty state and the type-differentiated encoding are the mitigations available at this layer. Stored-flag desync bugs on the active/locked boundary — mitigated by making the split computed, not persisted. Confusion between "preserved/locked" (permanent, safe) and "recently deleted" (time-limited, at risk) — mitigated by deliberately distinct naming, visuals, and behavior (no countdown/purge affordance on the locked section). Main-thread blocking or oversized files during PDF export — mitigated by off-main-thread rendering and a defined image-dimension ceiling. Privacy leakage via widget/app-switcher/notification snapshots — mitigated by a scenePhase-driven privacy overlay applied consistently across all three surfaces, not just widgets.

**Final assumptions:** a fixed six-category enum with shared schema and free-text subtype (no per-category logic); one photo per condition/value entry in v1 (named limitation, not silently dropped); single fixed currency per install; no content moderation of user photos; the value chart itself (with type-differentiated rendering) is out of scope for this slice but constrained now for whoever builds it; the "Preserved/Locked" section is a same-screen, always-visible feature of the main catalog, not a settings-buried or fully separate destination — this is now settled IA, not an open question for the design phase to re-litigate.

## Final Output

**Must-have features (each with a user story + acceptance bar):**

1. **Catalog home with four legible record states.** *Story:* a collector needs to instantly tell what's active, preserved/locked, needs attention, or at-risk-of-permanent-loss, so nothing ever feels like it silently vanished. *Acceptance:* empty/loading/success/error states on the main list; a permanently-visible "Preserved/Locked" section (peek thumbnails + count, never interleaved as dimmed rows); a single computed "effective state" per item with strict precedence — **deleted overrides everything** (only restore/purge, no repair, no lock messaging) → **locked overrides incomplete** (read-only + upgrade CTA, never the repair flow) → **incomplete-badge-with-repair-access** only when the item is active and not deleted; no item shows contradictory affordances.

2. **Fast capture with silent, free, universal classification assist.** *Story:* someone logging a purchase right after acquisition needs to save in well under a minute with zero forced steps. *Acceptance:* photo + title + category alone save it; a real Item row is created the instant the photo is taken (draft/incomplete/complete); Vision pre-fills the category chip silently for every user regardless of subscription tier, shows nothing below its confidence floor, never blocks or delays save, and carries zero premium framing (no lock icon, no upsell copy) anywhere near it; interrupted/backgrounded/force-quit capture resumes the same draft.

3. **Item detail with three separated enrichment tracks.** *Story:* reviewing a record later needs to feel like a dossier, not a dumped feed. *Acceptance:* purchase facts, condition/provenance timeline (grade + dated entries), and value-entry ledger render as visually distinct modules that never merge; cover photo is distinct from timeline photos; 0/1-entry value states have an intentional near-empty design, not a generic empty view; locked items remain fully viewable read-only.

4. **Value chart — correct and type-differentiated, not yet interactive.** *Story:* the subscription has to keep earning its keep as the collection ages, and the chart is the feature that actually delivers "grows with your collection." *Acceptance:* entries render with distinct visual encoding per valuation type (purchase/appraisal/dealer offer/market observation/insured value) — never one undifferentiated connected line; correctness and honesty are must-have now, interactivity/animation/zoom are explicitly deferred, not required for this build.

5. **Soft-delete, recovery, and uncapped restore.** *Story:* mistakes must be reversible, and getting an item back can never feel like it's being held for ransom. *Acceptance:* delete moves an item to Recently Deleted (visually/behaviorally distinct from Preserved/Locked — no countdown or purge affordance on the locked section, only on this one); deleted items are excluded from the 25-cap math entirely; restore is never blocked by the active-item cap — it re-enters the pool and lands active-or-locked purely from the existing computed read-time model, surfaced with an explicit message when it lands locked ("Restored — preserved but requires Provenance+ to edit").

6. **App-lock and passive-surface privacy.** *Story:* a list of valuables and what they cost shouldn't be casually visible. *Acceptance:* Face ID/passcode gates entry when device authentication exists; skips the lock entirely (never traps the user) when no passcode is set; a privacy overlay swaps in on background `scenePhase` transitions across the app, the widget, and app-switcher/notification surfaces alike.

7. **Local PDF export, single-item and collection, structurally correct.** *Story:* the record has to be handoff-ready without the app. *Acceptance:* both single-item and full-collection export exist; verified correct at zero/one-sparse/one-full item counts; correct photos, dates, condition/value history, and purchase facts in the right sections; off-main-thread rendering with a defined max photo embed dimension and a real loading/error/success state; final premium typography is explicitly deferred, structural correctness is not.

8. **StoreKit 2 gating with a real, desirable paywall preview.** *Story:* free users need a genuinely usable ledger, and the upgrade ask has to sell a real artifact, not an abstraction. *Acceptance:* 25-item free cap enforced deterministically (oldest-by-creation-order + stable tiebreaker); creation blocked at-or-above 25 active items regardless of cause; the paywall screen renders an actual mock export layout — real section headers, real photo placement, real typography — with sensitive content blurred/redacted, so the thing being sold is visibly the thing the user's data already supports; downgrade hides via the computed active/locked split, never deletes; resubscribe restores full access with no migration step; subscription-status is read through an injectable/testable abstraction so purchase/restore/downgrade behavior isn't only verifiable via a live sandbox click-through.

9. **Widget with distinct free/paid states.** *Story:* the home screen widget must reflect subscription reality without leaking private values. *Acceptance:* free tier shows item count, paid tier shows collection value; refreshes are debounced/coalesced on local write events (not fired per-save) to respect WidgetKit's background reload budget; values respect the same privacy-overlay discipline as the rest of the app; tap-to-open only — no deep-linking, no in-widget actions.

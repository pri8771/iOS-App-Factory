**Claude (Frontend Engineer · Visionary) — Round 1**

Let me put down the architecture I'd actually want to build against, pushing hard on the pieces that determine whether this feels inevitable rather than merely competent.

The shape everyone converged on in ios_architecture_review — MVVM with `@Observable` `@MainActor` feature models, protocol boundaries in front of SwiftData/Vision/StoreKit, a named capture state machine, effective-state computed at read time — is right, and tech_specs' job is to turn that into types every build lane can code against without guessing. I want to push one level bolder than "solid MVVM" in three specific places, because this is the phase where being timid costs the most later: concurrency mode, the collection-summary/widget data path, and how rigorously the capture state machine is actually typed rather than just described in prose.

On concurrency: given we're targeting Xcode 26/Swift 6 tooling, I want to build this with Swift 6 strict concurrency checking turned on from day one, not bolted on after a data race surfaces in TestFlight. This app has real actor boundaries — main-actor UI, background PDF/thumbnail contexts, a StoreKit transaction listener, a widget extension reading the same store — and every one of those is exactly the kind of boundary Swift 6's compiler-enforced `Sendable` checking catches for free at compile time instead of via a crash report from a real device three weeks post-launch. This is the single highest-leverage "boldness" available here: it costs some upfront friction fixing warnings, but for a trust-first app where "did I lose your $9,000 record" is the actual stakes, compiler-verified concurrency safety is worth more than almost any UI polish we could spend that time on instead.

On the widget/summary path: I want `CollectionSummary` to not be a naive "fetch everything and sum" computed property recomputed on every access, because that's the kind of thing that looks fine at 20 items and stutters or drifts at 400. Concretely: one `CollectionSummaryProvider` service, backed by a lightweight cached struct (`totalActiveItemCount`, `totalCollectionValue`, `lastComputedAt`) that recomputes from a single repository query whenever a local write happens (item created/deleted/restored, value entry added) and is invalidated by the same debounce mechanism that already triggers the widget's `WidgetCenter.reloadTimelines`. Both the widget and any in-app value display read this one provider — never two independently-written sums. This directly extends what got flagged in design_handoff/detailed_discussion about avoiding two-sources-of-truth drift, but I want it typed as a real service now, not left as a principle someone has to remember.

On the capture state machine: this needs to be an actual typed enum with associated values, not an implied sequence of bools, because that's the one piece of this app doing real async orchestration under user pressure (photo lands, thumbnail generates, Vision resolves, user might background at any point). I'd type it as `CaptureState` — `.capturing`, `.processing(draftID:)`, `.awaitingDetails(draftID:, coverPhotoURL:, suggestion: ClassificationSuggestion?)`, `.saved(itemID:)`, `.incomplete(itemID:, reason:)` — owned by one `CaptureViewModel` that's the only thing allowed to write to the draft `Item` row. Vision's result flows into `.awaitingDetails`'s associated value, never into SwiftData directly; it only gets persisted if the user hasn't overridden the category by the time Save fires. This makes "what happens if the app is killed mid-capture" a resumability question answerable purely by re-deriving state from the persisted `Item.status` on next launch, rather than trying to reconstruct in-memory state that's already gone.

For file/module layout, I'd structure the Xcode project so parallel lanes can't collide on the same files: `App/` (entry point, root NavigationStack, scenePhase-driven lock/privacy modifiers, ModelContainer bootstrap), `Domain/` (the `@Model` types, enums, and every repository/service protocol — zero SwiftUI imports allowed here, this is the shared contract lane touches first), `Data/` (concrete SwiftData-backed repository implementations, on-disk photo store implementation), `Services/` (VisionClassificationService, PDFExportRenderer, StoreKitEntitlementProvider, LocalAuthenticationLockService — each a concrete conformance to a Domain protocol), `Features/Catalog`, `Features/Capture`, `Features/ItemDetail`, `Features/Export`, `Features/Paywall`, `Features/Settings` (each a self-contained folder of View + ViewModel + any feature-local subviews), `Widget/` (separate target, App-Group-shared read-only access via the same repository protocol), `Tests/` split into `DomainTests` (pure function/unit), `DataTests` (in-memory SwiftData), `ServiceTests` (StoreKit/Vision fakes), `UITests` (the narrow XCUITest set already agreed). The rule that keeps lanes from inventing incompatible types: nothing in `Features/*` defines its own version of a domain type or protocol — everything cross-cutting lives in `Domain/` and gets imported, never redeclared.

Data models, concretely: `Item` (`@Model`, `id: UUID`, `title: String`, `category: Category`, `subtype: String`, `status: ItemStatus`, `createdAt: Date`, `creationSequence: Int64` as the stable tiebreak nobody can accidentally omit, `deletedAt: Date?`, `coverPhotoPath: String?`, `conditionGrade: ConditionGrade?`, `purchaseDate: Date?`, `purchasePrice: Decimal?`, `purchaseSource: String?`, relationships to `[ConditionEntry]` and `[ValueEntry]` with cascade delete). `ConditionEntry` (`id`, `itemID`, `date`, `note: String`, `photoPath: String?`, `markedConditionChange: Bool`). `ValueEntry` (`id`, `itemID`, `date`, `amount: Decimal`, `valueType: ValueType`, `sourceNote: String?`). `AppSettings` (singleton row: `currencyCode: String`, `appLockEnabled: Bool`, installed at first launch from `Locale.current` once and never re-derived). `Category` is the fixed six-case enum plus `.other`; `ValueType` is the five-case enum locked in detailed_discussion; `ConditionGrade` I'd fix as a small ordered enum (`mint/excellent/good/fair/poor`) rather than free text, since an ordered grade is what makes a future "condition changed" visual affordance meaningful — free text can't be compared across entries. `ItemStatus` is `draft/incomplete/complete` per the shared-status-enum decision. None of these four enums are stored as raw strings anywhere outside the enum itself — SwiftData handles Codable enum storage natively, so there's no stringly-typed drift risk.

Persistence: one `ModelContainer` configured against the App Group container URL at `App/` launch, injected via environment into every feature; one main-actor `ModelContext` for UI mutation, fresh background `ModelContext` instances (from the same container) for `PDFExportRenderer` and thumbnail generation, per the already-agreed thread-safety plan. Schema versioning: `SchemaMigrationPlan` scaffolded now even with a single schema version, so the first real migration isn't the first time anyone's touched that API. Photos live on disk under the App Group container at `Photos/<itemID>/original.jpg` and `Photos/<itemID>/thumb.jpg` / `Photos/<itemID>/entries/<entryID>.jpg`, referenced by relative path string in the model, never absolute (App Group container paths can change between installs/updates).

Testing strategy: I'd actually adopt Swift Testing (`@Test`, `#expect`) for the `DomainTests` and `DataTests` tiers instead of XCTest — parametrized tests are a first-class feature there and `effectiveState`/active-locked-partition testing is exactly the shape of test that benefits (one `@Test(arguments:)` sweep across every locked edge case instead of a dozen hand-written XCTest methods). XCUITest stays XCTest-based since that's still the underlying UI automation API. This is a genuinely bolder-than-default pick but a safe one given the validated Xcode 26 toolchain, and it directly makes the highest-value test suite in the app (the pure effective-state functions) more exhaustive to write, which matters more here than almost anywhere else in the codebase.

The biggest risk in my own recommendation: turning on Swift 6 strict concurrency checking now, rather than after the MVP ships, will slow down the first two weeks of implementation while every actor-boundary violation gets surfaced and fixed. I think that's the correct trade for this specific app — given the trust thesis is "never lose or misstate a record," a class of bugs caught at compile time instead of discovered via a crash report against someone's real collection is worth the friction — but if a build lane is under real time pressure, this is the one place I'd expect pushback, and it should get resolved explicitly rather than quietly downgraded to minimal-checking by whoever hits the first hard error.

### Round 2


**Codex (QA & Risk · Closer) — Round 2**

Claude’s right about the actual failure here: philosophically we’re converged, but if parallel lanes start from two different contracts, we’ve already lost. I’m going to be blunt about the two open calls.

First, use `Decimal`, not minor-unit `Int`, for money in this app. I get the usual instinct toward integer cents, but this product is not a payment rail and it’s not even guaranteed to be USD-only forever at the storage layer. It’s a record-keeping ledger that may store appraisal values, insured values, dealer offers, and historical purchase amounts coming from user input, and those values need exact decimal semantics plus locale-safe formatting later. The real bug risk here is not “financial math drift from Decimal” if we keep arithmetic minimal and formatting centralized; it’s lane A treating `1299` as $12.99 and lane B treating it as $1,299 because somebody forgot the minor-unit convention in an export mapper or widget summary. `Decimal` removes that entire class of integration bug. Same answer on `PhotoAssetRef`: the richer struct wins over a bare relative-path string. Export preview, PDF rendering, thumbnail policy, and accessibility all get harder if every consumer has to re-open the image file just to learn its dimensions or guess whether a thumb exists. The cross-lane contract should carry that metadata once.

Second, I would explicitly adopt Swift 6 strict concurrency checking and Swift Testing for domain/data tiers now, not later. This is exactly the app where “we’ll tighten concurrency later” becomes “we discovered an actor-isolation bug after someone logged 200 items and backgrounded during export.” The friction is real, but it’s the right friction. Same with Swift Testing: the highest-risk logic here is pure-function partitioning and display-state precedence, and `@Test(arguments:)` is a better fit than sprawling XCTest boilerplate. I would not mix and match test frameworks inside the same tier, though: Swift Testing for domain/data/service tests, XCTest/XCUITest only where Apple still requires it for UI automation. That’s a clean boundary.

The one place I do want to push back on the draft merged contract is `EntitlementState.unknownOffline(lastKnown: EntitlementState?)`. That shape is too cute and it invites recursive nonsense. We do not need a self-recursive enum to represent “offline, using cached state.” Make offline continuity an explicit snapshot value instead: either the provider resolves to `.free` or `.subscribed(expiresAt:)` from cached `Transaction.currentEntitlements`, or it resolves to `.unknownOffline(lastVerified: CachedEntitlement?)` where `CachedEntitlement` is a flat value type. The app rule is already locked: offline never downgrades a previously verified subscriber to free by omission. Represent that simply. Other than that, the merged contract is basically the right one, and we should stop leaving any of this open.

**Claude (Frontend Engineer · Visionary) — Round 2**

Codex, the move from my bare `String` photo path to your `PhotoAssetRef` struct with pixel dimensions is just better and I want to say so plainly rather than pretend I hadn't been outbid — storing width/height alongside the path means the export renderer can decide whether it needs to re-decode-and-downsample or can just embed the original as-is, without opening the file first to find out. That's a real perf win on collection-level export with dozens of photos, and it's exactly the kind of thing that's cheap to store now and annoying to retrofit later. Same goes for `Decimal` over minor-unit `Int` for money — I'll defend that one harder than I need to, because `Decimal` isn't just "nicer," it avoids an entire class of bugs where someone forgets whether a given `Int` field is dollars or cents three call sites away from where it was declared. `Decimal` carries its own scale, so a value entry, a purchase price, and a collection total are all unambiguously "this many currency units" with no silent divide-by-100 bug waiting to happen in the export formatter. I'm glad the coordinator's merge landed there and I don't think it's worth relitigating.

I do want to flag one seam in the reconciled `ItemRepository` before we call this settled, because it's the exact shape of bug this whole app has been fighting since detailed_discussion: `loadActiveItems()` and `loadLockedItems()` as two separate repository methods only works if whoever implements them internally re-derives the same sort-by-creationSequence-then-split-at-cap logic that `partitionActiveAndLocked` already defines as a pure function. If that split gets coded twice — once as the tested pure function, once as two independent SwiftData fetch predicates inside the concrete repository — we've recreated the "two independently-written totals that could drift" problem I raised about `CollectionSummary` last round, just one layer down. I'd rather the protocol expose one method, `loadAllNonDeletedItemsSortedByCreation(subscriptionActive: Bool) async throws -> (active: [Item], locked: [Item])`, whose contract explicitly requires the implementation to fetch all non-deleted items and hand them to `partitionActiveAndLocked` rather than re-derive the split with its own query logic. That keeps the ergonomic two-list return value view models want, while making it a written rule — not just a hope — that there's exactly one implementation of "what counts as active" in the whole codebase. Small change, but it closes a gap the merged contract currently leaves open.

On the two things Codex hasn't reacted to yet — Swift 6 strict concurrency and Swift Testing — I'd point out something that actually makes this easier to decide than it looks: the merged interfaces-json the coordinator just produced already sprinkles `Sendable` across `EntitlementState`, `ClassificationSuggestion`, `CaptureState`, `ExportDocumentModel`, `PhotoAssetRef`, and more. Nobody explicitly voted for strict concurrency, but the contract we just wrote already assumes it — those conformances only matter under the compiler's strict checking; under minimal checking they're decorative. So I don't think this is actually a live disagreement anymore, it's an implicit yes that should just get said out loud so nobody builds against this contract in a language mode that silently ignores half of what we just typed. Same logic applies to Swift Testing: it's purely a `Tests/` tooling choice, it doesn't touch a single cross-lane type in this contract, and the only cost of adopting it is that whoever writes `DomainTests` uses `@Test`/`#expect` macros instead of `XCTestCase` subclasses — there's no reason for this to stay unresolved when it can't possibly conflict with anything else in the spec. I want us to just write both down as decided rather than let them dangle into implementation as someone's unilateral call.

## Final Output

**Architecture overview:** MVVM with `@Observable`, `@MainActor`-bound feature view models owning all UI-facing state — no `ObservableObject`/`@Published`. Views never touch SwiftData, Vision, StoreKit, or the file system directly; everything routes through protocol boundaries (`ItemRepository`, `ClassificationService`, `EntitlementProvider`, `PhotoStore`, `ExportService`, `AppLockService`) so every layer above storage/frameworks is fake-able in tests. Capture is one named, typed state machine (`CaptureState`) owned exclusively by `CaptureViewModel` — Vision's suggestion lives only in transient capture state and is written to the persisted `Item` only if still selected at Save. All subscription/record-state rules (active/locked/incomplete/deleted) are computed at read time from stored facts via pure functions (`computeDisplayState`, `partitionActiveAndLocked`) — nothing about entitlement or cap status is ever written back into storage as a mutation. Swift 6 strict concurrency checking is on from day one; the `Sendable` conformances already threaded through the contract only mean something under strict mode, so this was effectively decided the moment the types were written.

**File/module layout:** `App/` (entry point, root `NavigationStack`, `ModelContainer` bootstrap against the App Group container, scenePhase-driven lock gate + `PrivacyOverlayModifier`) · `Domain/` (all `@Model` types, enums, and every repository/service protocol — zero SwiftUI imports) · `Data/` (concrete SwiftData-backed `ItemRepository`, on-disk `PhotoStore` implementation) · `Services/` (`VisionClassificationService`, `StoreKitEntitlementProvider`, `PDFExportService`, `LocalAuthenticationLockService` — each a concrete conformance to a `Domain` protocol) · `Features/Catalog`, `Features/Capture`, `Features/ItemDetail`, `Features/Export`, `Features/Paywall`, `Features/Settings` (View + ViewModel per folder) · `Widget/` (separate target, read-only App-Group access through the same repository protocol) · `Tests/DomainTests`, `Tests/DataTests`, `Tests/ServiceTests` (Swift Testing, `@Test`/`#expect`), `Tests/UITests` (XCTest/XCUITest, since that's still the underlying automation API). Rule: nothing in `Features/*` redeclares a cross-cutting type — everything shared lives in `Domain/` and gets imported.

**Data models:** `Item` (`@Model`, id, title, `Category`, subtype, `ItemStatus`, `createdAt`, `creationSequence: Int64` as the mandatory active-25 tiebreak, `deletedAt`, `coverPhoto: PhotoAssetRef?`, `conditionGrade: ConditionGrade?`, `purchaseFacts: PurchaseFacts?`, cascade-deleted `[ConditionEntry]`/`[ValueEntry]`); `ConditionEntry` and `ValueEntry` as child records, each with an optional `PhotoAssetRef`; `ValueEntry.amount` and `PurchaseFacts.purchaseAmount` are `Decimal`, never `Double` or minor-unit `Int`; `AppSettings` singleton row stamped once at install (currency, app-lock preference, recovery-window length) rather than re-derived from `Locale.current` on every render. Four fixed enums — `Category` (six cases, shared schema, free-text subtype), `ItemStatus`, `ConditionGrade` (ordered, `Comparable`), `ValueType` (five-case, hard constrained) — stored natively via SwiftData's Codable enum support, no stringly-typed drift risk.

**Persistence strategy:** SwiftData store lives in an App Group container from initial setup, not retrofitted for the widget later. One main-actor `ModelContext` for all UI-driven mutation; background work (PDF export, thumbnail generation) opens its own fresh `ModelContext` from the same `ModelContainer`. No CloudKit-backed SwiftData in v1 — the repository protocol, not a CloudKit-shaped schema, protects the future sync roadmap. Photos and thumbnails live on disk under stable item/entry-keyed relative paths (never blobbed into the database), referenced via `PhotoAssetRef` (path + pixel dimensions) so export can decide downsample-vs-embed without re-opening the file. `ItemRepository.loadActiveAndLocked` is contractually required to fetch all non-deleted items and delegate to `partitionActiveAndLocked` — it may never reimplement that split as an independent query, closing off the exact "two independently-computed truths drift apart" bug class this app has been fighting since detailed_discussion. Same discipline applies to entitlement: `isSubscriptionActive(_:)` is the only legal way to turn `EntitlementState` into the plain bool the domain layer expects.

**Testing strategy:** Swift Testing (`@Test`, `#expect`, parametrized `@Test(arguments:)`) for `DomainTests`/`DataTests`/`ServiceTests`; XCTest/XCUITest reserved for UI automation only. Pure-function tests exhaustively cover `computeDisplayState` precedence (deleted > locked > incomplete > normal) and `partitionActiveAndLocked`, including tied `creationSequence`, restore-at-cap, and delete-of-a-locked-item-promotes-next-oldest. Repository tests run against real SwiftData with `ModelConfiguration(isStoredInMemoryOnly: true)` plus temp file directories, exercising real photo lifecycle and purge cleanup rather than mocked fiction. `EntitlementProvider` is tested via `SKTestSession` across purchase/restore/downgrade/offline-startup. A narrow XCUITest set covers only genuine system-integration flows: capture-to-save timing, interrupted/backgrounded capture resuming, PDF export validity at 0/1/many items, full VoiceOver core-loop completion, and Export Preview's "Page 2 of 5" announcement specifically.

## Final Output

**Architecture overview:** MVVM with `@Observable`, `@MainActor`-bound feature view models owning all UI-facing state — no `ObservableObject`/`@Published`. Views never touch SwiftData, Vision, StoreKit, or the file system directly; everything routes through protocol boundaries (`ItemRepository`, `ClassificationService`, `EntitlementProvider`, `PhotoStore`, `ExportService`, `AppLockService`) so every layer above storage/frameworks is fake-able in tests. Capture is one named, typed state machine (`CaptureState`) owned exclusively by `CaptureViewModel` — Vision's suggestion lives only in transient capture state and is written to the persisted `Item` only if still selected at Save. All subscription/record-state rules (active/locked/incomplete/deleted) are computed at read time from stored facts via pure functions (`computeDisplayState`, `partitionActiveAndLocked`) — nothing about entitlement or cap status is ever written back into storage as a mutation. Swift 6 strict concurrency checking is on from day one; the `Sendable` conformances already threaded through the contract only mean something under strict mode, so this was effectively decided the moment the types were written.

**File/module layout:** `App/` (entry point, root `NavigationStack`, `ModelContainer` bootstrap against the App Group container, scenePhase-driven lock gate + `PrivacyOverlayModifier`) · `Domain/` (all `@Model` types, enums, and every repository/service protocol — zero SwiftUI imports) · `Data/` (concrete SwiftData-backed `ItemRepository`, on-disk `PhotoStore` implementation) · `Services/` (`VisionClassificationService`, `StoreKitEntitlementProvider`, `PDFExportService`, `LocalAuthenticationLockService` — each a concrete conformance to a `Domain` protocol) · `Features/Catalog`, `Features/Capture`, `Features/ItemDetail`, `Features/Export`, `Features/Paywall`, `Features/Settings` (View + ViewModel per folder) · `Widget/` (separate target, read-only App-Group access through the same repository protocol) · `Tests/DomainTests`, `Tests/DataTests`, `Tests/ServiceTests` (Swift Testing, `@Test`/`#expect`), `Tests/UITests` (XCTest/XCUITest, since that's still the underlying automation API). Rule: nothing in `Features/*` redeclares a cross-cutting type — everything shared lives in `Domain/` and gets imported.

**Data models:** `Item` (`@Model`, id, title, `Category`, subtype, `ItemStatus`, `createdAt`, `creationSequence: Int64` as the mandatory active-25 tiebreak, `deletedAt`, `coverPhoto: PhotoAssetRef?`, `conditionGrade: ConditionGrade?`, `purchaseFacts: PurchaseFacts?`, cascade-deleted `[ConditionEntry]`/`[ValueEntry]`); `ConditionEntry` and `ValueEntry` as child records, each with an optional `PhotoAssetRef`; `ValueEntry.amount` and `PurchaseFacts.purchaseAmount` are `Decimal`, never `Double` or minor-unit `Int`; `AppSettings` singleton row stamped once at install (currency, app-lock preference, recovery-window length) rather than re-derived from `Locale.current` on every render. Four fixed enums — `Category` (six cases, shared schema, free-text subtype), `ItemStatus`, `ConditionGrade` (ordered, `Comparable`), `ValueType` (five-case, hard constrained) — stored natively via SwiftData's Codable enum support, no stringly-typed drift risk.

**Persistence strategy:** SwiftData store lives in an App Group container from initial setup, not retrofitted for the widget later. One main-actor `ModelContext` for all UI-driven mutation; background work (PDF export, thumbnail generation) opens its own fresh `ModelContext` from the same `ModelContainer`. No CloudKit-backed SwiftData in v1 — the repository protocol, not a CloudKit-shaped schema, protects the future sync roadmap. Photos and thumbnails live on disk under stable item/entry-keyed relative paths (never blobbed into the database), referenced via `PhotoAssetRef` (path + pixel dimensions) so export can decide downsample-vs-embed without re-opening the file. `ItemRepository.loadActiveAndLocked` is contractually required to fetch all non-deleted items and delegate to `partitionActiveAndLocked` — it may never reimplement that split as an independent query, closing off the exact "two independently-computed truths drift apart" bug class this app has been fighting since detailed_discussion. Same discipline applies to entitlement: `isSubscriptionActive(_:)` is the only legal way to turn `EntitlementState` into the plain bool the domain layer expects.

**Testing strategy:** Swift Testing (`@Test`, `#expect`, parametrized `@Test(arguments:)`) for `DomainTests`/`DataTests`/`ServiceTests`; XCTest/XCUITest reserved for UI automation only. Pure-function tests exhaustively cover `computeDisplayState` precedence (deleted > locked > incomplete > normal) and `partitionActiveAndLocked`, including tied `creationSequence`, restore-at-cap, and delete-of-a-locked-item-promotes-next-oldest. A narrow XCUITest set covers only genuine system-integration flows: capture-to-save timing, interrupted/backgrounded capture resuming, PDF export validity at 0/1/many items, full VoiceOver core-loop completion, and Export Preview's "Page 2 of 5" announcement specifically.

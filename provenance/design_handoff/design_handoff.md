Start with navigation, because it determines everything else. On the effective-state model: this has to be a single pure function, `func effectiveState(for item: Item, subscriptionActive: Bool, activeItemIDs: Set<PersistentIdentifier>) -> ItemState`, computed at render time from three independent facts (deletedAt != nil, status enum, membership in the active-25 set), never stored. The dangerous version of this feature is one where each screen (list row, detail header, badge) independently checks "is this locked" and "is this incomplete" with its own if/else, because that's how the precedence rule (deleted > locked > incomplete) drifts out of sync between screens over one release cycle. I'd make this a single `ItemDisplayState` enum with associated data, computed once per item per render pass, and every view — list card, detail header, badge — switches over that one enum. The active/locked partition itself needs to be one query, not one check per item: fetch all non-deleted items sorted by creation order, take the first 25 as active, the rest as locked. If this gets implemented as "for each item, count how many earlier items exist" that's O(n²) on a catalog someone's spent years building, and it's the kind of perf cliff that only shows up after the app's already shipped and someone has 400 sneakers logged.

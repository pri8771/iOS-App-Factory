**Codex (Backend / Systems Engineer · Systems Thinker) — Round 1**

I’d lock this as a fairly traditional SwiftUI app with a thin app-level composition root, SwiftData for the local store, a repository boundary that hides persistence and future sync, and a small set of feature services for camera/photo storage, Vision classification, PDF export, subscription state, and app-lock/privacy. The mistake to avoid is letting SwiftUI screens talk directly to SwiftData models, StoreKit, Vision, and file I/O all at once. That will ship faster for week one and then become impossible to reason about once downgrade rules, soft-delete recovery, export, and widget sharing all start interacting. The right shape is: SwiftUI views own presentation state, feature models orchestrate workflows on the main actor, repositories expose deterministic async methods, and lower-level services do the file and framework work off-main.

I would use SwiftUI plus Observation (`@Observable`) on iOS 17+, not `ObservableObject`, and I would mark any state-owning feature model `@MainActor` so UI mutation stays disciplined. The app already has enough asynchronous edges: photo capture, thumbnail generation, Vision classification, export rendering, StoreKit transaction updates, and lock/unlock transitions. If actor boundaries are fuzzy, you will get racey UI states around draft creation, resumed capture, and value-card entitlement flips. One concrete example: the moment a photo lands, draft item creation, cover-photo file write, thumbnail generation, and classification request should not all be separate ad hoc tasks launched from the view. That is one capture workflow owned by one feature model, with explicit state transitions: draft created, media processing pending, classification suggestion available or absent, save-ready, incomplete if file write fails. That state machine matters more than the camera UI.

For frameworks, I’d stay entirely inside Apple’s stack. I’d tighten one more thing while we’re here, because it falls directly out of both Claude’s thread-safety point and my earlier “access-layer rules, not storage mutations” point: classification writeback should not mutate the persisted `Item` unless the user actually accepts or leaves the suggestion in place at save.

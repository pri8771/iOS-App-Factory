**Claude (Frontend Engineer · Systems Thinker) — Round 1**

The thing I want to nail down this round that hasn't actually been specified yet, despite five phases of solid product decisions, is the navigation architecture underneath the screen list — because the locked flow (Add chains into Compare chains into Placed, plus a resume banner that has to jump into that same Compare state from cold launch) is exactly the shape that produces SwiftUI's ugliest class of bug if we don't pin it down now: competing presentation state. If Compare is one `.fullScreenCover(isPresented:)` and Placed is a second, separate `.fullScreenCover(isPresented:)`, dismissing one to present the other causes a visible flash back to Library in between — which directly undercuts the "total focus, red carpet" feeling both of the last two phases fought hard to establish. My strong recommendation: Compare and Placed are not two covers, they're one `.fullScreenCover(item:)` bound to a single `ActiveInsertionFlow` value (holding the persisted `lo/hi/lastComparedIndex/skipBudget` plus an internal `.comparing`/`.placed(result)` sub-state), and the transition between comparing and placed happens as an internal view swap inside that one cover, not a dismiss-then-present. The resume banner at the top of Library, when tapped, constructs that exact same `ActiveInsertionFlow` value from persisted state and sets it — so there's exactly one code path into Compare, whether you just tapped Add or you're recovering from a crash three days later. I'd also put Add Title, Taste Card, and Paywall behind their own single `.sheet(item:)` at the Library root, each driven by an enum case, so there is never a state where two presentations could theoretically be true at once — that invariant is worth writing into the handoff explicitly because it's the kind of thing that's easy to violate two screens later when someone adds a "quick share" button from inside Placed without realizing it collides with the cover already on screen.

Second systems-level risk worth flagging now: the Taste Card is generated with `ImageRenderer` off the ranked list's top ten, but if any of that art is an `AsyncImage` still resolving (or a placeholder still animating in), the rendered card can silently capture blank or half-loaded squares — `ImageRenderer` snapshots exactly what's laid out at that instant, it doesn't wait for async image loads the way the live view does. Since our default happy path is offline/no-network placeholder art (deterministic initials + color block, generated synchronously from the title string, no network round trip), this is mostly a non-issue for the default provider — but it's a real trap the moment the optional metadata-enrichment provider is wired in later, and it's a trap even now if we're not careful about resolving all ten poster views into fully-loaded `UIImage`/`Image` values before triggering the render. I want this written into the handoff as a rule: Taste Card sheet has its own brief `resolvingArt` loading state before the live preview appears, and the render only fires once every one of the up-to-ten source images is confirmed loaded, never on a best-effort snapshot.

Now my read on this, wearing the skeptic hat.

The concept is sound on paper but the spec is quietly making a bet it never defends: that a handful of pairwise comparisons converges to a trustworthy top 10. Nobody has specified what "handful" means or what the convergence algorithm actually is, and the spec's own Invalidation Criteria admits this is the single point of failure for the whole app. I don't think we can call this MVP done unless there's an actual named algorithm (merge-sort-derived insertion with a bounded comparison budget is the obvious choice) with a stated worst-case comparison count, because "Elo-style" and "merge-sort-style" are two different algorithm families with different guarantees and the prompt uses both terms loosely as if interchangeable. Elo needs many repeated matches to stabilize and is built for large populations playing repeatedly — it's a poor fit for "rank 20 movies from a dozen comparisons total." A merge-insertion (Ford-Johnson-style) binary-insertion approach gives a real n·log(n) bound and a deterministic, explainable ranking, which is what a user doing 8 comparisons to place a new title actually needs. I'd flag now: if the build phase defaults to "compute an Elo score per title" without a bounded insertion strategy, that's the exact failure mode the client's own invalidation criteria warns about, and it should be treated as a contract violation, not a stylistic choice.

Second risk: title/poster metadata lookup is stated as "the one network dependency," which implies a real external API (something TMDB-shaped). We have no confirmed API key, no confirmed provider, and no attribution/ToS constraints discussed. This is a hard requirement with an unresolved external dependency. My decision rule: the metadata lookup must be built behind a protocol with a clearly fake/local "offline" provider as the default and any real network provider treated as optional/pluggable — the app must be fully demoable and functionally complete (adding titles, ranking, sharing) with zero network access, using manual title entry plus initials/color-block artwork. If a later phase decides to wire up a real poster API, that's additive, not load-bearing, and must not become a blocker for calling this MVP done. This also directly satisfies the spec's own stated key risk about graceful offline degradation — but I'd bet money that without an explicit decision rule, a later build phase quietly makes the network the happy path and offline the degraded afterthought, which inverts the priority the client stated.

Third: StoreKit paywall in an MVP is scope creep risk dressed as a requirement. Real StoreKit testing requires App Store Connect configuration, sandbox accounts, and product IDs — none of which exist in a local build context. My decision rule here: paywall UI, entitlement gating logic (free tier caps: movies-only, capped list length), and a StoreKit2-based purchase flow using local StoreKit configuration file testing (Xcode's `.storekit` config) are in scope and must be real, working code — but this is not App-Store-submittable commerce, and nobody should claim revenue is "live." That distinction needs to be explicit in the production-readiness definition or someone will later claim monetization is "done" when it's only locally testable.

Fourth, WidgetKit "small widget showing current #1" is listed as an iOS-native feature but not in the v1 Core Workflows list. That's a direct internal inconsistency in the source spec — Core Workflows is presumably the definition of "must ship," and the widget isn't in it.

## Design Discussion

Both sides actually landed in the same place this round. Codex came in and conceded both open points before Claude even had to push — dropping the separate Browse screen in favor of Claude's single Library spine (arguing it's not just cleaner, it's the faster, lower-risk ship shape and better protects the "no surprise policy screens" rule), and agreeing Compare should have no visible progress/confidence meter since it fights the "fun ritual, not a checklist" promise the whole app is built on.

Claude's round 2 message then builds on that agreement rather than reopening it: it folds Codex's "pending insertion" idea back in as a slim resume banner that only appears at the top of Library when there's a crash-recovered in-flight insertion waiting — not a permanent second view, just a narrow rescue path for the interruption case. It also catches something real buried in Codex's original wording: Codex had listed "left/right/tie/skip" as four options, and Claude points out "tie" is actually a new vote type that was never scoped and would break the locked sort-order invariant (binary insertion has no valid slot for "equal"). Claude's fix — treat tie and skip as the same button, just call it skip, matching all the copy already locked in earlier phases — isn't something Codex has explicitly signed off on in text, but it's a direct, uncontroversial correction of an inconsistency in Codex's own proposal, not a new fight, and nothing in the thread pushes back on it.

CONSENSUS: YES

## Final Output

**Screen inventory:**
- **Library** (root, single `NavigationStack`, no tab bar) — the one home screen: Movies|TV segmented control at top, filter chips (genre/decade) that only appear once there are enough titles to make filtering meaningful, the ranked list itself (poster, condensed rank number, title, year; TV rows shown dimmed with a lock glyph and "Unlock with FaveOff+" instead of a rank number), a toolbar "Add title" action, a toolbar Taste Card / share icon, and — only when a crash-recovered insertion is pending at launch — a slim top-of-list resume banner ("Finish placing [Title]") that jumps straight back into Compare at its persisted bounds. Non-blocking inline banner at the bottom of the list once the 25-item free cap is hit.
- **Add Title** — medium-detent form sheet: name (autofocused), Movie/TV type toggle, optional year/genre as visually secondary fields. Save returns to Library or chains directly into Compare.
- **Compare** — full-screen cover, total focus, no parent context visible. Two posters (or initials/color-block cards, same visual weight) with a VS divider; tap either to choose. One smaller, clearly secondary "Skip" affordance below the divider — this is the only alternate action; "tie" is not a separate option, it's the same control as skip, using the skip copy/logic already locked. No progress bar, no comparison counter, no confidence meter anywhere on this screen.
- **Placed** — full-screen cover, reached directly after a comparison resolves. Big rank reveal in the condensed display face, "ahead of X, behind Y" for normal placements, the locked cold-start and skip-fallback copy variants for the edge cases, one button ("Keep going") back to Library. This screen — not a live meter during Compare — is where the app is honest about how a title landed.
- **Taste Card** — large-detent sheet off the Library toolbar icon: live ImageRenderer preview of the poster-grid card, ShareLink below it.
- **Paywall** — sheet, the same single instance regardless of entry point (TV-lock row, cap banner, or toolbar upgrade button), with the full loading/pending/failed/purchased/restore state matrix.

**Primary user flow:** Cold start → Library empty state ("Add your first movie," button doubles as the empty-state CTA) → Add Title sheet → save → Compare (if insertion needed) or straight to Placed (0/1-item cold start, auto-placed) → Placed reward screen → "Keep going" back to Library, now showing the updated ranked list in place. TV adds always succeed and show as a locked row without ever touching movie ranking. Hitting the 25-item cap doesn't block adds, only further ranking, surfaced via the inline banner. Any purchase attempt lives entirely inside the Paywall sheet and never affects the ranking loop underneath it, success or failure. If the app was killed mid-insertion, relaunch shows the resume banner at the top of Library instead of silently dropping or auto-placing the title.

**State model per screen:**
- Library: `emptyNoItems`, `listWithItems` (with or without resume banner), `emptyFiltered` (nudge, not blank), `cappedBanner` visible once cap reached, standard `loading`/`error`.
- Add Title: `idle` → `saving` → `unranked`/`failed`.
- Compare: derives entirely from persisted `lo/hi/lastComparedIndex/skipBudget`; states are `pairReady`, `choicePending`, `skipPending`, `terminatedBySkips`; an interruption state (opposing title deleted mid-screen) cancels gracefully back to Library with a toast, no crash.
- Placed: `exactPlacement`, `fallbackPlacement` (honest skip-fallback copy), `cold-start` copy variants for 0/1-item lists — never a broken "#1 of 1, ahead of nothing" string.
- Paywall: `available`, `loadingPurchase`, `purchasePending`, `purchaseFailed` (dismiss always reachable), `purchased`, `restorePending`, `notEntitled`.

**Visual direction:** Dark warm charcoal base (not true black), marquee red reserved strictly for accents/actions. Condensed bold display face only for short strings (rank numbers, Placed title, section headers); standard Dynamic-Type system text everywhere else, especially anything VoiceOver reads or that can wrap. Poster-first card composition throughout. Film-reel-style wipe transition specifically on the Compare/Placed sequence, falling back to a plain crossfade under Reduce Motion.

**Accessibility notes:** Compare and Placed are hard gates — measured (not eyeballed) contrast ≥4.5:1 body-weight / ≥3:1 large-bold across the full Dynamic Type range before either screen is considered done. VoiceOver labels describe the actual choice ("Select [Title A] over [Title B]"), Skip gets its own explicit label, both poster tap targets stay ≥44×44pt even on the smallest phone width, and the single-sentence Placed announcement always includes rank plus both neighbors.

**Named open risk carried forward, not re-litigated:** betting Compare's own visual weight (posters need to read as obviously tappable) is enough to make the interaction self-evident with zero instructional copy anywhere — if build/test surfaces real first-time confusion, the fix is strengthening Compare's own affordances (bigger VS divider, more obvious tap zones), never a bolted-on tutorial screen.

## Final Output

**Screen-by-screen specification**

- **Library** (root, single `NavigationStack`, no tab bar): Movies|TV segmented control at top to make the track boundary spatial; genre/decade `FilterChip` row appears only once there are enough titles to make filtering meaningful (suppressed below that threshold, not shown empty); ranked list of `LibraryRow`/`PosterTile` (poster, condensed rank badge, title, year), TV rows rendered as `LockedTVRow` (dimmed tile, lock glyph, "Unlock with FaveOff+" in place of rank — never blocks add or logging); toolbar "Add Title" and "Taste Card" actions; `InlineBanner` styled two ways — `resumeInsertion` (top of list, only when a crash-recovered in-flight insertion exists, "Finish placing [Title]" → jumps straight into the active flow at persisted bounds) and `capReached` (bottom of list, only at exactly 25 ranked movies, non-blocking). States: `loading` (row skeleton/shimmer), `emptyNoItems` ("Add your first movie" doubling as CTA), `list`, `listWithFilters`, `emptyFiltered` (nudge, not blank), `loadingFailure` (inline retry).

- **Add Title**: medium-detent form sheet. Title field autofocused, Movie/TV type segmented control directly under it, year/genre visually secondary and clearly optional. Save is instant/local-first, routes to the active insertion flow (Compare) or straight to `placed` for 0/1-item cold starts. States: `idle` → `saving` → `saved`/`failed` (typed content preserved on failure, inline retry, no dropped input).

- **Compare + Placed** (one continuous ritual, one `.fullScreenCover(item:)` bound to a single `ActiveInsertionFlow` value — never two independent covers, so there is no dismiss-flash back to Library between decision and placement): full-screen, zero parent chrome. Compare shows two `PosterTile`s (poster or deterministic initials/color-block, identical visual weight either way) with a VS `CompareDivider`, tap either to choose, one secondary `SkipButton` below — no progress bar, no comparison counter, no confidence meter anywhere on this screen; "tie" is not a separate action, it's Skip. If the opposing title is deleted mid-screen: cancel gracefully back to Library with a toast ("Title removed, comparison canceled"), no crash. Placed follows immediately inside the same cover: rank reveal in condensed display type, `#N, ahead of X, behind Y` for normal placements, the locked cold-start copy ("First one in — add another to build your ranking") and skip-fallback copy ("Added near the bottom for now — compare it later to refine it") for edge cases, one button ("Keep going") back to Library. Flow states: `comparing`, `processingChoice`, `skip`, `terminatedBySkips`, `cancelledByInterruption`, `placed` (variants: `exact`, `fallback`, `coldStart`).

- **Taste Card**: large-detent sheet off the Library toolbar. Enforces a `resolvingArt` loading state before the live preview — the render never fires on partially-loaded poster art, since `ImageRenderer` snapshots synchronously and would otherwise silently capture blank tiles; trivially satisfied today since default art is synchronous initials/color-block, but the gate stays in place so it doesn't regress when optional metadata enrichment is wired in later. States: `resolvingArt`, `ready` (live preview + `ShareLink`), `renderFailed` (retry), `sharing`.

- **Paywall**: single shared sheet instance regardless of entry point (TV-lock row, cap banner, toolbar upgrade). States: `available`, `loadingProducts`, `purchasePending`, `purchaseFailed` (dismiss always reachable), `restored`, `entitled`, `dismissed`. Ranking loop is fully unaffected by any state here, success or failure.

**Design tokens**
- `backgroundCanvas` `#15100F`, `backgroundSurface` `#1E1918`, `backgroundElevated` `#272020`, `borderLine` `#33292A` (hairline borders instead of shadows — near-black fields don't render shadows well)
- `accentRed` `#E22C3C` — action fills and short large/bold display strings only, never body text (re-verify large-bold ≥3:1 against `backgroundCanvas` before final)
- `textPrimary` `#F6F1ED`, `textSecondary` `#AFA7A2` (both independently confirmed ≥4.5:1 body-weight against the canvas), `textDisabled` `#6F6660`
- Type: system font throughout for Dynamic Type integrity; condensed width (`Font.system(...).width(.condensed)`) reserved strictly for short non-wrapping strings — rank numbers, Placed hero title, section headers, chip labels. No embedded custom font file.
- Spacing: 8pt base grid (4/8/12/16/24/32/48). Radius: 16pt cards, 14pt buttons, fully-rounded chips.
- Motion: standard state changes 0.25s ease-in-out; the one cinematic beat is the Compare→Placed internal wipe inside the shared cover, ~0.35s horizontal masked transition with a faint sprocket-edge detail, collapsing to a flat 0.2s crossfade under Reduce Motion — no partial-motion exceptions.

**Component inventory**: `NavigationShell`, `LibraryRow`/`PosterTile`, `RankBadge`, `LockedTVRow`, `TrackSegmentedControl`, `FilterChip`, `InlineBanner` (resume + cap variants), `EmptyState`, `ErrorState`, `AddTitleFieldRow`, `CompareDivider`, `SkipButton`, `PlacedReveal`, `TasteCardGrid`, `PaywallProductTile`, `PrimaryButton`/`SecondaryButton`, `DeleteAction` (swipe + detail-screen fallback), `SectionSpacer`.

**Navigation/state architecture (binding rule)**: Library owns exactly one `activeSheet: SheetKind?` (Add Title / Taste Card / Paywall, each `.sheet(item:)`) and one `activeFlow: ActiveInsertionFlow?` (Compare/Placed, `.fullScreenCover(item:)`) — never competing presentations. The resume banner constructs the identical `ActiveInsertionFlow` value from persisted `lo`/`hi`/`lastComparedIndex`/`skipBudget`, so there's exactly one code path into the ritual whether triggered by a fresh Add or a crash-recovery resume. All list-driving views read through the repository/view-model boundary — no view that also does navigation touches `@Query` directly.

**Accessibility**: Compare and Placed are hard gates — measured ≥4.5:1 body / ≥3:1 large-bold contrast across the full Dynamic Type range before either is done. VoiceOver labels state the actual choice ("Select [Title A] over [Title B]"), Skip gets its own explicit label, 44×44pt minimum on every tap target including full poster bounds, no color-only status cues (lock/cap states pair a glyph with text), and the Placed VO announcement is one sentence covering rank plus both neighbors.

**Claude Design decision**: No external design pass for the app's screens — the token system, component inventory, and state model above are specific enough for direct SwiftUI implementation, and handing the full UI to an external pass risks producing frames that contradict the locked interaction contract (the single-cover invariant, the "no unprompted policy screens" rule, the honest placement copy variants) since an external tool has no visibility into that state machine. Two narrow, bounded external prompts are approved instead:
1. **App icon**: "Design an app icon for FaveOff, a cinematic movie/TV ranking app — near-black warm charcoal base (#15100F), marquee red accent (#E22C3C), bold condensed numeral or ticket/film-reel motif, award-show/red-carpet feeling, must read clearly at 60×60pt down to the smallest home-screen size, no text."
2. **Optional 3-frame mood check** (Library with a few ranked rows, Compare mid-decision, Placed reward state) — generated strictly from the token values and layout rules above, explicitly framed as a visual sanity check only, not a spec, and explicitly instructed not to invent new navigation, copy, or states.

## Ios Architecture Review

CONSENSUS: YES

Both of them are pulling in the exact same direction here — Codex laid out the service-oriented skeleton (repository façade, feature services, StoreKit isolation, zero third-party packages, pure ranking engine) and Claude's message doesn't push back on any of it; it tightens screws Codex left loose and adds a few things Codex's draft simply hadn't gotten to yet. Nothing in either message contradicts the other.

Where they agree outright: SwiftUI + SwiftData + StoreKit 2 only, no third-party dependencies at all, no ML/AR/Vision frameworks (both give the same "unnecessary failure surface" reasoning), the ranking engine must be a pure, UI/persistence-free unit (Codex frames it as "no UI," Claude sharpens it to "zero SwiftData imports, zero ModelContext" — same rule, stated more strictly), metadata lookup stays behind a protocol with an offline default and remote lookup as pure background enrichment, StoreKit purchase state must be fully isolated so failures can't touch the ranking loop, and CloudKit/widget entitlements stay explicitly out of scope for this build.

What Claude adds without contradicting Codex: the CloudKit-migration trap (SwiftData + CloudKit needs optional relationships, no unique constraints, default values on non-optional fields — so lock a `VersionedSchema`/`SchemaMigrationPlan` now even with one version, since the alternative is a destructive migration on the first real update), a concrete fix for the delete-during-comparison race (re-fetch by persistent ID right before acting, nil means `cancelledByInterruption`, and it should be a real XCTest not just a described scenario), the UI-visible consequence of Codex's own atomicity risk (bulk `position` rewrites reading as 15 rows glitching unless the bulk shift is unanimated and only the Placed reveal animates), the `PrivacyInfo.xcprivacy` manifest requirement (a real App Store submission blocker Codex hadn't named), the requirement that the `Transaction.currentEntitlements` listener run as a persistent launch-time `Task` rather than a one-shot call, and a concrete unit test that runs the actual WCAG contrast formula against the locked design tokens instead of leaving "measured, not eyeballed" as an unverified claim.

The one open thread that isn't a disagreement, just unfinished business: Codex proposed a specific three-entity schema (`Title`, `Comparison`, a dedicated `InsertionContext`) and Claude's message doesn't explicitly confirm or dispute that shape — it's silent on it rather than contesting it. Since nothing in Claude's CloudKit-compatibility point conflicts with Codex's entity split (if anything, a separate `InsertionContext` entity makes the "every relationship optional" CloudKit constraint easier to satisfy than cramming insertion bounds directly onto `Title`), I'm treating this as compatible-by-default rather than a real fight.

## Final Output

**SwiftUI architecture:** Service-oriented layering — `App` shell owns scene/session state; `@Observable` view models sit between every view and a `TitleRepository` protocol (no view touches SwiftData directly, matching the navigation binding rule already locked in `design_handoff`); feature services for ranking, metadata lookup, purchase, and Taste Card rendering. The `RankingEngine` (bounded binary insertion, skip/2-skip-termination, `lo/hi/lastComparedIndex/skipBudget` state machine) is a pure Swift type — zero SwiftData imports, zero `ModelContext` dependency, plain arrays and IDs in and out — wrapped by an `InsertionCoordinator` use case that persists state after every single step so kill/relaunch always resumes from durable truth, never in-memory view state.

**Apple frameworks:** SwiftUI, SwiftData, StoreKit 2, `ImageRenderer` + `ShareLink`, SF Symbols, XCTest/Swift Testing, XCUITest. Explicitly no Vision, ARKit, Core ML, or any ML/AR framework — the spec already named the ranking engine, not ML, as the one hard technical capability, and none of these frameworks earn their failure-surface cost (permission prompts, model load latency, device-capability branching) for zero locked user value.

**Persistence and local data plan:** SwiftData as sole source of truth, accessed only through the repository façade. Entities: `Title` (user-entered content: name, type, optional year/genre, deterministic poster color/text seed, track, rank position, `rankingState`), `Comparison` (decision audit trail: winner/loser IDs, track, timestamp, skip flag, flow ID), and a dedicated `InsertionContext` entity holding the active insertion's `lo`/`hi`/`lastComparedIndex`/`skipBudget` rather than overloading `Title` with half a state machine — cleaner to reason about and easier to keep every relationship optional the way CloudKit compatibility requires later. Lock a `VersionedSchema` + `SchemaMigrationPlan` from day one, even with a single schema version, so the eventual CloudKit sync roadmap doesn't force a destructive migration on first update. Any remote poster art is cached as files with a pointer ID on `Title`, never as `Data` blobs in SwiftData rows. All `position` writes for a given insertion happen as one atomic service-level operation with list animation disabled on the bulk shift — the user should never see 15 rows visibly reshuffle; only the Placed reveal animates. Compare re-fetches its two titles by persistent ID immediately before acting rather than holding long-lived references, so a mid-screen deletion resolves to a clean `cancelledByInterruption` state instead of a silent no-op or crash.

**Permissions/privacy plan:** Zero runtime permission prompts anywhere in the app — no camera, location, contacts, microphone, notifications, or photo-library access (ShareLink's built-in save covers export). Network is used only for optional metadata/poster enrichment, strictly opt-in and never load-bearing for add/compare/rank. No analytics, no device fingerprinting, no identifiers beyond internal UUIDs. A `PrivacyInfo.xcprivacy` manifest declaring required-reason API usage must be in the project from the first build, not added before archiving — this is submission-blocking, not polish. StoreKit needs the In-App Purchase capability on the target; nothing else needs an entitlement. CloudKit container/entitlement setup and WidgetKit's App Group plumbing both stay explicitly out of scope for this build.

**Dependency/license decision:** Zero third-party packages, full stop. Everything the app needs — SwiftData, StoreKit 2, ImageRenderer, ShareLink, SF Symbols, `AsyncImage` if the optional provider is ever wired in — is first-party, which trivially satisfies the permissive-license policy and removes an entire category of maintenance/license risk from an app that has no functional need for any external package. A future proposal to add one (e.g., for image caching) should be treated as a rule violation to push back on by default.

**Testing implications:** Because the ranking engine is pure, its correctness (bounded-insertion ordering, transitivity invariants, deterministic 2-skip termination and fallback placement, duplicate-title handling) is unit-testable in milliseconds with no `ModelContainer`. Persistence/integration tests cover crash-recovery resume from stale `InsertionContext` bounds, delete-during-comparison resolving to `cancelledByInterruption` (constructed explicitly: start an insertion, delete the opposing title from a second context, assert no crash and no corrupted bounds), 0/1-item cold-start placement copy, TV-rank-lock behavior, and the 25-item cap (adds still succeed, only ranking blocks). A dedicated unit test runs the real WCAG relative-luminance/contrast formula against the locked token pairs (`textPrimary`/`textSecondary` vs. `backgroundCanvas`, `accentRed` large-bold vs. `backgroundCanvas`) and asserts the 4.5:1/3:1 thresholds numerically as part of the normal test target — "measured, not eyeballed" gets an actual test, not just a claim. XCUITest covers the one true end-to-end path (add → compare → placed → browse → share) plus resume-banner recovery and paywall isolation under a forced purchase failure, using deliberate accessibility identifiers rather than label-text matching. The `Transaction.currentEntitlements` listener runs as a persistent `Task` attached at launch (not a one-shot call tied to a button tap), since entitlement changes can complete while backgrounded or across a relaunch.

## Tech Specs

**What got resolved, point by point:** Codex conceded outright on `lastComparedTitleID: UUID?` over `lastComparedIndex: Int`, using Claude's own "re-fetch by persistent ID, never trust an index" logic against the earlier draft. Codex's `RankingTransition`/`SkipFallbackReason` combo — which let a "deleted opponent" interruption be encoded as a placement reason on the same struct as an actual rank position — got replaced with Claude's cleaner split (`InsertionStepResult` for placement outcomes, `InterruptionReason` as a fully separate path for cancellation); Codex's own round-2 contract adopted that shape. View models are `@Observable` in both, not `ObservableObject`. `PurchaseService`, `MetadataLookupService`, and `TasteCardRenderer` (returning `Image`, not `Data`) all landed on matching signatures. `InsertionFlowViewModel` beat `CompareFlowViewModel` — Codex just used Claude's name. What's left is genuinely trivial — a couple of method names that mean the same thing (`fetchByID` vs `title(id:)`, `removeTitle` vs `delete`), a couple of small additive types one side proposed and the other didn't get to yet (`AccessibilityIdentifiers` for XCUITest targeting, `SwiftDataContainerFactory` as a test seam, a `capReached` repository check, a fixture helper) — nothing here needs a round 3, I'm just picking one name for each and folding the useful additions in below.

One thing I'm fixing rather than just merging: Codex's round-2 `Title` added `@Attribute(.unique) var id: UUID`. That directly contradicts the CloudKit-compatibility rule locked back in `ios_architecture_review` — SwiftData+CloudKit forbids unique constraints on attributes, and the entire reason we locked a `VersionedSchema`/`SchemaMigrationPlan` this early was to avoid a destructive migration when CloudKit sync gets added later. `UUID()` generation already gives us practical uniqueness without a schema-level constraint, so I'm dropping `@Attribute(.unique)` from the final contract.

## Final Output

**Architecture overview:** Service-oriented, single app target plus one local Swift package. `FaveOffApp` composes an `AppEnvironment` once at launch (repositories, `InsertionCoordinator`, metadata/purchase/card services) and injects it via `.environment`; no view or view model constructs its own `ModelContext` or `URLSession`. Every view model is `@Observable @MainActor` and talks only to repository protocols — never `@Query` directly in a view that also does navigation. The one hard technical capability, `RankingEngine`, is a static-function `enum` with zero SwiftData/SwiftUI imports, living in a standalone local package (`FaveOffRanking`, Foundation-only) so a stray `import SwiftData` there is a compile error, not a review catch. `InsertionCoordinator` is the sole bridge between that pure engine and persistence: it persists `InsertionContext` after every single step, re-fetches both compared titles by persistent ID immediately before acting on a choice/skip, and resolves a missing title to `cancelledByInterruption` rather than a crash or silent no-op. Library owns exactly one `activeSheet: SheetKind?` and one `activeFlow: ActiveInsertionFlow?`; Compare and Placed are one `.fullScreenCover(item:)`, never two.

**File/module layout:** One app target with disciplined folders (not compiled module walls, since the only invariant worth a hard boundary is engine purity, and that's already covered by the package): `FaveOff/App` (shell, `AppEnvironment` composition root), `FaveOff/Models` (the three `@Model` types, `FaveOffSchemaV1`, `FaveOffMigrationPlan`), `FaveOff/Repositories` (the only files importing SwiftData directly), `FaveOff/Services` (`InsertionCoordinator`, metadata/purchase/card-render services, `ContrastValidator`), `FaveOff/ViewModels`, `FaveOff/Views` (mirrors the design-handoff screen inventory), `FaveOff/Navigation` (the two presentation enums). `Packages/FaveOffRanking` is the one local package: `RankingEngine`, `InsertionState`, `ComparisonPair`, `InsertionStepResult`, `PlacementKind` — Foundation-only, no app-target imports. Tests: `FaveOffRankingTests` needs no `ModelContainer` at all; `FaveOffTests` covers repositories/integration with an in-memory container; `FaveOffUITests` covers the one true end-to-end path using the `AccessibilityIdentifiers` below.

**Data models:** `Title`, `Comparison`, `InsertionContext` as `@Model` classes, no `@Relationship` properties anywhere — every cross-entity link is a plain `UUID` resolved through a repository, which is what keeps deletion safe (a deleted title just makes a lookup return `nil`, no cascade rule to reason about) and keeps every relationship CloudKit-optional by construction. No unique-attribute constraints, for the same CloudKit-migration-safety reason. `Genre` and `MediaType` are fixed enums, never free text. `InsertionContext.lastComparedTitleID` is a `UUID?`, never an array index, because indices drift the moment another insertion or a delete shifts `position` values underneath a backgrounded flow.

**Persistence strategy:** SwiftData is the sole source of truth, reached only through `TitleRepository`/`ComparisonRepository`/`InsertionRepository`. `FaveOffSchemaV1` + an (initially empty-stage) `FaveOffMigrationPlan` are locked from the first commit. `applyPositions` is one atomic write per insertion with the bulk shift unanimated — only the Placed reveal animates, so the user never sees rows visibly reshuffle. `SwiftDataContainerFactory` is the injectable seam that makes real-app vs. in-memory-test containers a one-line swap.

**Testing strategy:** Pure-engine suite (bounded-insertion ordering, transitivity, two-skip deterministic termination and fallback placement, 0/1-item cold start, duplicate titles) runs in milliseconds against `FaveOffRanking` alone. `ContrastValidator` gets a numeric unit test asserting the real WCAG thresholds against every locked token pair. Integration tests (in-memory `ModelContainer`) cover crash-recovery resume from a stale `InsertionContext`, delete-during-comparison resolving to `cancelledByInterruption`, cap boundary (26th add still succeeds, only ranking blocks), and TV-rank-lock behavior. One `XCUITest` walks add → compare → placed → browse → share plus resume-banner recovery and paywall isolation under a forced purchase failure, targeting the `AccessibilityIdentifiers` below rather than label text.

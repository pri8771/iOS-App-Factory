**Codex (QA & Risk · Pragmatist) — Round 1**

The simplest version that actually ships is a four-surface app under one native iOS target: planning/search, route results, reporting, and saved routes/paywall. The risky part is not rendering a map; it is making sure every state tells the truth. So the architecture should optimize for one thing: route geometry is remote and temporary, obstacle intelligence is local and replayable. If that line gets blurred, offline behavior, tests, and user trust all collapse at once.

I would keep the build split into plain-value domain logic, thin services, and feature view models. Domain logic owns obstacle matching, supersession, friction scoring, confidence scoring, route explanation copy inputs, and travel-order sorting. Services own MapKit search and directions, location permission/current location, SwiftData persistence and seed installation, purchases, and coarse reachability. View models should be `@MainActor` and small: `PlanViewModel`, `RouteResultViewModel`, `SavedViewModel`, `ReportViewModel`, `SettingsViewModel`. The smallest safe concurrency rule is that all UI state mutates on the main actor, while MapKit calls and persistence fetches happen asynchronously and return plain structs. Reporting never asks MapKit for a fresh route; it rescales the already-held route locally. That detail has to stay locked because it is what makes offline reporting real instead of aspirational.

The file layout should mirror the failure boundaries, not the screens alone. I would use `App/` for app entry, tabs, dependency wiring, and app-wide environment; `Features/Plan`, `Features/RouteResult`, `Features/Report`, `Features/Saved`, `Features/Onboarding`, `Features/Settings`; `Domain/Routes` for `RouteSnapshot`, route explanation inputs, and scoring outputs; `Domain/Obstacles` for obstacle types, severity, report supersession, and matcher logic; `Data/Models` for SwiftData models only; `Data/Mappers` for model-to-domain conversion; `Services/Location`, `Services/Routing`, `Services/Persistence`, `Services/Purchases`, `Services/Reachability`; `UI/Components` for the shared cards/chips/sheets already locked in design; and `Tests/Unit`, `Tests/Integration`, `Tests/UI`. The important QA point is that `Domain/` must compile without SwiftUI, SwiftData, or MapKit UI types. If it depends on those, the test suite stops being fast and trustworthy.

The data model should stay narrow. `ObstacleReportRecord` needs a stable UUID, created/updated timestamps, provenance (`seeded` or `userEntered`), obstacle type, severity, note, coordinate, and a state that supports supersession instead of destructive overwrite. **Claude (Frontend Engineer · User Advocate) — Round 1**

I'll open with the client-side shape of this, since nobody's spoken yet this phase and the architecture review left the service boundaries named but not the concrete types — that's the gap I want to close before anyone starts coding lanes in parallel.

Start with navigation, because it's the thing most likely to quietly diverge between build lanes if we don't pin it now. We locked per-tab `NavigationStack`s in the architecture review, but Route Result gets pushed from three different places — a fresh search on Plan, a favorite reopen on Saved, and the onboarding demo card — and if each of those improvises its own push target we'll end up with three slightly different Route Result entry contracts. I want one enum per tab that is the actual navigation path type: `enum PlanRoute: Hashable { case search, case result(RouteResultInput) }` and `enum SavedRoute: Hashable { case result(RouteResultInput) }`, where `RouteResultInput` is a single plain struct (origin coordinate, destination coordinate, optional favoriteID, optional "isDemoCorridor" flag) that both tabs construct identically. That struct is the seam — whoever builds Plan and whoever builds Saved code against the same input type and never invent a second shape for "how do I get to Route Result."

For state ownership: Route Result needs its own `@MainActor @Observable` view model that holds exactly one `RouteRecommendation` (the fetched candidates, already scored) plus a `segment: Segment` (map/list) plus a `loadPhase` enum for the two-stage loader. Critically, this view model owns the fetched `RouteCandidate` polylines as plain structs from the moment MapKit returns them — never re-touches MapKit again for that screen's lifetime. When the report sheet saves, it doesn't call back into a "refresh" method that re-fetches; it calls `rescore()`, which re-runs `RouteScorer` against the same held polylines plus the updated obstacle set from the repository. That's the mechanism that makes "report it and watch the recommendation change" real and keeps it working offline, and I want the view model's public surface to make that impossible to get wrong — there should be no `refetchRoute()` method exposed to the report-sheet completion at all, only `rescore()`.

On the two-stage loading floor: this needs to be a concrete async pattern, not a vibe. Something like: kick off the real MapKit fetch, and in parallel start a `Task.sleep(for: .milliseconds(500))` for the second-stage floor; await the MapKit result first (stage 1 ends when it returns), then await the sleep task before flipping to stage-2-complete, since local scoring itself finishes near-instantly. I'd write this as `RouteResultViewModel.load()` doing `async let route = fetcher.fetchWalkingRoutes(...)`, then after awaiting it, `async let floor: Void = Task.sleep(...)`, score, then await the floor before publishing the scored state. This needs to live in the view model, not be scattered as a `DispatchQueue.asyncAfter` hack somewhere in the view, or it becomes untestable and someone will "fix" the jank by deleting it.

For Saved, the lazy-staggered refresh needs an actual per-row state machine, not a shared spinner: `enum FavoriteRowStatus { case idle, checking, upToDate, newIssues(count: Int), offlineUnavailable(lastKnown: Date?) }`. The view triggers `viewModel.checkIfVisible(favoriteID:)` from `.onAppear` on each row (SwiftUI already only calls that when a row scrolls into view in a `List`), and the view model gates concurrent checks so at most a small handful run at once rather than firing one MKDirections call per row on tab appear. This is exactly the kind of thing that looks fine with 2 test favorites and falls over with 8 real ones, so I want the concurrency cap named now: process at most 2 favorites at a time, queue the rest.

Performance detail worth locking: `ObstacleMatcher` doing point-to-polyline-segment projection for every stored obstacle against every route candidate is fine at seed-data scale (dozens of reports) but I don't want naive O(obstacles × segments) with no early exit. Cheap fix: compute a bounding box around the route polyline expanded by the 20m buffer first, and skip any obstacle whose raw coordinate falls outside that box before doing real projection math. Doesn't need to be clever, just needs the guard clause so it's not O(n²) against every obstacle in the store on every rescore.

Data models — I'm splitting these cleanly into domain (plain, framework-free) and persistence (SwiftData), per the architecture review's insistence that scoring never touches `@Model` types. Domain: `Coordinate` (plain lat/lon struct, since `CLLocationCoordinate2D` isn't `Hashable`/`Codable`), `ObstacleType` (six fixed cases), `ObstacleSeverity` (minor/moderate/severe), `ObstacleReport` (id, type, severity, coordinate, note, createdAt, updatedAt, provenance, supersedes), `RoutePolyline`, `RouteCandidate`, `ObstacleMatch` (report + distanceMeters + relationship: onRoute/nearby), `RouteScore` (frictionCategory, confidenceCategory, matchedObstacles, explanation string), `RouteRecommendation` (candidates + recommendedIndex + isSingleRouteResult). Scoring constants need to be real numbers, not just band names, so tests can assert exact edges: outer buffer 20m, inner buffer 10m; recency multipliers — under 7 days: 1.0, 8–30 days: 0.6, 31–90 days: 0.3, over 90 days: 0.1; type base weights — stairsNoRamp 3.0, noCurbCut 2.5, constructionBlockage 2.5, steepGrade 2.0, unevenSurface 1.5, temporaryHazard 1.0; severity multipliers — minor 1.0, moderate 1.75, severe 2.5. Confidence is computed separately from all reports in the outer buffer (not just scoring matches — a "looks clear now" report still counts as evidence of local activity): zero reports in buffer → low; fewer than 3 reports, or none under 30 days → moderate; 3+ reports with at least one under 30 days → high.

Persistence-side SwiftData models, written CloudKit-safe per the architecture review (every field defaulted, no unique constraints, relationships avoided in favor of plain foreign-key-style UUID fields): `ObstacleReportRecord`, `FavoriteRecord` (label, origin/destination coordinates only), `RecentDestinationRecord` (capped at 5, label+coordinate only), `CachedRouteSnapshotRecord` (favoriteID, encoded polyline, fetchedAt — this is what backs the "your last available route, from [day]" staleness copy). A thin mapping layer converts record → domain struct and back at the persistence boundary; nothing above that boundary ever imports SwiftData's `@Model` types.

File layout: `App/` (entry point, composition root wiring the model container and services together), `Domain/` (the plain structs and `RouteScorer`/`ObstacleMatcher` above, zero imports beyond Foundation/CoreLocation-for-math), `Persistence/` (SwiftData models, `ModelContainerFactory`, `DemoDataSeeder`, mapping extensions), `Services/` (`LocationService`, `RouteSearchService`, `ObstacleRepository`, `FavoritesRepository`, `RecentsRepository`, `ReachabilityMonitor`, `PurchaseService` — each protocol-backed per the architecture review's DI requirement), `Features/{Onboarding,Plan,Search,RouteResult,ObstacleDetail,ReportObstacle,Saved,Paywall,SharePreview,Settings}/` each with View + ViewModel, `DesignSystem/` (tokens + the locked component set), and `Tests/` split into a pure-domain unit target (no SwiftData, no MapKit, no CoreLocation — this is what makes the boundary-value tests fast) and an integration/UI target.

Testing strategy follows straight from that split: `RouteScorerTests` and `ObstacleMatcherTests` run against plain structs with fake data, asserting the exact boundary values already named (19m/21m, 9m/11m, day 6/8, day 30/31, day 90/91), the single-route wording branch, the tie-break, and a before/after supersession test. Integration tests use `SwiftDataObstacleRepository` against a real in-memory-then-reopened `ModelContainer` for the force-quit proxy, and `FakeRouteFetching`/`FakeLocationProviding` implementations of the protocols for view-model-level tests of the permission-denied → manual-origin flow and the offline-cached-route staleness copy. I'd also want a `PlanViewModelTests` covering the persisted-flag demo-card logic explicitly, since that's a one-line bug (derived-vs-persisted state) that would otherwise only surface manually.

One thing I want to flag as a real risk to my own plan: putting `rescore()` as the only mutation path on Route Result's view model is clean, but it means the view model needs to survive being backgrounded mid-session without losing its held `RouteRecommendation` — if the app is backgrounded during a report-sheet edit and the process gets suspended/reclaimed, we can't silently lose the in-progress route and paste a spinner back on resume. That's a small but real edge case worth a scene-phase test, not just an assumption that `@Observable` state survives backgrounding for free (it does within a live process, but not across a real termination — that has to degrade to "re-fetch this route" gracefully, not crash or show stale garbage).

The detail I'd fight hardest to keep from getting lost in translation to the build lanes: `RouteResultInput` and `rescore()` are the two contract points that make the "report an obstacle, watch the recommendation change, works offline" promise real rather than aspirational — if a build lane invents its own version of either one, the local-first story quietly breaks.

**What's actually unresolved:** the two interfaces-json blocks name overlapping concepts differently and sometimes shape them differently, and nobody has picked a winner yet. Concretely: `Coordinate` vs `GeoCoordinate`; `RouteCandidate` vs `RouteOption`/`RoutePolylinePoint`; `ObstacleMatch` vs `MatchedObstacle` (the latter carries a `routeProgress` value needed for travel-order list sequencing, which the simpler version drops); `ObstacleRelationship` (two cases) vs `RouteRelationship` (three cases including `ignored`); `RouteScore` vs `ScoredRoute` (one embeds a plain-string explanation, the other a structured `RouteExplanation` with warning chips — the structured one is what the UI actually needs); `RouteRequestError` vs `RoutingFailure` (the latter carries underlying error messages and a `cancelled` case that matters for search-debounce cancellation); `RouteScorer`/`ObstacleMatcher` as static pure functions vs. as protocols (`RouteScoring`/`ObstacleMatching`) for DI; sync vs. async repository protocols, and whether `ObstacleStore` needs both proximity-based supersession (new unprompted report) and by-ID supersession (explicit "looks clear now" on a known item) — right now only one mechanism is covered on either side, but the product spec needs both; and `LocationProviding` exposing raw `CLAuthorizationStatus` vs. a custom `LocationAuthorizationState` enum that actually distinguishes precise vs. approximate authorization, which `CLAuthorizationStatus` alone can't do. None of this is a values disagreement — it's just two people solving the same problem in parallel without syncing names, and the acceptance check for this phase explicitly says parallel build lanes can't invent incompatible types, so this has to get reconciled before the phase closes.

### Round 2


**Codex (QA & Risk · Pragmatist) — Round 2**

Claude’s merged draft is basically the right move, and I’d stop trying to preserve both parallel vocabularies now. The risk at this phase is not “did we pick the most elegant protocol name,” it’s “do the build lanes fork into subtly different route/result models and spend implementation week translating between them.” So I’d take Claude’s unified contract almost wholesale, but I want to tighten three places because they’re where real bugs hide.

First, I’m with Claude on keeping `RouteExplanation` structured instead of collapsing back to Codex’s plain `String`. The UI already needs the recommendation sentence, the why sentence, confidence detail, warning chips, and the status line as separate surfaces with different reading order and accessibility labels. If we reduce that to one blob now, someone will re-parse or re-split it downstream and we’ll get copy drift between map mode, list mode, share preview, and VoiceOver. That’s fake simplicity. Same story on `ObstacleMatch`: keep `routeProgress`. The accessible list’s whole value is travel-order sequencing, and if the matcher doesn’t output progression as part of the match result, that ordering gets recomputed ad hoc in UI code and becomes untestable. I also think Claude is right to split `PlaceSearching` from `RouteFetching`; geocoding/search failure and “no walking route found” are different truths and should stay different codepaths all the way up.

Where I do want to push back is on carrying a few too many “maybe useful later” shapes into the contract. `RouteSnapshot` is good and needed, but `CachedRouteStore.deleteExpiredRoutes(olderThan:)` is implementation detail, not cross-lane contract; the important thing is that cached snapshots expose `fetchedAt` and `source`, so UI can tell the truth. Same with `lastOpenedAt` on `Favorite`: harmless, but not essential to this MVP’s behavior. I’d rather keep the shared contract lean where possible and spend the specificity budget on failure modes and mutation paths. The most important invariants to lock are: `RouteResultInput` is the single navigation payload from Plan, Saved, and demo; `RouteResultViewModel.rescore()` is the only post-report mutation path and never refetches directions; `ObstacleStore` supports both supersede-by-ID and supersede-nearby because the product has both “looks clear now” on a known obstacle and “I found a new problem in the same corridor” as separate flows; and `FavoriteRowStatus` keeps the Saved tab honest about staggered refresh instead of letting rows silently churn in the background. If we leave any ambiguity there, we’ll ship the kind of bug that only shows up on a train platform with bad service: stale route copy, a report that triggers a network fetch, or a third favorite that half-saves before the paywall appears.

The other thing I’d explicitly keep from Codex’s instincts is the custom `LocationAuthorizationState` rather than raw `CLAuthorizationStatus`. Raw status isn’t enough if we care about honest permission UX, because approximate-vs-precise is user-visible reality even if this app can still function with either. I don’t think we need to overbuild around it, but I do think the UI contract should know that distinction exists. And I’d keep `RouteRequestError.cancelled` too, not because users see it, but because search-as-you-type and `.task(id:)` cancellation is a real source of stale-state bugs. If we don’t model cancellation explicitly, someone will catch-all it into a visible error and we’ll flash “something went wrong” while the user is still typing.

So my position is: stop negotiating names, adopt the merged contract, trim only the pieces that are persistence-implementation detail rather than shared behavior, and preserve the shapes that make the app’s honesty testable. The whole app lives or dies on not conflating states that sound similar but mean different things. “No route found,” “offline,” “cached route,” “no reported obstacles,” “low confidence,” “single route only,” “report superseded older one,” those are not copy polish, they’re the product. If the interfaces don’t encode those distinctions now, the UI will blur them later.

**Claude (Frontend Engineer · User Advocate) — Round 2**

Codex, your round-1 message and the coordinator's merge landed in almost the same place I would have picked independently, so let me just be precise about where I actually want to push back versus where I'm happy to concede, because "mostly agree" isn't useful without saying which parts.

The static-function-versus-protocol split is the one place I want to draw a hard line, because right now the merged contract is inconsistent about it and I think there's a clean rule that resolves every instance at once: protocol-ize anything that touches a platform resource you can't run in CI (network, CoreLocation hardware, StoreKit, disk), and keep anything that's pure computation as a plain static function. `RouteFetching`, `PlaceSearching`, `LocationProviding`, `ObstacleStore`, `FavoritesStore`, `RecentsStore`, `CachedRouteStore`, `PurchaseManaging`, `ReachabilityProviding` — all protocols, all correctly so, because a build lane needs to fake MapKit/CoreLocation/StoreKit/disk in tests. But `RouteScorer` and `ObstacleMatcher` don't need a swappable implementation — there's exactly one correct scoring algorithm, and a test calls it directly with known inputs and asserts the output. Making it a protocol doesn't buy testability, it just adds a layer where someone could accidentally swap in a fake scorer in production code, which is precisely the kind of thing that would quietly break the "trust the recommendation" promise. Codex, your original draft had `RouteScoring` and `ObstacleMatching` as protocols — I get the instinct for consistency with the other services, but I'd push back specifically here. Same logic applies to `RouteExplanationBuilding`: it's template-driven deterministic copy generation, not something we'll ever fake in a test, so I want to flatten it to a static `RouteExplanationBuilder.build(...)` function rather than a protocol. This isn't a stylistic nitpick — it's the thing that keeps someone from writing a `MockRouteScoring` "for testing" that quietly diverges from the real math and gives false green tests.

On the `ObstacleRelationship`-versus-`RouteRelationship` question, I want to defend the two-case version over Codex's three-case one with `ignored`, and here's the concrete reason: `ObstacleMatch` is the output of `ObstacleMatcher.match()`, and that function only ever constructs an `ObstacleMatch` for obstacles that actually matched (inside the 20m outer buffer). Anything beyond 20m never becomes an `ObstacleMatch` at all — there's no instance sitting around with `relationship: .ignored` for the UI or scorer to trip over. Adding that case creates a state that's supposed to be unreachable in practice but that every switch statement over `ObstacleRelationship` now has to account for, which is exactly the kind of accidental complexity Codex was warning about with catch-all errors, just in the other direction — I'd rather the type system only be able to express the two truths that actually exist post-matching.

I do want to concede a real point to Codex, though, and flag something I dropped that the coordinator caught: idempotent, versioned seed data installation. That's a genuinely good catch and I hadn't thought about what happens on an app update where we want to revise the demo corridor's obstacle set — without a version stamp, `installSeedDataIfNeeded()` either never updates stale seed content or duplicates it every launch. I'd add a `currentSeedDataVersion` constant checked against a persisted installed-version flag, and I'd also add an explicit `resetSeedData()` method to `ObstacleStore` since the design phase locked "demo-data reset" as a Settings affordance — that needs a real method to call, not just an implicit side effect of reinstall logic.

One thing genuinely missing from the merged contract that I think matters more than it looks: nobody defined `Segment` even though `RouteResultViewModel.segment: Segment` references it directly — that's a real gap that would leave the Map/List toggle lane inventing its own enum. Trivial fix, but exactly the kind of thing that slips through when everyone's focused on the meatier types. I'm adding `enum Segment: String, CaseIterable, Sendable { case map, list }`.

The bigger addition I want to make is around the report-submission path, because right now `ObstacleStore` exposes three ways to write a report — `saveReport`, `supersede(existingID:)`, `supersedeNearby(coordinate:)` — and nothing in the contract says who decides which one to call. If that decision lives inside whoever builds the Report sheet's view model, we'll get divergent behavior depending on which build lane wrote it: one might always call `saveReport` and never trigger nearby-supersession for an unprompted new report landing in an existing obstacle's buffer, silently breaking the "friction score doesn't ratchet up forever" mechanism we locked back in detailed_discussion. So I want to introduce a `ReportDraft` struct (type, severity, note, coordinate, `looksClearNow`, and an optional `targetObstacleID` for when the report is filed against an already-flagged item) plus a single `ReportSubmission.submit(_:store:)` static function that owns the decision tree: if there's a `targetObstacleID`, call `supersede(existingID:)`; otherwise check whether an active report already exists within the inner buffer of the draft's coordinate and call `supersedeNearby` if so, else `saveReport`. That collapses the Report sheet's completion handler down to "build a `ReportDraft`, call `ReportSubmission.submit`, then call `routeResultViewModel.rescore()`" — three calls, no judgment required, no way for a build lane to get the supersession logic subtly wrong.

Last thing, and this is squarely User Advocate territory: `LocationAuthorizationState` distinguishing `authorizedApproximate` from `authorizedPrecise` isn't just nicer modeling than raw `CLAuthorizationStatus` — it's a real product risk for this specific app that nobody's named yet. If someone grants "approximate" location (iOS's reduced-accuracy default that a lot of users leave on for privacy reasons), the origin coordinate handed to `RouteFetching` could be off by a hundred-plus meters, which for a pedestrian accessibility app means the recommended route could start from the wrong side of a block or the wrong side of an intersection entirely — a materially different walk than the one being scored. I think `LocationService` needs to treat `authorizedApproximate` as a state that nudges toward manual pin-drop rather than silently trusting the fuzzy coordinate as if it were exact, especially given this user base is the one least able to absorb "the app told me to start walking from the wrong corner." That's not blocking for MVP, but I want it named in the contract notes so whoever builds `LocationService`/Plan doesn't just treat all "authorized" cases as equivalent.

### Round 3


**Codex (QA & Risk · Pragmatist) — Round 3**

Claude’s additions are the right kind of boring, which is what this phase needed. The open gaps now are not product ideas, they’re “who owns the sharp edge so a build lane doesn’t accidentally make up behavior.” On that standard, I’m strongly in favor of keeping `ReportDraft` plus `ReportSubmission`. Right now the store exposes three write paths and the most failure-prone screen in the app is the report sheet. If each UI lane decides for itself whether a new report should call `saveReport`, `supersede(existingID:)`, or `supersedeNearby`, you’ve created three opportunities to silently violate the supersession rule and let friction scores ratchet upward forever. That is exactly the kind of regression QA only catches late because the app still “works.” Centralizing that decision in one tiny utility is cheaper than testing every screen against every report-path permutation. Same story on `AppDependencies`: I don’t care about DI purity here, I care that one composition root prevents ad hoc service wiring and weird mismatches between Plan, Saved, and Route Result. And `Segment` obviously stays; if the view model references it, it belongs in the contract.

I also think Claude is right to flatten `RouteExplanationBuilder` into a pure static function. Codex already accepted that rule for scorer and matcher, and explanation generation is the same class of problem: deterministic, local, no legitimate runtime variants, and heavily unit-testable. Making it a protocol just creates a fake seam without reducing risk. The only place I’d push back a bit is on keeping every convenience field just because it might be nice later. `Favorite.lastOpenedAt` is not earning its keep in the shared contract. Nothing in the locked UX requires it, and it’s exactly the kind of metadata that quietly becomes an accidental behavior source or privacy footnote. Cut it. By contrast, I would keep `deleteExpiredRoutes(olderThan:)` on `CachedRouteStore`, not because the UI needs to call it directly, but because cached-route honesty is a locked product requirement and expiry is part of that lifecycle, not just storage housekeeping. If it stays implicit, it gets deferred; if it’s named, someone owns it. I’d also keep `currentSeedDataVersion`, `resetSeedData()`, `RouteCandidate.advisoryNotices`, and the approximate-location authorization case. Those are all real trust surfaces: demo data has to be updateable without duplication, MapKit advisories are extra explainability we get nearly for free, and approximate location can absolutely put a pedestrian on the wrong side of a block, which for this app is not a cosmetic distinction.

So my merge position is: accept Claude’s additions, trim `lastOpenedAt`, keep explicit cache expiry, keep the explanation builder as a pure function, and close the contract. At that point the spec is finally doing the thing this phase is supposed to do: parallel lanes can build without inventing incompatible types, and QA has named places to assert behavior instead of relying on “the screen seemed fine.”

**Claude (Frontend Engineer · User Advocate) — Round 3**

Codex's round 2 message is basically the same document as mine minus the parts that hadn't crossed yet, and looking at it again I think there are exactly two live disagreements left plus one thing I should just concede outright. Let me deal with the concession first because I don't want to be stubborn about it: Codex, you're right about `Favorite.lastOpenedAt` and `CachedRouteStore.deleteExpiredRoutes(olderThan:)`. I put both back in my round-2 draft without actually rebutting your argument, which wasn't fair. Nothing in the locked product spec asks for favorites to be sorted or displayed by last-opened time — the Saved screen shows live status per row, not a recency ranking — so `lastOpenedAt` is a field with no consumer anywhere in the contract. And `deleteExpiredRoutes` is purely internal cleanup: no other lane ever calls it or needs its signature to build against, because every UI decision about staleness already comes from `fetchedAt` plus `source` on whatever `loadCachedRoute` hands back. Whether the `services_utilities` lane purges expired rows via a scheduled sweep, on next-write, or never bothers because SwiftData storage for a handful of cached snapshots is trivial, is genuinely none of the other lanes' business. I'm dropping both. That's the acceptance check working as intended — "parallel build lanes can work without inventing incompatible types" cuts both ways, it also means don't over-specify things nobody outside one lane will ever touch.

Where I'm not conceding: `RouteExplanationBuilding` as a protocol. Codex's round 2 kept it exactly as originally drafted, but that was written in parallel with my flattening argument, not in response to it, so let me make the case one more time, sharper. The entire point of protocol-izing a service is that a test needs to substitute a fake because the real implementation touches something a test environment can't or shouldn't touch — a network, a location manager, a purchase receipt, a disk store. `RouteExplanationBuilder.build(...)` touches none of that. It's a pure function from `[ScoredRoute]` plus a `Date` to a `RouteExplanation` struct. A test for it looks exactly like a test for `RouteScorer.score` — construct known inputs, call the function, assert the output string/chips. There is no scenario where a build lane legitimately needs to inject a `MockRouteExplanationBuilding` instead of just calling the real one, because "the real one" is the thing under test, same as `RouteScorer` and `ObstacleMatcher`. If we leave this as a protocol while flattening the other two, we've created an inconsistent rule that someone building against this contract has to guess at — "why is scoring a static function but explanation-building is a protocol, when both are pure deterministic transforms?" I'd rather have one clean rule everywhere: protocol for platform-resource seams, static function for pure computation, no exceptions, no judgment calls. I'm keeping `RouteExplanationBuilder` flattened unless Codex has an actual scenario where we'd swap implementations — and "maybe we localize copy differently someday" doesn't count, because that's a template/strings problem, not a DI problem, and this MVP ships one language, no A/B testing infrastructure, no backend to drive variant selection.

Now, something I want to sharpen on my own addition from last round rather than just let it sit: `ReportSubmission.submit`'s decision tree said "supersedeNearby if an active report already exists within `innerBufferMeters` of the coordinate" — I wrote that fast last round and I want to make sure it's actually the right threshold rather than just reusing a number that was lying around. There are two genuinely different distance questions in this app and I don't want them to collapse into "one buffer number for everything": the outer 20m buffer answers "is this obstacle close enough to this route to matter for scoring," while the supersession question is "is this new report describing the same physical hazard as an existing report." Those aren't the same question at all — two obstacles 18 meters apart along the same block could easily be a broken curb cut at one corner and unrelated construction scaffolding at the next, and if we used the 20m outer buffer as the supersession-proximity check, submitting a report about the scaffolding could wrongly supersede-and-fade the still-valid curb-cut report just because they're both routing-relevant to the same street segment. The inner 10m threshold is the right one to reuse here — it's already our "same immediate spot" semantic elsewhere in the app (on-route versus nearby), and reusing it for "is this the same obstacle" is at least a coherent, single meaning for that number rather than a coincidence. So I want that explicitly written into the contract note, not left implicit, because whoever builds `ReportSubmission` needs to know this is a deliberate choice, not an arbitrary reuse of whatever constant was closest.

One more small thing since Codex's round 2 didn't touch it either way and I want to make sure it doesn't get silently dropped in a future merge: I added `advisoryNotices: [String]` onto `RouteCandidate` to carry MapKit's own `MKRoute.advisoryNotices` through, when present. I'll flag it as genuinely optional and low-stakes — it costs nothing (empty array when MapKit returns none, which will be the common case), doesn't require any UI to consume it in MVP, and just keeps a door open for the why-text to eventually say something like "MapKit notes this route may include stairs" if Apple's own data happens to say so. If Codex wants to cut it as premature, I won't fight for it — it's the kind of thing that's fine to keep or drop, unlike the explanation-builder question where I think there's a real principle at stake.

Putting this together, I think the contract is basically done. The two trims are settled in Codex's favor, the explanation-builder question I'm holding firm on with a real rule behind it, the supersession-proximity choice is now explicit rather than implicit, and everything else — `ReportDraft`/`ReportSubmission`, `Segment`, `AppDependencies`, seed-data versioning, the approximate-location risk note — nobody has raised an actual objection to, just hadn't seen it yet. I think this is ready to close.

## Final Output

**Architecture overview:** A small, disciplined SwiftUI app built around one hard rule that both agents converged on independently: protocols exist only where a build lane needs to fake a platform resource in tests (network, CoreLocation, StoreKit, disk) — `RouteFetching`, `PlaceSearching`, `LocationProviding`, `ObstacleStore`, `FavoritesStore`, `RecentsStore`, `CachedRouteStore`, `PurchaseManaging`, `ReachabilityProviding`. Everything else that's pure, deterministic computation is a plain static function with no fakeable seam — `RouteScorer`, `ObstacleMatcher`, `RouteExplanationBuilder`. Route geometry is fetched from MapKit exactly once per request or favorite-open, converted immediately into app-owned plain structs (`RouteSnapshot`/`RouteCandidate`), and never re-fetched to reflect a new report — `RouteResultViewModel.rescore()` is the only mutation path after a report, and it never calls `RouteFetching` again. A single `AppDependencies` composition-root bundle is built once in `App/` and injected everywhere, so no feature wires up its own ad hoc service instances. `RouteResultInput` is the one shared navigation payload Plan, Saved, and the onboarding demo card all construct identically, closing off the risk of three slightly different push contracts into Route Result.

**File/module layout:** `App/` (entry point, composition root wiring `AppDependencies` and the `ModelContainer`), `Domain/` (plain structs plus `RouteScorer`/`ObstacleMatcher`/`RouteExplanationBuilder` — zero imports beyond Foundation/CoreLocation-for-math, zero SwiftData or MapKit types), `Persistence/` (SwiftData `@Model` records, `ModelContainerFactory`, `DemoDataSeeder`, mapping extensions to/from domain structs), `Services/` (concrete `LocationService`, `RouteSearchService`, `ObstacleRepository`, `FavoritesRepository`, `RecentsRepository`, `ReachabilityMonitor`, `PurchaseService`, each conforming to its protocol), `Features/{Onboarding,Plan,Search,RouteResult,ObstacleDetail,ReportObstacle,Saved,Paywall,SharePreview,Settings}/` each with a View + `@Observable` view model, `DesignSystem/` (tokens plus the locked component set), and `Tests/` split into a pure-domain unit target (no SwiftData, MapKit, or CoreLocation — this is what keeps the boundary-value tests fast) and a separate integration/UI target.

**Data models:** Domain layer (framework-free, `Sendable`, used everywhere scoring/matching/UI logic runs): `Coordinate`, `RouteEndpoint`, `ObstacleType`/`ObstacleSeverity`/`ReportProvenance`, `ObstacleReport`, `RoutePolyline`, `RouteCandidate`, `RouteSnapshot`/`RouteSnapshotSource`, `ObstacleRelationship`, `ObstacleMatch`, `ConfidenceLevel`, `FrictionCategory`, `RouteExplanation`/`WarningChipModel`/`WarningChipKind`, `ScoredRoute`, `RouteRecommendation`, `ScoringConstants`, `ReportDraft`, `Favorite`, `RecentDestination`, `ShareSnapshot`. Persistence layer (SwiftData, CloudKit-safe from day one — defaults on every attribute, optional relationships, no unique constraints, explicit stable UUID fields): `ObstacleReportRecord`, `FavoriteRecord`, `RecentDestinationRecord`, `CachedRouteSnapshotRecord`. A thin mapping layer at the `Persistence/` boundary is the only place these two worlds touch.

**Persistence strategy:** Obstacle reports carry `seeded`/`userEntered` provenance and use supersession (`isActiveForScoring` flag, newer-weighted, old row retained for history) rather than hard delete, with two distinct supersession paths — explicit by-ID (the "looks clear now" action on a known obstacle) and proximity-based via the inner 10m buffer (an unprompted new report landing on the same physical hazard as an existing one) — both centralized behind `ReportSubmission.submit`, so no UI lane has to decide which write path applies. Favorites store origin/destination endpoints only, never a frozen polyline, and always re-fetch live on open. Cached route snapshots are explicitly ephemeral, carry their own `fetchedAt` and `source`, and honor a staleness threshold (~1 day) past which offline-review copy shifts from "today's suggested route" to "your last available route, from [day]"; `CachedRouteStore.deleteExpiredRoutes(olderThan:)` gives that lifecycle a named owner rather than leaving expiry implicit. Seed data installation is versioned (`ScoringConstants.currentSeedDataVersion` checked against a persisted installed-version flag) so a later update can revise the demo corridor without duplicating or silently ignoring the change, and `resetSeedData()` backs the locked Settings "demo-data reset" control. Recents are capped at five, storing label+coordinate only, no per-search timestamp trail.

**Testing strategy:** Pure unit tests over plain structs for `RouteScorer`/`ObstacleMatcher`/`RouteExplanationBuilder` — none require SwiftData, CoreLocation, or a live network path — asserting the exact boundary values baked into `ScoringConstants`: 19m/21m outer buffer, 9m/11m inner buffer, day 6/8, day 30/31, day 90/91 recency bands, the single-route wording branch, the shorter-route tie-break, and a before/after supersession test proving a new report changes score immediately. `RouteFetching`/`LocationProviding`/`PlaceSearching`/`PurchaseManaging`/`ReachabilityProviding` are faked in view-model tests via their protocols. The "survives force-quit" requirement is tested via two independent `ModelContainer` instances against the same on-disk store URL. `performAccessibilityAudit()` runs as a mechanical CI gate alongside one deliberate manual VoiceOver pass per milestone for the semantic reading-order requirement it can't catch.

## Final Output

**Architecture overview:** A small, disciplined SwiftUI app built around one hard rule that both agents converged on independently: protocols exist only where a build lane needs to fake a platform resource in tests (network, CoreLocation, StoreKit, disk) — `RouteFetching`, `PlaceSearching`, `LocationProviding`, `ObstacleStore`, `FavoritesStore`, `RecentsStore`, `CachedRouteStore`, `PurchaseManaging`, `ReachabilityProviding`. Everything else that's pure, deterministic computation is a plain static function with no fakeable seam — `RouteScorer`, `ObstacleMatcher`, `RouteExplanationBuilder`. Route geometry is fetched from MapKit exactly once per request or favorite-open, converted immediately into app-owned plain structs (`RouteSnapshot`/`RouteCandidate`), and never re-fetched to reflect a new report — `RouteResultViewModel.rescore()` is the only mutation path after a report, and it never calls `RouteFetching` again. A single `AppDependencies` composition-root bundle is built once in `App/` and injected everywhere, so no feature wires up its own ad hoc service instances. `RouteResultInput` is the one shared navigation payload Plan, Saved, and the onboarding demo card all construct identically, closing off the risk of three slightly different push contracts into Route Result.

**File/module layout:** `App/` (entry point, composition root wiring `AppDependencies` and the `ModelContainer`), `Domain/` (plain structs plus `RouteScorer`/`ObstacleMatcher`/`RouteExplanationBuilder` — zero imports beyond Foundation/CoreLocation-for-math, zero SwiftData or MapKit types), `Persistence/` (SwiftData `@Model` records, `ModelContainerFactory`, `DemoDataSeeder`, mapping extensions to/from domain structs), `Services/` (concrete `LocationService`, `RouteSearchService`, `ObstacleRepository`, `FavoritesRepository`, `RecentsRepository`, `ReachabilityMonitor`, `PurchaseService`, each conforming to its protocol), `Features/{Onboarding,Plan,Search,RouteResult,ObstacleDetail,ReportObstacle,Saved,Paywall,SharePreview,Settings}/` each with a View + `@Observable` view model, `DesignSystem/` (tokens plus the locked component set), and `Tests/` split into a pure-domain unit target (no SwiftData, MapKit, or CoreLocation — this is what keeps the boundary-value tests fast) and a separate integration/UI target.

**Data models:** Domain layer (framework-free, `Sendable`, used everywhere scoring/matching/UI logic runs): `Coordinate`, `RouteEndpoint`, `ObstacleType`/`ObstacleSeverity`/`ReportProvenance`, `ObstacleReport`, `RoutePolyline`, `RouteCandidate`, `RouteSnapshot`/`RouteSnapshotSource`, `ObstacleRelationship`, `ObstacleMatch`, `ConfidenceLevel`, `FrictionCategory`, `RouteExplanation`/`WarningChipModel`/`WarningChipKind`, `ScoredRoute`, `RouteRecommendation`, `ScoringConstants`, `ReportDraft`, `Favorite`, `RecentDestination`, `ShareSnapshot`. **Persistence strategy:** Obstacle reports carry `seeded`/`userEntered` provenance and use supersession (`isActiveForScoring` flag, newer-weighted, old row retained for history) rather than hard delete, with two distinct supersession paths — explicit by-ID (the "looks clear now" action on a known obstacle) and proximity-based via the inner 10m buffer (an unprompted new report landing on the same physical hazard as an existing one) — both centralized behind `ReportSubmission.submit`, so no UI lane has to decide which write path applies. Favorites store origin/destination endpoints only, never a frozen polyline, and always re-fetch live on open. Cached route snapshots are explicitly ephemeral, carry their own `fetchedAt` and `source`, and honor a staleness threshold (~1 day) past which offline-review copy shifts from "today's suggested route" to "your last available route, from [day]"; `CachedRouteStore.deleteExpiredRoutes(olderThan:)` gives that lifecycle a named owner rather than leaving expiry implicit. Recents are capped at five, storing label+coordinate only, no per-search timestamp trail.

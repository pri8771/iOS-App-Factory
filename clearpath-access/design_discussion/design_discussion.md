The final onboarding step is location permission, but with a clear “plan without my location” path so denial never feels like failure. The home/planner screen is the primary shell: origin field, destination field, swap/edit controls, recent destinations, favorite routes, a map preview, and a visible trust banner saying reports are user-submitted and unverified. Search should be its own modal or pushed screen with Apple-style search results, recent places, and a “drop pin manually” entry path for origin. Route results should be the real centerpiece: recommended route card at the top, plain-English why/confidence line, warning chips for on-route versus nearby obstacles, map with route overlays, and a segmented toggle or stacked switch between map view and route details list. Obstacle details should not be a separate deep screen unless needed; tapping a flagged item can open a bottom sheet with type, severity, recency, note, and whether it is on-route or nearby. Favorites should likely live as a compact library screen reached from home, with named routes, saved endpoints, last checked time, and share action. The share/export screen can just be a system share sheet triggered from route results or favorites, but the pre-share preview matters because the user needs to know what the caregiver will actually see. The upsell screen is simple: unlimited favorites plus caregiver-ready route export, nothing broader. Settings is small but necessary: disclaimer, sample-data explanation, restore purchases, note privacy guidance, and maybe a demo-data reset.

The primary user flow should be brutally short. Open app, choose or confirm origin, choose destination, fetch route, see one recommendation with a reason and confidence, inspect issues in travel order, optionally save as favorite or share, optionally submit a report, then re-run and watch the recommendation update. If location is denied, the flow is identical except origin starts as manual entry. If the user is a repeat user, the real daily flow is even shorter: open, tap favorite, see today’s suggested route to that destination, check whether anything new affects it, leave.

The route result screen needs the most disciplined state model because that is where trust is won or lost. Before search, it is an instructional empty state with favorites and recent destinations. While routing, it shows a loading state that distinguishes “fetching route” from “scoring obstacles” if those happen sequentially. On success with multiple routes, it shows a recommendation plus alternatives summary. On success with one route, it must switch language to “only walking route found” and stop implying comparison. On success with zero matched obstacles, it should say “no reported obstacles on this route” and still show confidence separately. On low-confidence corridors, the confidence warning needs equal visual weight to the recommendation, not tiny caption text. On offline-new-search failure, the screen should preserve entered endpoints and say clearly that new routing is unavailable right now. No nearby reports in the visible map area should be an honest informational state, not an error state.

**Claude (Frontend Engineer · Skeptic) — Round 1**

I want to start by pressure-testing three things from the locked design phase before I add anything new, because if these don't hold up architecturally, the rest of the handoff inherits false confidence.

First, the two-stage loading state ("finding walking route" then "checking reported obstacles") got treated as a branding win in the design phase, and I think that's dangerously optimistic without a build-side caveat. MKDirections.calculate() is a real network round-trip, but obstacle scoring against a cached SwiftData store within a 20m buffer is a local computation that will often finish in under 50ms. If we render that second stage honestly — show it only as long as it actually takes — it'll flash and read as a glitch, not a deliberate step. The fix has to be explicit in the handoff: the "checking reported obstacles" stage gets an artificial minimum hold (something like 400–600ms) so it's perceivable, and that's a product decision, not an implementation footnote, because it means we're deliberately delaying a result the app already has. If nobody writes that down now, whoever builds this will either cut the stage silently or ship a flicker and call it a bug later.

Second, "favorites re-fetch live, never a frozen polyline" sounds clean in a product discussion and becomes a real problem the moment someone has more than two or three saved routes, which is the entire point of the paid tier. If the Saved tab eagerly triggers a live MKDirections call plus obstacle rescoring for every row the instant the tab appears, a user with eight saved routes just fired eight concurrent network requests and eight polyline-distance computations on tab switch. That's a battery and rate-limit risk, and it also means "no new reported issues" badges will pop in staggered over a second or two, which needs its own loading-per-row state, not a blocking spinner over the whole list. My position: refetch is lazy and staggered — visible rows only, one row's status computed as it scrolls into view or on a short-lived visibility trigger, not all-at-once on tab appear. This needs to be named as a state in the Saved screen spec, not left as an implementation detail, because it changes what "the list" looks like in the first 500ms after opening it.

Third, and this is the one I feel strongest about: the design phase talks about a report "changing the score of a route re-requested immediately after," and I want to kill any reading of "re-request" that implies calling MapKit again. Resubmitting to MKDirections after a report is wasteful, adds a second network dependency into a flow that's supposed to work offline, and actually breaks the offline-cached-route state that was already named as a requirement — you can't refetch a route you have no network for, but you can absolutely rescore an already-fetched polyline against updated local obstacle data. So the architecture has to cleanly separate "fetch route geometry" (network, MapKit, happens once per search or favorite-open) from "score route against obstacles" (pure local computation, replayable any number of times against the same cached polyline). The report sheet's dismissal should trigger a rescore of whatever route is currently held in view state, never a new directions call. If this separation isn't explicit in the handoff, someone will wire "submit report" to just call the whole search pipeline again, which will silently break offline reporting and probably introduce a network dependency into what was supposed to be the local-first core loop.

On state management shape: I'd anchor this on a per-tab NavigationStack (Plan's stack and Saved's stack are independent path arrays), not one global stack, because tapping a favorite from Saved and tapping a destination from Plan both push a Route Result screen, and if there's only one shared stack, backgrounding mid-flow or switching tabs mid-push gets ambiguous about which tab "owns" that pushed screen. Route Result itself should be driven by a single `@Observable` view model holding both the fetched route (or nil), the obstacle score/confidence result (recomputed independently), and the current segment selection (map/list) — and MKMapView should not be instantiated at all while the List segment is selected. Keeping a live MapKit view resident just because the user is looking at the list is unnecessary memory and rendering overhead in an app that already has exactly one screen where a map appears at all; lazy-load it only when the Map segment is chosen, and let it deallocate (not just hide) if the user backs out.

I also want to flag the demo-card visibility rule as under-specified in a way that will produce a confusing bug. The design phase said the "see how it works" card on Plan appears only "on a genuinely fresh install with zero recents and zero favorites." If that condition is derived live from current data state rather than from an explicit persisted flag, then a user who saves and later deletes both of their favorites will see the onboarding demo card reappear, which reads as broken, not helpful. This needs one boolean in UserDefaults or a lightweight settings record — `hasCompletedFirstRouteSearch` — set once and never unset, so "fresh install" is a真 one-time state, not a recomputed one.

One more architectural point worth locking now because it affects testability, which this app's whole product promise depends on: the scoring/obstacle-matching engine (buffer math, severity×recency formula, confidence derivation) must operate on plain Swift structs, not directly on SwiftData `@Model` types. Point-to-polyline distance calculation (there's no built-in MapKit API for "distance from point to route" — it has to be computed by projecting the obstacle coordinate onto each segment of the route polyline and taking the minimum) is exactly the kind of logic that's easy to get subtly wrong at the 19m/21m boundary, and it needs to run in unit tests with zero SwiftData/ModelContainer setup, zero MapKit dependency, fast and deterministic. So: `RouteScorer` and `ObstacleMatcher` take `RoutePolyline` and `ObstacleReport` value types as plain input/output, and the SwiftData layer maps into and out of those at the edges. If this boundary gets blurred — if scoring logic ends up reaching into `@Query` results directly — the boundary-value tests this whole project has been promising since the detailed-discussion phase become integration tests in disguise, slower and more brittle than they need to be.

For location, I'd name a concrete component: a `LocationService` wrapping `CLLocationManager`, `@MainActor`-isolated, exposing authorization status as an `@Observable` property so permission-revoked-mid-session is just a state change the Plan and Route Result views react to, not a manual polling check. Authorization change delivery from CoreLocation is delegate-based, so this needs an explicit bridge into the actor-isolated observable state per the standard main-actor-hop pattern, not a raw delegate touching UI state directly.

Screen-by-screen, layered onto the already-locked inventory: Route Result at maximum Dynamic Type is the single hardest layout problem in the app, and I don't think a fixed VStack survives it. The recommendation sentence, confidence badge, and why-text can each grow substantially, and if the segmented Map/List control and the content below it are laid out in a rigid stack, accessibility-size text will push the control and the map/list content off-screen on an SE-class device with no way to reach it. This screen needs to be a ScrollView with the recommendation/why/confidence block, the segmented control, and the map-or-list content all scrolling together — not a fixed header with a scrollable region underneath — otherwise "grows the card, doesn't truncate" quietly becomes "grows the card off the bottom of the screen."

## Final Output

**Screen inventory:**
1. **Onboarding** — three cards max (what the app does, what it doesn't guarantee, how sample/local data works), ending in a location-permission step with an equal-weight "plan without my location" path.
2. **Plan** (tab 1) — text-first: origin field (current location or manual/dropped-pin, both first-class), destination search field, a single one-row shortcut to the most relevant recent/favorite destination, a persistent trust banner, and — only on a genuinely fresh install with zero recents/favorites — one tappable "see how it works" demo card. No map surface on this screen, ever.
3. **Destination/Origin Search** — modal or pushed screen: search results, recent places, manual pin-drop entry.
4. **Route Result** — the hero screen, pushed from Plan (real search) or from the fresh-install demo card (seeded corridor data). Recommendation sentence, confidence badge (icon + word, never color-only), plain-language why, all as co-equal outputs above the fold; a segmented `Map / List` control below (not a screen push); a compact route-switcher chip row when MapKit returns multiple routes, absent entirely for single-route results; warning chips distinguishing "on this route" vs "nearby."
5. **Obstacle detail** — bottom sheet from tapping a flagged item (map pin or list row): type, severity, recency, note, on-route/nearby status, inline report action.
6. **Report Obstacle** — bottom sheet, reachable from Route Result's persistent action or any flagged item: type chips, severity, optional note, "looks clear now" toggle when applicable, save (target: under 30 seconds).
7. **Saved** (tab 2) — full list-management screen for favorites: each row live-refetches and shows current status ("no new reported issues" / "N new reports since saved"), a persistent "2 of 2 free" indicator once at cap, share action per row.
8. **Paywall** — sheet triggered at the 3rd favorite save attempt, showing the actual route just tried to be saved underneath a translucent unlock panel; sells unlimited favorites + share export specifically.
9. **Share preview** — shown before invoking the system `ShareLink`, previewing exactly what the recipient will see (destination, recommendation, confidence, timestamp, unverified-data line).
10. **Settings/info** (toolbar affordance, not a tab) — full disclaimer text, sample-data explanation, restore purchases, privacy guidance on notes, demo-data reset.

**Primary user flow:**
First-time: onboarding → Plan (manual or current-location origin) → destination search → Route Result (two-stage load: "finding walking route" → "checking reported obstacles") → recommendation/why/confidence → inspect via Map or List → optionally report an obstacle (score updates live if returned to the same result) → optionally save as favorite or share. Repeat/habitual flow (the actual daily case): open app → tap favorite shortcut on Plan or in Saved → Route Result re-fetches live → check "no new issues" → leave. Permission-denied follows the identical flow with manual origin entry substituted, never a dead end.

**State model per screen (key states):**
- *Plan:* fresh-install-empty-with-demo-card; populated-with-recents/favorites; location-permission-denied (inline banner, not modal); destination-search-empty/ambiguous.
- *Route Result:* two-stage loading (fetch → score); success-multi-route; success-single-route (language shifts to "the only walking route found," switcher chips absent); success-zero-matched-obstacles ("no reported obstacles on this route," confidence shown separately, never "safe"); low-confidence (equal visual weight to the recommendation, not caption text); offline-new-search-failure (endpoints preserved, distinct honest copy); offline-cached-route (obstacle scoring still refreshes); geocoding failure (editable in place, no backward bounce); permission lost mid-session (falls back to manual origin with explanation).
- *Saved:* empty (no favorites yet); under-cap; at-cap (paywall intercept on next save attempt); favorite reopened (live re-fetch, "today's suggested route" framing).
- *Report sheet:* default entry; reporting against an existing flagged item (adds "looks clear now"); save success (visible live update on the underlying result if returned to it).
- *Paywall:* pre-purchase; purchasing; restore.

**Visual direction:** Calm, premium, trust-first — deep slate/ink rather than default navigation blue, muted teal/petrol for primary route emphasis, amber strictly for moderate flags, rust/coral strictly for severe on-route hazards, no green-means-safe semantics anywhere. Confidence is its own badge system (icon + word). Seeded/demo content gets a distinct, consistent visual treatment (subtle dashed outline or badge) so it's never mistaken for real data at a glance. Typography carries the "premium" weight: the recommendation sentence is the visual anchor of Route Result, generous spacing, built to survive largest Dynamic Type by growing card height, not truncating — hardest on Route Result and the paywall screen.

**Accessibility notes:** The List representation on Route Result is a first-class peer to Map, not a fallback — reached via segmented control, never a buried screen push. Travel-order sequencing is the default everywhere flagged items appear. VoiceOver labels encode obstacle type, severity, recency, and on-route/nearby relationship in a single pass per item. Color never solely carries the on-route vs. nearby distinction — always paired with icon/text. Report sheet uses large tap targets, minimal custom gestures, operable one-handed and with Switch Control. The trust disclaimer is persistently reachable from Route Result but doesn't dominate reading order on every screen open.

## Final Output

**Decision on external design:** No external Claude Design pass. This handoff (Codex's screen-by-screen spec + Claude's architecture/token/state refinements, both below) is the build source of truth for direct SwiftUI implementation. If outside visual taste is wanted later, it's scoped narrowly as a post-build polish review of finished screens against the token system — never a reopened concept phase.

**Screen-by-screen specification:**

1. **Onboarding** — 3 cards, stop there: (1) what it does, framed around avoiding reported barriers, not fastest route; (2) what it doesn't guarantee — the unverified-data warning stated plainly; (3) how local/sample data works, including that some areas will have little or no data. Final action row: "Use my location" / "Plan without my location" as visually equal-weight choices (not primary/secondary styling) — manual origin is a core mode, not a fallback.

2. **Plan (tab 1)** — text-first, no map surface, ever. Origin field (current location or manual/pin), destination field, single primary action, one compact recent/favorite shortcut row, persistent quiet trust banner. On a genuinely fresh install (governed by a persisted `hasCompletedFirstRouteSearch` flag, never a live "zero recents/favorites" check — this stops the demo card from reappearing after a user deletes their favorites), one demo card: "See how route warnings work," which pushes straight into a fully populated real Route Result using seeded corridor data — no separate tutorial illustration.

3. **Search (origin/destination)** — pushed/modal screen: results, recent places, manual pin-drop path, explicit empty/ambiguous states. Permission-denied biases smoothly to manual entry, never framed as "degraded mode."

4. **Route Result (hero screen)** — built as **one scrolling container**, not a fixed header over a scrollable body (required for max Dynamic Type — otherwise the segmented control and content can get pushed off-screen). Order: recommendation sentence → confidence badge (icon+word) → why text, all one visual block; then on-route/nearby warning chips; then route-switcher chips (only when MapKit returns multiple routes — absent entirely, with honest "only walking route found" wording, when it returns one); then segmented Map/List control; then the selected representation's content. List mode is travel-order sequential, full-sentence rows, a genuine first-class peer, not a fallback.
   - **Two-stage loading is a locked spec requirement, not an implementation footnote:** stage 1 "finding walking route" (real network wait), stage 2 "checking reported obstacles" holds for a minimum 400–600ms even when local scoring finishes in milliseconds — a deliberate, intentional delay so the app's differentiator is legible on first use, must not be "optimized away" as dead code.

5. **Obstacle detail** — bottom sheet: type, severity, age, note, route relationship, inline report action. Staleness must be legible here — trust comes from showing evidence age, not hiding it.

6. **Report Obstacle** — bottom sheet: fixed six type chips, severity as segmented control/button group (never a slider — hard to operate with Switch Control), optional note, "looks clear now" toggle when applicable. Save triggers **local rescoring of the currently-held route polyline** — never a second MKDirections call. This keeps offline reporting working and is what makes the "watch the recommendation update live" moment real; dismissing the sheet should produce a visible (but Reduce-Motion-respecting) update on the recommendation card.

7. **Saved (tab 2)** — full list-management screen. Rows refetch **lazily and staggered** (visible-first, per-row "checking today's route…" loading copy) — never all-at-once on tab appear, which would otherwise fire a pile of concurrent MapKit calls for a paid user with several saved routes. Persistent "2 of 2 free" indicator at cap; share action per row.

8. **Paywall** — sheet on 3rd favorite save attempt, shown over the actual route just being saved (translucent unlock panel, blocked action still visible behind it). Sells unlimited favorites + share export only — never implies routing/scoring/reporting are premium.

9. **Share preview** — shown before the system `ShareLink`: destination, recommendation sentence, confidence, timestamp, unverified-data line. Reads as a route snapshot, not a social card.

10. **Settings/info** (toolbar affordance) — full disclaimer, sample-data explanation, restore purchases, note-privacy guidance, demo-data reset.

**Design tokens:**
- Colors: ink `#12161C` (primary text/chrome), warm off-white `#F7F5F1` (background), petrol `#1F5C56` (single primary/interactive accent), amber `#B8860B` (moderate severity only), coral/rust `#B4432E` (severe on-route hazard only), slate-gray `#6B7280` (secondary/meta text). - Plan: fresh-install-with-demo-card (flag-driven, one-time only), populated, permission-denied (inline banner), search-empty/ambiguous.

**Accessibility requirements:** Dynamic Type at largest accessibility sizes with no truncation on recommendation/why/confidence/paywall copy (enabled by the single-scroll-container fix above). ## Ios Architecture Review

Round two closed the one real open question cleanly. Codex dropped the hedge and committed: **non-consumable lifetime "Clearpath Pro" unlock**, not a subscription — because there's no recurring cost or live service behind unlimited favorites + share export, and Apple's own subscription guidance expects subscriptions to gate something that decays without ongoing investment. Claude independently landed on the exact same call for the exact same reasons and piled on a practical argument too: non-consumable collapses the entitlement model to one boolean check with no expiration/renewal/grace-period states to build or test. Nobody's holding out for subscription anymore.

Everything else from round one got explicitly confirmed rather than just left standing: CloudKit-safe SwiftData schema shape now (defaults, optional relationships, no uniqueness constraints, explicit stable IDs) without turning on the CloudKit entitlement itself; the three-way split of no-connectivity vs. `MKError.directionsNotFound` vs. geocoding failure as genuinely separate code paths; `RouteFetching`/`LocationProviding` protocol seams so tests never touch real MapKit/CoreLocation; two independent `ModelContainer` instances against the same store URL as the actual proxy for "survives force-quit"; `performAccessibilityAudit()` wired into CI as a mechanical gate alongside (not instead of) one manual VoiceOver pass; and zero third-party dependencies, full stop, no exception hedge.

Claude added two small, uncontested extensions in this round that nobody's pushing back on: cached route geometry needs its own staleness rule separate from obstacle recency (so "today's suggested route" doesn't get shown off a four-day-old offline snapshot — it should say "your last available route, from [day]" past a threshold), and recents should be capped at five, storing only label+coordinate with no per-search timestamp trail, plus a "clear recents" control next to demo-data reset — closing the same accidental-movement-history risk Codex flagged for obstacle notes.

## Final Output

**SwiftUI architecture:** Small, disciplined app — per-tab `NavigationStack`s (Plan, Saved), feature-scoped `@MainActor`-isolated `@Observable` view models, thin service layer underneath: `LocationService` (permission + current location), a search/routing service wrapping MapKit, an obstacle/persistence service, a `RouteScorer`/`ObstacleMatcher` pair operating purely on plain Swift structs (never SwiftData `@Model` types or MapKit-native types directly), and a purchases service. Route geometry is fetched from MapKit exactly once per request/favorite-open, immediately converted to app-owned plain structs, and never re-fetched just to reflect a new report — reporting triggers local rescoring of the already-held polyline only. `RouteFetching` and `LocationProviding` are defined as protocols so unit/view-model tests inject fakes and never touch real MapKit/CoreLocation.

**Apple framework choices:** SwiftUI + Observation for UI/state, MapKit for search, directions, and route rendering (native `Map`/`MapContentBuilder`, no `UIViewRepresentable`-wrapped `MKMapView`), CoreLocation for permission and current-location origin (When-In-Use only — no Always, no background location), SwiftData for local persistence, StoreKit 2 for the paid unlock, `NWPathMonitor` for coarse reachability context (to distinguish "no network" from a MapKit-specific failure, not as the sole truth source). No ML, AR, or LLM dependency in v1 — none clear the "materially improves the trust loop" bar.

**Persistence and local data plan:** SwiftData schema for obstacle reports, favorites, and recent destinations, written CloudKit-safe from day one (every attribute has a default, relationships are optional, no unique-attribute constraints, stable IDs as explicit fields) even though the CloudKit capability/entitlement stays off in this MVP — so "sync-ready" is a real property of the schema, not just a claim. Obstacle reports carry `seeded` vs. `userEntered` provenance and use supersession (soft state, newer-weighted) rather than hard delete. Favorites store origin/destination endpoints only, never a frozen polyline, and re-fetch live on open. A cached route snapshot (for offline review) is explicitly ephemeral and carries its own fetched-at timestamp with a staleness threshold (~1 day) — past that, offline-review copy shifts from "today's suggested route" to an honest "your last available route, from [day]" rather than implying currency it doesn't have. Recents are capped at five, store label+coordinate only with no per-search timestamp trail, and get an explicit "clear recents" control next to the demo-data reset in Settings.

**Permissions/privacy plan:** When-In-Use location only, manual/pin-drop origin entry as a first-class equal mode (never a fallback), permission-revoked-mid-session handled as an observed authorization-state change (not polled). No camera, no contacts, no notifications, no motion. Free-text obstacle notes and saved destinations use complete-until-first-unlock file protection, with inline copy discouraging personal/identifying info since sharing exists downstream. No hidden search/location history beyond the capped, visible recents list.

**Dependency/license decision:** Zero third-party packages, full stop — no conditional exception. Apple's own frameworks cover every requirement, including the ShareLink export.

**Testing implications:** Pure unit tests over plain structs for the scorer/matcher — boundary-value tests at 19m/21m, 9m/11m, day 6/8, day 30/31, day 90/91, the single-route wording branch, the shorter-route tie-break, and the supersession-changes-score-immediately case — none requiring SwiftData, CoreLocation, or a live network path. Integration tests cover persistence seeding, the third-favorite paywall intercept, permission-denied → manual-origin flow, and offline cached-route review (including the new staleness-copy rule). The "survives force-quit" requirement is tested via two independent `ModelContainer` instances against the same on-disk store URL, not an in-memory relaunch fake. `XCTestCase.performAccessibilityAudit()` runs as a mechanical CI gate (contrast, hit-target size, missing labels) alongside one deliberate manual VoiceOver pass per milestone for the semantic reading-order requirement, which the automated audit can't catch. StoreKit testing uses a local `.storekit` configuration file plus a real sandbox-account restore-purchases test after delete/reinstall — no server-side receipt validation in v1, consistent with the non-consumable, on-device-`VerificationResult`-only entitlement model now locked for the single "Clearpath Pro" unlock.

## Final Output

**Architecture overview:** A small, disciplined SwiftUI app built around one hard rule that both agents converged on independently: protocols exist only where a build lane needs to fake a platform resource in tests (network, CoreLocation, StoreKit, disk) — `RouteFetching`, `PlaceSearching`, `LocationProviding`, `ObstacleStore`, `FavoritesStore`, `RecentsStore`, `CachedRouteStore`, `PurchaseManaging`, `ReachabilityProviding`. Everything else that's pure, deterministic computation is a plain static function with no fakeable seam — `RouteScorer`, `ObstacleMatcher`, `RouteExplanationBuilder`. Route geometry is fetched from MapKit exactly once per request or favorite-open, converted immediately into app-owned plain structs (`RouteSnapshot`/`RouteCandidate`), and never re-fetched to reflect a new report — `RouteResultViewModel.rescore()` is the only mutation path after a report, and it never calls `RouteFetching` again. A single `AppDependencies` composition-root bundle is built once in `App/` and injected everywhere, so no feature wires up its own ad hoc service instances. `RouteResultInput` is the one shared navigation payload Plan, Saved, and the onboarding demo card all construct identically, closing off the risk of three slightly different push contracts into Route Result.

**File/module layout:** `App/` (entry point, composition root wiring `AppDependencies` and the `ModelContainer`), `Domain/` (plain structs plus `RouteScorer`/`ObstacleMatcher`/`RouteExplanationBuilder` — zero imports beyond Foundation/CoreLocation-for-math, zero SwiftData or MapKit types), `Persistence/` (SwiftData `@Model` records, `ModelContainerFactory`, `DemoDataSeeder`, mapping extensions to/from domain structs), `Services/` (concrete `LocationService`, `RouteSearchService`, `ObstacleRepository`, `FavoritesRepository`, `RecentsRepository`, `ReachabilityMonitor`, `PurchaseService`, each conforming to its protocol), `Features/{Onboarding,Plan,Search,RouteResult,ObstacleDetail,ReportObstacle,Saved,Paywall,SharePreview,Settings}/` each with a View + `@Observable` view model, `DesignSystem/` (tokens plus the locked component set), and `Tests/` split into a pure-domain unit target (no SwiftData, MapKit, or CoreLocation — this is what keeps the boundary-value tests fast) and a separate integration/UI target.

**Data models:** Domain layer (framework-free, `Sendable`, used everywhere scoring/matching/UI logic runs): `Coordinate`, `RouteEndpoint`, `ObstacleType`/`ObstacleSeverity`/`ReportProvenance`, `ObstacleReport`, `RoutePolyline`, `RouteCandidate`, `RouteSnapshot`/`RouteSnapshotSource`, `ObstacleRelationship`, `ObstacleMatch`, `ConfidenceLevel`, `FrictionCategory`, `RouteExplanation`/`WarningChipModel`/`WarningChipKind`, `ScoredRoute`, `RouteRecommendation`, `ScoringConstants`, `ReportDraft`, `Favorite`, `RecentDestination`, `ShareSnapshot`. Persistence layer (SwiftData, CloudKit-safe from day one — defaults on every attribute, optional relationships, no unique constraints, explicit stable UUID fields): `ObstacleReportRecord`, `FavoriteRecord`, `RecentDestinationRecord`, `CachedRouteSnapshotRecord`. A thin mapping layer at the `Persistence/` boundary is the only place these two worlds touch.

**Persistence strategy:** Obstacle reports carry `seeded`/`userEntered` provenance and use supersession (`isActiveForScoring` flag, newer-weighted, old row retained for history) rather than hard delete, with two distinct supersession paths — explicit by-ID (the "looks clear now" action on a known obstacle) and proximity-based via the inner 10m buffer (an unprompted new report landing on the same physical hazard as an existing one) — both centralized behind `ReportSubmission.submit`, so no UI lane has to decide which write path applies. Favorites store origin/destination endpoints only, never a frozen polyline, and always re-fetch live on open. Cached route snapshots are explicitly ephemeral, carry their own `fetchedAt` and `source`, and honor a staleness threshold (~1 day) past which offline-review copy shifts from "today's suggested route" to "your last available route, from [day]"; `CachedRouteStore.deleteExpiredRoutes(olderThan:)` gives that lifecycle a named owner rather than leaving expiry implicit. Seed data installation is versioned (`ScoringConstants.currentSeedDataVersion` checked against a persisted installed-version flag) so a later update can revise the demo corridor without duplicating or silently ignoring the change, and `resetSeedData()` backs the locked Settings "demo-data reset" control. Recents are capped at five, storing label+coordinate only, no per-search timestamp trail.

**Testing strategy:** Pure unit tests over plain structs for `RouteScorer`/`ObstacleMatcher`/`RouteExplanationBuilder` — none require SwiftData, CoreLocation, or a live network path — asserting the exact boundary values baked into `ScoringConstants`: 19m/21m outer buffer, 9m/11m inner buffer, day 6/8, day 30/31, day 90/91 recency bands, the single-route wording branch, the shorter-route tie-break, and a before/after supersession test proving a new report changes score immediately. `RouteFetching`/`LocationProviding`/`PlaceSearching`/`PurchaseManaging`/`ReachabilityProviding` are faked in view-model tests via their protocols. The "survives force-quit" requirement is tested via two independent `ModelContainer` instances against the same on-disk store URL. `performAccessibilityAudit()` runs as a mechanical CI gate alongside one deliberate manual VoiceOver pass per milestone for the semantic reading-order requirement it can't catch.

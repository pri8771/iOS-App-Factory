### Round 2


**Codex (Frontend Engineer · Systems Thinker) — Round 2**

Claude’s read on the ownership conflicts is stronger, and I’d rather resolve them cleanly now than pretend “we’ll coordinate later” means anything. `DesignSystem/` has to sit with `primary_ui`, not `polish_resilience`. That’s not because the design system is “more UI than polish” in some abstract sense; it’s because `Onboarding`, `Plan`, `Search`, and especially `RouteResult` cannot even stabilize their state/layout/accessibility contracts without the exact shared components and token semantics they render through. If the lane building the core trip flow has to wait on another lane to dribble over `RecommendationCard`, `ConfidenceBadge`, `WarningChip`, and the token file, you create fake parallelism and real churn. Same answer for `Features/Saved/`, but for the opposite reason: I think Claude is right that Saved belongs with `polish_resilience`, because Saved is not just “another screen.” It is the place where live-refresh policy, favorite-cap messaging, purchase gating, share export, and stale/offline framing all meet. That surface is mostly about second-order correctness: lazy refresh, capped concurrency, “today’s suggested route” wording, paywall intercept timing, restore state consistency. If `primary_ui` owns core trip flow and also owns Saved, then the same lane becomes the bottleneck for both milestone 3 and milestone 5, while the lane that already owns share/paywall/explanation polish is underloaded until late. That’s a bad split.

I also think we should stop carrying two backlog shapes. Claude’s exploded task list is the one workers can actually pick up without inventing work boundaries on the fly, so I’d take that list as the canonical backlog and just make the lane contracts explicit around it. The app should live at `/Users/pchordia/Documents/iOS-App-Factory/clearpath-access`, with one branch per lane off a shared integration branch: `clearpath-access/data-domain`, `clearpath-access/services-utilities`, `clearpath-access/primary-ui`, `clearpath-access/polish-resilience`, all merging back into `clearpath-access/integration`, then fast-forward or squash into the app’s main branch when the milestone gate passes. File ownership needs to be hard, not advisory: `data_domain` is the sole writer for `Domain/Models/` and `Domain/Scoring/`; `services_utilities` is the sole writer for `Persistence/`, `Services/`, and integration tests under `Tests/IntegrationTests/`; `primary_ui` is the sole writer for `App/` except the late integration wiring pass in `App/AppDependencies.swift`, plus `DesignSystem/`, `Features/Onboarding`, `Features/Plan`, `Features/Search`, `Features/RouteResult`, `Features/ReportObstacle`, `Features/ObstacleDetail`, and UI tests; `polish_resilience` is the sole writer for `Domain/Explanation/`, `Domain/Share/`, `App/Flags/`, `Features/Saved`, `Features/Paywall`, `Features/SharePreview`, `Features/Settings`, and accessibility/regression tests. Forbidden edit zones should be literal: no lane edits another lane’s folders; the only shared-edit zone is `App/AppDependencies.swift`, and even there only `primary_ui` edits the file directly, while the service lanes expose constructor seams and the integrator wires them. Conflict prevention should be branch + boundary driven: every task owns concrete files, every merge into `integration` requires its lane gate green, and anything that changes a cross-lane interface has to land in `T-001`/`T-014` style contract files first, before downstream implementation starts. Handoffs are straightforward: `data_domain` publishes structs/scoring API first; `services_utilities` builds against those exact contracts and never invents shadow models; `primary_ui` starts against fakes as soon as `T-001`, `T-012`, `T-013`, and `T-014` exist; `polish_resilience` can start explanation/share/flags early, then pick up Saved/paywall/share/settings after Route Result shape is real. That’s how we avoid stepping on each other: not “communicate more,” but “own narrower surfaces and merge only at explicit seams.”

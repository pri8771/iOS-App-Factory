## Final Output

**Architecture overview:** Builds on the design-handoff/architecture-review foundation (`@Observable` per-feature view models, SwiftData, `SummarizerService` protocol). New and load-bearing for this phase: long-running processing is split across two stores. A `@ModelActor` `ProcessingCoordinator` (off-main) does the real work — PDFKit/Vision extraction, `SummarizerService` calls, `DiffEngine` — and writes live stage ticks (queued/extracting/summarizing/diffing) into an in-memory `@MainActor @Observable ProcessingStatusBoard` keyed by document ID; SwiftData is only written at terminal states (ready/partial/failed) so the persistent store and `@Query`-driven views aren't hammered by transient progress updates. Because this creates two sources of truth for "state of a document," `ProcessingCoordinator` must run a reconciliation pass at app launch that requeues any persisted non-terminal stage rather than leaving a card stuck mid-spinner.

**Data model shape:** `ProcessingStage` is persisted as a flat `String, Codable` enum (`queued/extracting/summarizing/diffing/ready/partial/failed`) plus sibling fields (`completenessNote`, `missingSections`, `failureReason`) — not as a single Codable enum-with-associated-values directly on the model, which is friction-prone in SwiftData. The rich `ProcessingState` the UI switches on is a computed property reconstructed from those flat fields. `DigestItem.id` is deterministically derived from `(sourceID, meetingID, agendaItemNumber-or-sectionIndex)`, never array position, so `ForEach` identity survives reprocessing/reordering without swapping badges onto the wrong row.

**File/module layout:** `DocketApp` (entry + versioned `ModelContainer` schema); `Features/{Onboarding,Following,AddSource,SourceDetail,DigestDetail,Settings,Paywall}` (View + `@Observable @MainActor` view model each); `Core/Models` (four SwiftData `@Model` types); `Core/Processing` (`ProcessingCoordinator`, `ProcessingStatusBoard`, `ProcessingStage`); `Core/Summarization` (`SummarizerService`, `ExtractiveSummarizer`, `FoundationModelsSummarizer`); `Core/Extraction` (`PDFExtractor`, `OCRExtractor`); `Core/Diffing` (`DiffEngine`, pure, zero SwiftData imports, package-extractable); `Core/Persistence` (`PDFFileStore`); `Core/Notifications`; `Core/Entitlements` (`EntitlementStore`, StoreKit 2); `DesignSystem` (the nine design-handoff components + `Tokens`); `Shared`.

**Persistence strategy:** Carries forward "PDF binary on disk, never in SwiftData" (`PDFFileStore`, filename reference only). `DiffEngine` is a pure function decoupled from SwiftData and the summarizer, unit-testable against fixtures.

**Paywall decision (closes the open item from architecture review):** Free tier — up to 2 followed sources, full-quality digests always (summarization quality itself is never gated), visibility limited to current + 1 prior meeting per source, no push notifications (manual refresh still works). Docket Plus (single subscription, no sub-tiers) — unlimited sources, full historical archive, notifications. Presented as one paywall sheet with one message regardless of which of the three entitlement checks triggered it; `EntitlementStore` exposes three independent checks (`canFollowAdditionalSource`, `canViewHistoryBeyondMostRecent`, `canEnableNotifications`) backed by StoreKit 2 `Transaction.currentEntitlements`, checked fully offline. Paywall triggers contextually (3rd-source follow, archived-history tap, notification toggle), never at launch.

**Testing strategy:** `DiffEngine` and `ExtractiveSummarizer` — deterministic, fixture-based CI unit tests. `ProcessingCoordinator` — tested against an in-memory `ModelContainer` with a mock `SummarizerService`, specifically exercising the launch-reconciliation behavior. `EntitlementStore` — tested against a local `.storekit` config file, all three gates independently. `DigestItemCard` — `ImageRenderer` snapshot test at accessibility XXXL (the design-flagged risk). XCUITest smoke flows for onboarding→follow→"all caught up" and add-source→PDF import failure. BGTask stays untested in CI (physical-device only, per architecture review); manual refresh carries the real automated-test burden.

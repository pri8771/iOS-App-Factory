Picking up where design handoff left off — the `ProcessingState`/`.partial` model and SwiftData/`@Observable` architecture are the right foundation, and I want to build the architecture review directly on top of that rather than re-deciding it. But there's one thing sitting underneath all of it that nobody's stress-tested yet: the entire promise of this app is "on-device digest," and that phrase is doing a lot of unexamined work.

The realistic way to get "plain-language summarization" on-device on iPhone is Apple's Foundation Models framework, and it is gated hard — it needs Apple Intelligence enabled, which needs a specific chip tier (A17 Pro/M-series and up), a supported region and language, and the user hasn't turned it off in Settings. That is not a small slice of the install base to lose, it's potentially the majority of a typical resident's phone in year one of availability, and this app's core value prop literally does not exist without summarization. If we ship this as "call Foundation Models, and if it's unavailable show `.failed`," the app is dead on arrival for a large chunk of users on launch day. I want a hard requirement here: a rule-based/extractive fallback summarizer (first-sentence-of-section extraction, heading/agenda-item-title surfacing, no LLM at all) that runs on every device regardless of Apple Intelligence eligibility, with the Foundation Models path as an enhancement layered on top when available. This also has a testing upside — the fallback path is deterministic and can be fully unit tested and CI'd, where the Foundation Models path fundamentally can't be (there's no simulator support for the real model, so any architecture that hard-wires calls to it instead of going through a `SummarizerService` protocol is untestable by construction). Abstract it behind a protocol from day one so most of the app's logic — diffing, state transitions, persistence, UI — never actually depends on whether the model ran.

On frameworks: PDFKit for rendering and text-layer extraction, Vision (`VNRecognizeTextRequest`) for the OCR path on scanned agendas, SwiftData for `FollowedSource`/`Meeting`/`DigestItem`/`DiffRecord`, BGTaskScheduler for periodic "check for new agenda" background work, UserNotifications for per-source alerts, StoreKit 2 for the paid tier. One persistence detail I'd flag before it becomes a real problem: don't store the source PDF binary inside the SwiftData model. Government agenda PDFs can run tens of megabytes; storing that as an attribute blows up the store file and makes every fetch of unrelated fields slower. Store the PDF in Application Support/Documents keyed by a stable file name and keep only the file reference and extracted text in SwiftData. The diffing itself also deserves its own line item — there is no Apple framework that diffs two versions of a document's text, that's a from-scratch Myers-diff-style algorithm on our own extracted text, and it's real, testable-in-isolation engineering effort, not glue code. It should be a pure function taking two strings and returning change spans, unit tested against fixture pairs, completely decoupled from the summarizer and from SwiftData.

On background work, I'd push back gently on treating BGTaskScheduler as anything more than a nice-to-have. iOS gives zero execution guarantees on timing, and in practice "checks a few times a day, if the system feels like it" is the honest description. The design's `LastSyncedLabel` staleness indicator is the right way to be honest about that in the UI, but it means users need a reliable manual "check now" path (pull-to-refresh, already planned) because background fetch alone will produce inconsistent, hard-to-explain gaps that read as bugs. Also worth noting for QA: BGProcessingTask is close to impossible to reliably exercise in the simulator or in automated UI tests — that path needs to be tested on physical devices with the debug BGTaskScheduler launch trigger, and the manual refresh path needs to carry the real test burden in CI.

Permissions: I'd keep this list short on purpose. No CoreLocation — the "zip/location-assisted" onboarding search should be a plain text zip/city field, not a location permission prompt, because a location prompt on first launch before the user has seen any value is a proven churn point and there's no feature here that actually requires precise location (jurisdiction lookup by typed zip is sufficient and testable without a permission dialog at all). Document picker for manual PDF import doesn't need an entitlement beyond the system picker. Camera access only if we ever add "photograph a printed agenda," which I'd cut from MVP entirely — it's a v2 nice-to-have, not core. Notifications should be requested contextually, after the user follows their first source and sets a cadence, never as a blanket launch-time prompt. On privacy: no account, no login, all processing local — the one place this app touches the network is fetching public government PDFs/agenda pages, which means App Transport Security exceptions may be needed for specific municipal domains with weak TLS configs, and that should be scoped per-domain in the Info.plist exception list, never a blanket `NSAllowsArbitraryLoads`.

For StoreKit: nobody's actually defined the paywall yet since the product brief phase never got a real answer (spend-limit skipped), so I'll flag that as a real gap the tech spec phase needs closed, not invent one myself. What I will commit to architecturally is StoreKit 2 with `Transaction.currentEntitlements` for local entitlement checking — no server round-trip needed for MVP, which fits local-first and means paywall status is available offline, consistent with everything else in this app. Whatever the gate ends up being (source count cap, notification access, digest history depth), it should be checked at the model layer, not scattered across views, so it's testable independent of the StoreKit UI.

Dependency policy is easy to satisfy here: everything above is first-party, so the permissive-license question resolves itself by having zero third-party dependencies rather than by vetting licenses. The one place I'd flag as a legitimate "would a dependency reduce risk" question is snapshot/regression testing for the accessibility-XXXL `DigestItemCard` layout that the design phase already flagged as fragile — `swift-snapshot-testing` is MIT and would save real effort there, but it's not required; `ImageRenderer`-based home-grown snapshot comparison in XCTest gets us most of the value with zero external dependency, and I'd default to that unless it becomes a maintenance drag.

Last thing, because it's a QA concern that's easy to skip until App Review flags it: this app is telling residents what changed in decisions that affect their property, taxes, and civic life, based on an on-device model's summary of a messy-OCR'd PDF. A wrong or dropped summary is not a cosmetic bug, it's the failure mode that destroys the entire trust premise. Every digest screen needs a persistent, non-dismissible-on-first-view disclaimer that this is a summary and the original document is one tap away (the design already has that affordance, good), and the `.partial`/completeness indicator needs to be genuinely hard to miss, not a small badge — a resident should never be able to act on a partial digest believing it's complete.

**Persistence and local data plan:** SwiftData holds structured entities and extracted text only — the source PDF binary lives in Application Support/Documents keyed by a stable filename, referenced (not embedded) from the model, to avoid store bloat and slow fetches. Diffing is a pure, from-scratch Myers-diff-style function over two text versions, decoupled from SwiftData and the summarizer, unit-tested against fixture pairs.

**Permissions/privacy plan:** No CoreLocation — jurisdiction lookup is a plain typed zip/city field. No camera in MVP. One optional MIT-licensed aid (`swift-snapshot-testing`) flagged for the accessibility-XXXL card layout testing, not required, home-grown `ImageRenderer` comparison is an acceptable zero-dependency substitute.

**Codex (QA & Risk · Closer) — Round 1**

The risk lens here says the architecture has to be boring in exactly the right places. The app’s failure modes are not “the model was a bit off,” they’re state integrity failures: quota increments when nothing useful happened, a share-extension handoff silently loses text, a generated draft mutates after the user sends it, or the AI path and manual path drift into incompatible downstream objects. If those happen, the product stops being defensible at the moment it matters. So I’d lock a simple layered design now: SwiftUI screens on the main actor, a thin application layer of coordinators/use-cases, repository protocols over SwiftData, and one normalized analysis contract that every ingress path produces before Review.

I would keep four lane boundaries explicit so parallel work can proceed without type drift. `data_domain` owns persisted entities and pure domain enums. `services_utilities` owns repositories, extraction, entitlement, export, share-ingest, and clock/quota logic. `primary_ui` owns screen models and navigation state, but never invents its own business types. `polish_resilience` owns debug toggles, forced-failure harnesses, accessibility helpers, and test fixtures. If one lane starts making ad hoc versions of verdicts, baseline strength, or availability reasons, the build will fragment immediately. The one contract that must not move is: both AI analysis and manual tagging produce the same `ReviewItemDraft` array plus the same `AnalysisOutcome`, and only that normalized output is allowed into Review and Draft generation.

File layout should reflect those boundaries instead of feature soup. I’d use something like `App/` for entry, dependency wiring, navigation bootstrap, and long-lived listeners; `Domain/` for types like `Project`, `Baseline`, `ReviewVerdict`, `AnalysisOutcome`, `EntitlementState`; `Persistence/` for SwiftData schema models, migrations, repository implementations, App Group intake payload storage; `Services/Extraction`, `Services/Entitlements`, `Services/Export`, `Services/ShareIngress`; `Features/ProjectList`, `ProjectHome`, `BaselineEditor`, `Intake`, `Review`, `Draft`, `Settings`; `Debug/` for forced states and resettable counters; `Tests/Unit`, `Tests/Integration`, `Tests/UI`. Persistence strategy should be fail-safe, not just local-first. SwiftData is fine, but I’d lock these rules now:

The biggest technical trap is concurrency, not storage. I would put all UI-facing feature models on `@MainActor`, but keep repository and extraction services async and isolated. **Claude (Frontend Engineer · Visionary) — Round 1**

Taking the frontend/visionary angle here: the architecture questions from ios_architecture_review are already answered at the "what frameworks, what protocol boundary" level, so my job in this phase is to nail down the actual file layout, concrete type signatures, and the one place I think this build can be genuinely better than a generic "AI extraction app" rather than merely adequate — which is the manual fallback.

Here's the idea I want to push hard: don't build the manual path as "user reads raw pasted text and manually highlights/splits it into asks from scratch." That's the version that always ends up feeling like a punishment screen no matter how nice the visual chrome is, because it asks the user to do cognitive segmentation work a computer is good at. Instead, build a shared, deterministic, first-party `AskSegmenter` using the `NaturalLanguage` framework (`NLTokenizer` at the sentence unit, plus a cheap heuristic pass for imperative/request phrasing — modal verbs, question marks, "could you," "can we," "while you're at it") that runs identically for both paths. The AI path uses segmented candidates as the units it classifies; Foundation Models never sees a raw blob of text, it sees pre-chunked candidate asks and returns verdict/rationale/clause per candidate. The manual path runs the exact same segmenter and lands the user directly on pre-chunked candidates to *tag*, not an empty text box to *split*. This is instant (no model, no loading state — matches what's already locked), works on every device regardless of Apple Intelligence eligibility, and turns "genuinely usable fallback" into something that's actually faster to use than the AI path in some cases. It also means the extractor contract converges naturally: `ScopeExtractor.extract` always returns candidate items; the AI implementation fills in real verdicts, rationale, and confidence; the manual implementation returns the same candidates with verdict defaulted to flagged-for-your-call (never auto-approved, consistent with the bias-to-scrutiny rule) and a placeholder rationale of "not yet reviewed," which the Manual Tagging screen resolves by user action. One segmenter, one converging shape, two very different but equally fast experiences.

On concurrency: SwiftData's `ModelContext` isn't `Sendable`, and Foundation Models inference is genuinely async work that may legitimately run off the main actor. So the extractor protocol must never pass or return SwiftData `@Model` objects directly — it deals exclusively in plain `Sendable` DTOs (`ReviewItemDraft`, `ExtractionOutcome`, `BaselineSnapshot`), and the view model is the only place that takes those DTOs and materializes real `ReviewItem` `@Model` instances into the main-actor-bound `ModelContext`. This is the detail most likely to get skipped under time pressure because it's invisible until someone tries to hand a `ReviewItem` across an `await` boundary and the compiler (rightly, under Swift 6 strict concurrency) refuses. Every view model in this app is `@MainActor @Observable`; the only intentionally off-main work is inference inside `AIScopeExtractor`, which is itself `Sendable` and holds no SwiftData state.

File layout, organized as one Xcode project with two targets plus test targets — `ScopeGuard/App` (entry point, `ModelContainer` construction, `SchemaV1` registration, root routing logic computed synchronously before first render), `ScopeGuard/Models` (all `@Model` types plus the `VersionedSchema`), `ScopeGuard/DTOs` (the `Sendable` value types that cross actor/protocol boundaries — this separation from `Models` is deliberate, it's the thing that keeps the extractor protocol clean), `ScopeGuard/Services` (`ScopeExtractor` + both conformances, `AskSegmenter`, `EntitlementService`, `QuotaService`, `ExportService`, `ShareIntakeIngestService`), `ScopeGuard/Repositories` (`ProjectRepository`, `AnalysisRepository`, both protocol + SwiftData-backed implementation), `ScopeGuard/Features/{ProjectList,ProjectHome,BaselineEditor,Intake,ManualTagging,Review,DraftPreview,Settings}` (view + view model pairs, one folder per screen), `ScopeGuard/DesignSystem` (tokens, `ReviewRowView`, `VerdictChipView`, `BottomActionBarView`, `BaselineStrengthBadgeView`, shared empty/error state module), `ScopeGuardShareExtension` (its own tiny target — capture view, project picker, App Group JSON writer, nothing else), `ScopeGuardTests`, `ScopeGuardUITests`.

Data model, concretely: `Project` owns `Baseline` (cascade), an array of `AnalysisSession` (cascade), optional `clientName`. I want to resolve an ambiguity explicitly here rather than let it drift: "1 active project" for the free tier means total `Project` count, full stop — there is no archive/deactivate concept in v1, so the cap is simply `projects.count <= 1` for free-tier accounts. If we wanted archiving later that's a real feature, not something to fake now. `Baseline` carries Layer 1 (`agreementSummary: String`, required) and Layer 2 fields (all optional strings) plus `layer2DisclosureExpanded: Bool` persisted per-project as locked in design_handoff, and `lastEditedAt` for the "draft looks stale" comparison. `AnalysisSession` is the immutable record of one paste: raw text, origin (`.pastedCold`/`.shareExtension`), extractor origin, the Foundation Models availability at run time, a `quotaCharged: Bool` set exactly once, and its outcome. `ReviewItem` carries the full shared shape locked back in ios_architecture_review — source quote, normalized ask, verdict, rationale, optional matched clause (explicit nil renders as "flagged on general judgment," never fabricated), confidence, billability recommendation, `isResolved`, origin. `DraftDocument` is 1:1 with an `AnalysisSession`, every field set at `init` and never mutated again — I'd enforce this by making it a `struct`-like `@Model` with `let`-only stored properties where SwiftData allows, or at minimum by never exposing a setter path anywhere in the codebase, verified by the cascade/immutability tests already locked. `IntakeDraft` and `ManualTaggingChunk` are the autosaved mutable working state — separate types, not reused shapes, because their lifecycle guarantee (continuously overwritten, discardable) is the opposite of everything above.

Testing strategy carries forward what's locked (extractor conformance table, quota single-increment test, cascade-delete-zero-orphans test, draft-immutability-by-construction test, golden-set human-reviewed extraction regression) and adds one thing specific to the segmenter: since `AskSegmenter` is fully deterministic (no model involved), it gets real exact-match unit tests against a fixed set of messy real sentences — this is the one piece of "extraction-adjacent" logic that unlike the LLM path genuinely can be asserted precisely, and it's worth being explicit that it's tested differently than the AI classification step.

My honest objection to my own segmenter idea: it adds a real piece of shared logic that both paths now depend on, so a bad segmentation (splitting a sentence wrong, or merging two asks into one candidate) becomes a shared failure mode instead of an isolated one. I think that's still the right trade — a shared, testable, deterministic component is easier to get right and fix than two divergent hand-built splitting behaviors — but it means `AskSegmenter` quality deserves its own dedicated test coverage, not an assumption that "NLTokenizer will just work." The detail I most want preserved: the extractor protocol boundary must speak in `Sendable` DTOs only, never SwiftData `@Model` types, and the manual path is not "type from scratch," it's "confirm/adjust pre-segmented candidates" — if either of those get flattened during build for convenience, we're back to two parallel products and a concurrency-unsafe protocol.

Looking at round 1, Codex and Claude both did real architecture work, but this is the first round of this phase and there are enough live differences — not just naming, but actual structural choices — that I don't think this is ready to close yet.

Where they agree: the load-bearing seam is a `ScopeExtractor` protocol returning `Sendable` DTOs, not SwiftData models; AI and manual paths converge on the same `ReviewItem`/`ReviewItemDraft` shape; SwiftData `@Model` types stay separated from plain-value DTOs that cross actor boundaries; drafts are write-once/immutable, intake/manual-tagging drafts are mutable/autosaved; quota increments in exactly one place, only on completed AI runs; a debug-only forced-failure harness exists for entitlement/availability testing; the share extension hands off via a single App-Group JSON file, consumed once. All of that lines up cleanly with what ios_architecture_review already locked.

Where they haven't actually reconciled:

1. **Orchestration ownership.** Codex has a distinct `AnalysisOrchestrator` protocol as the single entrypoint that validates input, resolves entitlement/quota, calls the extractor, increments quota, and persists the session — kept deliberately separate from view models and repositories. Claude's version has `IntakeViewModel.analyze()` and `AnalysisRepository` doing that work directly, with no distinct orchestrator type. This isn't cosmetic — it's a real question of where a fairly sensitive piece of business logic (quota-increment-exactly-once, fail-closed entitlement checks) lives, and the two proposals put it in different places.
2. **AskSegmenter — a genuinely new idea, not yet reacted to.** Claude proposed a shared, deterministic `NaturalLanguage`-backed segmenter that both the AI and manual paths consume, so manual tagging becomes "confirm pre-chunked candidates" instead of "split raw text from scratch." That's a stronger, more concrete version of the "small deterministic pre-pass" Codex mentioned in the architecture-review phase, but Codex hasn't seen or responded to this specific shape yet — need his read on whether a shared segmenter becomes a single point of failure worth the trade, and whether he'd adopt `CandidateAsk` as Claude defined it.
3. **Duplicate/competing names for the same concept**, which is exactly what the acceptance check ("parallel build lanes can work without inventing incompatible types") is there to catch: `UsageMeterStore` (Codex) vs `QuotaService` (Claude); `DebugFailureHarness` (Codex) vs `DebugForcedFailureToggles` (Claude); Codex has an explicit `RootRouteResolver` function + `AppRoute` enum that Claude's interfaces-json doesn't include at all (only mentioned in prose). These need to collapse into one canonical name each before any build lane can start coding against this.

Next round needs Codex and Claude to actually merge into one interfaces-json rather than two parallel ones — pick one name per concept, settle the orchestrator-vs-viewmodel question, and get an explicit yes/no from Codex on the AskSegmenter design.

### Round 2


**Codex (QA & Risk · Closer) — Round 2**

Claude, I think your round-1 push on the shared `AskSegmenter` is the right move, and I think Codex’s instinct for a distinct orchestration layer is also right. The mistake would be treating those as competing ideas. They solve different risks. The segmenter solves “manual fallback turns into retyping raw prose from scratch and quietly becomes a punishment path.” The orchestrator solves “quota/accounting/entitlement/persistence rules get smeared across a view model and a repo and nobody can prove exact-once behavior anymore.” From a QA/risk angle, that second one is the one I refuse to leave soft. The app has several contract-grade invariants: AI quota increments only on completed AI runs, never on manual runs, never on cancelled runs, never twice; fail-closed entitlement checks happen before a paid-only path is entered; a session is either persisted as `.nothingToFlag` / `.itemsFound` or marked `.abandoned`, with no half-written middle state. If `IntakeViewModel.analyze()` directly coordinates extractor calls plus repo writes plus quota side effects, you’ve made the highest-risk business logic UI-owned. That’s where regressions hide. So I’d lock this: view models do intent and state, repositories do storage, services do narrow capabilities, and `AnalysisOrchestrator` owns the one cross-cutting transaction boundary for “run an analysis and commit the result correctly.”

On `AskSegmenter`, I’d adopt Claude’s exact shape with one guardrail Codex will care about: the segmenter is advisory, not authoritative. It is shared because we need the AI path and manual path to start from the same cleaned candidates, and because the deterministic part is exactly what *is* unit-testable. But it cannot become a hidden single point of failure that forces bad chunks downstream. So manual tagging has to support discard, retag, and a lightweight “split further” operation on a candidate if the heuristic over-groups two asks. That keeps the segmenter as a speed-up instead of a trap. On naming, we should stop carrying duplicates now. `QuotaService` is the cleaner name over `UsageMeterStore` because the behavior is quota semantics, not generic metering. `ShareIntakeIngestService` is clearer than `ShareIngressStore` because the important contract is consume-once ingestion, not abstract storage. `DebugForcedFailureToggles` is better than `DebugFailureHarness` because the UI literally presents toggles/settings, and the type should match what testers interact with. I’d also tighten one ambiguity from the merged draft: `ManualTaggingChunk` should stay persistent if we truly want autosave across relaunch during manual work, but it should be owned through a `ManualTaggingDraft` container rather than free-floating rows, otherwise cleanup semantics get sloppy. And I’d keep `RootRouteResolver` explicit; Claude described the requirement, Codex named it, and QA needs the startup routing contract written down so “no project-list flash” is testable instead of aspirational.

**Claude (Frontend Engineer · Visionary) — Round 2**

Codex, your orchestrator point actually lands harder on a second read than it did in my head when I wrote round 1. I said "quota increments in exactly one place" as a rule, but I never gave that rule a home — I just trusted `IntakeViewModel.analyze()` and `AnalysisRepository` to cooperate correctly, which is exactly the kind of thing that drifts the moment two people touch those files in parallel. Your `AnalysisOrchestrator` gives the rule an actual owner. I'm adopting it, but I want to split it into three methods instead of one `runAnalysis`, because AI and manual genuinely have different completion shapes: `runAIAnalysis` is a single atomic call (validate → check quota/availability → classify → charge quota once → persist session), but manual tagging is inherently multi-step and interactive — the user segments, then tags each item over however many taps, potentially across app backgrounding. So I'd split it as `runAIAnalysis(projectID:input:)`, `runManualSegmentation(projectID:input:)` (instant, persists a working `ManualTaggingDraftRecord`, never touches quota), and `finalizeManualAnalysis(projectID:draft:)` (persists the real immutable `AnalysisSession` once the user finishes tagging). That keeps your single-owner guarantee for the dangerous part (quota, entitlement, persistence) while not forcing a multi-minute interactive flow to pretend it's one function call.

I also want to flag something in your `ReviewVerdict` enum that isn't a naming quibble, it's an actual contract bug relative to what's already locked three phases back: you've got `nothingToFlag` as a case of the per-item verdict enum. But `design_discussion` and `detailed_discussion` both locked "nothing to flag" as a *session-level* outcome — the whole message produced zero items — not something an individual `ReviewItem` can be. If `nothingToFlag` is a verdict a single item can carry, then you can end up with a session that "has items" that are all individually nothing-to-flag, which directly contradicts the locked rule that nothing-to-flag is its own full-screen state and never a list variant. I'm keeping `Verdict` as the three real per-item buckets (`inScope`, `flaggedForReview`, `newAndBillable`) and `nothingToFlag` stays where I originally had it, as a case of the session-level `AnalysisOutcomeKind`. This one I'd push back on hard rather than just merge — it's not a style choice, it changes what states are even reachable.

Where you're clearly right and I was underspecified: the `BaselineClause` structuring. I had deliverables/exclusions as flat optional strings, which means "matched baseline clause" in a `ReviewItem` is either a fabricated-looking substring match against a blob, or nothing. Your `[BaselineClause]` with stable IDs per deliverable/exclusion is the right call — it lets both the AI path and manual path reference a specific, addressable clause instead of fuzzy-matching text. I'd add one refinement though: if `ReviewItem` only stores a `matchedClauseID: UUID`, and the user edits or deletes that clause later in the Baseline Editor, historical analyses silently go stale or dangling — and these are supposed to be immutable historical records. So I'd store both the ID and a text snapshot at match time (`matchedBaselineClauseID: UUID?`, `matchedBaselineClauseText: String?`) — the ID is useful for live linking back to the current baseline from Review, the text snapshot is what actually renders in history regardless of what happens to the baseline afterward. Revision policy, assumptions, timeline, and client-dependency notes stay flat free-text — those aren't things individual asks get matched against clause-by-clause, they're just context, so structuring them would be overbuilding.

I'm also taking your `EntitlementState` struct over my scattered `canCreateProject()/canRunAIExtraction()/monthlyAIUsageCount()` methods — a single resolved snapshot the UI observes is just better UI architecture, it means `SettingsViewModel` and `IntakeViewModel` bind to one value instead of independently polling three methods that could theoretically disagree with each other for a frame. And I like `BillabilityRecommendation` as its own axis separate from `Verdict` — it means a flagged item can still carry a soft lean ("likely billable, but no clause matched so it's flagged") instead of forcing billability into a binary the user has to infer from bucket placement alone. That's a real improvement to the one-tap-decision UX, not just a type-system nicety.

On naming collisions: I'll take `ShareIngressStore` over my `ShareIntakeIngestService` — yours actually has a write method (`writePending`), mine only had `consumePendingIntake()` and quietly assumed the extension would write the file some other undefined way, which is a real gap, not a style difference. Same with `PendingSharedIntake` over my `PendingIntake` — more specific, keep it. `DebugFailureHarness` over my `DebugForcedFailureToggles` — arbitrary, yours reads better, done. And I completely missed committing `RootRouteResolver`/`AppRoute` to my actual interfaces-json even though I described the identical synchronous-routing requirement in prose — that's exactly the kind of gap the coordinator was right to catch. Adopting yours, with one addition: it's missing a `manualTagging` case, which per design_handoff is a distinct pushed destination (screen #5 in the locked inventory), not folded into `intake`.

One place I'm not folding to your model: persisting `cancelled`/`failed` as `AnalysisOutcome` cases implies we're keeping a history row for runs that never produced anything. I don't think that's needed — the actual product requirement is "don't lose the user's typed text," which the autosaved `IntakeDraft` already guarantees, not "keep a record that an attempt failed." Persisting failure/cancellation as history adds a new surface (does it show in the past-analyses list? does tapping it do anything?) that nothing upstream asked for. I'd rather keep `AnalysisSession` reserved for actually-completed runs only, and let `IntakeViewModel`'s transient state carry the failure/cancel signal for the current screen without writing a permanent row.

Last thing — since `ReviewItemDraft` in your version already carries `isResolved`, I don't think we need a separate `ManualTaggingChunk` type at all. The manual path can just work directly over `[ReviewItemDraft]` from segmentation through tagging through finalize — one shape, no translation step between "chunk" and "review item." I'd persist the in-progress array as a single autosaved `ManualTaggingDraftRecord` row (Codable-encoded blob of `[ReviewItemDraft]` plus rawText and timestamp) rather than one SwiftData row per chunk, since it's discardable working state that doesn't need relational identity beyond what `ReviewItemDraft.id` already gives `ForEach`.

Still want your explicit yes/no on `AskSegmenter` before I call it locked — if you think a shared deterministic segmenter is a worse bet than two independently-built splitting behaviors, say why and I'll actually reconsider, but absent a counter-argument I think "manual mode confirms pre-chunked candidates instead of typing from scratch" is worth the shared-failure-mode risk.

**Coordinator (Claude) — decision after round 2**

Round 2 did a lot of real work — this is much closer than round 1, but there are still live technical disagreements, not just naming, so I don't think this is ready to close.

**What's actually settled now:** Both sides agree the `AnalysisOrchestrator` (not the view model) owns the dangerous cross-cutting stuff — quota increment, entitlement checks, extractor invocation, persistence commit — as one transaction boundary. Both landed on `Verdict` having exactly three per-item cases (`inScope`, `flaggedForReview`, `newAndBillable`) with "nothing to flag" living only as a session-level outcome, never a fourth verdict — Codex actually already fixed this in his own round-2 JSON, so this was a near-miss rather than a real fight. Both adopted `BaselineClause` as a structured, stably-identified list (not a flat string blob) so a `ReviewItem` can reference a specific deliverable/exclusion — Claude added storing both the clause ID (for live linking) and a text snapshot (so historical analyses don't go stale if the baseline is edited later), and Codex's round 2 didn't contest that, it just hadn't been said yet. Both took `EntitlementState` as one resolved snapshot the UI observes rather than three separate polling methods. Naming collisions from round 1 all resolved by explicit horse-trading: `QuotaService` (Claude's name) backed by Codex's persisted `UsageMeterRecord`, `ShareIngressStore`/`PendingSharedIntake`/`DebugFailureHarness` (Codex's names, Claude conceded these), `RootRouteResolver`/`AppRoute` adopted from Codex with Claude's missing `manualTagging` case added back in.

**What's still genuinely open, not yet reconciled:**

1. **Orchestrator method shape.** Codex proposed one `runAnalysis(...)` entrypoint; Claude split it into three (`runAIAnalysis`, `runManualSegmentation`, `finalizeManualAnalysis`) to handle the fact that AI completion is atomic but manual tagging is a multi-step interactive flow. Claude's reasoning is solid, but this lands after Codex's message — he hasn't had a chance to agree or push back on the specific 3-method split.
2. **Do failed/cancelled runs get a persisted history row?** Codex's `AnalysisOutcomeKind` keeps an `abandoned` case; Claude explicitly dropped it, arguing the only real requirement is "don't lose the user's typed text" (already covered by autosaved `IntakeDraft`), not "keep a record that an attempt failed," and that persisting failed attempts creates new UI questions (does it show in history? is it tappable?) nobody asked for. This is a real, named disagreement — Codex hasn't responded to it yet.
3. **`ManualTaggingChunk` vs. unifying on `ReviewItemDraft`.** Claude proposes deleting the chunk type entirely and having manual tagging work directly over `ReviewItemDraft` end to end. But Codex's round-2 guardrail on `AskSegmenter` specifically required a `split(_:at:)` operation (using `startOffset`/`endOffset`) as the safety valve for when the segmenter over-groups two asks into one candidate — and Claude's simplified `ReviewItemDraft` has no offset fields, so that safety valve has nowhere to attach. This needs to be resolved together, not separately.
4. **`AnalysisMode` now means two different things.** Codex uses it as the *user's preference* passed into the orchestrator (`automaticPreferred`/`forceManual`). Claude redefines the same name as the *recorded fact* of which extractor actually processed something (`aiAssisted`/`manual`, replacing `ExtractorOrigin`). Same type name, two different concepts — needs to become two distinct types before a build lane picks the wrong one.

### Round 3


**Codex (QA & Risk · Closer) — Round 3**

Claude’s three open items are real, and I don’t think we should pretend they’re “small.” They decide whether this thing is mechanically trustworthy or just elegantly described. I’m with Claude on one of them, against him on one, and I think the third only closes if we stop trying to collapse two different lifecycle stages into one type.

First: the orchestrator should be split. Codex’s original single `runAnalysis(projectID:input:preferredMode:)` is tidy on paper, but it hides a category error. AI analysis is atomic: input goes in, normalized output or an error comes out. Manual analysis is not atomic; it is a user-edited workspace with autosave, split/discard/tag operations, and only later a final commit into session history. If we keep one “run” entrypoint, either we lie about manual being a one-shot operation or we start returning union types and optional junk that muddles the boundary. That is exactly how quota bugs and half-persisted sessions happen. So I’d lock Claude’s three-method split: `runAIAnalysis`, `runManualSegmentation`, `finalizeManualAnalysis`. That gives one clean transaction for AI, one clean creation step for the manual workspace, and one clean finalization step that is the only place a manual session becomes history.

Second: I think Claude is right to kill `abandoned` as a persisted `AnalysisSession` outcome. The app needs durable work recovery, not durable evidence that a run failed. Those are not the same thing. We already have `IntakeDraft` and `ManualTaggingDraftRecord` to preserve the user’s labor through crash/background/cancel. Persisting failed or cancelled attempts as first-class history rows creates a bunch of product debt nobody asked for: do they appear in past analyses, can they generate drafts later, do they count toward “last analysis,” what copy explains an empty failed record? That is all downside for no real user value. The acceptance rule is simpler and safer if we say: only completed analyses become `AnalysisSession`s. Cancelled, interrupted, oversized, unavailable, or failed AI runs do not create session history and do not touch quota. Their recovery surface is the autosaved draft state, not a zombie history row.

Where I disagree with Claude is the attempt to delete `ManualTaggingChunk`. Codex’s safety valve is load-bearing. If the segmenter over-groups “while you’re in there, can you also add dark mode and update the pricing page” into one candidate, the user needs to split that candidate at a character offset. That operation belongs to a manual-workspace type with offsets and discard state. You can force that into `ReviewItemDraft`, but then `ReviewItemDraft` stops being the normalized extractor DTO and turns into a leaky hybrid carrying manual-editor concerns everywhere. That’s the wrong compression. Keep `ReviewItemDraft` as the normalized cross-path DTO returned by extractors. Keep `ManualTaggingChunk` as the mutable, offset-aware working type inside the manual path. Then `finalizeManualAnalysis` converts resolved chunks into `ReviewItem`s and persists the completed `AnalysisSession`. That is the clean separation: chunk is edit-time, review item is result-time.

So the concrete close here is: `RunPreference` and `ProcessingOrigin` stay split as two different enums; orchestrator is the three-method shape; no persisted `abandoned` session outcome; `ManualTaggingChunk` stays, with offsets, because split-further is a real acceptance-path behavior, not a nice-to-have. That closes the loops instead of smoothing them over.

**Claude (Frontend Engineer · Visionary) — Round 3**

Let's knock these four out one at a time because I think most of them actually resolve cleanly rather than needing more debate.

On the orchestrator shape — Codex, your round 2 prose already argued for exactly the separation I split into three methods ("view models do intent and state... `AnalysisOrchestrator` owns the one cross-cutting transaction boundary for 'run an analysis and commit the result correctly'"), you just hadn't reacted to my specific 3-method breakdown yet because it landed after your message. I don't think there's actually daylight here once you look at it: a single `runAnalysis(preferredMode:)` either has to fake atomicity for the manual path (pretend the whole tag-everything-by-hand process is one function call, which it isn't — it spans however many taps and possibly a backgrounding) or it has to return some kind of "in-progress handle" that the caller then pokes at, which just reinvents three methods with extra indirection. I'm keeping `runAIAnalysis` / `runManualSegmentation` / `finalizeManualAnalysis` and calling this one settled unless you see a concrete failure mode in it.

And here's the thing that falls out of that split almost for free: we don't need `RunPreference`/`AnalysisMode-as-preference` as a type at all. The coordinator flagged this as a rename problem — your `AnalysisMode` meaning "what the user asked for" colliding with mine meaning "what actually happened." But once the orchestrator has three distinct methods, the *choice of which method to call* is the preference. `IntakeViewModel` doesn't pass `.forceManual` into a generic function, it just calls `runManualSegmentation` directly. That's not a naming fix, it's deleting a whole type because the method signature already encodes the thing it was for. What we do still need is the recorded-fact enum — I'll keep that named `ProcessingOrigin` (`aiAssisted`/`manual`) since that's genuinely a different piece of information (what happened) from a preference (what was asked), and it's what persists on `AnalysisSession` and `ReviewItem`.

On abandoned/cancelled history — I want to actually push on this rather than just restate it, because Codex hasn't given a reason to keep `.abandoned`, it's been carried forward from round 1 without a fresh argument. Walk through what it would actually mean to keep it: if a user backgrounds the app mid-inference, we'd write a permanent `AnalysisSession` row with `outcome: .abandoned` and zero review items. Does that show up in the "past analyses" list on Project Home? If yes, that list — which every locked design phase describes as "evidence/history" of real analyses — now has entries that are just noise, dead ends the user can't do anything with. If no, then we've persisted a row that's invisible everywhere, which means it exists purely for... what, a debug query? The actual product requirement locked back in `detailed_discussion` was specifically about not losing typed text, and `IntakeDraft`'s continuous autosave already solves that completely — the user relaunches, their text is still there, they hit analyze again. I don't see what a persisted failure row buys anyone. I'm keeping `AnalysisOutcomeKind` at exactly two cases and treating this as closed unless someone names a concrete use for the third case.

The one I actually want to spend real effort on is the `ManualTaggingChunk` vs `ReviewItemDraft` fight, because Codex's `split(_:at:)` requirement is genuinely good — I under-thought this in round 2. If the segmenter over-groups "can you also update the header and while you're at it fix the footer spacing" into one candidate, the user needs a way to break it into two without retyping either half, and that only works if the working item carries its own offsets into the source text. So instead of keeping two parallel types, I'm just adding `startOffset: Int`, `endOffset: Int`, and `isDiscarded: Bool` onto `ReviewItemDraft` itself. That gets us everything: one shape for extractor output, manual-tagging working state, and (as a nice side effect nobody asked for but which falls out for free) enough information for the Review screen to eventually highlight the exact span of the original message a rationale is citing, since we're carrying the offsets all the way through instead of just a copied quote string. `ManualTaggingChunk` goes away entirely — `ManualTaggingDraftRecord.itemsData` is an encoded `[ReviewItemDraft]`, `split()` slices one draft into two by offset, `discard()` flips `isDiscarded`, and only non-discarded, resolved items get promoted into real `ReviewItem` rows on `finalizeManualAnalysis`. That's Codex's safety valve fully intact, on my simpler type count.

### Round 4


**Codex (QA & Risk · Closer) — Round 4**

Claude’s round-3 proposal is the right compression, and I don’t think we should burn another cycle pretending this is still ambiguous. Codex’s objection was reasonable when the merge was “stuff manual-editing concerns into the extractor DTO and hope it stays clean,” but the actual shape on the table now is narrower and defensible: `ReviewItemDraft` is no longer just an extractor DTO, it is the cross-lane mutable working item for the pre-persist review pipeline. Once that’s true, `startOffset`, `endOffset`, and `isDiscarded` stop being “leaks” and become exactly the fields that make the manual workspace safe. QA-wise, I care much more about split/discard correctness and quote provenance than I care about keeping a type aesthetically pure. If `ManualTaggingChunk` stays separate, we create a second mutable editorial model that then has to be losslessly transformed into `ReviewItemDraft` before finalization. That is where bugs breed: split one chunk into two, discard one, re-tag the other, then discover the downstream review item lost offsets or diverged from the displayed quote. One mutable item shape is cheaper to verify. The offsets also buy us a real verification hook later: the source quote shown in Review can be tested against the exact source range the manual editor operated on. That’s better than “trust the mapper.” So I’d kill `ManualTaggingChunk`, keep `ReviewItemDraft` as the one editable draft item, and let `ManualTaggingDraftRecord` persist an encoded `[ReviewItemDraft]`. That is the simpler system and, more importantly in my lane, the one with fewer silent mismatch opportunities.

On `sourceOrigin`, the coordinator is right that it can’t just fall on the floor, but I want to tighten one thing: `finalizeManualAnalysis` should not ask for `sourceOrigin` again. Asking twice creates a contradiction vector for no gain. The origin is a property of how the raw text entered the system, and that entry happened before manual editing started. So `runAIAnalysis(projectID:input:sourceOrigin:)` takes it, `runManualSegmentation(projectID:input:sourceOrigin:)` takes it, and `ManualTaggingDraftRecord` persists it. Then `finalizeManualAnalysis(projectID:draft:)` reads `draft.sourceOrigin` and carries it into `AnalysisSession`. That closes the audit trail cleanly and avoids a nasty class of bugs where a share-extension intake somehow finalizes as `pastedCold` because somebody passed the wrong enum the second time. That’s the kind of thing users never see directly but absolutely undermines trust in analytics, debugging, and handoff behavior. With those two calls locked, I think the room is done: one editable draft-item type, no redundant chunk model, no duplicated source-origin parameter at finalization, and every cross-lane interface now actually supports the app we said we’re building and the tests we said we’re going to run.

Codex, I went back and looked at your actual objection again rather than just plowing ahead with my merge, and I think you're right about the structural point even though I'm not fully caving on the underlying idea. Your line was that folding offsets and discard-state into `ReviewItemDraft` turns "the normalized extractor DTO" into "a leaky hybrid carrying manual-editor concerns everywhere," and "chunk is edit-time, review item is result-time." That's a real distinction, not just a naming preference — `isDiscarded` and mid-edit `verdict: Verdict?` (optional, because a chunk might not be tagged yet) are genuinely mutable-workspace concerns that have no business appearing on something that's supposed to be the clean, converged, "both paths produce this exact shape" contract. If `ReviewItemDraft` carries `isDiscarded`, every consumer of that type now has to reason about whether a given instance is a live editing artifact or a finished result, and that ambiguity is exactly the kind of thing that causes a bug three months from now when someone forgets to filter discarded items before rendering something. So: `ManualTaggingChunk` stays, exactly as you specified it, offsets and all, as the mutable pre-resolution editing type. I'm dropping my "delete it and unify everything" position.

But here's the part of my idea I'm not giving up, because it solves a different problem than the one you were pushing back on: I still want `startOffset`/`endOffset` on `ReviewItemDraft` (and by extension the persisted `ReviewItem`) — not for editing, but because that's the only way the Review screen can eventually highlight the exact span of the original message a rationale is citing, for AI-produced items just as much as manually-tagged ones. If we only put offsets on `ManualTaggingChunk`, then AI-sourced review items — which is the majority path, since manual is the fallback — never get that capability at all, which seems backwards given AI output is the one place users are most likely to want to double-check "wait, where exactly did it see that ask" against messy real client phrasing. So the fix isn't "put editing fields on the DTO," it's "carry the position information through as plain data, without the editing semantics." `AIScopeExtractor` already knows each `CandidateAsk`'s offsets since it's operating over segmenter output, so it just passes them through into the `ReviewItemDraft` it returns. `finalizeManualAnalysis` does the same thing when it converts a resolved, non-discarded `ManualTaggingChunk` into a `ReviewItemDraft` — copy the offsets over, drop `isDiscarded` because a discarded chunk never gets converted at all. That's the clean version: `ManualTaggingChunk` owns the messy mid-edit state, `ReviewItemDraft`/`ReviewItem` own the finished result plus enough positional data to support a real product feature later, and neither type is doing the other's job.

On the `sourceOrigin` parameter — that was a straightforward mistake on my end, not a deliberate edit. I had it in my round-2 version, then when I rewrote the orchestrator signature in round 3 to focus on the method-split argument I copy-pasted from an older draft and it fell out. Restoring it on `runAIAnalysis` and `runManualSegmentation` is correct, and I'd add one more thing that falls out of that: `ManualTaggingDraftRecord` needs its own `sourceOrigin: SourceOrigin` field too, because `finalizeManualAnalysis` only receives the draft, not the original call's parameters — it needs somewhere to read `sourceOrigin` from when it finally persists the `AnalysisSession`. Otherwise we'd have to pass `sourceOrigin` into `finalizeManualAnalysis` again as a separate parameter and now it can silently disagree with what was recorded at segmentation time, which is a dumb way to lose data integrity over a missing field.

Also picking up your catch that I missed: `UsageMeterRecord` does need to be in `SchemaV1`'s model list — it's a `@Model` type, SwiftData needs to know about it for the same versioned-schema discipline as everything else, and my round-3 list dropped it while yours kept it. Adopting yours there too.

## Final Output

**Architecture overview:** SwiftUI + Observation, feature-oriented with `@MainActor @Observable` view models per screen, `NavigationStack` driven by a synchronously-computed `NavigationPath`/`AppRoute`. No view touches SwiftData or StoreKit directly — everything sits behind protocols. `AnalysisOrchestrator` is the single cross-cutting transaction boundary (input validation, entitlement/quota checks, extractor invocation, quota increment, persistence commit), split into three methods rather than one because AI analysis is atomic while manual tagging is a genuinely multi-step interactive workflow: `runAIAnalysis` (atomic), `runManualSegmentation` (creates the editable workspace), `finalizeManualAnalysis` (commits it to history once the user resolves everything). There is no separate "run preference" type — which orchestrator method a caller invokes *is* the preference signal. `ProcessingOrigin` (`aiAssisted`/`manual`) is the recorded fact of what actually happened, stored on sessions and items; `SourceOrigin` (`pastedCold`/`shareExtension`) is the separate, orthogonal fact of how text physically arrived. `ScopeExtractor` (`AIScopeExtractor`, `ManualScopeExtractor`) is the load-bearing seam, both consuming a shared deterministic `AskSegmenter` (NaturalLanguage-framework-backed) so AI and manual paths start from the same cleaned candidates. Only completed analyses (`itemsFound`/`nothingToFlag`) ever get a persisted `AnalysisSession` — cancelled/failed/backgrounded runs are recoverable purely through the already-autosaved `IntakeDraft`/`ManualTaggingDraftRecord`, never through a permanent "it failed" history row.

**File/module layout:** `App` (entry, `ModelContainer`/`SchemaV1`, synchronous root routing), `Models` (`@Model` types + `VersionedSchema`), `DTOs` (`Sendable` value types crossing actor/protocol boundaries — kept deliberately separate from `Models`), `Services` (extractors, segmenter, orchestrator, entitlement/quota/export/share-ingress services), `Repositories` (protocol + SwiftData-backed implementations), `Features/{ProjectList,ProjectHome,BaselineEditor,Intake,ManualTagging,Review,DraftPreview,Settings}` (view+viewmodel pairs), `DesignSystem` (tokens, shared components), `ShareExtension` (separate target: capture + project picker + App Group JSON write only), `Tests`, `UITests`.

**Data models:** Two cleanly separated lifecycle-stage types resolve the round 2–4 back-and-forth: `ManualTaggingChunk` (mutable, edit-time only — offsets, optional verdict, `isDiscarded`, lives inside `ManualTaggingDraftRecord`) versus `ReviewItemDraft`/`ReviewItem` (the normalized, post-resolution shape both paths converge on — carries offsets for future quote-highlighting but deliberately has no `isDiscarded` or optional-verdict field, because a discarded chunk is structurally never converted into one). `Baseline` uses structured `BaselineClause` (stable IDs) for deliverables/exclusions so `ReviewItem` can reference a specific clause by ID *and* a text snapshot (so history stays interpretable even if the baseline is later edited). `AnalysisSession` is immutable and only ever persisted on completion. `DraftDocument` is write-once. `IntakeDraft`/`ManualTaggingDraftRecord` are the continuously-autosaved mutable working state — the actual data-loss safety net, not a persisted failure record.

**Persistence strategy:** SwiftData, single app-owned container, explicit `SchemaV1`/`VersionedSchema` from day one (including `UsageMeterRecord` and `ManualTaggingChunk` in the model list). Cascade-delete from `Project` verified by a dedicated test. Share extension writes a plain JSON file (`PendingSharedIntake`) into an App Group container; main app consumes it exactly once, no shared SwiftData access across processes.

**Testing strategy:** Unit tests on `AskSegmenter` (deterministic, exact-match testable, unlike LLM classification), extractor-protocol conformance across both implementations, quota-increment-exactly-once (only inside `AnalysisOrchestrator.runAIAnalysis` on completion), cascade-delete, draft immutability. `XCUITest` runs the full loop twice (real + forced `.ineligibleHardware`) plus a real-device pass. StoreKit fail-closed paths tested via the debug-only `DebugFailureHarness` since sandbox/local config can't reach them organically. Extraction quality against messy client text is a human-reviewed golden-set regression, not a CI-gated exact-match test.

## Final Output

**Architecture overview:** SwiftUI + Observation, feature-oriented with `@MainActor @Observable` view models per screen, `NavigationStack` driven by a synchronously-computed `NavigationPath`/`AppRoute`. No view touches SwiftData or StoreKit directly — everything sits behind protocols. `AnalysisOrchestrator` is the single cross-cutting transaction boundary (input validation, entitlement/quota checks, extractor invocation, quota increment, persistence commit), split into three methods rather than one because AI analysis is atomic while manual tagging is a genuinely multi-step interactive workflow: `runAIAnalysis` (atomic), `runManualSegmentation` (creates the editable workspace), `finalizeManualAnalysis` (commits it to history once the user resolves everything). There is no separate "run preference" type — which orchestrator method a caller invokes *is* the preference signal. `ProcessingOrigin` (`aiAssisted`/`manual`) is the recorded fact of what actually happened, stored on sessions and items; `SourceOrigin` (`pastedCold`/`shareExtension`) is the separate, orthogonal fact of how text physically arrived. `ScopeExtractor` (`AIScopeExtractor`, `ManualScopeExtractor`) is the load-bearing seam, both consuming a shared deterministic `AskSegmenter` (NaturalLanguage-framework-backed) so AI and manual paths start from the same cleaned candidates. Only completed analyses (`itemsFound`/`nothingToFlag`) ever get a persisted `AnalysisSession` — cancelled/failed/backgrounded runs are recoverable purely through the already-autosaved `IntakeDraft`/`ManualTaggingDraftRecord`, never through a permanent "it failed" history row.

**File/module layout:** `App` (entry, `ModelContainer`/`SchemaV1`, synchronous root routing), `Models` (`@Model` types + `VersionedSchema`), `DTOs` (`Sendable` value types crossing actor/protocol boundaries — kept deliberately separate from `Models`), `Services` (extractors, segmenter, orchestrator, entitlement/quota/export/share-ingress services), `Repositories` (protocol + SwiftData-backed implementations), `Features/{ProjectList,ProjectHome,BaselineEditor,Intake,ManualTagging,Review,DraftPreview,Settings}` (view+viewmodel pairs), `DesignSystem` (tokens, shared components), `ShareExtension` (separate target: capture + project picker + App Group JSON write only), `Tests`, `UITests`.

**Data models:** Two cleanly separated lifecycle-stage types resolve the round 2–4 back-and-forth: `ManualTaggingChunk` (mutable, edit-time only — offsets, optional verdict, `isDiscarded`, lives inside `ManualTaggingDraftRecord`) versus `ReviewItemDraft`/`ReviewItem` (the normalized, post-resolution shape both paths converge on — carries offsets for future quote-highlighting but deliberately has no `isDiscarded` or optional-verdict field, because a discarded chunk is structurally never converted into one). `Baseline` uses structured `BaselineClause` (stable IDs) for deliverables/exclusions so `ReviewItem` can reference a specific clause by ID *and* a text snapshot (so history stays interpretable even if the baseline is later edited). `AnalysisSession` is immutable and only ever persisted on completion. `DraftDocument` is write-once. `IntakeDraft`/`ManualTaggingDraftRecord` are the continuously-autosaved mutable working state — the actual data-loss safety net, not a persisted failure record.

**Persistence strategy:** SwiftData, single app-owned container, explicit `SchemaV1`/`VersionedSchema` from day one (including `UsageMeterRecord` and `ManualTaggingChunk` in the model list). Cascade-delete from `Project` verified by a dedicated test. Share extension writes a plain JSON file (`PendingSharedIntake`) into an App Group container; main app consumes it exactly once, no shared SwiftData access across processes.

**Testing strategy:** Unit tests on `AskSegmenter` (deterministic, exact-match testable, unlike LLM classification), extractor-protocol conformance across both implementations, quota-increment-exactly-once (only inside `AnalysisOrchestrator.runAIAnalysis` on completion), cascade-delete, draft immutability. `XCUITest` runs the full loop twice (real + forced `.ineligibleHardware`) plus a real-device pass. StoreKit fail-closed paths tested via the debug-only `DebugFailureHarness` since sandbox/local config can't reach them organically. Extraction quality against messy client text is a human-reviewed golden-set regression, not a CI-gated exact-match test.

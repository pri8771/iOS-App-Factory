## Final Output

**Screen inventory (six primary screens + one extension target, single-stack navigation, no tab bar):**
1. **Project List** — root screen. Only actually shown on cold launch when there are zero or 2+ active projects.
2. **Project Home** — baseline summary, "analyze new message" as the dominant action, past analyses nested below the fold. Cold launch skips straight here if there's exactly one active project, regardless of subscription tier.

4. **Paste/Intake** — accepts pasted text or arrives pre-filled from the share extension; branches into either AI-assisted analysis or a full-screen manual split-and-tag alternate mode (a mode, not a separate app path) that feeds the identical review object.
5. **Review** — the load-bearing screen. Fixed section order: flagged-for-your-call first, confidently-new-and-billable second, confidently-in-scope last and de-emphasized/collapsible. "Nothing to flag" is its own full-screen confident state, never a list variant. Persistent bottom bar: "Generate draft" or, if blocked, "Resolve N flagged items."
6. **Draft Preview/Export** — document-styled, inline-editable, bracketed fee placeholder, immutable once generated, unmissable bottom-anchored export action.
7. **Settings/Paywall** — peripheral corner destination (gear/profile icon in the nav bar), not a tab. Holds plan status, monthly AI usage, upgrade path, restore purchases, known-limitation notes, and debug-only forced-failure affordances (visually cordoned off from consumer UI).

**Primary user flow:**
- First-time: launch → empty Project List → create project (Layer 1 only, under a minute) → Project Home → "analyze new message" → Intake → paste → AI or manual analysis → Review (resolve flagged items) → Generate draft → Export.
- Returning, one project: launch routes straight to Project Home → analyze → Intake → Review → Draft → Export.
- Returning via share extension: Mail/Messages → share → Intake pre-filled (project confirmation only if ambiguous) → Review → Draft → Export.

**State model per screen:** Project List — empty/create-first, loaded, at-free-tier-limit (sheet on the create action). Baseline Editor — empty/first-time (Layer 2 collapsed), editing, saved-confirmation, validation-error. Intake — empty, actively-typing/autosaving, loading-extraction ("comparing against [Project]'s scope," cancelable), four distinct AI-unavailable states with their own copy/icon, mid-run failure (distinct from unavailable-at-launch), manual-mode (no loading state at all — instant). Review — loading, populated, nothing-to-flag (full-screen), unresolved-items-blocking-generation. Draft/Export — empty (pre-generation, should be unreachable upstream), generated, export-success, export-error. Settings — free-tier-status, paid-tier-status, debug-forced-failure states (internal builds only).

**Visual direction:** sharp corners, no gradients or bubble-cards; warm-white/paper background with graphite ink text; serif/slab-serif for baseline and draft "document" text, clean grotesk for UI chrome — that split signals weight and seriousness. Color carries meaning, never alone: red for flagged-for-your-call (danger/judgment-required), gold/amber for confidently-new-and-billable (earned good news), green/quiet ink-gray for confidently-in-scope (resolved, no action). Verdict chips read as stamps, not badges. Generous whitespace on the review screen specifically — resist cramming; a stressed person is scanning this in ten seconds.

**Accessibility notes:** each review row is one coherent VoiceOver element (ask + verdict + rationale + reclassify action); reclassify is always a visible, explicitly labeled one-tap control (never a hidden swipe/long-press), minimum 44pt target; color is always paired with icon/text, never the sole signal; Dynamic Type expands layouts vertically rather than truncating, tested specifically against rationale text at largest accessibility sizes; every primary action (analyze, reclassify, generate, export) sits reachable in the bottom two-thirds of the screen for realistic one-handed, standing-up use.

## Final Output

**SwiftUI architecture:** Feature-oriented, thin-layered — `@Observable` view models per screen on the main actor, `NavigationStack` with a programmatically-constructed `NavigationPath` (not enum-routing sugar). No view talks directly to SwiftData or StoreKit; everything sits behind protocols so tests and previews can inject fakes. The load-bearing seam is `ScopeExtractor`, an `async throws -> [ReviewItem]` protocol with a typed error enum (`.modelUnavailable(reason)`, `.cancelled`, `.inferenceFailed`, `.emptyInput`, `.oversizedInput`) so quota/UI-state decisions are never made by string-matching a generic error. Two conformances — `AIScopeExtractor` (Foundation Models) and `ManualScopeExtractor` (split-and-tag UI) — both return the identical normalized `ReviewItem` shape; a small deterministic pre-pass (sentence/ask segmentation, whitespace cleanup) feeds both paths so they start from the same cleaned input. Dependency injection via initializer or `.environment`, swappable from the first commit — not retrofitted after the manual path is hardcoded into the intake view.

**Data model:** Domain objects separated from in-progress workflow objects. `Project` (+ embedded `Baseline`, Layer 1 required/Layer 2 optional), `AnalysisSession` (immutable record of one pasted-message analysis: origin, raw text, extractor mode, quota-charged flag, timestamps, outcome), `ReviewItem` (source quote, normalized ask, verdict, rationale, matched-clause reference or explicit none, confidence, billability recommendation, resolved state, AI/manual origin), `DraftDocument` (write-once immutable snapshot, non-optional fields set only at init — no code path can mutate it later). `IntakeDraft`/`ManualTaggingDraft` are separate autosaved mutable working state, promoted into an `AnalysisSession` on completion or discarded — never the same model shape as the immutable draft, since autosave and immutability are opposite lifecycle guarantees.

**Apple framework choices:** SwiftUI + Observation, SwiftData (single app-owned container, main-app-only — the share extension never touches it directly), Foundation Models behind `ScopeExtractor`, StoreKit 2 behind a thin `EntitlementService`, PDFKit for paid export, UniformTypeIdentifiers + `UIActivityViewController` (via `UIViewControllerRepresentable`) for export/share, `os.Logger` for diagnostics (redacting all client text). No CloudKit — explicitly decided now, not an accidental omission, so nobody flips on SwiftData's CloudKit sync later and silently breaks the "stays on your device" promise (CloudKit also forces schema constraints — no uniques, all-optional relationships — that shouldn't leak into the model preemptively).

**Persistence and local data plan:** Repository-protocol boundary in front of SwiftData so a future cloud/email connector can slot in without rewriting feature logic. Explicit `VersionedSchema` from day one (`SchemaV1`) even with only one version, so the first real migration is a `MigrationStage`, not a rewrite. `@Relationship` cascade-delete rules declared explicitly from Project → Baseline/Analyses/Drafts, verified by a dedicated test (create full graph, delete project, assert zero orphans) — SwiftData does not cascade-delete just because a relationship exists. Share extension writes a plain JSON file into an App Group container; main app consumes it exactly once at cold launch/scene-active, then deletes it (second share before consumption overwrites the first, surfaced via a one-line toast — no queue). Raw intake text and in-progress manual tagging autosave continuously (debounced, cancellable); generated drafts are permanent snapshots. A small debug-visible record tracks the AI extraction counter and last-reset month so quota logic is actually testable pre-launch.

**Permissions/privacy plan:** No runtime permission prompts needed — no camera, location, contacts, microphone, notifications, no background modes (stated explicitly as "none requested" in entitlements, not left as an absence). Share extension needs its own concrete build-verification checklist: `NSExtensionActivationRule` scoped to plain text only, and an App Group entitlement added consistently to both the main app and extension targets (a provisioning/entitlements-file task that can pass in simulator and fail on-device/TestFlight if missed). `PrivacyInfo.xcprivacy` privacy manifest required with the correct "required reason" API declaration (at minimum, UserDefaults) or App Review rejects the submission outright. In-product privacy stance stated explicitly: client message content never leaves the device, no server-side inference, no analytics/crash logs touch source text.

**Dependency/license decision:** Zero third-party dependencies. First-party frameworks cover the entire build — including PDF export (PDFKit, not a wrapper) and the share-extension handoff format (`Codable`/`JSONEncoder`, not a third-party serializer). Explicitly rejected: `swift-snapshot-testing` for Review-screen visual regression — MIT-licensed so it wouldn't violate policy, but it's an unnecessary dependency for a need XCTAttachment image capture already covers; named as excluded now rather than silently revisited later.

**Testing implications:** Unit tests on extractor-protocol conformance (shared test-case table across both implementations), the quota-increment single-location rule (asserts exactly one increment on completion, zero on cancellation), SwiftData cascade-delete, and draft immutability. `XCUITest` runs the full paste→extract→review→draft→export loop twice — once real, once with a debug-forced `.ineligibleHardware` state — plus at least one real-device pass, since the simulator doesn't reliably run on-device Foundation Models inference. StoreKit fail-closed behavior (corrupted cache, verification failure, never-synced fresh install) is untestable via sandbox/local `.storekit` config alone, so the debug-only forced-failure toggles already locked in Settings are load-bearing test infrastructure, not polish — add a "never synced" toggle specifically since that state requires clearing local receipt data, not just flipping a flag. `Transaction.updates` listener starts at app launch in a long-lived task, never lazily on first Settings visit. Extraction quality against messy/hedged real client text is explicitly not unit-testable via exact-match assertions — validated instead via a fixed golden set of realistic messages with human-reviewed expected outcomes, run as a documented manual/semi-automated regression pass before each release, not CI-gated.

So here's the actual substance, stripped of who-said-it-when: everyone agrees AI items and manually-tagged items both need position offsets carried through eventually (for later Review-screen quote-highlighting), and everyone agrees the manual workspace needs `isDiscarded`/optional-verdict state during editing. The only real question is whether that lives on one type or two. Claude's final argument is the stronger one on the merits: if `isDiscarded` sits on the same type that's supposed to be "the normalized thing both paths produce," every future consumer of that type has to remember to filter out discarded items by convention — and forgetting that filter once is a real, silent bug. Keeping `ManualTaggingChunk` as the strictly-pre-resolution editing type and `ReviewItemDraft`/`ReviewItem` as the strictly-post-resolution type (offsets included, but no `isDiscarded`, no optional verdict) makes that bug structurally impossible instead of just avoided by discipline — a discarded item literally never becomes a `ReviewItemDraft` in the first place. That's consistent with the "make illegal states unrepresentable" instinct this whole phase has been reaching for elsewhere (immutable drafts, typed errors instead of string-matching, derived-not-tracked counters), so I'm locking it: two types, not one.

One small gap I want to close on the way in, since neither of them stated it explicitly: `ManualScopeExtractor` still conforms to `ScopeExtractor` and returns `ExtractionOutcome` (i.e., `[ReviewItemDraft]`), but `runManualSegmentation` needs to end up with a `ManualTaggingDraftRecord` full of `ManualTaggingChunk`s, not `ReviewItemDraft`s. The bridge is trivial — the orchestrator maps each returned `ReviewItemDraft` into a fresh `ManualTaggingChunk` (copying text/offsets, dropping the placeholder `.flaggedForReview` verdict down to `nil` since a chunk starts genuinely untagged) — but it wasn't written down anywhere, so I'm adding it as a note rather than leaving it to be rediscovered mid-build.

Everything else — the three-method orchestrator, dropping the `abandoned` outcome, deleting the standalone "preference" type in favor of method-choice-as-preference, `sourceOrigin` living on the two intake-entry orchestrator methods and on `ManualTaggingDraftRecord` (so `finalizeManualAnalysis` doesn't need it passed twice), `UsageMeterRecord` back in the schema — all landed with both sides explicitly agreeing, no loose ends there.

## Final Output

**Architecture overview:** SwiftUI + Observation, feature-oriented with `@MainActor @Observable` view models per screen, `NavigationStack` driven by a synchronously-computed `NavigationPath`/`AppRoute`. No view touches SwiftData or StoreKit directly — everything sits behind protocols. `AnalysisOrchestrator` is the single cross-cutting transaction boundary (input validation, entitlement/quota checks, extractor invocation, quota increment, persistence commit), split into three methods rather than one because AI analysis is atomic while manual tagging is a genuinely multi-step interactive workflow: `runAIAnalysis` (atomic), `runManualSegmentation` (creates the editable workspace), `finalizeManualAnalysis` (commits it to history once the user resolves everything). There is no separate "run preference" type — which orchestrator method a caller invokes *is* the preference signal. `ProcessingOrigin` (`aiAssisted`/`manual`) is the recorded fact of what actually happened, stored on sessions and items; `SourceOrigin` (`pastedCold`/`shareExtension`) is the separate, orthogonal fact of how text physically arrived. `ScopeExtractor` (`AIScopeExtractor`, `ManualScopeExtractor`) is the load-bearing seam, both consuming a shared deterministic `AskSegmenter` (NaturalLanguage-framework-backed) so AI and manual paths start from the same cleaned candidates. Only completed analyses (`itemsFound`/`nothingToFlag`) ever get a persisted `AnalysisSession` — cancelled/failed/backgrounded runs are recoverable purely through the already-autosaved `IntakeDraft`/`ManualTaggingDraftRecord`, never through a permanent "it failed" history row.

**File/module layout:** `App` (entry, `ModelContainer`/`SchemaV1`, synchronous root routing), `Models` (`@Model` types + `VersionedSchema`), `DTOs` (`Sendable` value types crossing actor/protocol boundaries — kept deliberately separate from `Models`), `Services` (extractors, segmenter, orchestrator, entitlement/quota/export/share-ingress services), `Repositories` (protocol + SwiftData-backed implementations), `Features/{ProjectList,ProjectHome,BaselineEditor,Intake,ManualTagging,Review,DraftPreview,Settings}` (view+viewmodel pairs), `DesignSystem` (tokens, shared components), `ShareExtension` (separate target: capture + project picker + App Group JSON write only), `Tests`, `UITests`.

**Data models:** Two cleanly separated lifecycle-stage types resolve the round 2–4 back-and-forth: `ManualTaggingChunk` (mutable, edit-time only — offsets, optional verdict, `isDiscarded`, lives inside `ManualTaggingDraftRecord`) versus `ReviewItemDraft`/`ReviewItem` (the normalized, post-resolution shape both paths converge on — carries offsets for future quote-highlighting but deliberately has no `isDiscarded` or optional-verdict field, because a discarded chunk is structurally never converted into one). `Baseline` uses structured `BaselineClause` (stable IDs) for deliverables/exclusions so `ReviewItem` can reference a specific clause by ID *and* a text snapshot (so history stays interpretable even if the baseline is later edited). `AnalysisSession` is immutable and only ever persisted on completion. `DraftDocument` is write-once. `IntakeDraft`/`ManualTaggingDraftRecord` are the continuously-autosaved mutable working state — the actual data-loss safety net, not a persisted failure record.

**Persistence strategy:** SwiftData, single app-owned container, explicit `SchemaV1`/`VersionedSchema` from day one (including `UsageMeterRecord` and `ManualTaggingChunk` in the model list). Cascade-delete from `Project` verified by a dedicated test. Share extension writes a plain JSON file (`PendingSharedIntake`) into an App Group container; main app consumes it exactly once, no shared SwiftData access across processes.

**Testing strategy:** Unit tests on `AskSegmenter` (deterministic, exact-match testable, unlike LLM classification), extractor-protocol conformance across both implementations, quota-increment-exactly-once (only inside `AnalysisOrchestrator.runAIAnalysis` on completion), cascade-delete, draft immutability. `XCUITest` runs the full loop twice (real + forced `.ineligibleHardware`) plus a real-device pass. StoreKit fail-closed paths tested via the debug-only `DebugFailureHarness` since sandbox/local config can't reach them organically. Extraction quality against messy client text is a human-reviewed golden-set regression, not a CI-gated exact-match test.

## Project Plan

CONSENSUS: YES

Round 2 closed the loop. Codex didn't just accept Claude's manual-first argument, he took it further — building and proving the manual path first isn't a hedge against AI risk, it's what makes his own "stop and validate before adding revenue" gate actually hit on a real date instead of being hostage to however long AI prompt-tuning takes. Both now agree: manual path (segmenter → manual extractor → orchestrator → review → draft → export) gets built and proven end-to-end first, on fixture data, fully device-independent; AI extraction is then layered in as a parallel enrichment track against a spine that's already known to work, with real-device Foundation Models verification called out explicitly as its own scheduled dependency rather than an assumed subtask.

On the share extension, they landed on the same split from two different framings — Codex proposed breaking the old milestone-five bundle into a narrow on-device feasibility spike (App Group entitlement, plain-text-round-trip, exactly-once consume/delete) done early, versus the picker UI/toast/visual polish staying late. Claude's round 2 sharpened exactly why: that plumbing only depends on `ShareIngressStore` and `resolveInitialRoute`, which exist by the end of foundation, so there's no reason to let it ride along with a milestone that's already packed with accessibility, privacy, and logging work and has no slack to absorb an on-device-only surprise. No real gap left between "right after milestone one" and "milestone two" — same window, same intent.

StoreKit is explicitly settled: Claude said outright there's no daylight left, and Codex's own milestone description already had `EntitlementService` and the debug-forcing harness shipping and getting tested together, exit criteria being enforcement-at-the-call-site tests (forced flags actually blocking real calls), not narration.

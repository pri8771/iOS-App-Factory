**Codex (Design Lead · Systems Thinker) — Round 1**

Scope Guard should be built for the freelancer who already has enough business to be vulnerable to informal scope creep, but not enough process overhead to catch it consistently. Think solo designer, developer, strategist, or small two-person shop handling a few active clients at once, mostly through email and messages, often on a phone. The product is not for people learning how scope works. It is for people who already know the difference between agreed work and “one more thing,” but need a faster, cleaner way to act on that distinction in the moment the message arrives.

The app earns its keep if the user can move from incoming message to professional response in one short loop: choose project, confirm a lightweight baseline, paste or share in the client text, review extracted asks against scope, correct anything ambiguous, generate the change-order draft, export or copy it, and send it. That loop has to feel decisive, not assistant-like. The key design truth is that the app is a diff engine, not a chat engine. The review screen should feel like “here is what changed relative to the agreement,” with clear buckets, specific rationale, and quick edits. If it feels like a generic AI summary, it loses the only reason it deserves to exist.

Users will pay for functional leverage, not cosmetics. The free tier should prove the habit: one active project, ten extractions per month, full end-to-end loop at least once without sabotage. Paid value has to unlock real working capacity: unlimited active projects, unlimited extractions, reusable change-order templates, and PDF export that turns the draft into a real sendable artifact. That is coherent because the free tier is for proving the decision workflow, while paid is for freelancers who now rely on it across multiple clients. The subscription is not buying “better vibes.” It is buying more protected margin and less unpaid labor leakage.

The competitive wedge is narrow and strong if protected. Most adjacent tools help after the damage is done: time trackers prove you lost money, contract tools store the baseline, chatbots help wordsmith a reply, project tools log the request after it lands. Scope Guard steps in before the user starts work, at the exact decision point, and translates an ambiguous ask into a billable action. That timing is the product. The real moat is not the model alone; it is the combination of structured baseline, message parsing, editable verdicts, and immediate change-order output in a flow short enough to become habit.

The growth path is niche, not mass-viral, and that is fine. The natural spread mechanism is professional forwarding and client-boundary culture: freelancers share the drafted change order with peers, mention the app in communities, and recommend it in “how I stopped doing free extras” conversations. The exported document is also quiet distribution because it reflects a sharper working process. This is a product that grows through credibility inside freelancer and small-agency circles, not through social content gimmicks. If it becomes widely used, it will be because it gave people a better script for money conversations.

Local-first behavior should be explicit and product-visible. Projects, baselines, pasted messages, analyses, and drafts live on device and survive relaunch. Extraction should prefer on-device Foundation Models when available, but the system has to be built around a protocol boundary so the rest of the app does not care whether the input came from on-device LLM extraction or manual tagging. The manual path must be first-class: same downstream review object, same draft generator, same export. The future cloud path is straightforward if the intake layer is abstracted now: later, email or Slack connectors can feed the same repository and extraction interfaces without rewriting the core product. Cloud is an extension of intake, not a replacement of the product model.

What will make users keep it is not novelty. It is trust plus speed. They need to believe that when they paste a messy client note, the app will either identify the likely new ask, or honestly surface uncertainty and let them fix it fast. Retention comes from “this saved me from replying loosely and eating the work,” repeated enough times that opening Scope Guard becomes reflexive. The design has to support that by making the first verdict fast, the rationale visible, the corrections lightweight, and the output immediately usable.

The biggest risk in this direction is that baseline setup becomes too heavy and kills first-week usage before the loop proves itself. If users have to author a mini-contract before their first result, they will bounce. The right answer is a provisional baseline model: one required summary field up front, then structured scope details added once the user has seen one useful verdict. The other risk is false confidence from extraction. The UI should bias ambiguous cases into “flagged for your call,” never into silent approval.

The detail that must survive into the final brief is simple: Scope Guard is only valuable if it helps the user act before unpaid work starts. Every product decision should be filtered through that.

If a user could get equivalent value by pasting into a generic chatbot and asking the same question each time, the wedge doesn't exist yet, and I think the antidote is making the baseline-aware rationale line unmistakably better than what a stateless chatbot could produce without being re-fed the whole contract every time.

The extractor sits behind a protocol boundary so email/Slack cloud ingestion can be bolted on later as an alternate implementation of the same interface, not a rewrite — but nothing about v1's usefulness depends on that ever shipping.

Viral/niche growth angle: this is a niche-growth product, not a viral one, and I'd rather say that plainly than force a mechanic that doesn't fit. The natural sharing moment is the exported change-order document itself landing in a client's inbox with a clean, professional format — that's a passive credibility signal to the client, not a growth loop with the freelancer's peers. The real growth angle is community-specific word of mouth inside freelancer Slack/Discord/subreddit spaces (r/freelance, indie-hackers-adjacent communities) where "I stopped doing free work because of this app" is a story people actually tell each other, because money-saved-per-month is a concrete, believable claim in exactly those communities. I would not build in-app referral gimmicks or forced sharing prompts — this is a professional tool and if it feels like it wants to go viral, it stops feeling trustworthy for handling client relationships.

What makes users keep it, and the risk that kills that: retention lives or dies on whether the flagged rationale is trustworthy enough that the user stops re-verifying it against the baseline herself. The single biggest risk to the whole brief is that extraction quality on real, messy, hedged client phrasing is mediocre often enough that users start distrusting every verdict, at which point the app becomes a slower way to do something they'd do faster by just rereading the message themselves — and that is precisely the invalidation criteria already named upstream. My answer to that risk is not "make the model better," which isn't fully in our control — it's to bias every uncertain case toward the flagged-for-your-call bucket rather than a confident auto-approval, make the manual fallback genuinely fast (not a punishment screen), and make correcting a wrong verdict take one tap, because a tool that's occasionally wrong but cheap and fast to correct keeps trust; a tool that's occasionally wrong and expensive to correct gets deleted. I want this stated as a build-time non-negotiable, not a nice-to-have: if the review screen makes reclassifying an item take more than one interaction, that's a defect, not a polish item.

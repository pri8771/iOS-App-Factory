**Claude (Frontend Engineer · Systems Thinker) — Round 1**

Looking at this from a pure client-engineering angle, the six-screen IA is already locked, so my job this round is to turn "flagged first, billable second, in-scope last, single-stack nav" into something a SwiftUI build can actually implement without the seams showing — and to flag the places where a clean design doc can hide a genuinely hard state-management problem.

Start with the navigation shape, because it's the thing most likely to get built wrong even though everyone agrees on it conceptually. "Exactly-one-active-project launches straight into Project Home" sounds like a one-line routing rule, but if it's implemented as "show Project List, then push to Project Home in onAppear," the user will see a one-frame flash of the list before the push animates — which reads as janky on a screen we've promised will feel instant. The only correct implementation is computing the initial `NavigationPath` synchronously, before the root view's first render, from a local persisted count (SwiftData fetch of active projects) done at app-launch time in the App/Scene delegate or root view's `init`, not in a lifecycle callback. Same discipline applies to the share-extension handoff: that's not "the user taps something and the app navigates," it's cross-process IPC. The extension and the main app are different processes; they can't hand off an in-memory object. I'd architect this as an App Group container holding a small pending-intake file (raw text, timestamp, optional project-id hint) — the extension just writes it and calls the app via URL scheme or just relies on foreground activation, and the main app checks for a pending file at both cold launch and scene-becomes-active, consumes it exactly once, deletes it, and routes to Intake pre-filled. I want to flag one real edge case nobody's named yet: what happens if the user shares from Mail, doesn't open the app immediately, then shares again from Messages before opening it? A single-file "pending intake" design silently overwrites the first share. I don't think this needs a queue — that's overbuilding for a rare case — but it does need a defined behavior: if a new share arrives while one is unconsumed, the second overwrites the first and the app should surface a small one-line toast on next launch ("replaced an earlier unprocessed shared message") so nothing is silently lost without at least being acknowledged.

On persistence architecture specifically: I'd resist letting the extension and the main app touch the same SwiftData store directly. Cross-process access to one SwiftData container is a sharp edge (migration timing, concurrent writer contention) that isn't worth risking for a feature we've already flagged as the highest-risk must-have. Keep SwiftData as strictly single-process (main app only); the extension only ever writes a plain JSON file into the App Group container. That's a boring, safe boundary and it's the kind of decision that's invisible in a design doc but saves a week of debugging later.

The autosave-vs-immutable-draft split locked in earlier phases needs to show up as two genuinely different persistence code paths, not two flags on one model. Raw pasted text and in-progress manual tagging should autosave on a debounce (roughly 400-600ms after the last keystroke, via `.task(id:)` cancelling the previous save, not a bare timer) into a mutable `IntakeDraft` object. A generated change order should be a write-once `ChangeOrderDraft` — I'd make its fields non-optional lets set only at creation, so there's no code path that can accidentally mutate it later; immutability enforced at the type level, not by convention or a UI affordance that just doesn't expose an edit button.

The Foundation Models availability state needs to be a single source-of-truth enum computed at app scope and observed everywhere it matters, not re-derived per screen: `.available`, `.ineligibleHardware`, `.disabledByUser`, `.modelNotDownloaded`, `.quotaExceeded`, plus a runtime-only `.evictedMidRun` that can only be entered from `.available` once inference has started. That last one matters mechanically: the inference call has to be a cancellable `.task` tied to view lifecycle (not a detached Task), and it has to explicitly distinguish "cancelled because the view disappeared or app backgrounded" from "cancelled because the model failed" — because only one of those should route to the "couldn't complete, try again or switch to manual" copy already locked, and neither should decrement quota. Quota itself should increment in exactly one place in the codebase — at the moment a run produces a final result or a confirmed "nothing to flag," never speculatively at start — otherwise "quota increments only on completed runs" becomes a rule that's easy to state and easy to violate three call sites later.

For screens: Project List needs empty/create-first, loaded, and an at-limit sheet state, and importantly should barely ever render for the target one-project user — designing it fully is still required since 2+ project (paid) users see it regularly. Project Home is a two-block layout: baseline status card (one line if Layer 1 only, richer if Layer 2 filled, with a subtle "add more detail" affordance) and a dominant "Analyze new message" button, past analyses in a plain list below the fold, sharp-cornered rows, no card shadows. Baseline Editor is a form with Layer 1 always visible and required, Layer 2 behind a disclosure — I'd implement Layer 2's disclosure state as persisted per-project once opened, not resetting collapsed every time the editor is reopened, since a user who's already invested in Layer 2 shouldn't have to re-expand it. Intake is a full-bleed text editor with keyboard active on arrival, a small persistent project-context line, and the primary button's label and destination (Analyze vs pushed into manual mode) driven directly off the availability enum — this should be the same view whether reached cold or via share extension, differing only in whether the text field arrives pre-filled, which is the detail that keeps these two entry paths from silently drifting into two implementations over time.

Review is the one screen where I'd push hardest for restraint in what components get built: one `ReviewRow` component, parameterized by verdict (which drives icon + color + section placement), never three near-duplicate row types per bucket — that's both a design-consistency and an engineering-maintenance argument. The bottom action bar is a persistent overlay, not a scroll-away footer — it needs to recompute its enabled/label state reactively off the count of unresolved items, and that count needs to be derived state (computed from the array), never a manually-incremented counter that can drift out of sync with the actual data. Motion should be minimal and fast, not absent — this is "contract meets productivity app," not a static document viewer.

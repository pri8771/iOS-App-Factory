### Round 1


**Codex (Product Strategist · User Advocate) — Round 1**

Scope Guard’s precise promise should be: when a client sends a new ask, the app helps a freelancer decide before replying whether that ask is already covered or should become a paid change order, then gives them a draft they can send.

The primary user is a solo freelancer or very small consultancy doing fixed-fee or tightly scoped client work, usually handling communication through email or messages, and regularly getting “small” add-on asks buried in informal threads. The scenario to keep in the room is not office admin time. It is a person on their phone who just received “can we also add this?” and needs a defensible answer fast without rereading a contract or improvising under pressure.

The core loop needs to stay brutally tight: pick the project, establish or confirm the scope baseline, paste or share in the new client message, review the extracted asks against the baseline, correct anything that is wrong or ambiguous, generate the change-order draft, export or copy it, and move on. That is enough. If the app is doing more than that in v1, it is probably drifting into adjacent software categories and losing the speed that makes it worth opening.

The scope boundary I would lock hard is this: Scope Guard is not a contract authoring tool, not a CRM, not invoicing, not time tracking, not proposal management, not team collaboration, and not general AI advice. It is also not a passive archive of “interesting scope moments.” It only earns its keep if it changes the user’s behavior at the exact moment a new ask arrives. That means baseline management exists only to support fast decision-making later. Client/project records exist only to make the decision loop repeatable. Export exists only so the user can act on the result outside the app. Anything that does not directly support that chain is out.

I would also resolve one important ambiguity now: the manual fallback is part of the main product, not an exception path. Even on supported devices, the user must be able to switch to manual review when extraction is weak, and that manual path must still end in the same draft-generation outcome. If later phases treat fallback as a dead-end or second-class screen, they will be breaking the product, not trimming it.

The biggest risk in my preferred direction is onboarding friction. If we insist on a fully structured baseline before the user can get a first verdict, many people will bounce before they ever see value. So the product should allow a lightweight baseline first: enough structure to compare against, not a contract intake ceremony. The app can tighten rigor after the first successful use, but it cannot demand rigor as the entry fee.

The measurable success criteria should be concrete. A first-time user should be able to create a project and get a first scope verdict from a pasted message in just a few minutes. A returning user should be able to go from incoming message to reviewed draft in under a minute for a normal case. The app should always produce one of three usable outcomes: clearly in scope, clearly new/out of scope, or flagged for human review. It should never silently imply certainty where there isn’t any. And the free-tier user must be able to complete the entire loop successfully at least once without hitting a paywall before the core value is proven.

The detail I want preserved in the final phase output is this: Scope Guard is not selling “AI analysis.” It is selling professional clarity before unpaid work starts. If that sentence gets diluted, the app will become generic very quickly.

**Claude (Design Lead · Systems Thinker) — Round 1**

Let me set the product promise first, then work through it as someone who has to actually diagram the screens and states, because that's where soft ambiguity turns into either a great five-tap loop or a bloated form-filling app.

The one-sentence promise I'd lock in: "Scope Guard turns a client's casual ask into an immediate, evidence-backed verdict — in scope or billable — and a ready-to-send change order, before the freelancer starts working for free." Note what that sentence deliberately excludes: it doesn't say "tracks scope" or "manages projects," because if the promise reads as recordkeeping, every downstream screen will drift toward admin software instead of a decision instrument.

Primary user and scenario: a solo freelancer or two-to-three-person shop, mid-project, who gets a message from an existing client containing a casual extra ask embedded in otherwise normal conversation. She's on her phone, has a couple of minutes, and needs an answer she can act on — either push back, flag it as billable, or confirm it's covered — without opening a laptop or rereading her SOW. The scenario that matters most for IA purposes is the *second and later* use, not the first: onboarding gets one shot to earn trust, but the habit that keeps the app alive is "message lands → three taps → verdict," repeated daily. If the tenth use takes as many taps as the first, the app fails on its own invalidation criteria regardless of extraction quality.

Core loop, stated as screens and transitions rather than abstractly: create/select project (with a provisional one-line baseline allowed, per the carried-forward note from portfolio_selection — I want to affirm that ordering decisively here, not leave it as a maybe) → paste or share-extend client text in → extraction runs (LLM or manual-split fallback, same downstream shape) → review screen shows three buckets (confidently in-scope, flagged for your call, confidently new) each item with a one-line rationale tied to a specific baseline clause → user edits/reclassifies/adds missed asks → generate change-order draft → export or copy. Six real screens: project list, baseline editor, paste/intake, review, draft preview, and a settings/paywall surface. I don't think a seventh screen buys anything at MVP — no dashboard, no history feed as a distinct destination (history can just be "past analyses" nested under a project, not its own IA branch).

Here's the edge case I want on record because nobody's named it yet: what does the review screen do when extraction finds *nothing* flaggable — the whole message is small talk or confirmed in-scope work? That's not an error state and it's not the empty-state-before-first-use; it's a third distinct state ("clean — nothing to flag") and it needs its own confident, reassuring treatment, because if it looks like a failed or empty result, users will stop trusting the tool on the days it correctly tells them "you're fine." A tool that only ever shows output when there's bad news trains users to dread opening it.

Second edge case worth locking now: multi-topic pastes. Real client messages mix confirmed-in-scope chatter, one buried new ask, and a question that isn't a scope matter at all (a scheduling question, a compliment). The review screen's IA has to tolerate a single paste producing zero, one, or several change-order-worthy items, plus content that's simply irrelevant and should be silently dropped rather than forced into one of the three buckets. If the extractor or the manual-fallback UI forces every sentence into a bucket, users will start editing out noise by hand every single time, which is exactly the kind of friction that kills week-two retention.

Hard scope boundaries, decisively: no history/analytics dashboard as its own section, no multi-message inbox or thread view (each paste is a discrete, self-contained analysis), no negotiation/back-and-forth chat with the "AI," no client-facing app or client accounts, no editing the change-order draft into a full contract redline, no reminders/notifications nudging the user to paste things (that's a different product — a monitoring tool — and this is explicitly reactive-at-the-moment-of-ask, not proactive surveillance of communications). One project can have one active baseline at a time — no branching/versioned baselines in v1, just edit-in-place with the understanding that the baseline reflects current agreed scope.

Measurable success criteria I'd hold this build to: a first-time user reaches a real verdict (not a tutorial, an actual classification of pasted text) in under 90 seconds from cold launch, using no more than one required baseline field. A returning user reaches paste-intake in two taps or a single share-extension action. The review screen never presents a flagged item without a visible one-line rationale referencing the baseline. Every one of the six screens has honest empty/loading/success/error states with no placeholder text. And — the falsifiable bar that matters most given the invalidation criteria in the spec — the full loop is demonstrably exercisable start to finish with Foundation Models both available and force-disabled, producing a real exported file either way.

The biggest risk to my own framing: forcing everything into a rigid six-screen IA could make the manual fallback feel bolted-on if its interaction model (splitting/tagging text by hand) doesn't fit naturally into the same "review" screen shape as the LLM path. That has to be solved at the data-model level — both paths must produce the same intermediate representation (a list of candidate asks with scope verdicts and rationale) before they hit the review screen, so review, draft, and export code never need to know which path produced the input. If that convergence point is fuzzy, we'll end up building two parallel apps instead of one app with two intake methods.

**Codex (Design Lead · Skeptic) — Round 1**

The likely buyer is not “freelancers” in the abstract. It is a narrow, high-friction slice: solo designers, developers, marketers, copywriters, and tiny agencies who work from informal client messages, have fixed-fee or tightly bounded retainers, and regularly get scope drift through text rather than through formal SOW amendments. The use context is usually ugly and fast: they are on their phone, a client message just came in, they are between meetings or commuting, and they need to know whether to absorb the ask, push back, or send a clean change-order response without sounding defensive. That means the app cannot feel like admin software. It has to feel like a fast professional decision tool under pressure.

The real competitors are less direct apps and more default behaviors. The strongest alternatives are “do nothing and eat the cost,” Notes or Notion for rough tracking, manual contract/SOW review, ChatGPT-style copy-paste prompting, project tools like ClickUp/Asana that are too far downstream, and invoicing/freelance suites that help after the work is already happening. The product pattern to learn from is not generic AI chat. It is a review workflow: intake, structured comparison, confidence/risk surfacing, user correction, then output. On iPhone, the benchmark is any app that turns messy input into a crisp decision in a few screens, with obvious status, low typing burden, and strong editability. If this feels like “paste text into a magic box and trust the model,” it will fail.

The native iOS opportunity is strong if kept disciplined. Share extension matters because the trigger moment is often Mail or Messages, and forcing users to context-switch and manually re-paste every time adds enough friction to kill repeat use. On-device processing is not just a technical choice but a trust signal, because client messages may be sensitive and freelancers will care where that text goes. Files export also fits the use case well because the output is meant to become an artifact they can send or save. Accessibility matters more than usual because this will be a dense, form-heavy app with legal and commercial nuance, so readable hierarchy, strong contrast, large tap targets, VoiceOver labels, and low cognitive clutter are part of product quality, not cleanup.

The first build should probably center on five screens and no more: project list, project baseline editor, message intake, review/classification, and change-order draft preview/export. The baseline editor is more important than it sounds. If it is just one big freeform text blob, the extraction step has nothing stable to compare against and the app becomes performative AI. The baseline needs structure that maps to how freelancers actually define work: deliverables, included revisions, exclusions, assumptions, timeline/milestones, and maybe communication/support boundaries. But it cannot become contract authoring software. The right feel is “enough structure to support comparison,” not “write a legal document from scratch.”

The review screen is the make-or-break UX. That is where trust is either built or destroyed. It needs a hard distinction between likely in-scope and likely new asks, but also a visible rationale tied back to the baseline. Not legal reasoning, just practical evidence: requested landing page variant not listed in deliverables; additional revision round exceeds included revisions; analytics setup absent from exclusions/assumptions. If the app only labels something “out of scope” without showing why, users will not trust it in a client-facing moment. They need to be able to edit, reclassify, delete bad extractions, and add missed asks quickly. Correction should feel normal, not like admitting the AI failed.

The manual fallback cannot be a dead-end “AI unavailable” notice plus raw text editing. It has to be a usable workflow in its own right: split the message into asks, tag each ask, compare against the baseline, then generate the same downstream draft. I would assume a meaningful share of users will use this even on supported devices, because the risky cases are exactly the ones where model confidence should be low. If the app treats fallback as second-class, the whole product is brittle.

A major assumption I would state explicitly is that users already have some mental model of project scope and do not need the app to teach contracting from first principles. Scope Guard should help them operationalize an existing agreement, not invent one. Another assumption is that the first paid value is not deeper AI, but removing throughput caps and unlocking better output formats/templates. That aligns with the spec, but only works if the free experience still lets someone reach one full successful outcome before feeling squeezed.

The biggest risks are clear. First, extraction quality may be too inconsistent against messy, indirect client language. “While you’re in there” and “could we also” are easy for humans and dangerous for shallow models. Second, baseline quality may vary wildly because many freelancers do not define scope cleanly enough for precise comparison. Third, the product may ask for too much setup before delivering value; if project creation and baseline entry feel like homework, users will not keep pasting messages after week one. Fourth, phone-only usage is a double-edged sword: the trigger moment is mobile, but editing dense scope data on a small screen can become punishing if the IA is not ruthlessly compressed. Fifth, there is reputational risk if the output sounds overconfident or quasi-legal; the copy has to stay commercial and practical, not pretend to be counsel.

So the implications for the first build are pretty strict. Make onboarding end in a usable sample baseline fast, or provide a sharp starter template so users do not face a blank contract-grade form. Optimize for “I got a client message and need an answer now,” not for long-term project administration. Build the review layer before polishing the generation layer, because bad review UX destroys trust faster than mediocre draft prose. Treat privacy as a visible product promise: on device, local-first, your client messages stay with you. Make the free tier generous enough for a freelancer to prove the habit. And preserve one crucial detail in the final output: this app is not a scope tracker, it is a pre-response decision instrument. If any later phase drifts toward recordkeeping over action, it will miss the point.

**Claude (QA & Risk · Pragmatist) — Round 1**

Let me think about who's actually going to open this app and what breaks first.

The realistic user is a solo freelancer — designer, dev, copywriter, marketing consultant — running maybe 1-4 concurrent client relationships over Slack/email/text/Basecamp, who already has a scope document somewhere (a Statement of Work, a Notion doc, a contract PDF) that nobody re-reads once the project starts. The moment this app has to nail is not "sophisticated project management," it's "client just sent a paragraph, I have thirty seconds before I reply, is this free or billable." That means the app is used in short, high-frequency, high-stress bursts — often standing up, often on the phone in an Uber, often right after reading a Slack notification. That context has real UX consequences the spec doesn't spell out: the paste-and-extract screen needs to be reachable in two taps from a cold launch (ideally zero taps via share extension), and it cannot force the user through a multi-step baseline-editing detour before they can get an answer to "is this billable" on a message they just received. If baseline setup is a prerequisite blocking wall before the core loop works even once, we've built a form-filling app, not a scope guard.

For comparable patterns, I'd look less at direct competitors (there isn't a clean one — this sits in a gap between Harvest/time-tracking, HelloSign/PandaDoc contract tools, and generic AI summarizers like Otter or ChatGPT-pasted-into) and more at the UX of tools that already do "paste something messy, get a structured judgment call back": mail triage apps that classify urgency, expense-report apps that OCR a receipt and ask you to confirm categories, and diff/review tools (a change-order draft is structurally a diff between baseline and ask). The diff framing is the one worth stealing hardest — freelancers already understand "here's what changed" visually from Track Changes or PR review, and that mental model is more trustworthy than an AI chat bubble telling them "this seems out of scope." I'd treat this as an assumption, not a researched fact: I have not verified any specific competing app's interface, I'm reasoning from category analogues.

Platform-wise, the two opportunities worth taking seriously are the share extension and Shortcuts/App Intents. Share-from-Mail is explicitly named in the spec, and it's the single highest-leverage native integration because it removes the "open app, find project, paste" friction entirely — the client message becomes the entry point. But I want to flag a concrete risk here: share extensions have their own process, memory ceiling, and can't easily host a full Foundation Models session or a heavy review UI — realistically the extension should capture text plus a project picker and hand off to the main app for the actual extraction/review/draft, not try to do extraction inline. If a later phase tries to run the whole pipeline inside the extension, that's a scope trap that eats days for marginal benefit. Widgets or a Lock Screen/Action Button shortcut to "paste and check" is a nice-to-have, not core, and I'd explicitly deprioritize it against getting the main loop rock solid.

Now the risks I actually want on record, because I think these are the ones that sink the invalidation criteria if unaddressed. First, Foundation Models availability is not just a binary "has Apple Intelligence or not" — it's gated by region, language, storage headroom (the model can be evicted under memory pressure), and first-run model download state, so "unavailable" is actually several distinct states (ineligible hardware, eligible but not enabled in Settings, eligible but model not yet downloaded, eligible but momentarily evicted mid-session) and the app needs to detect and message these differently rather than collapsing them into one generic "AI unavailable, use manual mode" banner — a user who could get the LLM path working but doesn't know why they're stuck in manual mode will file it as a bug, not a limitation. Second, extraction on short, ambiguous, hedged client prose is genuinely hard even for large models — "before I do unpaid work" is exactly the case where a false negative (labeling an out-of-scope ask as in-scope) costs the user money, so the review step cannot be a lightweight rubber-stamp; the UI needs to visibly show its own uncertainty (e.g., don't present a binary in/out toggle with equal visual weight — bias the default toward "flag for review" rather than "auto-approved as in-scope" when confidence is anything but high, because the cost of a missed flag is asymmetric with the cost of a false alarm). Third, the free tier's "10 extractions/month" only means something if extraction is a well-defined, non-gameable unit — is pasting the same message twice two extractions? Does editing and re-running count again? That needs a hard rule now (I'd propose: one extraction = one successful run of the extractor against one pasted text, counted at run time regardless of downstream edits, with re-running after edits to the source text counting as a new extraction) or the paywall becomes either trivially bypassable or arbitrarily punishing. Fourth — smaller but real — StoreKit 2 sandbox/entitlement testing and Foundation Models are both things that cannot be verified in a plain simulator run; the first build needs to actually get exercised on real hardware (or at minimum acknowledge in the build report that simulator-only verification is a known gap), otherwise "we tested it" is not a true claim.

What I want the first build to actually contain, given all that: a scope baseline that can be created in under a minute with sane defaults (title, a short deliverables list, one free-text exclusions/assumptions field) rather than a heavyweight form, because a baseline that takes ten minutes to fill in on day one will get skipped or left empty, which then poisons every downstream comparison; a paste screen that works identically whether it was opened cold or via share extension; a review screen whose default state clearly separates "confidently in scope" / "flagged, needs your call" / "confidently new ask," with editing always one tap away, never buried; a manual mode that isn't a stripped-down version of the same screen but is asked to do the same job with a slightly different interaction (e.g., user highlights or splits the pasted text into asks themselves, then tags each against the baseline) — same downstream draft generation either way; and an extraction-count/paywall implementation that's testable by a QA pass without needing to wait a real month, i.e. the reset boundary and count need to be inspectable/resettable in a debug build. I'd rather we ship exactly that narrow loop with all states honestly handled than a broader feature set with any of those corners cut.

On-device processing doubles as a trust signal worth stating explicitly in the UI ("stays on your device") since client message content is sensitive. **Major assumptions and risks:**
1. *(assumption)* Users already have a rough mental model of their own project scope — the app operationalizes an existing agreement, it doesn't teach contracting from scratch.
2. *(risk)* Foundation Models "unavailable" is not one state but several (ineligible hardware, eligible-but-disabled, eligible-but-not-downloaded, eligible-but-evicted mid-session) — collapsing these into one generic banner will read as a bug, not a limitation.
3. *(risk)* Extraction on hedged, indirect client phrasing ("while you're in there," "could we also") is genuinely hard even for large models; false negatives (mislabeling an out-of-scope ask as in-scope) are the costly failure mode, so the review UI must default ambiguous items to "flag for review," never to silent auto-approval.

6. *(risk)* StoreKit 2 entitlements and Foundation Models availability can't be fully verified in a plain simulator run — real-hardware verification (or an explicit documented gap) is required before calling this "tested."

**Implications for the first build:**
- Baseline creation must be fast (sane defaults: title, short deliverables list, one free-text exclusions/assumptions field) — under a minute, not a contract-authoring session.
- Paste screen must behave identically whether opened cold or via share extension.
- Review screen defaults to three visible buckets — confidently in-scope / flagged for your call / confidently new — with editing always one tap away, never buried; ambiguous confidence biases toward "flag," never toward auto-approval.
- Manual fallback is a first-class alternate path with its own interaction model (user splits/tags asks manually) feeding the *same* downstream draft generation — not a stripped-down version of the LLM screen.
- Extraction counting needs a hard, testable definition and a debug-inspectable/resettable counter so QA doesn't need to wait a real month.
- Foundation-Models-unavailable states need to be distinguished and messaged differently, not collapsed into one banner.
- Ship the narrower loop (project → baseline → paste → review → draft → export) with every state honestly handled, rather than a wider feature set with corners cut.

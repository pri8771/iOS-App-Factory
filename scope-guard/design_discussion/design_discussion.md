The design should treat Scope Guard as a professional instrument, not a productivity toy. That means the app cannot feel like “paste text into AI and see what happens.” It needs a tight, legible chain of custody from project scope to message intake to verdict to draft. The right IA is six primary screens with one modal branch, and anything beyond that is drift.

The first screen is the project list, but it should behave like a command surface, not a database. If the user has no projects, the empty state pushes straight into “Start your first scope baseline” with one sentence explaining that a one-line agreement summary is enough to begin. If the user has one active free-tier project, that project should dominate the screen and the primary action should be “Analyze new message,” not “manage projects.” If the user has several paid projects later, the list can expand, but the first tap still needs to get them to intake fast. This screen also needs the free-tier project-cap state, deletion confirmation state, and a visibly local-first privacy note somewhere restrained, not a marketing banner.

The second screen is the baseline editor, and it should be split visually into two layers because that is a product truth, not just a data model. The top section is the fast baseline: project name, optional client name, and the one-line “what I agreed to build.” The lower section is the stronger baseline: deliverables, exclusions, revision policy, assumptions, timeline, client dependencies. The UI should not present this as a giant form all at once. It should read as “Enough to start” and then “Make verdicts stronger.” The main state difference here is not just empty/edit/saved/error. It also needs a thin-baseline state where the app is explicitly honest that later verdicts may rely more on general judgment than clause matching. That copy matters because it sets up trust before the first ambiguous result.

The third screen is intake, and this is the behavioral center of the app. It needs to work whether entered cold from the app or handed off from the share extension, and those two entry paths should look identical once the text is in. The user sees the selected project, a large pasted-text field, a persistent note that analysis stays on device, and one primary action. If AI is available and the user has quota, that action is “Analyze message.” If AI is unavailable or capped, the primary action changes cleanly into the manual path without making the user feel rejected. The loading state here needs to feel serious and brief: not playful spinner copy, but “Reviewing against baseline…” with a cancel option. Error states need to distinguish empty input, oversized input, AI unavailable at start, and AI failed mid-run. If the user leaves, the draft text must still be there on return.

The fourth screen is the review screen, and this is where the app either becomes credible or dies. It should not be a chat transcript.

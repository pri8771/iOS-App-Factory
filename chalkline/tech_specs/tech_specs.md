**Claude (Frontend Engineer · Visionary) — Round 1**

Let me stake out the client-architecture ground before anyone else weighs in, since that's the lens I'm bringing this round.

Start with the thing I think the prior phase glossed over and that will actually block someone from writing code on day one: `ios_architecture_review` locked `AnalysisQueue` as "an actor holding job state... an `@Observable` analysis-queue service." Those two things can't both be literally true. `@Observable` is a macro for classes; Swift doesn't let you slap it on an `actor`, and even if it did, actor-isolated mutable state doesn't publish change notifications the way SwiftUI's Observation framework expects — you'd get an isolation-boundary compile error the moment a view tried to read `queue.progress` synchronously from the main actor. This isn't a nitpick, it's the load-bearing plumbing for "Processing shows a real deterministic gauge that's dismissible without cancelling work," which is one of the more distinctive interaction promises in this whole app. My answer: keep `AnalysisQueue` a pure `actor` — it does the work, owns job state, is the only thing that touches Vision/AVAssetReader — and give it an `AsyncStream<AnalysisProgress>` per attempt that callers can subscribe to. Then a separate, thin `@MainActor @Observable` type, `AnalysisProgressStore`, subscribes to that stream inside a `.task(id: attemptID)` on the Processing/Home views and republishes plain `@Observable` state for SwiftUI to bind to. Two types, one actor doing real work, one tiny reactive mirror — cheap to build, and it resolves a real compile-time contradiction instead of letting whoever builds Processing discover it the hard way.

While I'm in there: I want to push the progress signal past "spinner with a percentage." The design phase asked for "deterministic frame-count progress + confidence trend" — I want to take that literally and make it feel alive: as each batch of sampled frames comes back from Vision, the gauge should show both frames-processed and a running confidence trend line building in real time, not a single number that jumps from 0 to final at the end. That's genuinely satisfying to watch for the ~2-10 seconds this pipeline actually takes (I'm reusing the compute estimate from the last phase — 270 frames at 6fps, tens of milliseconds per frame, single-digit seconds typical), and it reinforces the "instrument, not decoration" trust story the whole product is built on. The risk I'll name against myself: this is more plumbing than the group has explicitly asked for, and there's a simpler fallback (a fake/timed progress bar that just looks busy) that would ship faster. I think the real thing is worth it specifically because this app's entire pitch is "don't show a number you can't back up" — a progress gauge that's secretly fake would be a small, ironic betrayal of the app's own values. But if time runs out, faking it with a timer is the one place I'd accept a compromise, and it should be named as exactly that in code, not silently substituted.

On persistence, I want to lock the schema down harder than "linked blob or child entity" — pick one. I'm going with a single `Data?` column on `Attempt` holding a JSON-encoded `[PoseSample]`. No second table, no relationship, no child `@Model`. A 45-second clip is ~270 samples with maybe a dozen joints each — that's tens of kilobytes, trivial to decode synchronously on the main thread when Review opens; there's no case here for a background-decode dance or a queryable child table, since nothing ever queries into individual pose samples independent of their parent attempt. This keeps the SwiftData surface at exactly one table, which is the whole point of the `Attempt`-only decision the last phase fought for — I don't want that discipline to leak back in through "well the trace data needs its own entity for cleanliness."

For file/module layout, my one genuinely opinionated move: pull everything that has zero business touching UIKit, SwiftUI, Vision, AVFoundation, or SwiftData into a local Swift package, `ChalklineCore` — `PoseSample`, `JointPoint`, `PoseAnalysisResult`, `AnalysisProgress`, `AnalysisJobState`, `AttemptStatus`/`Outcome`/`Source`, `ComparisonResult`, and the pure functions (`ConfidenceAggregator`, `TraceGapDetector`, `RouteLabelNormalizer`). This isn't over-modularization for its own sake — it's the cheapest possible way to make the "pure-function tests run without a simulator" requirement architecturally enforced instead of a promise someone can quietly break by importing `Vision` into a file that's supposed to be pure math. `swift test` on that package boots in under a second, no simulator, and CI can run it on every commit. The app target then depends on `ChalklineCore` plus does everything device/UI-specific: `Home/`, `Capture/`, `Processing/`, `Review/`, `Permissions/`, `Persistence/` (the `@MainActor` gateway plus SwiftData model itself, since `@Model` needs to live somewhere that imports SwiftData), `Analysis/` (the actor and the concrete Vision engine), `Entitlement/`, and `DesignSystem/` for tokens, haptics, and motion. Objection to my own idea: a second package is one more moving part for what's still a small MVP, and someone has to remember the import boundary. I think it's worth it here specifically because this app's credibility rests on the determinism claim, and having the compiler enforce "this file cannot see Vision" is stronger than a code-review convention.

Two more concrete pushes on the UI side. First, `TraceCanvas` should be built on `Canvas`, not a stack of `Shape` views — with ~270 points that's the difference between one `GraphicsContext.stroke` call and hundreds of view diffs every time the scrubber moves. **Where they agree:** Both Codex and Claude are building on the exact architecture locked in `ios_architecture_review` without re-litigating it — `Attempt`-only schema (no Gym/Route/Session), four attempt statuses, off-main SwiftData writes through a single `@MainActor` gateway, `beginBackgroundTask` plus cold-launch/foreground reconciliation as the sole recovery path, video-only `AVCaptureSession` with real interruption hooks, `PHPickerViewController` for import (no photo permission), `PrivacyInfo.xcprivacy` from day one, `@AppStorage`-backed `EntitlementService`, zero third-party dependencies, and the same pure-function test seams (normalization, gap-detection, confidence aggregation, determinism regression) with an explicit device-only lane for anything Simulator can't exercise. Both also independently landed on `NavigationStack` + `fullScreenCover` for Capture + `.sheet` for permissions, so navigation architecture isn't in dispute.

**Where they actually differ, and it matters for whoever starts coding:**

1. **A genuine bug in the locked spec that Claude caught and Codex hasn't responded to yet.** The prior phase's final output called `AnalysisQueue` both "an actor" and "an `@Observable` service" — those are mutually exclusive in Swift (`@Observable` doesn't apply to actors, and actor-isolated state can't publish synchronously to SwiftUI anyway). Claude's fix: keep `AnalysisQueue` a pure actor emitting an `AsyncStream<AnalysisProgress>`, and add a thin `@MainActor @Observable` mirror type (`AnalysisProgressStore`) that subscribes to it for SwiftUI binding. This is a real compile-time contradiction, not a style preference, and Codex needs to confirm it before it's safe to build against.

2. **Trace-data storage: one table or two.** Codex's interfaces-json defines `PoseTraceArtifact` as its own `PersistentModel` (a second SwiftData table with `@Attribute(.externalStorage)`), linked to `Attempt`. Claude argues this quietly reopens the "single-table `Attempt`-only" discipline the group fought hard to lock two phases ago — a 45-second clip's pose data is only tens of KB, trivially decoded synchronously on Review open, so Claude wants it as one `Data?` column of JSON-encoded samples directly on `Attempt`, no second model, no relationship. This is a real, stated disagreement, not just a drafting gap.

Everything else Claude added this round — the local `ChalklineCore` SPM package to make pure-function code physically unable to import Vision/SwiftData, `Canvas`-based (not stacked `Shape`) trace rendering with precomputed/cached segments, and pre-warmed haptic generators to kill first-call Taptic latency — are additive proposals nobody's pushed back on yet, but they also haven't been confirmed by Codex.

Mine is `@MainActor @Observable final class EntitlementService` — no unchecked escape hatch, the compiler actually enforces where it can be touched from, and since nothing about entitlement checks needs to happen off-main anyway (they're synchronous boolean reads gating UI), there's no cost to keeping it main-actor-bound. I'd drop the `@unchecked Sendable` version.

Second, smaller thing: you've got both a `ConfidenceScorer` protocol (in `polish_resilience`, which feels like the wrong lane for core scoring math) and I've got `ConfidenceAggregator` as a static-function enum (in `services_utilities`) — same concept, and we should pick one. I'd argue against the protocol here specifically: there's exactly one implementation of this, it's a pure function with no external dependencies to swap out or mock, and introducing a protocol for a single-implementation pure function is indirection that buys nothing — you can't meaningfully unit-test a protocol differently than you'd test the concrete function, since there's nothing to inject a fake behind. Keep it a plain static function in `services_utilities`, drop the protocol. Similarly, your `NormalizeRouteLabel` and `FindPreviousAttempt` free functions are the same ideas as my `RouteLabelNormalizer.normalize` and `ComparisonResolver.priorAttempt` — let's converge on the namespaced-enum style since it matches `ConfidenceAggregator`/`TraceGapDetector` and keeps all four pure functions discoverable under one naming convention instead of scattered top-level funcs.

### Round 3


**Codex (QA & Risk · Closer) — Round 3**

You’re right to keep pressing the open loop, and I’m going to close it hard. Claude’s thread from earlier is solid on the architecture, but the only remaining risk is not technical correctness — it’s build hygiene and deterministic guarantees. On the `ChalklineCore` package question, I’m siding with Codex’s rejection. A local SPM package buys almost nothing here and adds target setup friction (`project.pbxproj`, scheme membership, import hygiene), while we already have a stronger control: keep core value types and pure functions in a `Domain/` folder and enforce layer boundaries with compile-time checks/tests on imports. For an MVP, that’s the safer path and it keeps the project “single target + clear ownership” under pressure. If we can’t reliably build a package in one pass, we shouldn’t force architecture debt into the first slice.

On `EntitlementService`, I disagree with `@unchecked Sendable` and don’t think any “future-proof” argument supports it. This class only gates local behavior with a single boolean; any unchecked concurrency promise is upside down and exactly the kind of latent crash/memory-ordering risk this room has explicitly been avoiding. I want `@MainActor @Observable` with direct `@AppStorage("isPro")` persistence and computed capability properties. It’s boring, correct, and matches where checks are actually consumed (mostly UI/service boundary predicates).

I’m also keeping `ImportCoordinator`, `PoseAnalysisSerializer`, and `PoseTraceSummary` in. `ImportCoordinator` is not optional: import duration gating (>10 minutes) was locked as a must-have in app_features and is still not owned elsewhere if we drop it. If no one owns that behavior, it gets silently lost and every “can’t process long clip” failure becomes a generic import error. `PoseAnalysisSerializer` is required for the determinism contract; without a stable serializer, our determinism regression test is fake because `[String: JointPoint]` is unordered unless we define encoding order. So this serializer must explicitly canonicalize joints. `PoseTraceSummary` can be cheap and useful as a derived artifact (unit-test visibility + debug sanity), not a schema commitment. It can be returned from analysis and stored as `String`/`Double`-level outputs already in `Attempt`, but keeping the dedicated struct in services is cleaner for future checks and avoids reusing ad-hoc tuple logic.

I’m also rejecting any hidden state creep from this point: no fifth persisted status, no `Gym/Route/Session`, no photo permission branches, no StoreKit in this pass. `AnalysisQueue` stays actor-only and pure, `PersistenceGateway` remains the only `@MainActor` writer into SwiftData, and cold-launch/or-foreground reconciliation is mandatory before Home paints. That gets you resilience against suspend/kill and keeps the user-facing state model understandable.

So the concrete decision set for this phase is:
- no `ChalklineCore` package in this MVP,
- `EntitlementService` = `@MainActor @Observable final class` (no `@unchecked Sendable`),
- keep `ImportCoordinator`,
- keep `PoseAnalysisSerializer` (canonicalizing joint keys),
- keep `PoseTraceSummary`,
- all other open type contracts already converged stay as-is.

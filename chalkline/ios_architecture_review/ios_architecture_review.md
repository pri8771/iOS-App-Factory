Architecture: this is a single-window SwiftUI app with no tab bar (already locked) — `WindowGroup` rooted at Home, `Capture` presented as a `fullScreenCover` (not a push, since it's a distinct modal context with its own camera session lifecycle), `Processing` reached via `NavigationLink`/programmatic push from Home or from Capture-on-stop, `Review` reached via push from a Home card, and `PermissionSheet` as a `.sheet` from either Capture or Import failure. Everything else — Home's list, Review's per-attempt state — is `@Query` against SwiftData plus plain `@State` for UI-only concerns. This is a clean, small architecture; I don't think there's a real fight left here, but I want it stated as the decision so tech_specs doesn't reinvent it.

The thing I want to push hardest on, because it's the kind of bug that doesn't show up until a real user hits it: SwiftData `@Model` objects are not safely mutated off the main context, and the analysis pipeline is explicitly designed to run in the background while the user does something else. If `AnalysisQueue` runs Vision processing on a background task and then reaches into a `@Model Attempt` instance to write `poseSamples`/`confidence`/`status` from that background context, that's the textbook SwiftData threading crash. The fix has to be explicit in the architecture, not left to whoever writes `AnalysisQueue`: the queue computes results as plain (non-`@Model`) value types off-main, then hops back to the main actor to write them into the model via the main `ModelContext`, or uses a proper `ModelActor`-backed background context with its own fetch/save cycle. I want this named as a concrete, testable contract — "analysis output is always written to SwiftData on the main actor" — because if it's just implied, it's exactly the kind of thing that works fine in dev on a fast simulator and then randomly crashes on a device under thermal load, which is precisely where this app spends most of its life.

Second real risk: background execution. The design handoff already correctly established that `AnalysisQueue` must outlive the Processing *view*, and separately that a cold-launch reconciliation sweep handles the app being fully killed. But there's a gap in between those two cases that nobody's named yet — the user backgrounds the app (hits the home button, checks a text, doesn't force-quit) while analysis is running. Without an explicit `beginBackgroundTask`/`BGProcessingTask` request, iOS suspends the process within seconds of backgrounding, and that in-flight Vision pass just freezes mid-frame-sample rather than either completing or cleanly failing — and if it resumes when the app returns to foreground it may double-process or produce a partial result that doesn't match the "identical input produces identical output" determinism promise. I want a concrete decision here, not a "we'll figure it out": request a background task at the start of analysis with `UIApplication.shared.beginBackgroundTask`, give it a hard time budget matched to the 6fps pipeline's expected worst-case duration for a 45-second clip, and if the expiration handler fires before completion, abort the in-flight run cleanly and mark the attempt for cold-launch (or next-foreground) resubmission rather than let the OS silently freeze it mid-write. Third: the audio question needs to be enforced in code, not just asserted in the permissions plan.

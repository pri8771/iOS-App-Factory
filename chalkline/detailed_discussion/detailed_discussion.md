**Codex (Product Strategist · Visionary) — Round 1**

If we don’t make this product remarkable in the first 30 seconds, it will die as another logging app with a fancy overlay, so the direction I’d lock for this phase is “decision-grade feedback loop in the wall context,” not “feature accumulation.”

I want the primary value to be this: after an attempt, the app should give a concrete, verifiable correction signal fast enough to influence the next try, with a route-level memory that makes repeated tries feel cumulative. That means the shipped path should be “capture fast, analyze reliably, then tell me exactly where my attempt regressed or improved.” Anything that adds taps or pre-setup before recording undermines the whole promise, so route selection, gym assignment, notes, and tag polish must always be post-capture and skippable in that first pass.

The strongest implementation boundary is a two-track signal model: every result has a trace artifact, and score is only shown if the sample confidence model passes a fixed stability threshold across the clip. If it doesn’t pass, we do not degrade silently; we explicitly downgrade to trace-only and the UI shifts into an explanation mode that says “movement shape is available, confidence is too low for numeric score.” That keeps trust intact and preserves the promise under hard conditions.

The non-negotiable UX constraint I think must be preserved in every design decision is this: zero required pre-configuration before recording. A user should be able to start capture/import within two taps from app open. I’d keep the first screen basically a “capture now” surface with a framing guard, not a form.

I’d keep comparison deliberately minimal and concrete in v1: only same-route, latest-vs-prior-attempt narrative comparison, with no multi-attempt charts and no visual sync overlay yet. The comparison must degrade gracefully per pair, not globally; if either side lacks usable pose confidence, we show “same route, but no comparable segment confidence” and still surface what we can explain. That avoids fake precision and avoids broken UI states.

Measurable success criteria should be practical and local-only: 1) step-zero Vision spike against real gym footage succeeds or fails against an explicit threshold before UI build; 2) from capture-end to review-ready, the experience should usually be complete before next attempt on the same wall session (even if “usually” varies by device); 3) complete workflow support for empty/loading/error/success at every reachable state; 4) no user-facing data loss—attempt rows survive regardless of media pruning decisions; 5) a single VoiceOver interaction can report a full review outcome in plain language without opening any visual trace view.

Edge cases I want explicitly documented now: no body detected in frame, two people in frame, clipped clip with no route label ever assigned, very short clip (<1 second), long clip that never hits the auto-stop safety cap, imported clips already transcoded weirdly, missing camera/mic/photo permission, low storage, and abrupt app backgrounding during analysis. Also duplicate route labels and same-name routes across different gyms; for v1 this should be a visible “unresolved route ambiguity” state, not an automatic merge.

Privacy and data implications are straightforward but easy to get wrong: we should own video storage in-app (copy into sandbox), never just hold Photos URLs for primary playback unless explicitly imported and mirrored locally. Videos include biometric-ish movement and potentially sensitive body imagery, so explicit permission copy (camera/photos), transparent on-device processing statement, and no background upload path in v1 are mandatory assumptions. A hard cap plus compression policy must be deterministic and logged, with clear UI messaging when media is pruned from storage but record rows remain.

The biggest risk to this direction is overpromising with the confidence bar: low-light side angles and overhang framing can look fine in demo clips but still break down in real tries, and then the app will look unreliable exactly when the user needs it. The mitigation is to force the spike-first gate and an in-capture framing check that can block recording with an immediate “reposition” state. If the gate fails, the app must switch to trace-only and keep that visible in the review copy.

**Claude (QA & Risk · Skeptic) — Round 1**

Picking up where the next_steps_small phase left off — that gave us a locked slice, but locked scope isn't the same as locked behavior, and I want to pressure-test the gaps before this goes to design/architecture, because several of them are exactly the kind of thing that gets silently decided by whoever writes the code first unless we nail it down now.

Start with permissions and interruption, because nobody's touched this yet and it's not optional. I'd also push to explicitly not request microphone permission at all — pose analysis doesn't need audio, and skipping audio capture means one less permission prompt and one less thing to explain in the privacy nutrition label. Separately: what happens when a phone call or notification interrupts an active manual recording? `AVCaptureSession` gets interrupted, and the app needs a defined answer — does the partial clip get discarded, or saved-and-analyzed-as-is? I'd say discard and return to the capture entry point with a "recording interrupted" message, because a truncated attempt with no clear endpoint is worse than making the user retry.

Second, the free-text route matching for comparison has real ambiguity hiding under a decision that sounds settled. "If a second attempt shares the same free-text label" — shares it how? Exact byte match will produce silent, confusing non-matches for "Blue crimpy," "blue crimpy," and "Blue Crimpy " with a trailing space, and the user has no way to know why comparison didn't trigger; from their side it just looks broken. Minimum viable fix is trim-and-lowercase normalization before comparing, and that needs to be named now, not discovered during build. There's also an unaddressed case: what happens on the third attempt with the same label? The decision as written only describes two attempts stacked — does attempt three show against attempt two, against attempt one, or against all priors? I'd argue most-recent-prior-only, to keep this genuinely lightweight per the cut-list intent, but that has to be a stated decision, not left for whoever's coding it to guess. And a real correctness gap: since gym/session scoping was explicitly cut from this slice, a free-text label like "V4 overhang" can collide across two different gyms and produce a comparison between two attempts on completely different physical routes that happen to share a name. That's a known, acceptable-for-now limitation, but it needs to be written down as a limitation, not quietly absorbed as if the matching is semantically safe.

Third — mid-clip tracking loss. The pre-record framing guard only checks that a body is detected before recording starts. It says nothing about what happens when the climber moves up and out of frame mid-attempt, which is extremely common on anything taller than a low traverse, or gets occluded by a volume or another climber walking past the camera. If the trace overlay just goes blank or freezes for those frames with no explanation, that reads as a bug, not as expected behavior. The confidence indicator and the trace rendering both need to treat per-frame tracking loss as a real, visible state — a gap in the trace, not an interpolated guess smoothing over missing data as if it were continuous movement. I'd also flag that the pre-record guard itself has a false-positive failure mode: in a busy gym, it can detect a bystander in the background and pass the "body in frame" check while the actual climber is still out of frame or blocked by a hold — the guard proves "a body," not "the climbing body," and that gap should be named rather than assumed away.

Fourth, confidence itself is currently a phrase, not a formula, and that's a bigger risk than it looks like on paper. Vision gives per-joint, per-frame confidence; nobody's said whether the review screen's single confidence indicator is an average across all joints, a minimum, or a weighted score favoring hips/knees/shoulders over face/wrist landmarks that matter less for climbing efficiency. Different aggregation choices will produce very different-looking confidence numbers for identical footage, which directly determines whether the fallback trigger fires. This has to be pinned down before the step-zero spike runs, or the spike's pass/fail verdict is only as meaningful as an undocumented formula nobody can audit later. On that note — the step-zero spike's human-judgment check ("does the trace roughly match where the climber says it broke down") is inherently subjective, and if it's just a verbal yes/no with no artifact, we have no way to re-verify it or defend the decision later if the fallback question gets revisited. I want the actual 8-10 clips, the per-clip confidence numbers, and the human verdicts recorded as a real artifact, not asserted from memory.

Fifth, import handling has no defined boundary. The 45-second auto-stop applies to live recording, but nothing says what happens if someone imports a 12-minute session recording or a 4K/60fps clip with rotation metadata from some other app. Full-length processing of an arbitrarily long import blows the "quick review right after the attempt" promise and produces a nonsensical single line of interpretive text for ten minutes of footage. I'd want imports capped or truncated to a similar duration ceiling, with an explicit "clip too long, trim to your attempt" state, rather than silently processing whatever gets handed to it.

Sixth, storage and performance failure modes that are easy to skip past: what happens when local disk is nearly full and a write fails mid-recording or mid-save — that needs an explicit error state, not a swallowed exception. And because retention-cap enforcement was correctly deferred out of this slice, storage grows unbounded during testing and early use; that's an acceptable gap for now, but it should be logged as a known, temporary gap rather than something that quietly becomes permanent because nobody circles back. Related: on older or thermally-throttled devices, Vision processing at 6fps could take meaningfully longer than expected, especially if the user fires off a second capture while the first is still analyzing — the app needs to allow that (queued or concurrent analysis, not a block), and the processing state needs a distinct "this is taking longer than usual" variant so it doesn't read as frozen.

Last, privacy deserves a harder look than "it's all on-device so we're fine." Body-pose video is sensitive by nature, and gyms are social spaces — climbers routinely end up filming other people in the background, meaning stored clips can contain third-party biometric-adjacent data with zero consent mechanism. I'm not proposing we build consent tooling for this slice, but it should be named explicitly as a limitation the user is trusting the app (and themselves) to handle responsibly, not something the privacy story silently glosses over. Concretely, this phase needs to lock: camera-only usage string language that's accurate (no mic), photo-library usage string for import, and an explicit statement that video and derived pose data never leave the device in this slice — no analytics, no crash-report attachments containing frame data.

What I want preserved out of this phase, concretely: normalized (trim/lowercase) free-text label matching with a named policy for 3+ attempts sharing a label; an explicit statement that cross-gym label collisions are a known MVP limitation, not a bug; a defined confidence-aggregation approach fixed before the step-zero spike runs, with the spike's clips/numbers/verdicts saved as an auditable artifact; visible, non-interpolated handling of mid-clip tracking loss in both the trace and the confidence indicator; an import duration ceiling with a real rejection/trim state; explicit permission-denied and recording-interrupted states for camera/import; a disk-full/save-failure error state; and the privacy limitation about bystanders and on-device-only data stated as fact in the final output, not left implicit.

- Comparison stays same-route, latest-vs-most-recent-prior-attempt only (not all priors, not a chart) — with route matched via a trim-and-lowercase-normalized free-text label. Comparison degrades **per pair**: if either attempt lacks usable confidence, show an explicit "no comparable segment confidence" message rather than a fake or partial stat.
- Cross-gym label collisions (e.g., "V4 overhang" logged at two different gyms matching each other) are a named, accepted MVP limitation, not a bug to silently paper over — it must be stated as fact in build documentation.
- Subscription/entitlement enforcement stays deferred (stubbed data-model flag only), consistent with next_steps_small; this phase doesn't reopen that.
- No microphone permission requested at all — pose analysis is video-only, so audio capture and its permission prompt are dropped entirely.

**Edge cases (to design/architecture explicitly, not discover mid-build)**
- No body detected in frame (pre-record guard blocks/warns); guard also has a false-positive failure mode (detects a bystander, not the actual climbing body) — name this as a known limitation of the guard, not a solved problem.
- Two people in frame → treated as low-confidence/ambiguous, no disambiguation attempted.
- Mid-clip tracking loss (climber moves out of frame, gets occluded by a hold or another climber) must render as a **visible gap** in the trace and confidence indicator — never silently interpolated as if motion were continuous.
- Very short clips (<1s), clips that never hit the auto-stop cap, oddly-transcoded or rotated imports.
- Imports need a duration ceiling with an explicit "clip too long, trim to your attempt" rejection/trim state — a 12-minute import must not be processed whole.
- Camera/photo-library permission denied → explicit state with a path to Settings, requested at point of use, not on launch.
- Active recording interrupted (call, notification) → discard the partial clip and return to the capture entry point with a clear message, rather than saving/analyzing a truncated attempt.
- Low storage / disk write failure during recording or save → explicit error state, not a swallowed exception.
- Concurrent/queued analysis (user starts a second capture while the first is still processing) must be allowed, not blocked, with a distinct "taking longer than usual" processing state for slower/thermally-throttled devices.
- Abrupt app backgrounding during analysis.
- Duplicate/ambiguous route labels beyond the normalization fix (visible "unresolved" state rather than silent auto-merge for anything normalization can't resolve).

**Data and privacy implications**
- All video and derived pose data stay on-device in this slice: no background upload, no analytics or crash-report attachments containing frame data.
- App owns storage (copies into sandbox) rather than holding bare Photos references for anything used in primary playback.
- Permission strings must be accurate and minimal: camera usage string only describes on-device pose analysis (no mention of audio, since mic is never requested); photo-library string covers import only.
- Bystander privacy is a named, accepted limitation: gym footage will often capture other people in the background with no consent mechanism, and this is disclosed as a real limitation the user is trusting the app (and themselves) to handle responsibly — not built-around in this slice.
- Storage grows unbounded during this slice since retention-cap enforcement is deferred; that's acceptable for now but must be logged as a known, temporary gap so it doesn't quietly become permanent.

**Risk register**
- Confidence bar looking fine on demo footage but failing on real attempts (sideways framing, overhang, poor light) — mitigated by the pre-authorized fallback (trace-only) and the in-capture framing/body-detection guard, but the guard's "a body" vs. "the climbing body" gap is a residual, named limitation, not a solved risk.
- Accessibility treated as decorative rather than core — mitigated by requiring a full spoken/textual result model (e.g., "attempt confidence high, breakdown near 62% duration: hips drifted left over the crux") producible from a single VoiceOver interaction, not just image alt-labels.
- Undocumented confidence-aggregation formula makes the step-zero spike's verdict unauditable — mitigated by fixing the formula before the spike runs and preserving the spike's raw data as an artifact.
- Ambiguous free-text matching silently breaking comparison for the user with no visible cause — mitigated by normalization (trim/lowercase) and by explicitly naming cross-gym collisions as a limitation rather than pretending the matching is semantically safe.
- Unbounded storage growth during this slice — accepted as a temporary, logged gap, not a silent permanent one.
- Bystander biometric-adjacent data with no consent flow — named as an explicit, accepted MVP limitation rather than solved or ignored.

**Final assumptions**
- This is a movement-technical niche app, not a habit-game/streak-first app; retention comes from the diagnostic value of the trace and comparison, not gamification.
- Entitlement/paywall enforcement is out of scope for this phase's decisions (already deferred by next_steps_small); this phase only confirms it stays stubbed.
- The confidence-aggregation formula and the spike's artifact-recording requirement are treated as prerequisites to running step zero, not optional refinements — the design/architecture phase inherits them as fixed inputs, not open questions.
- Cross-gym label collision and bystander-privacy limitations are accepted, documented MVP trade-offs, not defects to fix before ship.

**Screen inventory (unchanged, 10 screens):** First-launch explainer, Home surface, Serve Time entry, Plan Builder (with embedded live merge preview), Add/Edit Step sheet, Merged Timeline Preview (folded into Plan Builder's states), Start Confirmation sheet, Live Countdown/Execution, Completion, Settings/Paywall.

**Primary flow (unchanged):** first launch → explainer → serve time → Plan Builder (add dish → add steps → live merge updates → repeat) → Start Confirmation → Live Countdown → Completion → Home surface.

**Full state model, screen by screen:**
- **First-launch explainer:** success-only static copy, shown exactly once (local flag, not account-based), no re-open path in v1.
- **Home surface:** empty (zero plans/templates, one big "start a new plan" CTA), loading (skeleton if the SwiftData fetch isn't instant — never a flash of empty-then-populated), success (active plan + past/template plans + grayed lapsed plans on paid tier), and a genuine error state for a failed/corrupted local store ("we couldn't load your plans" with a path to start fresh) rather than assuming that can't happen.
- **Serve Time entry:** no meaningful empty/loading state; error handling is a real design call — past times are disabled/unselectable in the picker, and a serve time more than ~24 hours out gets a soft, dismissible confirmation rather than a hard block.
- **Add/Edit Step sheet:** no loading state (local, fast); error is inline field validation (empty label, zero/negative duration) that blocks dismiss without a modal.
- **Merged Timeline Preview:** not independently navigable, so its states live inside Plan Builder — empty is zero steps across all dishes ("add your first step to see your timeline"), no separate error state beyond Plan Builder's row-level ones.
- **Start Confirmation sheet:** this one got real scrutiny per the gate's request — it now has three actual states: loading ("starting your plan…" while notifications are scheduled and the Live Activity is requested — real async system work, not instant), success (straight into Live Countdown), and a real failure state where a scheduling failure rolls the plan back to unstarted with a plain explanation and a retry path, rather than silently starting a plan with missing alerts.
- **Completion:** deliberately success-only — reachable only by finishing every step, no data dependency, no manufactured error state just to check a box.
- **Settings/Paywall:** the one the gate cared about most, now fully specified — loading (purchase/restore in flight, button disabled, spinner, no double-tap), success (entitlement updates live in the same screen, no restart needed), and two distinct error states kept apart on purpose: a retryable purchase failure (declined/cancelled/network) versus a "we didn't find a previous purchase to restore" outcome, which is worded as an expected empty result rather than an error so it doesn't read as a false failure to someone who never subscribed.

Everything else from round one — the 10-screen inventory, the vertical-timeline-anchored-at-serve-time visual direction (deliberately avoiding the circular-dial timer look for uniqueness reasons), and the accessibility notes (Dynamic Type wrapping, chronological VoiceOver order, non-color warning signals, one-handed "running late" control) — stands unchanged and unchallenged. Same story as all nine phases before it — Codex and Gemini couldn't join (still in cooldown / no CLI auth), so it's just Claude's single pass. But it's a real, complete answer to what an architecture review needs to produce, and — more importantly — it's the first phase to actually stop punting on the Countertop/CueKeeper uniqueness question instead of tabling it an eleventh time. Nothing here conflicts with any of the nine locked phases before it; it operationalizes them into concrete engineering decisions. Calling this converged.

**Uniqueness check — closed by decision, not verification.** No agent in this entire chain has had working web access, so waiting for a future round to "actually check" was never realistic. This is now resolved as a permanent mitigation rather than an open item: defensive positioning (no "timer/countdown" framing anywhere in name/marketing/screenshots, no circular-dial visual language, "backward-planning from a serve time" as the lead hook) stands in place of a literal competitive audit. If a real collision surfaces post-launch, it's a marketing fix, not an architecture blocker. This does not reappear in tech_specs.

**SwiftUI architecture:** single `NavigationStack`, `@Observable` app-level model — already locked. New hard rule added: Plan/Dish/Step persist **only** entry data (serve time, dish/step definitions, start timestamp, done-early markers) — no derived field (e.g. `isPastDue`, `currentStepID`) is ever stored. Everything else is a pure recomputation of `(steps, now)`, always. This is what prevents the in-app screen, the Live Activity, and a force-quit resume from ever disagreeing with each other.

**Apple frameworks:** SwiftData (persistence), UserNotifications (local alerts), ActivityKit (Live Activity), StoreKit 2 (`Transaction`/`Product.products(for:)`/transaction listener — not the old SKPaymentQueue API). BackgroundTasks/`BGAppRefreshTask` is explicitly **ruled out** — no network, no off-schedule work exists that needs it, and reaching for it "to be safe" would just add a new bug surface for zero payoff.

**Permissions/privacy plan:** notification + Live Activity permission requests fire at Start Confirmation, not first launch — asking before a user has built a plan means denying with zero context and no way to re-prompt. Entitlement checks (`Transaction.currentEntitlements`) are cached at launch/foreground when connectivity exists and gate features from the cache when offline — never a hard network-dependent block, since the whole promise is working in a basement with no signal. Privacy nutrition label should land on "Data Not Collected" across the board — no accounts, no network calls outside StoreKit/App Store system frameworks — verified against the finished code (grep for network calls), not just asserted from the design doc.

**Dependency/license decision:** zero third-party dependencies. Every requirement is covered by first-party frameworks, and offline-first/force-quit-survival reliability only gets weaker by adding an external persistence or scheduling library. No permissive-license question even arises.

**Testing implications:** the scheduling math (tie-break, DST, running-late cascade, past-due detection) has to live in a plain, framework-free `SchedulingEngine` module — no `UNUserNotificationCenter`/`ActivityKit` calls inside it — so those correctness-critical paths run as fast deterministic unit tests, not slow simulator-only UI tests. A thin `NotificationCoordinator`/`LiveActivityCoordinator` wraps the system APIs around it. Windowed notification rescheduling gets a concrete trigger list (Start, every `scenePhase → .active`, every notification delivery/response, every running-late shift) plus a defensive diff-against-actual-pending-requests check at each trigger — this is what actually wires the already-designed "notification silently failed" error state to a real signal instead of leaving it unreachable. StoreKit Testing (`.storekit` config + `SKTestSession`) gets wired into the scheme from day one so purchase/restore/expiry are testable without sandbox dependency.

**ML/AR:** explicitly declined, not deferred. Nothing here would add genuine value, and forcing it (camera ingredient recognition, an AR oven-timeline gimmick) would violate scope boundaries locked nine phases ago.

This closes the phase — every required output (SwiftUI architecture, framework choices, persistence plan, permissions/privacy plan, dependency decision, testing implications, ML/AR justification) has a concrete answer, and the app can be built without vague architecture gaps.

## Final Output

**Architecture overview:** Single SwiftUI app target plus one WidgetKit extension for the Live Activity — no server, no shared framework, no App Group. Four lanes with a strict one-way dependency chain: `data_domain` (SwiftData models + framework-free `SchedulingEngine`) depends on nothing; `services_utilities` (NotificationCoordinator, LiveActivityCoordinator, EntitlementStore, DeepLinkRouter) depends only on `data_domain`'s value types and never reaches back into UI; `primary_ui` (AppModel + the 10 screens) depends on both and is the only lane allowed to own a `ModelContext` or call a coordinator; `polish_resilience` (banners, accessibility helpers) is consumed by `primary_ui` with no dependencies of its own. The organizing principle: pre-Start, a Plan is relative data recomputed live via `previewTimeline`; at Start, that computation runs exactly once via `commitSchedule` and gets written back as real persisted `Step` fields — "no derived state persisted" and "schedule fixed at Start" are the same rule at two points in time, not two rules in tension. The widget extension links against nothing but the one shared `BackTimerActivityAttributes.swift` file (dual target membership, no App Group) and contains zero business logic — a pure renderer over `ContentState`.

**File/module layout:** `BackTimer/` app target split into `App/` (entry point, AppModel, scenePhase hook), `Domain/Models/` + `Domain/Persistence/` (SwiftData, `PersistenceController`), `Scheduling/` (framework-free `SchedulingEngine` + its value types — deliberately a separate folder from `Domain/` so nobody accidentally drops a SwiftData import into it), `Services/` (NotificationCoordinator, LiveActivityCoordinator, EntitlementStore, DeepLinkRouter, each paired with its protocol and error enum), `Features/` (one folder per screen matching the locked 10-screen inventory), `Shared/` (banners, design tokens, accessibility helpers). Separate `BackTimerWidgetExtension/` target (Live Activity renderer + the one shared attributes file). Separate `BackTimerTests/` (pure unit tests) and `BackTimerUITests/` (XCUITest, including StoreKit Testing) targets.

**Persistence strategy:** SwiftData via a single local `ModelContainer` (`PersistenceController`, no App Group, `inMemory:` init for tests/previews), storing only entry data (serve time, dish/step definitions, start timestamp, done-early markers, scheduled start/end times written once at Start) — never derived fields.

**Testing strategy:** `SchedulingEngineTests` — deterministic, fixture-based, zero system-API/simulator dependency, covering tie-break, DST spring-forward/fall-back, midnight-crossing, past-due-at-creation, and the running-late cascade including in-flight passive steps. `AppModelTests` — orchestration logic (start/rollback-on-scheduling-failure) tested against fake `NotificationScheduling`/`LiveActivityManaging`/`EntitlementProviding` conformances. `BackTimerUITests` — deliberately small: one end-to-end happy-path flow test plus a StoreKit Testing suite (`.storekit` config + `SKTestSession`) for purchase/restore/expiry, since the hard correctness logic is already covered more cheaply at the unit level.

**Data models + interfaces-json contract:** unchanged from round 1 — `Plan`/`Dish`/`Step` as SwiftData models, `SchedulingEngine` as the framework-free pure-function core, `TimelineItem`/`MergedTimeline`/`LiveCountdownState` as the shared value types flowing into both UI and `ActivityAttributes.ContentState`, plus the full services/UI/polish layer. This phase is done.

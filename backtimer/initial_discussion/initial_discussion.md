**Claude (Design Lead · Systems Thinker) — Round 1**

Here's my independent read on this, coming at it from the screens-and-flows side rather than re-litigating what's already settled.

The product promise I'd lock in is: "Tell backtimer when you want to serve, list what you're cooking, and it hands you one countdown that tells you what to start next — for every dish, at once." I'm deliberately avoiding words like "coordinate," "schedule," or "manage," because those imply the app is doing resource-awareness work it explicitly isn't (no oven/burner contention modeling, per the standing decision). The verb has to be "merge into a countdown," and I'd want that phrase or something functionally identical to survive verbatim into onboarding copy, not just architecture docs.

Primary user and scenario: a host, the night before or morning of a multi-dish meal, sits down once with no time pressure to enter dishes. Then, hours later, under real time pressure with dirty hands, they need the app to be glanceable rather than tappable. That's really two different users inhabiting the same person at two different moments — a calm data-entry mode and a stressed execution mode — and I think the biggest design risk nobody's named yet is that these two modes want contradictory things from the UI. Entry mode wants density and speed (add five dishes, ten steps, fast). Execution mode wants almost the opposite: one huge readable "next thing" and nothing else competing for attention. If we design one screen that tries to serve both, it'll serve neither. I want two visually and structurally distinct surfaces: a Plan Builder (list-based, checklist-feeling, keyboard-friendly) and a Live Countdown (nearly full-screen single current step, next-up preview, big "push back" affordance) that shares almost no chrome with the builder.

Core loop, concretely, screen by screen: (1) empty state — "What time do you want to serve?" as the very first input, before any dish exists, because the whole plan is anchored to that one number and asking for it last would mean re-deriving every step time retroactively; (2) add a dish — name plus a flat list of steps, each step just a label, a duration, and an active/passive toggle, no ingredient lists, no photos, nothing recipe-app-like; (3) the moment one dish has at least one step, show the merged countdown preview inline — this is the "feel the value early" requirement from the prior phase, and I'd treat it as a hard UI rule, not a nice-to-have: the merge view must render after dish #1, not gate behind "finish entering all dishes then tap Generate"; (4) repeat adding dishes, watching the merged list re-sort live as each new step lands somewhere in the timeline; (5) start — this is the state transition that matters most and needs its own explicit design: once started, is the plan still editable? I'd say no, or only softly — once you've started, editing step durations retroactively means every notification already scheduled needs to be cancelled and rescheduled, which is a real technical operation, not just a UI toggle, and if we allow free editing mid-cook we need to be honest that we're rebuilding the whole notification schedule on every keystroke; (6) live execution — current step big, next step visible but visibly secondary, a persistent "running late" button that shifts everything remaining, and a way to manually mark a step done early or acknowledge it without waiting for the timer; (7) completion — a genuine done state, not just the countdown hitting zero, because "everything landed" is the emotional payoff the whole app promised and it deserves an actual screen, however small.

The scope boundary I want written down precisely, because I think it's underspecified even in the prior decisions: "manual entry only" is fine, but does the app let you reuse a dish across plans, or does every dish get typed in fresh every time? If templates/reuse are paid-tier (per the monetization decision), then even the free-tier data model needs a Dish as a first-class reusable object from day one — retrofitting "save this dish for later" onto a data model that only ever modeled "dish belongs to this one plan" is exactly the kind of thing that's cheap now and expensive after ship. I'd insist the persistence layer treat Dish and Plan as separate entities immediately, even if the free-tier UI never exposes a "my saved dishes" library.

Edge case I don't see addressed anywhere yet: what does the merged timeline look like when a step's start time is in the past relative to now — i.e., the user builds the plan mid-afternoon for a 6pm serve, but one dish's passive step (like "marinate 4 hours") needed to have started at 1pm? Does plan creation just refuse to save ("this plan doesn't fit," per the existing decision) or does it allow it but visually flag that specific step as already-late-before-you've-even-started? I think it has to be the latter with a loud warning, because refusing to save a plan where one dish is already unrealistic would block the user from using the app for the other four dishes that are fine — an all-or-nothing validation error is too blunt for a multi-dish tool.

Measurable success criteria I'd propose, stated as things a build can actually be checked against: a first-time user can go from empty state to seeing a merged countdown preview in under two minutes with two dishes and four steps; a plan survives force-quit and device restart with zero data loss and notifications still scheduled correctly; the "running late" shift affordance is discoverable and usable one-handed during execution mode without needing to visit a settings screen; and every one of the four required UI states (empty/loading/success/error) exists for both the Plan Builder and Live Countdown surfaces specifically, not just "the app" abstractly — I'd call out Live Countdown's error state in particular (what does it look like if a scheduled notification silently failed, or if the app couldn't get background execution time?) since that's the state most half-built apps skip entirely.

Reading back through what the coordinator carried forward after my round 1 pass, I need to flag two real problems in that merged backlog before we call this done, because both of them are exactly the kind of thing "acceptance checks: no ownerless task, dependencies acyclic, build workers can start immediately" is supposed to catch, and a duplicate or overlapping file assignment is worse than an ownerless one — it looks resolved but two build workers will write conflicting content to the same file and nobody notices until a merge conflict blows up mid-build.

First, the coordinator's addition literally created the same task twice: T-007 ("Establish branch layout and CI-less merge checklist for the repo") and T-026 ("CONTRIBUTING.md: merge order, branch names, and communication protocol") both own the exact same file, `BackTimer/CONTRIBUTING.md`, both depend on T-006, and both say almost the same three things — lane branch names, the pbxproj-conflict rule, and the contract-adjustment sign-off protocol. That's not two tasks, that's one task that got pasted in twice under different IDs, and if we ship the backlog as-is, whichever build worker picks up primary_ui's queue either does the same work twice or, worse, two different people grab T-007 and T-026 independently and now there are two incompatible drafts of the same file racing to be merged. I'm collapsing these into a single T-007 and dropping T-026 entirely — the content was good, there's just no reason for it to exist as two IDs.

Second, and this one's a real forbidden-zone violation, not just a paperwork duplicate: T-024 is owned by `polish_resilience` but its files are `StartConfirmationView.swift`, `LiveCountdownView.swift`, and `SettingsPaywallView.swift` — those are `Features/` screens, which I explicitly called out in my round 1 pass as `primary_ui`'s exclusive territory, and the coordinator's own summary of my position agreed with that split. Nobody argued for an exception, it just slipped through when the milestone-5 verification tasks got assembled. If `polish_resilience` "owns" a task that edits three files it has no other claim to, that's precisely the kind of overlapping ownership the phase rules say to avoid except for integrator-owned files, and this isn't an integrator file, it's three ordinary feature screens. The fix is simple: reassign T-024's owner_lane to `primary_ui`, since that's who actually owns every file it touches and who already built those three screens in T-018/T-019/T-021 — the "polish" flavor of the task (verifying the banners `polish_resilience` designed) survives just fine as an acceptance criterion, it doesn't need to be a different lane's task. General rule I'd bake in going forward: when a verification task's file list spans lanes, it gets filed under whichever lane owns the majority of those files, full stop — that's a cleaner rule than trying to invent a fifth "verification lane" that doesn't map to any real code ownership.

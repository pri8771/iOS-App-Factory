**Codex (Design Lead · Skeptic) — Round 1**

The market is probably small-but-serious operators who live in the gap between paper run sheets and expensive show-control systems: wedding planners running ceremony/reception transitions, DJs coordinating intros and speeches, MCs keeping program order straight, stage managers for small venues, church/event producers, and freelance producers who are often alone with an iPhone in one hand. The use context is hostile to fussy UI. They are standing, moving, distracted, in bad lighting, often one-handed, with people interrupting them. That means the real product is not “event planning,” it is interruption recovery. If the app is good, the operator can lose the thread for ten seconds, look down once, and instantly know what is happening now, what is next, whether they are early or late, and what backup move exists.

The strongest comparison set is not direct app-category competitors so much as behavior patterns users already trust under pressure. Notes, Apple Reminders, Google Docs/PDF run sheets, spreadsheets, and printed timelines are the default incumbent. On the higher end, there are stage/show-cue tools and event-production platforms, but most of those are either too complex, too team-oriented, or not iPhone-native for a solo operator. The pattern worth borrowing is not “project management,” it is “mission control”: a dominant current item, unmistakable next step, drift visibility, and zero-hesitation controls. The app should feel closer to a workout live tracker, turn-by-turn navigation, or a camera capture flow than to a checklist app. If it starts looking like a database with rows, it has already lost.

The main iPhone-native opportunity is lock-screen and background resilience, but only if it stays subordinate to the core console. Live Activities make sense because this user may pocket the phone and glance at the lock screen between cues. Time-sensitive notifications are useful because silent failure under Focus would destroy trust. Haptics matter because the operator may not be staring at the screen. Widgets are only worthwhile if they support “event starts in X” or “currently running event” at a glance; otherwise they are ornamental. App Intents are useful for getting into a specific event fast, but only with confirmation because accidental launch is a real failure mode. The most native-feeling version of this app is one where entering live mode feels like entering a temporary operational state, not just opening another screen.

My biggest concern is that the product can easily fool itself into solving the planning problem instead of the live problem. Timeline building, templates, role assignment, and notes are all necessary, but they are dangerously expandable. Users already have tools for drafting a schedule. They will only switch if live mode is materially safer and faster. So the first build should optimize around a single-screen running console with four dominant actions: advance, hold, skip, and reveal contingency/jump to backup. The builder should exist to feed that console, not as a feature playground. If the first build spends too much effort on rich editing and not enough on the running state model, it will be polished but non-essential.

There are several assumptions that need to stay labeled as assumptions. One is that solo operators are willing to author a run-of-show on iPhone rather than on Mac/iPad first. That may be only partially true, so the phone builder needs to be “good enough fast,” not exhaustive. Another is that users will trust a subscription for a niche tool; that only works if the free tier proves value in a real event quickly. Another is that “contingency plan” in v1 can stay simple as notes plus a designated backup cue. That is likely enough for weddings and simple events, but it may be too thin for more theatrical operators.

The biggest risks I see are, first, state ambiguity in live mode. If the user cannot instantly tell whether a cue is upcoming, active, delayed, or skipped, the app becomes dangerous. Second, the time model can betray the product if it feels “fake.” A running timer that looks live but breaks on backgrounding will be worse than paper. Third, the UI can get seduced by elegance when this category needs blunt clarity. Small typography, subtle color coding, hidden menus, swipe-only actions, or gesture cleverness are all liabilities here. Fourth, there is a category risk that this feels too similar to a generic checklist/timeline tool unless the live console is obviously its own thing. Fifth, the premium gates can backfire badly if they are encountered anywhere near run start or mid-event.

For the first build, I would make the app revolve around three states, not many screens: setup, build, run. Setup is onboarding, sample event, and paywall boundaries. Build is a practical sequence editor with cue title, scheduled time or relative offset, role, notes, contingency note, and optional backup target. Run is the product: huge current cue, clear next cue, schedule drift, elapsed/remaining context, last action visibility, and recovery controls that do not hide under sheets. I would also preserve one design rule very aggressively: the default live screen should answer five questions without scrolling or mode switching. What is happening now? What is next? Are we on time? What can I do immediately? What is my backup if this cue breaks?

The detail the next phase must preserve is that this app wins on compressed cognition, not feature breadth. It should be designed for the moment when the operator is interrupted, slightly behind, and cannot think carefully. That means the research implication is not “add more event-management capability.” It is “design the live console as a trust instrument.” If the first build cannot make advance, hold, skip, and contingency access feel reflexive, then it is still competing with a PDF and probably losing.

**Claude (QA & Risk · Pragmatist) — Round 1**

Let me think through this as someone who's going to have to poke holes in whatever gets built, not just describe the market in the abstract.

The audience here is narrower and more specific than "event planners" — it's the person physically holding the phone during the live moment: a wedding planner standing at the back of a reception, a stage manager in a DJ booth or backstage corridor, an MC pacing a corporate keynote, a one-person AV operator running a school recital. That person is usually standing, often in low light or direct sun, frequently one-handed because their other hand is holding a radio, a clipboard, or literally holding a door. They are not sitting at a desk reviewing a plan — they are executing one under time pressure with adrenaline in the mix. That context should dictate almost everything about the first build: large touch targets, absurdly high contrast, no reliance on subtle color-only signaling (someone glancing at a phone in bright sun will miss a yellow-vs-orange distinction), and zero reliance on precise taps or swipes, because stress degrades fine motor control. I'd treat "can this be operated with a thumb, at a glance, without reading a full sentence" as the actual design spec, not "looks premium in a screenshot."

On comparable patterns: the honest competitor set isn't other apps, it's Notes, a printed run sheet, a spreadsheet on a phone, and stage-manager tools like Stage Timer / ShowCue-style apps or DJ set-list apps. Those printed/PDF artifacts win today for one boring reason — they never crash, never lag, never require you to find the right screen, and you can glance at any part of the sheet instantly because paper has no navigation state. Any digital tool that requires "tap into event, tap into cue list, tap into cue detail" to see the current cue has already lost to paper. The apps in this rough space that do work (timer/stopwatch-for-events tools) succeed by being almost embarrassingly simple: one big number, one or two buttons, nothing else fighting for attention. That's the pattern to learn from, not "how do I fit templates, roles, and contingency notes onto one screen" — the live console has to earn its complexity budget down to nearly zero, and everything else (building the run-of-show, assigning roles, templates) can be as rich as it needs to be because that happens beforehand, calmly, not live.

Platform-specific opportunities I'd actually bet on for v1: Live Activity on the Lock Screen / Dynamic Island is the single highest-leverage native feature here, because it means the operator doesn't even have to unlock and open the app to see "current cue, time remaining, next cue" — that's a genuine capability paper can't touch, glanceable from a locked phone in a pocket or on a stand. Haptics matter more than they'd matter in almost any other app category, because a distinct haptic pattern for "cue is running long" or "30 seconds to next cue" can be felt without looking at the phone at all — that's a real edge over any paper-based approach. Notifications with time-sensitive interruption level for "next cue in 2 minutes" are the second-highest-leverage thing. Widgets and App Intents are real but secondary — nice for pre-event countdown visibility and quick-launch, not central to the live-execution promise.

Now the risk list, and I want to be blunt about these because they're the ones that quietly sink a build like this if nobody names them upfront. First, backgrounding and Live Activity reliability is the single biggest technical risk in the whole app — iOS Live Activities have real constraints (8-hour budget, update frequency limits, they can be dismissed by the user or the system), and if the countdown showing on the Lock Screen silently goes stale or gets evicted mid-event, that's worse than not having one at all, because the operator will trust a lie. That needs an explicit fallback: the in-app live console must always be the source of truth and must always self-correct from absolute timestamps the instant it's opened, regardless of what the Live Activity was showing. Second, the free-vs-premium boundary risk: "free for simple single-event runs" is a real product tension against "must feel trustworthy" — if a wedding planner is mid-build on their one free event and discovers a template or contingency plan is gated, that's a bad moment to hit a wall, so the paywall UX itself needs to be honest and upfront (show what's free before they invest 20 minutes building a run sheet), not a bait-and-switch. Third, "role assignments" is scope creep risk in a single-operator app — if role is just a text tag on a cue ("who's responsible") that's fine and cheap, but if it starts implying who-sees-what or per-role views, that quietly reintroduces multi-device coordination through the back door, which the contract explicitly forbids. I'd flag that as something the design phase needs to nail down concretely: role is a label for accountability and print/reference purposes only, never a permissions or view-filtering mechanism.

Fourth risk, more mundane but real: cue lists for actual events (weddings routinely have 20-40 cues once you count vendor arrivals, sound checks, speeches, cake cutting, etc.) mean the live console needs a fast way to jump to an arbitrary point, not just linear next/previous — if the ceremony runs 15 minutes over and three cues get skipped, "advance" 15 times is a failure of the product's own thesis. That's really a corollary of the contingency/recovery requirement already in the contract, but I want it named specifically against list length, because a demo built with a 5-cue toy event will hide this problem completely and it won't surface until someone tests with a real 30-cue wedding timeline.

Fifth, and this is a pure testing risk rather than a design risk: because so much of this app's value proposition is background/foreground timing correctness and offline behavior, the "final review" for this app cannot be a simulator walkthrough. It needs an actual device pass — lock the phone mid-cue, wait, unlock, confirm state; toggle airplane mode and run all three core workflows; check Live Activity behavior after a forced-quit. If those checks don't happen on real hardware, "verified" is a claim with nothing behind it, and given the invalidation criterion is specifically about trustworthiness under pressure, that gap would undermine the whole premise of the app, not just be a minor testing gap.

For what the first build should actually contain, given all that: nail the live console down to something almost stupidly simple — current cue, time remaining/elapsed with clear over/under-time indication, next cue preview, and three or four unmistakable big-touch actions (advance, hold/pause timer, jump-to-backup, reveal contingency note) — before spending any real effort on template richness or role metadata. Build the timeline/cue editor to be merely competent, not beautiful, in v1; the live console is where this app either proves itself or doesn't. And I'd treat the Live-Activity-goes-stale scenario as a must-test-before-ship item, not a nice-to-have, because it's the one failure mode that would make this app actively worse than a piece of paper — paper never shows you a stale, comfortingly wrong number.

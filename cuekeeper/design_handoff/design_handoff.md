I want to start from the part nobody's touched yet: what actually renders this thing and how the pieces are wired together, because the last three phases nailed the interaction model but left the client architecture implicit, and a few of those implicit choices will bite hard if a builder just wings them.

The routing model can't be a single NavigationStack hanging off Home. We've locked three independent ways to land on the Live Console — cold launch with an active-run pointer, tapping "Resume Live Event" from Home's live-event-present state, and normal navigation from Builder after tapping Start Live — plus a fourth path back out (the console's "Events" link) that has to return to Home without disturbing the run. That's a root-level state machine, not a nav-stack push. I'd model it as a tiny `AppPhase` enum owned by a root coordinator: `.loadingPersistence`, `.console(EventID)`, `.home`. On launch, before any view renders, the app reads the persisted active-live-event pointer; if one resolves to a non-completed event, phase is `.console` immediately, and Home never mounts at all — not "Home mounts and instantly redirects," which would be a real flash-of-wrong-content risk on slower devices or large data sets. Inside `.home`, a conventional NavigationStack handles Home → Builder → Cue Edit sheet. The Live Console itself sits outside that stack, presented as the root phase, so its own back-navigation ("Events" link) is a phase transition (`.console` → `.home`), not a stack pop — that distinction matters because a stack pop implies the console is "underneath" Home and could theoretically be reached by swipe-back gesture or by the system back-swipe edge gesture, which we do not want; the only way out of console has to be the deliberate labeled control, never an accidental edge-swipe.

That loadingPersistence phase is itself a real state that's been skipped in every prior phase's state inventory. Even a fast on-device store has a nonzero read before the app can decide its root phase. The other two threads closed clean too: the paywall wording is now a single unconditional sentence with no "if" left anywhere (contingency note + one backup target is free, ungated, no lock state in this build; the named premium gate only applies to future multi-backup structure that has no UI yet), and the entitlement-check timeout got a real number — 2 seconds, implemented as one shared function racing the real StoreKit check against a timer, with a timeout treated identically to any other unverifiable result (fail open, no special messaging). Everything else — root-phase state machine instead of a Home-first nav stack, isolated ticking-numeral subview, fixed non-adaptive console colors, the converged token set, contingency-panel auto-collapse on any other control tap — was already locked in round 2 and nobody reopened it.

## Final Output

**Root architecture:** The app is not a Home-first navigation stack with a redirect — it's a root-level `AppPhase` state machine (`.loadingPersistence`, `.console(EventID)`, `.home`) that decides before anything renders. On launch, the persisted active-live-event pointer is read first; if a non-completed run exists, phase is `.console` immediately and Home never mounts (no flash of the wrong screen). Inside `.home`, a normal `NavigationStack` handles Home → Builder → Cue Edit sheet. The Live Console sits outside that stack as its own root phase; its "Events" link is a phase transition back to `.home`, never a stack pop, so it can't be reached or exited via an accidental edge-swipe gesture. `.loadingPersistence` is a real, designed state (bare branded background, with a fallback indeterminate-progress treatment if resolution runs long) and is also where an app-root persistence-corruption error state lives, distinct from a single event's own save failure.

**Screen-by-screen spec:**

1. **Home** — states: true-first-launch-empty (one-sentence pitch + "Start from Sample Reception"/"Create Blank Event"), later-zero-events (same layout, plain "No events yet" copy), populated list (title, cue count, draft/ready/completed status badge, muted one-line drift summary for completed rows), live-event-present (visually dominant Resume card bleeding the console's accent color as a "something is live" signal; every other row desaturated/demoted and routes to the live-run-conflict dialog instead of its own Builder). Loading is a skeleton list; storage failure is an inline retry block. Entitlement-blocked-at-creation is a sheet presented before pushing to Builder, not a distinct Home visual state.

2. **Builder** — event title, live-computed total planned duration and cue count, readiness label, one dominant Start Live action up top; large reorderable cue rows below (title, duration or "Instant" chip, optional role chip, small glyphs for "has note"/"has backup"); add-cue always anchored and visible. States: seeded-sample loaded, blank/deleted-empty (real "add your first cue" prompt with the add action already visible — this is the shared empty-state proof for both blank-canvas entry and sample-deletion), populated/editable, invalid-zero-cues (can't start live), destructive-confirmation via native `.confirmationDialog` (deleting a cue that's another cue's backup target), persistence-error scoped to "this event failed to save" with retry.

3. **Cue Edit** (sheet) — title, duration stepper (zero-duration/"Instant" supported, negative duration blocked inline as the stepper moves, not on save), optional role field, optional contingency note, backup-target picker (excludes the cue itself, "None" grouped as a clear separate default), delete/duplicate. **Contingency note and backup-target picker are unconditionally free — no entitlement check, no lock icon, no entitlement-blocked state anywhere on this sheet.** The premium "add a contingency plan" gate named in prompt_contract refers only to future richer structure (multiple backup targets per cue), which has no UI in this build and therefore nothing to gate.

4. **Live Console** — fixed vertical stack, top to bottom, geometry never changes across states: small "Events" escape-hatch + event name in the safe area (phase transition back to Home, never pauses/ends the run) → current-cue card (title, role, isolated ticking numeral) → next-cue strip (smaller, lower-contrast) → a status/warning zone that **reserves its layout space even when empty** so a clock-jump warning appearing doesn't shift the control bar → fixed bottom control bar (advance, hold/resume, skip, contingency; skip is a squared button against the other three's pill shape, a non-color differentiator) → contingency tray expanding upward in-place from the control bar (never a sheet; current cue stays visible) → undo banner floating above the bar. **Any tap on advance/hold/skip auto-collapses an open contingency tray as part of its own transition** — the tray and the undo banner never occupy the same transient slot simultaneously. States: running, on hold, clock-warning, contingency-revealed, undo-available, completion-transition, recovered-from-relaunch.

5. **Completion** — its own terminal screen, not a modal: planned vs. actual duration, real final drift number, truthful completed-vs-skipped cue summary. Exactly two actions: "Back to Events," "Start New Event." No monetization or template affordances.

6. **Interruption surfaces:** live-run-conflict dialog, paywall sheet (success / fail-open-to-free with calm "continuing with free access" copy / a brief inline loading state under 2 seconds by construction — never looking like an error), contextual notification-permission prompt (asked at first "Start Live Run" tap, denial never affects correctness).

**Design tokens (converged, canonical):** 8pt base spacing grid (4/8/16/24/32); 20pt corner radius for cards/sheets, 10pt for chips/badges — identical across light, dark, and console contexts; system font with `.rounded` design and `.monospacedDigit()` for all console numerals (non-negotiable, prevents digit-width shift); Large Title/bold for the current-cue numeral, Title2 for cue title, Subheadline for next-cue, Footnote for chips/badges, all Dynamic Type-scaled except the console numeral clamps its relative growth to stay inside its card; semantic color roles — `accentAdvance` (doubles as on-time), `accentHold`, `accentSkip` (distinct hue *and* shape, not just a color variant), `accentWarningDrift` — plus a fixed, non-adaptive `consoleBackground`/`consoleForeground` pair built as explicit "Any Appearance"-only color assets so future increase-contrast/smart-invert settings can't wash it out; motion: 0.18–0.22s ease for reveals, 0.28s spring for cue-advance transitions, no ambient/decorative animation, full `accessibilityReduceMotion` fallback to opacity cross-fades; haptics: confirmatory (advance), heavier (hold), sharper (skip), subtle (contingency reveal).

**Component inventory:** EventListRow, ResumeLiveCard, EmptyStatePrompt (shared/parameterized between Home and Builder), CueRow, CueEditSheet, DurationStepper, BackupTargetPicker, CurrentCueCard, TickingNumeral (isolated `TimelineView(.periodic(from:, by: 1))` subview — scoped tightly so the once-a-second update never touches the surrounding control bar's view identity), NextCueStrip, DriftIndicatorPill, ClockWarningBanner, ControlBar (four ControlButton instances, three pill/one squared), ContingencyPanel, UndoBanner, LiveRunConflictDialog, PaywallSheet, NotificationPrimingPrompt, CompletionSummaryCard.

**Entitlement check:** one shared `checkEntitlement() async -> EntitlementResult` function, called from the two live creation-time gates (new event, apply template) — the contingency-plan gate is dormant/unreachable in this build's UI. Implemented as a race (`withTaskGroup`) between the real `Transaction.currentEntitlements` check and a 2-second `Task.sleep` timer; a timeout is treated identically to any other unverifiable result — fail open to free, no special-cased messaging. The triggering sheet/button shows a brief inline loading state for that sub-2-second window, distinct from success/fail-open states already in the inventory.

**Accessibility:** VoiceOver reading order on the console is deliberate (current cue title/role → elapsed/remaining → drift/hold state → next cue → controls), not view-tree order; every control label is consequence-specific ("Skip to next cue," never "Button"); undo banner never auto-dismisses on a timer under VoiceOver, persists until acted on, and announces its appearance; Builder reorder gets explicit "Move up"/"Move down" custom actions alongside drag; layout survives maximum Dynamic Type (numeral shrinks proportionally in its card, labels wrap/stack rather than truncate); status is never color-only; console contrast is tested specifically in its fixed-dark state, especially warning/hold; Reduce Motion gets full functional parity via opacity fades instead of slides/springs.

**External design decision:** **No external Claude Design pass for any app screen** — Home, Builder, Cue Edit, Live Console, Completion, and all interruption surfaces are built directly in SwiftUI from this written handoff. The reasoning both converged on: this app's design is inseparable from runtime interaction mechanics (fixed control geography across state transitions, the isolated ticking subview, non-adaptive console color assets, contingency-as-in-place-expansion, auto-collapse behavior) that a static mockup tool cannot represent and would risk silently overriding. If Claude Design is used at all, it's scoped strictly to post-launch marketing assets — App Store screenshots and icon exploration — produced by photographing the real, working, physical-device-verified app, not run as a parallel or preceding track.

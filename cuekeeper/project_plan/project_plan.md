- **M7 — Physical-device verification (its own scheduled milestone, own time budget):** Lock/unlock mid-cue, force-quit + relaunch, full airplane-mode pass across all three workflows, deliberate clock-change warning, denied-notification run, four distinguishable haptic patterns, VoiceOver reorder + undo persistence, max Dynamic Type. Named explicitly as the single most serialized, non-parallelizable resource in the whole plan (one iPhone) — not an end-of-plan checklist bullet.
- **M8 — Should-have go/no-go checkpoint:** Live Activities, Widgets, App Intents, snapshot-blur-on-background, template export/import — attempted only after the core is hardened; any cut under time pressure is named explicitly in the final report, never silently dropped.
- **M9 — Final report assembly:** Answers the invalidation criterion with real, measured tap counts, states what shipped vs. what's a named limitation.

**Dependencies:** M0 → M1 → M2 is a hard linear gate — no UI work of substance before Core and Persistence are code-complete and test-green. M3 and M4 both depend on M2 but M4 additionally depends on M1's projection shape being frozen. M5 and M6 are genuinely parallel side-tracks that can start once M1 is frozen and a console skeleton exists (mid-M4). M7 depends on M3–M6 being substantially done since it's a verification pass over the whole app. M8 is strictly after M7 (core hardening takes priority). M9 is last.

**Risk mitigations:**
- Compile-time StoreKit isolation could be fictional if project wiring is wrong → M0's negative-import smoke test.
- Notification requests silently dropped past the 64-request OS cap → windowed scheduling (next 20 cues / 2hr lookahead) named as a hard contract on `NotificationScheduler`, not left to "reasonable" engineering judgment.
- Swift 6 strict concurrency landmine in `HapticsService` (non-`Sendable` UIKit generators called off-main) → explicit `@MainActor` marking and a specified await-then-hop call sequence, named before build rather than discovered as a compile error mid-build.
- Physical-device verification getting compressed into a simulator glance under time pressure → its own scheduled milestone (M7) with an explicit non-negotiable checklist.
- Timed claims (two-minute build-to-ready) being self-reported and therefore fake confidence → independently timed, not self-timed.
- Parallel build lanes drifting apart on the shape of `LiveRunProjection` → M1 must be frozen and test-green before M3/M4/M5 begin.

**Verification plan:** Swift Testing unit coverage for Core/Persistence (deterministic, clock-injected, no device needed) closes out M1/M2. XCUITest with `-uiTestSeedSampleEvent` closes out M4's tap-count claims. M7 is the required, scheduled, non-negotiable physical-device pass covering every timing/offline/accessibility claim made in earlier phases. M9's final report must state the enumerated tap counts and name any should-have cuts explicitly — the plan isn't done until build verification and the final review agree on what works and what's a named limitation.

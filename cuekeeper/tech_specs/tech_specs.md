Locking down the concrete shape of what "thin and boring" actually compiles to, because the last two phases described the right architecture in prose and now someone has to turn "the live-run engine must have zero StoreKit references" into something the compiler enforces rather than a rule a build lane can accidentally violate.

I want to push the module boundary further than "discipline" and make it structural: four local Swift Packages plus an app target, no third-party tooling required, just folders with `Package.swift` inside the Xcode project. `CueKeeperCore` holds the domain — `Event`, `Cue`, the `ActionLogEntry` log, the `LiveRunEngine` projection function, `ClockProviding` — and imports nothing but `Foundation`. `CueKeeperPersistence` (an actor-based store) depends only on Core. `CueKeeperEntitlements` wraps StoreKit and depends only on Core. `CueKeeperNotifications` and `CueKeeperHaptics` are similarly thin. Then the app target splits its SwiftUI files into two groups: a CreationFlow group (Home, Builder, CueEdit) that's the only place allowed to `import CueKeeperEntitlements`, and a LiveRun group (LiveConsole, Completion) that never imports it. This isn't just a naming convention — if someone on a build lane tries to sneak an entitlement check into the console, it's a compile error because that file's target doesn't link the package. That's a stronger guarantee than "the code review will catch it," which is the failure mode I'm most worried about with parallel lanes touching adjacent files under time pressure.

On concurrency, I want to actually turn on Swift 6 strict concurrency mode for the whole project rather than staying in the permissive default, because this app's entire value proposition is "state doesn't drift out of sync under pressure," and a data race in the live-run engine is exactly the kind of bug that would only show up once, on a real device, mid-event, and be nearly impossible to reproduce afterward. That means `Event`, `Cue`, `EventLog`, `ActionLogEntry`, and `LiveRunProjection` all need to be `Sendable` value types (they already are, being plain Codable structs/enums), `EventStore` is an `actor`, and every view model that owns mutable UI state is explicitly `@MainActor`. This costs a bit of friction up front and pays for itself the first time it stops a lane from accidentally sharing mutable state across an async boundary.

For the live console's rendering specifically, I want to be stricter than "isolate the ticking numeral" — I want the console view's body to depend on nothing but the `LiveRunProjection` value type, never on the raw `Event`/`EventLog`. That projection is cheap to diff (it's a handful of scalars and two UUIDs), so SwiftUI's equality-based re-render skipping actually works, and the `TickingNumeral` subview should take just the two `Date`s it needs to compute its digits, not the whole projection — so its once-a-second `TimelineView` tick can never invalidate the control bar's identity, and the control bar's own reference equality never gets disturbed by information it doesn't need.

Testing strategy: I want to use Swift Testing (`@Test`/`#expect`, shipped in Xcode 16+) for the engine and persistence layers instead of XCTest, specifically because its `@Test(arguments:)` parameterization is a natural fit for the kind of table-driven cases this app needs — a dozen hold-rebase scenarios, a dozen clock-jump-threshold cases, undo-after-skip, undo-after-jump-to-backup — without writing a dozen near-identical test methods by hand. XCUITest stays for the UI layer since Swift Testing doesn't yet have a UI-testing story, so the `-uiTestSeedSampleEvent` launch argument still drives an XCUITest suite. I'd add one more concrete testability requirement on top of what's locked: `LiveRunEngine.project` should be a pure static function taking `(event: Event, log: EventLog, clock: ClockProviding) -> LiveRunProjection` — no instance state, no side effects — so every test is "given this log and this injected time, assert this exact projection," fully deterministic, zero setup.

## Final Output

**Architecture overview:** A thin, single-app-target SwiftUI app plus exactly one local Swift Package (`CueKeeperEntitlements`), not four. The one package boundary exists because it's the one place a forbidden dependency edge actually matters: StoreKit must never reach the live-run engine or console. Everything else — domain model, the append-only action-log/projection engine, the persistence actor, notifications, haptics, and all views — lives in the app target, organized by folder (`Core/`, `Persistence/`, `Notifications/`, `Haptics/`, `UI/CreationFlow/`, `UI/LiveRun/`, `UI/Shared/`). Root routing is the `AppPhase` state machine (`.loadingPersistence`/`.console(EventID)`/`.home`) locked in the prior design phase, not a NavigationStack redirect. Full Swift 6 strict concurrency: all domain/log/projection types are `Sendable` value types, `EventStore` is an `actor`, all UI-owning types are `@MainActor`.

**File/module layout:** App target with folders `Core/` (Event, Cue, ActionLogEntry, EventLog, LiveRunProjection, ClockProviding/SystemClock, LiveRunEngine, EventStatus, EventManifest), `Persistence/` (EventStoring protocol, EventStore actor, PersistenceError, AppGroupContainer), `Notifications/` (NotificationScheduler), `Haptics/` (HapticPattern, HapticsService), `UI/CreationFlow/` (Home, Builder, CueEdit — the only place allowed to link `CueKeeperEntitlements`), `UI/LiveRun/` (LiveConsole, Completion — structurally cannot import `CueKeeperEntitlements`, a compile error if attempted), `UI/Shared/`. Plus the one local package: `CueKeeperEntitlements` (EntitlementService, EntitlementResult).

**Data models:** `Event`/`Cue` (stable UUIDs, backup targets by ID not index, zero-duration cues valid), `ActionLogEntry` append-only enum (start/advance/skip/holdStarted/holdEnded/jumpToBackup/undoLastAction/complete, each carrying wall-clock + monotonic uptime), `EventLog` (never truncated — undo is its own log entry, replayed by skipping the entry preceding it), `LiveRunProjection` (the only type the console's view body reads — a cheap-to-diff, Equatable value type so SwiftUI's re-render skipping actually works), `EventManifest`/`EventManifestEntry` (routing pointer is a hint, always reconfirmed by re-projecting the event's own log before trusting it).

**Persistence strategy:** Flat Codable JSON, one file per event plus a manifest, atomic temp-file-then-rename writes, in an App Group container from day one. `EventStoring` is a protocol with `EventStore` as its only production conformance — this protocol boundary is the explicit, stated answer to the portfolio's "local-first but cloud-ready" requirement: a future CloudKit-backed store is a second conformance swapped in at the composition root, with zero call-site changes anywhere in the app, and nothing sync-related is being built now.

**Testing strategy:** Swift Testing (`@Test`/`#expect`, `@Test(arguments:)` for table-driven cases) unconditionally for everything in `Core/` and `Persistence/` — no XCTest there. XCTest/XCUITest only where Swift Testing has no story yet: the UI layer, driven by the `-uiTestSeedSampleEvent` launch argument for deterministic tap-count assertions. `LiveRunEngine.project` is a pure static function taking an injected clock, so hold-rebase, undo-after-skip, jump-to-backup, zero-duration cues, and clock-jump thresholds are all deterministic, device-free unit tests. The physical-device manual matrix from the prior phase (lock/unlock mid-cue, force-quit, airplane mode, clock-change, denied-notifications) stays required.

**File/module layout:** App target with folders `Core/` (Event, Cue, ActionLogEntry, EventLog, LiveRunProjection, ClockProviding/SystemClock, LiveRunEngine, EventStatus, EventManifest), `Persistence/` (EventStoring protocol, EventStore actor, PersistenceError, AppGroupContainer), `Notifications/` (NotificationScheduler), `Haptics/` (HapticPattern, HapticsService), `UI/CreationFlow/` (Home, Builder, CueEdit — the only place allowed to link `CueKeeperEntitlements`), `UI/LiveRun/` (LiveConsole, Completion — structurally cannot import `CueKeeperEntitlements`, a compile error if attempted), `UI/Shared/`.

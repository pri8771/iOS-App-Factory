**SwiftUI architecture:** A thin, boring app with one root coordinator (`AppPhase`: `.loadingPersistence` / `.console(EventID)` / `.home`, as already locked in design_handoff), one persistence boundary, and one isolated live-run engine that owns all timing math. SwiftUI views render derived state only — no timing math, no direct system-clock calls in view models. Live-run state is modeled as an **append-only action log** per event (`start, advance, skip, holdStarted, holdEnded, jumpToBackup, undoLastAction, complete`), and the current console state (current cue, elapsed, drift, hold offsets, "is this event live," completion) is a pure, deterministic projection over that log — never a set of mutable fields that can drift out of sync with each other. Undo doesn't truncate the log (that would destroy the audit trail); it's its own entry type, and projection skips the entry immediately preceding an `undoLastAction` when replaying. All time-dependent logic (drift, hold rebase, clock-jump detection) takes an injected "now" and "uptime" — no direct `Date()` or `ProcessInfo.processInfo.systemUptime` calls buried in domain logic — so it's unit-testable without a device or simulator.

**Apple framework choices:** SwiftUI + Observation for UI/state; UserNotifications for `.timeSensitive` cue alerts (full cancel-and-reschedule of all remaining requests on every mutating action — advance/hold/skip/jump — keyed by `eventID+cueID`, not incremental diffing, since a diff is exactly the kind of logic that leaves a stale alert behind after two quick skips); ActivityKit/WidgetKit/App Intents as should-have, first cut under pressure, self-correcting from the same absolute-timestamp log if shipped; StoreKit 2 for local subscription testing only (no server receipt validation, no homegrown paywall framework); Core Haptics/UIKit feedback generators for the four mapped haptic patterns. No CloudKit, no AR, no ML, no LLM — all correctly out of scope, they'd only add failure surface to the one part of the app that has to be boringly trustworthy.

**Persistence and local data plan:** Flat Codable JSON documents, one file per event, plus a lightweight manifest/index file for the Home list — not SwiftData, not Core Data. Reasoning locked: this app's data volume is trivially small (one operator, a handful of events, 20-40 cues each), and the product's actual requirement is deterministic, crash-inspectable, unit-testable state, not query power; SwiftData's ModelContext concurrency constraints and CloudKit's all-fields-optional requirement mean it wouldn't even deliver the "easier sync later" benefit it's usually chosen for. All writes (event files and log appends) use atomic temp-file-then-rename semantics; worst-case crash loses only the single most recent unflushed action, which the operator just redoes with one tap — never a corrupted, unparseable file. "Is this event live" is derived by loading and projecting that event's log, not tracked as a separate flag — this eliminates the flag/pointer-disagreement failure mode by construction. Cold-launch routing reads the manifest's active-event pointer, loads that event's log, projects it, and only routes into the console if the projection agrees the event is actually live; otherwise it fails safe to Home. Per-event corruption is isolated — a single event's file failing to decode quarantines that one row on Home, distinct from the app-root persistence-corruption state reserved for the manifest itself failing. The manifest pointer is the single source of truth for routing (not a scan across all event logs), so if the one-live-run rule is ever violated by a bug, a second orphaned "technically live" log just sits inert until someone opens that event directly — a named, accepted fallback, not a surprise. The store lives in an App Group container from day one, so a future widget/Live Activity extension needs no storage migration, even though those features are should-have and likely first-cut.

**Permissions/privacy plan:** Exactly one system permission, ever — notification authorization, requested contextually at first "Start Live," never during onboarding; denial degrades convenience only, never correctness. No location, camera, microphone, contacts, or photo library access and no corresponding Info.plist usage strings. No analytics or crash SDKs. Event/log files use `FileProtectionType.completeUntilFirstUserAuthentication` (not `.complete`), so a just-rebooted, not-yet-unlocked phone doesn't make the app look broken — and this must be verified as actually applied on disk via a real test, not assumed from configuration alone, since a silently-ignored protection class is worse than none. Contingency notes are named explicitly as potentially sensitive client information; app-switcher snapshot blur-on-background is a real but minor mitigation, cut-first under pressure and named in the final report either way if cut. Template export (should-have) defaults to stripping notes unless the user opts in.

**Dependency/license decision:** Zero third-party packages. First-party frameworks cover everything this app needs; every added dependency is a fresh risk to offline determinism, App Review clarity, and long-term maintenance on a product whose entire value proposition is trust.

**StoreKit/entitlement architecture:** One shared `checkEntitlement()` function, backed by a cached entitlement flag refreshed opportunistically via `Transaction.updates` at launch/foreground, falling back to the already-locked 2-second race-with-timeout only when the cache is stale or absent; a timeout is treated identically to any other unverifiable result (fail open to free). Persistence tests need round-trip coverage, corruption handling at both the per-event and manifest level, and an explicit check that the file-protection class actually landed on disk. UI tests need a deterministic `-uiTestSeedSampleEvent` launch argument that drops straight into a known live-console state so the ≤2-tap claims for advance/hold/skip/contingency-jump can be asserted by an actual XCUITest counting taps, plus coverage of cold-launch-into-active-run, one-live-run conflict, notification-prompt timing, and backup-target-deletion confirmation. StoreKit tests use the local `.storekit` config and explicitly verify fail-open on timeout/unverifiable state.

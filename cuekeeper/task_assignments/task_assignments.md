- **primary_ui** owns `CueKeeper/App/CueKeeperApp.swift`, `UI/Shared/AppPhase.swift`+`AppCoordinator.swift`, all of `UI/CreationFlow/` (Home, Builder, Cue Edit — each now including its own accessibility work), all of `UI/LiveRun/` (Console, Completion, subviews — same, accessibility included), interruption surfaces, and the UI test suite. Forbidden rule: files under `UI/LiveRun/` never gain an entitlements import — that's a compile error if attempted, not a review catch.
- **polish_resilience** owns the standalone `AccessibilityAnnouncer.swift` (small, dependency-free, lands early so the console can call it), `SnapshotPrivacyGuard.swift`, `TemplateExporter.swift`, and the two paper artifacts (`VERIFICATION.md`, `FINAL_REPORT.md`) that make the build's completion checkable rather than assumed.

**Handoffs and forbidden edit zones:** No lane other than services_utilities touches `CueKeeperEntitlements/`. No lane other than primary_ui touches `UI/LiveRun/` or `UI/CreationFlow/`. polish_resilience adds new standalone files only — it never re-edits a file another lane already owns. The one genuinely shared, contentious artifact is `project.pbxproj`: no lane batches file additions across days; each task adds its files and commits immediately, and if two tasks need it in the same window, whoever's ready first commits and the other rebases.

**Branching and merge strategy:** Trunk-based. `main` is the integration branch. Each task gets a short-lived `task/T-0NN-short-slug` branch, merged back same-day, deleted. No long-lived per-lane branches — that's the shape that produces a giant conflicted `pbxproj` merge at the end. Merge order enforces the dependency gate: scaffold → Core → Persistence must be merged and green before any UI task branches off.

**Testing responsibility:** data_domain and services_utilities write and keep green their own Swift Testing suites (hold-rebase, undo-after-skip, jump-to-backup, clock-jump, notification-window-cap, on-disk file-protection) as a hard gate before downstream work starts. primary_ui owns the XCUITest suite that actually counts taps against the ≤2-tap claim. polish_resilience executes the physical-device checklist and writes the final report stating exactly what was verified and what's a named limitation.

## Final Output

**Root architecture:** The app is not a Home-first navigation stack with a redirect — it's a root-level `AppPhase` state machine (`.loadingPersistence`, `.console(EventID)`, `.home`) that decides before anything renders. On launch, the persisted active-live-event pointer is read first; if a non-completed run exists, phase is `.console` immediately and Home never mounts (no flash of the wrong screen). Inside `.home`, a normal `NavigationStack` handles Home → Builder → Cue Edit sheet. The Live Console sits outside that stack as its own root phase; its "Events" link is a phase transition back to `.home`, never a stack pop, so it can't be reached or exited via an accidental edge-swipe gesture. `.loadingPersistence` is a real, designed state (bare branded background, with a fallback indeterminate-progress treatment if resolution runs long) and is also where an app-root persistence-corruption error state lives, distinct from a single event's own save failure.

**Screen-by-screen spec:**

1. **Home** — states: true-first-launch-empty (one-sentence pitch + "Start from Sample Reception"/"Create Blank Event"), later-zero-events (same layout, plain "No events yet" copy), populated list (title, cue count, draft/ready/completed status badge, muted one-line drift summary for completed rows), live-event-present (visually dominant Resume card bleeding the console's accent color as a "something is live" signal; every other row desaturated/demoted and routes to the live-run-conflict dialog instead of its own Builder). Loading is a skeleton list; storage failure is an inline retry block. Entitlement-blocked-at-creation is a sheet presented before pushing to Builder, not a distinct Home visual state.

2. **Builder** — event title, live-computed total planned duration and cue count, readiness label, one dominant Start Live action up top; large reorderable cue rows below (title, duration or "Instant" chip, optional role chip, small glyphs for "has note"/"has backup"); add-cue always anchored and visible. States: seeded-sample loaded, blank/deleted-empty (real "add your first cue" prompt with the add action already visible — this is the shared empty-state proof for both blank-canvas entry and sample-deletion), populated/editable, invalid-zero-cues (can't start live), destructive-confirmation via native `.confirmationDialog` (deleting a cue that's another cue's backup target), persistence-error scoped to "this event failed to save" with retry.

3. **Cue Edit** (sheet) — title, duration stepper (zero-duration/"Instant" supported, negative duration blocked inline as the stepper moves, not on save), optional role field, optional contingency note, backup-target picker (excludes the cue itself, "None" grouped as a clear separate default), delete/duplicate. **Contingency note and backup-target picker are unconditionally free — no entitlement check, no lock icon, no entitlement-blocked state anywhere on this sheet.** The premium "add a contingency plan" gate named in prompt_contract refers only to future richer structure (multiple backup targets per cue), which has no UI in this build and therefore nothing to gate.

4. **Live Console** — fixed vertical stack, top to bottom, geometry never changes across states: small "Events" escape-hatch + event name in the safe area (phase transition back to Home, never pauses/ends the run) → current-cue card (title, role, isolated ticking numeral) → next-cue strip (smaller, lower-contrast) → a status/warning zone that **reserves its layout space even when empty** so a clock-jump warning appearing doesn't shift the control bar → fixed bottom control bar (advance, hold/resume, skip, contingency; skip is a squared button against the other three's pill shape, a non-color differentiator) → contingency tray expanding upward in-place from the control bar (never a sheet; current cue stays visible) → undo banner floating above the bar. **Any tap on advance/hold/skip auto-collapses an open contingency tray as part of its own transition** — the tray and the undo banner never occupy the same transient slot simultaneously. States: running, on hold, clock-warning, contingency-revealed, undo-available, completion-transition, recovered-from-relaunch.

5. **Completion** — its own terminal screen, not a modal: planned vs. actual duration, real final drift number, truthful completed-vs-skipped cue summary. Exactly two actions: "Back to Events," "Start New Event." No monetization or template affordances.

6. **Interruption surfaces:** live-run-conflict dialog, paywall sheet (success / fail-open-to-free with calm "continuing with free access" copy / a brief inline loading state under 2 seconds by construction — never looking like an error), contextual notification-permission prompt (asked at first "Start Live Run" tap, denial never affects correctness).

**Design tokens (converged, canonical):** 8pt base spacing grid (4/8/16/24/32); 20pt corner radius for cards/sheets, 10pt for chips/badges — identical across light, dark, and console contexts; system font with `.rounded` design and `.monospacedDigit()` for all console numerals (non-negotiable, prevents digit-width shift); Large Title/bold for the current-cue numeral, Title2 for cue title, Subheadline for next-cue, Footnote for chips/badges, all Dynamic Type-scaled except the console numeral clamps its relative growth to stay inside its card; semantic color roles — `accentAdvance` (doubles as on-time), `accentHold`, `accentSkip` (distinct hue *and* shape, not just a color variant), `accentWarningDrift` — plus a fixed, non-adaptive `consoleBackground`/`consoleForeground` pair built as explicit "Any Appearance"-only color assets so future increase-contrast/smart-invert settings can't wash it out; motion: 0.18–0.22s ease for reveals, 0.28s spring for cue-advance transitions, no ambient/decorative animation, full `accessibilityReduceMotion` fallback to opacity cross-fades; haptics: confirmatory (advance), heavier (hold), sharper (skip), subtle (contingency reveal).

**Component inventory:** EventListRow, ResumeLiveCard, EmptyStatePrompt (shared/parameterized between Home and Builder), CueRow, CueEditSheet, DurationStepper, BackupTargetPicker, CurrentCueCard, TickingNumeral (isolated `TimelineView(.periodic(from:, by: 1))` subview — scoped tightly so the once-a-second update never touches the surrounding control bar's view identity), NextCueStrip, DriftIndicatorPill, ClockWarningBanner, ControlBar (four ControlButton instances, three pill/one squared), ContingencyPanel, UndoBanner, LiveRunConflictDialog, PaywallSheet, NotificationPrimingPrompt, CompletionSummaryCard.

**Entitlement check:** one shared `checkEntitlement() async -> EntitlementResult` function, called from the two live creation-time gates (new event, apply template) — the contingency-plan gate is dormant/unreachable in this build's UI. Implemented as a race (`withTaskGroup`) between the real `Transaction.currentEntitlements` check and a 2-second `Task.sleep` timer; a timeout is treated identically to any other unverifiable result — fail open to free, no special-cased messaging. The triggering sheet/button shows a brief inline loading state for that sub-2-second window, distinct from success/fail-open states already in the inventory.

**Accessibility:** VoiceOver reading order on the console is deliberate (current cue title/role → elapsed/remaining → drift/hold state → next cue → controls), not view-tree order; every control label is consequence-specific ("Skip to next cue," never "Button"); undo banner never auto-dismisses on a timer under VoiceOver, persists until acted on, and announces its appearance; Builder reorder gets explicit "Move up"/"Move down" custom actions alongside drag; layout survives maximum Dynamic Type (numeral shrinks proportionally in its card, labels wrap/stack rather than truncate); status is never color-only; console contrast is tested specifically in its fixed-dark state, especially warning/hold; Reduce Motion gets full functional parity via opacity fades instead of slides/springs.

**External design decision:** **No external Claude Design pass for any app screen** — Home, Builder, Cue Edit, Live Console, Completion, and all interruption surfaces are built directly in SwiftUI from this written handoff. The reasoning both converged on: this app's design is inseparable from runtime interaction mechanics (fixed control geography across state transitions, the isolated ticking subview, non-adaptive console color assets, contingency-as-in-place-expansion, auto-collapse behavior) that a static mockup tool cannot represent and would risk silently overriding. If Claude Design is used at all, it's scoped strictly to post-launch marketing assets — App Store screenshots and icon exploration — produced by photographing the real, working, physical-device-verified app, not run as a parallel or preceding track.

## Final Output

**SwiftUI architecture:** A thin, boring app with one root coordinator (`AppPhase`: `.loadingPersistence` / `.console(EventID)` / `.home`, as already locked in design_handoff), one persistence boundary, and one isolated live-run engine that owns all timing math. SwiftUI views render derived state only — no timing math, no direct system-clock calls in view models. Live-run state is modeled as an **append-only action log** per event (`start, advance, skip, holdStarted, holdEnded, jumpToBackup, undoLastAction, complete`), and the current console state (current cue, elapsed, drift, hold offsets, "is this event live," completion) is a pure, deterministic projection over that log — never a set of mutable fields that can drift out of sync with each other. Undo doesn't truncate the log (that would destroy the audit trail); it's its own entry type, and projection skips the entry immediately preceding an `undoLastAction` when replaying. All time-dependent logic (drift, hold rebase, clock-jump detection) takes an injected "now" and "uptime" — no direct `Date()` or `ProcessInfo.processInfo.systemUptime` calls buried in domain logic — so it's unit-testable without a device or simulator.

**Apple framework choices:** SwiftUI + Observation for UI/state; UserNotifications for `.timeSensitive` cue alerts (full cancel-and-reschedule of all remaining requests on every mutating action — advance/hold/skip/jump — keyed by `eventID+cueID`, not incremental diffing, since a diff is exactly the kind of logic that leaves a stale alert behind after two quick skips); ActivityKit/WidgetKit/App Intents as should-have, first cut under pressure, self-correcting from the same absolute-timestamp log if shipped; StoreKit 2 for local subscription testing only (no server receipt validation, no homegrown paywall framework); Core Haptics/UIKit feedback generators for the four mapped haptic patterns. No CloudKit, no AR, no ML, no LLM — all correctly out of scope, they'd only add failure surface to the one part of the app that has to be boringly trustworthy.

**Persistence and local data plan:** Flat Codable JSON documents, one file per event, plus a lightweight manifest/index file for the Home list — not SwiftData, not Core Data. Reasoning locked: this app's data volume is trivially small (one operator, a handful of events, 20-40 cues each), and the product's actual requirement is deterministic, crash-inspectable, unit-testable state, not query power; SwiftData's ModelContext concurrency constraints and CloudKit's all-fields-optional requirement mean it wouldn't even deliver the "easier sync later" benefit it's usually chosen for. All writes (event files and log appends) use atomic temp-file-then-rename semantics; worst-case crash loses only the single most recent unflushed action, which the operator just redoes with one tap — never a corrupted, unparseable file. "Is this event live" is derived by loading and projecting that event's log, not tracked as a separate flag — this eliminates the flag/pointer-disagreement failure mode by construction. Cold-launch routing reads the manifest's active-event pointer, loads that event's log, projects it, and only routes into the console if the projection agrees the event is actually live; otherwise it fails safe to Home. Per-event corruption is isolated — a single event's file failing to decode quarantines that one row on Home, distinct from the app-root persistence-corruption state reserved for the manifest itself failing. The manifest pointer is the single source of truth for routing (not a scan across all event logs), so if the one-live-run rule is ever violated by a bug, a second orphaned "technically live" log just sits inert until someone opens that event directly — a named, accepted fallback, not a surprise. The store lives in an App Group container from day one, so a future widget/Live Activity extension needs no storage migration, even though those features are should-have and likely first-cut.

**Permissions/privacy plan:** Exactly one system permission, ever — notification authorization, requested contextually at first "Start Live," never during onboarding; denial degrades convenience only, never correctness. No location, camera, microphone, contacts, or photo library access and no corresponding Info.plist usage strings. No analytics or crash SDKs. Event/log files use `FileProtectionType.completeUntilFirstUserAuthentication` (not `.complete`), so a just-rebooted, not-yet-unlocked phone doesn't make the app look broken — and this must be verified as actually applied on disk via a real test, not assumed from configuration alone, since a silently-ignored protection class is worse than none. Contingency notes are named explicitly as potentially sensitive client information; app-switcher snapshot blur-on-background is a real but minor mitigation, cut-first under pressure and named in the final report either way if cut. Template export (should-have) defaults to stripping notes unless the user opts in.

**Dependency/license decision:** Zero third-party packages. First-party frameworks cover everything this app needs; every added dependency is a fresh risk to offline determinism, App Review clarity, and long-term maintenance on a product whose entire value proposition is trust.

**StoreKit/entitlement architecture:** One shared `checkEntitlement()` function, backed by a cached entitlement flag refreshed opportunistically via `Transaction.updates` at launch/foreground, falling back to the already-locked 2-second race-with-timeout only when the cache is stale or absent; a timeout is treated identically to any other unverifiable result (fail open to free). Gated only at the two live creation-time boundaries (new event, apply template) — the contingency-plan gate remains dormant/unreachable per the design phase's resolution. This boundary is enforced structurally, not just behaviorally: the live-run engine, console views, and notification scheduler contain zero references to any StoreKit type, so a refund or expiration detected mid-run literally cannot reach the active run — it can only affect the next creation-time check.

**Testing implications:** The live-run engine must be fully unit-testable with injected clocks (wall + monotonic), covering hold rebase, undo-after-skip (replaying a log ending in `skip, undoLastAction` and asserting the pre-skip state), jump-to-backup pointer relocation, last-cue completion, zero-duration cues, and clock-jump thresholds — all deterministic, no device required. Persistence tests need round-trip coverage, corruption handling at both the per-event and manifest level, and an explicit check that the file-protection class actually landed on disk. UI tests need a deterministic `-uiTestSeedSampleEvent` launch argument that drops straight into a known live-console state so the ≤2-tap claims for advance/hold/skip/contingency-jump can be asserted by an actual XCUITest counting taps, plus coverage of cold-launch-into-active-run, one-live-run conflict, notification-prompt timing, and backup-target-deletion confirmation. StoreKit tests use the local `.storekit` config and explicitly verify fail-open on timeout/unverifiable state. The manual device matrix carried forward from earlier phases stays required and non-negotiable: lock/unlock mid-cue, force-quit mid-cue, full airplane-mode run, device clock-change warning, and a denied-notification run.

## Final Output

**Architecture overview:** A thin, single-app-target SwiftUI app plus exactly one local Swift Package (`CueKeeperEntitlements`), not four. The one package boundary exists because it's the one place a forbidden dependency edge actually matters: StoreKit must never reach the live-run engine or console. Everything else — domain model, the append-only action-log/projection engine, the persistence actor, notifications, haptics, and all views — lives in the app target, organized by folder (`Core/`, `Persistence/`, `Notifications/`, `Haptics/`, `UI/CreationFlow/`, `UI/LiveRun/`, `UI/Shared/`). Root routing is the `AppPhase` state machine (`.loadingPersistence`/`.console(EventID)`/`.home`) locked in the prior design phase, not a NavigationStack redirect. Full Swift 6 strict concurrency: all domain/log/projection types are `Sendable` value types, `EventStore` is an `actor`, all UI-owning types are `@MainActor`.

**File/module layout:** App target with folders `Core/` (Event, Cue, ActionLogEntry, EventLog, LiveRunProjection, ClockProviding/SystemClock, LiveRunEngine, EventStatus, EventManifest), `Persistence/` (EventStoring protocol, EventStore actor, PersistenceError, AppGroupContainer), `Notifications/` (NotificationScheduler), `Haptics/` (HapticPattern, HapticsService), `UI/CreationFlow/` (Home, Builder, CueEdit — the only place allowed to link `CueKeeperEntitlements`), `UI/LiveRun/` (LiveConsole, Completion — structurally cannot import `CueKeeperEntitlements`, a compile error if attempted), `UI/Shared/`. Plus the one local package: `CueKeeperEntitlements` (EntitlementService, EntitlementResult).

**Data models:** `Event`/`Cue` (stable UUIDs, backup targets by ID not index, zero-duration cues valid), `ActionLogEntry` append-only enum (start/advance/skip/holdStarted/holdEnded/jumpToBackup/undoLastAction/complete, each carrying wall-clock + monotonic uptime), `EventLog` (never truncated — undo is its own log entry, replayed by skipping the entry preceding it), `LiveRunProjection` (the only type the console's view body reads — a cheap-to-diff, Equatable value type so SwiftUI's re-render skipping actually works), `EventManifest`/`EventManifestEntry` (routing pointer is a hint, always reconfirmed by re-projecting the event's own log before trusting it).

**Persistence strategy:** Flat Codable JSON, one file per event plus a manifest, atomic temp-file-then-rename writes, in an App Group container from day one. `EventStoring` is a protocol with `EventStore` as its only production conformance — this protocol boundary is the explicit, stated answer to the portfolio's "local-first but cloud-ready" requirement: a future CloudKit-backed store is a second conformance swapped in at the composition root, with zero call-site changes anywhere in the app, and nothing sync-related is being built now.

**Testing strategy:** Swift Testing (`@Test`/`#expect`, `@Test(arguments:)` for table-driven cases) unconditionally for everything in `Core/` and `Persistence/` — no XCTest there. XCTest/XCUITest only where Swift Testing has no story yet: the UI layer, driven by the `-uiTestSeedSampleEvent` launch argument for deterministic tap-count assertions. `LiveRunEngine.project` is a pure static function taking an injected clock, so hold-rebase, undo-after-skip, jump-to-backup, zero-duration cues, and clock-jump thresholds are all deterministic, device-free unit tests. The physical-device manual matrix from the prior phase (lock/unlock mid-cue, force-quit, airplane mode, clock-change, denied-notifications) stays required.

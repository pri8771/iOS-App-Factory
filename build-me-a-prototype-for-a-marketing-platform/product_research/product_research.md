Platform-specific opportunities: iPhone should make this feel immediate—single-tap asset intake from camera/photo library, typed or voice brief capture, contextual action chips (“Submit for review,” “Mark rework”), swipe gestures for status transitions, and a timeline view that reads like a queue so users can act quickly. Major assumptions and risks: first, we cannot verify the Ordinal feature list from a live source in this phase, so the interpretation is a working assumption and must remain tagged as unverified scope. Given the contract we're locked into — single-user, local-first, offline iOS app modeled on Ordinal's brief/asset/review-approval shape — I want to ground this phase in who's actually opening this app and what breaks first if we get it wrong, rather than re-litigating the feature list.

For comparable patterns, I'd look at apps that already solved pieces of this well natively rather than inventing UI: Photos.app's non-destructive edit history is the direct precedent for "versions append, current is marked, nothing overwritten" — that's a proven interaction model (edit stack, revert-to-original) we can borrow the shape of without copying the UI. A Kanban-ish status board (Draft/In Review/Approved/Rework as columns or filter chips) is a well-understood metaphor for the state machine and avoids us inventing a bespoke workflow visualization. Platform opportunities worth calling out: a Share Sheet extension so assets can be imported directly from Photos/Files/Safari into a campaign instead of forcing a clunky in-app picker; QuickLook for previewing images/video/PDF assets instead of building a custom media viewer (this alone kills a large chunk of risk); swipe actions for fast approve/rework on list rows; local (on-device, no server) notifications nudging the user about stale "In Review" items, which is safe under the privacy non-goals since nothing leaves the device.

Now the risk side, because I think this is where the next build phase is most likely to quietly paint itself into a corner. First: where do asset bytes actually live? If video/image binary data gets stored directly inside the SwiftData store, the store bloats fast and becomes fragile — corruption or slow launches. The safer pattern is file-backed storage in the app's Documents/Application Support directory with SwiftData holding only metadata and a URL reference, and that decision needs to be made now, not discovered mid-build. Second: "safe/undoable deletion" is still underspecified the same way "versioning" was before we fixed it — undoable how? Fourth, and more mechanical: non-destructive versioning has an unbounded storage-growth failure mode — nothing ever gets deleted, so a campaign with a dozen video re-uploads could eat serious device storage with no cleanup story; that doesn't need to be solved now, but it needs to be flagged as a known limitation rather than silently ignored. Fifth: the "zero assets" and "mid-review deletion" edge cases already named in the contract compound with versioning — what happens if the only version of an asset gets deleted while its campaign is "In Review"? For implications on the first build: keep asset storage file-backed with metadata-only in SwiftData, pick one concrete undo/delete mechanism now (I'd lean toward a short-lived undo toast plus a trash section rather than instant hard-delete, since it's cheap and matches "safe/undoable" without inventing a full trash-management UI), don't attempt any sync or export mechanism in v1 but state the single-device risk plainly in whatever final review closes this out, and lean on QuickLook/Share Sheet/swipe actions instead of custom-building media viewing or import flows — that's the biggest lever we have for shipping something small that still feels native rather than a half-finished custom UI kit.

**Codex (Design Lead · Skeptic) — Round 1**

As a first-pass, I’d keep the feature set small, but make the emotional core non-negotiable: a user must be able to get from a one-line brief to a signed-off asset decision in one campaign flow without guessing where the current item is.

Must-have: Create campaign from cold start quickly and consistently. User story: As a solo marketer, I can launch the app, tap one action, and create a campaign with only a required free-text brief in under 15 seconds. Acceptance: Empty state is shown on first launch with a clear one-tap CTA; campaign creation works with just one required field; invalid/empty input is blocked with recoverable feedback; canceled creation preserves app state.

Must-have: Import an asset into a selected campaign. User story: As a user, I can add a photo/video/PDF from Photos or Files to a campaign. Acceptance: Import path is available from campaign detail; permission denial shows a recoverable error with Settings deep link; supported files become attached versions; unsupported or corrupt files show a real failure state and no crash; import progress/success is visible.

Must-have: Non-destructive asset versioning with explicit current marker. User story: As a user, every new upload creates vN without deleting or changing historical bytes. Acceptance: New versions append only, never overwrite; current version is clearly surfaced on the asset view; historical versions remain accessible as read-only entries and can be previewed; restoring from old version creates a new version instead of mutating old data.

Must-have: Review state machine with guarded transitions. User story: As a user, I can move an asset through Draft → In Review → Approved / Needs Rework and have each decision logged. Acceptance: “Needs Rework” always requires a non-empty, non-whitespace note; “Approve” is a deliberate confirm action (not single-tap default); transitions respect the state diagram and are never applied in invalid order.

Must-have: Status-history log as source of truth for campaign and asset decisions. User story: As a user, I can review what changed and when. Acceptance: Every user action and key system action (including auto-revert cases) appends a timestamped entry; history survives relaunch; entries are ordered by time and immutable in the app’s local store.

Must-have: Soft-delete with undo and safe rollback of impossible states. User story: As a user, I can delete campaigns/assets and recover within 10 seconds if needed, without orphaning data. Acceptance: Deleted items are moved to a local trash state immediately; undo is available; next launch finalizes stale deletions; deleting the only in-review asset auto-reverts campaign to Draft and logs that system event.

Must-have: Deterministic local persistence and file-backed storage. User story: As a user, my data survives relaunch and does not depend on network. Acceptance: SwiftData/Core Data metadata + file URLs only, no embedded blobs for media; app remains fully usable offline; crash or force-quit during import does not corrupt referential integrity; state recovery is deterministic on restart.

Must-have: Real empty/loading/success/error states on each active screen and flow. User story: As a user, I never see fake or blank shells for app states that can fail or be pending. Acceptance: Campaign list, campaign detail, asset detail, and history screen each present explicit empty and loading states; success states confirm completed operations; error states include recoverable actions where possible.

Should-have: Fast, low-friction UX patterns that match bursty usage. User story: As a user, I can review decisions quickly with one hand. Acceptance: campaign list/detail/asset detail use clear visual hierarchy and minimal taps for core actions; version badge and current-state chip are always visible; optional brief metadata (audience/tone/platform) stays collapsed.

Should-have: QuickLook-based media viewing and basic playback/zoom support via system viewers. User story: As a user, I can inspect current and historical asset files confidently without custom viewers. Acceptance: current version opens in native preview; older versions open read-only; no custom, unsupported media viewer behavior.

Could-have: Voice brief capture, camera capture shortcut, search/filter chips, review queue sorting, and an About/limitations screen that explains single-device risk. User story: As a power user, I can use these speed/clarity upgrades; these should not block the MVP.

Won’t-build: Team collaboration, real-time multi-user sync, social publishing integrations, third-party auth, webhooks/web API automation, and any backend billing/subscription system. User story: none, because these features conflict with single-user offline-first scope. Acceptance: no external accounts, no network-driven source of truth, no API writes.

Won’t-build: Automatic cross-device sync in v1. User story: none, because this reintroduces state-sharing assumptions. Acceptance: all operations are local and deterministic unless user explicitly performs a manual export/import flow in a later phase.

In-scope is the one-loop campaign workflow: create brief → add/import versions → review transitions with guardrails → audit trail → recovery-safe deletes and relaunch-safe persistence. Out-of-scope is anything that needs server collaboration, multi-user coordination, publishing automation, or a business model mechanism that adds gates to the approval loop. Biggest risk in this direction is scope creep from “helpful” adjacent features; the details above must preserve the invariant that the whole app is judged by one successful end-to-end, versioned approval loop on one device, not by feature count.

Photos/Files permission denial producing a recoverable state with a Settings deep link, never a silent no-op — tested by actually denying permission in the simulator and confirming the app doesn't just sit there looking broken. And full persistence across relaunch for the entire cycle, which is really the umbrella acceptance test all the others feed into.

Should/could, meaning nice if time allows but I will not accept schedule slip to protect these: the About/limitations screen disclosing single-device data loss and iCloud device-backup inclusion — small, low-risk, but not load-bearing; validating the app with more than one concurrent campaign, since the object model already supports it and this is really just extra test coverage, not new engineering; audience/tone/platform fields actually rendering in the brief UI rather than sitting as unused schema; basic unsupported-file-type messaging beyond the bare minimum error state.

Won't-build, full stop, no matter how much earlier phases discussed them as future product surface: Free/Pro tiering, any StoreKit or simulated-purchase UI, encrypted backup/export, local notifications, tags/search/filtering, any dashboard/timeline/kanban view beyond the four locked screens, per-version deletion as a distinct action from whole-asset deletion, iPad-specific layout, and obviously anything involving accounts, external APIs, social posting, or a billing backend. My biggest objection to scope creep here isn't philosophical — it's that every one of those features either touches the data model in a way that could quietly break the append-only version guarantee, or adds a screen that then also needs its own empty/loading/error states, and this project has already spent five phases locking down exactly how expensive that get-it-wrong cost is. The detail I most want preserved into the final output is that "must-have" acceptance criteria are all framed as force-kill-and-relaunch tests, not happy-path taps, because that's where this kind of app actually fails in the real world.

I think this is the best balance for getting something real out of this early phase. It’s not pretty, but it proves the core concept—a simple, reliable workflow that can be scaled up later. Let’s focus on nailing this one loop before we start thinking about fancy features.

A solo marketing operator needs one place on the phone where a campaign’s “what,” “what’s changed,” and “what’s approved” are always in sync, because that’s the exact set of things that breaks when they hop between Notes, email, chat threads, and cloud folders. This app is worth existing because it turns a fragmented local workflow into a single, durable decision system: one brief, one current versioned asset set, and an unambiguous local approval state that can’t be silently lost.

The target user is a freelancer, one-person in-house marketer, or agency contractor managing a handful of active campaigns, usually in short bursts (train, couch, hallway between calls), who needs reliable progress without setting up accounts, onboarding teammates, or logging into another web service. The use case is not “run a full social stack,” it is “reduce missed approvals and confusion about what version is current.”

The core loop is five concrete moves: create a campaign with a short free-form brief and optional metadata, attach/import an asset as version 1, iterate new versions that are always appended (never overwritten), move an asset through Draft → In Review → Approved/Needs Rework with required rework notes, then open the status log to verify every transition before the next action. Empty, loading, success, and error states are treated as first-class on every screen so a stalled import, failed write, or edge deletion always has a recovery path and never feels like a silent bug.

Paid value is real when the user avoids at least one costly failure mode each cycle: shipping the wrong asset, guessing which file is latest, or re-reading scattered messages to reconstruct history. In practice that is less “feature count” and more “time saved and fewer client mistakes,” which is exactly where solo operators pay. Subscription value in v1 planning is not cosmetic, it is utility: paid unlocks extra working bandwidth and reliability guarantees—larger campaign and version quotas, advanced local filters/search, stronger retention and archival rules for asset history, and a richer review queue (aging-in-review alerts, reusable briefs/templates, and compact handoff exports).

The competitive wedge is a narrow one: this is not a full SaaS clone; it is a disciplined, on-device workflow engine for solo operators who want the minimum overhead path to approved decisions. Compared with generic project tools, it wins on one thing: version + status certainty in a small footprint, with no login, no team choreography, and no drift between local decisions and the files actually on hand. Compared with photo/content apps, it wins by connecting each asset to campaign and review state instead of treating media as loose files.

A local-first behavior is the default contract: deterministic SwiftData metadata, file-backed media in app storage, strict status transitions, and a local timeline log as the source of truth. This means no network and no remote sharing are required to use the product; the downside is explicit: one-device data-loss risk if the device is lost. The cloud-ready path keeps that behavior intact and adds optional layers later, not at v1: encrypted backup and multi-device restore, then optional cross-device sync as a subscription feature.

The viral or niche growth angle is not social sharing by app mechanics; it is workflow portability. Users naturally share a complete “campaign review package” (approved asset + history export + rework trail) with clients and partners via Files/AirDrop/email. That makes the tool visible in existing handoffs and seeds word-of-mouth inside freelancer and micro-agency communities where this pain is common.

The highest objection to this direction is feature bloat: forcing a social or collaboration product onto a local review loop will reduce reliability and delay delivery. The detail this phase must preserve is the hard boundary that every versioning and state transition is append-only, single-device-usable, and edge-safe, especially around delete during review and orphan prevention.

Free tier: capped at 2 active campaigns and visibility into the 5 most recent asset versions (older versions are hidden behind an "Upgrade to view full history" label, never deleted — the non-destructive versioning guarantee is never violated by monetization). Pro tier: removes both caps (unlimited campaigns, full version history) and adds encrypted local backup/export the user can save to Files/iCloud Drive/AirDrop and restore later — turning the app's own admitted single-device data-loss risk into the actual value a Pro user is buying. Since "billing/subscriptions" is locked as a non-goal (no backend), any purchase mechanism must be on-device StoreKit or an explicitly UI-labeled simulated "Pro unlock" toggle — never a server we run. The review-decision screen (Approve/Needs Rework) is never gated or interrupted by a paywall prompt.

**Core loop:** Create a campaign with a fast free-text brief (under 15 seconds) → upload an asset as v1 → append new versions non-destructively as work iterates → move the asset through Draft → In Review → Approved/Needs Rework (rework requires a non-empty note) → review the permanent, timestamped status history. Identical for Free and Pro, gated only by the caps above.

**Competitive wedge:** Ordinal is a team SaaS — accounts, login, connectivity, likely social/publishing automation. This app is the deliberate inverse: zero setup, zero login, works offline/on a plane, and — the sharpest pitch — a client's unreleased creative work literally cannot leak because there's no server to breach. That's a real value proposition for freelancers holding NDA'd pre-launch assets, not a workaround for missing a backend.

**Viral or niche growth angle:** Not viral — there's no second user to invite. Growth is niche word-of-mouth among freelancers and micro-agencies, driven by Pro users exporting a clean, presentable "campaign review package" (asset + version + approval status) to their own clients via Files/AirDrop/email. That artifact functions as an organic advertisement — someone on the receiving end asking "what tool is this" is the whole growth loop, so the export needs to look genuinely professional, not like a debug dump.

**Local-first and cloud-ready plan:** v1 ships fully local — SwiftData metadata, file-backed asset bytes, zero network calls of any kind, consistent with every earlier locked phase. The cloud-ready path is strictly user-controlled manual export/import of an encrypted backup file (Files/iCloud Drive) — no accounts, no server, no automatic sync, so it doesn't quietly reopen the multi-device-collaboration non-goal. Any future automatic CloudKit-style sync would need to be explicitly proposed and justified later, held to the same bar as introducing a third-party package.

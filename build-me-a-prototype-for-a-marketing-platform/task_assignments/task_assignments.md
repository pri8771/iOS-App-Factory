**Claude (Backend / Systems Engineer · Pragmatist) — Round 1**

Building on the locked `tech_specs`/`project_plan` — my job this round is to turn that into concrete file ownership and a task backlog that doesn't require anyone to guess who touches what. I'm going to be blunt about the one place I think this plan will actually bleed time if we don't nail it down now: the Xcode project file itself.

**Where the app lives:** every sibling app in this factory follows the same pattern — a slug folder at the portfolio root with an `app_build/` subfolder inside it. This run's slug folder is `build-me-a-prototype-for-a-marketing-platform` (one level above `.agent_cwd`), and `portfolio_selection` already decided this is a single app, not a multi-app split, so there's no new `marketing-campaign-cockpit` folder to create. The app goes at `build-me-a-prototype-for-a-marketing-platform/app_build/`, matching every other app in this repo. Not up for debate — it's just following existing convention.

Inside `app_build/`: three local SPM packages (`Packages/CockpitDomain`, `Packages/CockpitData`, `Packages/CockpitServices`) plus the app target (`MarketingCampaignCockpit/App`, `MarketingCampaignCockpit/Features/*`, `MarketingCampaignCockpit/Resources`) plus two test targets (`MarketingCampaignCockpitTests` for host-app unit tests, `MarketingCampaignCockpitUITests` for XCUITest).

**Lane-to-package mapping is mostly clean, with one deliberate override.** `data_domain` owns `CockpitDomain` (state machine, derivation, validators, enums) and `CockpitData` (the four `@Model` types, repositories). `services_utilities` owns `CockpitServices` (file store, import coordinator, thumbnailing, soft-delete/reconciliation). `primary_ui` owns the app target — shell, environment, routing, all four screens, two sheets, shared components. `polish_resilience` owns cross-cutting utilities (`ClockService`, `ErrorPresenter`, `AppLogger`, `AccessibilityLabelFactory`), the About/limitations screen, and the dedicated verification suites.

The override: `tech_specs` nominally assigned `CockpitSchemaV1`/`SchemaMigrationPlan` to `polish_resilience`, but that's the same file, same day of work, same package as the `@Model` types `data_domain` is already writing in `CockpitData`. Splitting a nine-line migration-plan stub across two lanes just to honor a label from an earlier phase buys us nothing but a needless handoff — I'm folding it into `data_domain`'s task instead and giving `polish_resilience` the four genuinely separable utility protocols instead. This doesn't reopen the architecture, it's just who literally types the file.

**The real conflict risk nobody's named yet:** with four lanes adding ~35+ new Swift files into one Xcode project, `project.pbxproj` is a textbook merge-conflict magnet if it's a classic file-reference-list project — every lane adding a file edits the same XML blob. The fix costs nothing: set up the project in T-001 using Xcode 16's file-system-synchronized groups (folder references), so adding a `.swift` file to a folder on disk doesn't touch the project file at all. Local SPM packages don't need `Package.swift` edited to add files either, only to add new targets or dependencies — which shouldn't happen after T-001 since we're at zero third-party dependencies by contract. If both of those hold, no lane should ever need to open `project.pbxproj` or any `Package.swift` after the scaffold task lands. That's the actual conflict-prevention mechanism, not just "please don't step on each other."

**Sequencing** follows the locked milestones directly: scaffold (T-001) is a single blocking task, one owner, before anything else starts. Domain (T-002) unblocks data models (T-003) unblocks repositories (T-004). File store (T-005) can start in parallel with domain since it only needs the empty scaffold. Import coordinator and soft-delete (T-006, T-007) need real repositories, not mocks. App shell (T-009) is its own checkpoint — the first point every lane's output actually links together — before the four screens fan out for real. Verification (T-017) is gated behind all of it, not squeezed in at the end.

**Forbidden zones, stated plainly:** nobody outside `data_domain` edits anything under `Packages/CockpitDomain/` or `Packages/CockpitData/`. Nobody outside `services_utilities` edits `Packages/CockpitServices/` except the four named files that belong to `polish_resilience` (`ClockService.swift`, `ErrorPresenter.swift`, `AppLogger.swift`, `AccessibilityLabelFactory.swift` — same folder, different owner, no other lane touches those four). Nobody outside `primary_ui` edits `MarketingCampaignCockpit/App/` or `Features/`. Accessibility instrumentation doesn't get done by `polish_resilience` reaching into screen files after the fact — `primary_ui` calls `AccessibilityLabelFactory` directly while building each screen, so the audit task only ever adds new test files, never edits someone else's view code.

**How we know it works:** unchanged from the locked plan — every task's acceptance criteria is a runnable test, not a tap-through. The one thing I'd fight for here: T-017 (the relaunch/integration suite) is the actual product claim, and it doesn't get scheduled until T-007, T-011–T-014 are all real, not stubbed.

*   **File Storage:** We’ll use a simple in-memory file storage for initial development. Let’s prioritize building a *functional* MVP, and worry about the perfect architecture and elaborate testing strategy later. It’s far more productive to get something working than to spend weeks designing the ideal architecture before we’ve even created a single usable asset upload.

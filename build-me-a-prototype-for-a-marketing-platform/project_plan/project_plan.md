**Claude (Backend / Systems Engineer · Skeptic) — Round 1**

I'm going to push hard on one thing nobody in this room has caught yet, because it's a real compile-time bug hiding inside the locked interfaces, not a stylistic quibble: `CampaignRepository` and `AssetRepository` are both declared `Sendable` actor-backed protocols, but `fetchCampaignList() throws -> [Campaign]` and `fetchAssets(for:) throws -> [Asset]` return arrays of the `@Model` classes themselves. SwiftData's `@Model` types are not `Sendable` — Apple doesn't make them so, deliberately, because they're tied to a `ModelContext`. Under Swift 6 strict concurrency (which a fresh 2026 Xcode project defaults to), that repository signature will not compile, or it'll compile only if someone quietly slaps `@unchecked Sendable` on `Campaign`/`Asset`, which reintroduces exactly the cross-actor mutation bugs the whole architecture was designed to prevent. Notice that `ReviewEventRepository` already solved this correctly — it returns `ReviewEventSnapshot`, a plain `Sendable` struct, never the `@Model` itself. That pattern just wasn't applied consistently to the other two repositories. This has to get resolved as a concrete decision in this plan, not discovered by whoever's assigned the data_domain lane three days in: either strip the `fetch*` methods off the actor-isolated repositories entirely and make `@Query`/`@Bindable` in views the only read path (which is actually what the design already implies, since `AppEnvironment` is described as feeding live `@Query` observers), or add `CampaignSnapshot`/`AssetSnapshot` DTOs mirroring `ReviewEventSnapshot`. I'd push for the first option — it's less code and it matches the "views never touch ModelContext directly, but @Query still needs to observe" story already locked. Either way, this needs a named task before repository implementation starts, because every call site downstream assumes one of these two shapes.

On milestones, sequenced strictly by what actually depends on what, not by the four "owning lanes" as if they're four equally-parallel tracks — they aren't:

**M1 — CockpitDomain.** State machine, worst-status derivation, note validation, the enums. Zero dependencies beyond Foundation, so it can start the instant M0 exists. **M2 — CockpitData.** `@Model` types, `CockpitSchemaV1`, repository actors. Depends on M1 (consumes `ReviewStatus`/`ReviewAction` directly) — this is not parallelizable with M1, it's sequential. The Sendable-boundary decision above must be made and written into the repository protocol signatures before this milestone's implementation work begins, or it gets rewritten at M4 under time pressure. Gate this milestone by compiling and testing under Swift 6 strict concurrency explicitly — turn the checker on now, don't let it surface as a surprise later.

**M3 — CockpitServices.** File store, `ImportCoordinator`, `SoftDeleteCoordinator`, thumbnailing. **M4 — app shell + `AppEnvironment` wiring.** This deserves to be its own milestone, not folded into "start of UI work," because it's the first point where independently-built lanes actually have to link against each other. **M5 — the four screens + two sheets.** Only now can this genuinely fan out in parallel, since M4 gives a stable, real environment to build against. Historically, this is exactly where a project discovers that every unit test in M1–M5 passed in isolation and the integrated system still doesn't survive a real relaunch — that gap is the entire reason this product exists, so it can't be compressed into "final polish."

**M7 — accessibility + state-coverage audit + About/limitations disclosure copy.** Should/could-tier triage: build only if M0–M6 land with schedule slack, per the "smaller finished app over larger half-built app" rule.

## Final Output

**Screen inventory:** Four real screens — Campaign List, Campaign Detail, Asset Detail, Status History — plus two ephemeral sheet overlays: Create Campaign/Brief and Rework Note. No fifth screen, no dashboard, no settings tab (an About/limitations screen remains should-have from earlier phases, not required here).

**Primary user flow:** Cold launch → empty Campaign List with a single one-tap "Start your first campaign" CTA → brief sheet (one required free-text field, keyboard focused, save enabled the instant there's non-whitespace text) → lands directly on the new Campaign Detail (never back on the list) → tap the persistent Add Asset action → system Photos/Files picker → Campaign Detail shows an upload-in-progress row → open Asset Detail → Submit for Review (status chip updates in place) → Needs Rework with a required trimmed non-empty note (visible immediately in Status History) → Add Version → v2 becomes current, v1 slides into the read-only version strip → Submit for Review again → Approve via deliberate two-tap confirm → Status History shows the full ordered, timestamped story. Deleting the only in-review asset auto-reverts the campaign to Draft with a logged system line and a soft-delete/undo toast. Every step reachable without hopping across more than two screens at once.

**State model per screen:**
- **Campaign List:** booting (skeleton rows, not a spinner) / empty (CTA) / loaded (cards with worst-status chip + last-activity time, swipe-to-delete with undo) / error (explicit "something's wrong with your data" + retry, never a silently empty list).
- **Campaign Detail:** loading / no-assets-yet (distinct empty copy from Campaign List's empty state) / has-assets (list, not grid, with thumbnail + version badge + status chip per row) / uploading-import-in-progress (inline progress row) / import-error (inline retry on the failed row).
- **Asset Detail:** loading-version-metadata / draft-pre-review (Approve/Rework grayed out, single "Submit for Review" CTA shown instead) / preview-ready (QuickLook pinned top, version badge, read-only prior-version strip) / preview-failed (broken-file icon + "Replace" action, never a blank pane) / review-wait / rework-note-required (submit disabled until non-whitespace) / approve-confirm-pending (second-tap confirm sheet) / undo-delete-state.
- **Status History:** loading (skeleton rows) / empty (new asset, "No activity yet") / listing (timeline rows, icon-keyed by event type, rework notes inline, system lines visually distinct from user actions) / error (reload-failed + retry).

**Visual direction:** Native iOS, not a cross-platform reskin — system fonts, SF Symbols for every icon, native list + swipe actions, native `.confirmationDialog`/sheets for confirms. One consistent status color mapping used identically everywhere it appears (gray=Draft, blue=In Review, orange=Needs Rework, green=Approved). Version chips (`v2 of 2 · Current`) and status chips always pair color with text/icon. Approve and Needs Rework are deliberately different visual weights so they can never be confused under thumb, one-handed, in motion.

**Accessibility notes:** Every status/version chip carries a full-state accessibility label (not just a colored dot); Dynamic Type must reflow gracefully on Asset Detail specifically (the densest screen) without clipping; VoiceOver gets explicit announcements on state transitions ("Moved to In Review," "Needs rework note required"); confirm sheets trap focus so VoiceOver can't swipe past into background content; the read-only old-version banner is announced, not just visually shown; color is never the sole signal anywhere; contrast is explicitly checked for orange-on-white (Needs Rework) in both light and dark mode.

**Named, deliberately accepted risk carried forward:** a list-based Asset Detail and text-heavy Status History are optimized for the locked MVP scale (one campaign, one asset, full cycle) — density at multi-asset scale is an explicitly deferred problem, not solved here.

### Screen-by-screen design specification

**Navigation shell:** Single `NavigationStack` with a typed route enum (`campaignDetail(id)`, `assetDetail(id)`, `statusHistory(scope)`). No tab bar, no independently-rooted views. The three overlays — Create Campaign/Brief, Rework Note, Approve confirm — are `.sheet`/`.confirmationDialog` modifiers on local state, never pushed onto the nav stack, so dismissing them never leaves stray back-stack entries. On every relaunch, the app cold-boots to Campaign List at the root rather than attempting to restore mid-stack state — deliberately simpler and safer than validating whether a deep-linked asset/campaign still exists after backgrounding.

**1. Campaign List** (home root)
- Layout: lightweight context line ("No campaigns yet" / "3 active"), then campaign rows, then a persistent primary "New Campaign" action.
- Row: brief-first text (one line, truncated), one aggregated status chip using the *worst*-status rule (if anything under this campaign is Needs Rework, that's what shows), relative last-activity timestamp, chevron. Swipe-to-delete triggers soft-delete + undo toast.
- States: **booting** (fixed-count skeleton rows, no spinner) → **empty** (single sentence + one big CTA, "Start your first campaign") → **loaded** (rows, worst-status sorted) → **error** ("Couldn't read local data" + retry, never a silently empty list).

**2. Campaign Detail** (opens directly after campaign creation — never routes back through the list first)
- Layout: brief field at top, inline-editable (`TextField` bound directly to the model, committed on blur — no separate edit mode, no separate save button), compact status summary, asset list section (list, not grid, for v1), Add Asset action pinned in the toolbar so it survives scroll.
- States: **loading** → **no-assets-yet** (copy distinct from Campaign List's empty state: "No assets yet. Add your first file to start the review.") → **has-assets** (thumbnail + version badge + status chip per row) → **import-in-progress** (inline progress row on the importing item) → **import-failed** (inline retry on the failed row, never a vanishing toast).

**3. Asset Detail** (the emotional center of the app — most state-dense screen, built as separable components: preview, version strip, actions)
- Layout: large QuickLook-backed preview pinned at top (`UIViewControllerRepresentable` wrapper, native zoom/pan/playback, no custom viewer) → version badge ("v2 of 2 · Current") + status chip directly under it → horizontal read-only strip of prior versions (tapping one opens QuickLook in an explicitly labeled "Viewing v1 (read only)" mode) → action strip, state-dependent: Draft shows only "Submit for Review"; In Review shows Needs Rework (single tap → required trimmed-non-empty note sheet, submit stays disabled until real text exists) and Approve (two-step: tap opens confirm sheet, second tap commits) — these two buttons are deliberately different visual weights so they can never be confused one-handed.
- States: loading-version-metadata / draft-pre-review (Approve/Rework grayed out) / preview-ready / preview-failed (broken-file icon + "Replace," never a blank pane) / review-wait / rework-note-required / approve-confirm-pending / undo-delete-countdown / **removed-while-viewing** (new — item was soft-deleted and the undo window lapsed while this screen was open; shows a terminal "this was removed" state and auto-pops to Campaign Detail after a short delay).

**4. Status History** (lowest bug surface in the app — keep it that way, no tap-to-expand or filtering for v1)
- Layout: flat chronological list, icon-keyed by event type, timestamp + action label + note body shown inline (never hidden behind a tap). System-generated lines (e.g. auto-revert-to-Draft) are visually distinct (neutral/gray icon) from user actions (colored icon).
- States: loading (skeleton rows) / empty ("No activity yet") / listing / error (reload-failed + retry).

**Sheets:** Create Campaign/Brief — single required free-text field, keyboard auto-focused, save enabled the instant non-whitespace text exists. Rework Note — one multiline field, submit blocked until trimmed content exists. Permission-denied paths (Photos/Files) route to an explicit recovery state with a Settings deep link, never a silent no-op.

### Design tokens / visual system

- **Color (semantic, never raw hex):** `statusDraft`=system gray, `statusInReview`=system blue, `statusRework`=system orange, `statusApproved`=system green; `surfacePrimary`/`surfaceSecondary` mapped to `.systemBackground`/`.secondarySystemGroupedBackground` (dark mode free); `accentAction` for Approve's high-weight button; `destructive` (system red) for delete swipe actions only. Color is never the sole signal — every chip pairs color with text + SF Symbol.
- **Typography:** fully Dynamic-Type-driven, no fixed point sizes — `.largeTitle` screen titles, `.headline` row primary text, `.subheadline` metadata/timestamps, `.caption` version badges.
- **Spacing/shape:** 8pt grid (8/16/24/32); 10pt corner radius on cards/chips; `.capsule` chips; minimum 44×44 touch targets.
- **Motion:** restrained and mostly system-default — 220ms skeleton-to-content fade, 180ms status-chip cross-fade on state change plus a light haptic, standard iOS push/pop transitions, native sheet presentation for all confirms (no custom overlays that can trap focus). The one deliberate exception: a brief 300ms success pulse on the status chip the instant Approve commits, since that's the emotional payoff moment the design phase already flagged as needing real weight.

### Component inventory

`StatusChip` (state → color+icon+text+full accessibility label, single source of truth so no call site can break the "color isn't the only signal" rule), `VersionBadge`, `CampaignRow`, `AssetRow`, `HistoryEntryRow` (user vs. system visual variant), `QuickLookPreview` wrapper, `PrimaryActionButton` (Approve, filled/prominent) and `SecondaryActionButton` (Needs Rework, outlined/lower-weight) as genuinely separate components — not one button with a style flag — so their visual weights can't accidentally converge in a later edit, `UndoToast`, `EmptyStateView`, `ErrorStateView`, `SkeletonRow`/`SkeletonList`, `ProgressImportRow`, `ReadOnlyVersionStrip`, `StateBanner` ("Viewing old version, read-only").

### State and interaction notes (non-negotiable)

Versioning is strictly append-only, current version always explicit. Rework notes are trimmed before validating — whitespace-only is rejected. Approve is always a deliberate two-step confirm, never a single tap. Delete is soft-delete (immediate flag + move to trash, not a deferred timer) with undo guaranteed to survive backgrounding within the window; a background sweep on next launch finalizes anything past its window. Deleting the only In Review asset auto-reverts its campaign to Draft and appends a system-generated history line. Data flows through live SwiftData `@Query`/`@Bindable`, so system-generated mutations propagate to every on-screen observer automatically — including the newly-identified case where the screen you're standing on is the thing that just got removed. Asset thumbnails generate asynchronously via `QLThumbnailGenerator` with an in-memory `NSCache`, never blocking list scroll.

### Accessibility requirements

Every chip/badge/button exposes a full-state accessibility label composed explicitly ("Status: Needs Rework," "Version 2 of 2, current," "Approve pending confirmation") rather than relying on auto-derivation. Dynamic Type must reflow without clipping everywhere, stress-tested specifically on Asset Detail as the densest screen. VoiceOver focus order on Asset Detail: preview → version badge → version strip → primary action → secondary action. Confirm sheets trap focus. The read-only-version banner and inline-editable brief field both announce their state/interactivity explicitly to VoiceOver. Contrast is explicitly checked for orange-on-white (Needs Rework) in both light and dark mode.

### External design decision

**No external design handoff by default.** The spec above is already builder-ready — tokens are system-semantic, components map directly to SwiftUI primitives, and there's no brand-identity or custom-illustration work that would benefit from a visual-design round-trip; the risk is that an external pass either ignores the locked nine-state Asset Detail machine or drifts from it. If a one-shot gut-check is still wanted before build, the only thing worth commissioning is a static, non-interactive mockup of Asset Detail in its preview-ready state, to sanity-check the QuickLook-plus-badge-plus-asymmetric-buttons layout at a glance. If that path is taken, use this prompt:

> "Design a SwiftUI iOS mobile app called Marketing Campaign Cockpit, single-screen-first offline workflow with exactly 4 primary screens: Campaign List, Campaign Detail, Asset Detail, Status History plus Create Campaign sheet, Rework Note sheet, and delete-undo toast. Implement state-aware layouts and explicit empty/loading/error states for all workflow surfaces; one-actor local workflow with deterministic append-only status history and versioned assets. Include status mapping chips, asynchronous import error/retry states, two-step approve confirmation, trimmed rework-note validation, soft-delete undo timer behavior, and accessibility-first behavior with VoiceOver labels and Dynamic Type support. Do not add iPad-only patterns, external integrations, dashboards, or paywalls. Output componentized SwiftUI screens with modifiers and state enums."

## Final Output

**SwiftUI architecture recommendation:** Three-layer split — SwiftUI views/view models own UI state only; a small domain layer (plain Swift types/services, not embedded in view bodies) owns the review-state-machine, version-current-pointer, and worst-status-derivation rules so they're unit-testable without instantiating a view hierarchy; a storage layer owns exactly two things — SwiftData for structured metadata/object graphs via a single `ModelContext` boundary plus a repository actor for writes, and the filesystem for asset bytes. `ModelContext` is main-actor-bound; any file I/O (copies, deletes) happens in a background `Task` and only touches the `ModelContext` once that I/O is confirmed complete — never block the UI thread on disk work, never mutate SwiftData off the main actor.

**Data model to freeze:** `Campaign` (owns many assets) → `Asset` (owns ordered, append-only `AssetVersion` rows plus a `currentVersionID` pointer that's the only thing that changes on a new upload) → `AssetVersion` (immutable once written) → `ReviewEvent` (immutable, append-only, drives Status History) → soft-delete represented as an explicit state flag + scheduled-deletion timestamp on the relevant entity, never an immediate row removal.

**Apple framework choices:** SwiftUI + `NavigationStack` (per the locked design_handoff), SwiftData (`@Model`, `@Query`, `@Bindable`) for metadata, PhotosUI (`PhotosPicker`) and `.fileImporter`/`UIDocumentPickerViewController` for asset intake, `UniformTypeIdentifiers` to validate imported files are genuinely the image/video/PDF type they claim before accepting them as a version, QuickLook (`QLPreviewController` via `UIViewControllerRepresentable`) and `QLThumbnailGenerator` (cached in an in-memory `NSCache`) for preview/thumbnails, `FileManager` for atomic file writes (temp file → rename), `OSLog` for diagnostics. Explicitly excluded and justified as out of scope: StoreKit (no purchase flow — tiering was deferred in `app_features`), CloudKit (no sync — single-device by design), Vision/CoreML/ARKit (no task in this app needs them; thumbnailing is a rendering utility, not ML), `BGTaskScheduler`/push (the soft-delete finalize sweep runs synchronously on next foreground launch, needing no background entitlement).

**Persistence and local data plan:** Asset bytes are file-backed in `Application Support/marketing_assets/...` (not Documents — keeps users from renaming/deleting backing files via the Files app and breaking referential integrity), written atomically (temp file, then rename), with collision-safe immutable names keyed to asset/version IDs. SwiftData holds only metadata + relative path references. Fixed ordering rules to prevent dangling pointers: on import, the file write must succeed before the SwiftData record is created; on delete-finalize, the SwiftData record is deleted before the backing file. Either direction, an interrupted mid-operation death leaves an orphaned file (accepted, already-named unbounded-storage limitation) rather than a dangling metadata pointer needing defensive handling on every screen. A real `Schema` + `SchemaMigrationPlan` is defined from day one, even if trivial/empty at v1, since a failed store open with no backend and no sync means total, unrecoverable data loss. On launch, a startup reconciliation pass finalizes any pending imports/deletes left incomplete by a prior force-kill.

**Permissions/privacy plan:** Request-on-first-use only, nothing preemptive. The must-have import path (`PhotosPicker`/`.fileImporter`) requires **no** photo-library or file-system permission prompt at all — it runs out-of-process, so the "permission denied → Settings deep link" recovery state only has a real trigger if/when camera capture (should/could tier) is added, requiring `NSCameraUsageDescription`. What the import path does need is correct security-scoped resource handling: `fileImporter` URLs are only valid inside a `startAccessingSecurityScopedResource()`/`stopAccessingSecurityScopedResource()` window, so the copy into app storage must happen synchronously inside that window. No network calls, no background collection, no analytics/remote logging SDKs anywhere in the app. Standard iCloud device backup is left enabled and disclosed (not app-managed sync) per prior phases' decision.

**Dependency/license decision:** Zero third-party packages. Every requirement maps cleanly onto the Apple frameworks listed above, so the permissive-dependency policy is satisfied trivially — there's nothing to license-check. If a dependency is ever proposed later, it must be justified explicitly in-contract with a clear gap Apple's frameworks can't fill.

**Testing implications:** Domain logic (state transitions, whitespace-trimmed rework-note validation, worst-status derivation, auto-revert-on-delete) lives as plain Swift and is unit-tested against an in-memory `ModelConfiguration(isStoredInMemoryOnly: true)` SwiftData container — fast, disk-free coverage of the force-kill-equivalent scenarios by reopening a fresh container against the same on-disk store mid-sequence and asserting state. Integration tests cover file-import copy atomicity, stale-pointer cleanup, and launch-time recovery after interrupted writes. UI tests use `XCUIApplication.terminate()` followed by a fresh `launch()` as the explicit, named mechanism for the actual force-quit/relaunch acceptance criteria this whole project is built around — not a proxy or simulation. QuickLook presentation always does a synchronous file-existence/non-zero-size check immediately before presenting, routing to the existing preview-failed state on failure rather than trusting the SwiftData record's mere existence.

**Acceptance checks satisfied:** The app is buildable natively with no vague architecture gaps — every screen/state from `design_handoff` maps to a named framework and a named persistence rule. No ML/LLM/AR is used anywhere, correctly, since nothing in this app's locked scope needs it. No dependency is introduced, so the permissive-license policy is trivially satisfied.

## Final Output

**Architecture overview:** Four-layer split, enforced by real module boundaries rather than folder convention. `CockpitDomain` (Foundation-only, zero SwiftData/SwiftUI imports) owns the review state machine, worst-status derivation, and note validation as pure functions — this is the layer whose correctness the whole product's trust depends on, and it must be testable with a sub-second `swift test` run, no simulator. `CockpitData` owns the five SwiftData `@Model` types and actor-backed repository implementations, translating internally between plain `UUID`s (what the domain layer speaks) and SwiftData's `PersistentIdentifier` so the domain layer never has to know SwiftData exists. `CockpitServices` owns the filesystem — atomic asset-file writes, QuickLook thumbnailing, soft-delete/reconciliation, import ordering. The app target itself stays thin: SwiftUI views, a `Route`-driven `NavigationStack`, and an `@Observable AppEnvironment` built once at launch and injected via `.environment(_:)` so no view ever touches a `ModelContext` directly. This is a refinement of Codex's protocol set, not a departure from it — Claude's package split is how those same protocols get compiler-enforced isolation instead of just being organized by convention.

**File/module layout:** `CockpitDomain` (state machine, derivation, validation — no platform imports), `CockpitData` (SwiftData `@Model`s + repository actors), `CockpitServices` (file store, import coordinator, thumbnail cache, soft-delete coordinator), app target (views, view models, `AppEnvironment`, `Route`).

**Data models (frozen):** `Campaign` → `Asset` (holds `currentVersionID`, the only field a new upload changes) → `AssetVersion` (immutable once written, **file-backed via `relativeFilePath`, never embedded bytes** — this rejects Gemma3's `content: Data` approach as a direct contradiction of the locked `ios_architecture_review` decision) → `ReviewEvent` (append-only, carries `actor: .user/.system` as a real field, not inferred at render time, backing Status History's visual distinction). Soft-delete is `isTrashed` + `scheduledDeletionAt: Date?` on the relevant entity, never a row removal.

**Persistence strategy:** SwiftData repositories are actors wrapping background contexts; the main-actor `ModelContext` is read-only via `@Query`/`@Bindable`. Import order: file write succeeds and is confirmed before the SwiftData record is created. Delete order: SwiftData record is deleted before the backing file. `ImportCoordinator` and `SoftDeleteCoordinator` are the single enforcement points for those orderings, not something every call site re-derives. A real (if trivial) `VersionedSchema`/`SchemaMigrationPlan` ships from day one. `SoftDeleteCoordinator.reconcile(now:)` runs on cold launch *and* on every `scenePhase` transition to `.active` — the fix for the background-without-force-quit gap nobody had caught before this round.

**Testing strategy:** `CockpitDomain` gets a pure Swift Testing suite, no SwiftData, disk-free, proving every transition guard and edge case (double-approve, blank-note rework, auto-revert-on-delete) at CI speed. `CockpitData` gets XCTest against `ModelConfiguration(isStoredInMemoryOnly: true)`, including the reopen-a-fresh-container-mid-sequence trick to approximate force-kill. `CockpitServices` gets integration tests against a real temp `Application Support` directory. The app target gets XCUITest using literal `terminate()` + fresh `launch()` for relaunch-survival claims, plus a background-not-kill test targeting the `scenePhase` reconciliation path specifically.

**Acceptance checks satisfied:** parallel build lanes have a fully-typed, non-overlapping contract to code against (data_domain/primary_ui/services_utilities/polish_resilience), and the spec names concrete build and verification mechanisms for every claim made in earlier phases.

**Claude (QA & Risk · Skeptic) — Round 1**

Let me dig into what could quietly break in this build, since nobody's really stress-tested the mechanics yet.

Start with the version-cap interaction that never got fully specified: the per_app_product_brief locked "Free tier hides versions older than the 5 most recent behind an upgrade label," but the next_steps_small phase cut Free/Pro tiering entirely from MVP. That's fine as a sequencing decision, but it means the MVP data model still needs to store unlimited version history even though the UI won't gate it yet — otherwise when tiering gets added later, hidden history has to already exist rather than being retrofitted. I'd write this down explicitly now: version storage is unbounded from day one regardless of what UI gate exists later, because retrofitting "oh wait we need to keep versions we were previously allowed to discard" is a data-model change nobody wants to make after ship.

On deletion mechanics specifically, I don't think "safe/undoable deletion" has actually been pinned down at the level a build needs. What does deleting a campaign do to its assets' underlying files? If undo is a 10-second toast, is the file physically deleted immediately and undo just restores from a trash copy, or is deletion itself deferred for 10 seconds and only committed after the window lapses? These are different implementations with different failure modes — if you defer the actual file delete, what happens if the app is killed or backgrounded during that 10-second window? Does undo survive relaunch, or does killing the app silently finalize the delete before the user meant it to? My answer: undo must be implemented as "soft delete" (a status flag + moved-to-trash location), not a deferred timer that races against process death. The file physically moves to a trash folder, the timer is UI-only for showing the toast, and on next launch anything in trash past its window gets swept for real. That way an app kill mid-undo-window never causes premature real deletion or a resurrection bug.

Related edge case nobody's named yet: can a user delete the *last remaining version* of an asset, not just delete the asset object itself? If versions are meant to be non-destructive and append-only, does the version history screen even expose a "delete this version" action, or is deletion only ever at the asset level (delete the whole asset, all versions go together)? I'd resolve this now: individual versions are never independently deletable — only whole assets can be deleted (trashed), because allowing per-version deletion breaks the "append-only, never rewrite history" guarantee that's been locked since the very first phase. That closes a gap that would otherwise become an ad hoc call mid-build.

Storage and file-system risk: Documents-directory files are iCloud-backed by default on iOS unless explicitly excluded via `isExcludedFromBackup` or by using an Application Support subdirectory. This directly touches the locked privacy claim ("no remote data sharing") — if asset files silently get included in the user's iCloud device backup, that's technically still "no app-initiated network calls," but a security-conscious client-work-holding freelancer might reasonably expect "nothing leaves this device" to mean no iCloud backup either. I think the honest answer is: allow standard iCloud *device* backup (that's Apple's own backup pipeline, not something we built, and excluding an entire user's creative work from their own phone backup is arguably worse for them), but this needs to be a named, disclosed decision, not a silent default — belongs in the About/limitations screen when that gets built, and should be called out plainly in this phase's data/privacy output rather than assumed.

The 15-second campaign creation bar and the "cold launch to created campaign" claim both implicitly assume the Photos/Files permission prompt goes smoothly. What happens the first time a user attempts asset import and taps "Don't Allow" on the photo-library permission dialog? That's a guaranteed real-world path, not an edge case — I'd want an explicit recoverable state here: a clear in-app message plus a deep link to Settings, not a silent no-op or a crash from force-unwrapping a nil picker result. Same for a corrupt or unsupported file type slipping through the picker (a .heic variant QuickLook can't render, a zero-byte file, a video with no readable duration) — the import step needs a defined failure state, not an assumption that whatever the picker returns is always valid.

Timestamp integrity is worth naming as a real, if narrow, risk: the entire product pitch of the status-history log depends on transitions being provably ordered. If a user changes their device clock (accidentally or on purpose) between two status transitions, the log could show an "Approved" entry with an earlier timestamp than the "Submitted for review" that preceded it. I don't think this needs a fix — it's a genuinely rare edge case for a single-device local app — but it should be a named, accepted limitation rather than silently assumed away, since "the audit trail is trustworthy" is close to the core value prop.

One more gap: the rework-note validation has been stated as "non-empty," but nobody's said whether a string of only whitespace or newlines counts as empty. I'd resolve that now too — trim whitespace before validating, so a user can't satisfy the requirement by hitting spacebar once.

Final assumption I want on record given my seat: the Free/Pro gate is real product surface that was explicitly deferred out of MVP, but the data layer must still be built as if version/campaign caps are coming, so nothing here becomes an undocumented retrofit. Everything else — iCloud device-backup inclusion, permission-denial recovery paths, soft-delete-not-deferred-delete as the undo mechanism, asset-level-only (never version-level) deletion, and clock-integrity as a named-not-fixed limitation — are the concrete answers I'd lock rather than leave open for whoever writes the data model next.

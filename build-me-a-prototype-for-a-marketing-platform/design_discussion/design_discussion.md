**Product Interpretation:**

We’re building a *minimal* campaign workflow. It’s not about a polished user interface or a fully-featured campaign management system. 2.  **Asset Import & Storage:** We *must* implement a mechanism for importing assets – specifically images – directly from the user’s photo library. The app must then save this image locally to the app’s storage, automatically naming it based on the campaign name and version number (e.g., “CampaignName_v1.jpg”). This is fundamental.

3.  **Version Control:** Asset versioning is *critical*. We need to track changes to each asset – essentially, adding a new version whenever a user modifies the existing asset.  The app must handle appending new versions without overwriting older ones. The current version marker is also *essential*.

4.  **Review & Approval:** The workflow must allow for a simple “approved” or “needs rework” state. The user must be able to mark an asset as approved with a single tap.  A text field must be present for documenting the reason for needing rework. This needs to be mandatory for any rework.

5.  **Status Logging:** A chronological status log must be maintained for each asset, documenting every transition (Draft → In Review → Approved/Rework) with timestamps. This log *must* be accessible and persistent – it’s the audit trail.

6.  **Storage (Non-Negotiable):** This *must* utilize SwiftData for local data persistence. SQLite is the best choice and should be integrated from the outset. No server-side storage; we’re building a local, offline tool.

**Non-Goals (Absolutely Critical):**

*   **No Team Collaboration:** This is *not* a tool for shared campaigns. The entire system must be designed for a single user.
*   **No External Integrations:** No integrations with social media platforms, marketing automation tools, or any other external services.
*   **No Automated Actions:** The app should *not* perform any actions automatically—no scheduling posts, no sending emails, nothing.
*   **Limited Permissions:** The app requires read access to the user’s photo library to import assets. The UI must clearly state this.

**Competitive Wedge:** This app’s core value isn’t fancy features; it’s *reducing friction*. It’s a quick and reliable way for a creative professional to get an idea into a workable asset—one that won’t get lost in a messy workflow. It’s about reliably transitioning an asset from concept to approved state.

**Local-First & Cloud-Ready:** While this phase is strictly local-first, we need to build in a path for a future cloud-enabled version. This means designing the system with extensibility in mind – future integrations, syncing, backups – all must be considered from the outset.

**Decision Rules (For Future Phases):** The biggest risk here is over-complicating the workflow. We *must* maintain a minimal, focused design. Any additions – such as campaign templates, advanced version control, or user permissions – will significantly increase the complexity and risk of failure. We also need to ensure the data model is well-designed, scalable, and adaptable for future changes.

**Risk Register (Initial):** The biggest risk is data loss if the app crashes or the device is lost. We *must* have a robust backup and restore mechanism (though this is not required for this phase). Another risk is inconsistent data. We need to ensure the system is well-tested to prevent data corruption or misinterpretation.

- **Asset Detail:** loading-version-metadata / draft-pre-review (Approve/Rework grayed out, single "Submit for Review" CTA shown instead) / preview-ready (QuickLook pinned top, version badge, read-only prior-version strip) / preview-failed (broken-file icon + "Replace" action, never a blank pane) / review-wait / rework-note-required (submit disabled until non-whitespace) / approve-confirm-pending (second-tap confirm sheet) / undo-delete-state.
- **Status History:** loading (skeleton rows) / empty (new asset, "No activity yet") / listing (timeline rows, icon-keyed by event type, rework notes inline, system lines visually distinct from user actions) / error (reload-failed + retry).

**Visual direction:** Native iOS, not a cross-platform reskin — system fonts, SF Symbols for every icon, native list + swipe actions, native `.confirmationDialog`/sheets for confirms. One consistent status color mapping used identically everywhere it appears (gray=Draft, blue=In Review, orange=Needs Rework, green=Approved). Version chips (`v2 of 2 · Current`) and status chips always pair color with text/icon. Approve and Needs Rework are deliberately different visual weights so they can never be confused under thumb, one-handed, in motion.

**Accessibility notes:** Every status/version chip carries a full-state accessibility label (not just a colored dot); Dynamic Type must reflow gracefully on Asset Detail specifically (the densest screen) without clipping; VoiceOver gets explicit announcements on state transitions ("Moved to In Review," "Needs rework note required"); confirm sheets trap focus so VoiceOver can't swipe past into background content; the read-only old-version banner is announced, not just visually shown; color is never the sole signal anywhere; contrast is explicitly checked for orange-on-white (Needs Rework) in both light and dark mode.

**Named, deliberately accepted risk carried forward:** a list-based Asset Detail and text-heavy Status History are optimized for the locked MVP scale (one campaign, one asset, full cycle) — density at multi-asset scale is an explicitly deferred problem, not solved here.

**Codex (Product Strategist · User Advocate) — Round 1**

One-sentence product promise: Build a local-first SwiftUI iOS app that lets a solo marketer move a campaign from brief capture to versioned asset intake and approval decisions entirely on-device, with a clear audit trail and safe recovery when things change.

Primary user and scenario: One person marketer (freelancer, one-person in-house operator, or agency contractor) who works in short bursts and needs to stage assets, review them quickly between meetings, and make reliable approve/rework decisions without team handoffs, internet access, or account setup.

Core loop: Create or open a campaign with a brief, add or import media assets, attach a new version when changes come in, send an asset to review, approve it or send it back with a required rework note, and carry that status history forward so the next action is always obvious.

Hard scope boundaries: One native iOS app only, single-user, no backend, no login, no team collaboration, no social posting or platform automation, no external webhooks/OAuth/web integrations, no analytics pipeline beyond local history, no multi-device sync, and no third-party libraries unless Apple frameworks truly cannot cover a requirement. Every user flow must include empty/loading/success/error behavior and explicit recovery.

Measurable success criteria: A new user can complete a full end-to-end campaign cycle—create campaign, capture brief, import at least one asset, add a new version, submit for review, complete an approval or needs-rework decision with note, and view status history—on-device without crashes or silent failures; all state persists across relaunches; and each of the above workflows exposes a recoverable error path (undo/restore/ retry) for the edge cases users hit first in real use. biggest risk I’m calling out is trying to chase unverified Ordinal-parity features instead of shipping this local workflow loop, so the contract must preserve “best-effort matching of the workflow, not web-parity claims” and “asset bytes remain local file-backed with metadata in persistence” as non-negotiable constraints.

**Claude (Design Lead · Systems Thinker) — Round 1**

Building on the locked contract, here's my read as design lead on how this actually needs to work as a system of screens, not just a features list.

The product promise I'd write down: a local-first iPhone app where a solo marketer turns a one-line brief into a campaign, uploads and versions creative assets against it, and pushes each asset through an explicit Draft → In Review → Approved/Rework loop, with every decision permanently logged — no login, no network, no collaborator, ever.

Primary user and scenario: someone managing 2-6 client campaigns solo, opening the app in short bursts — on the train drafting a brief, on the couch reviewing a photo a subcontractor texted them, between meetings tapping Approve. They are never at a desk with focused time. That constrains the IA hard: everything has to be reachable in two taps from launch, and the review decision itself has to survive being done one-handed while distracted.

Core loop, concretely as screens: Campaign List (with a persistent, unmissable "new campaign" affordance) → Campaign Detail (brief text + asset grid + status summary) → Asset Detail (QuickLook preview of current version + version history strip + Approve/Rework controls) → Status History (append-only log, both per-asset and rolled up per-campaign). That's the whole app. I want to resist the urge to add a fifth screen — a dashboard, a settings screen, a "insights" tab — none of that serves the loop and all of it just gives QA more surface to half-finish.

Where I think the real design risk lives is in the transitions, not the screens themselves. A few things I don't want left as undocumented judgment calls at build time: what a swipe-to-approve gesture looks like versus a deliberate button tap — I'd say no swipe-to-approve at all, since an accidental full-approve is much worse than an accidental rework (rework just costs a re-tap; a wrongly-approved asset going out the door is the actual failure mode this app exists to prevent), so Approve should require a tap-and-confirm while Rework can be a single tap since it's the safer direction to err toward. Second, the interaction between deleting an asset and campaign status — if a campaign's only asset gets deleted while "In Review," the campaign can't silently stay In Review with nothing to review; it should auto-drop to Draft with a visible system-generated line in its status history ("asset removed, reverted to Draft"), not just vanish from the board. Third, versioning UI: the asset detail screen needs the current version pinned at the top with a clear "v3 of 3, uploaded Tuesday" label and older versions in a collapsed history strip below — Photos.app's edit-stack pattern, but I want it explicit that tapping an old version previews it read-only, it does not "restore" it as current (restoring would need to itself become a new version, appended, never a silent rewrite — otherwise we've broken the non-destructive guarantee we just spent three phases locking in).

Hard scope boundaries, restated from the design side rather than the feature-list side: no tab bar with more than the one navigation stack this loop needs; no settings screen beyond maybe an "About/limitations" static screen that honestly discloses the single-device data-loss risk (that's a UX obligation, not just an engineering footnote — the user should see it once, not discover it after losing their phone); no in-app onboarding wizard beyond the empty state itself doing the job.

Measurable success criteria I'd hold this to: a first-time user can go from cold launch to a created campaign with a brief in under 15 seconds and zero required fields beyond the free-text brief; every one of the four core screens demonstrably renders correct empty/loading/success/error states (this should be checkable by literally deleting all data and relaunching, and by force-killing mid-upload); a full campaign can go create → brief → upload v1 → upload v2 → review → rework with note → upload v3 → approve → view history, relaunch the app, and have every version and every timestamped transition still present, byte-identical; deleting an asset mid-review and deleting a zero-asset campaign both complete without crash and without leaving orphaned state (no asset pointing at a dead campaign, no campaign stuck in a status it can't be in). If any of those four can't be demonstrated, the app isn't done, regardless of how much of the feature list got implemented.

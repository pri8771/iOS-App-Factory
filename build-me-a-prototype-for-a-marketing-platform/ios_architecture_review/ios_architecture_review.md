**Data model to freeze:** `Campaign` (owns many assets) ‚Üí `Asset` (owns ordered, append-only `AssetVersion` rows plus a `currentVersionID` pointer that's the only thing that changes on a new upload) ‚Üí `AssetVersion` (immutable once written) ‚Üí `ReviewEvent` (immutable, append-only, drives Status History) ‚Üí soft-delete represented as an explicit state flag + scheduled-deletion timestamp on the relevant entity, never an immediate row removal.

**Apple framework choices:** SwiftUI + `NavigationStack` (per the locked design_handoff), SwiftData (`@Model`, `@Query`, `@Bindable`) for metadata, PhotosUI (`PhotosPicker`) and `.fileImporter`/`UIDocumentPickerViewController` for asset intake, `UniformTypeIdentifiers` to validate imported files are genuinely the image/video/PDF type they claim before accepting them as a version, QuickLook (`QLPreviewController` via `UIViewControllerRepresentable`) and `QLThumbnailGenerator` (cached in an in-memory `NSCache`) for preview/thumbnails, `FileManager` for atomic file writes (temp file ‚Üí rename), `OSLog` for diagnostics. Explicitly excluded and justified as out of scope: StoreKit (no purchase flow ‚Äî tiering was deferred in `app_features`), CloudKit (no sync ‚Äî single-device by design), Vision/CoreML/ARKit (no task in this app needs them; thumbnailing is a rendering utility, not ML), `BGTaskScheduler`/push (the soft-delete finalize sweep runs synchronously on next foreground launch, needing no background entitlement).

**Persistence and local data plan:** Asset bytes are file-backed in `Application Support/marketing_assets/...` (not Documents ‚Äî keeps users from renaming/deleting backing files via the Files app and breaking referential integrity), written atomically (temp file, then rename), with collision-safe immutable names keyed to asset/version IDs. SwiftData holds only metadata + relative path references. Fixed ordering rules to prevent dangling pointers: on import, the file write must succeed before the SwiftData record is created; on delete-finalize, the SwiftData record is deleted before the backing file. Either direction, an interrupted mid-operation death leaves an orphaned file (accepted, already-named unbounded-storage limitation) rather than a dangling metadata pointer needing defensive handling on every screen. A real `Schema` + `SchemaMigrationPlan` is defined from day one, even if trivial/empty at v1, since a failed store open with no backend and no sync means total, unrecoverable data loss. On launch, a startup reconciliation pass finalizes any pending imports/deletes left incomplete by a prior force-kill.

**Permissions/privacy plan:** Request-on-first-use only, nothing preemptive. The must-have import path (`PhotosPicker`/`.fileImporter`) requires **no** photo-library or file-system permission prompt at all ‚Äî it runs out-of-process, so the "permission denied ‚Üí Settings deep link" recovery state only has a real trigger if/when camera capture (should/could tier) is added, requiring `NSCameraUsageDescription`. What the import path does need is correct security-scoped resource handling: `fileImporter` URLs are only valid inside a `startAccessingSecurityScopedResource()`/`stopAccessingSecurityScopedResource()` window, so the copy into app storage must happen synchronously inside that window. No network calls, no background collection, no analytics/remote logging SDKs anywhere in the app. Standard iCloud device backup is left enabled and disclosed (not app-managed sync) per prior phases' decision.

**Dependency/license decision:** Zero third-party packages. Every requirement maps cleanly onto the Apple frameworks listed above, so the permissive-dependency policy is satisfied trivially ‚Äî there's nothing to license-check. If a dependency is ever proposed later, it must be justified explicitly in-contract with a clear gap Apple's frameworks can't fill.

**Testing implications:** Domain logic (state transitions, whitespace-trimmed rework-note validation, worst-status derivation, auto-revert-on-delete) lives as plain Swift and is unit-tested against an in-memory `ModelConfiguration(isStoredInMemoryOnly: true)` SwiftData container ‚Äî fast, disk-free coverage of the force-kill-equivalent scenarios by reopening a fresh container against the same on-disk store mid-sequence and asserting state. Integration tests cover file-import copy atomicity, stale-pointer cleanup, and launch-time recovery after interrupted writes. UI tests use `XCUIApplication.terminate()` followed by a fresh `launch()` as the explicit, named mechanism for the actual force-quit/relaunch acceptance criteria this whole project is built around ‚Äî not a proxy or simulation. QuickLook presentation always does a synchronous file-existence/non-zero-size check immediately before presenting, routing to the existing preview-failed state on failure rather than trusting the SwiftData record's mere existence.

**Acceptance checks satisfied:** The app is buildable natively with no vague architecture gaps ‚Äî every screen/state from `design_handoff` maps to a named framework and a named persistence rule. No ML/LLM/AR is used anywhere, correctly, since nothing in this app's locked scope needs it. No dependency is introduced, so the permissive-license policy is trivially satisfied.

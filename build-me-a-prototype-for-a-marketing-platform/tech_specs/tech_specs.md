I’m taking a hard line on risk containment: this should ship as a strict local-first iOS SwiftUI app with a fixed, testable contract and one explicit persistence rule set, because the whole product promise depends on surviving relaunch and not corrupting local history. I’m choosing a three-layer model—domain, data/services, and UI—and enforcing all mutation through a repository actor boundary so writes are serialized, restart-safe, and observable by SwiftData fetches without ad-hoc cache invalidation logic.

Architecture overview: domain owns immutable business rules (state transitions, validity checks, version/current derivation), services owns storage and file IO, UI owns event-driven presentation state only. `ReviewStateMachine`, `Versioning` and `Rework validation` are pure domain logic, not view logic. The app shell is one `NavigationStack` that always cold-starts at campaign list; no deep stack restore. SwiftUI view models drive short-lived sheet/dialog state only, not navigation history. Concurrency is explicit: file I/O in background tasks, `ModelContext` mutations on main actor only.

File/module layout: `App/` contains app entry, dependency graph, DI container, and route state; `Domain/` contains states, rules, errors, and pure services; `Data/` contains `SwiftData` models, repositories, file-store, migration plan, and startup reconciliation; `Services/` contains file copy/thumbnail/undo/recovery helpers and OS logging; `Features/` has `CampaignList`, `CampaignDetail`, `AssetDetail`, `StatusHistory`, `SharedComponents`; `Resources/` has app strings and accessibility labels; `Tests/` has `Domain`, `Services`, and `UI` targets. One accepted constraint stays in scope from the start: no third-party dependencies.

Data models: persisted entities are `Campaign` -> `Asset` -> `AssetVersion` and `ReviewEvent`; plus a soft-delete marker embedded on `Campaign` and `Asset` with `isDeleted + deletedAtScheduled` and no immediate row removal. `Asset.currentVersionID` is the only mutable pointer on asset updates; versions themselves are append-only and immutable once written. `ReviewEvent` is append-only and timestamped, with both user and system events. Derived fields (current asset status, campaign worst status, last activity) must be computed from persisted state or via single-domain selectors, never hand-authored in view code. The domain must explicitly reject empty/whitespace-only briefs and rework notes and must block illegal transitions.

Persistence strategy: store only metadata in SwiftData and bytes on disk under `Application Support/marketing_assets/` with immutable IDs in filenames. Import rule: copy-to-disk succeeds before SwiftData metadata is created; delete-finalize rule: SwiftData metadata is deleted before backing file removal. This prevents dangling metadata pointers and turns interrupted crashes into orphaned files only (named limitation: storage growth remains unbounded in v1). Repositories run a startup reconciliation pass for incomplete imports/deletes after force-kill, and every QuickLook open must synchronously validate file existence and size > 0 first. `Schema` and `SchemaMigrationPlan` must exist from v1 with at least a baseline migration plan so model changes cannot silently brick local stores on update. `NSPhotoLibraryUsageDescription` and file picker denial states are not required for `PhotosPicker`/`fileImporter` path; permission-recovery UI should only exist for true gated features like camera (if added later).

Testing strategy: unit-test domain as pure Swift with in-memory SwiftData for transition safety, status derivation, and trimming/validation rules; integration tests for atomic copy/delete ordering and stale-import/deletion recovery; UI tests for all required state surfaces (empty/loading/error/success) and end-to-end cycle including relaunch integrity (`XCUIApplication().terminate()` + fresh `launch()`), plus the `only in-review asset deleted` auto-revert edge case and undo behavior. Biggest risk if we don’t do this: we’ll pass happy-path UI demos and still lose referential integrity after a kill/relaunch; the spec must preserve the startup recovery pass and both ordering rules above as non-negotiable.

**Claude (Frontend Engineer · Visionary) — Round 1**

Let me set the technical shape for this phase, thinking as the person who has to actually wire the four locked screens into a running SwiftUI app that survives force-kill.

My architecture bet is a strict three-package split underneath the app target, not just a "layered folder structure" — because the acceptance check for this phase is explicitly that parallel build lanes can't invent incompatible types, and folders don't enforce that, compiler boundaries do. I want a local Swift Package called `CockpitDomain` that has zero imports beyond Foundation — no SwiftData, no SwiftUI, nothing platform-flavored — and owns exactly the review state machine, the worst-status derivation, rework-note trimming/validation, and the event/status enums. That package should be testable with `swift test` in under a second with no simulator boot, which is the boldest thing I want out of this build: the actual business logic that this whole product's trust depends on (can Approve happen twice, can Rework fire with a blank note, does deleting the last in-review asset really auto-revert) should be provable in a CI-speed unit suite, not something we only find out by tapping through the simulator. On top of that sits `CockpitData`, which owns the five SwiftData `@Model` types plus repository protocols and their SwiftData-backed implementations, translating between plain `UUID`s (what the domain layer speaks) and SwiftData's `PersistentIdentifier` internally so `CockpitDomain` never has to know SwiftData exists. Alongside it, `CockpitServices` owns the filesystem: the atomic asset-file writer, the QuickLook thumbnail cache, the import coordinator that does the "write file, confirm, then create record" ordering we already locked, and the soft-delete/reconciliation coordinator. The app target itself is thin: SwiftUI views, a `Route` enum driving a single `NavigationStack`, and an `@Observable AppEnvironment` that's constructed once at launch and handed down via `.environment(_:)`, assembling the concrete repositories and services so no view ever touches a `ModelContext` directly.

On the data model, I'd freeze it exactly as the architecture review already sketched but make the concurrency-safe shape explicit: `Campaign` holds `id`, `briefText`, `createdAt`, `isTrashed`, `scheduledDeletionAt: Date?`, and a cascade relationship to `Asset`. `Asset` holds `id`, `currentVersionID: UUID`, `status: ReviewStatus`, `isTrashed`, `scheduledDeletionAt: Date?`, and cascade relationships to `AssetVersion` and `ReviewEvent`. `AssetVersion` is immutable once written — `id`, `versionNumber`, `relativeFilePath`, `fileType` (a UTType identifier string, checked against `UniformTypeIdentifiers` at import time so a renamed `.mov` claiming to be a `.jpg` can't sneak past QuickLook), `fileSizeBytes`, `createdAt`. `ReviewEvent` is append-only — `id`, `kind` (an enum: created, versionAdded, submittedForReview, approved, needsRework, autoRevertedToDraft, assetTrashed, assetRestored, campaignTrashed, campaignRestored, deletionFinalized), `actor` (`.user` or `.system`, so Status History's visual distinction has a real backing field instead of inferring it from the event kind at render time), an optional `note`, and `timestamp`. The one technical detail I want to nail down now, because nobody's actually specified it and it's exactly the kind of thing that becomes an undocumented judgment call at 11pm during build: the soft-delete undo countdown must not be driven by a live `Timer` or `Task.sleep` as its source of truth, because those die the instant the app backgrounds or gets killed, and we've already locked "undo must survive backgrounding within the window" as a hard requirement. The real mechanism is a stored `scheduledDeletionAt` timestamp written at the moment of soft-delete; the UI countdown is just a cosmetic poll against that timestamp, and the actual eligibility check — "has this passed its window" — is computed by comparing `scheduledDeletionAt` to `Date()` both in the startup reconciliation pass and on every `scenePhase` transition to `.active`, not only on cold launch. Without that second trigger, a user who backgrounds and foregrounds without a full force-quit would see stale trashed items sitting past their window indefinitely, which quietly breaks the "undoable within 10 seconds" contract nobody would notice until QA specifically tried it.

For the state machine itself, I want `ReviewStateMachine` to be a pure function — `transition(from: ReviewStatus, action: ReviewAction) -> Result<ReviewStatus, TransitionError>` — where `ReviewAction` is its own enum (`submitForReview`, `approve`, `needsRework(note: String)`) so the "rework requires a non-empty trimmed note" rule and the "invalid transitions are impossible" rule live in one switch statement that every call site (view, test, future feature) goes through, instead of being re-derived per screen. Same instinct for `WorstStatusDerivation.worstStatus(for: [ReviewStatus]) -> ReviewStatus?`, which is what Campaign List's aggregate chip and Status History's campaign-level rollup both need to agree on, so it can't drift into two different orderings of "urgency" in two different views.

Persistence-wise, repositories are actors — `CampaignRepository`, `AssetRepository`, `ReviewEventRepository` — each wrapping a background `ModelActor`-conforming context so writes never touch the main-thread `ModelContext` used for `@Query`/`@Bindable` reads. Every mutating call is `async throws` and returns plain `UUID`s or throws a typed error (`ImportError`, `PersistenceError`), never a raw SwiftData error bubbling into view code. The file-write-before-record-create and record-delete-before-file-delete ordering from the architecture review lives inside `ImportCoordinator` and `SoftDeleteCoordinator` respectively, so it's enforced in exactly one place rather than trusted to every call site to get right.

Testing strategy: `CockpitDomain` gets a pure Swift Testing (`@Test`, not XCTest) suite with no SwiftData dependency at all — this is where every transition-guard, note-trimming, and worst-status edge case gets proven, fast and disk-free. `CockpitData` gets XCTest with `ModelConfiguration(isStoredInMemoryOnly: true)` for repository and reconciliation-ordering tests, including the "reopen a fresh container mid-sequence" trick already agreed on to approximate force-kill without literally force-killing. `CockpitServices` gets integration tests against a real temp `Application Support` directory for atomic-write and orphan-cleanup behavior. And the app target gets XCUITest with real `terminate()` + fresh `launch()` for the actual relaunch-survival acceptance criteria, plus a test that specifically backgrounds (not kills) mid-undo-window to catch the `scenePhase` reconciliation gap I just flagged.

My honest objection to my own direction: three local packages is more upfront ceremony than a single-screen-flow MVP strictly needs, and if the domain/data boundary is drawn sloppily — say, a repository protocol that leaks `PersistentIdentifier` into `CockpitDomain` — the whole point of the split (a SwiftData-free, instant-test-suite core) quietly evaporates and we're left with the overhead of three packages and none of the benefit. The thing that must survive into the final spec to prevent that: domain-layer protocols speak only `UUID` and plain Swift value types, never SwiftData types, full stop — that boundary is the one rule I'd fight to keep un-negotiated.

## Final Output

**Architecture overview:** Four-layer split, enforced by real module boundaries rather than folder convention. `CockpitDomain` (Foundation-only, zero SwiftData/SwiftUI imports) owns the review state machine, worst-status derivation, and note validation as pure functions — this is the layer whose correctness the whole product's trust depends on, and it must be testable with a sub-second `swift test` run, no simulator. `CockpitData` owns the five SwiftData `@Model` types and actor-backed repository implementations, translating internally between plain `UUID`s (what the domain layer speaks) and SwiftData's `PersistentIdentifier` so the domain layer never has to know SwiftData exists. `CockpitServices` owns the filesystem — atomic asset-file writes, QuickLook thumbnailing, soft-delete/reconciliation, import ordering. The app target itself stays thin: SwiftUI views, a `Route`-driven `NavigationStack`, and an `@Observable AppEnvironment` built once at launch and injected via `.environment(_:)` so no view ever touches a `ModelContext` directly. This is a refinement of Codex's protocol set, not a departure from it — Claude's package split is how those same protocols get compiler-enforced isolation instead of just being organized by convention.

**File/module layout:** `CockpitDomain` (state machine, derivation, validation — no platform imports), `CockpitData` (SwiftData `@Model`s + repository actors), `CockpitServices` (file store, import coordinator, thumbnail cache, soft-delete coordinator), app target (views, view models, `AppEnvironment`, `Route`).

**Data models (frozen):** `Campaign` → `Asset` (holds `currentVersionID`, the only field a new upload changes) → `AssetVersion` (immutable once written, **file-backed via `relativeFilePath`, never embedded bytes** — this rejects Gemma3's `content: Data` approach as a direct contradiction of the locked `ios_architecture_review` decision) → `ReviewEvent` (append-only, carries `actor: .user/.system` as a real field, not inferred at render time, backing Status History's visual distinction). Soft-delete is `isTrashed` + `scheduledDeletionAt: Date?` on the relevant entity, never a row removal.

**Persistence strategy:** SwiftData repositories are actors wrapping background contexts; the main-actor `ModelContext` is read-only via `@Query`/`@Bindable`. Import order: file write succeeds and is confirmed before the SwiftData record is created. Delete order: SwiftData record is deleted before the backing file. `ImportCoordinator` and `SoftDeleteCoordinator` are the single enforcement points for those orderings, not something every call site re-derives. A real (if trivial) `VersionedSchema`/`SchemaMigrationPlan` ships from day one. `SoftDeleteCoordinator.reconcile(now:)` runs on cold launch *and* on every `scenePhase` transition to `.active` — the fix for the background-without-force-quit gap nobody had caught before this round.

**Testing strategy:** `CockpitDomain` gets a pure Swift Testing suite, no SwiftData, disk-free, proving every transition guard and edge case (double-approve, blank-note rework, auto-revert-on-delete) at CI speed. `CockpitData` gets XCTest against `ModelConfiguration(isStoredInMemoryOnly: true)`, including the reopen-a-fresh-container-mid-sequence trick to approximate force-kill. `CockpitServices` gets integration tests against a real temp `Application Support` directory. The app target gets XCUITest using literal `terminate()` + fresh `launch()` for relaunch-survival claims, plus a background-not-kill test targeting the `scenePhase` reconciliation path specifically.

**Acceptance checks satisfied:** parallel build lanes have a fully-typed, non-overlapping contract to code against (data_domain/primary_ui/services_utilities/polish_resilience), and the spec names concrete build and verification mechanisms for every claim made in earlier phases.

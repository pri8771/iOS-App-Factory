**Codex (Product Strategist · Closer) — Round 1**

The right design for cleanline is not “creative studio on a phone.” It is a guided production tool that keeps one sketch moving forward with as little ambiguity as possible. The app should feel like a calibrated workbench: every screen either helps the user qualify the source, understand the transformation, make one of the few allowed corrections, or get the file out. If a screen doesn’t reduce uncertainty or time-to-export, it should not exist.

I would lock the app to six real screens plus a couple of modal surfaces, and I would not let it grow past that in MVP.

The first screen is the recents home. This is the app’s root, not a splashy landing page. It has a top-level “New Trace” action, a flat recent-traces list, and a small settings entry. The empty state is important: if there are no traces yet, the screen should still teach the product in one glance with a large new-trace button, a plain sentence about “turn one paper sketch into an editable SVG or PDF on this device,” and a compact privacy line saying nothing leaves the phone. The loading state is just for storage fetch on launch. The success state is the recents list with thumbnail, title or timestamp, last export status, and maybe a small status badge like “Needs export” or “Exported.” The error state is failed local-load with retry, but it should be rare. From here, the user can start new, reopen a prior trace, delete a trace, or enter settings.

The second screen is intake, and it should be brutally simple. Two large choices: camera and import photo. If camera permission is denied, the camera card should stay visible but clearly explain the denial and offer system settings, while import remains fully active. This screen also needs a tiny supported-input note: dark linework on plain paper, one sketch at a time. That is not marketing copy; it is scope defense. The empty state is the default choice screen. The loading state appears only while opening camera/import. The error state is unsupported image or failed load. The success path immediately advances into quality checking after an image is chosen.

The third screen is capture quality and correction review. I think this should be one screen, not two, because splitting it adds ceremony. The app runs the quality gate first; if the image is out of bounds, the user lands in a failure variant of this same screen with the original image, the fixed named reason, a short “what to do next,” and clear actions like retake, choose another photo, or cancel back home. If the image passes, the user lands in the review variant with a segmented or swipe comparison between original and corrected image. This is where the app proves it actually understood the paper. The user should be able to inspect the corrected result, not tune ten controls. MVP only needs maybe one threshold/cleanup adjustment if absolutely necessary, but I would rather not turn this into a settings panel unless testing proves it is required. The important actions are proceed to trace, go back, and recapture/import another photo. The screen states here are especially important: checking, rejected with named reason, corrected preview ready, correction failed, and resuming-from-checkpoint.

The fourth screen is processing. I do not want this hidden behind a spinner overlay because the pipeline is the product. This should be a dedicated progress screen with the corrected image faintly visible in the background and one clear current stage label: checking geometry, correcting perspective, isolating linework, tracing paths, preparing preview. The user needs confidence that something deterministic is happening. There should be no fake progress bar unless it reflects real stage advancement. If processing fails, this same screen flips into a recoverable error with actions to retry or go back to review. If the app relaunches after interruption at this stage, it should come back here only if it is safely rerunning from the last checkpoint; otherwise it should reopen at the last completed stage and explain that tracing will restart.

The fifth screen is review. This is the emotional payoff screen and the one with viral potential, so it needs the most care. The core interaction should be a clean before/after comparison between the corrected source image and the vector overlay/result, with a high-contrast canvas and obvious zoom support. The user’s question here is simple: “Is this faithful enough to keep going?” Primary actions are continue to cleanup, export now, or retry from earlier. I would also show a quiet confidence/warning note only if needed, not a score. Something like “Looks good for export” or “Some fragments may need cleanup” is enough. The screen states are loading preview, ready, low-confidence warning, trace failure, and resumed-from-saved-trace. This screen should feel proud but still technical, more like inspecting a proof than browsing artwork.

The sixth screen is cleanup. This is the only place where the app should feel tool-like rather than wizard-like. The layout should be a large vector canvas with a bottom tool strip for select/delete, simplify, merge, and undo/redo. I would keep the tool count fixed and visible; hiding them in menus weakens the scope fence and makes the app feel more capable than it is. Selection should be explicit. When nothing is selected, destructive actions are disabled. Simplify should be presented as one slider with immediate preview feedback on a downsampled render, then committed on release. Merge should only appear when the selection model makes sense. There also needs to be an alternate list mode for accessibility and precision cases: a modal list of detected paths/fragments with names like “Path 12” plus metadata like size or open/closed status, and actions for select, delete, and merge. This is not optional polish; it is part of the cleanup screen’s design. Screen states here are no selection, one selection, multi-selection, reprocessing-preview, committed edit, undo available, and error if a reprocess fails. If the user kills the app here, it must return to the last committed edit state, never to an in-between drag.

Export should be a modal flow, not its own full screen. The cleanup and review screens both need an export action that opens a bottom sheet or full-screen sheet with format choice, filename, and destination via share sheet or file export. The states are ready to export, exporting, export success, and export failed-but-saved. The critical UX rule is that export failure can never feel like work loss. If sharing fails, the modal should say the vector is still saved in recents and can be exported again later.

Settings can be a small seventh screen if you count it separately, but it should stay minimal. This is where the plain-language privacy promise lives, along with the honest “Pro coming soon” note and maybe lightweight defaults if they exist later. No growth hooks, no subscription shell, no fake locked controls.

The primary user flow should read cleanly in one line: home recents, start new trace, choose camera or import, pass or fail quality gate, inspect corrected image, run trace, inspect before/after, optionally clean up, export, return to recents with saved result. Every detour should route back into that spine, not into side journeys. Reopened traces should skip directly to review or cleanup depending on their last saved checkpoint.

Visually, I would not design this like a bright consumer scanner app. It should look precise, calm, and premium without going dark-and-purple-generic. My best direction is warm paper tones against graphite and ink-black UI accents, with one sharp technical accent color like oxidized teal or safety orange used sparingly for active states and warnings. The home and review screens should have a subtle drafting-table feeling: off-white backgrounds, thin grid or paper-fiber texture at very low contrast, crisp black typography, and clean hairline dividers. The vector preview itself should feel almost print-proof-like: white or warm-white artboard, true black paths, no decorative chrome around it. Typography should lean editorial or industrial rather than startup-default; the product is about craft and output quality, not social energy. Motion should be functional: stage transitions that reinforce pipeline progress, not flourish. The before/after interaction is where the delight lives; everything else should feel confident and restrained.

Accessibility has to be designed in, not layered on. The quality-gate reasons need stable wording and consistent placement so repeat users learn them. Every important comparison view needs a non-gesture fallback; don’t rely on pinch, scrub, or tap-only interactions to make the app usable. Cleanup must support the list-based path management mode end to end with VoiceOver. Color cannot carry system meaning by itself; warning, ready, and failure states need iconography and text. Dynamic type needs to work at least on home, intake, settings, and all error/rejection states without truncating the recovery action. On the canvas-heavy screens, the right compromise is scalable supporting text with a fixed functional workspace, not pretending the whole vector editor should reflow like a text article.

The biggest risk in this direction is that the app becomes too stepwise and loses the feeling of speed. A quality gate screen, review screen, processing screen, and cleanup screen can start to feel like ceremony if the transitions are slow or if users are forced to confirm obvious things. The fix is not to collapse the screens into ambiguity; the fix is to make state changes decisive and keep optional steps optional. If the trace is good, the user should be able to go from corrected preview to processing to review to export in a fast, almost conveyor-belt rhythm. Cleanup should feel available, not mandatory.

The detail the final phase output must preserve is this: cleanline is a trust-building sequence, not a canvas app. The view hierarchy should make the pipeline legible, the state model should privilege checkpoint recovery and honest failure over flexibility, and the visual design should make the output feel precise enough to use professionally. If later phases muddy that by making every stage editable, hide the correction step, or turn cleanup into a vague floating toolbar, they will break the product even if the code still compiles.

**Codex (QA & Risk · Pragmatist) — Round 1**

The smallest architecture that still honestly ships this app is a document-style pipeline with four hard boundaries: ingest, gate/normalize, vectorize/edit, and export/persist. I would keep SwiftUI thin and stateful only at the workflow edges. One `@MainActor @Observable` `AppModel` owns recents and app-wide settings; one `@MainActor @Observable` `TraceSession` owns the open job, current screen state, selection, undo/redo stack, and committed checkpoint metadata. Everything expensive or failure-prone stays out of views and behind async service protocols so the app can be tested with fixtures and resumed without UI involvement. The critical choice is to treat vectorization as the risky core, not as a Vision helper. Vision only gives rectangle/contour detection; the product quality bar lives in our cleanup, simplification, curve fitting, path classification, and export correctness.

I would split the code into local Swift packages or at least module-like targets even for MVP: `TraceDomain`, `TracePersistence`, `TraceProcessing`, `TraceExport`, and the app target for UI/composition. `TraceDomain` holds pure types like `TraceDocument`, `VectorPath`, `QualityGateFailure`, `ProcessingStage`, edit commands, and checkpoint metadata. `TraceProcessing` owns image ingest, EXIF normalization, perspective correction, thresholding, ruled-paper suppression, quality gating, contour extraction, simplification, merge heuristics, and Bezier fitting. `TracePersistence` owns per-trace folders, atomic writes, index rebuild, thumbnails, and resume-from-checkpoint loading. `TraceExport` owns SVG/PDF serialization and export artifact prep. The UI target should know protocols and domain types, not file layouts or CI filters. That keeps parallel work sane and prevents one lane from inventing incompatible types under deadline pressure.

The data model should stay document-shaped. A `TraceDocument` is the durable unit and maps to one folder. It needs an `id`, created/updated timestamps, a `TraceStatus` enum, a `lastCommittedStage`, a `workspaceMode`, optional `QualityGateResult`, normalized image metadata, vector model metadata, export status, and a thumbnail reference. The vector itself should not be hidden in generic blobs. Use explicit `VectorPath` values with `id`, `points` or fitted segments, `isClosed`, `boundingBox`, `nodeCount`, `sourceFragmentLength`, and per-path simplify parameters if needed later. Keep the edit history command-based: `deletePaths([UUID])`, `mergePaths([UUID])`, `setSimplifyStrength(pathIDs:[UUID], value: Float)`. That gives deterministic undo/redo and clean checkpointing because only committed commands become durable. Persistence should be a per-trace package under Application Support, OS-default backup enabled, with no Core Data or SwiftData. The raw camera image is transient only through correction review, then discarded. All writes must be temp-file-then-rename atomic writes. The recents list should be a rebuildable cache, not the source of truth. That is a real field-failure issue, not polish. Resume behavior should be defined mechanically: if the last committed stage is `.normalized`, restart vectorization; if `.traced`, reopen workspace; if `.edited`, reopen workspace with saved vector state. **Claude (Frontend Engineer · User Advocate) — Round 1**

Looking at this from the client side, my job is to make sure the thing a solo illustrator actually taps on Tuesday night at her kitchen table holds together — state doesn't get lost, screens don't jank, and the four-lane parallel build doesn't produce four different opinions about what a `VectorPath` is.

The architecture I want locked is: one thin SwiftUI app target that owns navigation and screens, and everything else lives in local Swift Package targets with real `internal` access boundaries, exactly as the ios_architecture_review phase called for. But I want to go one level more specific than "isolated services," because the thing that will actually break a solo build-lane split is two workers independently inventing two different shapes for `VectorPath` or `TraceDocument`. So there's a `CleanlineModels` package that contains every shared data type — no lane may redeclare its own copy of a cross-lane type, full stop. Every service consumes and returns those shared types, never a private DTO that gets converted later. That's the single rule that makes "parallel build lanes can work without inventing incompatible types" actually true instead of aspirational.

Navigation: a single `NavigationStack` driven by an explicit `AppRoute` enum path array, not ad-hoc pushes scattered across views. The reason I care about this specifically is checkpoint resume — the whole product promise is "reopen a trace and land exactly where you left off," and that only holds if there's one function, `AppRouter.route(for: TraceDocument)`, that deterministically computes the NavigationPath from a loaded document's status field (needsReview → Workspace/Compare, mid-edit → Workspace/Edit, traceFailed → Processing's failed state). If this logic gets embedded ad hoc in fifteen different `.onAppear` handlers instead of one router, someone will "temporarily" make interrupted traces land back on Recents Home to unblock a build, and that regression will be invisible until a real user loses their place after a phone call.

State ownership: `TraceSession` is an `@Observable`, `@MainActor` reference type, one instance per open trace, and it is the only mutable source of truth for zoom/pan/selection/current mode/undo-redo stack. Views bind to it, they never own parallel copies of its state in `@State`. The undo/redo stack is a command log (`deletePath`, `mergePaths`, `setSimplifyStrength`), and — this is the detail I most want to protect — a command is only ever appended on gesture *commit*, never on `onChanged`/mid-drag. That's not a style preference, it's what makes "resume from last committed checkpoint" mechanically true instead of a hope. If a slider's live value ever leaks into the checkpoint before release, the resume promise silently breaks the first time someone backgrounds the app mid-drag.

On the vectorization and pipeline services specifically: these need to be real `async` boundaries, ideally an `actor VectorizationEngine`, because Vision requests and Douglas-Peucker/Bezier-fitting are genuinely CPU-heavy and must never run on the main actor while the Processing screen is animating stage labels. `TraceSession` calls into these actors and gets back plain `Sendable` value types (`[VectorPath]`, `QualityGateResult`) — no shared mutable state crosses that boundary, which also makes the pipeline stages trivially unit-testable as pure functions, exactly the fixture-based testing model the architecture phase already committed to.

Performance is a first-class requirement, not a hope, because the device floor here matters: this app has to feel instant on something like an iPhone 12, not just the dev's Pro Max. Concretely: the simplify slider drag renders against a downsampled preview (capped at something like 1024px on the long edge) with a cancel-aware debounce around 150–250ms, and only re-fits at full resolution on release — if this gets implemented as a naive `onEditingChanged` call straight into the full-res fitter, it will visibly stutter and the "premium craft tool" feeling evaporates in the one interaction users touch the most. Camera Capture's live preview must hold real frame rate — no synchronous Vision or Core Image work on the capture delegate queue. And I want to name the trust-curve concern explicitly as an engineering requirement, not just a design nicety: trace #2 and #3 should feel faster than trace #1, which means remembering the last-used simplify/threshold preference per user (small `UserDefaults`-backed struct) so repeat traces don't force the same manual dial-in every time.

Persistence strategy, from the client side: `TraceStore` is a protocol with atomic temp-file-then-rename writes for `normalized.heic`/`vector.json`/`thumbnail.jpg`/`metadata.json`, and the Recents list is populated by scanning the traces directory and reading each `metadata.json`, never by trusting a separate index file as ground truth — that index is a cache that gets rebuilt, matching what's already locked. Testing strategy: unit tests own the pipeline stages (pure functions, fixture-in/fixture-out, snapshot-diffable), the edit reducer gets adversarial sequence tests (delete-all, merge-with-self, redo-after-new-edit invalidates stack), persistence gets a literal kill-mid-write simulation, and XCUITest is reserved narrowly for things that can only be proven on a real render tree: the gesture-arming contract (wipe only arms via handle at base zoom, tap-to-select never fires during a wipe drag, no double-tap-zoom delay), a real VoiceOver pass over the path-list cleanup flow, and permission-denied → import-still-works. I don't want XCUITest carrying pipeline-correctness weight; it's slow and flaky, and the pipeline is provably deterministic enough to not need it there.

The biggest risk to my own direction: the four-lane split is only healthy if `CleanlineModels` ships first and is treated as frozen truth for the rest of the build — if lanes start building against guesses about its shape before it's nailed down, we'll get exactly the incompatible-types problem the acceptance check warns about. The thing that must survive into the final tech spec, word for word: one `TraceSession` object, one `AppRouter` function deciding where a reopened trace lands, commands appended only on gesture commit, and every cross-lane type living in a single shared models package that nobody duplicates.

### Round 2


**Codex (QA & Risk · Pragmatist) — Round 2**

Codex, I think the coordinator is right that the only thing actually blocking consensus now is contract drift, not architecture drift. If we shipped the current round-1 output to parallel builders, we'd get avoidable breakage immediately: one lane coding against `TraceSession` as a value type, another against an `@Observable @MainActor` reference owner; one lane persisting eight `TraceStatus` cases, another rendering only the three status chips the design spec actually allows. That's not “we'll sort it out in code review” territory, that's “resume routing and recents UI become nondeterministic” territory. So I would stop trying to preserve both shapes and just collapse to one canonical split: `TraceStatus` stays tiny and user-facing (`needsReview`, `exported`, `traceFailed`), while pipeline progress lives somewhere else (`TraceStage` for committed checkpoints, `ProcessingStage` for transient on-screen progress). That separation is the smallest thing that actually ships cleanly, and it lines up with what Claude already locked in the design handoff.

I also think Claude's separation of `TraceDocument` metadata from `VectorDocument` payload is the safer call, but only if we stay disciplined about what each file is for. `metadata.json` should answer “what trace is this and where do I resume,” while `vector.json` answers “what paths do I render/export.” If you cram `paths` into `TraceDocument`, every lightweight recents refresh risks dragging heavy geometry through decode paths that don't need it. But if you split them, you need one rule nobody violates: recents and routing never require loading vector geometry. That's the practical performance win on an iPhone 12, and it's exactly the kind of thing that quietly regresses if the contract is vague. Same story on `EditCommand`: keep the command signatures exactly as the design phase wrote them, even if they feel slightly narrow today. The point of the command log is resumability and testability, not theoretical elegance. I'd also keep `QualityMetrics` in the contract even if it's internal-facing, because otherwise the quality gate turns into a boolean oracle that's impossible to debug when “shadow coverage too high” starts misfiring on ruled-paper fixtures. We don't need to expose those metrics in UI, but we absolutely need a typed place for them in tests and logs.

The one place I want to tighten the coordinator's proposed reconciliation is `AppModel.startNewTrace(from:source:)`. Raw `Data` there is too lossy for the capture/import split we've already said matters. We already have `CapturedImage`, with explicit orientation, and that's the safer cross-lane input because orientation bugs are one of the easiest trust-killers in this whole app. I also want `TraceSession` to own `interruptionRecoveryState`; if that lives elsewhere, somebody will implement the resume banner as a separate view concern and the invariant “never resume a half-applied gesture” gets mushy. Otherwise I think the coordinator's direction is the right one: one router function, one observable session object, one shared models package, one persistence contract with atomic checkpoint writes and rebuildable recents, and no pretending these names are still negotiable.

**Claude (Frontend Engineer · User Advocate) — Round 2**

Reading the coordinator's reconciliation, I think it lands in the right place on basically every conflict, and I want to say clearly: Codex, your `VectorDocument` as its own checkpoint-file entity beats my flat `paths: [VectorPath]` on `TraceDocument`. I was sloppy there — the persistence architecture we already locked describes four separate files per trace (`metadata.json` / `normalized.heic` / `vector.json` / `thumbnail.jpg`), and embedding the vector array directly on `TraceDocument` would mean either serializing the whole thing into `metadata.json` (defeating the point of having a separate `vector.json`) or maintaining two representations that drift. Your shape maps cleanly onto the actual file layout, mine didn't. Same story on the three-value `TraceStatus` vs your eight-value one — the coordinator's fix of splitting persisted user-facing status (`needsReview`/`exported`/`traceFailed`, matching the Recents chip vocabulary word-for-word from design_handoff) from internal pipeline progress (`TraceStage` for checkpoints, `ProcessingStage` for the Processing screen's stage labels) is the right disambiguation — you and I were both reaching for one enum to do two jobs.

One thing I want to flag before we call this closed, because it's a real gap neither of our drafts actually resolved: what happens to disk state when the quality gate *rejects* a photo? We've both been assuming `TraceStage.captured` is the first checkpoint, written the moment a photo is captured/imported, so an app-kill during gate evaluation can resume by re-running the (cheap, deterministic) gate against the already-saved raw bytes rather than losing the source — that's consistent with detailed_discussion's "autosave checkpoints at capture" and design_handoff's "resuming-from-checkpoint" being a legitimate state for the Quality-Gate+Correction screen. But if we write that checkpoint unconditionally and the gate then says "shadow-coverage-too-high," what happens to that folder? If nobody resolves this, whoever builds Intake/Gate will improvise, and the two bad improvisations are equally likely: either dead folders pile up on disk for every retake attempt (a real user who retakes a photo three times before getting a good shot now has two ghost folders sitting in Application Support forever, which is a privacy smell too — a rejected photo might have someone's whole studio table in frame), or someone "fixes" it by not writing the checkpoint until after the gate passes, which quietly breaks the resume-during-gate-evaluation guarantee we already locked.

The clean resolution: a trace folder gets created and the `captured` checkpoint written the instant we have image bytes, but if the gate rejects, `AppModel` deletes that folder immediately as part of handling the rejection — the user sees the rejection card with retake/choose-another, and nothing ever surfaces in Recents, nothing lingers on disk. Only a gate-*pass* leaves the folder in place and becomes a real, browsable, resumable trace. That means `AppModel.startNewTrace` shouldn't just return a bare `UUID` — I want it to return an outcome the Gate screen can branch on directly:

```swift
enum NewTraceOutcome: Sendable { case rejected(RejectionReason); case accepted(traceID: UUID) }
```

with `AppModel.startNewTrace(from:source:) async throws -> NewTraceOutcome` internally doing capture-checkpoint → gate-evaluate → (delete-and-return-.rejected) or (keep-and-return-.accepted). This closes the loop cleanly: `AppRouter.route(for:)` only ever operates on documents that have already passed the gate, so there's no ambiguity about routing a half-alive rejected folder anywhere.

On Codex's `TraceWorkflowControlling` protocol — I get the instinct (a UI-facing facade so views don't call services directly), but I don't think it earns its keep here on top of what we already have. The concrete `TraceSession` class the coordinator wrote already *is* that facade — `apply`, `undo`, `redo`, `exportCurrentTrace` are right there as methods, and for testing you inject fake `VectorizationEngine`/`ExportService`/`TraceStore` implementations at the service layer, which gives you the same fixture-testability without a second abstraction layer wrapping the first. For an app this size, one more protocol whose only implementation is `TraceSession` itself is ceremony, not safety net — I'd rather keep the surface area small and let the compiler-enforced package boundaries do the decoupling work.

Also, small thing worth just stating out loud so nobody second-guesses it mid-build: `CGPoint`/`CGSize`/`CGRect` are genuinely `Codable` and `Sendable` as of the SDKs backing iOS 17+/Swift 5.9+, so using them directly in `PathSegment`/`VectorPath`/`VectorDocument` instead of hand-rolled `PointDTO`/`RectDTO`/`CGSizeDTO` wrapper types (which Codex's draft had) is safe and cuts a layer of pointless conversion code. I'd rather lock that now than have someone "play it safe" and reintroduce DTO shims nobody asked for.

### Round 3


**Codex (QA & Risk · Pragmatist) — Round 3**

Claude’s three loose ends are real, but they’re also the kind of loose ends that turn into stupid field bugs if we pretend “close enough” is enough. I don’t think we need another architecture debate here because the direction is already settled; we need to stop leaving lifecycle edge cases half-specified.

On the rejection cleanup point, I think Claude’s `NewTraceOutcome` is the right fix and I’d lock it exactly because it makes the failure mode impossible to hand-wave. If `startNewTrace` can only return a `traceID`, then a rejected capture either has to create a fake document that never belongs in Recents, or it has to throw a generic error and make the caller infer “this was a known, user-facing rejection, not a broken pipeline.” Both are sloppy. A typed `.rejected(RejectionReason)` outcome lets the UI show the designed rejection state while the model deletes the transient folder internally and guarantees no ghost trace survives. That is the smallest thing that actually ships cleanly. Same story on `CapturedImage` as the input to `startNewTrace`: Codex was right to push on that. The services already traffic in `CapturedImage`, and orientation is not incidental metadata here, it is part of correctness. Reverting to raw `Data` at the app-model boundary just forces an avoidable conversion and makes it easier for someone to lose orientation handling in exactly the layer that decides whether a trace is accepted or rejected.

I also agree with Codex that `interruptionRecoveryState` belongs on `TraceSession`, not floating as some separate view concern. If the promise is “resume from the last committed checkpoint, never from a half-applied gesture,” then the canonical session object should be the thing that knows whether it was resumed and from where. Otherwise the resume banner logic and the actual resume semantics can drift apart. That’s the kind of mismatch that passes happy-path testing and then looks amateur the first time someone kills the app during tracing and relaunches into a workspace that says nothing or says the wrong thing. One small tightening I’d add, because this is where QA pain usually shows up: `TraceStore` should not be asked to persist rejected work at all beyond the temporary capture write needed to run the gate. The durable contract starts only once the gate passes. That keeps Recents rebuild logic simple and avoids weird “folder exists but isn’t a document” branches. At this point I think the contract is closed: keep `TraceSession` as the UI-facing facade, don’t add a redundant protocol over it, keep geometry in native CoreGraphics types, and make the rejection path mechanically incapable of leaving disk residue.

## Final Output

**Architecture overview:** A thin SwiftUI app target owns navigation and screens only. All real logic lives in isolated local Swift Package targets with real `internal` access boundaries — capture, normalize, quality-gate, vectorize, edit, export, and persist are each their own module, plus one shared `CleanlineModels` package that every other module and the UI target depends on, so no lane can invent a competing shape for a cross-lane type. Two observable model layers: `AppModel` (`@Observable @MainActor`) owns the Recents list and trace lifecycle (start/open/delete); `TraceSession` (`@Observable @MainActor`, one instance per open trace) owns the active workflow — document, vector data, undo/redo stack, zoom/pan/selection, mode, and interruption-recovery state. A single `AppRouter.route(for:)` function is the only place that decides where a reopened/resumed trace lands. Heavy processing (Vision contour detection, Douglas-Peucker simplification, Bezier fitting, winding/fill classification) runs off the main actor in an actor-isolated `VectorizationEngine`; `TraceSession` calls into it and gets back plain `Sendable` value types. Undo/redo is a command log (`EditCommand`) applied through a pure reducer (`EditEngine.apply(_:to:)`), and commands are appended only on gesture commit/release — never mid-drag — which is what makes "resume from last committed checkpoint" mechanically true rather than aspirational.

**Data models:** `TraceDocument` is the durable metadata record (`metadata.json`) — id, timestamps, `TraceStatus` (the 3-value Recents-chip vocabulary: `needsReview`/`exported`/`traceFailed`), `lastCommittedStage` (`TraceStage`: `captured`/`normalized`/`traced`/`edited`), workspace mode, gate result, export status, and file references. It never carries vector geometry or EXIF/location/device data. `VectorDocument` (`vector.json`) is its own file-backed type — canvas size, `[VectorPath]`, global simplify strength — kept separate so Recents/routing never has to decode heavy geometry. `VectorPath`/`PathSegment` use native `CGPoint`/`CGSize`/`CGRect` directly (both Codable and Sendable on this deployment target) rather than hand-rolled DTO wrappers. `EditCommand` matches the exact signatures already locked in design_handoff (`deletePath(id:)`, `mergePaths(ids:resultID:)`, `setSimplifyStrength(pathID:value:)`). `RejectionReason` is the fixed seven-case named vocabulary. `NewTraceOutcome` (`.rejected(RejectionReason)` / `.accepted(traceID:)`) is the return type of starting a new trace — this is the mechanism that guarantees a gate-rejected capture never leaves a ghost folder or Recents entry: the folder is deleted internally on rejection, and a `TraceDocument`/status only exists once the gate has passed.

**Persistence strategy:** Per-trace file packages under Application Support, OS-default backup enabled (this is deliberate — "local-first" means not shipping sketches to our servers or third-party SDKs, not opting a user's own work out of Apple's encrypted device backup). No Core Data/SwiftData. All checkpoint writes are atomic temp-file-then-rename, with an explicit test that simulates a kill mid-write and asserts the prior checkpoint survives intact. The Recents list is a rebuildable cache, not a source of truth — on any inconsistency it's reconstructed by scanning trace folders and reading each `TraceDocument`. Durable storage begins only after the quality gate passes; a rejected capture's transient folder is deleted by `AppModel` as part of handling the rejection, never left as orphaned disk state. `TraceStore` owns the atomic write/read/delete contract exactly as specified in the interfaces block below. `PrivacyInfo.xcprivacy` is a locked MVP requirement (file-timestamp and UserDefaults usage trigger required-reason declarations even with zero third-party SDKs). No `BGProcessingTask`; a short `beginBackgroundTask` grace window covers finishing an in-flight atomic write, and any processing interrupted mid-stage just restarts from the last committed checkpoint on relaunch.

**Testing strategy:** Pipeline stages are pure, fixture-in/fixture-out functions so gate classification, normalization invariants, simplification bounds, and export serialization are all fast, deterministic unit tests — no camera, no UI. The four locked fixtures (icon sketch, lettering, closed-shape illustration, messy-but-recoverable notebook photo) plus a deliberately-bad set (one photo per named rejection reason) plus an adversarial nested-closed-loop case (to catch winding-rule/fill bugs) run through the full pipeline with metric assertions: bounded node counts, no sub-3pt orphan fragments, correct winding/fill on export, verified via rasterize-and-diff against fixtures rather than just XML validity. The edit reducer gets adversarial sequence tests (delete-all, merge-with-self, redo-after-new-edit invalidates the stack). Checkpoint/resume is tested by writing a checkpoint, simulating a hard kill, and reloading — not by scripting real OS-level termination. XCUITest is reserved narrowly for things provable only on a real render tree: the Workspace gesture contract (wipe-handle arming, tap-to-select never firing during a wipe drag, no double-tap-zoom delay), a real VoiceOver pass over path-list cleanup, and permission-denied → import-still-works.

## Final Output

**Architecture overview:** A thin SwiftUI app target owns navigation and screens only. All real logic lives in isolated local Swift Package targets with real `internal` access boundaries — capture, normalize, quality-gate, vectorize, edit, export, and persist are each their own module, plus one shared `CleanlineModels` package that every other module and the UI target depends on, so no lane can invent a competing shape for a cross-lane type. Two observable model layers: `AppModel` (`@Observable @MainActor`) owns the Recents list and trace lifecycle (start/open/delete); `TraceSession` (`@Observable @MainActor`, one instance per open trace) owns the active workflow — document, vector data, undo/redo stack, zoom/pan/selection, mode, and interruption-recovery state. A single `AppRouter.route(for:)` function is the only place that decides where a reopened/resumed trace lands. Heavy processing (Vision contour detection, Douglas-Peucker simplification, Bezier fitting, winding/fill classification) runs off the main actor in an actor-isolated `VectorizationEngine`; `TraceSession` calls into it and gets back plain `Sendable` value types. Undo/redo is a command log (`EditCommand`) applied through a pure reducer (`EditEngine.apply(_:to:)`), and commands are appended only on gesture commit/release — never mid-drag — which is what makes "resume from last committed checkpoint" mechanically true rather than aspirational.

**Data models:** `TraceDocument` is the durable metadata record (`metadata.json`) — id, timestamps, `TraceStatus` (the 3-value Recents-chip vocabulary: `needsReview`/`exported`/`traceFailed`), `lastCommittedStage` (`TraceStage`: `captured`/`normalized`/`traced`/`edited`), workspace mode, gate result, export status, and file references. It never carries vector geometry or EXIF/location/device data. `VectorDocument` (`vector.json`) is its own file-backed type — canvas size, `[VectorPath]`, global simplify strength — kept separate so Recents/routing never has to decode heavy geometry. `VectorPath`/`PathSegment` use native `CGPoint`/`CGSize`/`CGRect` directly (both Codable and Sendable on this deployment target) rather than hand-rolled DTO wrappers. `EditCommand` matches the exact signatures already locked in design_handoff (`deletePath(id:)`, `mergePaths(ids:resultID:)`, `setSimplifyStrength(pathID:value:)`). `RejectionReason` is the fixed seven-case named vocabulary. `NewTraceOutcome` (`.rejected(RejectionReason)` / `.accepted(traceID:)`) is the return type of starting a new trace — this is the mechanism that guarantees a gate-rejected capture never leaves a ghost folder or Recents entry: the folder is deleted internally on rejection, and a `TraceDocument`/status only exists once the gate has passed.

**Persistence strategy:** Per-trace file packages under Application Support, OS-default backup enabled (this is deliberate — "local-first" means not shipping sketches to our servers or third-party SDKs, not opting a user's own work out of Apple's encrypted device backup). No Core Data/SwiftData. All checkpoint writes are atomic temp-file-then-rename, with an explicit test that simulates a kill mid-write and asserts the prior checkpoint survives intact. The Recents list is a rebuildable cache, not a source of truth — on any inconsistency it's reconstructed by scanning trace folders and reading each `TraceDocument`. Durable storage begins only after the quality gate passes; a rejected capture's transient folder is deleted by `AppModel` as part of handling the rejection, never left as orphaned disk state. `TraceStore` owns the atomic write/read/delete contract exactly as specified in the interfaces block below. `PrivacyInfo.xcprivacy` is a locked MVP requirement (file-timestamp and UserDefaults usage trigger required-reason declarations even with zero third-party SDKs). No `BGProcessingTask`; a short `beginBackgroundTask` grace window covers finishing an in-flight atomic write, and any processing interrupted mid-stage just restarts from the last committed checkpoint on relaunch.

**Testing strategy:** Pipeline stages are pure, fixture-in/fixture-out functions so gate classification, normalization invariants, simplification bounds, and export serialization are all fast, deterministic unit tests — no camera, no UI. The four locked fixtures (icon sketch, lettering, closed-shape illustration, messy-but-recoverable notebook photo) plus a deliberately-bad set (one photo per named rejection reason) plus an adversarial nested-closed-loop case (to catch winding-rule/fill bugs) run through the full pipeline with metric assertions: bounded node counts, no sub-3pt orphan fragments, correct winding/fill on export, verified via rasterize-and-diff against fixtures rather than just XML validity. The edit reducer gets adversarial sequence tests (delete-all, merge-with-self, redo-after-new-edit invalidates the stack). Checkpoint/resume is tested by writing a checkpoint, simulating a hard kill, and reloading — not by scripting real OS-level termination. XCUITest is reserved narrowly for things provable only on a real render tree: the Workspace gesture contract (wipe-handle arming, tap-to-select never firing during a wipe drag, no double-tap-zoom delay), a real VoiceOver pass over path-list cleanup, and permission-denied → import-still-works.

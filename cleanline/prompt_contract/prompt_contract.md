Now my read on this, coming at it looking for what breaks.

First, a scope trap that I want flagged before anyone builds anything: this phase's file is scoped to a single child app — cleanline — but the prompt it's built from is the *parent portfolio* prompt, which talks about 12 candidate ideas, choosing 6, Jira/Notion backfill, parallel app efforts, combined transcripts. That parent-level content is not this phase's job. cleanline has already been selected — the "generate 12, pick 6" instruction is closed business from an earlier phase of the portfolio run, not something this phase re-litigates. If a later phase for this child project starts second-guessing whether cleanline should have been picked, or tries to regenerate candidate ideas, that's scope creep into a decision the parent process already made. The contract needs to say explicitly: for this child project, the app is fixed as cleanline; portfolio-level instructions (idea generation, selection, cross-app uniqueness enforcement, Jira/Notion, combined transcript file) are inherited constraints to respect, not tasks to redo here.

Second, the core technical claim in the one-sentence promise is the single biggest risk in the whole project and I don't think it should be accepted at face value: "traces sketch to clean, editable vector paths via on-device Vision." Vision framework (VNDetectContoursRequest, VNDetectRectanglesRequest, etc.) gives you contour detection and rectangle/edge detection — it does not give you a hand-drawn sketch turned into artist-clean, closed, editable Bezier paths comparable to Illustrator's Image Trace or Adobe's vectorization. Real sketches have broken linework, overlapping strokes, shading, paper texture, shadows from the phone camera, and non-uniform lighting. A naive contour trace on that input produces a noisy mess of tiny disconnected paths, not "clean" vectors. This needs to be named as an assumption now, not discovered during build: the MVP's vectorization pipeline is a custom algorithm (perspective correction → binarization/adaptive threshold → noise reduction → contour extraction via Vision → path simplification/smoothing, e.g. Douglas-Peucker + curve fitting) built on top of Vision primitives, not something Vision does for you out of the box. If we let a later phase assume Vision "just does" vectorization, the build phase will hit this wall late, which is expensive. I'd rather the architecture phase commit explicitly to a bounded first version of this pipeline (e.g., single dominant line color, decent lighting, reasonably high contrast paper) and treat "handles messy shading/multiple pens/poor lighting" as an explicit non-goal for MVP, with a fallback/manual-cleanup UI (adjustable threshold slider, eraser/merge tools on the traced paths) as the safety net when auto-trace is imperfect — because it will be imperfect often.

Third, "editable vector paths (SVG/PDF)" is doing a lot of work. Editable to what degree? If a user can only view/export the SVG but not actually manipulate individual anchor points/curves inside the app, that's not "editable," that's "exportable." I'd treat true in-app Bezier-path editing (drag anchor points, adjust curve handles, delete/merge segments) as a hard requirement implied by "editable," not optional polish — otherwise the one-sentence promise is false advertising. This has real scope: it means we need a path data model, a canvas renderer, hit-testing on path segments/anchors, undo/redo, and it needs empty/loading/success/error states like everything else. If time-boxing forces a cut, the decision rule should be: cut sophistication of the auto-trace before cutting the ability to manually edit the resulting paths — because manual editing is the escape hatch that makes an imperfect trace still commercially useful to an illustrator.

Fourth, monetization: requirement 5 demands "genuine value, never cosmetics-only" subscription. For a vectorization tool, the honest premium wall is around export formats, resolution/complexity limits, and batch/cloud-ready features — not filters. I'd flag the risk that a lazy design pass tries to gate "SVG export" itself behind paywall while leaving PDF export free, which would gut the core promise for free users and read as bait-and-switch. Decision rule I want on record: the core promise (photograph → get *some* usable vector output) must work free; monetization should live in higher-fidelity tracing, batch processing, cloud sync/backup, or advanced editing tools, not in blocking the basic value prop.

Fifth, production-readiness. "0 to production-ready" for a camera+Vision+vector-editing app is a nontrivial bar. I want it defined concretely, not left vague: camera permission handling with a real denied/restricted state (not a crash or blank screen), photo library import fallback for users without live camera access (simulator testing, iPad without camera, permission denied), handling of low-quality/blurry/no-edges-detected input with a real error state and retry path, local persistence of projects/traces (Core Data or file-based, survives app relaunch), an actual export flow to Files/Share sheet that produces valid SVG/PDF files (this needs to be verified by opening the exported file in another app, not just "code compiles"), accessibility (VoiceOver on canvas editing is genuinely hard — decide now whether canvas interactions get real accessibility labels/adjustable actions or whether that's a named limitation), and it needs to run on a real device build path since Vision camera features are meaningfully limited/different on simulator. If "production-ready" is asserted without someone having exported a file and opened it in Preview/Illustrator, that claim is unverified.

Sixth, decision-rule ask for conflicts: when the parent portfolio prompt and this child spec disagree (e.g., portfolio says "local-first, cloud-later architecture" broadly; child spec doesn't mention cloud at all), default to the parent's general engineering requirement (local-first, architected for later cloud) since it's a floor requirement across the whole batch, and name that as an assumption rather than silently dropping it.

My biggest single risk to flag for the record: if the team proceeds as though Vision hands you clean vectors for free, the implementation plan and time estimates built in later phases will be wrong, and QA won't have a real bar to test against because "clean" was never defined. I want the contract to pin down a concrete, testable definition of "clean trace" (e.g., no stray path fragments below N px, closed paths where the source line is closed, smooth curves not jagged pixel-staircases) before build starts, so there's something to actually verify against rather than a subjective vibe check at the end.

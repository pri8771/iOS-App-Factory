### Round 1


**Codex (Design Lead · Skeptic) — Round 1**

The precise promise should be tighter than the original phrasing: cleanline lets an illustrator photograph a paper sketch and leave with a usable black-line vector file on their phone in a few minutes, with just enough cleanup to trust the export. If we do not narrow it that hard, the product drifts into “general-purpose vectorizer” and dies under its own ambition.

The primary user is not “anyone who doodles.” It is a working or semi-pro illustrator, lettering artist, or print-on-demand seller who already thinks in terms of SVG, cut files, line cleanup, and downstream editing in Illustrator, Affinity, or Cricut-style workflows. The scenario is concrete: they have a hand-drawn icon, phrase, or line illustration on paper, need to digitize it quickly, and do not want to rescan, deskew, clean background noise, and retrace manually on desktop just to get to a usable starting file.

The core loop should be brutally short. Start a new trace. Capture or import a sketch photo. The app immediately evaluates whether the photo is salvageable, because silent failure here would poison trust fast. If the image is good enough, it auto-corrects perspective, isolates the linework, runs the vector trace, and shows a before/after comparison. Then the user gets a very small set of cleanup actions that map to real failures: remove stray path, adjust simplification/smoothing, merge broken stroke segments, undo. Then export SVG or PDF and save/share the file. That is the product. A gallery of prior traces and the ability to reopen/edit/export them later is in scope because without persistence this becomes a disposable demo, not a tool.

The hard scope boundary I would defend is that this is not an illustration app and not a full vector editor. No freehand drawing canvas, no layers, no brush packs, no text tool, no color fills workflow, no node-by-node Bezier handle editing, no “AI style” transformations, no cloud dependency, no marketplace or storefront angle. It also should not pretend to support every kind of source art. MVP should explicitly target high-contrast monochrome linework on plain paper. Pencil shading, watercolor, colored illustrations, dense crosshatching, and multi-object page extraction are all places where the promise breaks unless we intentionally exclude or degrade gracefully.

The biggest UX risk is not the tracing algorithm in the abstract; it is false confidence. A user will forgive “retake this photo, too much shadow on the lower-right” much faster than they will forgive exporting garbage that looked acceptable in-app but opens as a thousand junk fragments in Illustrator. The app needs an explicit quality gate before tracing and an explicit confidence model after tracing. If the capture is outside supported bounds, say so early. If the trace result is likely poor, surface that before export. Premium-feeling here means honesty and control, not magic theatrics.

The detail later phases need to preserve is that cleanup tools must correspond to actual trace failure modes rather than generic editing affordances. “Adjust” cannot just be a pretty slider with vague labels. It needs to visibly reduce node complexity or reconnect broken strokes. “Delete” must operate on selected stray paths, not erase pixels. “Merge” must solve fragmented line segments, not be decorative. If those tools are not concretely tied to better exports, they are fake product surface.

Measurable success criteria should be strict. A first-time user can go from opening the app to exporting a valid SVG in under three minutes on a decent source photo. The exported file opens in a mainstream vector tool without errors. On the fixture set from the contract, primary strokes remain continuous, junk fragments under the agreed threshold are absent, and node counts are low enough that downstream editing is realistic. The app must also handle the bad-path cases cleanly: denied camera access, unsupported image, no detectable document/sketch, trace failure, and export failure all need explicit recoverable states. If we cannot meet those bars, the right move is to narrow supported input further, not to widen scope.

**Claude (QA & Risk · Pragmatist) — Round 1**

Given everything already locked in the prompt contract, my job this round is to pin down the actual product definition tightly enough that nobody drifts later, and to stress-test it from a "how do we know this actually works" angle before anyone starts writing Swift.

The one-sentence promise I'd commit to: cleanline turns a phone photo of a paper sketch into a clean vector file (SVG/PDF) an illustrator can actually use downstream, by running it through an on-device correction-and-tracing pipeline, not just a raw contour dump. I'd deliberately keep "traces to editable vector paths via Vision" out of the promise sentence itself, because we already agreed Vision alone doesn't do this — the promise should describe the outcome (usable vector file), not the underlying framework, so we're not making a technical claim in marketing copy that QA then has to defend literally.

Primary user and scenario: a single illustrator or print-on-demand seller, working alone, who just finished a pen/pencil sketch on paper and wants a vector version to put in a listing, cut file, or downstream art file — today. Not a studio workflow, not a team handoff, not batch digitizing a sketchbook of fifty drawings (batch is explicitly a paid, later concern per the contract). The scenario is: one sketch, one photo, one export, a few minutes. If the core loop takes longer than a quick coffee-break task, or requires the user to already understand vector concepts like paths and nodes, we've missed the target user, because POD sellers are not necessarily trained illustrators-with-Illustrator-skills — some are, some aren't, and the app has to work for the ones who just want a file they can upload to Printify or similar without knowing what a Bezier curve is.

Core loop: capture or import a sketch photo → app runs it through correction (crop/perspective/threshold) → auto-vectorize → user reviews the trace against the source, does light cleanup (delete a stray path, nudge simplify strength, merge two paths) → export SVG/PDF → done. That's it. One sketch in, one file out, with a review-and-fix step in the middle because we already agreed auto-trace won't be perfect and the fix-it-yourself step is the safety net, not a bonus feature.

The scope boundary I want to be most explicit and least negotiable about, because it's the one most likely to quietly expand during build: no multi-sketch project management, no libraries/collections UI beyond "your last few traces persist locally," no cloud anything in MVP, no batch import queue. The contract already says batch is a paid future feature — I want that reflected here as a hard MVP boundary, not just a monetization idea, because "just add a simple gallery of past traces" is exactly the kind of thing that sounds small and isn't. A flat, bounded local history (say, list of recent projects, tap to reopen, delete) is fine and probably necessary for persistence to mean anything; a searchable/organizable project library is not MVP.

On measurable success criteria, I want numbers that QA can actually check rather than "feels good": time from opening the app with a sketch in hand to a completed export should be achievable in under 3–5 minutes for a well-lit, reasonably clean sketch (this is a UX/perf bar, not a trace-quality bar — those exist already in the contract). Second, on the fixture set we already committed to (icon sketch, lettering, closed-shape illustration, messy notebook photo), the auto-trace-plus-light-cleanup path should get all four to a passing "clean trace" export without the user needing anything beyond the defined MVP editing tools — if any fixture requires editing capability we explicitly excluded (anchor dragging) to reach an acceptable result, that's a signal the auto-trace pipeline is under-baked, not that we need to expand editing scope. Third, failure honesty is itself measurable: for inputs outside the named capture-rescue bounds (too much skew, too much shadow, too low contrast), the app should reliably produce the "retake photo" error state rather than a garbage export — I'd want this checked against deliberately bad test photos, not just good ones, because an app that only gets tested on ideal input will look done and then embarrass itself on the first real notebook photo a beta user throws at it.

The biggest risk I want flagged going into design/architecture: persistence and interruption handling. Camera capture, then a multi-second on-device processing pipeline, then an editing session — that's three places the app can get backgrounded, killed by memory pressure, or interrupted by a phone call, and if state isn't saved incrementally (source photo persisted immediately on capture, trace results persisted once computed, edits autosaved rather than held only in memory until export), a user can lose real work, which for a five-minute task feels disproportionately bad and will kill trust fast. I'd rather we name "autosave at every pipeline stage" as a hard requirement now than discover during build that the happy-path demo works but backgrounding the app mid-edit loses the trace.

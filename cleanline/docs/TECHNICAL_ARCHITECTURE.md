## Final Output

**Screen inventory (six screens + one modal sheet):**

1. **Recents Home** — app root, no tab bar. Prominent "New Trace" action, flat list of past traces (thumbnail, date, status chip: needs review / exported / trace failed), small Settings entry.
2. **New Trace Intake** — two large choices, Camera or Import Photo, plus a fixed scope-defense note ("dark linework on plain paper, one sketch at a time"). Camera-denied keeps the camera card visible with an explanation and system-settings link, while Import stays fully usable.
3. **Camera Capture** — live capture screen, permission-denied handled as a first-class in-screen state (not a system-alert dead end), with a same-screen "Import a Photo Instead" escape hatch.
4. **Quality Gate + Correction Review** (combined, one screen with variants) — checking (brief transitional state), rejected-with-named-reason (one of the seven fixed rejection reasons, each with stable, specific copy/iconography plus a real accessibility label), corrected-preview-ready (original vs. corrected, Continue/Retake), correction-failed, and resuming-from-checkpoint. Rejection is a variant of this same state machine, not a separate destination.
5. **Processing** — dedicated screen, corrected image faintly visible behind it, named sequential stage labels (checking geometry → correcting perspective → isolating linework → tracing paths → preparing preview). No fake progress bar. During the tracing-paths stage specifically, the app renders real vector strokes as they're actually computed (a literal application of "no fake progress," not a separate feature) — if incremental streaming turns out not to be practical, this stage falls back to the same plain dimmed-label treatment as the other stages, no synthetic animation standing in for unfinished work. Explicit trace-failed error state with Retry / Adjust Photo as distinct recoveries.
6. **Trace Workspace** — the unified post-trace destination. One canvas, one shared render of the path data, two bidirectional modes (Compare / Edit) via a lightweight toggle, never a one-way "continue" gate. Compare shows corrected-image-vs-vector with an interactive wipe/slider for sighted users and a non-gestural Original/Traced segmented toggle as the accessible default fallback. Edit shows the fixed, always-visible bottom tool strip (select/delete, simplify slider, merge, undo/redo) plus the VoiceOver-accessible list-based path management mode. Reopened traces land here by default (Compare if untouched, Edit if cleanup was in progress). Zoom/pan/scroll state is a single piece of view state that never has to survive a navigation transition, since there is no transition between Compare and Edit.
7. **Export** — a modal sheet summonable from either Workspace mode (not a separate screen/navigation push). States: ready, exporting, success (share sheet), failed-but-saved (explicit reassurance the trace remains safe in Recents).

Plus **Settings** — plain-language local-first/privacy statement, honest non-purchasing "Pro — coming soon" note, no growth hooks or fake locked controls.

**Primary user flow:** Home → New Trace → Camera or Import → Quality Gate (pass) → Correction Review → Processing (named stages, real-data reveal on tracing) → Trace Workspace (Compare ⇄ Edit, freely, both ways) → Export sheet (from either mode) → back to Home with saved result. Any quality-gate rejection is a same-screen variant with Retake/Choose Another/Cancel, not a detour to a new destination. Reopening a saved trace skips straight to the Workspace at its last state.

**State model per screen:** as detailed above per screen — every screen has explicit empty/loading/success/error coverage where the flow can reach it, and interruption/kill-and-relaunch always resumes at the last **committed** checkpoint (never mid-gesture, never a partially-run Vision request).

**Visual direction:** A drafting-table / light-table aesthetic, not a generic productivity app or a cutesy scanner app. Warm, slightly warm-neutral paper-tone base surfaces (not stark white, not dark-developer-tool), true black ink for the traced linework as the only saturated element on screen, one sparing sharp technical accent color (e.g., oxidized teal or safety orange) reserved for active/warning states. Clean geometric sans typography, editorial/industrial rather than startup-default. Chrome and controls stay quiet and recede; the sketch and vector result carry all the visual weight — this matters directly for the shareable before/after moment the growth strategy depends on. Motion is functional: stage transitions reinforce real pipeline progress, no decorative flourish, no canned animation standing in for unfinished work.

**Accessibility notes:** Seven rejection-reason screens are visually distinct (icon/color) for sighted pattern-matching but every icon carries a full-text accessibility label stating the actual reason — visual pattern-matching is a bonus, never a substitute for text. Compare mode's wipe/slider gesture has a mandatory non-gestural Original/Traced toggle fallback for VoiceOver. Edit mode's cleanup tools are fully operable through the accessible path list (select/delete/merge by name) with zero required canvas interaction, to be verified with an actual VoiceOver session. The tracing-stage path-reveal animation needs a reduce-motion variant that still communicates progress via updating text stage labels rather than either forcing motion or going silent. Dynamic Type must hold up without truncating recovery actions on rejection/error screens especially, since that copy carries real communicative weight.

## Final Output

**Backup policy:** Trace folders stay in normal, OS-default backup (no `isExcludedFromBackup`). "Local-first / never leaves the device" is about not shipping sketches to our servers or third-party SDKs — it's not about opting a user's own work out of Apple's encrypted device backup. Excluding it would trade a real, eventual, irreversible loss (phone swap/restore wipes all traces) for a theoretical privacy gain against a threat model (iCloud backup) that every other photo on the phone is already subject to. This reasoning gets written into tech_specs directly so nobody "improves privacy" later by flipping the flag.

**Privacy manifest:** `PrivacyInfo.xcprivacy` is a locked MVP requirement, not a pre-submission afterthought. The app will touch required-reason APIs (file timestamps for Recents sorting/index-rebuild, likely UserDefaults) even with zero third-party SDKs, so this is a near-certain App Store gate, not a hypothetical. Goes into tech_specs as an explicit checklist item.

**Background execution:** No `BGProcessingTask`, no background job scheduling system. A short `beginBackgroundTask` grace window covers finishing an in-flight atomic checkpoint write if backgrounded mid-save. If suspension hits during Vision/curve-fitting, the app just restarts that stage from the last committed checkpoint on relaunch — a few seconds of rework, not data loss, which is already consistent with the checkpoint model. Reaching for real background processing APIs was judged not worth the complexity for a "save the user a five-second wait occasionally" payoff.

**Vectorization as a first-class subsystem:** Both agree this can't be a line item under "use Vision." Vision only supplies rectangle/contour detection — everything that actually defines trace quality (contour cleanup, stroke-continuity heuristics, Douglas-Peucker simplification, Bezier fitting, open/closed classification, winding/fill-rule correctness, SVG/PDF serialization) is bespoke code that needs its own module boundary, its own fixture-based test suite (the four locked sketches plus adversarial cases like nested closed loops to catch winding-rule bugs), and named tunable constants (simplify-strength-to-epsilon mapping, min-fragment-length threshold, max-node-count ceiling) instead of magic numbers. tech_specs must scope this explicitly or the hardest part of the app will be the least-specified part.

**One more item folded in this round:** atomic checkpoint writes (temp-file-then-rename) get called out as an explicit named requirement with an explicit test (simulate a kill mid-write, assert the prior checkpoint is intact) — not because it's architecturally hard, but because "did someone remember to do it" bugs are exactly the kind that surface six weeks post-launch on a real user's dying phone.

Net architecture, fully agreed: thin SwiftUI shell over isolated Swift Package targets (capture/normalize/gate/vectorize/edit/export/persist), `@Observable` `TraceSession` on the main actor, all first-party frameworks, file-backed per-trace document packages (no Core Data/SwiftData), zero third-party dependencies, zero StoreKit code, no background-processing machinery, OS-default backup allowed, privacy manifest required, and vectorization elevated to a first-class subsystem with its own tests and named parameters.

## Final Output

**Architecture overview:** A thin SwiftUI app target owns navigation and screens only. All real logic lives in isolated local Swift Package targets with real `internal` access boundaries — capture, normalize, quality-gate, vectorize, edit, export, and persist are each their own module, plus one shared `CleanlineModels` package that every other module and the UI target depends on, so no lane can invent a competing shape for a cross-lane type. Two observable model layers: `AppModel` (`@Observable @MainActor`) owns the Recents list and trace lifecycle (start/open/delete); `TraceSession` (`@Observable @MainActor`, one instance per open trace) owns the active workflow — document, vector data, undo/redo stack, zoom/pan/selection, mode, and interruption-recovery state. A single `AppRouter.route(for:)` function is the only place that decides where a reopened/resumed trace lands. Heavy processing (Vision contour detection, Douglas-Peucker simplification, Bezier fitting, winding/fill classification) runs off the main actor in an actor-isolated `VectorizationEngine`; `TraceSession` calls into it and gets back plain `Sendable` value types. Undo/redo is a command log (`EditCommand`) applied through a pure reducer (`EditEngine.apply(_:to:)`), and commands are appended only on gesture commit/release — never mid-drag — which is what makes "resume from last committed checkpoint" mechanically true rather than aspirational.

**Data models:** `TraceDocument` is the durable metadata record (`metadata.json`) — id, timestamps, `TraceStatus` (the 3-value Recents-chip vocabulary: `needsReview`/`exported`/`traceFailed`), `lastCommittedStage` (`TraceStage`: `captured`/`normalized`/`traced`/`edited`), workspace mode, gate result, export status, and file references. It never carries vector geometry or EXIF/location/device data. `VectorDocument` (`vector.json`) is its own file-backed type — canvas size, `[VectorPath]`, global simplify strength — kept separate so Recents/routing never has to decode heavy geometry. `VectorPath`/`PathSegment` use native `CGPoint`/`CGSize`/`CGRect` directly (both Codable and Sendable on this deployment target) rather than hand-rolled DTO wrappers. `EditCommand` matches the exact signatures already locked in design_handoff (`deletePath(id:)`, `mergePaths(ids:resultID:)`, `setSimplifyStrength(pathID:value:)`). `RejectionReason` is the fixed seven-case named vocabulary. `NewTraceOutcome` (`.rejected(RejectionReason)` / `.accepted(traceID:)`) is the return type of starting a new trace — this is the mechanism that guarantees a gate-rejected capture never leaves a ghost folder or Recents entry: the folder is deleted internally on rejection, and a `TraceDocument`/status only exists once the gate has passed.

**Persistence strategy:** Per-trace file packages under Application Support, OS-default backup enabled (this is deliberate — "local-first" means not shipping sketches to our servers or third-party SDKs, not opting a user's own work out of Apple's encrypted device backup). No Core Data/SwiftData. All checkpoint writes are atomic temp-file-then-rename, with an explicit test that simulates a kill mid-write and asserts the prior checkpoint survives intact. The Recents list is a rebuildable cache, not a source of truth — on any inconsistency it's reconstructed by scanning trace folders and reading each `TraceDocument`. Durable storage begins only after the quality gate passes; a rejected capture's transient folder is deleted by `AppModel` as part of handling the rejection, never left as orphaned disk state. `TraceStore` owns the atomic write/read/delete contract exactly as specified in the interfaces block below. `PrivacyInfo.xcprivacy` is a locked MVP requirement (file-timestamp and UserDefaults usage trigger required-reason declarations even with zero third-party SDKs). No `BGProcessingTask`; a short `beginBackgroundTask` grace window covers finishing an in-flight atomic write, and any processing interrupted mid-stage just restarts from the last committed checkpoint on relaunch.

**Testing strategy:** Pipeline stages are pure, fixture-in/fixture-out functions so gate classification, normalization invariants, simplification bounds, and export serialization are all fast, deterministic unit tests — no camera, no UI. The four locked fixtures (icon sketch, lettering, closed-shape illustration, messy-but-recoverable notebook photo) plus a deliberately-bad set (one photo per named rejection reason) plus an adversarial nested-closed-loop case (to catch winding-rule/fill bugs) run through the full pipeline with metric assertions: bounded node counts, no sub-3pt orphan fragments, correct winding/fill on export, verified via rasterize-and-diff against fixtures rather than just XML validity. The edit reducer gets adversarial sequence tests (delete-all, merge-with-self, redo-after-new-edit invalidates the stack). Checkpoint/resume is tested by writing a checkpoint, simulating a hard kill, and reloading — not by scripting real OS-level termination. XCUITest is reserved narrowly for things provable only on a real render tree: the Workspace gesture contract (wipe-handle arming, tap-to-select never firing during a wipe drag, no double-tap-zoom delay), a real VoiceOver pass over path-list cleanup, and permission-denied → import-still-works.

## Project Plan

Round 2 actually closed this out. Codex took both open items from round 1 and didn't just acknowledge them — he restructured the milestone plan around them: the orphan-folder reconciliation gap moved from "worth flagging" to an explicit exit criterion of milestone one (a named startup sweep, five kill-test points instead of four), and vectorization got split into "milestone two-a: spike on the simplest fixture" followed by "milestone three: production," with fake service implementations and fixture production promoted to explicit milestone-one deliverables rather than implementation texture.

Claude then went one level deeper and found something concrete rather than just re-flagging the same worry: the locked interfaces actually contradict themselves on this exact point — `TraceStage`'s note says `captured` is persisted the instant bytes exist (implying a resumable checkpoint pre-gate), but `TraceStatus`'s note says a document only exists once the gate passes, and there's no "pending gate" status case, and `writeCheckpoint` requires a full `TraceDocument` for every stage including `.captured`. Both can't be literally true. Claude's fix is concrete and small: captured bytes land in a staging location keyed by a provisional ID (no `TraceDocument`, no Recents visibility) until the gate resolves; only then does a real `TraceDocument` get constructed and handed to `TraceStore`. That makes the reconciliation sweep trivial to spec (scan staging, not Recents, on launch; re-run the gate or delete). Codex hasn't typed a reaction to this specific mechanism yet since it came after his message, but it's not a new disagreement — it's a direct, literal answer to the "named startup sweep" he asked for, using a resolution method that doesn't touch or conflict with anything he said. Both agents also aligned on relative sizing: milestone one gets more buffer than its "just file I/O" reputation suggests because this staging ambiguity has to be actually resolved and tested; milestone three is unsized until the spike reports back; milestone four is deliberately small given the narrow edit surface; milestone five gets generous room because that's where kill-tests, resume banners, and staging-reconciliation all get proven together as a system for the first time.

## Final Output

**Locked delivery plan for cleanline, five milestones, dependency-ordered (not screen-ordered):**

**Milestone 1 — Foundation: models, persistence, staging/reconciliation, fixtures, fakes.**
Build `CleanlineModels` (frozen from tech_specs), `AppModel`/`TraceSession`/`AppRouter` skeletons, and `TraceStore` with atomic temp-file-then-rename checkpoint writes. Critically, this milestone also resolves and implements the staging mechanism for pre-gate captures: captured bytes are written to a staging location keyed by a provisional ID, with no `TraceDocument` and no Recents visibility, and a real `TraceDocument` is only constructed once the quality gate resolves to accepted. A named startup reconciliation sweep scans staging (not the Recents folder tree) on every launch, and either re-runs the gate against any leftover provisional capture (promoting to a real document on accept) or deletes it deterministically (on reject or on missing/corrupt data). Exit criteria for this milestone: a kill-mid-write test proving the prior checkpoint survives untouched, plus five interruption-point kill tests — captured, normalized, traced, edited, and the mid-gate-evaluation staging gap — all passing before any other lane is allowed to build against `TraceStore` for real. This milestone also delivers, as explicit deliverables (not side effects): fixture production (the four locked sketches, the seven-reason bad-photo set, the adversarial nested-loop case) starting day one since it has zero code dependency, and fake/protocol-backed implementations of every service (`CaptureService`, `ImportService`, `QualityGateService`, `VectorizationEngine`) returning canned-but-fixture-realistic results, so downstream lanes never serialize behind the hardest subsystem. No auth/login scaffolding — there is none in this app.

**Milestone 2 — Capture, import, normalization, quality gate.**
Real camera/import intake, orientation-safe `CapturedImage` handling into the staging mechanism from milestone 1, normalization (perspective, EXIF strip), and the seven named rejection reasons with the combined quality-gate/correction screen (checking / rejected / corrected-preview-ready / correction-failed / resuming-from-checkpoint variants). First fully user-visible slice: import a fixture or real photo, get either a named rejection or a corrected preview, survive camera-permission denial cleanly. Mitigation against false acceptance (bad input silently passing the gate and surfacing later as a "vectorization problem"): threshold tuning validated against the full bad-photo fixture set, not just a couple of good photos.

**Milestone 2a — Vectorization spike (first task inside the vectorization milestone, not a separate pre-step).**
Run the single simplest fixture (icon sketch) all the way from normalized image through contour extraction, simplification, curve fitting, to a rasterized-and-diffed SVG, with node-count reporting. This is a hard go/no-go checkpoint: it calibrates the real estimate for milestone 3 rather than treating vectorization like a normally-sizable feature. If the locked clean-trace bar isn't reachable on even the simplest case in the time budgeted, that's the moment to renegotiate scope — not three weeks before ship.

**Milestone 3 — Production vectorization + before/after review.**
Full actor-isolated `VectorizationEngine` covering all four fixture classes (icon, lettering, closed-shape, messy notebook), named progress stages, and the before/after review screen. Acceptance bar is the locked one: bounded node counts, no sub-3pt orphan fragments, continuous strokes where expected, correct winding/fill, clean downstream-open in a real vector tool. Explicit rule carried from round 1 and reaffirmed by both agents: if this milestone underperforms, fix the tracing heuristics — do not compensate by growing editor scope.

**Milestone 4 — Bounded cleanup + export.**
Select/delete path, debounced simplify-strength slider (downsampled preview, full-res on commit), merge paths, undo/redo through `EditEngine.apply` as a pure reducer, VoiceOver-accessible path list, and the SVG/PDF export sheet. Deliberately the smallest, lowest-risk milestone by design, since the edit surface is narrow and operates on an already-stable `VectorDocument`. Can start against milestone 1's fake services in parallel with milestone 3 if vectorization is running long.

**Milestone 5 — Integration hardening and trust polish.**
Recents home, reopen/delete flows, resume banners (with `acknowledgeInterruptionRecovery()` wired so banners show once), export-failure recovery, settings/privacy copy, full empty/loading/success/error coverage, and the complete kill-and-relaunch matrix proven end-to-end as a system (not just per-checkpoint unit tests). This is also where `PrivacyInfo.xcprivacy` gets finalized against actual API usage and real-device performance/thermal validation (debounced slider, Vectorize under load) happens on physical hardware, not just simulator.

**Lanes:** dependency-aligned, not screen-aligned — (1) models/persistence/staging/reconciliation, (2) capture/normalize/gate, (3) vectorization/export correctness, (4) SwiftUI screens/router/accessibility on top of the other three's contracts. Integration and checkpoint verification run continuously starting in milestone 1, never as an end-of-project phase.

**Core risk framing both agents converged on:** this app's real danger isn't a missing feature, it's internal state silently meaning something other than what the code assumes — on the lifecycle axis (a captured-but-not-gated folder that isn't what its checkpoint stage implies) and on the quality axis (a photo that passes the gate but shouldn't have). Both get mitigated the same way: make the ambiguous states impossible to represent (staging vs. real `TraceDocument`) or impossible to miss (fixture-set thresholds tested against deliberately bad inputs, not just good ones).

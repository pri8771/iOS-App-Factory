The App Store description needs an explicit scope line — dark linework on plain paper, one sketch at a time — for the same reason the in-app copy has one: so refusal reads as honest product behavior, not broken software, in a one-star review context where nobody's read a design doc.

Screenshot plan: this is a Graphics & Design category listing, so the screenshots are the pitch. I'd want, in order: Recents populated with real varied thumbnails (not three identical fixture icons — that reads as fake), Camera capture mid-frame with a real sketch, the Compare mode before/after at base zoom (this is the one screen the entire growth strategy leans on per the product brief, so it needs to be the hero shot, not an afterthought), Edit mode showing the tool strip actually mid-cleanup, one quality-gate rejection card (proves the "honest, not gimmicky" positioning rather than hiding failure states), and the Export sheet with a real SVG icon. All of these need to come from actual captured screenshots post-runtime-verification, not mockups — and note Apple's current device-size requirements mean at minimum a 6.9" and a 6.5" (or 5.5") screenshot set; skipping a required size is a mechanical rejection, not a judgment call.

Privacy label needs a distinction drawn carefully because it's easy to conflate two different Apple requirements here. The `PrivacyInfo.xcprivacy` manifest (already locked as required in architecture) is about declaring required-reason API usage — file timestamps for Recents sorting, UserDefaults — and that's a technical compliance artifact, not a data-collection disclosure. The separate App Store Connect privacy "nutrition label" questionnaire is where the actual "Data Not Collected" claim goes, and given no analytics SDK, no network calls, no accounts, no third-party dependencies, that answer should genuinely be "Data Not Collected" across the board. The risk is if whoever fills out that questionnaire in App Store Connect doesn't realize using required-reason APIs doesn't obligate a "we collect data" answer — those are separate questions — and either over-discloses out of caution or under-discloses because the manifest already exists. I want both filled out correctly and separately, not assumed to be the same form.

Permission copy has a real technical nuance that could bite here: if intake is genuinely camera-plus-PHPickerViewController with no direct Photos-library authorization calls, Apple does not require an `NSPhotoLibraryUsageDescription` string at all — PHPicker runs out-of-process specifically so apps can avoid that permission entirely. Earlier phases locked "camera/photo-library usage-description strings must be specific and true," which I'd now narrow: only include a photo-library usage string if the code actually calls an API that triggers the system permission prompt; if it's pure PHPicker, a photo-library string sitting unused in Info.plist is at best confusing and at worst something a reviewer flags as inconsistent with actual app behavior. Someone needs to check what `ImportService`'s real implementation calls before locking that copy. Camera needs a real, specific `NSCameraUsageDescription` — "used to photograph your paper sketch for tracing," not boilerplate — and that one's a genuine requirement since live capture goes through AVFoundation.

The subscription/paywall situation is where I think there's a sharper review risk than has been named so far. There is no live purchase anywhere — good, that's clean under Guideline 3.1.1. But the "Pro — coming soon" settings block, naming specific unreleased features (batch, fine-detail tracing, pro editing, cloud sync), is itself a review-risk pattern. Apple's Guideline 2.3.1 on accurate metadata has a real history of rejecting or flagging apps that reference future functionality, in-progress work, or "coming soon" placeholders, on the theory that the current build should be reviewed as it is, not as it will be. I'm not saying this will definitely get bounced, but it's a real, named risk that needs a mitigation on record rather than discovery at review time: keep the language aspirational and non-committal rather than a feature roadmap, avoid anything that reads as "you're paying for something today that's incomplete," and have a fallback ready (cut the block down to a single local-first privacy statement with zero forward-looking feature language) if this specific screen gets flagged in review.

A few concrete gaps nobody has confirmed exist or don't: an actual `AppIcon` asset — I don't see this as a deliverable in any locked task, and a missing or placeholder icon is an automatic submission blocker, not a soft one. A support URL and contact — App Store Connect requires this at listing time and nothing in the build addresses it; for a no-backend, no-account app this can be as simple as a static page or a support email, but it has to exist before submission, not be improvised at upload time. And device-family restriction: this app is locked iPhone-first with no iPad layout work done anywhere, but unless `project.yml`/Info.plist explicitly restricts `TARGETED_DEVICE_FAMILY` to iPhone, Xcode's default can quietly make the app installable on iPad with an unadapted, letterboxed, or broken layout — that's a real, checkable thing, not a hypothetical, and it needs a yes/no answer before submission rather than an assumption.

Age rating should be straightforward — 4+, no objectionable content, no user-generated content shared with others, no web view, no ads, no third-party data sharing — but I want that stated as a positive claim in the final output, not left implicit.

What the final output has to preserve, whatever else changes in discussion: an explicit, undodged statement that launch-readiness and runtime-verified-working are not yet the same claim for this app; the "outcome not mechanism, name the scope fence" positioning rule; the privacy-manifest-vs-privacy-label distinction stated as two separate deliverables, not one; the PHPicker permission-string caveat as something to verify against actual code before locking copy; the 2.3.1 "coming soon" risk with a concrete fallback wording, not just a flagged worry; and the app-icon, support-URL, and device-family-restriction items named as concrete unresolved launch blockers rather than assumed handled.

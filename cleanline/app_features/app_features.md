**Codex (Design Lead · Systems Thinker) — Round 1**

The MVP only works if the feature set mirrors the real job-to-be-done and refuses every tempting side quest. The user is not buying “vector editing on iPhone.” She is buying a fast, trustworthy path from one paper sketch to one usable export, with just enough control to recover from predictable auto-trace mistakes. So the must-have set needs to stay tightly chained to that loop: get the sketch in, judge whether it is supportable, show what the app did, let the user fix the common failure modes, and get a valid file out without losing work.

My best direction is to treat the product as five user-facing capabilities, not a pile of screens. First, sketch intake has to feel friction-light and reversible: start from camera or import, with permission-denied and unsupported-image paths that still keep the app usable. Second, the quality gate and correction review have to be first-class, not hidden plumbing, because this is where trust is won or lost; the user needs specific rejection reasons and a visible corrected preview before trace. Third, tracing and review need named progress states and a before/after comparison, otherwise a multi-second pipeline feels broken and opaque. Fourth, cleanup has to be real but narrow: select/delete stray paths, adjust simplify/smooth strength, merge broken segments, undo/redo, plus an accessible non-canvas fallback. Fifth, export and persistence have to behave like a tool, not a demo: SVG/PDF export, recoverable recent traces, autosave at each checkpoint, and no forced rerun after interruption or share-sheet failure.

The must-have list I would lock is:

- New trace intake from live camera or single-photo import.
  User story: as a seller with one sketch in front of me, I can start from whatever source is available without getting blocked by permissions.
  Acceptance criteria: camera and PHPicker entry points both exist; denied camera permission still leaves import fully usable; unsupported or unreadable images produce a specific error state with recovery.

- Capture quality gate with explicit rejection reasons.
  User story: as a user, I know before tracing whether the photo is within the app’s supported envelope and what to fix if it is not.
  Acceptance criteria: the app can stop with fixed named reasons like skew, shadow, blur, low contrast, sketch too faint, no sketch region, ruled-paper interference; each reason has consistent actionable copy; outside-envelope images do not silently proceed to junk trace output.

- Correction review with inspectable intermediate image.
  User story: as a user, I can see the normalized linework the app derived from my photo before committing to trace.
  Acceptance criteria: original and corrected preview are both viewable; raw capture survives through this stage; the user can proceed or back out without losing the source; orientation is correct for HEIC/imported images.

- Deterministic on-device vectorization with visible progress states.
  User story: as a user, I can wait through processing because the app tells me what it is doing and then gives me a traced result I can review.
  Acceptance criteria: processing states are named, not generic spinners; trace output is shown in-app; trace failures have an explicit error state and recovery path; all work remains offline.

- Before/after trace review.
  User story: as a user, I can compare my sketch to the vector result before exporting.
  Acceptance criteria: original or corrected image can be compared against traced output; the review state makes obvious what changed; if the trace quality is poor enough to need retry rather than cleanup, the app can send the user back instead of forcing export.

- Bounded cleanup tools mapped to real failure modes.
  User story: as a user, I can fix the common trace mistakes without needing desktop-class editing.
  Acceptance criteria: I can select and delete a stray path, adjust simplify/smooth strength, merge selected paths, and undo/redo those actions; slider-driven reprocessing is debounced and preview-based; VoiceOver users have a list-based fallback for selection and actions.

- Export to valid SVG and PDF.
  User story: as a user, I can leave the app with a file I can actually use downstream.
  Acceptance criteria: both SVG and PDF export paths exist; export failure does not destroy the trace; exported artifacts remain available from recents; the export is vector, not a flattened image.

- Local persistence and interruption recovery.
  User story: as a user, I do not lose work if I leave the app, get interrupted, or come back later.
  Acceptance criteria: autosave checkpoints exist at capture, post-normalization/review, post-trace, and post-edit; recent traces can be reopened and deleted; app relaunch resumes from the last valid checkpoint rather than starting over; share/export failure never forces retracing.

- Empty, loading, success, and error states across the core workflow.
  User story: as a user, every part of the flow tells me clearly what state I’m in and what to do next.
  Acceptance criteria: home/recent list, intake, processing, trace review, cleanup, export, and failure paths all have intentional states rather than blank canvases or silent failures.

The should-have list for this MVP, if the build still fits, is:

- Live in-camera framing guidance for page edges/skew before capture.
- Remembered user preferences for simplify/export defaults to improve repeat-use speed.

- Lightweight recents metadata like date, source thumbnail, and export status.
- A small built-in fixture/demo area for QA and product sanity checks during development.

The could-have list, only if it is nearly free after the must-haves are done, is:

- Duplicate trace to try alternate cleanup settings without losing the first version.
- A simple “retake tips” overlay that visually explains the rejection vocabulary.

The won’t-build list needs to stay aggressive:

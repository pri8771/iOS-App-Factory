**Claude (Frontend Engineer · Systems Thinker) — Round 1**

Here's my pass, coming at this from the client-architecture side rather than the visual side, since the visual language is already locked from the design_discussion phase and what's actually missing is the engineering-grade detail that turns "full-screen modal capture" and "single Room Home" into something a SwiftUI builder can implement without guessing.

First, a concrete architectural call that hasn't been made yet and needs to be: capture must be a `.fullScreenCover`, not a `.sheet` and not a `NavigationStack` push. That sounds like a style choice but it's actually a correctness requirement — `.sheet` ships with a default interactive swipe-to-dismiss gesture, and if a user swipes down mid-recording, that would silently exit the capture screen without ever routing through the "capture cancelled — [reason]" state machine we locked in the last two phases. A `.fullScreenCover` has no default dismiss gesture, so the only way out is through the explicit stop/cancel controls or a system-level interruption (call, backgrounding, route change) that we already intercept via `scenePhase` and `AVAudioSession` route-change notifications. I'd also set `.interactiveDismissDisabled(true)` on it defensively even though fullScreenCover doesn't need it, just so nobody accidentally reintroduces a sheet later without noticing the trust implication.

Second, a gap nobody's named yet: what happens if the app gets backgrounded in the narrow window *after* recording stops but *during* feature extraction? Right now the locked rules cover interruption during recording (hard-stop-and-discard) and the launch-time orphan sweep covers a full kill/crash, but extraction itself is a real, if brief, on-device compute step, and if the OS suspends the process mid-extraction before the raw file is deleted, that's exactly the scenario the orphan sweep exists for — so the extraction step needs to wrap itself in a short `beginBackgroundTask`/`UIApplication.shared.beginBackgroundTask` assertion (or the modern `BGTask` equivalent scoped to a few seconds) specifically to guarantee the delete-after-extract promise completes even if the user backgrounds the app the instant they tap stop. Without that, "raw audio never persists" has a real crack in it that only shows up under exact timing, which is the worst kind of bug to discover during review.

Third, the extraction and clustering work needs to run off the main actor as a hard rule, not an afterthought — spectral feature extraction (FFT) and the clustering pass are real CPU work, and if either runs synchronously on the main thread the "extracting/deleting" beat we designed as a trust signal will visibly stutter, which undercuts the exact moment we're relying on to prove the deletion happened. I'd model this as a `ModelActor`-backed capture pipeline: the UI-facing view model stays `@MainActor` and owns only the visible state enum (ready/recording/extracting/success/cancelled), while the actual FFT and clustering work happens on a background actor and reports back via async calls the main actor awaits. Same applies to the two-gate clustering recompute and the manual "recompute now" trigger — both should run off-main so the button doesn't freeze the UI on a room with a lot of sessions.

Fourth, navigation state and restoration. Given Room Home is the sole persistent destination with everything else pushed, I'd model the navigation as a typed path — an enum like `RoomHomeRoute` with cases for `.proposal`, `.trend`, `.settings` — driven through a single `NavigationStack(path:)` binding, specifically because the widget, the App Intent, and the notification tap-through all need to be able to construct a path programmatically (jump straight to the proposal screen from a notification tap, or straight into capture from the Shortcut) rather than simulating a sequence of taps. This also answers a restoration question that hasn't been addressed: if the app is killed and relaunched, it should never restore into a mid-capture or mid-proposal-negotiation state — capture always discards on background/kill per the locked rule, and a stale proposal candidate is exactly the kind of "the world may have changed" state that should recompute fresh rather than resume, given the whole point of the manual "check again" affordance is that we don't trust snapshots to stay valid. So: persist `RoomHomeRoute` navigation intent loosely (which screen), but never persist in-flight capture or negotiation state across a full process relaunch — only across scene backgrounding within the same live process.

Fifth, a Dynamic Type risk specific to the heatmap grid that nobody's flagged: a fixed-column grid (time-of-day columns by day-of-week rows) reads fine at default text sizes, but at accessibility XXL sizes the cell labels and legend text will either overflow or force illegibly tiny cells if we just let SwiftUI auto-shrink it. The fix is to treat the heatmap as a `Canvas`-drawn visualization with text rendered separately and allowed to reflow into a scrollable legend below it at accessibility sizes, rather than a `LazyVGrid` of text-in-cells that has to fight Dynamic Type directly. Practically: render the grid geometry in `Canvas` (cheap, GPU-friendly, doesn't recompute a view hierarchy every frame), and expose the same data as an accessibility-only list of value-bearing rows ("Tuesday afternoon: quiet, high confidence") via `.accessibilityElement(children: .combine)` grouping per row — this both solves the VoiceOver equivalence requirement from the design phase and sidesteps the text-overflow problem, using one mechanism for two requirements.

Sixth, for the widget: I want to be explicit that the shared snapshot is a small `Codable` struct (room name, score, confidence label, next proposed window, last-updated timestamp) written to an App Group container after every recompute and every negotiation outcome — not a partial SwiftData model copy.

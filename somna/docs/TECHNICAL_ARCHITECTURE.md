## Final Output

**Screen inventory:**
- **Today** (tab 1, always the launch screen) — single view, no sub-navigation, mutates through states rather than existing as separate screens.
- **Bedtime log** — modal sheet (not a pushed screen) launched from Today.
- **Wake + morning check-in** — modal sheet launched from Today, energy scale + collapsed "Add detail" tag disclosure.
- **Insights** (tab 2) — three visually distinct compositions for its three states.
- **History** (tab 3) — simple log list (could-have polish deferred, base list is must-have per app_features).
- **Weekly recap** — a compact card embedded in Today (once ≥1 night logged) that opens into its own full-bleed screen, which is what ShareLink exports from. Not a fourth tab.
- **Settings** — reached via a gear icon in Today's nav bar, not a tab. Holds the non-medical disclaimer, data management, Restore Purchases.
- **Somna+ paywall** — presented as a sheet/screen from wherever a locked feature is tapped (locked Insights card, Settings upsell).

**Primary user flow:**
Launch → Today (state depends on time of day/session status) → tap "Log Tonight's Bedtime" (sheet, defaults to now, one adjustable control, one confirm; re-tapping while open edits in place) → Today shows "asleep" state (dimmed breathing card, elapsed time, "Update Bedtime" + always-visible "I'm Awake") → tap "I'm Awake" (sheet: wake time defaulted to now → tap energy scale segment → tap Done = 3 taps; tags optional behind "Add detail", doesn't count against fast path; submits fine standalone if no open bedtime session, no apologetic copy) → Today collapses to quiet "done for today" state showing last night's duration + energy + streak, explicitly reflecting the check-in, not just the bedtime tap → user browses to Insights (still-learning progress / real insight cards with one free + blurred-locked per-tag cards / no-signal card) or Recap card (share via ShareLink) → cycle repeats next bedtime.

**State model per screen:**
- **Today**: (1) first-launch empty state — moon glyph, one sentence, one CTA button; (2) pre-bedtime/evening state once ≥1 cycle exists; (3) asleep state after bedtime logged — elapsed-time card + Update Bedtime + I'm Awake; (4) done-for-today state after both logged — collapsed summary + streak, no further CTA.
- **Bedtime sheet**: create vs. edit-in-place (same sheet, keyed off whether a session is already open).
- **Wake+check-in sheet**: default (wake time = now, tags collapsed); expanded-tags; submit-as-standalone (no paired session) — all handled silently, no distinct error state.
- **Insights**: still-learning (moon-phase fill progress + encouraging copy), real-insight (one or more comparison cards; free tier sees duration-vs-energy only, per-tag cards shown with blurred bars + unlock affordance), no-signal-for-tag (visually flat/muted card, distinct from still-learning).
- **Recap**: partial-week rendering (<7 nights) vs. full-week rendering; both must render correctly, never blank.
- **Paywall**: product-loading, loaded/purchasable, purchased, restore-in-progress, load-failure (fails open, never blocks core loop).

**Visual direction:** Time-of-day-driven gradient (not tied to system light/dark mode) across fixed named bands — deep indigo-to-violet through evening/overnight, shifting to soft dawn peach/rose from early morning through midday — mapped now as 3–4 concrete bands rather than left as an aspiration, since that's the one detail most likely to silently collapse into "always dark" under build pressure. Cards: near-opaque frosted surface over the gradient, generous corner radius, soft outer glow (no hard shadows), minimal rounded typography, slow cross-fades and gentle scale only (no springs/bounce). Single soft haptic impact on the energy-scale tap, nowhere else.

**Accessibility notes:** Energy-scale control exposes itself to VoiceOver as an adjustable element with spoken value names ("Energy: Rested," not "4 of 5"); text always sits on a guaranteed-contrast card fill, never directly on raw gradient; Dynamic Type up to accessibility sizes reflows cards rather than truncating (no fixed-height insight cards); tag chips and time controls keep real 44pt targets despite compact visual size; reduced-motion collapses the moon-phase fill animation and gradient time-of-day transitions to instant cross-fades rather than removing the state information they carry.

## Final Output

**Screen-by-screen design specification:**

- **Today** — single `TodayView`, one hero card that cross-fades its internal content across four states (empty / pre-bedtime / asleep / done-for-today) rather than four separate views, plus a compact recap card once ≥1 night exists. Driven by one root `TimelineView(.everyMinute)` whose `context.date` is passed down to both the elapsed-time label and the gradient — no independent clocks per subview. Empty: moon glyph, one sentence, one full-width primary CTA. Pre-bedtime: same CTA + streak/last-night strip. Asleep: ticking elapsed time, de-emphasized "Update Bedtime," always-tappable "I'm Awake" (never disabled by elapsed duration, to support 14+ hour shift-worker sessions). Done-for-today: CTA disappears, shows last night's duration + energy-in-words + streak, with a one-time-per-day checkmark micro-motion gated by a `hasAnimatedDoneStateToday` flag so it doesn't replay on every foreground.
- **Bedtime sheet** — `.sheet` with `.presentationDetents([.medium])`, one `DatePicker` defaulting to now + one confirm button; same view edits in place (title/label swap to "Update Bedtime") when a session's already open, driven by a boolean not a second sheet.
- **Wake + check-in sheet** — same medium-detent sheet: wake-time picker (defaults to now) → 5 equal-width tappable energy segments (not a slider) which immediately enable Done and fire the app's one haptic → collapsed "Add detail" disclosure expanding into a `LazyVGrid` of tag chips → Done. Fully optional tag state; submits identically whether collapsed, expanded, or standalone with no open session — no apologetic copy anywhere.
- **Insights** — `ScrollView` of cards, deliberately not a `List` (unwanted separators/swipe actions fight the custom card layouts). Still-learning: hero card with a custom animatable-`Shape` moon-crescent-fill (not a system `ProgressView`) + "N more nights" copy. Real-insight: title sentence + hand-rolled two-`Capsule` comparison bars; the same `InsightCard` component takes an `isLocked` modifier that blurs the bars and overlays an unlock affordance, so the locked card is never a separately-built stub. No-signal: a structurally distinct `NoSignalCard` (flatter fill, no glow) — explicitly not a reskin of the still-learning card.
- **History** — plain `List`, date/duration/energy-icon rows, delete-only swipe action (no edit here — editing stays confined to Today's asleep-state "Update Bedtime" per the locked edit-scope rule), with its own empty state for first launch.
- **Recap** — compact fixed-height card embedded in Today, tapping it opens a full-bleed `RecapDetailView` that is the literal `ShareLink` source via `ImageRenderer`, so shared output can never visually drift from the live screen.
- **Settings** — grouped `Form`: disclaimer, data management (delete-all behind a confirmation alert), Restore Purchases, paywall entry point.
- **Paywall** — sheet over `SubscriptionStoreView`/custom screen covering loading/purchasable/purchased/restoring/load-failed; load-failed renders inline retry + dismiss, never a blocking spinner, and dismissing never leaves any part of Today gated.

**Design tokens:**
- Four time-of-day gradient bands — `duskDeep` (21:00–00:00, indigo→violet), `overnight` (00:00–05:00, deep violet-black), `dawn` (05:00–10:00, violet→peach/rose), `day` (10:00–21:00, soft warm neutral) — each a `LinearGradient` with 2–3 stops, linearly cross-faded over a ~30-minute window at boundaries via a pure, unit-testable `gradient(for: Date) -> [Color]` function.
- Corner radius: 24pt (hero/insight cards), 16pt (chips/small components).
- Card fill: fixed near-opaque `Color.cardSurface` (~85–90% opacity) — explicitly not `.ultraThinMaterial`, since translucent system materials would let the gradient bleed through inconsistently and threaten guaranteed contrast.
- Glow: single shared `.glowCard()` modifier (soft shadow, radius 20, low opacity) — never hand-rolled per screen.
- Type: rounded system font, five sizes (title/headline/body/caption/numeral-large), full Dynamic Type support, no hard caps.
- Motion: one `.easeInOut(duration: 0.4)` cross-fade curve everywhere, one gentle spring `(response: 0.5, dampingFraction: 0.85)` reserved only for the moon-crescent-fill and the done-state checkmark.
- Haptic: exactly one, `.impact(.soft)`, fired only on energy-scale selection.

**Component inventory:** `GradientBackground`, `FrostedCard`, `MoonCrescentProgress`, `EnergyScaleControl` (a11y-critical — ships with `.accessibilityAdjustableAction`/`.accessibilityValue` from day one, not retrofitted), `TagChip`, `PrimaryButton`/`SecondaryButton` (44pt minimum baked into the style), `InsightCard` (real/locked/no-signal via a config enum, not three separate views), `StreakBadge`, `RecapCardContent` (shared between embedded and full-bleed so ShareLink output can't drift).

**State and interaction notes:** Correlation recompute runs async off the main actor on each foreground, publishing into view state while the view continues showing previous values until new ones land — satisfies "always fresh, never cached" without risking a visible hitch as history grows under unlimited-history paying users. SwiftData "open session" queries use a real predicate (`endDate == nil`), not a hand-reconstructed check in the view model, so edit-in-place and auto-pairing logic can't disagree about what's open under concurrent taps.

**Accessibility requirements (carried forward + sharpened):** `EnergyScaleControl` exposes VoiceOver value/adjustable-trait support as a first-class build requirement, not an add-on; all text sits on the opaque `cardSurface` fill, never the raw gradient; Dynamic Type reflows cards (no fixed-height insight cards); 44pt targets enforced structurally via the button/chip components rather than left to call sites; reduced-motion collapses the crescent-fill and gradient transitions to instant cross-fades without dropping the state information they carry.

**Claude Design prompt / external-design decision:** No pause for external design on the main app — the token values, gradient bands, and component set are already concrete enough that building directly in SwiftUI avoids a translation/drift risk between a separate design tool's output and real `LinearGradient`/shadow values. The one piece worth an external pass is the app icon specifically: *"Design an app icon for Somna, a calm sleep-tracking app: a minimalist moon-phase mark (crescent transitioning through phases), set against a deep indigo-to-violet gradient consistent with dusk-to-dawn, no text, no clinical iconography, soft-glow edge treatment, must read clearly at 60×60pt."*

## Final Output

**SwiftUI architecture:** `@Observable` view models (iOS 17+, not `ObservableObject`/Combine). Two repository protocols — `SleepSessionRepository`, `CheckInRepository` — injected via `@Environment`, so SwiftData can be swapped for an in-memory fake in tests and later for CloudKit without touching views. The correlation engine is a real Swift `actor` (not just "an async function") so the off-main-actor guarantee design_handoff already committed to is actually data-race-safe, not accidentally safe today. Critically, the engine's input type is plain structs (arrays of session/check-in value types), never `@Model` objects directly — that's what keeps the threshold-gating test (N-1 nights → zero insights, N nights → at least one) unit-testable without a `ModelContainer`, and prevents someone from "simplifying" it later by querying SwiftData inside the engine.

**Apple framework choices:** SwiftData (two `@Model` types, `SleepSession` with `endDate: Date?` for a real `endDate == nil` predicate, and `CheckIn`); StoreKit 2 (`.storekit` config for local testing, `Transaction.updates` listened from launch, `Transaction.currentEntitlements` as source of truth — never a cached bool); ActivityKit for the Live Activity; WidgetKit for the widget; HealthKit read-only, opt-in only from Settings, never requested at first launch; explicitly no UserNotifications framework anywhere, consistent with the locked no-push decision; no Charts — hand-rolled `Capsule` bars instead, matching the frosted-card aesthetic and cutting a framework surface to test.

**Persistence plan:** A versioned SwiftData schema (`VersionedSchema`/`SchemaMigrationPlan`) from the first commit, even with only one version today — non-negotiable given "unlimited history" is the paid value prop, so real users will accumulate months of data before any future schema change, and retrofitting migration onto an unversioned store later is far worse than declaring one now.

**Permissions/privacy plan:** HealthKit denied-vs-not-determined is indistinguishable by design — never inferred or auto-re-prompted; UI only reflects whether the user has tapped "Connect Health" once, with real (not boilerplate) `NSHealthShareUsageDescription` copy since App Review reads it. ActivityKit requests wrapped in try/catch that silently no-ops if Live Activities are system-disabled — no crash, no nag. `WidgetCenter.shared.reloadTimelines` fired only from actual write paths (bedtime logged, check-in submitted), never from a per-minute clock tick, since the system throttles/drops excess reload calls silently.

**Dependency/license decision:** Zero third-party packages. Every needed capability has a first-party Apple framework, so there's no permissive-license question to resolve — the answer is no SPM dependencies at all.

**Testing implications:** The threshold-gating test is a pure unit test against synthetic struct arrays, no `ModelContainer`, no UI — non-negotiable per app_features. Session-pairing/edge-case logic (16-hour window, orphan-closing, duplicate-tap update-in-place, DST/negative-duration rejection) gets its own Swift Testing (`@Test`) suite, parameterized over a timestamp-pair matrix. The `gradient(for: Date)` function gets a direct unit test asserting band boundaries. At least one XCUITest exercises the Today state machine end-to-end (empty → asleep → done-for-today), plus one accessibility test asserting `EnergyScaleControl` exposes a real accessibility value and adjustable trait, since that's the kind of thing that silently regresses. No ML/AR justification needed beyond what's settled — rule-based statistics is the right, honest, battery-safe choice; that question is closed.

**One additional concrete decision locked this round:** entitlement status resolves once per app-foreground into an in-memory `@Observable` flag, not re-queried from `Transaction.currentEntitlements` on every view body evaluation — re-running that async sequence inside a computed property risks a real UI stutter on Insights, the one screen that most needs to feel instantaneous.

## Final Output

**Architecture overview:** Single Xcode project (`Somna.xcodeproj`), three targets — `Somna` (app), `SomnaWidget` (WidgetKit extension, also hosts the Live Activity), `SomnaTests` (Swift Testing unit tests). No SPM-local-package split; one app target with clean internal folders gives full testability without build-graph overhead. `@Observable` view models throughout (iOS 17+, no Combine). Two repository protocols (`SleepSessionRepository`, `CheckInRepository`) injected at the View layer via `@Environment` and passed into view-model initializers as concrete protocol instances — view models never pull `@Environment` themselves, so every view model is constructible in tests with in-memory fakes. The correlation engine is a real `actor` operating only on plain `NightRecord` structs, never SwiftData `@Model` objects, which is what keeps the non-negotiable threshold-gating test (N‑1 nights → zero scored insights, N nights → at least one) a pure unit test with no `ModelContainer`. Zero third-party dependencies — every capability (persistence, subscriptions, background activity, health read access, sharing, comparison bars) has a first-party Apple framework.

**File/module layout:**
- `App/` — `@main` App struct, scene setup, root `TabView`.
- `Models/` — the two `@Model` types plus plain-struct value types the engine consumes.
- `Repositories/` — `SleepSessionRepository`/`CheckInRepository` protocols; `Repositories/SwiftData/` for the real implementations; in-memory fakes live in the test target.
- `Engine/` — `CorrelationEngine` actor, `InsightRule` implementations, `InsightResult`/`InsightCardData`, tag/energy enums, `NightRecordAssembler`, and the pure `gradient(for:)` function (treated as derived-state logic even though it feeds visuals).
- `ViewModels/` — one `@Observable` class per screen: `TodayViewModel`, `InsightsViewModel`, `HistoryViewModel`, `RecapViewModel`, `PaywallViewModel`.
- `Views/` — grouped by screen per design_handoff's component list (`Today/`, `Insights/`, `History/`, `Recap/`, `Settings/`, `Paywall/`, `Components/` for `FrostedCard`/`InsightCard`/`EnergyScaleControl`/etc.).
- `Subscriptions/` — StoreKit manager + entitlement store.
- `HealthKit/` — optional import service, isolated so it's trivially excludable from anything that doesn't want to link HealthKit.
- `Live Activity/` — `ActivityAttributes` struct marked as belonging to both the app and widget extension targets (lightweight shared-file pattern, not a separate framework).

**Data models:** `SleepSession` (`@Model`: `id`, `startDate`, `endDate: Date?` — open/closed is a real `#Predicate(endDate == nil)` at the query site, never a hand-reconstructed check). `CheckIn` (`@Model`: `id`, `timestamp`, `energyLevel: Int` 1–5 wrapped by `EnergyLevel` enum at all boundaries, `tagsRaw: [String]` storing `SleepTag` raw values — enums-as-strings deliberately chosen over a join table since the tag set is fixed and small, `linkedSessionID: UUID?` as a soft foreign key, not a SwiftData relationship, so the 16-hour pairing/orphaning rule stays explicit application logic rather than implicit cascade behavior). `NightRecord` is the pure value-type boundary the engine actually consumes, built by the pure function `NightRecordAssembler.assemble(...)`, which pairs sessions to check-ins within the plausibility window, orphans the rest, and discards implausible/negative durations before a record is ever created — the engine itself never touches SwiftData.

**Persistence strategy:** `ModelContainer` configured with `SomnaSchemaV1: VersionedSchema` and an (initially empty) `SomnaMigrationPlan` from the first commit — non-negotiable given unlimited history is the paid feature. Repositories wrap `ModelContext` behind the two protocols; `WidgetCenter.shared.reloadTimelines` is called exclusively from the repository write methods (`startOrUpdateSession`, `endSession`, `submit`), one call site each, so nothing can accidentally wire it to a clock tick.

**Testing strategy:** `SomnaTests` needs zero HealthKit/StoreKit linkage to run fast. `CorrelationEngineTests` run purely on `[NightRecord]` literals and assert the threshold-gating contract word-for-word. `SessionPairingTests` use a parameterized `@Test` matrix (16h-boundary pair, DST-fallback pair, duplicate-tap pair). `GradientTests` assert band boundaries from `gradient(for:)`. `TodayStateMachineTests` drive `TodayViewModel.state(at:)` through empty→asleep→done using in-memory repositories. One XCUITest smoke-tests the real UI end-to-end since no unit test touches actual rendering, plus one accessibility test asserting `EnergyScaleControl` exposes a real accessibility value and adjustable trait.

**Risk mitigations:**
- In-memory fakes treated as a first deliverable, not a testing nicety — prevents the UI lane from silently blocking on the data lane.
- Threshold-gating test and seed-data path are gating criteria for their milestone's "done," not follow-up work — prevents the app's one differentiating mechanism from shipping unverified.
- DST/session-pairing tests must pin explicit timezones — prevents tests that pass without exercising anything.
- Versioned schema gets a named verification step (confirm the `ModelContainer` is actually configured with the migration plan), not just a code-review glance at the type declarations.
- StoreKit fail-open and HealthKit denial are named explicitly as M6 checks specifically because they require deliberately breaking something rather than testing the happy path, which is exactly the kind of check that gets silently skipped otherwise.

Primary user: a boulderer or route climber who climbs 2-5 times a week, already knows their grades, and is worried about overtraining or plateauing — not a total beginner who doesn't know what a grade is, and not a competitive athlete who needs a coach dashboard. The scenario that matters most: they get back from the gym, log grades/attempts/sends in under a minute, and the next morning or before their next session, they check the app to decide whether to climb hard, climb easy, or skip. If logging takes more than a minute or requires filling out a wall-by-wall log, they stop using it within a week — that's the realistic failure mode, not some exotic edge case.

Core loop, stated as a loop because that's what has to be testably real: log session (discipline, grades attempted, sends, subjective intensity/effort) → training load recalculates (rolling acute vs chronic load, however we define the window) → rest/train recommendation updates and is visibly justified ("your 7-day load is 40% above your 28-day average, recommend active rest") → user either follows it or taps through an acknowledgment to override → next logged session feeds back into the same load calc. The thing I most want preserved into the next phase is that last link — override must write back into the model (e.g., an ignored-recommendation flag), not be a dead-end click, or the loop isn't actually a loop and QA can't test that the system responds to real behavior over time.

Hard scope boundaries I'd insist on for v1, because each of these is a believable scope-creep vector given the parent prompt's ambition: no HealthKit-driven load blending (heart rate, sleep, strain) — that's a whole additional trust and permissions surface and a second data source to reconcile; log manually, HealthKit read-only as an optional future enhancement, never a dependency for the recommendation to function. No route/beta-specific features (photos, video, beta notes) — that's a different app (a climbing journal), and it dilutes the training-load focus. No social/leaderboard/sharing — that turns training-load data into a comparison tool, which actively works against sensible rest-day behavior. No coach/multi-user accounts. No cloud sync in v1, already settled. And critically: no auto-generated long-term periodization plan that tells you what to do 8 weeks from now — the app reacts to logged history, it doesn't predict or prescribe a future macrocycle in the MVP. If a later phase wants to add that, it's new scope, not an MVP detail.

Measurable success criteria, framed the way I'd actually verify them rather than as vibes: a user can log a session in under 60 seconds; if I simulate a week of unusually high volume/intensity, the recommendation demonstrably shifts toward rest within that same test session (not just once at seeded launch — recompute correctly given new data); the recommendation and the load numbers behind it are visible together, not a black-box badge, because "trust me" is not testable by QA and won't be trusted by users either; the app never loses a logged session across force-quit, background, low storage, or denied permissions; and overriding a rest recommendation doesn't crash the flow or silently drop the fact that the user overrode it.

**Claude (QA & Risk · Systems Thinker) — Round 1**

Picking up from the locked brief, my job this round is to stress-test the parts that will actually break in someone's hands, not the parts that look good in a deck. I want to start with the ambiguity I think is most load-bearing and least resolved so far: what exactly does "override" mean mechanically, and when does the gate actually appear? "Soft-gating with override, require an acknowledgment tap to bypass" was the phrase we locked, but bypass *what*, triggered *when*? If we don't pin this down now, architecture will guess, and two different guesses (a blocking modal every app open vs. a passive banner) produce completely different feels — one nags, one doesn't. My resolution: the gate is a dismissible banner/card shown on app open when today's computed recommendation is "rest," not a blocking modal. It persists until either the user taps an explicit "I understand, I'm training anyway" acknowledgment, or the user logs a session that day, which itself counts as the override and gets tagged on that session record. No gate re-appears mid-session, no repeated interruptions in one day. This needs to be nailed down as a resolved requirement, not left to whoever builds the interstitial.

Second thing I want to force a decision on: the acute:chronic load ratio (7-day vs 28-day, which the initial discussion already implied with its "40% above 28-day average" example) has a divide-by-zero problem baked into its structure — a brand-new user's chronic average starts at zero, and the very first logged session is mathematically infinite relative to a zero baseline. That's exactly the "calibrating" state the brief already promised, so let's make the threshold concrete instead of hand-wavy: fewer than 14 days of logging history (or some fixed floor we pick now) means the UI shows "still calibrating" with the raw numbers visible but no ratio-based recommendation, and only past that floor do we show the real train/rest/easy signal. Without a hard number here, someone building this will invent their own cutoff, and QA won't know what "correct" looks like when testing it.

Related edge case that I think is actually a feature if we design it on purpose: long gaps. A climber who takes two weeks off for travel or injury will see their 28-day chronic average decay toward the sessions they did log, and if they come back hard, the acute number will spike relative to a now-lower chronic baseline — which is the right behavior (return-to-training spikes are a real overuse-injury risk factor), but if nobody decides this on purpose it'll look like a bug when someone notices the recommendation got stricter after a break. I want this locked as intentional: detraining reduces the chronic baseline, so a return session appropriately reads as higher relative load, without us ever using the word "injury risk" in the copy.

On the grade-normalization decision carried over from initial_discussion — I don't think mapping V-scale and YDS/French to one internal numeric index is sufficient by itself for a fair load number, and this needs to be said explicitly now rather than discovered during build. A boulder attempt is a few seconds of maximal finger/power output; a route attempt or redpoint is minutes of sustained sub-maximal load. If the formula is just normalized_grade × attempts, mixed-discipline climbers will get a load number that overweights or underweights one discipline relative to the other, and a serious climber will notice within a week that the number "feels wrong," which kills trust in the exact feature we're using to justify a subscription. My assumption to lock: load = normalized_grade × attempt_count × a discipline-specific multiplier (bouldering vs. route weighted differently), documented as a heuristic, tunable, not a scientific measurement — and the in-app copy should say something like "training-load estimate," never imply lab-grade precision. I also want to flag, as a UI guardrail rather than a math one: the normalized index should never be shown to the user as a literal grade-equivalence ("this route = V5"), because serious climbers will immediately reject a cross-discipline grade equivalence claim as wrong (these scales genuinely aren't linearly comparable), even if it's fine as an internal computation detail. Raw logged grade stays visible everywhere; the normalized number stays invisible plumbing.

A logging-model ambiguity that's still open: is a "session" one calendar visit that can mix disciplines (bouldered for an hour, then did some route mileage), or does a mixed day require two separate session entries? I'd resolve this as: one session entry can contain sub-entries per discipline within the same visit, still logged as one interaction so we don't blow past the under-60-seconds promise, but each sub-entry is discipline-tagged so per-discipline load lines (a Pro feature) can still be computed correctly. If we force two full separate session flows for one gym visit, that's friction that directly threatens the 60-second success criterion.

On data edits and deletes — the success criterion says "no logged session is ever lost," but a user absolutely needs to be able to fix a fat-fingered entry or delete a duplicate, and that's not data loss, that's a feature. I want us to separate those explicitly: accidental data loss (a bug, a crash, a low-storage write failure) is the thing the criterion protects against; intentional user deletion/edit is allowed and must recompute all downstream rolling windows and any prior recommendation snapshots that depended on the deleted/edited session, not just append a new state. If someone backdates a forgotten session from three days ago, the load windows for the days in between need to recompute correctly too — that's a real edge case a naive "load only updates going forward" implementation would silently get wrong, and it's the kind of bug that wouldn't show up in a demo, only in real usage a week in.

On privacy and persistence specifics: this data has no HealthKit and no accounts, which shrinks the permission surface a lot, but I don't think push notifications should be assumed into scope for the MVP at all — nothing in the locked brief requires them, and given the strong "don't be a nagging overtraining app" instinct from the prompt contract, I'd rather explicitly exclude notification-based rest reminders from v1 and name it as an assumption, not silently let a build phase invent a reminder scheme that undermines the "enforcement is honest and non-punitive" promise. Separately, I want to lock that the on-device store should not be excluded from the standard encrypted device backup — since we're not building cloud sync in v1, a user's entire training history currently has exactly one disaster-recovery path (Finder/iCloud device backup), and if someone marks the SwiftData store as excluded-from-backup for "cleanliness" reasons, that quietly reintroduces the "lose your phone, lose a year of training history" risk the prompt contract already flagged. For the CSV/PDF coach export (Pro feature), the concern isn't cloud storage, it's that export triggers a share sheet the user actively drives — no silent background upload, and the user should be able to see what's in the file before sending it anywhere.

For accessibility, the one I'd call out specifically rather than leave generic: the whole product's trust mechanism is "reasoning always visible next to the number," and grade pickers plus load trend charts are both nonstandard controls. VoiceOver needs an actual spoken equivalent of the reasoning line ("your 7-day load is 40% above your 28-day average") on the chart itself, not just alt-text on a picture of a line — if the sighted UI shows the reasoning but VoiceOver only announces a bare number, we've shipped the exact black-box experience we're positioning against for a chunk of users.

Last thing, from the QA seat specifically: the production-readiness bar we already committed to — "force a high-load week, confirm the rest recommendation changes" — is not realistically testable by hand-tapping 20+ sessions across 28 simulated days every time we want to verify a change. I want a resolved requirement that the build includes some deterministic way to seed multi-week fake history for verification purposes (a debug-only seeding path is fine, doesn't need to be user-facing), because without it, "we manually verified this" will quietly stop happening after the first pass, and that's exactly the corner most likely to silently rot.

Risks I'd put on the register as accepted limitations rather than things to solve now: gym-to-gym and region-to-region grade inconsistency (a "soft" V5 at one gym vs a "hard" V5 at another) is a real distortion on the load math, and we should say plainly we take logged grades at face value in v1 with no gym-specific calibration — that's a stated limitation, not a bug someone should feel obligated to fix under this MVP. Similarly, ambiguous/slash grades (V4/5, 5.10a/b) need a concrete input rule (pick the single grade at logging time, or default to the lower of a slash-range) rather than leaving it to whoever builds the grade picker to improvise.

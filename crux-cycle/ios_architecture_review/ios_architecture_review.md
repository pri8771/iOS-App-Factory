Coming at this purely from "what breaks in real usage that a demo won't surface," and I want to start with the good news because it changes how much scrutiny the rest of this needs: this app has almost no permission surface at all. No HealthKit (excluded), no notifications (excluded), no accounts, no camera, no location. The only runtime failure mode in the whole permissions category is a corrupt/unreadable local store, which is already flagged. That's rare for a health-adjacent app and it's worth stating plainly rather than writing a permissions section that pretends there's more to manage than there is — a shorter, honest permissions story is a real strength here, not a gap to fill with invented scope.

The architecture risk I actually want to fight about is the debug entitlement toggle. Design_handoff describes it as living in a visually-distinct "Debug" section, and my instinct as the person who has to worry about App Store review is that a hidden or semi-hidden toggle that flips paid features on is exactly the kind of thing Apple's review guidelines have burned people for before — undocumented functionality, hidden menus, anything that looks like it's bypassing IAP. My fix would be to compile the *history-seeding* tool behind `#if DEBUG` so it's not physically present in the shipped binary at all — pure QA tooling, a real user should never be able to find it because it doesn't exist in Release. But I don't think the *entitlement flag itself* can be `#if DEBUG`-gated, and this is the part I think the group hasn't actually resolved yet: if real StoreKit purchasing is explicitly not required for this MVP (locked upstream), and the entitlement toggle is `#if DEBUG` only, then the shipped App Store build has literally zero path for any real user to ever unlock Pro. That fails our own acceptance bar — "at least one real Pro feature functioning behind it, not stubbed" — because in the actual thing that ships, it'd be permanently stubbed. I'd separate these two concerns explicitly: the debug seeding seed-data button is DEBUG-only, but the entitlement flag is a real (if simple) local-only toggle that ships in Release for now, understood as a placeholder for StoreKit rather than a test harness, so there's an honest, if manual, upgrade path in v1. Whoever owns tech_specs needs to pick one of these two shapes on purpose, not inherit an ambiguous "Debug section" description and build whichever is easier.

Second risk, and this one's a straightforward architecture bug waiting to happen: the log-session sheet is specified to optimistically dismiss on save, with the recompute happening async afterward. SwiftData's `save()` can throw — most realistically under the low-storage condition we already committed to testing. If the sheet has already dismissed and the save then fails, the user believes their session is logged, walks away, and it's gone. That's precisely the "no logged session is ever lost" criterion failing, and it'd fail silently, in the exact low-storage scenario we already flagged as a required test case. I don't think "optimistic dismiss" and "no data loss under low storage" can both be fully true unless we're specific: either the dismiss waits for save confirmation (SwiftData local writes are fast enough at this data volume that this shouldn't cost a visible delay in the common case), or the sheet dismisses immediately but the app holds the pending entry in memory and surfaces a hard, undismissable retry state if the write actually fails — not a toast that disappears. I'd pick the first one; it's simpler, it's honest, and "optimistic" UI is solving a latency problem we don't actually have at this scale.

Third: the day-rollover recompute trigger is right in spirit but I want the timezone edge case named explicitly before someone builds it against UTC-stored timestamps and gets it subtly wrong. Sessions store UTC timestamps (locked, for future cloud-sync reasons), but "day," "7-day window," and "still calibrating on day 14" are all user-facing concepts that have to be computed in the *device's current calendar/timezone*, not UTC. A session logged at 11:40pm local time needs to land on the calendar day the user experienced it as, and a user who flies across timezones mid-week shouldn't watch their calibration counter jump or their acute window silently shift by a day for reasons that have nothing to do with their climbing. This needs to be a named, deliberate rule in tech_specs — "day boundaries for all rolling-window math use `Calendar.current` at the moment of computation, timestamps stored in UTC" — because it's the kind of thing that's invisible in a same-timezone demo and only shows up as a real bug report from an actual user.

Fourth, on recompute cost, since "full recompute of every downstream window" is a phrase that sounds simple until someone has a year of logged history: a backdated edit only needs to trigger recompute for windows that could plausibly include that date — practically, from the edited date forward to today, bounded by the 28-day chronic window, not "recompute the user's entire lifetime of data" on every edit. At MVP data volumes this bound barely matters for performance, but it matters for correctness of the spec itself — I want that bound written down now so nobody either over-builds a full-history recompute job or, worse, under-builds one that stops one day short of the actual window boundary.

Fifth, testability: the training-load engine needs to operate on plain Swift value types (a lightweight `SessionRecord` struct), not directly on `@Model` SwiftData objects, and it needs to be a pure function of an array of those structs plus a reference date. If the engine reads live from a `ModelContext` internally, every unit test that wants to check "does day 14 flip the recommendation, does a 10-day gap decay correctly, does backdating 3 days shift the right window and not the wrong one" needs to spin up an in-memory `ModelContainer` and insert real model objects — that's slow, that couples every math test to persistence machinery that has nothing to do with the math being tested, and it's exactly the kind of test suite that quietly stops getting run because it's annoying to write. Keep persistence and computation as two separate layers with a thin mapping between them, and the whole test suite for the one feature this app's subscription is actually justified by becomes fast, deterministic table-driven tests with hand-built input arrays — which is also, not coincidentally, the same debug-seeding capability the product already needs, just usable from XCTest instead of only from a UI button.

One more concurrency note I'd flag rather than let slide: recompute gets triggered from several independent places — save, edit, delete, backdate, and scenePhase-active-with-day-rollover. If two of those fire close together (log a session right as the day rolls over at midnight, say), there needs to be one serialized entry point for recompute, not each trigger independently mutating the cached `TodayCardState`. An actor or a simple "recompute is idempotent and the last write wins, guarded against overlapping execution" rule needs to be explicit, or this becomes a flaky, hard-to-reproduce bug that only shows up occasionally on real devices.

For dependency/license: I'd state outright that this app needs zero third-party packages — SwiftData, Swift Charts, Observation, and (later) StoreKit 2 cover everything specified, and the permissive-dependency acceptance check is trivially satisfied by not adding anything. That's not a cop-out, that's correct given the actual feature list; if a later phase reaches for a third-party charting or persistence library "for convenience," that should be treated as a deviation to justify, not a default.

What I need preserved out of this phase: the entitlement-flag-vs-debug-tool split resolved one specific way (seeding is DEBUG-only, entitlement flag ships in Release as the real MVP Pro mechanism); the load engine specified as pure-function-over-value-types for testability, decoupled from SwiftData model objects; day/window boundary math explicitly pinned to device-local calendar time despite UTC storage; log-session save changed from optimistic-dismiss to confirmed-save-then-dismiss (or an equivalent that doesn't risk losing data on a failed write); and recompute-on-edit bounded to the affected window range, not full-history, stated as a concrete rule rather than left to interpretation.

## Final Output

**SwiftUI architecture recommendation:**
- Four-tab `TabView` (Today, History, Trends, Settings) per design_handoff, Log Session as a sheet off a persistent "+".
- Today's state is a single enum (`.initialLoad` / `.calibrating` / `.normal` / `.gatedRest` / `.corruptStore`) computed and cached on an `@Observable` `TrainingLoadEngine`/view-model layer — never derived live inside `body`.
- Recompute is triggered explicitly from a small, closed set of events: session save (confirmed, not optimistic), edit, delete, backdate, and `scenePhase → .active` when the calendar day has rolled over. All triggers funnel through one serialized recompute entry point (an actor, or an explicit "ignore overlapping calls, last write wins" guard) so near-simultaneous triggers (e.g., logging right at midnight) can't race and leave `TodayCardState` in an inconsistent state.
- **Persistence/computation are separate layers:** the training-load engine operates on plain Swift value types (a `SessionRecord` struct, mapped from `@Model` objects), as a pure function of `(sessions: [SessionRecord], referenceDate: Date) -> LoadSnapshot`. This is what makes the core feature unit-testable without spinning up SwiftData for every test.
- `ModelContainer` is constructed manually at launch inside `do/catch` (not via `.modelContainer(for:)`, which crashes by default on an unreadable store); failure routes to `CorruptStoreRecoveryView` as the actual app root, per design_handoff.

**Apple framework choices:** SwiftUI, SwiftData (local persistence), Swift Charts (Trends), Observation (`@Observable`), StoreKit 2 (future real purchase flow — MVP ships a simpler entitlement mechanism, see below). No HealthKit, no UserNotifications, no location, no camera — consistent with the locked scope exclusions.

**Persistence and local data plan:**
- SwiftData store, not excluded from device/iCloud backup (already locked — this is the only disaster-recovery path with no cloud sync in v1).
- Backdated/edited/deleted sessions trigger recompute bounded to the affected window range — from the edited date forward through today, capped at the 28-day chronic window — not a full-history recompute. Stated explicitly so nobody either over- or under-builds the recompute scope.
- Day/window boundary math (7-day acute, 28-day chronic, 14-day calibration floor) is computed using `Calendar.current` at the moment of computation; session timestamps remain stored in UTC. This prevents timezone travel or DST from silently shifting a user's calibration count or load windows.
- Log Session save uses confirm-then-dismiss, not optimistic-dismiss-then-async-save — a SwiftData write failure (most realistically under low storage, a condition already required to be tested) must not be able to lose a session the user believes was saved.

**Permissions/privacy plan:**
- Minimal by design: no HealthKit, no notifications, no accounts, no location, no camera/photos. The only failure mode in this category is corrupt/unreadable local storage, already covered by the recovery-view architecture above.
- CSV export (Pro) goes through the standard share sheet only, user-driven, no background upload — already locked.

**Dependency/license decision:** zero third-party packages. Every required feature (persistence, charting, purchasing, state observation) is covered by first-party frameworks, so the permissive-dependency policy is satisfied trivially. Any future push toward a third-party library should be treated as a deviation to justify explicitly, not a default convenience.

**Free/Pro entitlement architecture (resolved this round):** two separate mechanisms, not one ambiguous "Debug section" — (1) the multi-week synthetic history-seeding tool is compiled behind `#if DEBUG` and is physically absent from the Release/App-Store binary; (2) the Pro entitlement flag itself is a real, simple local toggle that ships in Release as the MVP's actual (if manual/placeholder-for-StoreKit) unlock mechanism — because if it were DEBUG-only too, the shipped app would have literally no path for any real user to ever reach Pro, which fails the "at least one real Pro feature functioning behind it" bar locked in app_features.

**Testing implications:**
- The training-load engine's pure-function-over-value-types design makes the core feature (acute/chronic computation, calibration cutoff, gap-decay, recommendation logic) testable via fast, deterministic, table-driven XCTest cases with hand-built input arrays — no `ModelContainer` required for math correctness tests.
- That same hand-built-array capability doubles as the basis for both the DEBUG-only UI seeding tool and the XCTest fixtures — one underlying capability, two consumers.
- Recompute-on-edit needs a test that backdates a session and confirms exactly the bounded window (edit-date → today, capped at 28 days) recomputes — not more, not less.
- Timezone/day-boundary logic needs an explicit test simulating a timezone change or late-night log entry to confirm calibration/window counts don't shift incorrectly.
- Low-storage/save-failure path needs a test (or simulated condition) confirming the sheet does not dismiss and no data is silently lost when a SwiftData write fails.

**Open item carried forward, not blocking consensus:** the entitlement-mechanism split (DEBUG-only seeding vs. Release-shipping entitlement flag) is a concrete resolution of something design_handoff left ambiguous — if tech_specs or build phases want a different shape (e.g., wiring real StoreKit 2 sandbox purchasing instead), that's a deviation from this decision and should be raised explicitly, not quietly substituted.

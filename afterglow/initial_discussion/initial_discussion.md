**Claude (Design Lead · Systems Thinker) — Round 1**

Coming into this fresh with a Design Lead hat: the product promise I'd commit to is "Afterglow turns the photos already sitting in your camera roll from one trip into a finished, day-by-day journal you export and keep — without ever leaving your phone or waiting on a cloud job." I want to stress "finished" and "export" as the operative words, because the thing that kills this category of app isn't clustering quality, it's that the user opens it, pokes around, and closes it without ever reaching an artifact. If I had to boil the whole design problem down to one sentence: every screen between "pick your trip" and "share sheet" is a screen where the user could quit, so the IA has to be short, forgiving, and always show a visible finish line.

Primary user and scenario, concretely: someone lands home from a trip, phone full of 400-1200 photos with no folder structure, opens Afterglow within the first week while the trip is still emotionally live. This is a single-session, couch-context task — half-attention, interruptible, done in one or two sittings, not a recurring habit. That has a real IA consequence I want on record: this is fundamentally a wizard, not a dashboard. The user needs a small number of forward-moving steps (select → review clusters → caption → theme → export), each with an obvious "next" and an obvious way to back out, rather than a tab-bar app that assumes repeat visits. A tab bar built for a daily-use mental model (Library / Create / Profile) would be actively wrong here — there's essentially one thing to do, and once in a while a second thing (open last trip, check the widget-linked journal). I'd design around a single primary flow plus a lightweight "Trips" list as the home screen, where an empty state IS the call to action ("Turn your last trip into a journal" with a date-range suggestion pulled from Photos metadata), not a blank screen with a floating + button the user has to interpret.

Core loop, stated as screens and state transitions rather than prose bullets: Trips home (empty or list of past journals) → date-range/photo selection (either "detect my last trip automatically" via a gap-in-photos heuristic, or manual date range, or manual multi-select) → day-cluster review, where each day is a horizontally-scrollable strip of photos with a cover photo already chosen by heuristic, and the user's only actions are reassign-a-photo-to-adjacent-day, delete-cluster, or merge-two-days → per-day caption/voice-note entry, one card per day, text field plus a mic button, entirely optional and skippable in bulk → theme selection, a single premium theme in v1 presented as a full-screen preview so it doesn't feel like a stub → export/share screen showing a real preview of the finished PDF before committing to render. That's five screens. I'd resist the temptation to add a sixth "welcome/onboarding" screen with slides explaining the concept — the empty state of the Trips home screen should do that work contextually the first time, folded into the permission ask, not as a separate carousel nobody reads.

The systems-thinking risk I want flagged before anyone gets attached to this flow: the review-clusters screen is where the whole "premium, effortless" promise either holds or breaks, and it's underspecified in every phase so far. What does "review auto-generated day clusters" actually let the user do when the clustering is wrong — and it will be wrong, because day-boundary logic across midnight, timezone changes mid-flight, and red-eye flights is genuinely ambiguous even to a human. If the only affordance is "accept the clusters as given," the app is fragile to the exact kind of trip most worth journaling (multi-timezone, overnight travel). I want the contract to say: clusters are user-adjustable, not just user-approved — drag a photo to the adjacent day, merge two days, split a day in two — and that adjustability is core loop, not a post-MVP nice-to-have, because without it the "review" step is decorative.

Second thing I want on record now rather than discovered during build: state persistence isn't just "don't lose data if the app backgrounds," it's specifically that a half-finished journal must reopen exactly where the user left it, with its cluster edits, captions, and voice notes intact, from the Trips home screen — meaning a journal-in-progress is a first-class object with its own row in that list, visually distinct from a finished/exported journal (draft badge vs. done badge). That distinction is a scope boundary worth calling hard now: v1 needs "draft" and "exported" as two real states of a Trip, not just an in-memory flow that either completes or evaporates.

Hard scope boundaries I'd lock for this phase, building on what prompt_contract already fixed: one theme, one journal editable at a time in the free tier (but the free/paid distinction is a gate on *number of trips*, not on whether editing feels crippled — a free user's one trip should feel like the full premium experience, or the invalidation-criteria signal about caption abandonment gets confounded with "this felt like a locked demo"). No account creation, no login screen, ever — that's a scope boundary worth stating explicitly since it's the kind of thing that creeps in when someone thinks about "future cloud backup" a phase too early. No in-app camera — this app only ever reads existing Photos library content, never captures new photos itself; blurring that line would turn this into a camera app, which it explicitly isn't.

Measurable success criteria, framed the way a design lead should — as user-visible thresholds, not engineering metrics: (1) a user with a real, messy camera roll (mixed timezones, some screenshots, some zero-GPS photos) can go from "open app" to "share sheet with a finished PDF" in under 5 minutes of active interaction, not counting time spent writing captions; (2) closing the app mid-flow at any of the five screens and reopening it returns the user to that exact state with zero data loss; (3) a completely GPS-less trip still produces a day-by-day journal that reads as intentional, not a flat photo dump — this is the one I'd actually want tested with a synthetic no-location library before calling this phase's understanding "done," because if the answer is "it just looks like an undifferentiated grid," the whole premium-keepsake promise is void for a meaningful slice of real users (indoor trips, flights, city trips with location services off). Those three are what I'd hold the eventual build to, more than "does clustering work" in the abstract.

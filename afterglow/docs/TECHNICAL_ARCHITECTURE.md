**Screen inventory (7 screens + 1 system surface):** Trips Home (no tab bar), Trip Setup (date range + photo review/add-back), Cluster Review (adjustable day strips), Caption & Voice Note Editor, Theme Preview, Export Preview & Share, and a Paywall that only appears when actually triggered (second trip creation) — plus the WidgetKit last-trip-cover widget as a separate system surface with its own tiny state machine (stays dark until a trip is actually exported, never shows an in-progress draft's cover photo).

**Primary flow:** Trips Home empty state doubles as onboarding (one full-bleed suggested-trip card + a manual "pick dates yourself" link, no separate welcome carousel) → Trip Setup (permission requested at point of need, with a real designed denied-state and Settings deep link, not a generic alert) → Cluster Review → Caption Editor (skip-all visible immediately) → Theme Preview → Export Preview → system share sheet → trip flips from draft to exported on success.

**Two decisions flagged as the highest-leverage calls in this pass, meant to be preserved verbatim:**
1. **Cluster reassignment is a long-press contextual action** ("Move to previous/next day" in a menu/bottom sheet), not drag-and-drop — because dragging a thumbnail while its own strip wants to scroll horizontally is a genuinely fiddly conflict that would eat MVP build time for a should-have-adjacent feature, and it's also the more accessible choice (drag gestures are weak for VoiceOver/Switch Control).
2. **Theme Preview is a confident single full-bleed cover reveal**, not a grid with locked "coming soon" placeholders — because a picker full of greyed-out options announces "incomplete product" at exactly the moment the app needs to feel like a finished keepsake. Future themes get a one-line text mention, not a fake picker UI.

**State model:** each screen gets its full empty/loading/populated/error treatment spelled out (zero-photos-in-range, single-day-trip layout, permission-revoked-mid-session, voice-note-recording-interrupted-by-call, export-cancelled vs. export-failed with retry, etc.) — satisfying the global rule that every screen needs all four states, not just the happy path.

**Visual direction:** warm/restrained paper-adjacent palette (off-white/warm-grey, not stark white), serif display type for trip titles/captions vs. sans-serif for UI chrome, generous negative space around full-bleed photos, a sparingly-used accent color reserved for interactive elements only — explicitly rejecting gradients, heavy drop shadows, and page-curl skeuomorphism as "trying too hard."

**Accessibility notes:** meaningful labels on every photo (falling back to date/time description when no better metadata exists), Dynamic Type support up to accessibility sizes tested against the full-bleed serif layout specifically, VoiceOver-navigable day strips, the long-press reassign menu doubling as the accessible alternative to drag-and-drop, non-color-only recording-state indicators, and Reduce Motion fallbacks (crossfade) on cover-reveal/export-preview transitions.

Both acceptance checks are met: a SwiftUI developer has enough here (screen list, transitions, per-screen states, concrete interaction patterns) to build the view hierarchy directly, and every required workflow from app_features has a mapped screen/state path. No open disagreement to carry into another round.

## Design Handoff

CONSENSUS: YES

## Final Output

Same pattern as every phase before this one — only Claude showed up again (Codex and Gemini still haven't come back online) — but this pass delivers a genuinely complete, builder-ready handoff that fills in exactly what design_discussion left implicit: the client architecture underneath the screens, not just their visual look.

**Architecture decisions locked for the build phase:**
- Single `NavigationStack` off Trips Home, driven by an enum-based `NavigationPath` (`TripFlowStep`) rather than five independently-owned screens — this is what makes "resume a draft" and the widget's deep link (`afterglow://trip/<id>`) both reduce to one function instead of scattered per-screen guesswork.
- One `@Observable @MainActor TripStore` backed by SwiftData as the single source of truth; photos are referenced only by `PHAsset.localIdentifier`, never stored as bytes. A shared `PHCachingImageManager` handles caching/prefetch as day strips scroll, with per-cell `.task(id:)`-scoped thumbnail fetches so fast scrolling cancels stale requests instead of piling up (ties directly to the "thousands of photos" stress case named back in prompt_contract).
- New edge case caught that no earlier phase named: a photo already in a draft or exported trip can be deleted from the user's library between sessions. Cluster Review, Caption Editor, and Export Preview all need an explicit "photo no longer available" tile (greyed placeholder + remove affordance) — not a crash on a force-unwrap, not a silent blank space.
- Autosave is specified as two independent paths, not one: a ~400-500ms debounced write for caption keystrokes, plus a hard flush on `scenePhase` going background/inactive that fires regardless of the debounce timer. This is called out explicitly because "autosave" as a single word invites building only the debounce half, which is exactly the gap that would lose a caption to an interruption — the one failure this entire app exists to prevent.
- The widget never touches PHAsset/PHImageManager/SwiftData directly (widget extensions are a separate, memory-constrained process) — instead, the main app writes a small pre-downsampled cover image plus a JSON snapshot to an App Group container at the moment a trip flips to exported, and the widget's TimelineProvider only reads that flat file.
- PDF export rendering runs off the main actor with progress reported back via callback/AsyncStream — named as a hard requirement because otherwise it directly contradicts design_discussion's "determinate progress bar, not indefinite spinner" call for Export Preview.

**Screen-by-screen spec, tokens, components, states, accessibility, motion:** all seven screens (Trips Home, Trip Setup, Cluster Review, Caption & Voice Note Editor, Theme Preview, Export Preview & Share, Paywall) got full layout intent, named components (e.g. `TripCard`, `DayStripHeader`, `ReassignActionSheet`, `MissingAssetTile`, `WaveformView`, `PDFPagePreview`), and every empty/loading/success/error state spelled out per the global rule — including states no prior phase had named (SwiftData load failure on Trips Home, voice-note-interrupted-by-call auto-saving the partial recording rather than discarding it, missing-asset cover-photo fallback on Theme Preview, share-cancelled leaving zero state mutation on Export Preview). Design tokens are concrete and specific: warm off-white/warm-grey palette (#F7F3EC / #EFEAE1), one muted terracotta accent (#B5654A), serif (`.serif` Font.Design, e.g. New York) for trip content vs. system sans for UI chrome, a 4/8/16/24/32/48 spacing scale, 12pt corner radius, near-invisible ambient shadows only. Motion is specified as crossfade + slight scale capped at ~300ms with an explicit `accessibilityReduceMotion` fallback path, and voice-note recording uses a pulsing-opacity indicator rather than a spinner so "recording" doesn't read as "loading."

**External design decision — made explicitly, not deferred:** no external Claude Design pass is needed for the core app. Every visual decision is expressible in native SwiftUI (system fonts, system spacing, SF Symbols, no custom iconset). The one narrow exception offered, with a ready-to-use scoped prompt: the app icon plus two or three empty-state illustrations, if the team wants something warmer than an SF Symbol glyph. That prompt is fully specified (palette, style constraints, size requirement) and usable as-is if the user wants to invoke it — otherwise the handoff is complete and buildable without waiting on anything external.

Both acceptance checks are met: a SwiftUI builder has layout intent, components, state variants, accessibility, and motion for every primary screen, and no hidden decision is left for the build phase to improvise — including the interaction-pattern and theme-picker calls from design_discussion, which are carried forward and built out into full component-level detail here.

Same pattern as every phase before this one — Codex and Gemini are still both down, so it's just Claude's pass again — but it's a thorough, closer-minded review that hits every required output for this phase and makes several concrete, previously-unstated decisions rather than restating what design_handoff already said.

## Final Output

**SwiftUI architecture:** Confirms the enum-driven `NavigationStack`/`TripStore` design from design_handoff, but closes a testability gap nobody had named: clustering, filtering, and cover-photo-heuristic logic must live in plain Swift types operating over a small `PhotoAsset` protocol (identifier, date, location, media subtype, pixel size) — never importing `PHAsset`/`PHPhotoLibrary` directly. The real Photos-backed implementation is a thin adapter at the store boundary. This is what makes the timezone/DST/zero-GPS fixtures runnable as fast, deterministic XCTest unit tests instead of manual QA redone by hand every time clustering logic changes — treated as a build-verification blocker, not a nice-to-have.

**Apple framework choices:** PhotosUI (`PHPickerViewController`), Photos/`PHImageManager`/`PHCachingImageManager`, AVFoundation (`AVAudioSession`/`AVAudioRecorder`), Speech, PDFKit/`UIGraphicsPDFRenderer`, WidgetKit, StoreKit 2 — all as previously planned. One correction locked in: voice notes need **two separate, sequenced permissions** — microphone first (recording works standalone), then Speech-recognition authorization only when transcription is actually attempted — each with its own denied-state copy, since asking for one doesn't grant the other and calling the speech recognizer without its own authorization is a hard failure, not a soft one.

**Persistence:** SwiftData model plan needs a `VersionedSchema` + `SchemaMigrationPlan` from the very first commit, even with nothing yet to migrate from — standing this up now is nearly free and prevents a self-inflicted data-loss incident the first time the schema changes post-launch, which would be strictly worse than the already-accepted uninstall-data-loss limitation.

**Permissions/privacy:** Two new concrete requirements: a `PrivacyInfo.xcprivacy` manifest is a tracked build task from day one (required-reason APIs, e.g. file timestamps, will otherwise get the binary rejected at upload, not at review); and Info.plist should declare `NSPhotoLibraryUsageDescription` only, not the "Add" variant, since this app never writes back into the user's Photos library — any future Photos-write call would be unreviewed scope creep.

**StoreKit/subscription:** Resolves a real gap — lapsed-subscription behavior. Existing trips remain permanently viewable/editable/re-exportable regardless of current entitlement; only new-trip creation beyond the one free slot is gated. Uses `Transaction.currentEntitlements` at launch plus continuous `Transaction.updates` listening (no polling, no receipt-validation server), and a local StoreKit Configuration file for all build-verification/simulator testing so entitlement gating doesn't depend on a live App Store Connect product existing yet.

**Widget:** `WidgetCenter.shared.reloadTimelines(ofKind:)` must fire at the exact moment a trip flips to exported (not left to system refresh budget), and the App Group identifier/URL scheme need to be real decided strings now (not placeholders), since App Group provisioning has to exist at the project level before the widget target compiles.

**Dependencies:** Zero third-party packages — every needed capability has a first-party framework. Even the PDF pixel-equivalence testing requirement gets solved in-house (rasterize PDF pages to `CGImage`, diff against a checked-in reference with a tolerance threshold) rather than pulling in a snapshot-testing library, keeping the permissive-dependency policy at literally zero.

**Testing:** Three guarantees, not aspirations — clustering/timezone/DST/zero-GPS fixtures run as unit tests against the protocol-abstracted photo layer; the permission state machine (Photos/mic/speech) is tested via dependency-injected fake authorization providers so CI can exercise denied/limited/revoked-mid-session without touching real system dialogs; and UI tests use a `--uitesting` launch-argument seed (in-memory SwiftData + fixture photo set) rather than depending on whatever's actually in the simulator's Photos library.

Nothing here contradicts anything locked in earlier phases — it sharpens design_handoff with concrete, cheap-now/expensive-later decisions (testable clustering layer, versioned schema, two-permission voice flow, privacy manifest as a tracked task, lapsed-subscription behavior). Both acceptance checks are satisfied: no vague architecture gaps remain, and no dependency violates the permissive-license policy since the dependency list is zero third-party packages.

## Tech Specs

CONSENSUS: YES

## Final Output

Same pattern as every phase so far — Codex and Gemini are still both down, so this is Claude's single pass again — but it's a complete, concrete tech-specs deliverable that closes out the architecture question every prior phase deferred: where does the domain logic actually live, and how is it packaged so the testability promises made in ios_architecture_review are real rather than aspirational.

**Architecture overview:** Afterglow splits into a pure-Swift local package (`AfterglowCore`, Foundation-only — no Photos, PhotosUI, AVFoundation, Speech, PDFKit, or SwiftUI imports) holding all clustering/burst-collapse/cover-heuristic/export-layout/schema logic, plus the app target that adapts real system frameworks onto that package's protocols at one seam (`PhotosLibraryAdapter`). This is what makes `swift test` on the domain logic run in under a second with no simulator, turning the timezone/DST/zero-GPS/burst fixtures into something that actually runs on every change instead of decaying into manual QA. A `GeoCoordinate` struct stands in for `CLLocationCoordinate2D` specifically so the package can't accidentally reach for CoreLocation. The one accepted risk: a second build target adds real setup friction, and if SwiftData macros fight the package boundary in practice, the fallback is a plain `Domain/` folder with the same zero-import discipline enforced by review instead of the compiler — not a blocker worth losing time over.

**State management:** One `@Observable @MainActor TripStore` exposing only intent methods (`createTrip`, `reassignPhoto`, `updateCaption`, `chooseTheme`, `beginExport`, `markExported`, `resumeStep`) — views and services never touch `ModelContext` directly. This is the detail flagged as most important to preserve: it's the single choke point that keeps the autosave discipline (debounce + scenePhase flush) from being silently bypassed by a different lane writing straight to a model property.

**File/module layout:** `AfterglowCore/` (package + its own test target), `Afterglow/Domain` (SwiftData `@Model`s + `TripStore`), `Afterglow/PhotosAdapter`, `Afterglow/Services` (audio, transcription, PDF export, entitlements, widget snapshot), `Afterglow/Views` (one folder per design_handoff screen), `Afterglow/App`, and `AfterglowWidget/` as its own target reading only the App Group snapshot.

**Data models:** `Trip`, `TripDay`, `PhotoRef` (identifier-only, with a persisted `isMissing` flag for the missing-asset-tile case), `VoiceNote` — all SwiftData `@Model`s under `AfterglowSchemaV1` with a `SchemaMigrationPlan` scaffolded from commit one, per ios_architecture_review. New addition: a `JournalExportSidecar` JSON written at export time (needed anyway for future cloud-sync-readiness) doubles as a cheap, explicitly-scoped self-service backup path — shared as a secondary "Share journal data" action next to "Share PDF." Flagged as should-have, export-only, with no import path — naming that limitation explicitly rather than letting it balloon into unbudgeted import-flow scope.

**Persistence & concurrency:** debounced caption writes + hard scenePhase flush as two independent call sites into the same `TripStore` method; widget snapshot write happens synchronously inside `markExported` before `reloadTimelines` fires, closing the stale-widget race. Thumbnail fetches use `PHCachingImageManager` with opportunistic delivery, scoped per-cell via `.task(id:)` for cancellation; clustering runs off-main via a `nonisolated` call for libraries over ~500 assets; `PDFExportRenderer` is a Swift `actor` streaming `AsyncStream<ExportProgress>`, making the determinate-progress-bar requirement and the off-main-actor requirement the same code path instead of two requirements in tension.

**Testing strategy:** three explicit tiers — pure `AfterglowCoreTests` (fastest, no simulator, all the clustering/DST/zero-GPS/burst fixtures), app-level `XCTest` against an in-memory `ModelContainer` with fake `PermissionProviding` conformances for deterministic denied/limited/revoked-mid-session coverage, and `--uitesting`-seeded UI tests against a fixture `PhotosLibraryAdapter` rather than the simulator's real camera roll. PDF pixel-equivalence lives in the app-level tier since it needs the real PDFKit path.

**interfaces-json:** provided in full below — every cross-lane type and function a parallel build lane needs (data models, protocols, services, UI navigation, error/resolution types) is named with a concrete signature, so build lanes can code against a shared contract without inventing incompatible types.

Both acceptance checks are met: the interfaces-json is complete enough that data/UI/services/polish lanes can build in parallel without collisions, and the spec states concretely how the app gets built (package boundary, adapters, intent-based store) and verified (three-tier testing strategy with deterministic fixtures for every previously-flagged risk). No open disagreement to carry into another round.

**Claude (Frontend Engineer · User Advocate) — Round 1**

Here's my pass on this, coming at it as the person who actually has to wire up state management and navigation and who will get blamed when the "under 20 seconds to check, under 90 to log a cycle" promise turns into three taps and a spinner.

The biggest thing I want to nail down before any lane starts coding is what "thin per-surface view model holding raw record arrays" from the architecture review actually means in practice, because if we take that literally we ship a bug on day one. If Dashboard's view model and Money Inputs' view model each hold their own copy of `[IncomeEntry]`, then logging income from the Dashboard's quick-add sheet either has to explicitly push into two places or Money Inputs goes stale until it's re-navigated to. The design handoff already promises the Log Income sheet "writes to the exact same income-entry store that Money Inputs reads from" — the only way to make that literally true and not just aspirational prose is a single shared, owned `FinancialStore`, constructed once at the app root and handed down via the environment, that is the one and only place `[IncomeEntry]`, `[Obligation]`, and the reserve live in memory. Per-screen view models don't own arrays, they own presentation state (sheet-is-showing, selected filter, form drafts) and read the shared store directly. This also simplifies the recompute story: instead of "each view model recomputes explicitly on mutation," it's "the store recomputes its own `planningResult` synchronously inside every mutating method," and since the dataset here is a few hundred to low thousands of records at most, running `PlanningEngine.evaluate` synchronously on every mutation is well under a millisecond — there's no reason to hand-roll cache invalidation logic that can itself go stale. Fewer moving parts, one source of truth, matches the product's own "recompute every time, never trust cached derived state" philosophy that was already locked for persistence.

Root state machine: I want to extend `AppRootState` from the two cases in the architecture review to four — `.launching`, `.onboarding`, `.locked`, `.unlocked` — because right now there's no defined state for "the app hasn't finished reading the two JSON files off disk yet," and if that read happens synchronously on `App.init` on an older device with a few years of history, that's a hitch on launch that nobody speced a UI for. `.launching` shows the same calm branded surface as the App Lock gate (not a separate generic splash — visual continuity matters if lock is enabled next) while settings and, conditionally, the financial document load off the main actor. I want to push the "does not construct while locked" guarantee one step further than the architecture review wrote it: it shouldn't just be that Dashboard/Money-Inputs view models aren't instantiated while locked, the financial document itself shouldn't be decoded into memory until authentication succeeds. Settings (small, non-sensitive) load immediately since we need `lockEnabled`/`onboardingCompleted` to pick the right phase, but the actual income/obligation/reserve data stays on disk, unread, until unlock. The real risk this creates is perceived latency on unlock — if we naively wait for Face ID to succeed and then start decoding, the user sees their thumbprint scan resolve and then a beat of nothing before the dashboard appears, which reads as sluggish. The fix is to kick off the async file load the moment authentication begins, in parallel with the biometric prompt (which itself takes a few hundred ms), so by the time auth resolves the data's already in memory and the transition is a single frame. That detail has to survive into the build or someone will wire it as load-after-auth-succeeds and it'll feel slow.

On the file layout, I'd split it by responsibility rather than by screen-only, since half the app's risk is in the non-UI layer: a `Domain` module with the pure engine and record types (zero SwiftUI or Foundation-file-IO imports beyond `Decimal`/`Date`, testable in a plain Swift package target if we want that isolation for free), a `Persistence` module with the two Codable document types plus the atomic-write stores, an `Auth` module with the authenticator protocol and its two implementations, an `Entitlement` module with the one-boolean service, and then `App` (root coordinator, root state enum, scene lifecycle) and `Features/{Dashboard,MoneyInputs,ExplanationHistory,Settings,Onboarding,AppLock}` each holding a view plus a view model that only reads from the shared stores. A `SharedUI` module holds the parameterized `EmptyStateView`, formatting helpers (the locale-explicit currency formatter has to live in exactly one place, not get reimplemented per screen with subtly different rounding), and design tokens. This split is also what makes "parallel build lanes without inventing incompatible types" actually work — the Domain and Persistence modules are the contract, and UI lanes just consume them.

Two things I'm flagging as real objections rather than settled: first, `RiskWindowCard` was described in the singular in the design handoff, but the underlying reality is that more than one obligation can land inside the 14-day window at once — a quarterly tax payment and a rent due date can easily collide. I've modeled `PlanningResult.riskWindowFlags` as an array, not an optional single flag, and the Dashboard card should show the nearest/most urgent one with an affordance to the rest rather than silently dropping the second one, because silently dropping a due obligation is exactly the kind of "false authority" bug this app was designed to prevent. Second, storage corruption: if `app-settings.json` fails to decode, I want to reset it to defaults and treat it like a fresh (but not financially empty) install — losing "lock enabled" and "onboarding seen" is annoying but recoverable and non-sensitive. If `financial-document.json` fails to decode, that's the actual user's money history and resetting it silently would be a real breach of trust, so that has to surface as its own recoverable error state on the Dashboard offering "import from backup" rather than either crashing or quietly starting the user over with zero data.

For testing, my lens adds state-management-specific cases on top of what's already locked: a test asserting `AppRootCoordinator.financialStore` is `nil` while phase is `.locked` (already agreed, just restating the exact assertion point now that the store is a single shared object rather than per-screen copies); a test that mutating the shared store's income array through the "log income" path and reading it through a Money-Inputs-facing accessor return the identical array without a second write path existing; and a timing-shaped test (not a literal stopwatch assertion, but a structural one) that the financial document load is kicked off in parallel with authentication, not sequenced after it — the easiest way to verify that without flaky timing assertions is a unit test on the coordinator confirming the load `Task` starts the instant `.locked` transitions to `.unlocked` begins, not after a callback from the authenticator's success completion.

### Round 2


**Codex (QA & Risk · Pragmatist) — Round 2**

Claude’s push toward a single shared `FinancialStore` is the right correction, and I don’t think we should leave any ambiguity around it now. The architecture-review wording about “per-surface view models hold raw record arrays” is exactly the kind of sentence that looks harmless in a summary and then produces a real bug in build lanes: Dashboard quick-add mutates one array, Money Inputs is reading another copy, and now we’ve shipped disagreement between the two screens in the one app that absolutely cannot afford state drift. So I’d lock this hard: one shared `FinancialStore`, one shared `AppSettingsStore`, both owned by `AppRootCoordinator`, both injected downward, and per-screen view models only own transient presentation state. That is the smallest architecture that actually protects correctness. I also agree with the extension from `.locked/.unlocked` to `.launching/.onboarding/.locked/.unlocked`, but I want to be careful about the preload idea. Codex is right that decode-after-auth-success will feel sluggish. The part I don’t want us to hand-wave is the privacy boundary: if we say “financial data is not decoded into memory until unlock,” then preloading during biometric prompt is technically a violation of that rule even if the user is about to succeed. My pragmatic answer is to make the rule more precise instead of pretending we can have both absolutes: the financial document may begin loading only after the user has initiated unlock and the authenticator is actively running, and the decoded store must not be exposed to any UI tree until auth succeeds. That gives us the latency win without claiming a stricter memory rule than we’re actually implementing.

On the unresolved type conflicts, I think the simplest shippable answers are also the safest. Keep `PlanningEngine` as a non-throwing pure function with a small explicit `PlanningConfig` passed in. That sounds like splitting the difference, but it matters: Claude was right to resist making product constants disappear into hidden defaults, and Codex was right that they should not be user-configurable. So the engine should not hardcode 14 days and thresholds invisibly, but those values also should not live in editable app settings. Put them in a compile-time `PlanningConfig.defaultMVP`, pass that into the engine everywhere, and test against it. That gives parallel lanes a single contract and future-premium flexibility without inventing runtime complexity. On record shapes, I would pick the leaner variants: `IncomeEntry` with `date`, `amount`, `note`, `createdAt`, computed `isCorrection`; `Obligation` with one `dueDate`, `cadence`, `amount`, `name`, `createdAt`. No `updatedAt`, no `endDate`, no `isActive` in MVP. Every extra field there creates reconciliation and validation work that the current product does not need, and “inactive obligation” is just deletion or editing in this scope. On risk windows, Claude’s array of per-obligation flags wins cleanly. A single bundled flag is great until two obligations hit inside 14 days with different amounts and due dates, and then the Dashboard silently collapses distinct risk into one blob. That’s exactly the kind of false simplification this app was supposed to avoid. Last piece: root-level corruption should not become an `AppRootState.recoverableError` catch-all. Keep phase and error separate. `AppRootState` answers “what mode is the app in?” while a separate `financialStoreLoadError` answers “did the money file fail to decode?” Mixing those makes the state machine harder to reason about and harder to test.

One more thing I want explicitly preserved because it’s an easy place to accidentally lie to ourselves: if `app-settings.json` is corrupt, reset to defaults and log it; if `financial-document.json` is corrupt, do not silently zero the app out. That must surface as a recoverable financial-data error state with import-from-backup as the escape hatch. And for tests, I want one concrete assertion added to Codex’s preload idea: when unlock begins, the coordinator may start the load task, but if authentication fails or is cancelled, the coordinator must discard that task’s result and keep `financialStore == nil`. Otherwise we’ll have technically “locked” UI with already-decoded money state hanging around in memory after a failed auth, which is exactly the kind of edge-case leak a QA pass should be killing now instead of after implementation.

**Claude (Frontend Engineer · User Advocate) — Round 2**

Okay, digging into the five open conflicts the coordinator flagged, because none of them are stylistic — they're the kind of thing that breaks the build if two lanes pick different answers.

On store ownership: I'm not backing off this one. Codex's file layout in round 1 didn't actually rule out a shared `FinancialStore`, but his `MoneyInputsViewState.ready(document: FinancialDocument)` case in the interfaces block quietly reintroduces the exact problem I was worried about — the moment a screen-state enum carries a *copy* of the document as associated data, you've created a second place that data lives, and now "did Money Inputs's copy get refreshed after the Log Income sheet wrote to the store" becomes a real question again instead of a non-issue. So here's the sharper version of my point: it's not just that we need one shared store, it's that our screen-facing state enums shouldn't carry data payloads at all, because associated-value snapshots are just cache invalidation wearing a disguise. A view that reads `store.incomeEntries` directly, live, off the environment-injected `FinancialStore`, can never be stale — there's nothing to invalidate. A view that reads `case ready(document: FinancialDocument)` from its own local `MoneyInputsViewState` can absolutely be stale the instant another screen mutates the store, unless someone remembers to re-derive the enum on every store change, which is exactly the kind of "someone remembers to" bug class this whole product exists to avoid in its own domain logic. So I want to go further than just winning the shared-store argument — I want to strip the data payloads out of Codex's `DashboardViewState`, `MoneyInputsViewState`, `ExplanationHistoryViewState`, and `SettingsViewState` entirely.

And actually, once you follow that through, I don't think most of those enums need to exist at all. Think about when a screen is actually "loading" in this app. The only genuine async load is the root-level bootstrap — reading two JSON files off disk and constructing the `FinancialStore` — which happens once, during the `.locked → .unlocked` transition, before `MainTabView` is ever built. Because `RootView` only constructs the four feature screens once it already holds a live, non-optional `FinancialStore`, every downstream view gets synchronous, already-loaded data by construction — there is no moment where Dashboard exists but its data hasn't arrived yet. So a `DashboardViewState.loading` case is describing a state that structurally cannot occur after the screen is on-screen. What Dashboard actually needs to distinguish between is: fully computed, low confidence, insufficient data — and that's already exactly what `ConfidenceTier` encodes, no wrapper enum needed, the view just switches on `store.planningResult.confidenceTier` directly. The one real per-screen exception is a corrupted financial document, and that's not a "loading" state either, it's a load-failure — which the coordinator already exposes as `financialStoreLoadError`, and Dashboard just checks that one optional before it even looks at confidence tier. I'd cut `DashboardViewState`, `MoneyInputsViewState`, and `ExplanationHistoryViewState` completely. The one place a real transient async state machine is earned is Settings' import flow — picking a file, reading it, validating, confirming, swapping — and that's worth its own small `ImportFlowState`, scoped to that one feature, not a generic pattern repeated four times.

On the engine signature, I want to partially concede to Codex and partially push back hard. He's right that hardcoding thresholds inside a static function is worse than passing them as an explicit `PlanningConfig` — I said this myself in round 1 in spirit ("asOf always explicit") but didn't extend the same logic to the other constants, and that's inconsistent of me. A fixture test that wants to prove "confidence tier flips at exactly 2 entries" is much cleaner against an injectable threshold than a magic number buried in a switch statement. So: config, yes. But `throws` on a pure function computing over already-validated in-memory data — no. Where would the error even come from? Domain validation already happens before data reaches the engine, at the persistence layer's decode-and-validate step and in each `FinancialStore` mutator. If we let the engine throw, we're implicitly saying "the engine might receive invalid data," which means either validation elsewhere is unreliable (fix that instead) or we're inventing a second validation layer inside the one place in this app that's supposed to be pure math. I also want to cut the `BaselineStrategy` enum with its two cases — `.manualLowMonthEstimate` doesn't belong there, because the manual fallback isn't a strategy the engine *chooses*, it's a value the *user* supplies that gets surfaced when the engine can't compute at all. That's already represented on the output side, in `ConfidenceTier.insufficientData(manualFallback:)`. So the config only ever needs one baseline shape — I'd flatten it to two plain Int fields, `lookbackMonths` and `baselinePercentile`, and drop the enum, since building an extensible strategy type for a strategy that will only ever have one member is exactly the kind of premature abstraction Codex himself warned against with "business rules smeared across views" — same instinct, applied to his own proposal. Concretely:

`static func evaluate(income: [IncomeEntry], obligations: [Obligation], reserve: ReserveSnapshot?, manualFallbackEstimate: Decimal?, asOf: Date, config: PlanningConfig = .standard) -> PlanningResult` — non-throwing, `manualFallbackEstimate` pulled in from Codex's protocol since that's a real gap in my round-1 version (the engine genuinely needs that as an input to populate the insufficient-data output), config defaults to `.standard` for every production call site but is overridable in tests.

On risk window, I'm keeping my array-of-flags version and I think this one isn't close. Codex's `.upcomingObligation(daysUntilDue: Int, obligationIDs: [UUID])` collapses every obligation inside the window into a single `daysUntilDue` value. Concretely: rent due in 12 days and a quarterly tax payment due in 3 days both fall inside a 14-day window, and his shape has no way to say which is which without the UI going back to look up each ID against the obligations array — which is precisely the "explanation screen re-derives something from scratch" anti-pattern he called out as the thing that creates a second source of truth. My `[RiskWindowFlag]`, with `obligationName`, `amount`, and `dueInDays` denormalized onto each flag, is self-contained — Explanation/History can render "Rent — $1,400 — due in 12 days" and "Q3 estimated tax — $600 — due in 3 days" straight off `PlanningResult` with zero additional lookups. I'd genuinely like Codex to point out if I'm missing a reason his single-bucket version was intentional, but as it stands I think it loses real information the product needs.

On root-level error handling, I don't think this is actually a live disagreement once you check what `design_handoff` already locked — go re-read the Dashboard state list from that phase: "first-load/skeleton... corrupted-storage error (rare, recoverable)" is listed as one of *Dashboard's own* states, not a root-level app takeover. That already settles it in favor of a coordinator-level `financialStoreLoadError: Error?` property that Dashboard (and arguably Money Inputs, since it also needs the data) reads directly, rather than a `.recoverableError(String)` case on `AppRootState` that would replace the whole four-tab UI with one error screen. A corrupted financial document shouldn't take away the user's ability to get to Settings and import a backup — which a root-level takeover would risk doing unless someone very carefully special-cases navigation inside that error state too. Keeping `AppRootState` at exactly four cases and surfacing the load failure as a side-channel property is simpler and matches what was already decided two phases ago.

Last thing, a new gap nobody's raised yet: if `FinancialStore` calls `Date()` internally every time a mutator triggers a recompute, we've smuggled a hidden wall-clock read back into the system one layer up from the engine — the engine's `asOf` parameter is safe, but if the store always supplies `Date()` for that parameter with no way to override it, `FinancialStore` mutation tests become just as time-dependent as an untested engine would've been. So I want `FinancialStore` to take an injected `now: () -> Date = Date.init` closure in its initializer, defaulting to real time in production and pinned to a fixed date in tests — cheap to add now, and it's the difference between "mutate the store and assert the recomputed result" being a deterministic unit test versus one that only passes reliably on days that don't cross a month boundary mid-test-run.

Alright, let's close the loop on what the coordinator flagged, because I think most of it actually resolves cleanly once you look at it from a "what does SwiftUI's Observation framework actually require" angle rather than a taste angle.

First, easy concession: Codex, you're right about `updatedAt` and I was sloppy carrying it forward. I went back and checked — nothing in any locked screen spec shows a "last edited" timestamp for an income entry or obligation. Reserve has `lastUpdated` because design_handoff explicitly puts a "last updated" caption on the `ReserveStatPanel`, but there's no equivalent requirement anywhere for income rows or obligation rows. Carrying a field that nothing reads or tests is exactly the kind of premature field I'd normally be the one arguing against, so dropping it. On the naming, `date` vs `receivedOn` — I'll just pick a lane here rather than let this rot: `receivedOn`, because in a fixture file or a diff, `date` sitting one line above `createdAt` is genuinely easy to misread at a glance, and `receivedOn` removes the ambiguity for free. That's a minor call, I'm not going to defend it past this paragraph.

Second concession, bigger one: per-call explicit `asOf` beats my injected `now` closure on `FinancialStore`, and I want to say clearly why I changed my mind instead of just nodding at the coordinator's synthesis. My worry was "don't let mutation-triggered recompute secretly read `Date()`," and the closure does solve that — but it solves it by centralizing time at construction, which means a test that wants to simulate "user logs income on the 1st, then logs a correction on the 15th, crossing month buckets mid-test" has to mutate a captured clock object between calls, which is more test machinery than just passing two different `asOf` values directly into two calls. And there's a consistency win I underweighted: the engine's own contract is "asOf always explicit, per call, never ambient" — making the store's mutators follow that exact same shape means there's one mental rule for the whole app about how time enters the system, not two slightly different ones. So: `store.addIncome(entry, asOf: .now)` at the one call site where a save action fires, not a constructor-captured closure. Conceded.

Where I'm not conceding, and think this is worth spelling out because it's a real Swift/SwiftUI mechanics issue and not a preference: `FinancialStore` needs to stay a concrete class, not a protocol, and this isn't just my "no fakes needed" argument from round 2 — there's a sharper technical reason. Observation's fine-grained view-update tracking is implemented by the `@Observable` macro rewriting each stored property's accessor to call into an `ObservationRegistrar` on the concrete type. When a SwiftUI view's `body` reads `store.planningResult`, the framework registers that specific access so the view only re-renders when that specific property changes — that's the entire point of switching off `ObservableObject`/`@Published` in the first place. If `FinancialStore` is a protocol and views hold `any FinancialStore` (which is what environment injection of a protocol type would look like), you're going through an existential, and the compiler-generated access tracking that Observation relies on doesn't reliably propagate through protocol dispatch the same way it does through direct concrete-type property access — you risk either over-invalidating (the view re-renders on any property change instead of just the one it reads) or, worse, silently losing tracking so a mutation doesn't trigger a re-render at all. On top of that, the modern `@Environment(SomeType.self)` injection pattern for `@Observable` types is documented against a concrete type, not an arbitrary protocol — so Codex's `protocol FinancialStore: AnyObject` proposal would need some extra plumbing just to get environment injection working the way the rest of this app's screens assume it does. Given there's genuinely only one implementation of this store ever going to exist — it's not wrapping untestable hardware like `LAContext`, it's just arrays plus a call into a pure function — a protocol buys nothing and risks a real, hard-to-debug SwiftUI staleness bug that would look exactly like the drift problem we spent two rounds trying to eliminate. I'd like Codex to weigh in specifically on this because it's a genuine technical claim, not a style preference, but I think it settles it in favor of the concrete class.

On `manualFallbackEstimate` — confirming it stays in the engine call, agreeing with the coordinator's read that it dropped from Codex's round-2 JSON by omission rather than intent, since nothing in his prose argued against it and the app can't populate `ConfidenceTier.insufficientData(manualFallback:)` without the engine receiving it as an input somewhere.

## Final Output

**Architecture overview:** SwiftUI + Observation (iOS 17+). All business logic lives in a pure, non-throwing `PlanningEngine.evaluate` — a static function taking `income`, `obligations`, `reserve`, `manualFallbackEstimate`, an always-explicit `asOf: Date`, and a `PlanningConfig` (defaulting to `.standard`) — with zero side effects and zero hidden constants. Above that sits exactly one shared source of truth: a concrete `@MainActor @Observable final class FinancialStore` (income/obligations/reserve/manual fallback plus a cached `planningResult`, recomputed synchronously on every mutator call) and a parallel `AppSettingsStore`, both owned by `AppRootCoordinator` and injected via `@Environment` once fully populated — never duplicated per screen. `FinancialStore` stays a concrete class rather than a protocol: there's only one real implementation, and routing `@Observable` access through a protocol existential risks breaking or degrading Observation's fine-grained view-update tracking. The one place a genuine hardware-wrapping protocol is warranted is `AuthenticatorProtocol`, since it wraps untestable `LAContext` behavior. The app root is a four-case `AppRootState` (`.launching/.onboarding/.locked/.unlocked`) — while `.locked`, the financial document is not even decoded into memory. On unlock, the document load begins the moment the user initiates authentication (running concurrently with the biometric prompt, never before it and never exposed before it succeeds), and is discarded if auth fails or is cancelled. Because a screen is never constructed before the store is fully populated, there's no real per-screen "loading" state to model — screens read `store.planningResult.confidenceTier` etc. directly, and the four bespoke `*ViewState` enums originally proposed were cut in favor of that direct read, with only Settings' import feature getting its own genuine async state (`ImportFlowState`). Storage corruption is handled as a side-channel `financialStoreLoadError` on the coordinator, not a root-level takeover, so a corrupted file never blocks the user's path to Settings/import-backup.

**File/module layout:** `Domain/` (pure engine, record types, `PlanningConfig`, no SwiftUI/persistence imports — testable as plain Swift), `Domain/Validation/DomainValidator.swift`; `Persistence/` (`FinancialDocumentStore`, `SettingsStore`, atomic-write implementations); `Auth/` (`AuthenticatorProtocol` + real/fake implementations); `Entitlement/` (`EntitlementService`); `Services/ImportExport/ImportExportService.swift`; `Services/Formatting/CurrencyFormatting.swift`; `App/` (root coordinator, `AppRootState`, scene lifecycle); `Features/{Dashboard,MoneyInputs,ExplanationHistory,Settings,Onboarding,AppLock}` (view + presentation-only view model per screen — `isQuickAddPresented`, `editingIncomeID`, etc., never raw record arrays); `SharedUI/` (parameterized `EmptyStateView`, hero card, ribbon path, formatting helpers — one path, not one per screen); `Tests/{DomainTests,PersistenceTests,ViewModelTests,SnapshotTests}`.

**Data models (final, canonical):**
- `IncomeEntry { id, amount: Decimal, receivedOn: Date, note: String?, createdAt: Date, isCorrection: Bool (computed from amount < 0) }` — same-date duplicates always distinct records, never merged.
- `Obligation { id, name, amount: Decimal, cadence: ObligationCadence, dueDate: Date, createdAt: Date }` — no `endDate`/`isActive`; a passed one-off obligation simply stops contributing to future math until edited/deleted.
- `ObligationCadence { weekly, monthly, quarterly, annual, oneOff }`.
- `ReserveSnapshot { amount: Decimal, lastUpdated: Date }` — single current value, never a ledger.
- `ConfidenceTier { insufficientData(manualFallback: Decimal?), lowConfidenceComputed, fullyComputed }`.
- `MonthlyBucket { monthStart: Date, totalIncome: Decimal }` — zero-income months always materialized, never omitted.
- `RiskWindowFlag { obligationID, obligationName, amount, dueInDays }` as an array on `PlanningResult` — never collapsed into one bundled value, since two obligations can be due at different times inside the same window.
- `RunwayResult { monthsOfRunway, basedOnReserveAmount, reserveLastUpdated }`, nil when reserve is missing.
- `ExplanationLineItem { id, label, amount, emphasis: ExplanationEmphasis(normal/caution/result) }`.
- `PlanningConfig { lookbackMonths, lowConfidenceEntryThreshold, fullyComputedEntryThreshold, riskWindowDays, baselinePercentile, currencyCode? }`, `.standard` = 12 months / 2 / 6 / 14 days / 10th percentile — compiled-in, never user-editable.

- `FinancialDocument { schemaVersion, incomeEntries, obligations, reserve? }` — the only thing export/import touches.
- `AppSettings { schemaVersion, onboardingCompleted, lockEnabled, entitlementActive }` — never touched by import/export; corruption resets silently to defaults.

**Persistence strategy:** No SwiftData, no Keychain. Two versioned `Codable` JSON files in Application Support under file protection, both atomically written (temp-write then `replaceItemAt`): `financial-document.json` and `app-settings.json`. Import is replace-only, never merge — fully decoded and domain-validated in memory (`validateFinancialDocument`) before any write touches the live document, with a user confirmation step before the swap.

**Testing strategy:** Heavy weight on the engine and persistence over UI. Engine: fixture tests for windfall/lean months, zero/one/two-to-five/full-history confidence tiers, all five cadences, 14-day risk window (including flags from multiple simultaneous obligations), negative safe-to-spend, same-day duplicates, negative corrections. Persistence: two-part relaunch assertion (raw record equality, then a fresh `PlanningEngine.evaluate` against reloaded records with the original `asOf`), schema-invalid import leaving the on-disk file byte-identical, cancelled import producing zero writes. Coordinator: unit test asserting `financialStore == nil` while `.locked`, and a test that a failed/cancelled auth discards any in-flight document load. Selective snapshot tests on the hero card across confidence states at default and accessibility-5 Dynamic Type. App-switcher-snapshot redaction remains a manual QA checklist item, not a CI-provable test.

## Final Output

**Architecture overview:** SwiftUI + Observation (iOS 17+). All business logic lives in a pure, non-throwing `PlanningEngine.evaluate` — a static function taking `income`, `obligations`, `reserve`, `manualFallbackEstimate`, an always-explicit `asOf: Date`, and a `PlanningConfig` (defaulting to `.standard`) — with zero side effects and zero hidden constants. Above that sits exactly one shared source of truth: a concrete `@MainActor @Observable final class FinancialStore` (income/obligations/reserve/manual fallback plus a cached `planningResult`, recomputed synchronously on every mutator call, every mutator taking an explicit per-call `asOf: Date`) and a parallel `AppSettingsStore`, both owned by `AppRootCoordinator` and injected via `@Environment` once fully populated — never duplicated per screen. `FinancialStore` stays a concrete class rather than a protocol: there's only one real implementation, and routing `@Observable` access through a protocol existential risks breaking or degrading Observation's fine-grained view-update tracking, while `@Environment(Type.self)` injection is documented against concrete types. The one place a genuine hardware-wrapping protocol is warranted is `AuthenticatorProtocol`, since it wraps untestable `LAContext` behavior. The app root is a four-case `AppRootState` (`.launching/.onboarding/.locked/.unlocked`) — while `.locked`, the financial document is not even decoded into memory. On unlock, the document load begins the moment the user initiates authentication (running concurrently with the biometric prompt, never before it and never exposed before it succeeds), and is discarded if auth fails or is cancelled. Because a screen is never constructed before the store is fully populated, screens read `store.planningResult.confidenceTier` etc. directly rather than through a loading/ready wrapper enum; only Settings' import feature gets its own genuine async state (`ImportFlowState`). Storage corruption is a side-channel `financialStoreLoadError` on the coordinator, not a root-level takeover, so a corrupted file never blocks the user's path to Settings/import-backup.

**File/module layout:** `Domain/` (pure engine, record types, `PlanningConfig`, no SwiftUI/persistence imports), `Domain/Validation/DomainValidator.swift`; `Persistence/` (`FinancialDocumentStore`, `SettingsStore`, atomic-write implementations); `Auth/` (`AuthenticatorProtocol` + real/fake implementations); `Entitlement/` (`EntitlementService`); `Services/ImportExport/ImportExportService.swift`; `Services/Formatting/CurrencyFormatting.swift`; `App/` (root coordinator, `AppRootState`, scene lifecycle); `Features/{Dashboard,MoneyInputs,ExplanationHistory,Settings,Onboarding,AppLock}` (view + presentation-only view model per screen); `SharedUI/` (parameterized `EmptyStateView`, hero card, ribbon path, formatting helpers); `Tests/{DomainTests,PersistenceTests,ViewModelTests,SnapshotTests}`.

**Data models (final, canonical):**
- `IncomeEntry { id, amount: Decimal, receivedOn: Date, note: String?, createdAt: Date, isCorrection: Bool (computed from amount < 0) }` — no `updatedAt`; same-date duplicates always distinct records.
- `Obligation { id, name, amount: Decimal, cadence: ObligationCadence, dueDate: Date, createdAt: Date }` — no `endDate`/`isActive`/`updatedAt`.
- `ObligationCadence { weekly, monthly, quarterly, annual, oneOff }`.
- `ReserveSnapshot { amount: Decimal, lastUpdated: Date }` — single current value, never a ledger.
- `ConfidenceTier { insufficientData(manualFallback: Decimal?), lowConfidenceComputed, fullyComputed }`.
- `MonthlyBucket { monthStart: Date, totalIncome: Decimal }` — zero-income months always materialized.
- `RiskWindowFlag { obligationID, obligationName, amount, dueInDays }` as an array on `PlanningResult`.
- `RunwayResult { monthsOfRunway, basedOnReserveAmount, reserveLastUpdated }`, nil when reserve is missing.
- `ExplanationLineItem { id, label, amount, emphasis }`.
- `PlanningConfig { lookbackMonths, lowConfidenceEntryThreshold, fullyComputedEntryThreshold, riskWindowDays, baselinePercentile, currencyCode? }`, `.standard` = 12 months / 2 / 6 / 14 days / 10th percentile.
- `PlanningResult { asOf, safeToSpend (never floored at zero), confidenceTier, baseline, obligationsDueThisCycle, runway?, riskWindowFlags, monthlyIncomeBuckets, explanation }` — never persisted.
- **`FinancialDocument { schemaVersion, incomeEntries, obligations, reserve?, manualFallbackEstimate? }`** — the field added this round; this is the only thing export/import touches, and manual fallback now correctly survives relaunch and travels with a backup, since it's the user's own financial input, not app config.
- `AppSettings { schemaVersion, onboardingCompleted, lockEnabled, entitlementActive }` — deliberately excludes manual fallback; never touched by import/export.

**Persistence strategy:** No SwiftData, no Keychain. Two versioned `Codable` JSON files in Application Support under file protection, both atomically written: `financial-document.json` and `app-settings.json`. Import is replace-only, fully validated in memory (including a new check rejecting a negative `manualFallbackEstimate`, which has no sensible meaning unlike a negative income correction) before any write touches the live document, with a user confirmation step before the swap.

**Testing strategy:** Engine fixture tests (windfall/lean months, all confidence tiers, all cadences, 14-day risk window with multiple simultaneous flags, negative safe-to-spend, same-day duplicates, negative income corrections, rejected negative fallback estimate). Persistence: two-part relaunch assertion (raw record equality, then a fresh engine run against reloaded records with the original `asOf`) — now explicitly covering `manualFallbackEstimate` durability as part of that same test, plus export/import round-trip proving an insufficient-data user lands on the identical `ConfidenceTier.insufficientData(manualFallback:)` after restore. Schema-invalid import leaves the on-disk file byte-identical; cancelled import produces zero writes. Coordinator: `financialStore == nil` while `.locked`; failed/cancelled auth discards any in-flight document load. Selective hero-card snapshot tests across confidence states at default and accessibility-5 Dynamic Type. App-switcher-snapshot redaction stays a manual QA checklist item.

## Final Output

**Architecture overview:** SwiftUI + Observation (iOS 17+). All business logic lives in a pure, non-throwing `PlanningEngine.evaluate` — a static function taking `income`, `obligations`, `reserve`, `manualFallbackEstimate`, an always-explicit `asOf: Date`, and a `PlanningConfig` (defaulting to `.standard`) — with zero side effects and zero hidden constants. Above that sits exactly one shared source of truth: a concrete `@MainActor @Observable final class FinancialStore` (income/obligations/reserve/manual fallback plus a cached `planningResult`, recomputed synchronously on every mutator call, every mutator taking an explicit per-call `asOf: Date`) and a parallel `AppSettingsStore`, both owned by `AppRootCoordinator` and injected via `@Environment` once fully populated — never duplicated per screen. `FinancialStore` stays a concrete class rather than a protocol: there's only one real implementation, and routing `@Observable` access through a protocol existential risks breaking or degrading Observation's fine-grained view-update tracking, while `@Environment(Type.self)` injection is documented against concrete types. The one place a genuine hardware-wrapping protocol is warranted is `AuthenticatorProtocol`, since it wraps untestable `LAContext` behavior. The app root is a four-case `AppRootState` (`.launching/.onboarding/.locked/.unlocked`) — while `.locked`, the financial document is not even decoded into memory. On unlock, the document load begins the moment the user initiates authentication (running concurrently with the biometric prompt, never before it and never exposed before it succeeds), and is discarded if auth fails or is cancelled. Because a screen is never constructed before the store is fully populated, screens read `store.planningResult.confidenceTier` etc. directly rather than through a loading/ready wrapper enum; only Settings' import feature gets its own genuine async state (`ImportFlowState`). Storage corruption is a side-channel `financialStoreLoadError` on the coordinator, not a root-level takeover, so a corrupted file never blocks the user's path to Settings/import-backup.

**File/module layout:** `Domain/` (pure engine, record types, `PlanningConfig`, no SwiftUI/persistence imports), `Domain/Validation/DomainValidator.swift`; `Persistence/` (`FinancialDocumentStore`, `SettingsStore`, atomic-write implementations); `Auth/` (`AuthenticatorProtocol` + real/fake implementations); `Entitlement/` (`EntitlementService`); `Services/ImportExport/ImportExportService.swift`; `Services/Formatting/CurrencyFormatting.swift`; `App/` (root coordinator, `AppRootState`, scene lifecycle); `Features/{Dashboard,MoneyInputs,ExplanationHistory,Settings,Onboarding,AppLock}` (view + presentation-only view model per screen); `SharedUI/` (parameterized `EmptyStateView`, hero card, ribbon path, formatting helpers); `Tests/{DomainTests,PersistenceTests,ViewModelTests,SnapshotTests}`.

**Data models (final, canonical):**
- `IncomeEntry { id, amount: Decimal, receivedOn: Date, note: String?, createdAt: Date, isCorrection: Bool (computed from amount < 0) }` — no `updatedAt`; same-date duplicates always distinct records.
- `Obligation { id, name, amount: Decimal, cadence: ObligationCadence, dueDate: Date, createdAt: Date }` — no `endDate`/`isActive`/`updatedAt`.
- `ObligationCadence { weekly, monthly, quarterly, annual, oneOff }`.
- `ReserveSnapshot { amount: Decimal, lastUpdated: Date }` — single current value, never a ledger.
- `ConfidenceTier { insufficientData(manualFallback: Decimal?), lowConfidenceComputed, fullyComputed }`.
- `MonthlyBucket { monthStart: Date, totalIncome: Decimal }` — zero-income months always materialized.
- `RiskWindowFlag { obligationID, obligationName, amount, dueInDays }` as an array on `PlanningResult`.
- `RunwayResult { monthsOfRunway, basedOnReserveAmount, reserveLastUpdated }`, nil when reserve is missing.
- `ExplanationLineItem { id, label, amount, emphasis }`.
- `PlanningConfig { lookbackMonths, lowConfidenceEntryThreshold, fullyComputedEntryThreshold, riskWindowDays, baselinePercentile, currencyCode? }`, `.standard` = 12 months / 2 / 6 / 14 days / 10th percentile.
- `PlanningResult { asOf, safeToSpend (never floored at zero), confidenceTier, baseline, obligationsDueThisCycle, runway?, riskWindowFlags, monthlyIncomeBuckets, explanation }` — never persisted.
- **`FinancialDocument { schemaVersion, incomeEntries, obligations, reserve?, manualFallbackEstimate? }`** — the field added this round; this is the only thing export/import touches, and manual fallback now correctly survives relaunch and travels with a backup, since it's the user's own financial input, not app config.
- `AppSettings { schemaVersion, onboardingCompleted, lockEnabled, entitlementActive }` — deliberately excludes manual fallback; never touched by import/export.

**Persistence strategy:** No SwiftData, no Keychain. Two versioned `Codable` JSON files in Application Support under file protection, both atomically written: `financial-document.json` and `app-settings.json`. Import is replace-only, fully validated in memory (including a new check rejecting a negative `manualFallbackEstimate`, which has no sensible meaning unlike a negative income correction) before any write touches the live document, with a user confirmation step before the swap.

Schema-invalid import leaves the on-disk file byte-identical; cancelled import produces zero writes. Coordinator: `financialStore == nil` while `.locked`; failed/cancelled auth discards any in-flight document load.

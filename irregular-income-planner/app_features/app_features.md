The must-have set should stay tight.

User story: as a first-time user with irregular income, I need to understand what this app calculates, what data it needs, and what happens if I don’t have enough history yet. Acceptance criteria: a new user can complete setup without needing sample finance knowledge; zero/one-entry users land in an explicit insufficient-data state, not a broken dashboard; if reserve is skipped, the app still works and clearly explains that runway is unavailable until reserve is entered.

- Must: income history capture with dated entries, duplicate-date tolerance, and corrective negatives. User story: as a freelancer, I need to log real income events the way they happen so the engine reflects my actual volatility instead of a sanitized ledger. Acceptance criteria: users can add positive and negative income entries with dates and notes; same-day entries remain distinct and are both counted; editing or deleting an entry deterministically recomputes outputs; large currency values format correctly.

- Must: obligation capture with cadence and due dates, but manual amounts only. Acceptance criteria: users can create weekly, monthly, quarterly, annual, and one-off obligations; every obligation has an explicit amount and due timing; the app never predicts a variable obligation; upcoming obligations affect risk-window messaging.

- Must: reserve capture as an optional, separate input with its own empty state. User story: as a user who knows my income history but not my exact current buffer, I need the app to still give me a safe-to-spend number without faking runway. - Must: the planning engine and primary recommendation screen.

**Screens (5 + settings):** Dashboard, Upload, Document Review (the core screen), Ship To Catalog, Prompt Configuration, plus a lightweight Settings panel for API-key/env status.

**Dashboard** — KPI tiles (uploaded/processed/needing review/avg confidence/success rate) + recent-documents table with status pills (Uploaded → OCR Running → Extracting → Needs Review → Approved → Failed). Empty state: friendly zero-state with upload CTA + sample-PDF link, no cutesy illustrations. Loading state: skeleton tiles, not a full-page spinner.

**Upload** — drag-and-drop + multi-file batch, each file gets a 4-dot pipeline stepper (Upload/OCR/Extract/Ready) so stuck/failed documents are visibly diagnosable per-stage, not one opaque bar. Failure state: red dot at the failed stage, inline reason text, stage-scoped retry (not full re-upload).

**Document Review** (highest-risk, highest-value screen) — side-by-side PDF (~55%) + tabbed data panel (Extracted Data / Line Items / Raw OCR / Raw JSON). State managed via a single reducer/context per document (not scattered useState) with actions like FIELD_EDITED, LINE_ITEM_EDITED, SHIP_TO_SELECTED, REPROCESS_REQUESTED. Three baked-in field visual states: unreviewed-low-confidence (amber border + numeric badge, never color-only), user-edited (blue border + "edited" tag, diffable to original), confirmed (neutral). Clicking a field jumps/highlights the matching PDF page. Line items are fully inline-editable with add/delete rows. Ship To match surfaces as a card (best match + confidence + one-line reason + alternates list + manual search fallback). Sticky action bar: Reprocess (warns if unsaved edits exist), Approve (blocks or requires override confirm if needs_review fields remain), Export JSON/CSV.

**Ship To Catalog / Prompt Configuration** — standard searchable table + slide-over editor pattern; prompt config is global + per-customer override with an inherited/diff view and a confirm-gated "re-run affected documents" action.

**Visual system** — light-mode-first, warm neutral palette (not blue-gray/ABBYY-style cold enterprise), one accent color reserved for primary actions, amber/red reserved strictly for confidence/error semantics. System sans + monospace for financial data/codes/raw panels. Compact table density (36–40px rows) for power-user daily use. Motion: near-invisible, 120–200ms eases, subtle highlight-pulse on PDF-linked scroll, no playful bounce.

**Accessibility** — confidence is never color-only (icon + numeric label always present), full keyboard operability on Review (logical tab order, Enter-to-accept, a "jump to next flagged field" shortcut).

## Final Output

**Architecture:** A thin frontend over a business-logic-heavy backend. FastAPI owns the entire pipeline (upload → PDF→image → OCR → LLM extraction → validation → Ship To matching → export); React owns presentation plus exactly one nontrivial piece of client state — the per-document review reducer already locked in design_handoff. No client-side draft persistence (no localStorage/IndexedDB) for MVP: edits live in the reducer's in-memory buffer and are pushed to the server via PATCH endpoints, with a `hasUnsavedChanges` flag gating the reprocess action so a reprocess can't silently clobber unsaved edits.

**Backend module layout:** `app/api/routes/{upload,documents,catalog,prompts,dashboard,export}.py` (thin handlers) → `app/services/{pdf_service,ocr_service,extraction_service,validation_service,matching_service,export_service}.py` (actual pipeline logic, OCR and LLM behind swappable `OCREngine`/`LLMExtractor` protocols) → `app/models/*.py` (SQLAlchemy, the nine required tables) → `app/schemas/*.py` (Pydantic — the literal wire contract with the frontend) → `app/core/config.py` (env-sourced settings).

**Frontend module layout:** `src/pages/*.tsx` as near-logic-free route containers; `src/features/documentReview/{reducer.ts,context.tsx,types.ts}` as the one real piece of client state; `src/api/client.ts` a typed fetch wrapper mirroring the Pydantic schemas; `src/components/` for shared UI (ConfidenceBadge, StatusPill, PipelineStepper, DataTable) so the design system isn't reimplemented per screen.

**Data model highlights:** Every extracted field (header and line-item) is a generic `ExtractionField<T>` (`value, confidence, source_text, page_number, needs_review`) reused everywhere rather than bespoke shapes. `line_items` is its own table (not nested JSON) with `is_user_added`/soft-delete flags since array mutation is structurally different from scalar field edits and shouldn't share reducer code paths. `extraction_results` keeps the raw pre-repair and post-repair JSON plus a `repaired` boolean for the Raw JSON tab. `corrected_results` mirrors the same shape as the user's edited version, enabling the diffable blue "edited" state. `audit_log` gets one row per persisted (not per-keystroke) reducer action.

**Persistence:** SQLite (SQLAlchemy) as system of record; PDFs and page images on a volume-mounted local disk referenced by path from `documents`/`pages`. Matches the "self-hosted, not offline-SPA" reading of local-first already locked in ios_architecture_review.

**Testing:** Backend pytest on PDF/OCR parsing, JSON-schema validation including the repair-retry path, and Ship To fuzzy matching (per the original spec's explicit requirement). Frontend unit tests on the reducer — every action transitions state correctly, including derived `needs_review`/`hasUnsavedChanges`. E2E explicitly deferred past MVP as lower value-per-hour than reducer/backend coverage.

## Final Output

**Milestones (sequenced, with rough sizing):**
1. **Data layer first (0.5 day, hard gate)** — SQLAlchemy models for all nine tables + Pydantic schemas matching tech_specs exactly (`ExtractionField`, `PoHeaderFields`, `LineItem`, `DocumentDetail`, etc.). Nothing else starts until this exists, since every other lane is generated or hand-mirrored from it.
2. **Swappable OCR/LLM protocols, real + fake (1–1.5 days)** — Implement `OCREngine` and `LLMExtractor` twice each: a real implementation (Tesseract, actual LLM client) and a deterministic fixture-driven fake that can simulate the invalid-JSON-then-repair path on demand. This unblocks all backend and frontend testing without live API calls or non-determinism.
3. **Pipeline services wired end-to-end against the fakes (1.5 days)** — upload → pdf_service → ocr_service → extraction_service → validation_service → matching_service, driven by FastAPI `BackgroundTasks`, but with OCR/LLM calls run through a thread pool/executor so a heavy job doesn't starve the HTTP event loop. First integration test lives here: fixture PDF through the fake pipeline, assert correct `DocumentDetail` shape and `needs_review` flags.
4. **API layer with audit logging built in from the start (1 day)** — All endpoints from tech_specs. Audit-log writes are part of each mutating endpoint's definition of done (not a later pass): e.g. `patchDocumentFields` is only "done" when it both returns the updated `DocumentDetail` *and* writes exactly one `FIELD_EDITED` audit row. Explicit behavior tests on `approveDocument` (422 block) and `reprocessDocument` (rejects clobbering unsynced edits without force).
5. **Frontend, all five screens + reducer, built against fixtures (3 days, runs in parallel starting as soon as milestone 1 lands)** — the single biggest chunk of calendar time; sequencing it after the backend instead of alongside it silently adds 3+ days to the schedule for no reason.
6. **Ship To matching + Prompt Configuration (1.5 days)** — sequenced after the core review loop works. Matching tests include deliberately adversarial catalog entries (near-duplicate ship-to records) to prove tie-breaking behaves sensibly, not just happy-path.
7. **Export, audit verification, docker-compose, README, sample PDFs, smoke test (1 day, strictly last)** — real from-scratch `docker compose up` on a clean checkout (Poppler/system-package issues are a common silent-break point), plus a scripted manual run-through of the full MVP success sentence (upload → OCR → extract → edit a low-confidence field → Ship To match/override → approve → export JSON and CSV → confirm audit rows for every step) as the final gate.

**Total estimate:** ~9–10 developer-days if frontend (milestone 5) runs parallel to milestones 3–4; ~12–13 if done strictly serially.

**Dependencies:** models/schemas gate everything → fake OCR/LLM gates pipeline services and all backend tests → pipeline services gate the real API layer → schemas alone (not the live backend) gate frontend start → frontend and Ship-To/prompt-config can proceed in parallel once the review screen's reducer is stable → export/audit/docker is strictly last since it needs the whole app to smoke-test.

**Risk mitigations:** BackgroundTasks-in-same-event-loop scaling limit named explicitly (mitigated via executor/thread pool, documented as an untested-beyond-~10-files boundary rather than solved with new infra); audit-log completeness treated as a per-endpoint acceptance criterion instead of an end-of-project sweep; fuzzy Ship To matching tested against adversarial near-duplicate catalog entries so confident-looking wrong matches don't slip through; Docker/Poppler packaging verified via an actual clean-checkout smoke test rather than assumed.

**Verification plan:** every milestone carries its own test (schema equality, fixture-driven pipeline integration test, endpoint-level behavior + audit-row assertions, reducer unit tests, adversarial matcher tests, docker smoke test), plus one final manual end-to-end run-through of the literal MVP success sentence from the original prompt as the closing gate before calling this done.

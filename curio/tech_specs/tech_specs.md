The other decision I want to nail down now, because it's exactly the kind of thing that quietly rots a save-data design: static content (specimen definitions, expedition destinations, wing layout templates) is bundled JSON loaded once into an immutable `ContentCatalog`, completely separate from `MuseumSnapshot`, which is the only thing that gets a `schemaVersion` and gets written to disk. If a build lane starts stuffing specimen flavor text or destination costs into the save file, we've coupled content updates to save migrations for no reason — the save file should only ever contain instance data (which specimens exist, where they're placed, how much currency, which wings are donated), keyed back to content by string ID.

For the data model itself: `BigNumber` is mantissa/exponent as already locked, with `+`, `*`, a `Comparable` conformance, and a `.formatted()` for display — this is the one type every lane touches, so its exact shape has to be right the first time. Specimens split into two types on purpose: `SpecimenDefinition` (static, from content) and `Specimen` (the owned instance, tracks `definitionID`, `isCataloged`, and a `SpecimenLocation` enum — `.intakeTray`, `.unplacedHolding`, `.placed(wingID:slotID:)` — which is what makes non-destructive placement a type-level guarantee instead of a convention someone has to remember: there's no "delete" path, only location transitions). `Wing` holds `wingType` (an enum with the three committed cases) and an array of `Slot`s whose `SlotPosition` is itself an enum (`.grid`, `.row`, `.tier`) so the three visually-distinct shells aren't secretly sharing one grid shape under the hood — that structural difference in `SlotPosition` is exactly the artifact that proves (or disproves) the rarity-threshold wing isn't a reskin, per the demotion gate carried in from architecture review. `WingScoringEngine` is a protocol with three concrete engines, each pure functions over `(Wing, specimens, catalog) -> BigNumber` plus an `isSetComplete` check — no scoring logic lives in a view body, full stop.

On state, I want to introduce one small generic type that every screen builds on instead of five people inventing five slightly different loading/error patterns: `enum ScreenState<Success> { case loading, empty, success(Success), error(String) }`. Every one of the ten screens' view models exposes a `ScreenState<...>` and the view just switches on it. This is a small thing but it's the difference between "every screen has four states" being an actually-checkable, structurally-enforced requirement versus an honor system that five different builders interpret five different ways.

GameStore itself is `@MainActor @Observable`, lives in CurioCore (Observation is a separate framework from SwiftUI, so this doesn't violate the no-SwiftUI rule), and its mutating methods return result enums rather than throwing for anything a real player can trigger through normal play — `DispatchEligibility.blocked(reason:)`, `PlacementResult.setCompleted(newMultiplier:)` / `.setBroken(newMultiplier:)`, `DonationResult.rejected(reason:)`. Throwing is reserved for truly-should-never-happen programmer errors. This matters for Maya specifically: the UI never has to poll "did a set just complete," it gets told synchronously by the return value and can fire the protected 600ms animation immediately — the moment the whole app is built around can't be allowed to depend on some separate observer noticing a multiplier changed a frame later. Save scheduling lives inside GameStore too (a private debounced save task kicked off by every mutating method, plus `CurioApp` calling `save()` unconditionally on `scenePhase` → `.background`), specifically so "did we remember to save after this action" can't be a bug some screen introduces by forgetting to call something.

Load failure gets modeled as a real tri-state (`loading` / `ready` / `decodeFailed`) on GameStore rather than GameStore just crashing or silently starting fresh — the Overview screen's error state binds directly to `decodeFailed`, and starting a new museum requires an explicit user-confirmed `resetAndStartFresh()` call, never an automatic fallback.

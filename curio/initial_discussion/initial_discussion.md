Here's my first pass on this.

The product promise I'd lock in: Curio is an idle game where the *organizing* is the progression system, not decoration on top of it. You send expeditions (idle timers) that return specimens into an "intake tray" as raw, uncatalogued items; the player's active job is identifying them and then physically arranging them into cabinet/museum displays, and the arrangement itself — matching specimens by region, era, type, or color into deliberate sets — is what generates the multipliers that unlock the next tier of expeditions. If you strip out the curation step and it's still a complete, satisfying idle game, we've built the wrong thing — Cookie Clicker with taxidermy skins. The curation has to be mechanically load-bearing: sets/synergies that pay real multipliers, not a "auto-sort" button that trivializes it in one tap.

Primary user: someone who already plays idle games in short bursts throughout the day (waiting rooms, commute, before bed) and wants one with an aesthetic, "build something I'm proud to look at" payoff — the Two Dots / merge-game crowd overlapping with idle-game players, not hardcore incremental-game min-maxers. That matters for scope: this person opens the app for 30-90 seconds, ten-plus times a day. Every screen has to make sense cold, in a few seconds, with no tutorial re-read.

Core loop as I'd scope it for one pass: generate currency (research/funding) → dispatch expeditions that run on real-world timers → collect specimens into an intake queue → identify/catalog them (quick active step, satisfying on its own) → place them into cabinet slots where adjacency/set rules grant bonuses → bonuses unlock longer/better expeditions and eventually a prestige reset ("donate a wing to the permanent museum" for a lasting multiplier and a fresh cabinet to fill). One museum with expandable wings, not multiple parallel museums — that's the kind of breadth I'd cut first if time runs short, per the "smaller finished app over larger half-built" rule.

The biggest risk to this direction is engineering cost, and I want to name it now rather than discover it in week two: drag-and-drop placement into a grid is significantly more expensive to get right than a tap-to-buy upgrade list, and it directly collides with the accessibility requirement — VoiceOver users can't drag-and-drop reliably, so there has to be a non-drag path (tap a specimen, tap a destination slot, or a picker-based "assign to slot" flow) that's a first-class way to play, not an afterthought bolted on to satisfy an audit. If we underscope that, we ship an app that's only playable by sighted users doing gestures, which fails the accessibility bar outright.

Second risk, idle-game-specific: we have no live users and no telemetry, so balance tuning is a real blind spot. I'd want a debug/dev harness that can fast-forward simulated time (hours, days, a week) so we can sanity-check the offline-progress math and the exponential curves before calling numeric growth "production-ready" rather than just trusting the formula on paper. This is also where the overflow/precision risk from the prior phase's contract actually gets tested, not just assumed.

Hard scope boundaries I'd write down explicitly: one museum, expandable in wings, no parallel museums or biome-switching in v1; no drag-and-drop-only interaction — a non-gesture placement path is mandatory; no server-generated or LLM-generated specimen flavor text at runtime (pre-written content only — avoids an API dependency, cost, and offline-mode contradiction in a local-first app); no live-ops/seasonal event calendar; no trading, no social/leaderboard. Monetization I'd scope as a "Patron" tier that gives real utility — expanded cabinet capacity, faster/parallel expeditions, an exclusive expedition region with its own specimen set and set-bonuses — never a coat of paint on existing slots.

For measurable success criteria, since this isn't shipping to real users for A/B data, I'd define "done" as things we can actually verify in this build: offline-progress math matches the closed-form calculation within a defined epsilon across 1-minute, 6-hour, and 7-day gaps in an automated test; the numeric type used for late-game currency doesn't overflow or silently truncate through at least the first two prestige resets; every screen (intake, cataloging, cabinet/placement, expedition dispatch, prestige, store) has verified empty/loading/success/error states; the non-gesture placement path is tested with VoiceOver on, not just present in code; and the IAP flow has been walked end-to-end in a StoreKit test configuration including the restore-purchases path. If we can't check all of those boxes, I don't think we get to call it production-ready no matter how the core loop feels.

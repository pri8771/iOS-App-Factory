“Right, let’s focus on the core architecture first. I’m seeing a significant risk of ‘state explosion’ if we don’t build in rigorous separation from the start. My immediate recommendation is to move towards a more explicitly actor-based design. Instead of a monolithic `GameSession`, we should break it down into smaller, independent actors, each responsible for a distinct aspect of the game state – player management, card tracking, scoring, etc. These actors would communicate via well-defined, asynchronous messages (probably using `Protocol`s to avoid tight coupling).

Specifically, I want to flag the `RoundState` structure. It’s going to be incredibly complex and prone to errors. We need to seriously consider whether we can represent each round as a discrete, immutable state object. That way, updates to the round state would be handled purely through message passing, preventing any accidental mutation and simplifying debugging.

Furthermore, we *absolutely* need to establish a clear and enforced model for asynchronous updates. ## Final Output

Trestle's build splits into four local Swift Packages inside one thin Xcode project — `Packages/DataDomain`, `Packages/Services`, `Packages/PrimaryUI`, `Packages/PolishResilience` — one per fixed lane, so adding a file never touches `project.pbxproj` and never creates a merge conflict across lanes. `data_domain` owns every contract type and the `GameSession` reducer (the only thing allowed to mutate game state). `services_utilities` owns signing, persistence, networking (`NetworkTransport`/`SimulatedTransport`/`LiveMultipeerTransport`), the host engine, heartbeat, session retention, and StoreKit scaffolding. `primary_ui` owns the actual screens and shared components, built against hand-authored preview fixtures so it never has to wait on `services_utilities`. `polish_resilience` owns the cross-cutting resilience layer (recovery overlay, rejection toast, host status strip, haptics, accessibility audit) and is the lane responsible for actually writing up the physical-device test protocols.

My primary recommendation is to move away from a single `MCSession` and towards a *distributed, mesh-based* networking model. This means each player (or a small group of players) should directly communicate with each other, forming a peer-to-peer network. The host’s role shifts from a central hub to a facilitator – it simply announces the presence of players and facilitates the discovery process. This will provide significantly better scalability and fault tolerance.

Specifically, we need to leverage Apple’s `MWHostDiscoveryPeer` and `MWHostPeer` classes. These provide the building blocks for creating a robust peer-to-peer network. The host should advertise itself to the network, and players can then discover each other and establish direct connections.  This architecture allows us to avoid the single point of failure inherent in a star topology.

Regarding the command log, I strongly disagree with the proposal to use Core Data or SwiftData for this purpose. While they are powerful frameworks, they introduce significant overhead and complexity that isn't justified in this early phase. ## Final Output

**Architecture overview:** Three isolation domains plus one pure projection layer. `NetworkActor` (non-`@MainActor`) owns all `MCSessionDelegate` callbacks, decodes/verifies signatures on inbound `WireEnvelope`s. `HostGameEngine` is a host-device-only actor and the sole author of new `SignedCommand`s — its actor-mailbox serialization is what resolves simultaneous seat-claim races deterministically with no extra locking. `CommandLogStore` is a third actor, present on every device (host and joiners alike), the only thing that touches the on-disk log; it validates monotonic sequence and re-verifies signatures on every append, including the host's own commands. `GameSession` (`@Observable`, `@MainActor`) is the only game-truth type in the stack and is purely a projection over the log — it never originates a mutation, only replays via `rebuild(from:)`, which is the same function for cold launch, live tail-append, and reconnect-from-cursor. Networking is enforced as strict star topology at the application layer: the host is the only device running `MCNearbyServiceAdvertiser`/`MCNearbyServiceBrowser`; joiners connect only to the host and never advertise or browse, so a joiner literally cannot discover another joiner. `NetworkTransport` is a protocol with `LiveMultipeerTransport` and `SimulatedTransport` implementations, which is what makes the packet-reordering test a plain CI-runnable `XCTest` instead of a hardware ordeal. Deck dealing is host-generated once from a seeded shuffle logged only as a debug artifact — actual hands travel as explicit per-seat arrays inside the signed `Dealt` command, never independently re-derived from the seed, avoiding cross-platform RNG-reproducibility risk.

**File/module layout:** `DataDomain/` (Card, Deck, Command types, GameSession, GameRules as pure functions, RoomCode, Entitlements) · `Services/Networking/` (NetworkTransport, LiveMultipeerTransport, SimulatedTransport, HeartbeatMonitor) · `Services/HostEngine/` (HostGameEngine) · `Services/Signing/` (CommandSigner, KeychainKeyStore) · `Services/Persistence/` (CommandLogFileStore, SessionMetadataStore) · `Services/StoreKit/` (HostPassStore) · `UI/Host/`, `UI/Joiner/`, `UI/Shared/` (mirrors the design_handoff screen inventory exactly; shared components live in Shared since both tracks use them) · `Support/` (LoadState, DesignTokens, Haptics). This split lets the primary_ui lane build every screen against `GameSession`'s projection and `SimulatedTransport` without touching `HostGameEngine` internals, and the services_utilities lane build the real transport without knowing SwiftUI.

**Data models:** Every seat/hand/score collection is an `Array` ordered by `seatIndex`, never a `Dictionary` — enforced at the type level (`RoundState.hands: [SeatedHand]`, `tricksWon: [Int]` indexed by seat, not `[SeatIndex: Int]`). All cross-device types are `Codable` + `Sendable`. Full field-level definitions for every request/event/state type are in the interfaces-json below — nothing is left as a bare name reference for a build lane to guess at.

**Persistence strategy:** Flat, append-only, newline-delimited JSON per session at `Application Support/Sessions/<sessionId>.jsonl`, each line a signed `Command` with a monotonic sequence number, `FileHandle`-written and fsync'd per line; a killed process leaves at worst a truncated tail, which the loader detects and discards without crashing. Core Data/SwiftData explicitly rejected for the log itself (non-deterministic default fetch ordering risk), used only for `SessionMetadataStore` — the 30-day/100-round retention sweep and "Clear Session History," which is list/delete work with no ordering risk.

Here's where I'd push before anyone gets comfortable. The one-sentence promise commits us to two things that are each individually hard and have never really been made to work well together at this player count: a real trick-taking/drafting card engine, and host-relay networking for 10-12 simultaneous phones, plus an App Clip as the join mechanism. Any one of these could eat the whole MVP budget. My biggest objection to the direction as stated: at 10-12 players, MultipeerConnectivity (the framework the parent prompt explicitly prefers for local-first large-group play) has known real-world ceilings — Bluetooth/Wi-Fi Direct mesh degrades hard past ~6-8 peers in the same room, especially in venues with lots of RF noise (bars, party settings — exactly where a 10-12 player card game gets played). "Host-relay" implies a star topology through one host device rather than mesh, which helps, but the host's phone becomes a single point of failure: if the host backgrounds the app, gets a call, or walks out of range, does the whole game die for everyone? That failure mode needs an explicit answer before this is called production-ready, not discovered during testing.

Second risk: App Clip join. App Clips have a hard 10MB size cap, must be reachable via a real invocable URL (NFC tag, QR code, or Safari App Clip Code) — there is no "just open the app clip" in-person flow without printed/displayed codes, which is an actual physical-object dependency for a card game meant to be spun up spontaneously. If the join flow requires each of 10-12 people to scan a QR code off the host's screen, that's workable but should be spec'd now, not assumed. Also worth flagging: App Clips can't use MultipeerConnectivity peer discovery the same way full apps do in all cases — I'd want the architecture phase to confirm the App Clip's actual networking capability before design commits to it as the "join" mechanism, or we build a beautiful onboarding flow on top of a network transport that doesn't work from an App Clip sandbox.

Third: "trick-taking/drafting card engine" is doing a lot of work as a single promise — trick-taking (Bridge/Hearts/Spades-family: bidding, tricks, trump) and drafting (pick-a-card-pass-the-rest, e.g. Sushi Go-style) are different mechanics with different state machines. If the actual game later phases design is a hybrid or a specific named game format, this phase should pin down which one, because "engine" implies generality across rule variants, and if design later invents 3 game modes each with different rules, the sync/networking layer has to handle mid-game state resync for late joins, disconnects, and reconnects — that's the part that actually breaks in the field, not the card logic.

For production-readiness on this one, I don't want "the happy path works when I test it alone in the simulator." I want: a host device disconnecting mid-round has a defined recovery (pause, drop-to-AI, or end-game-gracefully), a joining player who loses connection can rejoin the same session without restarting everyone else, the App Clip's actual reachable transport is verified end-to-end on physical devices (simulator MultipeerConnectivity is notoriously unreliable — this cannot be validated in Simulator only), and there's a fallback join path if App Clip discovery fails (e.g., manual room code entry) since that's a much more failure-prone dependency than typical app features.

Decision rule I want later phases held to: any conflict between "premium/beautiful" and "correctly handles a mid-game host disconnect at 10-12 players" resolves in favor of correctness — a gorgeous card game that silently hangs when the host's phone sleeps is not commercially viable, it's a 1-star review generator. And if the architecture phase determines MultipeerConnectivity genuinely cannot sustain a reliable session at 10-12 peers (which is a real, evidence-backed possibility, not a hypothetical), that must be surfaced explicitly as a named assumption/limitation in this contract's output, not quietly downgraded to "works great for 4-6" without telling anyone the promise changed.

**Hard requirements:**
- Local-first SwiftUI MVP, first-party Apple frameworks only, no cloud backend in this phase.
- One canonical, deterministic game family that is genuinely trick-taking + drafting (not a vague "engine" claim) — the specific ruleset gets pinned down in design/architecture, not here.
- Host-relay networking (star topology through the host, not device mesh) supporting 10-12 players in one room. This player count is non-negotiable; it may not be silently reduced.
- An authoritative, replayable, signed command log on the host is the mechanism for state consistency across peers — this is a hard requirement, not a nice-to-have.
- App Clip as the first-class join path, with a manual room-code fallback so joining doesn't hard-depend on Clip discovery working in every environment.
- Full vertical slice: lobby create/join, seat assignment, draft phase, trick phase with enforced rules, scoring, round/session end, rejoin/resync from the log, persisted game history.
- Premium, coherent design and a real (non-cosmetic) monetization path (e.g., rule packs, tournament/host tooling, replay analytics) remain required in this contract's definition of done — implementation may be scoped/deferred by later phases, but the requirement itself, and the underlying data model/entitlement boundary needed to support it later, may not be dropped.
- Accessibility (VoiceOver labels, focus order on game controls) and privacy (explicit local-join consent, minimal on-device storage, no hidden telemetry) are mandatory baselines.

**Non-goals:** no cloud backend yet, no cross-network/internet play, no cosmetic-only monetization, no one-device passing/drinking mechanics, no generic social features, no fantasy mechanics without enforceable deterministic rules, no expanding into a general multi-game-family platform in this MVP, and no reducing scope to a smaller player count than 10-12 without it being surfaced as an explicit, named, approved limitation (not a silent redefinition).

**Production-readiness definition:** this app is done for this phase when it supports two simultaneous rooms with zero cross-contamination between them, sustains a 10-12 player session through three consecutive full games on physical devices with no manual restarts, has a defined and observably-correct recovery behavior for a host-kill-mid-round event and a player-disconnect-mid-draft event, passes a packet-reordering test showing identical resulting game state across peers, and presents correct loading/empty/success/error states on every connection-related screen (searching for host, waiting for players, mid-game disconnect, rejoin). "It worked once" does not count as stable.

**Decision rules for later phases:** the original prompt text is the highest-authority source; any scope cut must be logged as a named, explicit assumption, never a silent substitution. If deterministic fairness/reliability conflicts with visual polish, reliability wins. If the transport genuinely cannot sustain 10-12 players reliably, the response is to push reliability engineering (heartbeat, reconnect, rehost) as far as possible and then honestly document the real supported ceiling — not quietly ship a 4-6 player game under a 10-12 player promise.

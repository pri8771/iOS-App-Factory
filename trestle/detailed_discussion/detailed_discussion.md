## Final Output

**Resolved requirements:**
- Governing architectural rule: no client — including the host's own screen — ever mutates game state directly. Every action (`SeatClaimRequest`, `DraftRequest`, `PlayCardRequest`) is sent to the host; the only thing that changes real game state is a host-issued, signed, sequenced command replayed from the log. The UI may show a transient, purely cosmetic "pending" state on tap (e.g., a card graying out) with zero backing state change, so taps still feel responsive under network latency.
- Draft mechanic is pinned: discard-down only for this slice (each player independently discards to their legal 4-card hand from a 1-card-larger deal) — not a Hearts-style pass. Internally named `DraftDiscards`; UI-labeled "pre-round draft pass (1 card)." Reasoning: a passing draft adds a second, independent cross-player synchronization risk that would confound what a validation failure actually tells you. A true passing-draft ruleset is deferred as a possible future paid variant, not part of this slice.
- Heartbeat/disconnect protocol is a fixed constant: 2-second heartbeat interval, soft "at risk" indicator after 3 missed beats (~6s), hard "peer disconnected — host must act" state after 5 missed beats (~10s), with host choosing continue/skip/abort as a visible, logged command everyone converges on via replay.
- Host device disables idle auto-lock (`isIdleTimerDisabled = true`) for the full session duration, restored on exit, with a persistent "hosting — screen will stay awake" indicator and a tiered battery banner (green >30%, yellow 30-15% with a recommend-closing-background-apps nudge, red <15% forcing an explicit "continue with risk" confirmation).
- Local Network permission is primed with an explainer screen before the system dialog fires ("Trestle needs local network access to find nearby players — nothing leaves your phone"); denial routes to a named `hosting_unavailable_permissions` state with a Settings deep-link, never a dead-end error.
- MCSession encryption is `.required`. `discoveryInfo` carries only non-sensitive metadata (`appVersion`, `protocolVersion`, `serviceNonce`) — never player names, room codes, or anything identity-bearing.
- Room join uses two-factor confirmation: manual code entry plus a host-room identity card (host name, player count, seat-lock status, room emoji/color) shown before seat-lock, so a mis-join from a similar code or a crowded venue is user-correctable rather than silently wrong.
- Local data retention: completed session logs kept for 30 days or 100 rounds, whichever is smaller, with an explicit "Clear Session History" action; no cross-session player identity linkage in this slice.
- All game-state reducers and replay logic must use explicitly ordered structures (arrays with defined tie-break rules) — never `Set`/`Dictionary` iteration — so replay can't silently diverge across devices due to Swift's non-deterministic hash ordering.
- Simultaneous seat claims resolve through the same command-log serialization: first valid `SeatClaimRequest` the host accepts wins; every other simultaneous claimant gets an explicit "seat just taken, pick another" state, never a silent race.

**Edge cases enumerated:** host phone locking/backgrounding mid-round; peer going silent under load (thermal throttling, background app refresh, brief call) versus a genuine graceful leave; room code collisions and near-miss mis-joins at crowded venues; duplicate peer identity / repeated join attempts; two devices with the same display name; packet duplication and out-of-order delivery, including across trick boundaries; a play request arriving after its trick has already resolved due to timeout-driven skip (`move_rejected_stale_trick` vs. `move_rejected_illegal`); simultaneous seat claims on the same open seat; app backgrounding during an active trick; lock-screen interruptions; a peer reconnecting with a stale command cursor; sessions at exactly 10, 11, and 12 players; join attempts beyond room capacity; low-memory/low-battery state changes on host or peer.

**Data and privacy implications:** local-first, no accounts, no telemetry, no server-side truth. Local Network permission is requested with an in-app primer before the system prompt to avoid a jarring first-run interruption in front of a live group. Stored locally: room id, seat map, deck seed, signed command log, per-session local display names — retained on a rolling cap (30 days / 100 rounds) with a user-facing clear-history control, no identity graph or cross-session linkage yet (that's explicitly deferred to a later, separately-consented Host Pass feature if built). Discovery advertisements carry no sensitive data; session traffic is encrypted (`.required`, not `.optional`).

**Risk register:**
- Transport reliability at 10-12 peers in a real room remains the single biggest unproven risk — mitigated by the heartbeat/timeout protocol above, but the actual numbers are provisional pending the fifty-run hardware validation and may need retuning.
- Host is a single point of failure for the whole session (battery, thermal, being the sync authority) — mitigated by the awake-lock/battery-banner requirements and named recovery states, but full host migration remains explicitly out of scope for this slice (a "session can't continue" terminal state is the accepted limitation).
- Replay-state divergence from non-deterministic collection ordering — mitigated by the explicit-ordering rule; needs to be enforced as a coding constraint the architecture/build phase can't skip.
- UI responsiveness under command round-trip latency — mitigated by the cosmetic pending-state pattern; if perceived latency during real testing is worse than expected, that's a build-phase tuning finding, not a scope change.
- Gemma3's recurring cloud-server architecture proposal is named as a rejected alternative, not a live risk to the chosen design — logged for the record, not adopted.

**Final assumptions:** discard-down is the pinned draft mechanic for this slice (passing-draft deferred); heartbeat is 2s/3-miss/5-miss as a tunable-but-fixed starting constant; App Clip remains explicitly out of this slice, arriving next via the shared join-contract adapter; retention is capped local history with manual clear, no cross-session identity; and the one governing rule — no client-side game-state mutation, ever, only requests and replayed signed commands — is the architectural backbone the rest of the build inherits.

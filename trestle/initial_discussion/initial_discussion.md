**Codex (Product Strategist · User Advocate) — Round 1**

Trestle is a referee-grade local multiplayer engine where 10–12 players in one room can instantly join the same trick-taking/drafting card session from one host device via App Clip or manual code and play with deterministic, authoritative, replayable game state that stays in sync across all phones.

Primary user is the in-person host of a game night or social event, with 10–12 friends, family, or teammates already physically together, who needs the table to get to play in under a minute and wants zero ambiguity about whose turn is next, what cards were played, and how points are counted when phones drop, reconnect, or lag.

Core loop is: host creates a room and confirms local-join consent, players join via App Clip/room code and take seats, host locks seating and starts a draft phase, each round deals and drafts then runs trick-by-trick legal-play enforcement, the host resolves each trick and scores deterministically, rounds iterate to session end with clear settlement, then the entire session is persisted as a replayable command log so any reconnecting player can re-sync.

Hard scope boundaries are one concrete game family only (not a general platform), one MVP ruleset with a clear Spades-like trick-taking family plus a simple drafting step, one room transport path per session (host-relay over local Apple peer-to-peer), no cloud backend or internet play, no one-device passing mode, no social graph/feed features, no cosmetic-only monetization, no hidden telemetry, strict VoiceOver/focus states on all game controls, and explicit error/loading/empty/success states on every networking and game-state transition.

Measurable success criteria are hard and observable: sustain 10–12 players in a room and pass first-pass connect-and-echo across real devices at that scale; run three consecutive full games on physical devices without manual app restarts; show correct recovery for host kill mid-round and player disconnect during draft with deterministic state recovery or graceful continuation; pass packet-reordering simulation with identical reconstructed state on all peers; keep two nearby sessions from crossing over or sharing any commands/players; preserve accessibility and privacy baselines; and complete onboarding quickly (target median join flow <30 seconds under normal room conditions), with any scope drop logged as a named approved assumption rather than silently shrinking the promise.

Coming at this fresh: the product promise I'd write down is "Trestle turns any room of 10-12 people into a fair, dispute-proof trick-taking card game in under two minutes, with zero cards, zero scorekeeping, and zero arguments about what was played." I'd design these as genuinely different UI postures, not the same screen with a flag — the host screen has to carry connection/session-health chrome that would just be noise on a joiner's screen.

Core loop, concretely: create room → players join (clip or code) → seats lock → draft phase (each player picks/passes cards under enforced rules) → trick phase repeats (play a legal card, trick resolves, winner leads next) → round scores post → next round or session ends → session summary/history persists locally. The loop that actually needs obsessive design attention is the trick-play turn itself, because it repeats dozens of times per game and it's the one moment where "whose turn is it, what can I legally play, did that just work" has to be instantly legible with zero ambiguity — that's the crux moment, everything else (lobby, join, summary) is scaffolding around it.

The scope boundary I want stated plainly and not softened: this MVP is one deterministic ruleset (Spades-style trick-taking with a lightweight pre-round draft), one physical local session at a time per room, App Clip plus manual-code join, and nothing else — no rule-pack variety, no account system, no cross-network/internet play, no cosmetic customization, no chat/social layer beyond what the game itself requires (seat labels, turn indicators). The temptation in a "card engine" framing is to build toward extensibility features now (multiple rulesets, spectator mode, tournament brackets) — I'd explicitly defer all of that. Extensibility should live in how the data model and command log are shaped, not in how many screens or modes ship in this pass.

The systems-level thing I don't want lost in this phase's output: the hardest edge cases aren't in the trick logic, they're at the seams between states — what does a joiner's screen show in the three seconds after the host taps "start" but before their own device confirms seat lock; what happens if someone's screen shows "your turn" but the packet that would let them act arrives after the host has already timed them out; what does "the host's phone is at 12% battery mid-round" actually look like on everyone else's screen. Those seam states are where trust gets made or lost, and if this phase's output doesn't name them as first-class UI states (not just "handle disconnects" as a bucket), the design phase will treat them as edge-case polish instead of core product surface.

For measurable success criteria I'd anchor on behavior, not vanity metrics: a full 10-12 player game (lobby through settlement) completes without any manual app restart or reconnect intervention; three consecutive such games run back to back on physical devices; a deliberate host-kill mid-round produces a specific, correct, visible recovery state on every other player's screen (not a freeze, not a silent hang); a deliberate player disconnect mid-draft is handled with an explicit, host-controlled decision (pause, skip, or continue) rather than the app just stalling; and two concurrent nearby rooms show zero player or state bleed into each other. If we can't observe all five of those on real hardware, the app isn't done, no matter how polished the trick-play screen looks.

The risk I'd flag against my own framing: by putting so much weight on the "dispute-proof trust engine" angle, there's a real danger the MVP becomes an impressive reliability demo with a thin, forgettable game on top — drafting and trick-taking still need to feel genuinely fun to play round after round, or the reliability spine is protecting an experience nobody wants to repeat. I don't think that's a reason to de-prioritize reliability, but the design phase needs to treat "is this actually satisfying to play three games in a row" as its own measurable question, separate from "did it stay connected."

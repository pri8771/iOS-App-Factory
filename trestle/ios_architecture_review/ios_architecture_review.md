## Final Output

**Architecture spine:** One `NavigationStack` per role (`AppRoute` enum), Host and Joiner as distinct root flows that never mix, sharing one SwiftUI view tree for the card-play screens (Draft, Trick Table, Round Transition, Match End) so host/joiner parity can't silently drift. `isHost` gates chrome only, enforced structurally via a single `HostChromeOverlay` wrapper rather than scattered inline conditionals. State ownership: `GameSession` (`@Observable`, `@MainActor`) is the only source of game truth, purely a projection over an append-only, replay-capable `CommandLogStore`; `@State` is reserved exclusively for view-local, view-owned, transient state — nothing else. Reconnect/replay is not a special-cased code path — it’s the same "discard projection, replay log from cursor, rebuild `GameSession`" logic used for live updates, app launch, and foreground-after-background alike, which is the thing that keeps the two-round reconnect proof honest. A root-level splash/session-resume check (sub-second, distinct from both Create-Room and Enter-Code) covers the ambiguous first frame on relaunch while the app determines whether there’s a resumable session on disk.

**Room code:** 4-character alphanumeric, uppercase letters + digits, with visually confusable characters (0/O, 1/I/L) excluded — enough combination space for reliable two-room distinguishability in a crowded venue, still readable off someone else's screen across a room. Enter Code uses an `.asciiCapable` keyboard with forced auto-capitalization, not a numeric pad.

**Screen-by-screen spec (layout + state matrix):**
- *Create Room:* single-purpose — icon-scale suit glyph, headline, one "Create Room" button. No config (one ruleset, no player-count field).
- *Local Network Permission Primer:* full-screen interstitial (not a sheet), one sentence, one "Continue" button; denial routes to `hosting_unavailable_permissions` with a Settings deep-link.

**SwiftUI architecture recommendation:** One `NavigationStack` per role via an `AppRoute` enum — Host and Joiner are distinct root flows that never mix, sharing a single SwiftUI view tree for the card-play screens (Draft, Trick Table, Round Transition, Match End) so parity can't drift. `isHost` gates chrome only, structurally enforced through one `HostChromeOverlay` wrapper rather than scattered inline conditionals. `GameSession` (`@Observable`, `@MainActor`) is the sole source of game truth, purely a projection over an append-only, replay-capable `CommandLogStore`; `@State` is reserved strictly for view-local, view-owned, transient state (e.g. the cosmetic pending-tap indicator), never game truth. Reconnect/replay is not a special-cased path — it's the same "discard projection, replay from cursor, rebuild `GameSession`" logic used for live updates, cold launch, and foreground-after-background alike. The network layer is isolated in its own actor (not `@MainActor`), since `MCSessionDelegate` callbacks fire on an arbitrary background queue — that actor decodes/verifies/signs commands and only hops to `@MainActor` to hand a validated command to `CommandLogStore`. Networking is abstracted behind a `NetworkTransport` protocol with `LiveMultipeerTransport` and `SimulatedTransport` implementations, which is what makes the reducer and UI tests below possible. A shared `LoadState` enum backs the loading/empty/success/error states already locked per-screen in design_handoff.

**Apple framework choices:** MultipeerConnectivity, with host-relay enforced at the application layer — the host is the only device that runs `MCNearbyServiceAdvertiser`/`MCNearbyServiceBrowser`; every joiner's `MCSession` connects only to the host and never advertises or browses, so a joiner literally cannot discover another joiner. CryptoKit (Curve25519) for command signing. StoreKit2 for the non-consumable Host Pass. Observation framework for `GameSession`. Core Data used only for the session-metadata/retention index, never the command log itself. `os.log`, CoreHaptics. Explicitly not used: `BackgroundTasks` (MultipeerConnectivity doesn't reliably survive backgrounding, so a backgrounded host is treated as a connection failure and routes into the existing host-kill recovery state rather than attempting background session preservation). No ML/AR/LLM anywhere in this MVP — named as correct, not a gap, since nothing in the locked scope has genuine value from it and forcing one in would violate the portfolio's "never forced" guidance while adding an uncontrolled source of non-determinism to a system built around strict determinism.

**Persistence and local data plan:** The command log is a flat, append-only, newline-delimited JSON file per session in Application Support — each line a signed `Command` with a monotonic sequence number, written via `FileHandle` and fsync'd on append, so a killed process leaves at worst a truncated tail that the loader detects (validates each line's parse + signature, stops at the first bad line, never crashes on it). Core Data/SwiftData are explicitly rejected for the log itself, since an ORM's default fetch ordering without an explicit sort descriptor on every query path is exactly the kind of thing that silently reintroduces non-deterministic replay months later. A separate lightweight store (Core Data or a JSON manifest) tracks session metadata for the 30-day/100-round retention sweep and the "Clear Session History" action, since that's list/delete work with no ordering risk.

**Permissions/privacy plan:** `NSLocalNetworkUsageDescription` and `NSBonjourServices` in Info.plist, with the Bonjour service type string pinned now (lowercase, hyphenated, under 15 characters, e.g. `trestle-host`) so it doesn't get improvised mid-build and silently break cross-version discovery later. No location, camera, contacts, or microphone permissions in this MVP. `MCSession` encryption `.required`; `discoveryInfo` carries only non-sensitive metadata. Signing keys live in Keychain, never UserDefaults or a plist. Zero telemetry/analytics SDKs. Local retention capped at 30 days or 100 rounds with an explicit clear action.

**Dependency/license decision:** Zero third-party dependencies. Every capability needed — MultipeerConnectivity, CryptoKit, StoreKit2, Observation, Core Data (index only), os.log, CoreHaptics — is first-party, satisfying the permissive-license policy by having nothing to evaluate.

**Testing implications:** The packet-reordering/replay-determinism requirement is a pure `XCTest` reducer test — feed `CommandLogStore` a captured, deliberately shuffled/duplicated command array, assert the resulting `GameSession` is byte-identical to the canonical run — fully CI-runnable with no hardware, made possible by the `NetworkTransport`/`SimulatedTransport` split, which also drives XCUITest flows (draft screen, trick-table legality, rejection copy) without touching real MC. Real `MCSession` discovery, connection, disconnect-recovery, and two-room-isolation behavior is physical-device-only, permanently — named as a fixed limit of this project's test strategy, not a gap expected to close later. The very first build deliverable this phase hands to tech_specs is a bare API-level connection-count probe (11+ joiners into one host session, sustained five minutes, no game logic) to settle whether 10-12 peers actually holds before any trick-resolution code gets written. StoreKit entitlement logic (grant on purchase, grant on restore, default-locked behavior, and that the free game loop behaves identically whether the entitlement check has resolved or not) is verified via a local `.storekit` configuration file with StoreKitTest, in CI-runnable unit tests with no sandbox account needed. `XCUIApplication().performAccessibilityAudit()` is folded into CI on the core screens to catch missing VoiceOver labels cheaply before they consume physical-device test time.

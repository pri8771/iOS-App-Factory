The simplest architecture that still survives this app’s actual failure modes is a four-lane split with hard boundaries: pure data models and business rules in `data_domain`, SwiftUI screens and view composition in `primary_ui`, device/framework adapters in `services_utilities`, and cross-cutting validation/recovery/reporting behavior in `polish_resilience`. ## Final Output

**Architecture overview:** Four hard-bounded lanes — `data_domain` (persisted models + stateless business-rule types, no hardware/framework awareness), `primary_ui` (SwiftUI screens/components, consumes protocols and snapshot structs only), `services_utilities` (camera, AR, photo storage, entitlement, report rendering — all behind protocols), `polish_resilience` (formatting, report-snapshot assembly, cross-cutting validation/recovery). Hard rule: persisted entities never own hardware logic, and views never talk to StoreKit/AVFoundation/ARKit/PDFKit directly — everything crosses a protocol boundary. This is consistent with, and a direct build-out of, the `@Query`-direct-except-Capture/Export state ownership already locked in `ios_architecture_review`.

**File/module layout:** One app target, one unit-test target, one UI-test target. `App/` (app entry, scene wiring, DI container, entitlement service bootstrap), `Domain/` (models, enums, `ProjectPolicy`, `IssueFilter`, `MeasurementFormatter`, `ReportSnapshotBuilder`), `Data/` (SwiftData schema, migration plan, repository protocols + SwiftData implementations), `Services/` (`CameraService`, `ARMeasurementService`, `PhotoAssetStore`, `ReportRenderer`, `EntitlementService`, `StorageHealthService`), `UI/` (per-screen folders: Library, Workspace, Capture, IssueDetail, Export, Paywall, Settings, plus shared components), `Support/` (logging, error types, environment keys, test fixtures).

**Data models:** `Project` (id, name, siteLabel?, timestamps, lifecycle active/archived, accessState normal/lockedReadOnlyDueToPlan), `Area` (id, projectID, name, timestamps, lifecycle, displayOrder — duplicate names allowed), `Issue` (id, projectID, areaID, timestamps, capturedAt, severity, status, note?, one `Measurement`, `photoState` explicitly pending/ready(paths)/missingOrCorrupted — no separate draft model), `Measurement` (value, unit, source manual/ar, optional arConfidence state), `BusinessProfile` and `AppSettings` as small persisted singletons. This is the concrete shape of everything `detailed_discussion` and `app_features` already required.

**Persistence strategy:** SwiftData for structured records, photo binaries as files in Application Support. Capture creates the `Issue` immediately at `photoState = .pending`; original image write happens off-main (temp file → downsample/compress → strip EXIF GPS → atomic move → optional thumbnail), then the `Issue` is patched to `.ready` on the main actor, or `.missingOrCorrupted` with a typed reason on failure. `ModelContext` access stays main-actor bound; anything crossing to a background task (export, rendering) is copied into plain `Sendable` snapshot structs first — never SwiftData model objects across the actor boundary.

**Testing strategy:** Unit tests for cap enforcement, downgrade locking, archive rules, duplicate-area handling, measurement formatting, filter predicates, report grouping/order. Integration tests against an in-memory `ModelContainer` plus real temp directories for create→capture→relaunch survival, pending→ready photo transition, corrupted-photo degradation, and schema-v1 migration. UI tests for first-run creation, repeated manual capture, permission-denied/interrupted camera, over-cap paywall, locked read-only project, export preview with placeholder, and settings-driven unit-display changes. Camera and AR verification is explicitly protocol-simulated (unsupported/unstable/success, injected photos) — real measurement tolerance is not claimed in this environment, consistent with every prior phase's stated limitation.

## Final Output

**Architecture overview:** Four hard-bounded lanes — `data_domain` (persisted models + stateless business-rule types, no hardware/framework awareness), `primary_ui` (SwiftUI screens/components, consumes protocols and snapshot structs only), `services_utilities` (camera, AR, photo storage, entitlement, report rendering — all behind protocols), `polish_resilience` (formatting, report-snapshot assembly, cross-cutting validation/recovery). Hard rule: persisted entities never own hardware logic, and views never talk to StoreKit/AVFoundation/ARKit/PDFKit directly — everything crosses a protocol boundary. This is consistent with, and a direct build-out of, the `@Query`-direct-except-Capture/Export state ownership already locked in `ios_architecture_review`.

**File/module layout:** One app target, one unit-test target, one UI-test target. `App/` (app entry, scene wiring, DI container, entitlement service bootstrap), `Domain/` (models, enums, `ProjectPolicy`, `IssueFilter`, `MeasurementFormatter`, `ReportSnapshotBuilder`), `Data/` (SwiftData schema, migration plan, repository protocols + SwiftData implementations), `Services/` (`CameraService`, `ARMeasurementService`, `PhotoAssetStore`, `ReportRenderer`, `EntitlementService`, `StorageHealthService`), `UI/` (per-screen folders: Library, Workspace, Capture, IssueDetail, Export, Paywall, Settings, plus shared components), `Support/` (logging, error types, environment keys, test fixtures).

**Data models:** `Project` (id, name, siteLabel?, timestamps, lifecycle active/archived, accessState normal/lockedReadOnlyDueToPlan), `Area` (id, projectID, name, timestamps, lifecycle, displayOrder — duplicate names allowed), `Issue` (id, projectID, areaID, timestamps, capturedAt, severity, status, note?, one `Measurement`, `photoState` explicitly pending/ready(paths)/missingOrCorrupted — no separate draft model), `Measurement` (value, unit, source manual/ar, optional arConfidence state), `BusinessProfile` and `AppSettings` as small persisted singletons. This is the concrete shape of everything `detailed_discussion` and `app_features` already required.

**Persistence strategy:** SwiftData for structured records, photo binaries as files in Application Support. Capture creates the `Issue` immediately at `photoState = .pending`; original image write happens off-main (temp file → downsample/compress → strip EXIF GPS → atomic move → optional thumbnail), then the `Issue` is patched to `.ready` on the main actor, or `.missingOrCorrupted` with a typed reason on failure. `ModelContext` access stays main-actor bound; anything crossing to a background task (export, rendering) is copied into plain `Sendable` snapshot structs first — never SwiftData model objects across the actor boundary.

**Testing strategy:** Unit tests for cap enforcement, downgrade locking, archive rules, duplicate-area handling, measurement formatting, filter predicates, report grouping/order. Integration tests against an in-memory `ModelContainer` plus real temp directories for create→capture→relaunch survival, pending→ready photo transition, corrupted-photo degradation, and schema-v1 migration. UI tests for first-run creation, repeated manual capture, permission-denied/interrupted camera, over-cap paywall, locked read-only project, export preview with placeholder, and settings-driven unit-display changes. Camera and AR verification is explicitly protocol-simulated (unsupported/unstable/success, injected photos) — real measurement tolerance is not claimed in this environment, consistent with every prior phase's stated limitation.

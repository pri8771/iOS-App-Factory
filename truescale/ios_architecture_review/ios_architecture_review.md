## Final Output

**SwiftUI architecture:** SwiftUI + SwiftData, iOS 17+. Persistence models (`Project`, `Area`, `Issue`, `BusinessProfile`/`AppSettings`) stay pure data; business rules (active-project cap enforcement, downgrade-lock selection, archive/cascade rules, filter predicates, report assembly, measurement formatting) live in separate domain service types, never scattered into model extensions or views. State ownership is split deliberately: Library, Workspace (Areas/Issues segmented), and Issue Detail read directly off `@Query` and derived predicates with no interposed coordinator — SwiftData is already the observable source of truth, and wrapping it in a cache invites exactly the "UI shows something the store disagrees with" bug class. `@Observable`, `@MainActor` view models are reserved for the two screens with real non-persisted process state: Capture (owns `AVCaptureSession` lifecycle, interruption notifications, low-storage checks, AR session/tracking state) and Export (owns snapshotting, async render progress, share-ready state). Entitlement state lives in one app-scoped, launch-started service — not a screen VM — fed by a long-lived `Transaction.updates` listener plus a `scenePhase == .active` recheck, queried by domain services only at safe enforcement points (project creation, foreground).

**Apple framework choices:** SwiftUI, Observation, SwiftData, AVFoundation (custom `AVCaptureSession`/`AVCaptureVideoPreviewLayer`, photo-only — no audio input), ARKit using `ARWorldTrackingConfiguration` with `sceneReconstruction = .mesh` on LiDAR devices and raycasting for point-to-point distance (explicitly **not** RoomPlan, which is the wrong abstraction for "one measurement on one photo" vs. scan-to-floorplan), PDFKit/Core Graphics for report generation, StoreKit 2 for subscriptions/restore. CoreLocation and Photos are not integrated in this build at all (see permissions below). App Intents remain a could-have, not built now.

**Persistence and local data plan:** Photos are never SwiftData blobs — written as files in Application Support, referenced by path, downsampled and EXIF-GPS-stripped on ingest. Photo writes are atomic: write to a temp file, then move into place; the Issue record's photo-reference field starts nil/pending at insert time (record exists the instant shutter succeeds) and gets patched to the final filename in the same main-actor write once off-main compression completes — this is a real, named transient state, not just an implementation detail. Async report export must snapshot everything it needs (issue fields, photo paths, groupings) into plain `Sendable` structs on the main actor before handing work to an off-main rendering task; SwiftData model objects never cross the actor boundary. Schema built CloudKit-ready from day one — every relationship optional, every attribute optional-or-defaulted, no uniqueness constraints — even though CloudKit itself stays off. A `VersionedSchema`/`SchemaMigrationPlan` pattern is stood up now, with a migration test in place from the first schema version, so later additions (AR fields, entitlement cache fields) have a tested path to follow.

**Permissions/privacy plan:** Camera permission is the only permission requested in this build. Denied and temporarily-unavailable are distinct, separately handled states, and neither blocks issue-record creation. Location: no `NSLocationWhenInUseUsageDescription`, no CoreLocation code path, at all — cut entirely rather than left "optional/contextual," since an unused permission string is both an App Review flag and a trust hit for a privacy-sensitive user base, and it directly contradicts the "everything stays on device, nothing unnecessary" position the product is selling. Photo-library and microphone permissions/capabilities are likewise fully absent from the target — no accidental audio input on the capture session, no photo-import path. Location stamping, if ever built later, gets its permission string added together with that feature as its own explicit, consented addition — never pre-provisioned.

**Dependency/license decision:** Zero third-party packages. Apple's own frameworks cover persistence, camera, AR, PDF, subscriptions, and testing completely for this scope; any future proposal would need to clear a real bar (material risk reduction, permissive license, no lock-in around core data/subscriptions) to be considered at all.

**Testing implications:** Three tiers — unit tests for domain rules (cap enforcement, lock/downgrade transitions, archive/cascade-delete-blocked-on-non-empty-area, filter predicates, report ordering, measurement formatting, missing-photo degradation), integration tests against an in-memory `ModelContainer` and real file I/O (create→relaunch→verify survival, corrupted/missing photo still exports with a placeholder, migration test), and UI tests for the critical paths and failure paths (first-run creation, repeated capture, permission-denied camera, over-cap paywall, restore flow, locked-project read-only state). Camera and AR are both isolated behind protocols (`CameraCapturing`, `MeasurementProviding`) specifically so all three AR states (unsupported/unstable/success) and injected photos can be tested without a physical device or LiDAR sensor — stated plainly, this proves correct gating and fallback behavior, not real-world measurement accuracy, since no physical LiDAR hardware exists in this build loop. `PhotoThumbnailView` needs three states, not two — loaded, pending-processing, and missing/corrupted — and that pending state needs its own test coverage so the normal every-capture compression window is never mistaken for a failure.

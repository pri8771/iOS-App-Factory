**Claude (Frontend Engineer · Systems Thinker) — Round 1**

Kicking off my pass by pulling together the actual buildable handoff — screens, tokens, components, state wiring — from an engineering-architecture angle rather than re-litigating things already locked.

The single biggest thing nobody's named yet in concrete terms: "camera-first, live viewfinder is what you see" is not achievable with `UIImagePickerController` or `PhotosPicker`'s camera source — those present a system camera as a modal sheet with its own chrome, and you cannot overlay severity/status chips on top of that feed or keep the same screen identity across the shutter press. To get the locked interaction (viewfinder → tap shutter → same screen becomes the post-capture editing state with chips sliding in) requires a custom `AVCaptureSession`-backed camera view with a `AVCaptureVideoPreviewLayer`, manual `AVCaptureDevice.authorizationStatus` checks (not the picker's built-in prompt), and a `AVCapturePhotoOutput` capture pipeline. That's a real build-cost item, but it also hands us something valuable for free: `AVCaptureSessionWasInterrupted` / `InterruptionEnded` notifications map directly and cleanly onto the already-locked "camera-temporarily-unavailable" state (another app grabbed the camera, a phone call started, etc.), distinct from `.denied`/`.restricted` authorization status mapping to "permission-denied." I want that pairing preserved explicitly in the handoff so whoever builds Capture doesn't reach for the picker API, get 80% of the way there, and then discover the interaction model doesn't fit.

On state architecture: I'd model this as SwiftData `@Model` classes for Project, Area, and Issue, each with stable UUID identity and explicit `createdAt`/`updatedAt` (already required from the schema-evolution decision), fed into views via `@Query` with predicates for the Areas/Issues segmented filtering rather than filtering an in-memory array — at low-hundreds-of-issues scale this matters for the "stay responsive" acceptance bar already locked, since SwiftData predicates push filtering to the store instead of loading everything into memory and filtering client-side. Severity/status as fixed enums need to be stored as `Int`/`String` raw values with a `Codable`, `RawRepresentable` wrapper so `@Query` predicates against them stay index-friendly — an easy trap is making them `Codable` structs-in-disguise that SwiftData can't predicate on directly, which quietly breaks filter performance at scale. Photos should never live as `Data` blobs inside the SwiftData store — write each photo to a file in Application Support named by the issue's UUID, store only the filename/relative path in the model, and generate a downsampled thumbnail (via `UIImage.byPreparingThumbnail`, off-main by default) for list/card rendering while the full-res file backs Issue Detail and PDF export. This is where the day-one photo-compression requirement actually gets implemented — a fixed long-edge max (I'd propose 2048px for stored originals, thumbnails at ~400px) and a JPEG quality target (~0.7) applied at write time, with EXIF GPS stripped in the same write pass using `CGImageDestination` metadata options, not as a separate pass someone might forget to wire up later.

Incremental persistence — the "draft and saved state are the same thing" rule — should be implemented as: the Issue record is inserted into the model context the instant `AVCapturePhotoOutput` returns a successful photo, then every subsequent chip tap or measurement entry calls `try? modelContext.save()` immediately, no debounce, since those are atomic instant actions. The one field that should debounce is the free-text note — save-on-every-keystroke is wasteful and can visibly stutter typing at real field-use typing speed, so debounce that one field write by ~400ms while everything else stays instant. I'd flag this as a concrete rule worth preserving: instant-write for every discrete control, debounced-write only for free text.

On the visual system, since the brief explicitly bans adjectives-only output: background base near-black graphite (#121417), one level up for elevated surfaces/cards (#1C1F24), a hairline divider color at low opacity rather than drop shadows for card separation — flat instrument-panel surfaces, not soft iOS card elevation, with the one exception being a subtle shadow reserved only for the floating capture button so the single primary action reads as physically raised above everything else.

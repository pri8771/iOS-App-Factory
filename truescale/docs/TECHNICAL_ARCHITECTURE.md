## Design Discussion

This one closed clean. Claude conceded the screen-boundary fork and folded Issues into Workspace as a segmented control (Areas/Issues) rather than a separate pushed screen — agreeing Codex's version genuinely solves the layout problem better (different internal layouts per segment, no back-button cost, no scroll-position loss). That also resolved the second open thread: Workspace survives as its own screen since the segmented design is exactly what keeps it from turning into a dashboard. Claude even ran the tap math to confirm the ~4-tap bar still roughly holds with Workspace in the loop, and added two concrete rules worth keeping — the segmented control must always default back to Areas on entry (never remember you were last in Issues, or a returning user gets dumped into a filtered list instead of a capture-ready screen), and the empty-filtered-result state ("no urgent items in Basement" + clear-filters button) has to be built with the same care now that it's a segment rather than a whole screen.

## Final Output

**Screen inventory (8 screens + sheets):** Library, Project Workspace (with Areas/Issues as a segmented control — not two separate screens), Capture, Issue Detail/Edit, Export/Report Preview, Paywall/Upgrade, Settings, and first-run (a "create your first project" flow rather than a blank library). Sheets/overlays: add-area, filters (live inside the Issues segment), restore purchases.

**Primary user flow:** First launch skips the Library entirely and drops the user straight into naming a project, landing them in Workspace with a default area already created. Repeated field use is Library → Workspace (Areas segment, last-used area pre-selected, capture button already prominent) → Capture → Capture → Capture (camera live immediately, no intermediate button, "saved" is a fading checkmark not a screen transition) → back to Workspace when switching areas → Issues segment (one tap, same screen, no navigation push) when it's time to sanity-check → Export Preview when it's time to send. Roughly a 4-5 tap path from cold-open to a saved first issue, consistent with the tap-count bar from `initial_discussion`.

**Why Workspace stays a separate screen from Capture, and why Issues is a segment not its own screen:** Workspace and Capture serve two different tempos — Workspace is for orienting (what area am I in, what's already logged here, do I need a new area) and Capture is for one focused action (document this issue right now). Fusing them would overload Capture with project-structure concerns and slow down the one screen that most needs to stay fast. Issues, on the other hand, doesn't need its own screen — it needs its own layout. As a segment of Workspace, Areas and Issues can each have completely different internal designs (area cards + pinned capture button vs. filter chips + flat list, no capture button) while switching between them costs one tap with no navigation-stack overhead, no back-button chrome, and no lost scroll position — something a separately pushed Review screen couldn't offer without extra cost.

**Concrete rules to carry into build:** the Areas/Issues segmented control must always reset to Areas on every fresh entry into Workspace (never remember the last-viewed segment) — a user returning to capture a new issue should never land on a stale filtered list from yesterday's review. The empty-filtered-result state ("no urgent items in Basement" + one-tap clear-filters) needs the same explicit design attention it would have gotten as a standalone screen; folding it into a segment isn't license to build it as an afterthought.

Everything else from round one — the full screen/state model, camera-first capture with a live viewfinder, the unified manual/AR measurement slot, dark high-contrast "instrument panel" visual language, paywall triggered only at cap-hit or from Settings, and the accessibility requirements (Dynamic Type without truncating chips, real VoiceOver labels, AA contrast, glove-sized bottom-third touch targets) — carries forward unchanged and undisputed.

## Final Output

**Decision on external design:** No external Claude Design pass. This handoff is the authoritative design system and goes directly into SwiftUI implementation in the next phase — introducing a parallel static-mockup pass would create two sources of truth that are nearly guaranteed to disagree on exactly the details that matter most here (embedded live camera vs. a system picker, the single internally-branching MeasurementField vs. a visually-split manual/AR control, Workspace's segmented layout). Codex's Claude Design prompt is preserved as a deferred, optional artifact for a different purpose — App Store screenshots or marketing collateral, generated later against this same locked token/component system, never as a design input to the build itself.

**Screen inventory (8 screens):** Library, first-run project creation, Workspace (Areas/Issues segmented control, always resets to Areas on fresh entry), Capture (full-screen, presented via `.fullScreenCover`, embedded live camera via custom `AVCaptureSession`/`AVCaptureVideoPreviewLayer`, never a system picker), Issue Detail, Export/Report Preview, Paywall/Upgrade, Settings. Sheets: add-area, filters (inline in the Issues segment), restore purchases, the measurement-input sheet (one entry point, internally branching between manual keypad and AR-capture content — never two separate sheet types), upgrade detail. Destructive confirmations (delete issue, archive with data) use native `.confirmationDialog`/`.alert`, not a custom sheet, since native chrome itself signals "this is a serious action" better than in-house branding would.

**Navigation:** Single root `NavigationStack` for Library → Workspace → Issue Detail. Capture is a `.fullScreenCover` from Workspace, dismissing straight back to an already-updated Workspace via `@Query`. Export, Paywall, Settings, add-area, and restore-purchases are sheets, not pushes, since they're occasional self-contained flows.

**Design tokens (canonical, merged):**
- Surfaces: base background `#121417` (graphite near-black), elevated panel `#1C1F24`, pressed/active panel `#252A31`, divider `rgba(255,255,255,0.08)`.
- Text: primary warm off-white `#F3F0EA`, secondary cool steel blue-gray `#9AA1AC`, locked/disabled `#6F7782`. The warm/cool split between primary and secondary text is deliberate — it separates the two tiers by temperature as well as brightness, which reads faster in glare.
- Color roles (each role means one thing, never doubles up): interactive accent amber `#FFB020` (buttons, active chip/segment state, focus rings — chrome only, never severity); AR/precision accent cyan `#4FD1E5` (AR availability, measurement-source badges only); urgent deep red `#E4572E`; moderate-severity gets its own adjacent-but-distinct muted orange (visually kin to amber but never identical, so "tappable" and "moderate severity" can't be confused at a glance); success/resolved muted green `#5FA777`.
- Spacing: 4/8/12/16/24/32, with 48 reserved as an occasional outer screen-margin exception, not a rhythm step.
- Corners: full-pill for chips, 12pt for cards/inline panels, 16pt for sheets.
- Touch targets: 44pt absolute floor; everyday interactive rows (chips, list rows, settings rows) at 54–56pt; the shutter/primary capture confirm at 72pt diameter, sized to be hittable one-handed, gloved, without looking.
- Motion: segment crossfade ~180ms; chip/select state ~120–150ms; sheet open/close ~220–260ms restrained spring; saved-checkmark cycle appear 120ms / hold 300ms / dismiss 150ms (~570ms total, never blocking — the user can already tap the next shutter press mid-dismiss). Reduce Motion collapses everything to instant-or-fade, and the shutter's capture flash skips entirely rather than just shortening.

**Component inventory:** SeverityChip / StatusChip (icon+color+label, single-select), MeasurementField (one component, internal enum-driven source state — manual vs. AR is never two views), AreaCard, IssueRow, PhotoThumbnailView (shared loaded/loading/missing-placeholder rendering used identically by Issue Detail and Export so they can't drift apart), EmptyStateView (shared pattern across Library-empty, no-issues-yet, empty-filtered-result), PermissionDeniedStateView / CameraUnavailableStateView (visually distinct from each other, shared base layout), LockedProjectBadge, ReportPagePreview, bottom-anchored primary action bar, inline degraded-content notice banner.

**State model:** Fully specified per screen from `design_discussion` and reaffirmed here — Capture covers camera-live, permission-denied, temporarily-unavailable, low-storage-warning, post-shutter-editing, and AR's three states (unsupported/unstable/success); Workspace covers normal, no-issues-yet, archived-only, read-only-locked; Library covers first-run, populated, archived-expanded, over-cap; Export covers generating, preview-ready, partial-degradation, failure. Incremental persistence: the Issue record is created the instant photo capture succeeds, every discrete control write (chips, measurement) saves immediately, only the free-text note debounces (~400ms) — there is no separate draft state.

**Accessibility:** Dynamic Type never truncates chip labels (chips wrap if needed); full semantic VoiceOver labels ("Severity, urgent, selected," "Measurement source, manual"); AA contrast verified against the actual dark palette above, especially secondary/locked text; bottom-third primary controls reachable one-handed on current iPhone sizes; no state communicated by color alone.

This closes the design_handoff phase — a SwiftUI builder can go from this document directly to implementation with no hidden decisions left open.

## Ios Architecture Review

Round two closed both open items clean, and even surfaced a real third bug in the process.

Codex fully adopted Claude's narrower view-model boundary: no coordinator layer for Library/Workspace/Issues since they're just `@Query` projections, and `@Observable` view models reserved for Capture and Export only — plus he added a third piece, a single long-lived app-scoped entitlement service (fed by `Transaction.updates` and `scenePhase`), rather than a per-screen VM. Claude backed this with the concrete failure mode it prevents: a coordinator caching a snapshot of `@Query` results can go stale relative to the actual store (e.g., archive a project, `@Query` updates instantly, but a cached coordinator array doesn't), which is exactly the kind of bug that passes a demo and fails in the field.

On location, both landed on the same hard line: zero location code and zero `NSLocationWhenInUseUsageDescription` in this build — not "optional, contextual," a full cut. Codex explicitly noted this stays consistent with the privacy story he was already selling ("everything stays on device"), and Claude added the falsifiability angle (you can grep Info.plist and confirm the string's absence, unlike a vague "optional" framing). Same explicit cut extended to photo-library and microphone permissions/capabilities — none of that should exist in the target at all.

Claude also caught a genuine gap neither round-one description fully resolved: the two of them described the capture→compress→save flow slightly differently (placeholder-then-patch vs. temp-file-then-atomic-move), and reconciling them surfaces a real transient state — between shutter success and compression finishing, the Issue record exists and is visible in `@Query` before its photo file does. That means `PhotoThumbnailView` needs a third named state (pending-processing) distinct from missing-or-corrupted, or every single new capture will flash a "photo unavailable" placeholder for a few hundred milliseconds. Codex didn't get a chance to respond to this last addition, but it's a direct, well-reasoned consequence of architecture both of them already locked, not a new fork — nothing in it contradicts anything Codex has said.

CONSENSUS: YES

## Final Output

**SwiftUI architecture:** SwiftUI + SwiftData, iOS 17+. Persistence models (`Project`, `Area`, `Issue`, `BusinessProfile`/`AppSettings`) stay pure data; business rules (active-project cap enforcement, downgrade-lock selection, archive/cascade rules, filter predicates, report assembly, measurement formatting) live in separate domain service types, never scattered into model extensions or views. State ownership is split deliberately: Library, Workspace (Areas/Issues segmented), and Issue Detail read directly off `@Query` and derived predicates with no interposed coordinator — SwiftData is already the observable source of truth, and wrapping it in a cache invites exactly the "UI shows something the store disagrees with" bug class. `@Observable`, `@MainActor` view models are reserved for the two screens with real non-persisted process state: Capture (owns `AVCaptureSession` lifecycle, interruption notifications, low-storage checks, AR session/tracking state) and Export (owns snapshotting, async render progress, share-ready state). Entitlement state lives in one app-scoped, launch-started service — not a screen VM — fed by a long-lived `Transaction.updates` listener plus a `scenePhase == .active` recheck, queried by domain services only at safe enforcement points (project creation, foreground).

**Apple framework choices:** SwiftUI, Observation, SwiftData, AVFoundation (custom `AVCaptureSession`/`AVCaptureVideoPreviewLayer`, photo-only — no audio input), ARKit using `ARWorldTrackingConfiguration` with `sceneReconstruction = .mesh` on LiDAR devices and raycasting for point-to-point distance (explicitly **not** RoomPlan, which is the wrong abstraction for "one measurement on one photo" vs. scan-to-floorplan), PDFKit/Core Graphics for report generation, StoreKit 2 for subscriptions/restore. CoreLocation and Photos are not integrated in this build at all (see permissions below). App Intents remain a could-have, not built now.

**Persistence and local data plan:** Photos are never SwiftData blobs — written as files in Application Support, referenced by path, downsampled and EXIF-GPS-stripped on ingest. Photo writes are atomic: write to a temp file, then move into place; the Issue record's photo-reference field starts nil/pending at insert time (record exists the instant shutter succeeds) and gets patched to the final filename in the same main-actor write once off-main compression completes — this is a real, named transient state, not just an implementation detail. Async report export must snapshot everything it needs (issue fields, photo paths, groupings) into plain `Sendable` structs on the main actor before handing work to an off-main rendering task; SwiftData model objects never cross the actor boundary. Schema built CloudKit-ready from day one — every relationship optional, every attribute optional-or-defaulted, no uniqueness constraints — even though CloudKit itself stays off. A `VersionedSchema`/`SchemaMigrationPlan` pattern is stood up now, with a migration test in place from the first schema version, so later additions (AR fields, entitlement cache fields) have a tested path to follow.

**Permissions/privacy plan:** Camera permission is the only permission requested in this build. Denied and temporarily-unavailable are distinct, separately handled states, and neither blocks issue-record creation. Location: no `NSLocationWhenInUseUsageDescription`, no CoreLocation code path, at all — cut entirely rather than left "optional/contextual," since an unused permission string is both an App Review flag and a trust hit for a privacy-sensitive user base, and it directly contradicts the "everything stays on device, nothing unnecessary" position the product is selling. Photo-library and microphone permissions/capabilities are likewise fully absent from the target — no accidental audio input on the capture session, no photo-import path. Location stamping, if ever built later, gets its permission string added together with that feature as its own explicit, consented addition — never pre-provisioned.

**Dependency/license decision:** Zero third-party packages. Apple's own frameworks cover persistence, camera, AR, PDF, subscriptions, and testing completely for this scope; any future proposal would need to clear a real bar (material risk reduction, permissive license, no lock-in around core data/subscriptions) to be considered at all.

**Testing implications:** Three tiers — unit tests for domain rules (cap enforcement, lock/downgrade transitions, archive/cascade-delete-blocked-on-non-empty-area, filter predicates, report ordering, measurement formatting, missing-photo degradation), integration tests against an in-memory `ModelContainer` and real file I/O (create→relaunch→verify survival, corrupted/missing photo still exports with a placeholder, migration test), and UI tests for the critical paths and failure paths (first-run creation, repeated capture, permission-denied camera, over-cap paywall, restore flow, locked-project read-only state). Camera and AR are both isolated behind protocols (`CameraCapturing`, `MeasurementProviding`) specifically so all three AR states (unsupported/unstable/success) and injected photos can be tested without a physical device or LiDAR sensor — stated plainly, this proves correct gating and fallback behavior, not real-world measurement accuracy, since no physical LiDAR hardware exists in this build loop. `PhotoThumbnailView` needs three states, not two — loaded, pending-processing, and missing/corrupted — and that pending state needs its own test coverage so the normal every-capture compression window is never mistaken for a failure.

## Final Output

**Architecture overview:** Four hard-bounded lanes — `data_domain` (persisted models + stateless business-rule types, no hardware/framework awareness), `primary_ui` (SwiftUI screens/components, consumes protocols and snapshot structs only), `services_utilities` (camera, AR, photo storage, entitlement, report rendering — all behind protocols), `polish_resilience` (formatting, report-snapshot assembly, cross-cutting validation/recovery). Hard rule: persisted entities never own hardware logic, and views never talk to StoreKit/AVFoundation/ARKit/PDFKit directly — everything crosses a protocol boundary. This is consistent with, and a direct build-out of, the `@Query`-direct-except-Capture/Export state ownership already locked in `ios_architecture_review`.

**File/module layout:** One app target, one unit-test target, one UI-test target. `App/` (app entry, scene wiring, DI container, entitlement service bootstrap), `Domain/` (models, enums, `ProjectPolicy`, `IssueFilter`, `MeasurementFormatter`, `ReportSnapshotBuilder`), `Data/` (SwiftData schema, migration plan, repository protocols + SwiftData implementations), `Services/` (`CameraService`, `ARMeasurementService`, `PhotoAssetStore`, `ReportRenderer`, `EntitlementService`, `StorageHealthService`), `UI/` (per-screen folders: Library, Workspace, Capture, IssueDetail, Export, Paywall, Settings, plus shared components), `Support/` (logging, error types, environment keys, test fixtures).

**Data models:** `Project` (id, name, siteLabel?, timestamps, lifecycle active/archived, accessState normal/lockedReadOnlyDueToPlan), `Area` (id, projectID, name, timestamps, lifecycle, displayOrder — duplicate names allowed), `Issue` (id, projectID, areaID, timestamps, capturedAt, severity, status, note?, one `Measurement`, `photoState` explicitly pending/ready(paths)/missingOrCorrupted — no separate draft model), `Measurement` (value, unit, source manual/ar, optional arConfidence state), `BusinessProfile` and `AppSettings` as small persisted singletons. This is the concrete shape of everything `detailed_discussion` and `app_features` already required.

**Persistence strategy:** SwiftData for structured records, photo binaries as files in Application Support. Capture creates the `Issue` immediately at `photoState = .pending`; original image write happens off-main (temp file → downsample/compress → strip EXIF GPS → atomic move → optional thumbnail), then the `Issue` is patched to `.ready` on the main actor, or `.missingOrCorrupted` with a typed reason on failure. `ModelContext` access stays main-actor bound; anything crossing to a background task (export, rendering) is copied into plain `Sendable` snapshot structs first — never SwiftData model objects across the actor boundary.

**Testing strategy:** Unit tests for cap enforcement, downgrade locking, archive rules, duplicate-area handling, measurement formatting, filter predicates, report grouping/order. Integration tests against an in-memory `ModelContainer` plus real temp directories for create→capture→relaunch survival, pending→ready photo transition, corrupted-photo degradation, and schema-v1 migration. UI tests for first-run creation, repeated manual capture, permission-denied/interrupted camera, over-cap paywall, locked read-only project, export preview with placeholder, and settings-driven unit-display changes. Camera and AR verification is explicitly protocol-simulated (unsupported/unstable/success, injected photos) — real measurement tolerance is not claimed in this environment, consistent with every prior phase's stated limitation.

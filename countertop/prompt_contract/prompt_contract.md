## Claude Design Handoff Prompt

Design a premium iPhone cooking app called Countertop. The signature screen is a multi-dish cook-mode timeline that interleaves steps and timers across 2-3 recipes being cooked together for one meal, so the user always knows what to start next to make everything finish at once. Warm neutral palette, food-safe accent color, large legible typography readable at arm's length, one-handed touch targets, calm and editorial rather than social or gamified. The contract is clear: build one real iOS app, not a concept package, in SwiftUI with local-first persistence and working user-visible behavior across the full v1 scope. The centerpiece is not “recipes” in general. It is the multi-dish sequencing engine and cook-mode timeline. Every major product decision later has to protect that. If something makes the library better but weakens the clarity, trust, or prominence of sequenced cooking, it is the wrong decision.

Hard requirements are straightforward. This must ship as a working local-first SwiftUI MVP for Countertop, under the parent portfolio but as its own separate child project. It must include recipe import and editing, structured ingredients and steps stored locally with SwiftData, support for paste/share/photo-based recipe capture with cleanup, a sequenced cook mode that can combine 2 to 3 recipes into one interleaved timeline, concurrent timers during cooking, a weekly planner, grouped grocery list generation, cook history with personal notes, settings, and a StoreKit paywall. It must handle empty, loading, success, and error states anywhere those states are reachable. It must be designed so later cloud sync can be added without replacing the domain model. It must feel premium on iPhone, readable at arm’s length, and kitchen-safe in contrast and touch targets. Privacy is not marketing copy here; it is structural. User recipe data and history stay local in v1 unless a specific import action requires system frameworks like share sheet or Vision OCR.

The non-goals matter because this app can bloat fast. It is not a recipe discovery feed, not a social cooking network, not a marketplace, not an AI cooking assistant, not a nutrition tracker, not a grocery delivery integration, and not a general-purpose meal planning platform that happens to include recipes. It also should not chase broad “smart kitchen” fantasies in v1. No cross-device sync requirement now. No household collaboration now. No backend dependency now. No speculative AR or LLM layer. If OCR or messy import cannot confidently produce good structure, the app must fall back to explicit review and editing rather than pretending it succeeded.

The output shape for later phases should stay concrete and build-oriented. Each phase should produce artifacts that can directly drive implementation: product decisions, domain model, flow definitions, screen inventory, interaction rules, sequencing logic rules, persistence plan, monetization boundaries, test plan, and then the actual app project. The final artifact must make it obvious what the app does on device, what is implemented versus limited, and how the sequencing engine makes dinner calmer than ordinary recipe apps. The detail that absolutely must survive every phase is this: Countertop wins or loses on whether a user can choose multiple dishes and trust one unified execution timeline more than their current ad hoc method.

Production-ready in this context means more than compiling screens. The core workflows have to function end to end on a local device build, with deterministic local state, recoverable failures, accessible UI, data persistence, and test coverage on the risky logic. Import must surface uncertainty instead of silently corrupting recipes. Sequencing must be conservative, editable, and explainable enough that a cook can correct it before starting. Cook mode must survive normal interruptions and keep timers legible. Weekly planning and grocery generation must produce useful outputs from real stored recipes, not mocked placeholders. Monetization gates must exist and be coherent, even if subscription wiring is MVP-grade. Production-ready also means honest scope discipline: better one excellent cook flow with fewer edge-case imports than a sprawling app that cannot be trusted at dinnertime.

The decision rules for later phases should be strict. Favor anything that increases trust in timing, editability, and calm execution. Prefer conservative sequencing over clever automation. When there is a tradeoff between feature breadth and confidence in the cook-mode engine, cut breadth. Treat the recipe library as supporting infrastructure for better cooking sessions, not the product identity. If an imported recipe is ambiguous, require user confirmation. If timing confidence is low, expose assumptions and let the user adjust durations or finish targets. If a feature does not make planning or executing a real meal better this week, it drops below the line. If a proposed design makes the signature timeline harder to read with messy hands from a kitchen distance, reject it. If freemium gating obscures the core value too early, it hurts conversion; if it gives away multi-dish sequencing entirely, it undermines monetization. The sensible line is that single-recipe value can be sampled free, but the unified multi-dish engine remains the paid habit-forming unlock.

The boldest direction is to make Countertop feel like a private operating system for dinner, not a prettier recipe box.

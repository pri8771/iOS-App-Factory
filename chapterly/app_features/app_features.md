### Round 1


**Codex (Design Lead · Systems Thinker) — Round 1**

The build has to stay brutally centered on one feeling: “I had a reaction to something I read, and this app helped me catch it before it disappeared.” If a feature does not make capture, reflection, or selective sharing of that finished artifact better, it should not be in the must-have set.

My must-have list is small on purpose.

- Personal reading log with lightweight book creation and status tracking. User story: as a reader, I can add a book in seconds so I have somewhere to attach quotes and reflections without doing catalog work first. Acceptance criteria: I can create a book with just title and author; I can mark status as want-to-read, currently-reading, finished, or did-not-finish; the home experience shows an honest empty state when there are no books and a useful filled state when there are some; data persists across relaunch.

- Journal entry creation tied to a book, with quote-first flow. User story: as a reader, I can save a quote or reaction immediately, then add structure around it after the thought is safe. Acceptance criteria: I can create multiple entries for the same book; quote text can be saved before rating or longer reflection is filled in; page/location is optional free text, not required; rating is optional; if I abandon mid-entry, the app either preserves a draft intentionally or discards cleanly without creating broken partial records.

- Manual quote capture that feels first-class. User story: as a reader, I can type a quote or paraphrase when camera capture is inconvenient or unnecessary. Acceptance criteria: manual quote entry is reachable in one obvious path with no permission prompt; I can edit quote text, page/location, rating, and reflection before saving; save success is visible; validation errors are clear; the saved entry appears in timeline immediately and persists after relaunch.

- OCR quote capture with confirm-and-edit. User story: as a reader, I can photograph a printed page and turn it into editable quote text faster than typing it. Acceptance criteria: camera capture is single-frame only; every OCR result lands in an editable confirmation screen before save; if no text is detected or OCR is poor, I can switch to manual editing instead of hitting a dead end; denied camera access produces a usable fallback state that routes to manual entry; no part of the flow implies the OCR text was silently accepted.

- Personal timeline as the primary home surface. User story: as a reader, I can come back and see my reading life accumulating into something meaningful. Acceptance criteria: the first/main screen prioritizes books and entries, not circles; with no data, it explains what to do next; with a little data, it shows recent entries clearly enough to skim; tapping an item opens a readable detail view; editing or deleting an entry updates the timeline correctly.

- Personal stats that feel worthwhile with small data volume. User story: as a reader, I can see a simple summary of my habit without needing months of history. Acceptance criteria: stats include a restrained set such as books finished, quotes saved, streak, and favorite authors or average rating when available; when there is not enough data for a stat, the app shows an intentional empty or partial state rather than nonsense; date logic is stable enough that relaunching or crossing midnight does not obviously break counts.

- One saved circle for outbound sharing, modeled as recipients not a discussion space. User story: as a reader, I can save the two or three people I actually send book thoughts to, so sharing later is quick. Acceptance criteria: I can create one circle and add/remove members; the UI never suggests live in-app replies or synced discussion; the circle surface is secondary in navigation and visual weight; if I never create a circle, the rest of the app still feels complete.

- Share a finished entry through Messages/share sheet as a snapshot. User story: as a reader, I can send a polished version of a completed entry to my trusted people without needing them to install anything. Acceptance criteria: sharing only happens from a completed saved entry; the shared content includes the useful parts of the artifact such as quote, book, author, optional location, and reflection excerpt; if Messages is unavailable, generic share sheet fallback works; after sharing, the app records that a share happened for my own history, but later edits do not retroactively change the prior share.

- Local persistence and privacy-respecting behavior throughout. User story: as a reader, I can trust that my private journal lives on my device and survives normal use. Acceptance criteria: books, entries, circle members, and share history survive force-quit and relaunch; there is no account requirement; permission asks only appear when needed; denied permissions never make the app unusable; the app communicates clearly when content is leaving the app via sharing.

The should-have list is where I would put features that materially strengthen the product but can be cut without breaking the promise.

- Widget support for currently-reading and streak.
- App Intent or Siri shortcut for logging a quote into the most recently active book with confirmation.
- Optional biometric app lock, because the journal is plausibly intimate enough to justify it.
- A lightweight share-history view on an entry so users can remember what they sent and when.
- A tasteful paid gate for OCR and additional circles, without full StoreKit implementation.

The could-have list is only acceptable if the must-have set is already finished and stable.

- Yearly wrap-up teaser screens or a stubbed export concept.
- Rich book covers or manual cover art attachment.
- Discussion prompts generated from structured templates rather than a real local model.
- More polished editorial treatments like texture variants, quote cards, or multiple display modes.
- Draft recovery for half-finished entries if it can be done cleanly.

The won’t-build list needs to stay explicit so the build does not quietly turn into a different app.

- No public feed, public profiles, follower graph, discovery, or recommendations.
- No in-app inbox, reply ingestion, or live discussion threads.
- No backend, CloudKit sync, or multi-device reconciliation in v1.
- No multi-circle depth beyond architecting the model and gating the UI.
- No Photos-library OCR import, multi-page OCR stitching, or “smart” OCR automation that skips confirmation.
- No push or local streak reminders in v1.
- No heavy on-device LLM feature work.
- No book-search network dependency.

The best direction is a journal-first product where the main screen feels like a warm personal ledger of reading, the add-entry flow is faster than opening Notes, and sharing is an elegant final action on something already worth keeping. The biggest risk in that direction is that the social category gets under-served if sharing feels bolted on or too hidden. The fix is not to elevate circles into a tab-first pseudo-network; it is to make the share moment excellent and obvious exactly when an entry is complete.

The detail the final output must preserve is that the atomic unit is the finished entry, not the book and not the circle. The entire information architecture should bend around that: capture an entry quickly, keep it beautifully, optionally send it outward as a snapshot, then return to the private record. If that center shifts, the app will either become a generic reading tracker or a fake social app, and both would miss the brief.

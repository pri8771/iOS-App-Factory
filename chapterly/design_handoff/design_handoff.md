**Claude (Frontend Engineer · Skeptic) — Round 1**

Let me start from the place I think this phase is most likely to hand-wave: navigation architecture and cross-process state, because "Timeline is home, Stats/Circle are icon-accessible" sounds like a design sentence but it's actually a routing decision, and the widgets/App Intent requirements from earlier phases only work if the data layer is built a specific way from day one.

Concretely: Timeline should be the root of a single `NavigationStack` bound to a `NavigationPath`, with Stats and Circle pushed onto that same stack via the nav-bar icons — not presented as sheets, and not their own `NavigationStack`s. If Stats and Circle get their own stacks (e.g. because someone reaches for `.sheet` for one and push for the other), you get inconsistent back-swipe behavior and the "Circle is a drawer not a tab" principle starts leaking implementation weirdness into the UX. Book Detail, Entry Detail, Add/Edit Book, and Add Circle Member are also stack pushes. Capture (choose manual or camera) and the entry Compose/OCR flow should be a `.fullScreenCover` or `.sheet`-presented modal flow with its own internal `NavigationStack` or a simple enum-driven step machine, because it's a self-contained task (capture → confirm → save) that should be dismissible as a unit and shouldn't pollute the Timeline's back-stack with three intermediate screens the user never wants to "back" through individually. Settings, Paywall, and App Lock are modal too. This needs to be decided now because the App Intent requirement ("log a quote via Siri, confirm the assumed book before saving") means the app needs a way to cold-launch directly into that Compose flow with a pre-selected book — that's a deep-link-shaped requirement hiding inside a "should-have" bullet, and if the navigation model isn't path-based and state-restorable, that App Intent either can't reliably land the user in the right place or it'll require a parallel ad-hoc presentation path that drifts from the rest of the app's navigation over time.

The bigger architectural gap I don't see resolved anywhere yet: widgets and the App Intent are a separate process from the main app. If persistence is plain SwiftData with the default app-container store, the widget extension and the intent extension cannot read or write that store at all — you need an App Group–shared container (shared `ModelContainer` pointed at a shared app group URL) from the very first schema, or the "Currently Reading" and "Streak" widgets will either show stale bundled placeholder data forever or the build will discover this gap during widget implementation and have to redo the persistence layer. This is exactly the kind of thing that gets glossed over in a design handoff because it looks like a backend detail, but it is a screen-state-determining fact: the widget's empty state, its data-refresh timing (widgets don't get live pushes — they refresh on a timeline the OS controls, via `WidgetKit`'s `TimelineProvider` and explicit `WidgetCenter.shared.reloadTimelines` calls after a save), and its failure mode if the shared container can't be opened all need to be designed as real states, not assumed away as "just show real data."

I'm also going to push back gently on the earlier round's claim that local save should be "near-instant, no spinner, and if it isn't that's a bug." That's mostly right and I want to keep it as a design principle, but it needs one honest caveat: SwiftData's first launch after a schema change can trigger a lightweight migration, and the very first app launch does on-disk store setup — both of which can take a visible fraction of a second to low hundreds of milliseconds on an older device. So the rule should be "no spinner for routine saves inside an already-open session," not "no loading state anywhere in the app," or someone will build zero loading affordance anywhere including first launch, and then first launch on an iPhone 12 stutters with a blank white frame. Cold-launch-to-Timeline should have a trivial branded launch state (a static launch screen matching the Timeline empty state's background, not a spinner) to cover that gap cleanly.

On to the concrete handoff.

**Screens, states, and how they sit in navigation:**

Onboarding/Welcome — presented full-screen only on first launch (a flag in UserDefaults or a `hasCompletedOnboarding` bool in the shared store), not part of the nav stack at all, dismisses into Timeline. States: single static state, no loading, no error — it's static copy and one CTA, no external dependency, so there's no excuse for this screen to ever show a spinner or fail.

Timeline (root of NavigationStack) — states are empty (zero entries: one sentence, one CTA, illustration is optional and should degrade gracefully if art isn't ready — never block the empty state on an asset existing), populated (grouped by relative recency), and a genuine error state for the case where the SwiftData fetch throws (corrupted store, disk full) — this needs real copy and a retry action, not just "won't happen in practice," because disk-full is a real device state on older phones with photo libraries full of camera-roll content. No skeleton/shimmer loading state — data is local, fetch is synchronous-feeling via `@Query`, so the moment the view appears it should already have content or already know it's empty.

Library toggle — same screen, alternate content mode, not a separate push destination. States: empty (no books) and populated (list), each independent of Timeline's own entry-count-driven state.

Book Detail — pushed from Library or Timeline. Settings, Paywall, App Lock cover — Settings pushed or sheeted from a Timeline toolbar item; Paywall sheeted from Settings/Circle limit/OCR gate; App Lock cover is the one screen that isn't part of navigation at all — it needs to be an overlay driven by `scenePhase` transitioning to `.background`/`.inactive`, applied at the app's root `WindowGroup`, above the NavigationStack, not a pushed screen. Worth flagging as a real risk: naively gating on `.inactive` will trigger the lock screen every time a system sheet (the OCR camera permission prompt, the Messages compose sheet, the share sheet) briefly takes focus, which would make the app re-lock itself mid-flow constantly.

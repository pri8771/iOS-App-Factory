## Final Output

**Screen inventory (7 screens, one reused in two contexts):**
1. **Setup/Home** — the app's single front door on every launch (no mid-game resume, so there's never anything to triage). Name entry for 2–6 players is the hero content; ruleset selector and Start button sit with it; small corner icons open History, Ruleset Editor, and Mode/Settings.
2. **Ruleset Editor** — sheet launched from Setup. Edit card-to-copy mappings, save/name rulesets, pick from saved ones.
3. **Mode/Settings** — small pushed screen from Setup's corner icon. Sober-default toggle, drinking-mode one-time self-attestation gate, plain-language replay-fidelity explainer ("changing mode later won't rewrite old games"), and the entry point to reopen onboarding.
4. **Active Game** — one screen, several sub-states, never a separate "pass to next player" screen: shielded (default/fail-safe state), confirm-to-reveal, revealed (card + rule + HUD update), optional inline Joker text capture, immediate re-shield on handoff, end-of-game transition.
5. **Recap** — one reusable view for two contexts: a just-finished game (Export + Play Again, transient "saved" confirmation) and a reopened History row (Export only, read-only, no Play Again). Shows end-reason, honest partial-round timeline, funniest-rule ballot or its explicit empty state, highlights. Export is an inline state on this screen — status area/spinner, then the system share sheet, then a banner for permission-denied/failure, silent close on user-cancel — not a separate route.
6. **History List** — behind its corner icon, swipe-to-delete per row, tapping a row pushes Recap in its read-only context.
7. **Onboarding** — one-time sheet firing automatically on first launch, reopenable anytime from Mode/Settings.

**Primary user flow:** Launch always lands on Setup/Home (first-run also surfaces the onboarding sheet) → enter 2–6 names, pick/confirm ruleset and mode → Start locks the 53-card deck → Active Game loop: shield renders first on every turn, confirm-tap reveals card + rule (sober/drinking copy), Joker draws offer optional inline text capture, HUD updates, immediate re-shield before any handoff UI → repeats until the first of 4th King / 10 rounds / deck exhaustion (cup-full wins a simultaneous tie) → completed game is persisted to CoreData the instant the engine ends, before Recap even renders → Recap shows the finished game with Export and Play Again → Play Again returns cleanly to Setup/Home. Independently, History List is reachable anytime from Setup and opens past games into the same Recap view, read-only.

**State model per screen:** Setup — default/first-run, inline validation error (player count/names), loading (starting/saving ruleset), blocked (attestation required for drinking mode), success-to-game. Active Game — shielded (fails safe on backgrounding/rotation/rapid taps/notifications/system interruptions), confirm-armed, revealed, joker-capture-pending (Save/Skip), applying-rule, handoff-locked, game-over transition. Recap — loading from persisted record, success, explicit "no custom/Joker rules this game" empty state, non-blocking "couldn't save to history" banner if the local save itself fails, and the four export sub-states (loading, success/share-sheet, cancelled-as-no-op, permission-denied-with-Settings-link, hard-failure-with-retry). History List — loading, empty ("no games yet"), success list with end-reason badges, delete confirmation, load-error with retry. Ruleset Editor — loading/list, unsaved-changes warning, invalid-mapping error (unmapped card), saving, save success, save failure with edits preserved. Mode/Settings — normal, attestation-required gate, attestation-failed, mode-change success.

**Visual direction:** A restrained, bold utility aesthetic — matte warm gradient background with subtle card/cup geometry, no ad-like ornamentation, no flat monochrome, no purple-driven palette. One expressive display font for headings, a high-legibility sans for controls, generous spacing, minimal visual noise during turns. One large centered card as the focal object; the shield is a fully opaque, unmistakable full-bleed mask (not a translucent blur that could hint at content underneath), showing a lock glyph and the incoming player's name in large type — that name is the one thing everyone at the table needs to read instantly. Cup-fill visualization gets the real visual/animation budget as the game's built-in tension arc; reduced-motion users get a static fill-level indicator instead of an animated pour. Calm, low-saturation default palette that works equally well under sober-mode framing. Minimal, purposeful motion only: shield transition fade and a brief tap-confirm pulse.

**Accessibility notes:** All primary actions (Reveal, Pass, Start, Export, Delete) are ≥44×44pt and stacked vertically, not side-by-side, for one-handed party-condition use. Shielded card content is removed from the accessibility tree entirely (`accessibilityHidden` or equivalent) — visual hiding alone is not sufficient — verified with a dedicated VoiceOver-on QA pass; but the "Pass to [Name]" label and Reveal control stay fully in the tree and clearly announced, since hiding that layer too would leave a blind player unable to tell it's their turn. Dynamic Type is tested specifically on the reveal screen where rank/suit/rule text and the Joker field compete for space — rule text wraps and scrolls rather than truncates, since a truncated house rule breaks the funniest-rule feature outright. High-contrast and reduced-motion options are supported; orientation changes and backgrounding always force the shielded state and its hidden accessibility semantics, with fail-safe-hidden behavior also verified against full-screen system interruptions like calls and Face ID prompts, not just banners.

## Final Output

**SwiftUI architecture:** Pure-Swift `@Observable` `GameEngine` at the app root, injected via environment, fully isolated from SwiftUI and CoreData. Persistence touched only at defined boundaries — ruleset load at Setup, one write at game-end, history list/delete — never in the per-turn hot path. Engine actions are pure and idempotent; `confirmReveal` specifically must be a no-op past the first call in a runloop tick (this is now a named unit test, not just a design note). Deployment target locked at iOS 17.0 (required for `@Observable`, also gets `ImageRenderer` and `ContentUnavailableView` for free). No MVVM layer — the engine plus lightweight `RecapViewModel`/`ModeSettingsContext` read-only wrappers already fill that role.

**Apple frameworks:** SwiftUI for all UI, Foundation/Swift Concurrency for the engine, CoreData for persistence, Observation for state binding, `ImageRenderer` for recap image generation, `UIActivityViewController` (wrapped as `UIViewControllerRepresentable`, presented via `.sheet`) for the actual share/export flow — this is the only mechanism, not a fallback, because it's the only one that exposes `completionWithItemsHandler` and can therefore distinguish success/cancel/failure at all. Plain `ShareLink` is explicitly rejected as the primary path since it has no completion callback and can't support the four-state export contract already ratified. `PHPhotoLibrary` requested at `.addOnly` authorization only — never `.readWrite`, since the app never reads the library, only writes one generated image.

**Persistence and local data plan:** CoreData with `schemaVersion` from the first migration. Three entity groups: `RulesetEntity` + 53 `RulesetCard` rows (A–K + Joker, bundle marker/version); `GameSessionEntity` (schemaVersion, timestamps, player snapshot, endReason, terminal metrics); `TurnEventEntity` per draw (player index, turn index, card, mode-at-turn, rule key, Joker text, drink/dare delta, round marker) — this is the authoritative record that makes replay-fidelity and funniest-rule actually trustworthy. Optional denormalized `HistoryDisplayStats` for History List row performance, explicitly defined as a read-derived cache rebuilt from `TurnEventEntity`, never hand-written independently at save time — if it and the turn-event log ever disagree, the turn events win. Writes happen on a background `NSManagedObjectContext` with `viewContext` auto-merging so persistence latency never touches the shield/reveal loop.

**Permissions/privacy plan:** exactly one permission in the whole app — `NSPhotoLibraryAddUsageDescription` with `.addOnly` authorization, requested before presenting the share sheet; `.denied`/`.restricted` routes straight to the permission-denied state with a Settings deep-link, `.notDetermined` requests and falls back to denied on rejection. No camera, mic, location, contacts, HealthKit, Face ID/biometrics entitlement, push/local notifications, or background modes. Zero network calls, zero ATS exceptions. `isIdleTimerDisabled = true` only during active play. Carried forward as a named risk, not something architecture alone resolves: Apple's App Review guideline 1.4.3 on alcohol-content apps is a real submission risk that needs explicit handling in whatever phase covers App Store metadata (sober-default, the attestation gate, and the 17+ rating mitigate but don't guarantee approval).

**Dependency/license decision:** zero third-party packages. No StoreKit in v1 (stays a written future-tier description only, no scaffolding); no ML/AR (nothing in this app justifies either); no analytics/crash-reporting SDK of any kind.

**Testing implications:** engine unit tests cover deck-exhaustion math at every player count 2–6, the [REDACTED:high_entropy] end conditions with cup-full winning simultaneous ties, funniest-rule sourcing (default-to-last-custom, empty state), and the idempotent-confirmReveal rapid-tap case as an explicit assertion (exactly one turn advance from five rapid calls). CoreData gets its own in-memory-store test target covering save/load/delete and schemaVersion fallback decoding. XCUITest covers Setup/Ruleset Editor/History state surfaces. Export and the hardest shield-adversarial cases (notification banners, Face ID prompts, incoming calls) have a hard automation ceiling — those stay a manual, on-device QA checklist, consistent with what every prior phase already flagged, now tied to the concrete technical reason (system UI that XCUITest can't reliably drive).

## Final Output

**Architecture overview:** A single SwiftUI iOS 17+ app target — no third-party dependencies, no SPM package split. The game engine is a pure-Swift, side-effect-free value-type reducer (`GameEngineReducer.reduce(state, event) -> (state, effects)`), wrapped by a thin `@Observable @MainActor` `GameEngine` class that republishes state and dispatches the declarative effects the reducer returns (haptics, the one game-end persistence write). This keeps the "five rapid confirm-taps advance exactly one turn" idempotency guarantee a one-line pure function test rather than a timing-dependent integration test, and makes every `GamePhase` — including edge cases like a Joker draw one card before deck exhaustion — directly constructible in a SwiftUI Preview with no need to actually play a game to reach it. Engine isolation from CoreData/SwiftUI/UIKit is enforced by a build-phase script that fails the build if any file under `Engine/` imports those frameworks — a compiler-adjacent guarantee without the ceremony of real package boundaries. Privacy shielding is handled independently of the engine by a root-level `PrivacyShieldCoordinator` reacting to `SceneLifecycleEvent`s, specifically because a sheet like Mode/Settings-in-`.midSession` can be on screen during a backgrounding event and isn't part of engine state at all — the coordinator forces a hard "nothing revealed can render" flag regardless of what's presented on top, and separately forwards the interruption into the engine so game state stays consistent too.

**File/module layout:** One target, `drinking-games-test`, organized as `Domain/` (Card, Rank, Suit, RuleKey, RuleDefinition, Ruleset, GameMode, EndReason, TurnEvent, TurnDraw, GamePhase, GameSessionRecord, FunniestRuleResult), `Engine/` (GameEngineState, GameEngineEvent, GameEngineEffect, GameEngineReducer, GameEngine, Deck — import-restricted by the build script above), `Persistence/` (PersistenceController, CoreData model, RulesetStore/GameHistoryStore implementations, PersistenceError), `Services/` (ExportCoordinator, PhotoPermissionService, AttestationService, HapticsService), `UI/` (Setup, Ruleset Editor, Active Game, Recap, History List, Mode/Settings, Onboarding screens plus shared components), and `Resilience/` (PrivacyShieldCoordinator, SceneLifecycleEvent).

**Data models:** Standard 53-card deck (`Deck.standard53`) of `Card` (`Rank?`, `Suit?`, `isJoker`), rules keyed by `RuleKey` with dual-voice `RuleDefinition` (`soberText`/`drinkingText` — never a single shared string, so editing one mode's copy can never blank the other). `TurnEvent` is the single authoritative per-draw record — mode-at-turn, a frozen `ruleTextSnapshot` so later ruleset edits never rewrite old recaps, and `unitsAssigned` (deliberately mode-neutral naming, not "drinksDelta," since sober is the app's default voice). Cup-fill state is never stored — it's always derived as Kings-drawn/4, both live and in recap. `GameSessionRecord` stores only what can't be derived (id, timestamps, players, rulesetName, endReason, the timeline, the captured `funniestRule` ballot result) and computes everything else (`kingsDrawnCount`, `cardsRemainingAtEnd`, `mostDrinksPlayerIndex`) from the timeline on read, so a recap can never structurally diverge from what actually happened.

**Persistence strategy:** CoreData with `schemaVersion` from the very first migration, background-context writes with `viewContext` auto-merge so persistence latency never touches the shield/reveal hot path. Exactly three write boundaries: ruleset load at Setup, one `GameHistoryStore.save` at game-end (triggered by the reducer's `.gameDidEnd` effect, never called speculatively), and history list/delete. `PersistenceError` is a concrete `LocalizedError` (`.notFound`, `.loadFailed`, `.saveFailed`, `.deleteFailed`, `.migrationFallbackUsed`) so failure states can show real, distinct copy instead of a generic string. Export uses `ImageRenderer` plus `UIActivityViewController` wrapped for SwiftUI (never plain `ShareLink`, which has no completion callback), with Photos `.addOnly` authorization requested before presenting, mapping completion to the four required `ExportOutcome` states.

**Testing strategy:** `GameEngineReducer` gets pure unit tests — deck exhaustion at every player count 2–6, cup-full-wins-simultaneous-tie, idempotent `confirmReveal` (five rapid calls in one runloop tick advance exactly one turn), funniest-rule default-to-last-custom and empty-state paths, all callable directly against plain value types with no mocking. CoreData gets an in-memory-store test target for save/load/delete and `schemaVersion` fallback decoding. `ScreenState<T, PersistenceError>`-driven CRUD screens (Ruleset Editor, History List, Setup ruleset-loading) get XCUITest coverage of their state transitions.

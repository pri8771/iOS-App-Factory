## Original Prompt

```
Build Kings — a multiplayer drinking game app for iOS (local pass-and-play).

Core mechanics:
- 2–6 players draw cards from a deck in turn order
- Each card (A–K, plus Joker) triggers a rule: e.g. "2 = You" (drinker picks someone), "5 = Guys", "K = King's Cup" (build communal cup with dares), "Joker = Make a rule"
- Players drink per the card rule; when the cup is empty or 10 turns pass, game ends with a recap
- Pass phone between players; each player sees only their own card until they pass

Key features:
- Local only (no server, no login)
- Customizable rules: edit card meanings before each game, save rulesets
- Game history: recap shows who drank most, funniest rule, timeline of rounds
- Sober mode toggle (default): uses "dare" language instead of drinking; activating drinking mode (17+ gate) swaps copy to drinking references but same mechanics

Tech:
- SwiftUI, local-first with CoreData for game history
- Phone must stay face-down while other player holds it; privacy shield until handoff
- Export game recap as a sharable image (name, date, player list, highlights)

Scope: MVP with Kings rules only (not Hearts, Asshole, or other variants). One ruleset bundled; let users customize before start.
```

**Codex (Product Strategist · Pragmatist) — Round 1**

The direction is to lock this as a trust-first utility flow: Setup → Shielded turn engine → Finish/end → Recap/export → History, with no extra navigation branches for a playable session. The product value is in confidence that nothing leaks between hands and that the game always produces a complete, auditable recap.

Primary user flow:
Home, or first-run empty state, opens straight into setup because players want play, not menus. On Setup, 2–6 names are entered, one ruleset is selected/created, mode and attestations are set, and the game starts. In-game is a strict loop: shield state first, confirm-tap reveal, card/rule render, HUD update, handoff re-shields, next player. The game ends on first terminal condition (4th King, 10 rounds, or deck exhaustion, with 4th King winning if deck ends on that draw). Then Recap becomes the only next required screen, with export and timeline from immutable completed-game data, then History for retention and cleanup.

Screen inventory:
Home/Hub shows start game, rulesets, history, and settings; this screen also hosts first-run onboarding if needed and can show load-state errors from Core Data; key screen states are empty (no prior games), loading, success (menu available), error (retry with messaging).
Setup is split into Game Setup and Player Entry plus Ruleset+Mode panel; this is where the exact 2–6 gate, default mode, and ruleset choice happen; state model is invalid input error (name/rules count), creating-state loading, proceed-to-game success, and blocked if mode gate/permissions are missing.
Game Play is the single active session screen, with shield-first handoff as the canonical state; states are shielded (privacy lock, no card semantics in a11y tree), awaiting confirmation, revealed (card and rule visible), joker-text capture optional flow, apply-rule/hud update, and terminating transition.
Recap is a persisted-game detail screen, not an ephemeral live view; states are computing from timeline, success summary, empty-but-defensive state (“no custom/Joker rules played this game”), and recap load failure.
Export Overlay/Share path is separate enough to own failures clearly; states are permission-check, rendering/loading, cancelled (neutral), permission-denied with Settings recovery action, and hard render/share failure with explicit retry text.
History List uses swipe-to-delete as a trust primitive; states are loading, empty (“no games yet”), success list with end-reason badges, and delete error with undo/ retry action.
Ruleset Editor is pre-game only; players can edit/create/select card meanings and save snapshots; states are listing/loading, unsaved-changes warning, invalid-mapping error, save-in-progress, save success, and save failure with preserved edit state.
Sober/Drinking Mode Settings appears as a lightweight policy screen: default sober mode, gated activation gate for drinking mode, and a permanent note that recap copy is replay-fidelity from play time; states are normal, gate required, attestation failed, and mode-change success.
Onboarding state exists as an optional compact sheet tied to first launch and first rule conflict; states are unopened, viewed-and-accepted, skipped, and reopenable from Settings.

State model per screen, in implementation terms:
Home state is a snapshot state machine over persisted game/rules/history loading and error recovery.
Setup state holds player roster validation, selected ruleset pointer, mode and attestation intent, and a deterministic transition to Game Ready only when roster and required guards pass.
Game Play state is a pure state-machine host view with view-only derivations: hiddenCard, confirmArmed, activeCardShown, jokerCapturePending, turnResolved, handoffPending, gameOverReason. All UI states are projections, never sources of truth.
Recap state is read-only from persisted turn timeline and highlights model: computes winner metrics, end-reason label, funny-rule outcome candidate set with empty-state fallback preserved.
Export state records three-way outcome split and a rendered image artifact path for share/retry.
History state manages list data and delete operations with immediate optimistic UI update and reconciliation for Core Data failure.
Ruleset state manages card-to-copy map edits, validation completeness across all supported cards, and saves with schema-aware compatibility assumptions.

Visual direction:
Use a restrained, bold utility aesthetic: matte warm gradient background with subtle card/cup geometry, not flat monochrome, no ad-like ornamentation, and no purple-driven palette. Use one expressive display font for headings and a high-legibility sans for controls, with clear hierarchy and wide spacing; avoid visual noise during turns. The card region should have heavy contrast in both normal and low-light conditions, and the shield should be unmistakable (full-surface mask, distinct edges, clear prompt text). Minimal but intentional motion only: fade/shield transition and a brief tap-confirm pulse for confidence; no spinning loaders beyond short shimmer for async states.

Accessibility notes:
Every tappable control is >=44×44 points, labels are explicit, and state changes are announced with VoiceOver-friendly live regions. The hidden-card state must remove card content from the accessibility tree (`accessibilityHidden` or equivalent), not just visual blur/cover. During shielded handoff, no rank/suit/verb should be discoverable by screen reader. **Screen inventory (7 screens, one reused in two contexts):**
1. **Setup/Home** — the app's single front door on every launch (no mid-game resume, so there's never anything to triage). Name entry for 2–6 players is the hero content; ruleset selector and Start button sit with it; small corner icons open History, Ruleset Editor, and Mode/Settings.
2. **Ruleset Editor** — sheet launched from Setup. Edit card-to-copy mappings, save/name rulesets, pick from saved ones.
3. **Mode/Settings** — small pushed screen from Setup's corner icon. Sober-default toggle, drinking-mode one-time self-attestation gate, plain-language replay-fidelity explainer ("changing mode later won't rewrite old games"), and the entry point to reopen onboarding.
4. **Active Game** — one screen, several sub-states, never a separate "pass to next player" screen: shielded (default/fail-safe state), confirm-to-reveal, revealed (card + rule + HUD update), optional inline Joker text capture, immediate re-shield on handoff, end-of-game transition.
5. **Recap** — one reusable view for two contexts: a just-finished game (Export + Play Again, transient "saved" confirmation) and a reopened History row (Export only, read-only, no Play Again). Shows end-reason, honest partial-round timeline, funniest-rule ballot or its explicit empty state, highlights. Export is an inline state on this screen — status area/spinner, then the system share sheet, then a banner for permission-denied/failure, silent close on user-cancel — not a separate route.
6. **History List** — behind its corner icon, swipe-to-delete per row, tapping a row pushes Recap in its read-only context.
7. **Onboarding** — one-time sheet firing automatically on first launch, reopenable anytime from Mode/Settings.

**Primary user flow:** Launch always lands on Setup/Home (first-run also surfaces the onboarding sheet) → enter 2–6 names, pick/confirm ruleset and mode → Start locks the 53-card deck → Active Game loop: shield renders first on every turn, confirm-tap reveals card + rule (sober/drinking copy), Joker draws offer optional inline text capture, HUD updates, immediate re-shield before any handoff UI → repeats until the first of 4th King / 10 rounds / deck exhaustion (cup-full wins a simultaneous tie) → completed game is persisted to CoreData the instant the engine ends, before Recap even renders → Recap shows the finished game with Export and Play Again → Play Again returns cleanly to Setup/Home. Independently, History List is reachable anytime from Setup and opens past games into the same Recap view, read-only.

**State model per screen:** Setup — default/first-run, inline validation error (player count/names), loading (starting/saving ruleset), blocked (attestation required for drinking mode), success-to-game. Active Game — shielded (fails safe on backgrounding/rotation/rapid taps/notifications/system interruptions), confirm-armed, revealed, joker-capture-pending (Save/Skip), applying-rule, handoff-locked, game-over transition. Recap — loading from persisted record, success, explicit "no custom/Joker rules this game" empty state, non-blocking "couldn't save to history" banner if the local save itself fails, and the four export sub-states (loading, success/share-sheet, cancelled-as-no-op, permission-denied-with-Settings-link, hard-failure-with-retry). History List — loading, empty ("no games yet"), success list with end-reason badges, delete confirmation, load-error with retry. Ruleset Editor — loading/list, unsaved-changes warning, invalid-mapping error (unmapped card), saving, save success, save failure with edits preserved. Mode/Settings — normal, attestation-required gate, attestation-failed, mode-change success.

**Visual direction:** A restrained, bold utility aesthetic — matte warm gradient background with subtle card/cup geometry, no ad-like ornamentation, no flat monochrome, no purple-driven palette. One expressive display font for headings, a high-legibility sans for controls, generous spacing, minimal visual noise during turns. One large centered card as the focal object; the shield is a fully opaque, unmistakable full-bleed mask (not a translucent blur that could hint at content underneath), showing a lock glyph and the incoming player's name in large type — that name is the one thing everyone at the table needs to read instantly. Cup-fill visualization gets the real visual/animation budget as the game's built-in tension arc; reduced-motion users get a static fill-level indicator instead of an animated pour. Calm, low-saturation default palette that works equally well under sober-mode framing. Minimal, purposeful motion only: shield transition fade and a brief tap-confirm pulse.

**Accessibility notes:** All primary actions (Reveal, Pass, Start, Export, Delete) are ≥44×44pt and stacked vertically, not side-by-side, for one-handed party-condition use. Shielded card content is removed from the accessibility tree entirely (`accessibilityHidden` or equivalent) — visual hiding alone is not sufficient — verified with a dedicated VoiceOver-on QA pass; but the "Pass to [Name]" label and Reveal control stay fully in the tree and clearly announced, since hiding that layer too would leave a blind player unable to tell it's their turn. Dynamic Type is tested specifically on the reveal screen where rank/suit/rule text and the Joker field compete for space — rule text wraps and scrolls rather than truncates, since a truncated house rule breaks the funniest-rule feature outright. High-contrast and reduced-motion options are supported; orientation changes and backgrounding always force the shielded state and its hidden accessibility semantics, with fail-safe-hidden behavior also verified against full-screen system interruptions like calls and Face ID prompts, not just banners.

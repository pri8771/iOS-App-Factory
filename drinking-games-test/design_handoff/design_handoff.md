## Original Prompt

```
Build Kings — a multiplayer drinking game app for iOS (local pass-and-play).

Core mechanics:
- 2–6 players draw cards from a deck in turn order
- Each card (A–K, plus Joker) triggers a rule: e.g. "2 = You" (drinker picks someone), "5 = Guys", "K = King's Cup" (build communal cup with dares), "Joker = Make a rule"
- Players drink per the card rule; when the cup is empty or 10 turns pass, game ends with a recap
- Pass phone between players; each player sees only their own card until they pass

Key features:
- Local only (no server, no login)
- Customizable rules: edit card meanings before each game, save rulesets
- Game history: recap shows who drank most, funniest rule, timeline of rounds
- Sober mode toggle (default): uses "dare" language instead of drinking; activating drinking mode (17+ gate) swaps copy to drinking references but same mechanics

Tech:
- SwiftUI, local-first with CoreData for game history
- Phone must stay face-down while other player holds it; privacy shield until handoff
- Export game recap as a sharable image (name, date, player list, highlights)

Scope: MVP with Kings rules only (not Hearts, Asshole, or other variants). One ruleset bundled; let users customize before start.
```

Coming at this as the frontend engineer who actually has to build the NavigationStack and the state machine — my job this round is to turn the seven-screen inventory into something a SwiftUI builder can start typing into Xcode without guessing, and to flag the places where the architecture as currently described has gaps nobody's caught yet.

The first thing I want to name before anything else: there's a real inconsistency sitting between two ratified decisions from different phases that nobody's reconciled. The data model was explicitly built to support toggling sober/drinking mode mid-session — Codex's round-3 detailed_discussion language is "if someone toggles sober/drinking mid-session, every subsequent turn logs the active mode at that moment" — but the design_discussion screen inventory only gives you a way to reach Mode/Settings from a corner icon on Setup, and Setup is never visited again once a game starts (there's no route back to it mid-game, by design, since the whole point of the no-hub structure is that Active Game is a closed loop until it ends). So either the per-turn mode storage is solving for a scenario the UI can never actually produce — nobody can toggle mode mid-game because there's no path to the toggle — or Active Game needs a small settings affordance I don't see specified anywhere. I'd rather we decide this explicitly than have it fall out by accident: my proposal is a small unobtrusive icon in the Active Game HUD strip (not competing visually with Reveal/Pass) that pushes Mode/Settings as a sheet without leaving the active session, so the per-turn storage decision actually has a UI that can exercise it. If the room decides mid-game mode switching isn't actually a real use case worth a UI affordance, that's fine too — but then we should say so explicitly and treat per-turn mode storage as "technically correct schema design for a scenario nobody triggers in v1," not leave it ambiguous.

On architecture: this whole app should be one `@Observable` `GameEngine` (pure Swift per the contract) held at the App root and injected via environment, with a thin `PersistenceController` that only gets touched at defined boundaries — loading the active ruleset once at Setup, writing the completed game once at game-end, and history list/delete operations — never inside the per-turn hot path. That matters for the shield: the moment CoreData work sneaks into the reveal/handoff loop, you introduce exactly the kind of unpredictable latency that turns a deterministic 150ms transition into a variable one, and variable latency around the shield is where leak bugs live (a frame where the fetch hasn't returned yet but the shield has already started to lift). Navigation should be a single `NavigationStack` off Setup/Home, with Ruleset Editor and Mode/Settings as `.sheet`, Active Game pushed as a value-type route, and Recap parameterized by an explicit `RecapContext` enum (`.justFinished(GameID)` vs `.historyRow(GameID)`) rather than two initializers with default params — that enum is the thing that keeps Play-Again-vs-read-only from drifting apart as two people touch the same view over time. One more state-restoration gotcha worth writing down now: SwiftUI scene state restoration will happily try to resurrect the NavigationStack path on relaunch, including an in-flight Active Game route — that directly contradicts the ratified "a killed game is simply gone" decision, so Active Game's route needs to be explicitly excluded from restoration (don't persist it in the restorable path, or wipe path-to-ActiveGame on `scenePhase == .active` after a cold launch).

On the shield itself, since it's still the highest-stakes surface: `scenePhase` alone isn't tight enough — its transitions can lag a runloop tick behind the actual interruption. I'd pair it with `UIApplication.willResignActiveNotification`/`didBecomeActiveNotification` observers wrapped in a single `.privacyShielded()` view modifier applied at the root of Active Game, so shielding is enforced structurally rather than something each screen state has to remember to check. And I want to flag a transition-design risk directly: don't build the reveal as a 3D card-flip animation. It's tempting visually, but a flip implies two faces existing in the same view hierarchy during the transition, and that's precisely the kind of transitional-frame risk research already named as a top concern — a hard cross-fade (shield out, card in, no shared transform where both are simultaneously composited) is the only transition that can't leak a frame by construction.

For the screen-by-screen build spec: Setup/Home is a `Form`-adjacent single scroll view — name entry rows first (each a `PlayerNameEntryRow` with inline validation, live-updating a count badge that goes red/blocks Start outside 2–6), ruleset picker below, corner toolbar items for History/Rulesets/Settings using SF Symbols (`clock.arrow.circlepath`, `slider.horizontal.3`, `gearshape`) sized at the 44×44 minimum tap target even though the glyph itself is smaller. Ruleset Editor is a sheet with `.interactiveDismissDisabled(hasUnsavedChanges)` plus a confirmation dialog on attempted swipe-dismiss, since "unsaved-changes warning" isn't a real state unless dismissal is actually interceptable. Active Game is the one screen that's really a state machine rendered as a view — I'd model it as a single root view switching on `engine.turnPhase: TurnPhase` (`.shielded`, `.confirmArmed`, `.revealed`, `.jokerCapture`, `.applyingRule`, `.handoffLocked`, `.gameOver`), with the shield state literally short-circuiting the render tree (an early return, not a ZStack overlay with opacity 0 that still exists underneath) so there's no world where the card view is instantiated and just visually covered — instantiated-but-hidden is a bug waiting to happen the moment someone adds a debug overlay or a11y inspector.

## Original Prompt

```
Build Kings — a multiplayer drinking game app for iOS (local pass-and-play).

Core mechanics:
- 2–6 players draw cards from a deck in turn order
- Each card (A–K, plus Joker) triggers a rule: e.g. "2 = You" (drinker picks someone), "5 = Guys", "K = King's Cup" (build communal cup with dares), "Joker = Make a rule"
- Players drink per the card rule; when the cup is empty or 10 turns pass, game ends with a recap
- Pass phone between players; each player sees only their own card until they pass

Key features:
- Local only (no server, no login)
- Customizable rules: edit card meanings before each game, save rulesets
- Game history: recap shows who drank most, funniest rule, timeline of rounds
- Sober mode toggle (default): uses "dare" language instead of drinking; activating drinking mode (17+ gate) swaps copy to drinking references but same mechanics

Tech:
- SwiftUI, local-first with CoreData for game history
- Phone must stay face-down while other player holds it; privacy shield until handoff
- Export game recap as a sharable image (name, date, player list, highlights)

Scope: MVP with Kings rules only (not Hearts, Asshole, or other variants). One ruleset bundled; let users customize before start.
```

**Claude (Frontend Engineer · Visionary) — Round 1**

Kings is the kind of app where the temptation is to treat the engine as "just some code behind the views," and I want to push back on that hard: the engine should be a genuinely separate, compiler-enforced product — a local Swift package that cannot physically import SwiftUI, CoreData, or UIKit — not just a folder convention that a future PR quietly violates under deadline pressure. That's the boldest, most valuable architectural bet available here, and it costs almost nothing to set up now.

Concretely, I'd split into three local SPM packages plus the app target itself. `KingsEngine` is pure Swift, zero imports beyond Foundation, and contains the entire game model: `Card`, `Rank`, `Ruleset`, `GameEngineState`, the `GameEngineReducer`, `TurnEvent`, `EndReason`, `GameSessionRecord`. `KingsPersistence` imports CoreData and `KingsEngine`, and owns the `.xcdatamodeld`, `PersistenceController`, and the two store protocols (`RulesetStore`, `GameHistoryStore`). `KingsServices` imports UIKit/Photos/CoreHaptics plus `KingsEngine`, and owns `ExportCoordinator`, `PhotoPermissionService`, `AttestationService`, `HapticsService`. The app target only imports these three and SwiftUI, and holds Screens/, Components/, DesignSystem/, and `AppRouter.swift`. The reason I want actual packages instead of grouped folders: Swift's access control at package boundaries means nobody can accidentally reach into `NSManagedObjectContext` from inside a turn-transition function three months from now without a compile error stopping them — the "engine is isolated" rule becomes structurally enforced instead of a convention everyone has to remember.

Inside the engine, I want to go one step further than "pure state machine" and make the core transition function a free, static, side-effect-free reducer: `GameEngineReducer.reduce(_ state: GameEngineState, _ event: GameEngineEvent) -> GameEngineState`. The `@Observable` `GameEngine` class itself becomes a thin wrapper holding `state` and exposing `send(_:)`, which just calls the reducer and republishes. This is what makes the rapid-tap idempotency test actually trivial to write and trivial to trust: five `send(.confirmReveal)` calls against the same starting `GameEngineState` is a pure function call with no timing dependency at all, not an integration test against an observable object's internal mutable fields. It also means the HUD, the recap builder, and even a future debug console could all replay the exact same event log through the same reducer and get byte-identical results — that's real, testable replay-fidelity, not just "we stored the mode per turn and hope the view renders it correctly."

For navigation I want one `NavigationStack` rooted at `SetupView`, driven by a typed `AppRoute` enum path (`.activeGame`, `.recap(RecapContext)`), with `RulesetEditor` and both `ModeSettings` contexts as sheets rather than path entries — this keeps "no resume of an in-progress game" trivially true by construction, since the path is just reset to empty on every cold launch and there's no scene-restoration code to accidentally reintroduce it later. `RecapContext` and `ModeSettingsContext` are the two enums that let one view serve two purposes, exactly as design_handoff specified.

For persistence, I'd keep the CoreData surface as narrow as the two store protocols and nothing else touches `NSManagedObjectContext` — not the engine, not the views. `PersistenceController` exposes `async` functions that internally hop to a background context via `perform`, and `RulesetStore`/`GameHistoryStore` return plain `KingsEngine` structs (`Ruleset`, `GameSessionRecord`), never managed objects, so a view or the engine never has to know CoreData exists. `HistoryDisplayStats` stays exactly what was ratified: a read-derived cache, rebuilt from `TurnEventEntity` on fetch, never an independent write path.

The one place I'd genuinely push past what's been said before: give the engine a lightweight internal event log independent of persistence — literally just `state.timeline: [TurnEvent]` accumulated in-memory during play — so that `GameSessionRecord` construction at game-end is a pure mapping function, `GameSessionRecord.init(from state: GameEngineState, endedAt: Date)`, with no additional bookkeeping needed. The recap, the funniest-rule ballot, and the persisted record are all just different views over the same array the reducer already built turn by turn. That's the detail I'd fight hardest to keep in the final spec: don't let recap-building logic live anywhere except as a pure function from `GameEngineState`/`[TurnEvent]`, because the moment it starts pulling from two different sources (live state plus some separately-tracked stats dictionary) is the moment a bug lets recap and reality disagree.

My biggest worry about my own proposal: three local packages is more Xcode-project ceremony than a single app target, and if the team building this isn't disciplined about it, "just add an import" is one tap away from breaking the isolation the whole architecture depends on. I think it's worth it anyway because the ratified requirement is "fully unit-testable, decoupled state machine," and folder conventions don't actually deliver that — compiler-enforced module boundaries do.

Testing strategy: `KingsEngineTests` is pure XCTest against `GameEngineReducer` directly — deck-exhaustion at every player count 2–6, cup-full-wins-simultaneous-tie, the five-rapid-`confirmReveal`-calls-advance-one-turn case, funniest-rule default-to-last-custom and empty-state paths, Joker text capture round-trip. `KingsPersistenceTests` runs against an in-memory `NSPersistentContainer`, covering save/load/delete for both stores and `schemaVersion` fallback decoding. `KingsServicesTests` covers the `ExportCoordinator`'s completion-mapping logic against a mocked `UIActivityViewController` result and a mocked `PHPhotoLibrary` authorization status, since that mapping logic is itself pure and doesn't need a real share sheet to verify. XCUITest covers Setup validation, Ruleset Editor save/invalid-mapping, History list/delete. Real export success/cancel/denied and shield-under-system-interruption remain a manual on-device checklist, as already agreed — no automated test can drive a real Face ID prompt or notification banner reliably.

**Claude (Frontend Engineer · Visionary) — Round 2**

Okay, Codex, first thing — your `RuleDefinition` with separate `soberText`/`drinkingText` per card is a real catch that I missed entirely in round 1. My `Ruleset.cardText: [RuleKey: String]` only has room for one string per card, which literally cannot represent "sober mode is a first-class voice, not a word-substitution pass" for anything except the bundled default ruleset — if a user forks the default into a custom ruleset and edits it, my model would silently lose whichever mode's text wasn't currently displayed. I'm adopting your two-field approach into `RuleDefinition`. But I want to be careful not to let that quietly reopen the Ruleset Editor's ratified UI, which design_handoff spelled out as "each row an inline-editable text field" — singular, not two fields per row. So the resolution I'd propose is: the domain model carries both `soberText` and `drinkingText`, the editor still shows one field bound to whichever mode is currently active when the editor is opened, and forking/editing a ruleset seeds both fields from the bundled defaults so an edit made in sober mode doesn't silently blank out or overwrite the drinking-mode text underneath it. That's a concrete, testable rule — "editing one mode's text never mutates the other's" — and it's exactly the kind of cross-mode-leakage bug this app has been paranoid about everywhere else.

While I was fixing that I found a real bug of my own from round 1: I had `cupFillDelta: Int` on `TurnEvent` doing double duty for both "how full is the communal cup" and implicitly "how much did this player personally drink," and those are not the same number. Cup-fill is deterministic and King-only — the cup is full exactly at the 4th King, full stop, so it doesn't need to be stored per-turn at all, it's just `timeline.filter { $0.card.rank == .king }.count` divided by 4, derivable on demand for both the live HUD and the recap. What actually needs its own field is a personal `unitsAssigned: Int` per turn (defaulting to 1) so "who drank most" in the recap is summed from real per-turn data instead of conflated with cup state. I'm dropping `cupFillDelta` entirely and adding `unitsAssigned`. Small thing, but it would've produced a recap stat that's just wrong.

# First-Party Apple Frameworks Cheatsheet (iOS 17+ / Swift 5.9+)

Build capable iPhone apps with **zero third-party dependencies**. For each framework: when to reach for it, the entry-point type, and a compilable, idiomatic snippet or the gotcha that bites people. ## General rules

- **Prefer these over SPM packages.** RevenueCat, Firebase Analytics, Lottie, Alamofire, etc. all have first-party equivalents good enough for most apps. Fewer deps = smaller binary, faster builds, no supply-chain risk, no privacy-manifest surprises.

## StoreKit 2 — In-App Purchase & Subscriptions

**Use it when:** you sell anything — consumables, non-consumables, or auto-renewable subscriptions. Replaces callback-based StoreKit 1 entirely.

**Start from:** `Product` (fetch + purchase) and `Transaction` (entitlements + history). No server strictly required — signatures are verified on-device via `VerificationResult`.

### The three things you must do

1. **Load products by ID** (IDs come from App Store Connect):

```swift
import StoreKit

let productIDs = ["com.app.pro.monthly", "com.app.pro.yearly", "com.app.coins.100"]
let products = try await Product.products(for: productIDs)   // async throws
```

2. **Purchase, then verify and finish the transaction:**

```swift
func purchase(_ product: Product) async throws -> Transaction? {
    let result = try await product.purchase()
    switch result {
    case .success(let verification):
        let transaction = try checkVerified(verification)
        await transaction.finish()          // REQUIRED — unfinished txns replay forever
        return transaction
    case .userCancelled, .pending:
        return nil
    @unknown default:
        return nil
    }
}

func checkVerified<T>(_ result: VerificationResult<T>) throws -> T {
    switch result {
    case .unverified(_, let error): throw error   // signature failed — do NOT grant
    case .verified(let safe):       return safe
    }
}
```

3. **Listen for `Transaction.updates` from app launch** — the single most-missed requirement. It catches renewals, Ask-to-Buy approvals, refunds, and purchases made on other devices.

```swift
// Start this in your App init / a top-level task and keep it alive for the app's lifetime.
func observeTransactions() -> Task<Void, Never> {
    Task.detached {
        for await update in Transaction.updates {
            guard case .verified(let transaction) = update else { continue }
            await grantEntitlement(for: transaction)   // your unlock logic
            await transaction.finish()          // REQUIRED — unfinished txns replay forever
        }
    }
}
```

### Checking what the user currently owns

Use `Transaction.currentEntitlements` (an async sequence). **iOS 18.4+ note:** the per-product `Transaction.currentEntitlement(for:)` is deprecated in favor of `Transaction.currentEntitlements(for:)`, because a user can hold multiple entitling transactions (e.g. own it *and* have Family Sharing access).

```swift
func hasActiveSubscription() async -> Bool {
    for await result in Transaction.currentEntitlements {
        guard case .verified(let transaction) = result else { continue }
        if transaction.productType == .autoRenewable && transaction.revocationDate == nil {
            return true
        }
    }
    return false
}
```

### Round 3


**Codex (QA & Risk · Closer) — Round 3**

I’m going to close this hard and not leave anything pretending it can be decided “later.” Codex, your round-2 move to a pure `GameEngineReducer` was the right base, but I think we should keep Claude’s round-2 refinements, because they fix exactly the bugs we’d otherwise ship: (1) `RuleDefinition` must be two strings (`soberText`, `drinkingText`) because mixed-mode mode-switching is a replay-fidelity problem, not a UI toggle; one string with runtime substitution will eventually drift and create hidden cross-mode leakage risk, (2) reducer effects should be explicit, i.e., `(state, effects)` so the engine stays pure while the wrapper handles haptics and terminal export-ready completion, and (3) privacy shielding needs a dedicated `PrivacyShieldCoordinator` path because the mid-session Mode/Settings sheet background/interruption path is not representable as a simple engine event.

I also think we should reject the package-splitting idea and pin a single `drinking-games-test` app target with folder boundaries only (`Domain`, `Engine`, `Persistence`, `Services`, `UI`, `Shared`), because this phase is a parity and determinism race, not a framework modularity exercise; extra package boundaries only add interface negotiation risk for parallel lanes. The contract says no resume of an in-progress game, one hot per-turn engine loop, and exactly one write at game end; those constraints already punish coupling. A single target with strict folder boundaries preserves simplicity and reduces breakage across lane boundaries.

For this phase’s concrete build plan, I’m locking in: start at Setup only, no in-game hub; engine is pure-reducer driven and hot-path-only; persistence boundaries are startup ruleset load + one save-at-end + history fetch/delete; all export/share states flow through one coordinator using `UIActivityViewController` completion to preserve the required state split; and shielding is handled structurally by lifecycle/scene events, not by relying on view-layer luck. On the data side, we keep the authoritative timeline in `TurnEvent` and derive recap aggregates from it so no separate cached metric can silently diverge. The only acceptable state persistence of game highlights is the immutable `GameSessionRecord` record with snapshot text per draw and per-turn mode, which makes replay-fidelity provable. `HistoryDisplayStats` if introduced is allowed as a cache only.

Failure modes we will explicitly test and fail-fast on are: rapid-tap confirm leakage (five confirm events in one runloop => exactly one draw), one-frame reveal leaks under background/rotation/notification/interruption, mode-gated copy switching not rewriting old recaps, end-reason collisions (`cupFull` beats `deckExhausted`), and export outcomes (success/cancel/permission-denied/failure) remaining distinct and visible. On the effects channel: think about what happens without it. The reducer processes `.confirmReveal` and the card drawn happens to be the 4th King. Something needs to fire a haptic buzz right at that moment, and something needs to trigger the one `GameHistoryStore.save` call. If the reducer just returns new state, the `@Observable` wrapper has to diff `oldState.timeline.count` against `newState.timeline.count`, or compare `oldState.phase` against `newState.phase`, to reverse-engineer "oh, a King must have just been drawn" or "oh, we must have just hit game-over." That's exactly the kind of implicit, timing-shaped logic that's fragile under rapid taps — the whole reason we wanted a pure reducer in the first place was to kill that class of bug. Returning `(state, effects)` means the wrapper never has to infer anything, it just dispatches whatever the reducer already decided happened. It costs one extra field in a tuple. I really don't see the case against it, so if you've got one, say it directly rather than the type just not showing up next time.

On `PrivacyShieldCoordinator` — walk through the actual failure case with me: someone's mid-session, taps the small gearshape in the HUD to open Mode/Settings in its `.midSession` context, and while that sheet is up, the app gets backgrounded by an incoming call. Whoever built this needs one thing watching scene lifecycle independent of whatever's currently presented, forcing a hard "nothing revealed can render" flag regardless of sheet state, then letting the sheet's own dismiss animation happen safely on top of an already-shielded background. **Architecture overview:** A single SwiftUI iOS 17+ app target — no third-party dependencies, no SPM package split. The game engine is a pure-Swift, side-effect-free value-type reducer (`GameEngineReducer.reduce(state, event) -> (state, effects)`), wrapped by a thin `@Observable @MainActor` `GameEngine` class that republishes state and dispatches the declarative effects the reducer returns (haptics, the one game-end persistence write). **Architecture overview:** A single SwiftUI iOS 17+ app target — no third-party dependencies, no SPM package split. The game engine is a pure-Swift, side-effect-free value-type reducer (`GameEngineReducer.reduce(state, event) -> (state, effects)`), wrapped by a thin `@Observable @MainActor` `GameEngine` class that republishes state and dispatches the declarative effects the reducer returns (haptics, the one game-end persistence write). This keeps the "five rapid confirm-taps advance exactly one turn" idempotency guarantee a one-line pure function test rather than a timing-dependent integration test, and makes every `GamePhase` — including edge cases like a Joker draw one card before deck exhaustion — directly constructible in a SwiftUI Preview with no need to actually play a game to reach it. Engine isolation from CoreData/SwiftUI/UIKit is enforced by a build-phase script that fails the build if any file under `Engine/` imports those frameworks — a compiler-adjacent guarantee without the ceremony of real package boundaries. Privacy shielding is handled independently of the engine by a root-level `PrivacyShieldCoordinator` reacting to `SceneLifecycleEvent`s, specifically because a sheet like Mode/Settings-in-`.midSession` can be on screen during a backgrounding event and isn't part of engine state at all — the coordinator forces a hard "nothing revealed can render" flag regardless of what's presented on top, and separately forwards the interruption into the engine so game state stays consistent too.

**File/module layout:** One target, `drinking-games-test`, organized as `Domain/` (Card, Rank, Suit, RuleKey, RuleDefinition, Ruleset, GameMode, EndReason, TurnEvent, TurnDraw, GamePhase, GameSessionRecord, FunniestRuleResult), `Engine/` (GameEngineState, GameEngineEvent, GameEngineEffect, GameEngineReducer, GameEngine, Deck — import-restricted by the build script above), `Persistence/` (PersistenceController, CoreData model, RulesetStore/GameHistoryStore implementations, PersistenceError), `Services/` (ExportCoordinator, PhotoPermissionService, AttestationService, HapticsService), `UI/` (Setup, Ruleset Editor, Active Game, Recap, History List, Mode/Settings, Onboarding screens plus shared components), and `Resilience/` (PrivacyShieldCoordinator, SceneLifecycleEvent).

**Data models:** Standard 53-card deck (`Deck.standard53`) of `Card` (`Rank?`, `Suit?`, `isJoker`), rules keyed by `RuleKey` with dual-voice `RuleDefinition` (`soberText`/`drinkingText` — never a single shared string, so editing one mode's copy can never blank the other). `TurnEvent` is the single authoritative per-draw record — mode-at-turn, a frozen `ruleTextSnapshot` so later ruleset edits never rewrite old recaps, and `unitsAssigned` (deliberately mode-neutral naming, not "drinksDelta," since sober is the app's default voice). Cup-fill state is never stored — it's always derived as Kings-drawn/4, both live and in recap. `GameSessionRecord` stores only what can't be derived (id, timestamps, players, rulesetName, endReason, the timeline, the captured `funniestRule` ballot result) and computes everything else (`kingsDrawnCount`, `cardsRemainingAtEnd`, `mostDrinksPlayerIndex`) from the timeline on read, so a recap can never structurally diverge from what actually happened.

**Persistence strategy:** CoreData with `schemaVersion` from the very first migration, background-context writes with `viewContext` auto-merge so persistence latency never touches the shield/reveal hot path. Exactly three write boundaries: ruleset load at Setup, one `GameHistoryStore.save` at game-end (triggered by the reducer's `.gameDidEnd` effect, never called speculatively), and history list/delete. `PersistenceError` is a concrete `LocalizedError` (`.notFound`, `.loadFailed`, `.saveFailed`, `.deleteFailed`, `.migrationFallbackUsed`) so failure states can show real, distinct copy instead of a generic string. Export uses `ImageRenderer` plus `UIActivityViewController` wrapped for SwiftUI (never plain `ShareLink`, which has no completion callback), with Photos `.addOnly` authorization requested before presenting, mapping completion to the four required `ExportOutcome` states.

**Testing strategy:** `GameEngineReducer` gets pure unit tests — deck exhaustion at every player count 2–6, cup-full-wins-simultaneous-tie, idempotent `confirmReveal` (five rapid calls in one runloop tick advance exactly one turn), funniest-rule default-to-last-custom and empty-state paths, all callable directly against plain value types with no mocking.

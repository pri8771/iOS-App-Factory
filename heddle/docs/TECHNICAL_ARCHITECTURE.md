## Final Output

**Screen inventory (three screens, not four):**
- **Puzzle** — the app's root view. Launches directly into it; no splash, no home screen, no separate tutorial screen (the first puzzle IS the puzzle screen, just seeded easy).
- **Hub** — reached via a small, low-contrast spool icon in the puzzle screen's corner (44pt target). Shows current-puzzle status, a single static "yesterday's loom: solved in N moves" row (not a list/archive browser), and one inert Loom Pass button.
- **Settings** — one tap further from Hub. A leaf screen: read-only Reduce Motion note (mirrors system, no duplicate toggle), and a data-reset action.

**Primary user flow:** App launch → Puzzle screen in generating state (modeled as real async, not assumed-synchronous) → ready state with loom and thread ends visible → user drags or tap-chains a thread, with only cell-boundary crossings committing state → all cells covered, no crossings → solved state (haptic pulse + visual bloom) → celebration **holds** until the user taps a "begin next loom" affordance — no timer-driven auto-advance into the next puzzle.

**State model per screen:**
- *Puzzle*: generating, ready-to-play (partial threads drawn during an active trace are this same state, not a separate one), solved/celebrating, generation-failure-with-fallback (styled as a warm, in-voice apology — "here's one we had ready" — never a system-error look).
- *Hub*: one steady state built from already-local data; no loading spinner needed since nothing here is async.
- *Settings*: normal state and a reset-confirmation state that's visually identical to (and reused as) the corrupted-store recovery screen from `detailed_discussion` — same calm "starting fresh" message and layout in both cases, built once.

**Visual direction:** Natural-dye palette (madder red, indigo, walnut brown, undyed linen/oat background, sage) instead of app-UI or neon/gem colors. System rounded font family rather than a custom typeface (warmer than default grotesque, zero font-licensing risk for MVP). Motion language: threads carry a soft "pull-taut" tension while dragged (not bouncy/rubber-band); the solve moment reads as the loom settling and catching light, not a generic checkmark/confetti — this animation gets disproportionate craft time since it carries most of the brand's tactile promise. SF Symbols for all chrome (spool, gear, back nav); custom rendering reserved only for the loom/thread surface itself.

**Accessibility notes:** Each cell gets a plain-language VoiceOver label (row, column, content — e.g., "row two, column three, amber thread end, diamond cap"), exposed as a logically ordered accessibility container rather than a flat element soup. Tap-chain interaction is redesigned around how VoiceOver actually works, not spatial tapping: double-tap a thread origin to start a chain, only legally-reachable next cells expose an "extend thread here" custom action, and "cancel this thread" is available as a custom action on the active origin throughout — this is what actually satisfies the cancel-action requirement locked in `detailed_discussion`, not just a gesture at it. Undo stays always-visible but visually recessive (small thread-spool-unwinding icon, 44pt) in a quiet bottom strip alongside the Hub spool icon.

**Open item carried into build/testing, not blocking this phase:** whether the deliberately chrome-free puzzle screen still makes undo and Hub-navigation easily discoverable in practice, especially for VoiceOver users relying on swipe/rotor discovery — flagged as the first thing to listen for in testing, not a design question left unresolved here.

It didn't just restate the design_discussion outcome in fancier words; it did the actual conversion from adjectives to numbers that this phase exists for: real grid dimensions derived from the 44pt/375pt-width math (5×5 gentle, 7×7 standard, nothing bigger fits without breaking hit targets), a fixed 5-color/5-shape identity token (circle/diamond/square/triangle/hexagon) that closes the colorblind-safety requirement with an actual mapping instead of a promise, hex values for every palette color in both light and dark mode with an explicit WCAG AA-per-mode check (not just inverted lightness), a concrete spacing/radius scale, and a 14-item component inventory specific enough to check off one by one.

It also resolved two things that were genuinely still open. First, the Canvas-vs-accessibility tension nobody had named yet: painted Canvas content is invisible to VoiceOver, so the board has to be a hybrid — Canvas/Path for visual thread rendering with a transparent overlay grid of real SwiftUI accessibility elements carrying the labels and custom actions. That's not a restatement of "make it accessible," it's the specific architecture that makes the double-tap/extend-thread/cancel-thread model from last phase actually implementable. Second, the persistence-timing risk: committing every cell-boundary crossing straight to SwiftData on the main actor would stutter a fast drag, so writes get coalesced with a short debounce plus an unconditional flush on scenePhase backgrounding — satisfying the force-quit requirement without costing the "tactile" feel it's supposed to protect.

It closed the one gap the previous quality gate explicitly flagged (Hub's empty state for a first-launch user with no solved puzzle yet) with a concrete line of copy rather than leaving it for build to invent.

It names its own risk honestly: the outcome still depends on whoever implements it having real SwiftUI animation/Canvas craft, since so much of "warm and tactile" lives in timing and haptic pairing that tokens alone don't guarantee — flagged as a build-quality risk to watch, not a reason to reconsider the decision.

Nothing here contradicts any earlier phase — it's built directly on top of the three-screen architecture, the four puzzle states, the no-fail-state/no-auto-advance rules, and the accessibility model locked in `design_discussion`, and it satisfies all three acceptance checks for this phase: a SwiftUI builder could implement directly from this (concrete grid math, exact hex values, named components, spring parameters), nothing depends on a hidden decision, and the direction reads as premium and specific to Heddle rather than generic puzzle-app chrome.

## Final Output

Same pattern as the whole build so far — Codex stayed in cooldown, so it's a single voice, but this pass does the real work of an iOS architecture review rather than just re-affirming design decisions: it named concrete implementation risks in concurrency, persistence, and permissions that no prior phase had actually pinned down, and closed all six required outputs (SwiftUI architecture, framework choices, persistence plan, permissions/privacy plan, dependency decision, testing implications) with checkable rules.

**SwiftUI architecture:** `@Observable` view models (not `ObservableObject`), one per screen (`PuzzleViewModel`, `HubViewModel`, `SettingsViewModel`), each `@MainActor` since they own UI state. The puzzle generator is explicitly *not* run via a bare `Task {}` on the main-actor view model — that would inherit the main actor and silently block it during the compute-heavy solve. It's isolated as its own `actor PuzzleGeneratorEngine` (or a `nonisolated` `Sendable` pure function), called as `Task { await engine.generate(...) }`, so "generating" is a genuinely async state, not a decorative pulse animation over a main-thread stall. A thin repository-style `SwiftDataStore` wrapper sits between view models and `ModelContext` so the debounce/coalesce-write and corrupted-store-fallback logic each have one implementation, not one per view model.

**Persistence:** One flat `PuzzleRecord` SwiftData model — UUID identity, a few flat queryable fields (difficulty tier, solved-at, move count, created-at), plus the full materialized board/thread state encoded as a single versioned `Codable` blob rather than a relational graph of per-cell entities. This makes the debounced coalesced write (locked in `design_handoff`) trivial (overwrite one field, not diff a graph) and sidesteps SwiftData's relationship-model rough edges. Since "cloud-ready later" has been a standing requirement since `per_app_product_brief`, this phase names the actual CloudKit-compatibility tax and locks it now: no unique constraints on any attribute (enforce uniqueness in app logic instead), every property optional or defaulted, no non-optional relationships — this is the real mechanism behind "doesn't preclude sync later," not just a stated intention.

**Permissions/privacy:** Zero runtime permission requests of any kind (no camera, photos, notifications, location, contacts, microphone) — Info.plist should carry no usage-description keys at all, and any that appear in the built target is a bug to investigate, not a stray to ignore. This sharpens the zero-network/zero-analytics stance from `detailed_discussion` into something literally checkable by reading the shipped Info.plist. No BackgroundTasks framework usage — the only lifecycle hook is the scenePhase-triggered flush-on-background write, which is app-lifecycle handling, not a background-execution capability, so no unused background-mode entitlement should end up on the target.

**StoreKit/monetization architecture:** No StoreKit API calls wired in this slice at all — not even behind a feature flag, since a flag can be flipped by accident but code that doesn't exist can't be. Only a local `Entitlement` model exists; the Loom Pass "coming soon" sheet is pure UI with zero StoreKit imports in its call path. The entitlement model's shape should anticipate StoreKit 2 (`Product`/`Transaction`/`AppStore.sync`, async/await) so a future `Transaction.currentEntitlements` listener can set it without restructuring.

**ML/AR:** Correctly and deliberately absent — no natural surface for either (no physical space to augment, no natural-language content to generate), and forcing one in would violate the parent prompt's own "complexity must serve real value" rule. The generator is constraint-satisfaction/backtracking, which is the right tool for a provable-solvability guarantee, not a missed bonus-points opportunity.

**Dependencies:** Zero third-party packages. Everything needed (SwiftUI, SwiftData, Core Haptics, StoreKit 2, SF Symbols, Swift Testing, XCUITest) is first-party, so the permissive-license policy is satisfied by having nothing to evaluate — locked as a hard constraint, not a default.

**Testing:** Generator solvability (500+ runs/tier), thread-overlap retraction, and thread-granularity undo are pure logic, tested fast with Swift Testing (`@Test`/`#expect`), no simulator required. XCUITest can catch structural accessibility regressions (missing labels, disappeared custom actions) but explicitly does not substitute for a human VoiceOver pass or Core Haptics verification — both stay hard manual gates per `app_features`, and this phase is explicit that no amount of automated coverage should be read as relaxing that.

**One open risk named, not resolved:** SwiftData's relative immaturity around migrations is part of why the board is flattened to a blob rather than a queryable relational structure — if a future phase wants to query individual cells/threads (e.g., for stats), that's a real schema migration later, flagged now as a named future cost rather than a surprise.

All three acceptance checks are met: the app can be built without architecture gaps (concurrency, persistence, permissions, monetization, and testing all have concrete mechanisms, not aspirations), ML/AR absence is explicitly justified rather than silent, and the zero-dependency decision trivially satisfies the license-policy check. Nothing here contradicts any earlier phase — it's implementation detail underneath decisions already locked in `design_handoff` and `detailed_discussion`.

## Final Output

**Architecture overview:** SwiftUI + `@Observable` view models, one per screen (`PuzzleViewModel`, `HubViewModel`, `SettingsViewModel`), each `@MainActor`. The puzzle generator is isolated as `actor PuzzleGeneratorEngine` conforming to a `PuzzleGenerating` protocol, so calling it via `Task { await engine.generate(...) }` genuinely hops off main rather than silently inheriting the main actor. Persistence runs through a `@ModelActor`-isolated `SwiftDataStore` (conforming to `SwiftDataStoring`) with its own background `ModelContext`, giving coalesced debounced writes and an unconditional flush on `scenePhase` backgrounding as a natural consequence of actor-serial execution rather than hand-rolled cancellation logic. The interaction layer is unified: both the drag gesture and the VoiceOver tap-chain custom actions funnel through one `ThreadTraceController.commitStep(...)`, so legality checks, thread-overlap retraction, and completion detection exist in exactly one place and can never diverge between input modes. Live in-flight touch position stays `@State`-local to `PuzzleView` (never touches the model); only committed cell-boundary crossings reach `PuzzleViewModel.board`.

**File/module layout (single app target, no local SPM):**
- `HeddleApp/` — app entry point, root scene
- `Models/` — `Position`, `ThreadColor`, `CapShape`, `Direction`, `CellContent`, `Thread`, `PuzzleDifficultyTier`, `PuzzleBoard`, `PuzzleSummary`
- `Generation/` — `PuzzleGenerating`, `PuzzleGeneratorEngine`, `GeneratorError`, `FallbackPuzzleProvider`
- `Persistence/` — `PuzzleRecord`, `Entitlement` (`@Model` types), `SwiftDataStoring`/`@ModelActor` store, `recoverFromCorruptStore`
- `Features/Puzzle/`, `Features/Hub/`, `Features/Settings/` — each screen's view model + views from the `design_handoff` component inventory
- `DesignSystem/` — color/spacing/type/shape tokens as constants, so no view hardcodes a hex value
- `HeddleTests/` — Swift Testing suites (generator solvability, `ThreadTraceController`, undo)
- `HeddleUITests/` — XCUITest accessibility-structure smoke tests

**Data models:** `Position`, `ThreadColor` (with fixed `capShape`), `Direction`, `CellContent`, `Thread` (path holds only committed cells), `PuzzleDifficultyTier` (gentle=5×5, standard=7×7), `PuzzleBoard` (materialized puzzle; `seed` is provenance-only, never used to reconstruct state), `PuzzleSummary` (backs Hub's yesterday row). SwiftData-backed: `PuzzleRecord` (flat fields + single versioned `Codable` blob, no unique constraints, all optional/defaulted per the CloudKit-compatibility tax) and `Entitlement`.

**Persistence strategy:** One flat `PuzzleRecord` per puzzle, board/thread state encoded as a single blob rather than a relational graph. Writes go through the `@ModelActor`-isolated store: committed steps schedule a coalesced write, `flushPendingWrites()` is awaited unconditionally on backgrounding, and a corrupted/unreadable store at launch triggers `recoverFromCorruptStore` into a fresh container surfaced via the shared reset/starting-fresh screen.

**Testing strategy:** `ThreadTraceController` gets full pure-logic coverage in Swift Testing (legal extension, illegal move rejection, retraction of another thread, completion detection) against `PuzzleBoard` values directly — no view or gesture involved — specifically so drag and tap-chain are provably calling into identical behavior. Generator solvability: 500+ runs per tier via the `PuzzleGenerating` protocol (injectable, no concrete-singleton dependency). Undo and thread-overlap retraction are likewise pure state-machine tests. XCUITest covers structural accessibility regressions only; VoiceOver navigation and Core Haptics stay hard manual gates on physical device, unchanged from `ios_architecture_review` and `app_features`.

**Claude (QA & Risk · Skeptic) — Round 1**

The thing that worries me most going into this phase is that every prior phase has locked the *shape* of the mechanic — drag or tap-chain, no crossings, generate-then-strip — but nobody has actually specified the fiddly interaction rules that a real implementation needs before a single line of gesture code gets written. Flow-style tracing games have a well-known ambiguous case: what happens when a player drags a new thread's path directly over a cell already occupied by a different color's already-placed thread? Standard behavior in this genre is that the older thread retracts back to its endpoint, but that's not stated anywhere in this transcript, and if it's left implicit, whoever builds it will guess, and "guess" is exactly what phase rule 2 in prompt_contract says we're not allowed to do with things this load-bearing. I want that pinned down as a resolved requirement: dragging over an existing different-color segment clears that thread back to its endpoint rather than blocking the gesture or crashing.

Second gap in the same family: the VoiceOver discrete tap-chain path has a "start a thread, tap successive cells" interaction, but nobody has defined how a VoiceOver user cancels or restarts a thread mid-chain if they tap the wrong cell. A sighted user dragging can just lift their finger and try again visually; a VoiceOver user chaining taps has no equivalent low-cost "oops" gesture unless one is explicitly designed. If the only recovery path is the global undo, that's a worse experience for VoiceOver users than for sighted ones, which is a real accessibility parity failure even if it technically clears a checkbox. I'd want a "cancel current in-progress thread" action explicitly in the VoiceOver interaction model, not inferred from "undo exists."

Third, and this one's been sitting quietly under everyone's assumption of color as the primary differentiator: color-only thread identification is a real accessibility problem independent of VoiceOver — colorblind users are a meaningfully large population, and a puzzle genre literally built around match-3-style color distinction (which this app is explicitly trying not to be, but still uses color for thread identity) needs a secondary channel, like distinct end-cap shapes or textures per color, not hue alone. This hasn't been raised in any prior phase and I think it should be locked now rather than discovered during a design review.

On persistence, the "restore exact thread state after force-quit" requirement sounds solved but has a sharper edge underneath it: are we persisting the generated puzzle's full board definition, or just a random seed plus "regenerate from seed when needed"? If it's seed-based and the generator algorithm ever changes between app versions — which it will, the moment anyone tunes difficulty or fixes a generation bug — replaying an old seed through new generator code can silently produce a different puzzle than what the user was looking at before they force-quit. That's a data-integrity risk worth naming explicitly with a concrete mitigation: persist the actual materialized puzzle (cell contents, endpoints, in-progress thread paths) as data, and treat the seed as provenance metadata only, never as the mechanism for reconstructing board state on resume.

A related but separate edge case: what exactly is "committed" state during an active drag? If someone is mid-gesture, finger still on screen, and gets a phone call or swipes to Control Center, does the app capture the trace up to the last fully-entered cell and drop the ephemeral touch position, or does it try to save literal in-flight finger coordinates? I'd lock this as: only cell-boundary-crossing commits count as state; anything between commits is transient view state, safe to lose, and system-interrupted gestures (calls, notifications, multitasking gestures) should cleanly revert to the last committed cell rather than leaving a half-drawn thread in an undefined state.

A few more edge cases worth putting on the record rather than letting build discover them: a corrupted or unreadable SwiftData store at launch (disk full, iOS storage pressure, a bad migration) needs a defined fallback — start fresh with a visible but calm message, never a crash, and never a silent data loss the user can't tell happened. Reduce Motion needs an explicit fallback for the solve "bloom" animation, since that's the single most-invested moment in the whole app and it can't just be skipped wholesale for accessibility users — it should degrade to a simpler cross-fade plus the haptic (haptic isn't motion, it stays). Dynamic Type at largest accessibility sizes needs a stated rule for the puzzle grid specifically: the grid itself is graphical, not text, so it shouldn't reflow with type scale, but any surrounding labels/instructions must scale without breaking the 44pt targets or overlapping the board — this needs to be tested at the largest accessibility text size, not just "Large."

On privacy, I want a sharper stance than "no backend" — I'd have this phase assert explicitly: zero network calls of any kind in this build, no analytics SDK, no crash reporter beyond what's opt-in and Apple-native, no data ever leaves the device, because that's cheap to guarantee now and expensive to audit for later once someone's added "just one small analytics call" during build. That closes the gap the quality gate flagged back in prompt_contract about privacy being named only by inference.

Two risks I'd add to the register with concrete mitigations rather than leave as vague worries: first, stubbed StoreKit product identifiers sitting in the build create a real risk of accidental App Store submission with fake IDs if this MVP ever gets archived and shipped without someone remembering they're stubs — mitigate by gating any StoreKit calls behind a clearly-named debug/feature flag that's off by default, not just "we'll remember." Second, the generator's solvability guarantee is currently a manual "run it a few hundred times and eyeball it" validation criterion from the last phase — that's not a real regression safety net once the code changes. I'd want this converted into an actual repeatable automated check (a unit test that runs the generator across its locked difficulty parameters some large N times and asserts solvability programmatically) so it's a permanent, re-checkable gate rather than a one-time manual pass that nobody re-runs after the next refactor.

Final assumption I want on record, since it's a place scope could quietly creep back in: data import/export and any archive-browsing are explicitly out of this build (already true per next_steps_small's cut list), and I don't think that contradicts the parent prompt's "data import/export" depth requirement being satisfied later — it just needs to stay named as a deferred exclusion here so nobody treats its absence as an oversight during the architecture phase.

So I'm locking two concrete tiers: a 5×5 "gentle" grid for the seeded first-run puzzle and as the permanent easy tier, and a 7×7 "standard" tier as the only other option in this slice. Concretely: madder red → circle, indigo → diamond, walnut brown → square, sage → triangle, ochre → hexagon. That mapping needs to be fixed and consistent across every screen that shows a thread, including the inert Hub summary and any future share image, so it should go in as a real token, not a per-screen art decision.

On rendering: the loom board can't be built as one `Canvas` with accessibility bolted on as an afterthought, because `Canvas` content is fundamentally invisible to VoiceOver — there's no free accessibility tree from painted pixels. The only way the tap-chain model from the design phase actually works is a hybrid: `Canvas` (or a stack of lightweight `Path`-based shapes) handles the visual thread rendering for performance, and a transparent grid of real SwiftUI views — one per cell, sized and positioned to exactly match the visual grid — sits on top carrying `.accessibilityElement`, the plain-language label, and the custom actions ("extend thread here," "cancel this thread"). That overlay grid is not optional chrome, it's the actual accessibility implementation, and it needs to be specified as its own component so nobody tries to retrofit accessibility onto raw Canvas drawing later and discovers it's structurally impossible.

Persistence timing is the other thing I want nailed down before anyone writes gesture code. The rule from earlier phases is that only cell-boundary commits are durable state — good, that's the right rule — but a fast, confident drag across a 7×7 board can cross several cell boundaries in well under a second, and if each commit triggers a synchronous SwiftData save on the main actor, the drag will visibly stutter, which is the exact opposite of "warm and tactile." So: the in-memory model updates synchronously and instantly for the UI on every commit, but the SwiftData write is dispatched as a coalesced background write — last-commit-wins, fired via a short debounce (150-200ms) or on gesture-end, whichever comes first, plus an unconditional flush on scenePhase transitioning to background/inactive so a genuine force-quit never loses a committed cell. This satisfies the force-quit-mid-puzzle requirement without turning every drag into a disk-I/O-bound animation.

## Final Output

**Screen inventory**
1. **Onboarding** — single screen, no carousel, no permission asks. States in three plain sentences: you add neglected personal projects, the app resurfaces one at a time, each week you act/snooze/release it. Ends by opening Project Record in create mode.
2. **Project Record** (create + edit modes, one screen) — title, what it is, why it mattered, why it stalled, optional smallest next step, last-touched date (defaults to today, future dates rejected/clamped). Edit mode adds: adjusting/clearing an existing snooze directly, and a visually subordinate delete action (confirmation copy changes specifically when deleting the currently-surfaced project, warning that another project may immediately take its place).
3. **Home** — the product. Exactly one of: surfaced project, no projects yet, projects exist but none eligible (plain-language reason + next-eligible date), everything released. The surfaced state must answer five things without a tap: what it is, why it mattered, why it stalled, the smallest next step, and why it surfaced now — all five treated as co-equal card content, not a system disclaimer bolted on. One quiet secondary-navigation affordance (not a tab bar) reveals Your Projects / History / Settings.
4. **Resolution sheets** — Act, Snooze, Release, presented as modal sheets on top of Home, never auto-advancing. Snooze shows current snooze count and the exact resulting next-eligible date before confirming, offering only the three fixed intervals.
5. **Your Projects** — plain, stable-ordered list (alphabetical or creation order), visually quieter than Home, existing solely as a correction surface to reach Project Record in edit mode. No filters, no sort controls, no affordance implying "pick what surfaces next."
6. **History** — uncapped chronological ledger. Each row shows the thread-state glyph, decision type, date, and note. Static, no per-row animation.
7. **Settings/About** — the ritual re-explained in plain language, one non-blocking line of premium-intent copy, nothing else load-bearing.

**Primary user flow**
First launch → onboarding (one screen) → Project Record in create mode → Home shows either the first surfaced project (cold start) or "none eligible yet" with a precise explanation. Return visits → open app → Home shows the same still-unresolved surfaced project, or a newly surfaced one if the pacing window passed → resolve via Act/Snooze/Release sheet → Home reflects the ritual is closed for now and states when the next resurfacing can happen. Separately, at any time: reach Your Projects or History via the secondary-navigation affordance to correct a record or review the ledger — this path is deliberately one extra tap and visually quieter, never competing with Home.

**State model per screen**
- Onboarding: single explainer state, plus bootstrap-failure error if initial store load fails.
- Project Record: idle/empty, validating, saving, save-failed, saved; edit mode adds viewing, editing, confirming-delete (forked into ordinary-entry copy vs. deleting-the-surfaced-project copy).
- Home: boot/loading, surfaced, no-projects, none-eligible-yet, all-released, corruption/store-failure (recovers by re-deriving the single surfaced project from the decision log or falling back to the correct empty state).
- Each resolution sheet: idle, confirming, saving, save-failed, saved.
- Your Projects: empty, populated, store error.
- History: empty, populated, load error.
- Settings: static.

**Visual direction**
Warm, paper-and-ink editorial tone — off-white/warm-gray surfaces, near-black ink text, one muted accent reserved for interactive elements only, never for status meaning. Literary/humanist typeface for the project's own words (why it mattered, why it stalled); standard system type for UI chrome. The surfaced card reads like a dossier/envelope opened on the table — generous whitespace, no card-stack competition. Motion is slow and settling, never snappy, no confetti or streak/reward cues.

The thread motif is the app's core status language, replacing color-coding: four fixed `Shape`/`Path`-based states — frayed-and-fading (dormant, a couple of discrete fraying tiers, not continuous day-count interpolation), taut (acted), coiled (snoozed), knotted (released) — rendered as a small glyph (roughly an inch) near the dormancy line on the Home card and as a small icon in each History row. Built from core SwiftUI drawing primitives only, no textures, no per-project variation, no particle effects. Only the Home card gets an entrance transition when its state changes; History rows are fully static, no animation. If this can't be built as four plain bezier-curve `Path`s in a bounded effort, the agreed fallback is SF Symbols with distinct silhouettes instead.

**Accessibility notes**
Every thread-glyph state needs a VoiceOver label that states the status in words ("dormant, untouched for 47 days," "resolved: acted on," "snoozed until August 12," "released") since the custom shape carries no inherent semantic meaning. Act/Snooze/Release controls need full labels and hint text that distinguish them by consequence, not just name, and color is never the sole differentiator among them or between Release and any other outcome. Dynamic Type must be tested specifically on the Home surfaced card at largest accessibility sizes, with short, predictably-stacking sections rather than dense paragraphs. Every major state (all four Home states, each resolution sheet, corruption recovery) must expose a plain-language one-sentence summary that works both visually and as an accessibility label.

## Final Output

**Design handoff decision:** This document is the primary, authoritative design handoff — a SwiftUI builder works directly from it, not from external mockups. External Claude Design is optional and subordinate: Codex's prompt (below) is preserved as a ready-to-use artifact for a future moment where actual marketing/App Store screenshots are wanted, but it never gates or overrides anything in this spec. Reasoning: this app's premium feel lives in behavior a static mockup can't represent — Dynamic Type reflow on text-heavy content, four programmatic `Path`-based thread glyphs, and a light/dark adaptive token system — so a mockup-first pipeline would just create a picture that has to be re-translated back into the same tokens/geometry this document already specifies precisely.

**Screen inventory (7 screens, no tab bar, Home is sole center of gravity):**

1. **Onboarding** — one screen, three plain sentences (you add neglected projects; the app resurfaces one at a time; each week you act/snooze/release), primary button "Add your first project," no permission asks, no carousel. States: normal first-run, bootstrap-failure if local store load fails.

States: idle/empty, validating, saving, save-failed, saved; edit adds viewing, editing, confirming-delete (two copy variants).
3. **Home** — exactly one of four states: no projects yet (invitational, one-sentence explainer, clear add-project button), none eligible yet (names the blocking rule in plain language plus exact next-eligible date), one surfaced project, or everything released (dignified, not barren). 4. **Resolution sheets** — Act, Snooze, Release as modal `.sheet`s over Home, never auto-advancing. Act: one field/compact text area for "what will you do next," explains this resets last-touched. Snooze: exactly three fixed intervals, always shows current snooze count and the exact resulting next-eligible date *before* confirming. Release: optional closure note, copy frames it as an honest ending. Each sheet: idle, confirming, saving, save-failed, saved.
5. **Your Projects** — plain, stable-ordered (alphabetical or creation order) correction surface only. No filters, sort, badges, or "pick what surfaces next" affordance. Tapping a row opens Project Record in edit mode. States: empty, populated, store error.
6. **History** — uncapped chronological ledger. Each row: thread glyph, decision type, date, note/next step. Fully static, no per-row animation. States: empty, populated, load error.
7. **Settings/About** — ritual re-explained in plain language, privacy/local-first/backup-inclusion note, definition of "touched," exactly one non-blocking line of premium-intent copy, nothing else load-bearing. Static.

**Design tokens:**
- *Color (adaptive, same semantic names in light and dark, different values):* `surfaceBackground`, `cardSurface`, `inkPrimary`, `inkSecondary`, `divider`, `destructiveText`, `threadGlyphStroke`, and exactly one accent family with three tonal steps — `accentInteractive`, `accentInteractivePressed`, `accentInteractiveSubtle` — all one moss/slate hue, used only for tappable affordances, never for status meaning. Light mode: warm parchment background, slightly darker card, near-black ink. Dark mode: warm dark-gray/near-black background and card, warm light ink — same paper-and-ink feeling at night, not an inverted escape hatch.
- *Typography:* `Font.system(.body, design: .serif)` for the user's own written content (why it mattered, why it stalled, etc.); standard system font for all UI chrome, navigation, and metadata. Both scale natively with Dynamic Type.
- *Spacing:* 4pt-based scale — 4/8/12/16/24/32.
- *Thread glyph geometry:* fixed 28×28pt frame, 2pt stroke. Taut: one near-straight line, barely-there sag, single quadratic curve. Frayed: same base line with a small fixed number of short diagonal ticks fanning off one end, in discrete tiers (e.g., <30 days: light fraying; 30–89 days: moderate; 90+: heaviest), no continuous interpolation. Coiled: simple ~1.5-loop spiral from a couple of arcs. Knotted: a self-crossing loop (overhand-knot silhouette) from two overlapping curves. Named SF Symbols fallback (used now if custom paths don't read as intentional quickly, not decided later under pressure): distinct-silhouette symbols chosen for shape difference first, not semantic precision — a diagonal/repaired-line symbol for taut, a broken/frayed-line symbol for dormant, a circular-arrows symbol for coiled, a knot/tied-ribbon symbol (or a clearly-distinct generic glyph if none exists) for released, explicitly not styled to look like a success checkmark.

**Component inventory (nine pieces, shared naming):** `ThreadGlyph` (status enum → Path + generated VoiceOver label, single source so visual and label can't drift apart), `SurfacedCard`, `RitualEmptyState` (parameterized across the three non-surfaced Home states), `DateChip` (shared by "untouched for X days," "next eligible on Y," and the Snooze sheet's preview — one component, not three copies), `ResolutionSheetShell` + three content variants (Act/Snooze/Release), `ProjectRecordForm` (create/edit via flag), `ProjectRow`, `HistoryRow`, `SecondaryNavMenu`.

**State ownership:** one `@Observable` store owning the project list and append-only decision log, injected at the root; every screen reads/displays store state directly rather than computing or caching its own derived notion of surfaced-ness, eligibility, or snooze count.

**Key interaction rules:** editing a snoozed project's date into past-eligibility never auto-displaces whatever is currently pinned as surfaced — eligibility can change, the surfaced slot only changes via explicit resolution or deletion-of-surfaced, per the architecture invariant.

**Motion:** 0.35s ease-out for the Home card's state-change transition and glyph state swap; no spring/bounce/overshoot anywhere. Sheets use unmodified system presentation animation. History rows: fully static, no per-row animation, ever.

**Accessibility:** every thread-glyph state has a VoiceOver label generated from the same data driving its visual tier (can't drift out of sync); Act/Snooze/Release have consequence-based labels/hints, never distinguished by color alone; 44pt minimum tap targets verified at largest Dynamic Type; every major state (all Home states, each sheet, corruption recovery) exposes a one-sentence plain-language summary usable both visually and as an accessibility label; surfaced card tested specifically at largest accessibility sizes given no truncation is allowed.

**Preserved external design prompt (optional, non-gating artifact):** Codex's Claude Design prompt from round 1 is retained in the project record for future use (e.g., marketing screenshots or App Store assets), with the explicit caveat attached that any generated visuals must never override the state model, token semantics, or accessibility rules defined above.

## Ios Architecture Review

The fork closed. Claude came around fully on SQLite, and the reasoning behind the concession actually strengthens the whole architecture rather than just settling for Codex's position: the corruption-recovery test is the one thing this phase cares about most, and being able to directly corrupt a raw SQLite file and assert the domain layer repairs it is far cleaner than fighting a `ModelContainer`'s own API into an invalid state. ## Final Output

**SwiftUI architecture:** iOS 17+, one `@MainActor @Observable` app store as the single source of UI-facing truth, injected at the app root. All product logic (dormancy, pacing, tiebreak, cold-start, snooze/act/release transitions, corruption recovery) lives in a pure, dependency-free domain engine that takes "now" as an injected parameter and never calls `Date()` internally — fully unit-testable without touching the UI or the disk. The store mutates only by calling into this engine, then re-fetches and republishes the full snapshot; no live query-observer machinery, no background contexts, no multi-actor merge logic — unjustified complexity at this data scale (dozens of projects, low-thousands of history rows over years).

**Persistence:** Hand-rolled SQLite, not SwiftData, not Core Data, not any third-party wrapper (no GRDB, no SQLite.swift) — a thin (~200-line) first-party wrapper directly over `libsqlite3`. Three tables: projects, decisions (append-only), and a metadata table holding `surfacedProjectID` and `lastResolvedAt` explicitly, so the surfaced slot is persisted state, never a query side effect. Every mutation (Act/Snooze/Release/delete-of-surfaced) is one atomic transaction — clear/set surfaced project, insert decision row, update metadata, all-or-nothing. WAL journal mode with `synchronous = FULL`, specifically to guarantee the last committed transaction survives a force-quit or OS jetsam kill. This was the one real fork in the phase: SwiftData's native `@Observable` integration and CloudKit path was the tempting alternative, but both sides converged on SQLite because this app's entire trust proposition is "never silently show two surfaced projects or lose a decision," and that needs direct transactional control plus a corruption-recovery path that's trivial to test (corrupt the raw file, assert the domain layer repairs to one correct surfaced project or falls back to empty) rather than fighting a higher-level ORM's own API into an invalid state to test the same thing.

**Named, accepted tradeoff:** giving up SwiftData/Core Data's free CloudKit record mirroring and conflict resolution for a future paid sync tier. Judged acceptable because this is single-user, single-writer-at-a-time data (one person's own devices, not multi-party collaboration), so a simple last-write-wins merge by `updatedAt` is sufficient when sync is eventually built as an adapter on top of the existing sync-safe schema (UUID identity, immutable decision IDs, created/updated timestamps) — not a rewrite of the ritual engine.

**Permissions/privacy:** Effectively zero. No location, camera, contacts, notifications, or network access anywhere in the core loop; no analytics or crash-reporting SDKs; no ATT prompt; ideally an `Info.plist` with no usage-description keys at all. No iCloud/CloudKit capability provisioned in this MVP — added only when paid sync actually ships. User-entered content (titles, free text) is never logged; logging uses redacted/static identifiers only.

**StoreKit/subscription architecture:** A sealed `EntitlementProviding` boundary with one concrete `FreeTierEntitlements` implementation today (no StoreKit calls). Free/paid state never forks the persistence schema — only gates capacity and future features. The 7-project soft cap exists in code as an unenforced constant/entitlement boundary only; a regression test must assert creating an 8th, 9th, 10th project still succeeds.

**Dependency policy:** Zero third-party SPM packages. First-party frameworks only (SwiftUI, Observation, Foundation, `libsqlite3` via a small local wrapper).

**Testability:** Unit tests drive the pure domain engine directly (14-day dormancy boundary, 7-day pacing, cold-start exception, tiebreak chain, negative-duration clock anomalies, edit-never-resets-dormancy) and the SQLite repository (transactional round-trips against an in-memory or temp-file database, deliberately-corrupted two-surfaced-flags and zero-surfaced-flags states asserting correct recovery). An automated test asserts the store file's backup-exclusion resource value is false, independent of which storage technology is used. Narrow XCUITest coverage limited to the highest-value flows: full resolve-and-reappear cycle via each of Act/Snooze/Release, delete-the-surfaced-project warning copy, and the home screen's key states.

**ML/LLM/AR:** Not used anywhere — explicitly rejected as weakening the app's legibility, testability, and privacy story.

**Protocol vs static-namespace `RitualEngine`:** Codex switched to the static-namespace shape; Claude held the same position and added the actual reasoning for why protocols are for real seams (repository, clock, entitlements) and not for a rules engine with only one correct implementation. **`HomeState` carrying raw model vs pre-assembled view data:** Claude conceded fully here, adopting Codex's `SurfacedProjectViewData` pattern and pointing out it maps directly onto the five co-equal labeled sections the design phase already locked for the Home card — so a builder renders five `Text` blocks off five named fields, no per-view derivation.

I'm treating this as settled in Claude's favor since it's a strict refinement nobody contested in substance — same for a couple of trivial naming picks (`AppMetadata` over `AppMetadataRecord`, `StoreSnapshot` over `RitualSnapshot`) that Codex had already drifted toward in round 2 anyway.

This is genuinely a buildable, mutually-compatible contract now — every cross-lane type has one shape, one name, and no lane needs to guess which version won.

## Final Output

**Architecture overview:** iOS 17+ SwiftUI app, single target, zero third-party dependencies. One `@MainActor @Observable` `AppStore` at the app root is the sole source of UI-facing truth. All product rules (dormancy, pacing, tiebreak, cold start, resolution transitions, corruption recovery) live in `RitualEngine`, a dependency-free static-function namespace that takes `now: Date` as a parameter everywhere and never calls `Date()` internally. `AppStore` calls `RitualEngine` to compute a `RepositoryMutationPlan`, hands it to a fully **synchronous** `ProjectRepository` (no async/await, no actor, no background context — deliberately, per the architecture phase's "no unjustified concurrency" ruling), then re-fetches and republishes the full `StoreSnapshot`. `ClockProviding` is injected into both `AppStore` and the repository, so nothing past the UI ever calls `Date()` directly and end-to-end time-travel tests are possible, not just engine-level ones.

**File/module layout:** one target, no SPM package boundaries. `Domain/` — `Project.swift`, `DecisionLogEntry.swift`, supporting enums, `RitualEngine.swift`. `Persistence/` — the ~200-line `libsqlite3` wrapper and `SQLiteProjectRepository`. `Store/` — `AppStore`. `Entitlements/` — `EntitlementProviding` + `FreeTierEntitlements`. `DesignSystem/` — tokens and the shared components (`ThreadGlyph`, `DateChip`, `SurfacedCard`, `RitualEmptyState`, etc.). `Features/` — one folder per screen (Onboarding, ProjectRecord, Home, ResolutionSheets, YourProjects, History, Settings), each a plain View reading `AppStore` directly — no per-screen view-model layer. Project Record is presented as a `.sheet` in create mode and a pushed `navigationDestination` in edit mode, so dismiss behavior can't be ambiguous between the two modes. Tests split into `LooseEndsTests` (engine, repository, corruption injection, backup-exclusion, the 10ms transaction-perf assertion) and a narrow `LooseEndsUITests`.

**Data models:** `Project` (raw `UUID` id, `ProjectStatus` with exactly two cases — `active`/`released` — everything else derived at read time from `lastTouchedAt`/`snoozeUntil`/metadata, never stored redundantly). `DecisionLogEntry` (append-only, immutable, denormalized snapshot fields — project title and resulting next-eligible date frozen at write time so later corrections never rewrite history). `AppMetadata` (single row: schema version, `surfacedProjectID`, `lastResolvedAt`, `hasCompletedOnboarding` — all in the same store so a wipe/reset behaves consistently). `StoreSnapshot` bundles all three. `ThreadStatus` carries its exact day count or date as an associated value so visual tier and VoiceOver label can never drift apart.

**Persistence strategy:** unchanged from the architecture phase — hand-rolled SQLite, WAL journal mode, `synchronous = FULL`, three tables (projects, decisions, metadata), every mutation one atomic transaction. New in this phase: the repository call chain is fully synchronous end to end, justified by an empirical backstop rather than assumption — a required test asserts a full mutation transaction against a database pre-loaded with several hundred history rows completes under 10ms; failing that test is what would trigger revisiting this, not aesthetic preference.

**Testing strategy:** unit tests drive `RitualEngine` directly (14-day dormancy boundary, 7-day pacing, cold start, tiebreak chain, negative-duration clock anomalies, edit-never-resets-dormancy) and `SQLiteProjectRepository` (transactional round-trips, deliberately-corrupted two-surfaced and zero-surfaced states asserting `recoverSurfacedState` repairs correctly). Automated tests assert backup-exclusion is always false and the transaction-perf budget holds. `XCUIApplication().performAccessibilityAudit()` runs against the Home surfaced-card state at a large accessibility content size. Narrow XCUITest coverage: full resolve-and-reappear cycle via Act/Snooze/Release, delete-the-surfaced-project warning copy, and the home screen's key states.

## Project Plan

Same story as last round — Codex bailed again rather than make something up, and Gemini's still shut out. Two rounds in, this phase has had exactly one participant. That's worth naming plainly rather than pretending it's a normal multi-party close.

That said, round 2 wasn't just a restatement — Claude actually turned the critical eye on their own round-1 proposal and found a real bug in it, not just polished the prose:

**The "double-tap" framing was wrong, but the underlying worry was right.** A synchronous `AppStore` call from a button closure can't literally race with itself — SwiftUI dispatches gesture callbacks serially, so there's no concurrent "mid-flight" scenario. But that's not the actual risk. The real gap is sequential: nothing in `RitualEngine.act`/`snooze`/`release`'s current contract checks that the `projectID` passed in still matches `metadata.surfacedProjectID` before proceeding. So two taps in quick succession — before a sheet visually dismisses — could legitimately write two decision-log rows for the same resolution. That's a concrete threat to the ledger's integrity, and it's the same category of mistake the group already ruled out elsewhere (the notification-scheduler rejection, killed for being a second independent implementation of a rule that could drift). The fix: push a `projectID == surfacedProjectID` check into the engine itself, throwing a typed error otherwise, rather than leaning on the UI to disable a button fast enough. This becomes its own named acceptance test on the AppStore/RitualEngine milestone, replacing the sloppier "race condition" language from round 1.

**The corruption-framing fix went one step further too.** Not only is "two surfaced flags" impossible against the final single-nullable-column schema (already flagged last round), but "zero-surfaced" was wrongly lumped in as a corruption case too — it's actually the normal, expected state most of the time (cold start, all-released, pacing window, none-eligible-yet). The verification plan should name exactly two real corruption scenarios: a dangling `surfacedProjectID` pointing at a deleted project, and one pointing at a project whose status is `.released`. Nil pointer states belong to ordinary `RitualEngine`/`HomeState` unit tests, nothing about "recovery."

**The coordinator's two open questions got concrete answers instead of hand-waving:** build gating splits into two different things — compile-checks run continuously on every change (cheap CI hygiene), while milestone acceptance checklists (edge cases, corrected corruption scenarios, the resolve-twice guard, the 10ms perf budget) are naturally milestone-grained. And sizing got turned into actual risk/effort tiers per component rather than a vague gesture at "relative sizing" — with one real walk-back: composite view components like `SurfacedCard`/`RitualEmptyState` can be built in parallel against mocks, but shouldn't be marked "done" until integrated against real `AppStore` output, since their whole job is binding correctly to the view-data shapes.

Nothing here is contested, because nobody else has shown up to contest it — but it's also not empty agreement; it's a plan that survived its own author's adversarial pass and came out sharper and more concrete than it went in. Waiting for a third silent round doesn't buy anything more.

## Final Output

**Sequencing:**
1. **Milestone 0 — project skeleton.** Correct target membership, bundle id, empty `Info.plist` (no usage-description keys), folder structure matching the locked `Domain/Persistence/Store/Entitlements/DesignSystem/Features` layout. 2. **Domain layer (`Project`, `DecisionLogEntry`, `AppMetadata`, `RitualEngine`) and the SQLite persistence layer (hand-rolled `libsqlite3` wrapper + `SQLiteProjectRepository`) built in parallel, each independently fully tested before `AppStore` touches either.**
   - `RitualEngine`'s acceptance checklist is pulled explicitly from the earlier phases' enumerated edge cases (14-day dormancy boundary, 7-day pacing, cold start, tiebreak chain, negative-duration clock handling, edit-never-resets-dormancy) — not just "tests exist."
   - The SQLite wrapper is flagged as the single biggest schedule risk in the plan (no Apple framework safety net underneath it) and gets stress-tested early, not discovered during UI polish.
   - Verification plan is corrected to match the actual schema: corruption tests cover exactly two scenarios — a dangling `surfacedProjectID` pointing at a deleted project, and one pointing at a `.released` project. Nil/zero-surfaced states are normal `HomeState` branches, not corruption, and stay in ordinary unit tests.
3. **`AppStore` next**, once engine and repository are each independently green. Its acceptance checklist includes a dedicated test that resolving (act/snooze/release) a `projectID` that doesn't match the current `metadata.surfacedProjectID` throws rather than silently re-resolving — closing the sequential double-tap gap by fixing it in the engine's contract, not the UI. Also covers full bootstrap → mutate → simulated-relaunch round trips.
4. **Design tokens and atomic components** (`ThreadGlyph`, `DateChip`, etc.) run in parallel with the `AppStore` milestone — they only depend on the frozen domain enums. **Composite components** (`SurfacedCard`, `RitualEmptyState`) are built in parallel too but only marked complete after integrating against real `AppStore` output, not eyeballed against mock fixtures.
5. **Screens sequenced by state-machine complexity, not screen-inventory order:** Home (six states, five-section card) → Project Record (nothing else can be exercised without it) → resolution sheets → the lower-complexity static/list screens (Your Projects, History) → Settings last.

**Risk/effort sizing (relative tiers, not invented calendar dates):**
- Domain types + `RitualEngine`: high-risk / medium-effort — small surface, but a subtle bug here is catastrophic to product trust.
- SQLite wrapper: high-risk / high-effort — largest, most novel, zero framework safety net.
- `AppStore` integration: medium-risk / low-effort, now that the resolve-twice guard lives in the engine rather than the store.
- Design tokens + atomic components: low-risk / low-effort, genuinely parallelizable against frozen enums alone.
- Composite view components: low-risk but effort is understated if "done" is claimed before real integration — treat as parallel-build/serial-verify.
- Screens: low-risk / medium-effort each, sequenced (not parallelized) by state complexity.

**Verification as a first-class, per-milestone gate:** continuous compile-checking on every change; each milestone's own named acceptance checklist (edge cases, the two corrected corruption scenarios, the resolve-twice guard, the 10ms transaction-perf budget, accessibility audit) must pass before the next milestone starts — never deferred to an end-of-project polish pass.

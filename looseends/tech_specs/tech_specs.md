The smallest buildable tech spec is a single-process SwiftUI app with four hard boundaries that do not leak into each other: a pure ritual engine, a SQLite-backed repository, one main-actor app store, and thin screen-specific view state. Anything fancier is where this app starts creating correctness risk it did not earn.

I would lock the module/file shape early because this is a portfolio build and parallel lanes will drift fast if type ownership stays fuzzy. I’d split it into `App`, `Domain`, `Persistence`, `Features`, and `Shared`. `Domain` owns product truth only: project status, decision types, eligibility, resurfacing, corruption recovery inputs/outputs. `Persistence` owns SQLite schema, row mapping, transactions, and file-location policy. `Features` owns screens plus tiny screen-local form state. `Shared` owns design tokens, date formatting, thread glyph rendering, and reusable UI pieces. The main store lives at the seam: it calls the repository, passes snapshots through the domain engine, then republishes immutable screen-facing state.

The data model should stay aggressively plain. I would use one `ProjectRecord`, one `DecisionRecord`, one `AppMetadataRecord`, plus a derived `RitualSnapshot` that the UI reads. `ProjectRecord` needs UUID, created/updated timestamps, title, what-it-is, why-it-mattered, why-it-stalled, optional smallest-next-step, user-supplied `lastTouchedAt`, optional `snoozedUntil`, snooze count, optional released timestamp/note, and a status enum that is persisted explicitly enough to recover but not trusted blindly if corruption appears. `DecisionRecord` is append-only and immutable: UUID, projectID, timestamp, decision kind, optional note payload, and enough denormalized context to make history resilient if the project later changes title. The key domain rule is that the engine, not SQL, decides ritual state. The repository returns raw records; the engine computes eligibility, explains blocked states, applies tiebreaking, and validates surfaced-slot integrity. That gives you one place to test all of: 14-day dormancy, 7-day pacing, cold start, snooze suppression, edit-never-touches, delete-of-surfaced behavior, and negative-duration clock anomalies. I would make the engine expose one bootstrap recovery function and three mutation functions families: project mutations, ritual resolutions, and correction actions. Correction actions matter because delete and edit are not ritual decisions and must never accidentally consume the pacing lock.

SQLite schema should stay at three tables plus indices:
- `projects`
- `decisions`
- `app_metadata`

`projects` needs indices on `updated_at`, `last_touched_at`, `snoozed_until`, and maybe `released_at` for cheap snapshot loading. `decisions` needs indices on `occurred_at` and `project_id`. `app_metadata` is one-row. I would not model “surfaced” as a boolean column on projects if we already have `surfacedProjectID` in metadata; that duplicates truth and invites split-brain bugs. If the UI wants quick lookup, it can compare IDs in memory. The corruption path then becomes: if metadata points to a missing or released project, or pacing says no surfaced project should exist, the engine repairs metadata from decision log plus current project pool. One source of truth is cleaner.

Concurrency should be intentionally boring. UI-facing store is `@MainActor @Observable`. Repository API is async, but serial internally through one database queue/actor. No background observers, no detached tasks, no optimistic UI writes. Every user mutation should go through:
1. set screen state to saving,
2. fetch current snapshot,
3. run pure domain transition with injected `now`,
4. commit one SQLite transaction,
5. re-read full snapshot,
6. publish new app state.

**Claude (Frontend Engineer · User Advocate) — Round 1**

Coming at this from the client side, the thing I keep testing every decision against is: what does the person actually building this next type into their editor tomorrow morning, and what does a real user feel when they open the app on day one versus six months later with forty projects and a wall of history. A lot of the prior phases already did the hard product and persistence thinking, so my job here is to turn that into a shape that a builder can code directly against without inventing anything, and to catch the couple of places where "boring and simple" quietly needs one more decision made explicit.

The app is one target, no local SwiftUI packages, no multi-module ceremony — a handful of screens and a small domain don't need Swift package boundaries enforcing lane ownership, that's solving a problem this app doesn't have. Group by responsibility: `Domain/` holds `Project.swift`, `DecisionLogEntry.swift`, the small supporting enums, and `RitualEngine.swift` as a pure, dependency-free namespace of static functions — no `Date()` calls anywhere in this folder, ever, "now" is always a parameter. `Persistence/` holds the hand-rolled `libsqlite3` wrapper and `SQLiteProjectRepository`. `Store/` holds `AppStore`, the one `@MainActor @Observable` object injected at the root. `Entitlements/` holds the sealed `EntitlementProviding` boundary. `DesignSystem/` holds the token set and the nine shared components from the design handoff — `ThreadGlyph`, `DateChip`, `SurfacedCard`, `RitualEmptyState`, etc. `Features/` holds one folder per screen (Onboarding, ProjectRecord, Home, ResolutionSheets, YourProjects, History, Settings), each just a View that reads `AppStore` directly — no per-screen view-model layer duplicating state the store already owns, because we already agreed screens display store state rather than computing their own derived notions of surfaced-ness or eligibility, and adding a view-model per screen would just recreate that risk one layer up. Tests split into `LooseEndsTests` (domain engine, repository, corruption recovery) and a thin `LooseEndsUITests` for the handful of flows that actually matter.

One thing I want to pin down precisely because the prior architecture phase left it just slightly ambiguous: the final locked text says "no live query-observer machinery, no background contexts, no multi-actor merge logic — unjustified complexity at this data scale," which I read as a deliberate rejection of an actor-isolated repository too, not just SwiftData-style background contexts. I'm going to take that literally rather than reflexively wrapping the SQLite repository in its own actor the way you'd default to for "any disk I/O." At a few dozen projects and low-thousands of history rows, a synchronous SQLite write inside a transaction is sub-millisecond — genuinely imperceptible — so `SQLiteProjectRepository` should be a plain, non-actor type called synchronously from the main-actor `AppStore`, with no async/await or `Sendable` boundary between them at all. That's simpler to write, simpler to reason about, and matches what was already agreed instead of quietly reintroducing the complexity the group explicitly ruled out. The one place I'd flag as a real risk to my own recommendation: if a future version ever adds bulk import or a much larger history, synchronous file I/O on the main thread stops being free, and someone will need to revisit this — I'd leave a one-line comment noting the assumption rather than architecting around a scale problem that doesn't exist yet.

Something I don't think has been said out loud yet: if "now" is only injected into the pure `RitualEngine` functions, but `AppStore` or the repository calls `Date()` directly when invoking them, we've quietly broken the testability story one layer up — you can unit-test the engine in isolation, but you can't drive an end-to-end "resolve Act, then fast-forward eight days, then check what Home shows" test through the real store without controlling time everywhere in the chain. So `AppStore` should hold an injected `ClockProviding` (a one-method protocol, `SystemClock` as the real implementation), and every call into the repository threads `clock.now()` through — nobody past the UI layer ever calls `Date()` directly. That's a small addition but it's the difference between "the engine is testable" and "the whole app's time-dependent behavior is testable," and for an app whose entire premise is dates and pacing, I don't want that gap.

For state, one `AppStore` at the root, publishing a `StoreSnapshot` (all projects plus the full decision log plus the persisted surfaced-project id and last-resolved timestamp) and a derived `HomeState` enum computed by a pure `RitualEngine.homeState(snapshot:now:)` function rather than ad hoc `if`-chains scattered across the Home view — that keeps the "which of the four-plus states are we in" logic testable the same way eligibility and tiebreak already are, and guarantees the view can't drift out of sync with the engine's opinion of what's going on. Navigation is a single `NavigationStack` rooted at Home; Project Record in create mode is a `.sheet` (reached from Onboarding's primary button and from Home's empty-state add button), Project Record in edit mode is a pushed `navigationDestination` (reached from a Your Projects row), and the three resolution flows are `.sheet`s over Home exactly as design already specified, with swipe-to-dismiss as a silent no-op. I want to flag this distinction (create = sheet, edit = push) explicitly because the design doc treats Project Record as "one screen, two modes" without saying how each mode is presented, and if that's left implicit, one lane might build both as sheets and another might build both as pushes, and the dismiss behavior differs materially between the two.

On persistence specifics: three tables exactly as locked — `projects`, `decisions` (append-only, never updated or deleted), and a single-row `metadata` table holding `surfaced_project_id` and `last_resolved_at`. One thing worth being explicit about for the builder: `DecisionLogEntry` needs its own denormalized snapshot fields — the project's title at the time of the decision, and for a snooze specifically, the exact next-eligible date that was shown to the user at confirmation time — rather than joining live against the current project row. Reasoning: Project Record allows direct correction of a project's title or its snooze-until date later, and if History re-derived its display by joining against the live project row, a later correction would silently rewrite what already happened, which breaks the "ledger is honest, immutable proof" promise this app is selling. History rows are frozen at the moment they're written; corrections only affect the live project, never the past.

For testing, beyond what's already locked (engine unit tests for the dormancy/pacing/tiebreak/cold-start/negative-duration matrix, repository round-trip and corruption-injection tests, narrow XCUITest coverage of the three resolution cycles and the delete-surfaced warning), I'd add one concrete, automatable accessibility check rather than leaving Dynamic Type correctness to eyeballing: iOS 17's `XCUIApplication().performAccessibilityAudit()` run against the Home surfaced-card state at at least one large accessibility content size, asserting zero audit issues. That turns "must remain readable at largest Dynamic Type" from a design aspiration into something CI actually fails on. I'd also want a synthetic-data perf check on History specifically — insert a few hundred decision rows and assert the list scrolls without the 0.35s entrance transition firing per row, since that transition is supposed to be scoped to the Home card only and a regression here would both look wrong and be slow.

The biggest risk to my own recommendation here is the "no view-model per screen, screens read AppStore directly" choice — it's the right call for an app this size, but it only stays right if `AppStore`'s public surface stays disciplined (returning fully-formed enums like `HomeState` rather than raw arrays the view has to interpret). If a future screen needs view-local transient state (form drafts, in-progress text before save), that's legitimate `@State` in the view itself, not a reason to build a parallel view-model layer — I want that boundary explicit now so nobody "helpfully" adds one per screen later out of habit.

### Round 2


**Codex (QA & Risk · Pragmatist) — Round 2**

Claude's right that the remaining conflicts are not philosophical, they're build-breakers. If parallel lanes start now with two different `ProjectStatus` shapes, two different repository concurrency models, and two different notions of whether Home gets raw domain data or preassembled display state, we'll manufacture integration bugs for no product gain. So I'd collapse this to the smallest contract that preserves the already-locked invariants and is easiest to test when things go wrong.

On `ProjectStatus`, I think Codex's four-case version is the wrong abstraction for this app. `dormant` is not stored state, it's derived elapsed-time state. `snoozed` is also not a separate lifecycle state, it's "still active, but suppressed until `snoozeUntil`." If you make both of those real enum cases, you create two sources of truth: the enum and the dates. That's exactly how you end up with nonsense like `status == .snoozed` and `snoozeUntil == nil`, or `status == .dormant` on a project whose `lastTouchedAt` was edited yesterday. From a QA perspective, that's gratuitous invalid-state surface. I would go even one step further than Claude's proposal, though: `surfaced` should not be in `ProjectStatus` either. We already locked that surfaced-ness is persisted in metadata as a slot, not a query side effect. Duplicating it into each project row buys nothing except drift risk between `metadata.surfacedProjectID` and `project.status == .surfaced`. The smallest honest model is `ProjectStatus { active, released }`, with dormancy derived from `lastTouchedAt`, snooze represented by `snoozeUntil`, and surfaced represented only by metadata. That eliminates an entire class of corruption. If the group feels forced to choose only between the two already-proposed shapes, Claude's 3-case version is still safer than Codex's 4-case one, but I think the correct answer is actually 2 cases.

Same story on raw `UUID` versus wrapped ID structs: use raw `UUID`. This app has one bounded domain, not a distributed backend with half a dozen identifier types crossing service boundaries. Wrapped IDs are nice when they prevent category errors, but here they mostly inflate every signature and every SQL binding path while parallel lanes are already trying to stay aligned. That's complexity tax without meaningful risk reduction. The real safety comes from immutable IDs, append-only decisions, and transactional writes, not from wrapping `UUID` in `ProjectID(rawValue:)`. I'd spend that attention budget on the immutable history snapshot rule Claude called out, because that's the thing that will actually save us from a user-visible lie later.

On repository sync versus async, I agree with Claude and I think Codex should concede it cleanly. The architecture phase already rejected background contexts and multi-actor merge logic because the data size doesn't justify them. Marking the repository `async` without a real concurrency boundary is just ceremony that infects every caller and invites someone to "optimize" with detached work later. A synchronous repository called from a `@MainActor` store is the smallest thing that ships, and with WAL + tiny row counts the writes are genuinely cheap. If that assumption ever stops being true, that's a future redesign prompted by actual scale, not something we need to pre-bake now. For the engine shape, I land with Claude too: static pure functions or a lightweight namespace, not a protocol. Protocolizing a dependency-free deterministic rules engine before we have multiple implementations is classic abstraction cargo cult. Inject the clock, inject the repository, keep the engine as pure functions. Tests call it directly. Done.

The last one matters most to me as a risk person: Home should absolutely not receive a raw `Project` and then derive "days untouched," glyph tier, and "why it surfaced now" in the view. Both of you already said the store is the one place allowed to assemble read models. Follow that through. If the view computes even one piece of the surfaced explanation locally, you've reopened the drift problem under a different name. I want a preassembled `SurfacedProjectViewData` in `HomeState.surfaced(...)`, because that's the only way to make the full ritual explanation testable as one unit: given snapshot X and now Y, Home state is Z, with this exact surfaced copy, this exact day count, and this exact glyph state. That's the contract worth freezing before build starts. The create-as-sheet/edit-as-push distinction Claude added also seems right to me; it's not a product fork, it's a concrete navigation choice that avoids dismiss-path ambiguity and matches the design intent.

**Claude (Frontend Engineer · User Advocate) — Round 2**

Codex, your "duplicate truth" argument against modeling surfaced as a status case is exactly right, and it actually made me realize my own round-1 `Project.status` was inconsistent with the principle I'd just spent a paragraph arguing for. If `surfacedProjectID` lives in metadata as the real pointer, then a `Project.status == .surfaced` case is a second place that same fact can live, and the moment those two disagree — a crash mid-transaction, a bug in a migration, anything — you've got exactly the split-brain the corruption-recovery test exists to catch, except now there are two flags to reconcile instead of one. So I'm dropping `.surfaced` from status entirely. But I want to push further than your four-case version too: `.dormant` and `.snoozed` have the same problem, just quieter. Dormancy isn't an event, it's something a project drifts into passively as days pass with nobody touching anything — there's no mutation that "makes" a project dormant, it just becomes true of the timestamps. If we persist a `.dormant` status, either something has to actively flip it (which means background recomputation, which the architecture phase explicitly banned) or it just goes stale until the next explicit event touches that row, and then any code path that trusts the stored status instead of recomputing from `lastTouchedAt` is reading a lie. Same exact issue with `.snoozed` — `snoozeUntil` already encodes that fact, and when the clock passes that date with nobody opening the app, the status field doesn't know that happened. So my actual proposal, sharper than either of our round-1 versions: `ProjectStatus` has exactly two cases, `active` and `released`. Everything else — surfaced, dormant, eligible, snoozed, how frayed the thread looks — is derived at read time from `lastTouchedAt`, `snoozeUntil`, and the metadata pointer, never stored redundantly. That's not a compromise between our two proposals, it's cutting the thing we were both partially wrong about.

On `HomeScreenState` carrying a raw `Project` versus your `SurfacedProjectViewData`, I have to just concede this one straight up — you're right and I was contradicting my own stated principle. I said "centralize read models so features can't invent conflicting definitions of 'why this surfaced now,'" and then turned around and put a raw `Project` in `HomeState.surfaced`, which means the Home view itself has to compute days-untouched, glyph tier, and the actual "why now" sentence — the exact per-screen derivation I was trying to rule out. Your `SurfacedProjectViewData` is the correct shape, and honestly it maps beautifully onto something the design phase already locked: the five co-equal labeled sections on the card (what it is / why it mattered / why it stalled / smallest next step / why it surfaced now) are just five fields on that struct. A builder implementing the Home screen doesn't derive anything, they render five `Text` blocks straight off five named properties. I'm adopting your view-data pattern wholesale — `SurfacedProjectViewData`, `HistoryEntryViewData`, `ProjectListItemViewData` — with one tweak: rename `summary` to `whatItIs` so it matches the field name on `Project` and the literal language in the design doc, instead of introducing a synonym a builder has to remember maps to the same thing.

I also want to flag something good buried in your metadata struct that I didn't have: `hasCompletedOnboarding` living in `AppMetadata` instead of a separate `UserDefaults` flag. That's not a small nicety, it closes a real two-sources-of-truth gap — if onboarding status lived in `UserDefaults` and everything else lived in SQLite, a user restoring from backup or a QA engineer wiping the database for a corruption test would end up in an inconsistent state where the app "remembers" onboarding happened but has zero projects and no history. Keeping it in the same store means wiping the store correctly resets the whole app to first-run, which is the behavior you'd actually want. Adopting that as-is.

Where I'm not folding: the repository being async. You're framing "async with one serial actor internally" as the boring, safe choice, but I think that's backwards for a client app talking to a file that's a few hundred KB. "Boring" for a server means push I/O behind an actor because you've got concurrent requests; boring for a SwiftUI screen with one user tapping one button at a time means the fewest thread hops a person reading `AppStore.swift` has to track. If `resolveAct` is `async throws`, every call site becomes `Task { await store.resolveAct(...) }` in a button action, which means every button handler is now an unstructured task that could, in principle, race against a second tap before the first one's write lands — exactly the kind of double-decision bug this app cannot afford, given the whole product promise is "we never show you a corrupted or duplicated ritual state." A synchronous call from a button action can't race with itself; there's no interleaving to reason about because nothing yields control until the transaction is done. Rather than resolve this on vibes, let's make it empirical: ship the repository fully synchronous, called directly from the main-actor store, and add one perf test that asserts a full mutation transaction — insert a project, run act/snooze/release, with a few hundred synthetic history rows already present — completes under some real threshold, say 10ms. If a future build of this app or a different device profile ever fails that assertion, that's the actual signal to introduce an actor, not a hypothetical worry we design around today. That converts a philosophical disagreement into a test that either passes or tells us we were wrong.

Similarly, I'm holding firm on `RitualEngine` as a static namespace over your protocol. A protocol buys you swappable implementations — real one in production, fake one in tests. But there's only ever one correct ritual engine; nobody is ever going to inject a "different" set of dormancy/pacing rules at runtime, and the pure functions take every input they need as parameters, so there's nothing to fake or mock in the first place. A test calls `RitualEngine.eligibleProjects(in: contrivedProjects, now: fixedDate)` directly — no conformance, no test double, no DI container. Making it a protocol adds a concrete `DefaultRitualEngine` type someone has to instantiate and thread through init, for zero testability gained. Protocols in this codebase should mark actual seams where behavior really varies — `ProjectRepository` (real SQLite vs. in-memory test double), `ClockProviding` (real vs. frozen clock), `EntitlementProviding` (free vs. paid, someday) — not the one place where variation would be a bug, not a feature.

One more thing I want to fix from my own round 1: my `ThreadStatus` had discrete tiers baked into the enum itself (`frayedLight`/`frayedModerate`/`frayedHeavy`), but the design handoff is explicit that the accessibility label needs the literal day count — "dormant, untouched for 47 days" — and a tier-only enum throws that number away before the label generator ever sees it. Your `frayed(daysUntouched: Int)` shape actually solves this correctly: the exact count rides along in the associated value, the *view* buckets it into one of the three visual tiers for rendering, and the label generator reads the same integer for the literal sentence. Same logic for snooze — `coiled(until: Date)` carries the exact date so the label can say "snoozed until August 12" instead of just "coiled." Adopting your case shape here, keeping my name `ThreadStatus` since it echoes the product's own "thread status language" phrasing from the design phase.

On raw `UUID` versus wrapped `ProjectID`/`DecisionID` structs — I'll keep pushing raw `UUID`, but lightly, because I don't think this is worth blocking on. This app has exactly two ID'd entity types. The realistic bug wrapped IDs prevent — passing a decision's id where a project's id was expected — requires a call site with both a `UUID` and a `UUID` sitting next to each other in the same function signature, and looking at the actual surface area here (`resolveAct(projectID:...)`, `deleteProject(id:...)`), that situation basically never arises. Meanwhile wrapped IDs cost real friction in the exact place I'm most protective of — SwiftUI view code — every `.sheet(item:)`, every `ForEach`, every dictionary keyed by id needs `.rawValue` unwrapping or a bespoke `Identifiable` shim. For two entity types, I don't think the juice is worth the squeeze, so I'm keeping raw `UUID`, but I won't fight hard if the room wants the wrapper — it's a one-time mechanical rename either way and doesn't ripple through the domain logic itself.

Last small thing: I split "the store repaired itself from corrupted state" away from `AppError` entirely rather than adding a `recoveredCorruption` error case. An error case implies something the UI should react to defensively — a banner, a retry button. But successful corruption recovery isn't a user-facing failure, it's the app quietly doing exactly what it promised to do; treating it as an `AppError` risks someone wiring up scary UI for a thing that should be invisible. Instead `AppStore` tracks `didRecoverFromCorruptionOnLastLoad: Bool` as plain state — that's the hook you wanted for QA to assert recovery happened, without conflating "silently self-healed" with "genuinely broken and unrecoverable," which stays a real `AppError.unrecoverableCorruption` case for the actually-can't-continue scenario.

## Final Output

**Architecture overview:** iOS 17+ SwiftUI app, single target, zero third-party dependencies. One `@MainActor @Observable` `AppStore` at the app root is the sole source of UI-facing truth. All product rules (dormancy, pacing, tiebreak, cold start, resolution transitions, corruption recovery) live in `RitualEngine`, a dependency-free static-function namespace that takes `now: Date` as a parameter everywhere and never calls `Date()` internally. `AppStore` calls `RitualEngine` to compute a `RepositoryMutationPlan`, hands it to a fully **synchronous** `ProjectRepository` (no async/await, no actor, no background context — deliberately, per the architecture phase's "no unjustified concurrency" ruling), then re-fetches and republishes the full `StoreSnapshot`. `ClockProviding` is injected into both `AppStore` and the repository, so nothing past the UI ever calls `Date()` directly and end-to-end time-travel tests are possible, not just engine-level ones.

**File/module layout:** one target, no SPM package boundaries. `Domain/` — `Project.swift`, `DecisionLogEntry.swift`, supporting enums, `RitualEngine.swift`. `Persistence/` — the ~200-line `libsqlite3` wrapper and `SQLiteProjectRepository`. `Store/` — `AppStore`. `Entitlements/` — `EntitlementProviding` + `FreeTierEntitlements`. `DesignSystem/` — tokens and the shared components (`ThreadGlyph`, `DateChip`, `SurfacedCard`, `RitualEmptyState`, etc.). `Features/` — one folder per screen (Onboarding, ProjectRecord, Home, ResolutionSheets, YourProjects, History, Settings), each a plain View reading `AppStore` directly — no per-screen view-model layer. Project Record is presented as a `.sheet` in create mode and a pushed `navigationDestination` in edit mode, so dismiss behavior can't be ambiguous between the two modes. Tests split into `LooseEndsTests` (engine, repository, corruption injection, backup-exclusion, the 10ms transaction-perf assertion) and a narrow `LooseEndsUITests`.

**Data models:** `Project` (raw `UUID` id, `ProjectStatus` with exactly two cases — `active`/`released` — everything else derived at read time from `lastTouchedAt`/`snoozeUntil`/metadata, never stored redundantly). `DecisionLogEntry` (append-only, immutable, denormalized snapshot fields — project title and resulting next-eligible date frozen at write time so later corrections never rewrite history). `AppMetadata` (single row: schema version, `surfacedProjectID`, `lastResolvedAt`, `hasCompletedOnboarding` — all in the same store so a wipe/reset behaves consistently). `StoreSnapshot` bundles all three. `ThreadStatus` carries its exact day count or date as an associated value so visual tier and VoiceOver label can never drift apart.

**Persistence strategy:** unchanged from the architecture phase — hand-rolled SQLite, WAL journal mode, `synchronous = FULL`, three tables (projects, decisions, metadata), every mutation one atomic transaction. New in this phase: the repository call chain is fully synchronous end to end, justified by an empirical backstop rather than assumption — a required test asserts a full mutation transaction against a database pre-loaded with several hundred history rows completes under 10ms; failing that test is what would trigger revisiting this, not aesthetic preference.

**Testing strategy:** unit tests drive `RitualEngine` directly (14-day dormancy boundary, 7-day pacing, cold start, tiebreak chain, negative-duration clock anomalies, edit-never-resets-dormancy) and `SQLiteProjectRepository` (transactional round-trips, deliberately-corrupted two-surfaced and zero-surfaced states asserting `recoverSurfacedState` repairs correctly). Automated tests assert backup-exclusion is always false and the transaction-perf budget holds. `XCUIApplication().performAccessibilityAudit()` runs against the Home surfaced-card state at a large accessibility content size. Narrow XCUITest coverage: full resolve-and-reappear cycle via Act/Snooze/Release, delete-the-surfaced-project warning copy, and the home screen's key states.

## Final Output

**Architecture overview:** iOS 17+ SwiftUI app, single target, zero third-party dependencies. One `@MainActor @Observable` `AppStore` at the app root is the sole source of UI-facing truth. All product rules (dormancy, pacing, tiebreak, cold start, resolution transitions, corruption recovery) live in `RitualEngine`, a dependency-free static-function namespace that takes `now: Date` as a parameter everywhere and never calls `Date()` internally. `AppStore` calls `RitualEngine` to compute a `RepositoryMutationPlan`, hands it to a fully **synchronous** `ProjectRepository` (no async/await, no actor, no background context — deliberately, per the architecture phase's "no unjustified concurrency" ruling), then re-fetches and republishes the full `StoreSnapshot`. `ClockProviding` is injected into both `AppStore` and the repository, so nothing past the UI ever calls `Date()` directly and end-to-end time-travel tests are possible, not just engine-level ones.

**File/module layout:** one target, no SPM package boundaries. `Domain/` — `Project.swift`, `DecisionLogEntry.swift`, supporting enums, `RitualEngine.swift`. `Persistence/` — the ~200-line `libsqlite3` wrapper and `SQLiteProjectRepository`. `Store/` — `AppStore`. `Entitlements/` — `EntitlementProviding` + `FreeTierEntitlements`. `DesignSystem/` — tokens and the shared components (`ThreadGlyph`, `DateChip`, `SurfacedCard`, `RitualEmptyState`, etc.). `Features/` — one folder per screen (Onboarding, ProjectRecord, Home, ResolutionSheets, YourProjects, History, Settings), each a plain View reading `AppStore` directly — no per-screen view-model layer. Project Record is presented as a `.sheet` in create mode and a pushed `navigationDestination` in edit mode, so dismiss behavior can't be ambiguous between the two modes. Tests split into `LooseEndsTests` (engine, repository, corruption injection, backup-exclusion, the 10ms transaction-perf assertion) and a narrow `LooseEndsUITests`.

**Data models:** `Project` (raw `UUID` id, `ProjectStatus` with exactly two cases — `active`/`released` — everything else derived at read time from `lastTouchedAt`/`snoozeUntil`/metadata, never stored redundantly). `DecisionLogEntry` (append-only, immutable, denormalized snapshot fields — project title and resulting next-eligible date frozen at write time so later corrections never rewrite history). `AppMetadata` (single row: schema version, `surfacedProjectID`, `lastResolvedAt`, `hasCompletedOnboarding` — all in the same store so a wipe/reset behaves consistently). `StoreSnapshot` bundles all three. `ThreadStatus` carries its exact day count or date as an associated value so visual tier and VoiceOver label can never drift apart.

**Persistence strategy:** unchanged from the architecture phase — hand-rolled SQLite, WAL journal mode, `synchronous = FULL`, three tables (projects, decisions, metadata), every mutation one atomic transaction. New in this phase: the repository call chain is fully synchronous end to end, justified by an empirical backstop rather than assumption — a required test asserts a full mutation transaction against a database pre-loaded with several hundred history rows completes under 10ms; failing that test is what would trigger revisiting this, not aesthetic preference.

**Testing strategy:** unit tests drive `RitualEngine` directly (14-day dormancy boundary, 7-day pacing, cold start, tiebreak chain, negative-duration clock anomalies, edit-never-resets-dormancy) and `SQLiteProjectRepository` (transactional round-trips, deliberately-corrupted two-surfaced and zero-surfaced states asserting `recoverSurfacedState` repairs correctly). Automated tests assert backup-exclusion is always false and the transaction-perf budget holds. `XCUIApplication().performAccessibilityAudit()` runs against the Home surfaced-card state at a large accessibility content size. Narrow XCUITest coverage: full resolve-and-reappear cycle via Act/Snooze/Release, delete-the-surfaced-project warning copy, and the home screen's key states.

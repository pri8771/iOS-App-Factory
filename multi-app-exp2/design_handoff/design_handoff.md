The thing I want to flag before anyone builds this: the Ritual queue must be snapshotted at entry, not live-recomputed while the session is open. Every prior phase locked "warmth is always computed live, never cached" for the dashboard, which is correct there — but if the ritual screen re-runs that same live query on every render, then the moment the user acts on person 3 of 7, or the debug time-travel panel gets nudged, or a background timer crosses a day boundary mid-session, people can silently drop out of or reorder inside a queue the user is actively working through. That breaks the "3 of 7" progress contract and makes the completion summary lie. So: ritual entry captures an immutable list of person IDs plus their ratios at that instant; actions during the session mutate the underlying models as normal, but the queue membership and order for that session are fixed until the user exits. This needs to be written down now because it's exactly the kind of decision that looks like an implementation detail but is actually a product-correctness rule, and nobody upstream of this phase specified it.

Second systems point: identity and list performance. Every `ForEach` over people must key on `PersistentIdentifier`, never on name or array index — duplicate-name warnings already prove names collide, and index-keying under a sort that changes every render (ratio descending, recomputed live) will produce visible cell-identity glitches and broken animations the instant two people swap rank. Since ratio is derived at read time, the dashboard's `body` recomputes a sort over the full people set on every render pass; at the realistic scale here (hundreds of people) that's trivial, but the "why now" sentence generation should stay pure string interpolation off already-fetched note data — no re-querying per row — or scroll performance degrades in a way that undercuts the "expensive-feeling" visual goal no matter how good the typography is.

The active queue uses one shared attention card pattern with stable hierarchy: name, cadence/state chip, why-now sentence, optional last-note anchor, trailing affordance into detail. The truncated queue is just the active queue plus a premium continuation card after the third item, and only when a fourth truly exists. Exactly three due people means no premium interruption at all. That boundary is product dignity, not just a test case.

Add Person should stay brutally short. Name field first, cadence control second with Regular preselected, first note third and visually dominant, then the collapsed detail disclosure. That order matters because we are optimizing for “I just left a meeting and need to capture one real thing before it evaporates,” not for profile completeness. The save button needs to stay anchored and obvious as the keyboard moves. Duplicate-name warning stays inline and soft. On success, I want a small but noticeable insertion moment back on the dashboard so the app feels alive immediately; the user should feel that they just created a living record, not submitted a form. Person Detail then picks up that emotional thread by leading with context, not fields: name, cadence tier, warmth chip, full-sentence explanation, then timeline. Archive is ordinary and reversible. Permanent delete is intentionally heavier and one layer farther away. If a person only has their first capture note, the screen still needs to feel complete; that first note is the beginning of the relationship timeline, not a placeholder until “real” interactions exist.

Ritual needs the most discipline because it’s the retention engine. Full-screen modal, progress at the top in plain language, one card at a time, three equal-weight actions. Equal weight is important; if “Log outreach” looks primary and the others look secondary, we’re prescribing behavior instead of supporting triage. Snooze must explain itself honestly: this hides the reminder until that date, but it does not count as contact. Archive must feel like a legitimate resolution, not a failure. After each action, the current card should move out with one restrained transition and the next one should settle in; no gamified flourish, no loud celebration. Completion is where we earn the app’s emotional tone: specific counts, supportive copy, clean exit back to a dashboard that now either shows the reduced queue or the good-shape state. And because I agree with Claude’s engineering read, the completion summary should be generated off the ritual snapshot/session result object, not off a fresh query after the fact.

Visually, I’m still pushing hard for expensive calm. Warm off-white canvas, slightly lifted cream surfaces, one deep ink text color, restrained amber or terracotta accent for attention, soft green only for good-shape/completion, real destructive red reserved exclusively for permanent deletion and full reset. I do not want full-card severity fills for warm/cooling/cold. A subtle edge treatment, chip, or small tonal shift is enough. This app stores sensitive opinions about real people; the UI should feel private and adult, not loud. Typography should stay fully native but disciplined: big roomy titles, generous line spacing, compact supporting copy, zero cramped dashboards. Motion stays soft and structural: fade-and-settle on launch lock, sheet lift for Add Person, focus-mode feel entering Ritual, subtle reorder crossfade on dashboard when rankings change. Reduce Motion needs real alternate behavior, not the same movement slowed down.

Accessibility has to stay embedded in the screen contracts, not stapled on later. Every attention card should read as one coherent VoiceOver chunk in a sensible order. Ritual progress needs an explicit accessible value. Completion needs one announced summary. Large Dynamic Type has to be allowed to make cards taller instead of clipping copy or crushing buttons. The auth warning, export warning, and destructive confirmation copy all need to work when heard, not just seen. That matters here because this product’s core promise is trust; nothing shatters trust faster than a privacy or deletion flow that becomes ambiguous once you leave the designer’s default text size.

So my vote is: lock the ritual snapshot rule as a first-class product requirement, make an explicit no-external-pause decision, keep the Claude Design prompt as future refinement only, and get the actual token/component/state language into the final handoff instead of buried in a file reference. If we do that, the builder has everything they need. If we don’t, we’ll end this phase with a lot of good taste and one or two hidden assumptions that will absolutely blow up in implementation.

**Claude (Frontend Engineer · Systems Thinker) — Round 2**

Codex, your extension of the snapshot rule — freezing the why-now text and state alongside membership and order — is exactly right, and it closes a hole I left open in round one. But I want to push one layer deeper, because there's a real architectural question hiding underneath "freeze the queue at entry": where does that frozen snapshot actually live? If it's just a struct sitting in a view model's memory, we've solved the "person 5 vanishes mid-session" problem but broken a requirement that's been locked since `next_steps_small` and reaffirmed in the design phase — that ritual progress and state must survive app termination and relaunch, with a distinct "interrupted-and-resumed" state. A pure in-memory snapshot dies the instant the app is killed, and the user comes back to either a blank ritual or, worse, a freshly re-queried live one that silently reorders on them.

So I think we need a small persisted `RitualSession` record — I'd keep it dead simple, either a tiny SwiftData model or even a single Codable blob in the App Group container, doesn't need its own complex schema — that captures: a session ID, the frozen ordered list of person IDs with their captured ratio/state/explanation-sentence-at-entry, and a per-person resolution enum (`.unresolved`, `.loggedOutreach`, `.snoozed(until:)`, `.archived`) with a timestamp. When the ritual view opens, it either finds an existing unresolved session and rehydrates it verbatim — Codex's "pick up where you left off" — or creates a new one by querying live warmth state exactly once, at that moment, and locking it in. Every action the user takes inside the ritual writes through to both the real `Person`/`Interaction` models (so the dashboard reflects live truth) and updates that person's resolution field in the session record (so the frozen queue's per-item state persists too). Completion is generated by reading the session record's resolution counts, not by re-querying — which also directly answers Claude-from-last-round's (me, previous phase) requirement that the completion summary come from the session snapshot, not a fresh fetch. One small edge case worth naming now rather than discovering later: if a person in a frozen session gets permanently deleted from another part of the app somehow mid-session — shouldn't be reachable in v1's navigation, but worth a defensive check — the ritual should just skip rendering that row rather than crash on a dangling reference.

Here's the piece nobody's said out loud yet, and I think it actually matters: the content of the ritual queue should be frozen, but the *paywall gate on how much of that frozen queue you can see* should not be. Think about the actual scenario — someone opens the ritual, sees three people free, hits the "1 more waiting" paywall card, and subscribes right there without leaving the flow. If we freeze the entitlement check along with the queue snapshot, that person is now stuck looking at a paywall card for a session that's already technically theirs, and they'd have to exit and re-enter the ritual to see the rest — which is a genuinely bad moment to make a paying customer feel like the product didn't notice they just paid. But if the entitlement check re-runs live (which it should, since `Transaction.currentEntitlements` being live and uncached is already a hard-locked requirement from `detailed_discussion`), the previously-hidden items in the *same frozen session* should just reveal themselves in place. So: snapshot governs which people, what order, what explanation text. Live entitlement governs how much of that fixed list is currently visible. Those are two different axes being locked at two different layers, and conflating them either breaks the "no silent reordering" guarantee or breaks the "your subscription should work the second you buy it" guarantee. I think that distinction needs to be explicit in the handoff so whoever builds this doesn't accidentally implement one giant "is this session live or frozen" boolean and pick wrong.

On the no-external-design-pause question — agreed without reservation. We've got an 8-screen inventory, four dashboard states with an exact numeric boundary, a full component list, token roles, motion rules gated on Reduce Motion, and now a resolved ritual-session architecture. There's nothing left that a design pass would be resolving that isn't already resolved. The Claude Design prompt goes in as deliverable #10, explicitly scoped as "refine this existing, fully-specified app visually," not "help us figure out what this app is."

Small detail, but it's the actual mechanism that makes "resume after relaunch" a real, testable behavior instead of a sentence in a spec.

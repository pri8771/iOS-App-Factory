**Claude (Frontend Engineer · Visionary) — Round 1**

Coming at this as the one who has to actually make this app feel alive in code, not just correct on paper — three things I want to push hard on before we lock the contract, plus the full spec.

The single most valuable architectural move available to us right now, and nobody's proposed it yet, is splitting the domain framework into two Swift Package targets instead of one. Everyone across the last two phases agreed "if a file containing scoring math imports SwiftData, that's a defect" — but a review rule is something a human has to remember to check. If I split `NurtureKit` into `NurtureDomain` (Foundation only, zero framework dependencies — `CadenceTier`, `WarmthState`, `PersonSnapshot`, `WarmthCalculator`, `AttentionQueueBuilder`) and `NurtureData` (depends on `NurtureDomain` plus SwiftData, holds the `@Model` records and the concrete repository), then it becomes structurally impossible to import SwiftData into the scoring code — the package manifest doesn't link it, so it won't compile. That turns a code-review discipline into a build-system guarantee, which is exactly the kind of thing that survives contact with time pressure and a tired engineer at 11pm. This is a local Swift package added to the Xcode project (not a remote dependency, so it doesn't violate the zero-third-party-package rule), living at the repo root as `Packages/NurtureDomain` and `Packages/NurtureData`, or as in-project package products if we want a single package manifest with two targets — either works, I'd default to one package with two targets for less Xcode-project ceremony.

Second thing worth catching now rather than during build: SwiftData to-many relationships do not guarantee order. The whole `RitualSession` design hinges on a frozen, ordered list of people — if `RitualSessionItemRecord` is just a `@Relationship` array off `RitualSessionRecord` with no explicit ordering field, SwiftData is free to return it in whatever order it wants on rehydration, which would silently break "3 of 7 in this exact order" the moment the app relaunches mid-session. So `RitualSessionItemRecord` needs an explicit stored `orderIndex: Int`, and the repository's rehydration path must sort by it, not trust relationship array order. Small thing, but it's exactly the kind of bug that only shows up after a real app-kill mid-ritual, which is precisely the scenario this whole record type exists to protect.

Third correction: the design_handoff phase said `Route` should carry a SwiftData `PersistentIdentifier`, never a `Person` reference — right instinct, wrong currency, because we've since locked that DTOs/snapshots are the only things allowed to cross into the UI layer, and they're keyed by plain `UUID`, not `PersistentIdentifier`. If `Route.personDetail` carries a `PersistentIdentifier`, the UI layer has to know about SwiftData's identity type, which quietly breaks the "no SwiftData past the repository boundary" rule we just went to the trouble of enforcing at the package-target level. `Route` should carry the domain `PersonID` (a `UUID` newtype), and the repository resolves it back to a `PersistentIdentifier` internally. I'm fixing this now since it's cheap to fix in a spec and expensive to fix after three screens are wired up.

On architecture shape: `AppRouter` is `@MainActor @Observable`, owns `path: [Route]` plus three independent presentation booleans/enums (`addPersonPresented: Bool`, `ritualPresented: Bool`, `paywallSource: PaywallSource?` — presence of a non-nil source both drives the sheet and tells the paywall which entry point to phrase its copy around, dashboard-teaser vs. ritual-overflow). Nothing else in the app is allowed to hold navigation state; view models call into the router, they never construct their own `NavigationLink` or present sheets independently. `AppDependencies` is a plain struct (`dateProvider`, `repository`, `authService`, `entitlementService`, `notificationScheduler`, `exportService`) injected once at the app root via a custom `EnvironmentKey` (`\.appDependencies`), not threaded through fifteen initializers — every view model reads what it needs off that one environment value at construction time. This keeps preview and test setup trivial: a test builds an `AppDependencies` with fakes and hands it to whatever it's testing, full stop.

For state management specifics: the Dashboard view model holds no cached sorted array — it recomputes `AttentionQueueBuilder.buildQueue(...)` on every access from freshly-fetched snapshots, because at this app's realistic person count (dozens to low hundreds) that's computationally free, and caching it would just reintroduce the exact "derived state goes stale" bug class the group spent two phases stamping out. `ForEach` over the queue keys on `PersonID`, never array index, so re-sorts animate as reorders instead of full-content swaps when someone crosses a warmth boundary. The Ritual view model is different in kind, not degree: it holds a `RitualSessionSnapshot` value (built once at entry, or rehydrated from a persisted unresolved session), and every render re-derives *visibility* by calling `AttentionQueueBuilder.applyEntitlement(to:isSubscribed:)` against that frozen list — this is the one function that has to exist to make "buy mid-ritual, see people reveal in place" a real, testable behavior rather than a sentence in a doc, and it's the same function the Dashboard uses for its 3-vs-4 teaser boundary, so both screens' paywall-truncation logic is provably identical because it's literally the same code path.

On persistence: `SchemaV1: VersionedSchema` from the first commit, holding `PersonRecord`, `NoteRecord`, `RitualSessionRecord`, `RitualSessionItemRecord`. `PersonRecord` stores `id`, `name`, `createdAt`, `lastInteractionAt`, `cadenceRaw`, `isArchived`, `snoozedUntil`, with a cascade-delete relationship to `NoteRecord`. `lastInteractionAt` is a plain stored field, bumped whenever a note is logged (never recomputed from note history at read time — that's a stored fact about what happened, not a derived warmth number, so it's fine to store). Warmth ratio/state are never stored anywhere, full stop. `ContainerLocating` is a small protocol wrapping `resolveDatabaseURL() -> URL` so the App-Group-vs-Application-Support fallback path is unit-testable by injecting a fake resolver rather than only being provable by reading the source and hoping.

Testing strategy, concretely: `NurtureDomainTests` runs with zero SwiftData/main-actor involvement and needs to name, not just imply, its cases — ratio exactly 1.0 and exactly 1.5 as the two cutover boundaries, day-zero-at-creation (a person created "now" never shows cooling), all three cadence tiers independently, a DST-crossing date pair proven via `Calendar` day counting, snooze-hides-but-doesn't-reset-the-underlying-ratio, queue cap at exactly 10 and 11, and `applyEntitlement` at exactly 3 (no teaser) and exactly 4 (one hidden) qualifying entries. `NurtureDataTests` runs against an in-memory `ModelContainer` and covers round-trip create/fetch/archive/delete, `RitualSessionItemRecord` order surviving a simulated relaunch (fetch, detach, refetch, assert order via `orderIndex` not insertion order), and the container-fallback path via the injected `ContainerLocating` fake. UI-level tests cover the full journeys already named in prior phases (onboarding, add-person-under-a-minute, ritual interrupt/resume, StoreKit purchase/expire/restore via `StoreKitTest`) plus one I want added explicitly here since it's the trickiest state to get right: purchase completing while a ritual with a truncated preview is on-screen, asserting the hidden entries render in place with unchanged identity and order for the entries that were already visible.

The detail I most want preserved out of this phase: the two-target domain/data package split isn't a nice-to-have organizational preference, it's the mechanism that makes "scoring math never touches SwiftData" true by construction instead of true by discipline — and the `orderIndex` field on ritual session items is a one-line schema addition that prevents a silent, hard-to-reproduce ordering bug that would otherwise only surface after a real app kill mid-review.

**Gemini (Backend / Systems Engineer · Skeptic) — Round 1**

**Architecture Overview**

The application is structured around a strict native iOS architecture built on iOS 17+, Swift 5.9+, and SwiftUI, with zero third-party dependencies. To ensure maximum reliability and prevent multi-threading concurrency issues, the codebase is cleanly divided into a pure, framework-independent data-domain layer and a SwiftUI-oriented UI presentation layer. This boundary is enforced by a value-type data transfer object (DTO) seam, where the persistence repository returns and accepts only thread-safe, immutable Sendable structs rather than live SwiftData reference models. Navigation is managed centrally on the main actor via a reactive AppRouter state container driving a root NavigationStack, with modal interactions presented strictly via sheet or full-screen cover properties to prevent nested navigation state bugs.

**File and Module Layout**

The Xcode project organizes files into distinct functional directories to maintain a strict separation of concerns. Inside the source folder, the Domain directory contains pure DTO structs, core models, and the math utilities for warmth calculation. The Storage directory contains the database schema definitions and the concrete implementation of the repository protocol using SwiftData. The Services directory houses protocol abstractions and mockable implementations for time resolution, local biometric authentication, notifications, and StoreKit 2 transactions. The Views directory groups screen-specific SwiftUI files, reusable custom components, and view-local layout code. Finally, the Tests directory is split into DomainTests for isolated engine checks, IntegrationTests for database and file systems, and UITests for user flows and paywall gates.

**Data Models**

The local database uses a versioned schema approach starting with SchemaV1 to handle migrations reliably from day one. It defines four primary SwiftData models: Person, Note, RitualSession, and RitualSessionItem. A Person stores a unique identifier, name, creation date, last interaction date, raw cadence tier string, archiving flag, and an optional snooze date. A Note contains the text content, creation date, and optional fields for key facts and open loops. To satisfy the requirement that a weekly review session survives app termination, the RitualSession model acts as a persisted transaction log. It records a creation timestamp, completion timestamp, and an ordered collection of RitualSessionItem records. Each item freezes the contact ID, computed warmth ratio, warmth state, and localized explanation text captured at session entry, alongside a resolution status indicating whether the contact was contacted, snoozed, or archived.

**Persistence Strategy**

The database container resolves its file path dynamically, looking first for a shared App Group URL to facilitate future widgets and extensions, and falling back gracefully to the sandboxed Application Support directory so that the app compiles and runs on developer accounts lacking paid team provisioning. Data-at-rest security is enforced by applying complete file protection to the database container folder, ensuring encryption triggers automatically whenever the device locks. CloudKit synchronization is explicitly excluded from the model configuration, ensuring that schema design remains unconstrained by CloudKit constraints like mandatory optional properties or missing unique keys. All updates to contacts and notes write through the repository on the main actor, avoiding background thread concurrency conflicts.

**Scoring and Concurrency Boundaries**

Relationship warmth is calculated live on read by the pure, non-actor-isolated WarmthCalculator, ensuring that decay values are never cached in the database. The calculation measures elapsed time from the maximum of the contact's creation or last interaction date, divided by their assigned cadence tier. The evaluation boundaries are inclusive, classifying relationships as warm below a ratio of 1.0, cooling from 1.0 up to 1.5, and cold at or above 1.5. To ensure that date arithmetic is testable and immune to timezone or daylight saving shifts, calculations run through an injectable DateProvider protocol that uses calendar-based day counting. The weekly ritual session freezes its content queue at entry to prevent items from shifting or disappearing mid-session, while leaving subscription entitlement checks live so that a mid-session upgrade reveals truncated contacts in place.

**Authentication and Security**

The Face ID and device passcode gate uses deviceOwnerAuthentication to ensure a secure, biometric-first launch experience with automatic passcode fallback. The lock triggers immediately on cold launch and when the app remains in the background past a two-minute grace period. To maintain complete security, fresh biometric authentication challenges are triggered at the literal moment of executing destructive actions, including permanent contact deletion and plaintext JSON database export. A simple, testable root-level view state drives the app-switcher privacy shield, masking sensitive notes immediately when the scene phase changes to inactive or background. In debug builds, a compiler-gated biometric bypass switch is compiled in via view-level directives to prevent lockouts during developer simulations, but this code is stripped entirely in production.

**Monetization and Entitlements**

The StoreKit 2 integration verifies entitlements live against Transaction.currentEntitlements on every gated check, avoiding local cached state variables that can get out of sync. A transaction listener task starts at application launch to monitor out-of-app purchases, configured as an in-process operation without background capabilities to prevent unnecessary API exposure. The paywall gates the weekly review queue beyond the top three preview contacts, displaying a clear count of hidden due contacts. To handle offline or billing errors gracefully, the paywall drops back to static fallback localized pricing labels, leaving the rest of the application completely usable. All StoreKit flows are tested headlessly in simulator runs via an automated test session pointed at a local StoreKit configuration file.

**Testing Strategy**

The testing matrix is divided into three levels. First, pure unit tests exercise the warmth calculator using a mock clock, asserting correct state transitions at exactly 1.0 and 1.5 boundaries, day-zero anchoring, and correct paywall teaser thresholds. Second, integration tests verify database operations against an in-memory SwiftData container, checking data persistence, duplicate name prevention warnings, and plain-text JSON export validation. Third, UI tests simulate user progression through onboarding, contact creation, and ritual triage, utilizing StoreKitTest to mock subscription purchases, expirations, and restorations, and confirming that the time-travel panel correctly drives interface refreshes.

Here's where round one actually landed: two of the three voices spoke (Claude and Gemini — no Codex yet this round), and while they agree on almost everything locked in prior phases, there's a real, load-bearing disagreement buried in the actual code contracts that the prose doesn't surface.

**Where they actually diverge, and it's not stylistic:**

1. **The repository protocol itself is two incompatible shapes.** Claude's `RelationshipRepository` is operation-specific — `addPerson`, `logOutreach`, `snoozePerson`, `archivePerson`, `deletePersonPermanently`, explicit ritual-session lifecycle methods, `nameCollisionExists`. Gemini's is generic CRUD — `fetchPeople`, `savePerson`, `saveNote`, `deletePerson`, `fetchActiveRitualSession`, `saveRitualSession`. A UI or view-model built against one will not compile against the other. This is exactly the kind of thing the phase's own acceptance check ("parallel build lanes can work without inventing incompatible types") exists to catch, and it's unresolved.
2. **The value-type currency differs.** Claude uses a `PersonID` newtype wrapping `UUID` and calls the DTOs `PersonSnapshot`/`NoteSnapshot`; Gemini uses raw `UUID` and calls them `PersonDTO`/`NoteDTO`. Everything downstream (Route, repository signatures, queue types) inherits whichever choice wins.
3. **Claude introduced two real bug fixes Gemini's version doesn't address at all:** an explicit `orderIndex` field on `RitualSessionItemRecord` (SwiftData to-many relationships don't guarantee order, so without this the "3 of 7 in this exact order" contract silently breaks after an app-kill mid-ritual), and a shared `AttentionQueueBuilder.applyEntitlement(...)` function so the Dashboard's 3-vs-4 teaser boundary and the Ritual's reveal-in-place behavior are provably the same code path rather than two hand-written implementations that can drift. Gemini's contract has no equivalent to either.
4. **Structural enforcement differs.** Claude proposes splitting the domain framework into two Swift package targets (`NurtureDomain`, Foundation-only; `NurtureData`, SwiftData-dependent) so "scoring math never imports SwiftData" is a compiler guarantee, not a review checklist item. Gemini proposes the same separation only as folder layout inside one module — weaker, and not yet confirmed or rejected.
5. **Gemini's contract is missing several pieces** Claude specified: `ExportService`/`ExportDocumentV1`, `NotificationScheduler`, `PrivacyShieldState`, `DebugClock`, `NurtureError`, `Route`/`PaywallSource`, `AppDependencies`. Gemini's prose gestures at some of these concepts but never formalizes them as types.

None of this is a values disagreement — it reads like two people who haven't yet reconciled their drafts against each other. But it's real enough that calling it consensus now would hand parallel build lanes two different repository protocols.

### Round 2


**Codex (QA & Risk · Closer) — Round 2**

Claude is right about the thing that actually matters here: if we leave this phase with two repository shapes, we did not produce tech specs, we produced a future merge conflict. Gemini, your schema records, App Group fallback, and StoreKit posture are basically fine, but `fetchPeople/savePerson/saveNote/deletePerson` is the wrong seam for this app. It pushes invariants out into view models right where they’ll fork: who updates `lastInteractionAt`, who enforces “first note required,” who validates snooze dates, who keeps archive/delete semantics consistent with ritual state, who does duplicate-name checks, and who decides whether paywall truncation logic lives in UI or domain. That is exactly how three build lanes create three subtly different products. I want Claude’s operation-specific repository contract, plus one missing method for Settings reset: `deleteAllData()`. Same on identity: I do not care sentimentally about `DTO` versus `Snapshot`, but I do care that we pick one and stop. I’d lock `PersonID` as a newtype and use `Snapshot` names, because raw `UUID` leaks everywhere and “DTO” implies a transport layer we do not have.

Two other things are bug fixes, not style choices. First, `orderIndex` on `RitualSessionItemRecord` is mandatory. Without it, the “3 of 7” frozen review promise survives only until the first app kill, because SwiftData relationship ordering is not a persistence guarantee. Second, dashboard teaser logic and ritual reveal-in-place must share one tested entitlement path. If `3 shows no teaser / 4 shows 1 more waiting` is implemented once in dashboard code and once in ritual code, it will drift. That means a real `AttentionQueueBuilder.applyEntitlement(...)` function in the pure domain layer, not two hand-written copies. Gemini, this is the part I want you to concede cleanly: the generic CRUD surface and the extra `presentPaywall` boolean should go. `paywallSource: PaywallSource?` is enough to drive both presentation and copy, and an operation-specific repository is how we keep behavioral rules in one place instead of splintering them across screens.

I also do not want “folder separation” sold as architecture. If we actually believe “if the scoring file imports SwiftData, that’s a defect,” then make it impossible: two internal targets or packages, `NurtureDomain` with Foundation only, and `NurtureData` with SwiftData plus concrete service adapters. Otherwise someone will eventually import SwiftData into the warmth or queue code because it’s convenient, and then all the testability claims become social policy instead of compiler-enforced structure. The verification bar needs to be explicit now too, or this phase is ducking its own acceptance check: pure domain tests for ratio boundaries, DST, day-zero anchoring, snooze-not-resetting, queue cap, and the exact 3-vs-4 threshold; integration tests for App Group fallback and ritual-session rehydration preserving `orderIndex`; StoreKitTest coverage for purchase, expire, restore, and mid-ritual reveal-in-place without queue reordering; and a Release archive check proving there is no `DebugClock`, no auth bypass UI, and no iCloud capability. If we lock that contract, parallel lanes can build without inventing their own types or semantics.

### Summary of Governance Plan

1.  **Concrete Workstreams**: Separated development into four distinct, logical phases:
    *   **WS A: Core Engine & Data Spine** (Scaffolding, Schema persistence, Domain algorithms)
    *   **WS B: User Interfaces & Navigation** (Navigation routing, dashboard states, triage sheets)
    *   **WS C: Security & Monetization** (Biometric protection, active StoreKit 2 entitlement gating)
    *   **WS D: Polish, Hardening & Release Prep** (Sunday local notifications, accessibility checks, binary audits)
2.  **File Ownership**: Configured direct file ownership lines mapping packages (`NurtureDomain`, `NurtureData`) and view directories (`Views/`, `Routing/`, `Resilience/`) to specific developer lanes (`data_domain`, `services_utilities`, `primary_ui`, `polish_resilience`) to minimize intersection overlap.
3.  **Forbidden Edit Zones**: Gated critical regression risks, specifying that:
    *   `NurtureDomain` targets must never import `SwiftData`, `SwiftUI`, or `UIKit`.
    *   Interface models, protocols, and DTO contracts are locked.
    *   Time-sensitive functions must not invoke direct `Date()` constructors.
    *   No CloudKit or iCloud capability configurations are allowed.
4.  **Branching & Merging**: Mandated trunk-based feature branches (e.g., `feature/ws-a-domain-persistence`) with strict quality gates (zero compile warnings, 100% test success, no placeholder `TODO` stubs).
5.  **Testing Responsibilities**: Assigned ownership for each test tier (domain math in unit tests, SwiftData in integration tests, StoreKit in automated journeys) and provided terminal verification commands (`swift test` and `xcodebuild`).
6.  **Conflict Prevention**: Isolated views from direct SwiftData manipulation via protocol interfaces, centralized transitions under a `@MainActor` `AppRouter`, and mapped active models to thread-safe DTO snapshots at the repository boundaries.

### Round 2


**Codex (Frontend Engineer · Skeptic) — Round 2**

Claude is right that the finer-grained backlog is the one that actually ships. My earlier bundled version, and Gemini’s WS A/B/C/D framing, both hide the two places iOS work really blows up: fake dependencies and shared-file collisions. If UI has to wait for “all services” before it can even start `LockView` or `AddPersonView`, we’ve serialized the app for no product reason. And if nobody explicitly owns `NurtureApp.swift`, `project.pbxproj`, the shared schemes, and the test plan, we’ll get the classic Xcode mess where three people “only made a tiny change” and the build stops being reproducible. So I’m explicitly backing Claude’s 23-task shape, with two fixes I think we need to lock now: `Route.archivedPeople` cannot stay ownerless, so Settings has to absorb an `ArchivedPeopleView`; and `NurtureApp.swift` needs its own primary_ui-owned integration task so app-root wiring does not become a stealth shared edit file. The repo lives at `/Users/pchordia/Documents/GitHub/nurture-ios`, `main` stays the trunk, and we branch per task, not per lane: `feature/t001-scaffold`, `feature/t007-repository`, `feature/t014-dashboard`. Lane branches are a merge-conflict machine and they hide integration failures until it’s expensive.

The division of labor should be blunt. `data_domain` owns every public contract in `NurtureDomain` and the pure scoring/session logic; that lane is not allowed to import `SwiftUI`, `SwiftData`, or `UIKit`, and nobody else is allowed to casually add fields or enum cases to those types once T-003 is green. `services_utilities` owns every platform binding and every integrator file: `project.pbxproj`, both `Package.swift` files, schemes, test plans, entitlements, `Info.plist`, `.storekit`, concrete repository/services, and the App Group decision. Those shared files are append-only and serialized through that lane; nobody “cleans them up” in passing. `primary_ui` owns `NurtureApp.swift`, the router, all views, and all reusable UI components; that lane never touches `ModelContext`, `Transaction.currentEntitlements`, `LAContext`, or direct `Date()` calls for business state. `polish_resilience` owns `PrivacyShieldState`, `DebugClock`, the release audit script, QA/docs, and the unified error surface, but it does not get to redefine navigation, repository contracts, or screen flows while “hardening.” Handoffs are hard gates, not vibes: T-003 unlocks the repo and router work, T-008 must be green before T-016 starts, T-010/T-019/T-020 feed the app-shell integration, and T-022 is the only authority on whether Release is clean. Communication is equally simple: claim the task id first, list the exact files you’re touching before the first commit, and if you need a contract change or a shared-file edit outside your lane, stop and call it before coding. One open PR at a time for shared integrator files. Squash-merge each task into `main` only after its acceptance criteria and lane tests pass. If two lanes disagree at a seam, the upstream contract owner wins and the downstream lane rebases; we do not “meet in the middle” by smearing logic across layers.

**Claude (Backend / Systems Engineer · Pragmatist) — Round 2**

Codex's fixes are the right ones — Route.archivedPeople was genuinely ownerless in my draft, and NurtureApp.swift absolutely needs a single named owner instead of becoming the file everyone "just quickly adds one line to." Both of those were real bugs in the division of labor, not nitpicks, and I'm folding them in without argument.

Where I want to push further, staying in pragmatist mode: the single biggest real-world source of pain in a multi-agent iOS build isn't logic bugs, it's `project.pbxproj` merge conflicts, and I don't think we've actually named the mitigation. Here's the concrete point — `NurtureDomain` and `NurtureData` are Swift packages, so anything under their `Sources/`/`Tests/` directories is auto-discovered by SwiftPM. Nobody touches `project.pbxproj` to add `WarmthCalculator.swift` or a new test file; `Package.swift` just needs the target declared once in T-001, and every source file after that is free. That means T-002 through T-009 — the whole `data_domain` and half of `services_utilities` lane — can run fully parallel with zero pbxproj risk. The one place we're still exposed is the main `Nurture` app target, where T-011 through T-020 all add new Swift files under `Nurture/Views/`, `Nurture/Components/`, `Nurture/Debug/`. If T-001 sets that target up as an Xcode 16+ "file system synchronized group" (folder reference, not enumerated group membership), those files also get auto-picked-up with zero pbxproj diffs. I want that stated explicitly as a T-001 acceptance criterion, because if whoever builds the scaffold doesn't do it, every single primary_ui task turns into a pbxproj-edit race and Codex's whole "serialize shared integrator files" plan becomes the bottleneck for the majority of the backlog instead of the exception.

Second thing, smaller but real: Codex's T-006 acceptance criteria says "ProductIdentifiers constants are the single source of truth used by both EntitlementService and the StoreKit config wiring" — but a `.storekit` file is a JSON blob Xcode edits, it cannot literally import a Swift constant. That sentence reads like the two are structurally the same source, and they can't be. What actually has to happen is a test that parses the `.storekit` file's product identifier strings at runtime and asserts equality against `ProductIdentifiers.swift`. If we don't say that explicitly, someone builds T-006, eyeballs that the strings "look the same," and we've silently reintroduced the exact drift risk the project_plan phase already flagged as a named risk. I'd rather spell out "an automated test reads the .storekit JSON and diffs it against the Swift constants" than leave it implied.

Third, small gap: T-001's file list doesn't include a `.entitlements` file, but the App Group go/no-go decision from three phases ago requires one to exist (even if it ends up empty/App-Group-free once the team-less signing test runs). I'm adding `Nurture/Nurture.entitlements` to T-001 so that decision has an actual file to land in.

On process — Codex's branch-per-task naming (`feature/t007-repository` etc.), lane-owns-its-contracts rule, and "upstream contract owner wins, downstream rebases" tie-breaker are all exactly right and I'm not touching them. I'd just make the testing responsibility split explicit since the phase asks for it directly: `data_domain` is responsible for `swift test --package-path NurtureDomain` staying green, full stop, before anything merges; `services_utilities` owns `swift test --package-path NurtureData` plus the named `SKTestSession`/`xcodebuild test -testPlan` runs; `primary_ui` owns manually walking every named state (empty/loading/error/success, VoiceOver grouping, Dynamic Type XXL) in-simulator per screen before marking a task done, since that can't be scripted the way domain logic can; `polish_resilience` owns the one binding check nobody else can fake — the `verify_release.sh` archive-and-grep pass — plus auditing that docs claim only what's actually verified. Communication protocol: claim a task id before touching its files, state the exact file list in your first commit, and if you hit a shared/integrator file outside your lane's ownership, stop and flag it in the room before coding around it — don't quietly "just fix it while I'm in there."

One real thing I do want fixed before we call this locked: T-001 and T-006 both list `nurture-ios/Nurture/Nurture.storekit` and the `NurtureStoreKit.xctestplan` as owned files. That's not a hypothetical — it's the exact "overlapping file ownership" the phase rules tell us to avoid. In practice it's harmless *because* T-006 strictly depends on T-001, so there's no simultaneous edit, but I don't want to leave it implicit. Let's just say it plainly: T-001 creates these two files as minimal, build-provable placeholders (an empty-product `.storekit` file just to prove the scheme wires up and signing succeeds), and T-006 is the sole subsequent editor once T-001 has merged to `main` — nobody touches those two files concurrently, and T-006's branch must rebase on `main` after T-001 lands before touching them. Same pattern we already use for T-024 as the named integrator exception; this is just a second, smaller named exception, not a new category of problem.

Second thing, and this is a boring but real gap: nobody's task owns `git init`, a `.gitignore` (excluding `DerivedData/`, `xcuserdata/`, `.build/`, `*.xcuserstate`), and the first commit. If T-001 doesn't explicitly do this, we get four agents independently discovering "wait, is this even a repo yet" at different times, which is exactly the kind of silent-assumption bug this group has burned itself on before (see: three rounds of nobody checking whether the App Group capability breaks team-less signing until someone actually said "let's test it"). I'm adding `.gitignore` to T-001's file list and "git init with an initial commit containing the empty package skeletons exists before any task branch is cut" to its acceptance criteria. Small, but it's the kind of thing that either exists on day one or causes a scramble on day three.

Third — and this is the pragmatist point I actually care about most, because it's about how we *physically* avoid stepping on each other rather than just who owns what on paper: since several lane workers are going to be operating against the same real working directory concurrently, disjoint file ownership on paper doesn't prevent one agent's uncommitted working-tree state from being visible to, or clobbered by, another agent's checkout. I want each active task branch worked in its own git worktree (`git worktree add ../nurture-ios-T-007 task/T-007-repository`, etc.) rather than everyone sharing one working copy of `main` and switching branches under each other. That's cheap, it's exactly what worktrees are for, and it means nobody's mid-task file edits are sitting in a shared tree where a sibling task's tooling might see half-written files. Merge is still trunk-based into `main` one task at a time, fast-forward or rebase-merge after the acceptance criteria are green — worktrees just protect the interim state, they don't change the merge model Codex already laid out.

Last thing on process, not architecture: every PR/merge into `main` should paste the actual command output in the description — the real `swift test` summary line, the real `xcodebuild build` result, the real named test case output — not a checked box. That's not new policy, it's just carrying forward the exact discipline `project_plan` already fought for (scriptable verification, not eyeballing), and it's cheap to enforce if we say it explicitly here instead of assuming everyone remembers it from a different phase's document.

Everything else — lane ownership, branch names, forbidden zones (no bare `Date()`, no SwiftData in UI or domain, no navigation state outside `AppRouter`, no debug code outside `#if DEBUG`, nobody but T-024 touching `NurtureApp.swift`), and the testing split by lane — I'm signing onto as Codex wrote it. That's a real, buildable division of labor and I don't think there's anything left to litigate.

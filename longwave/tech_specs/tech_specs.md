The smallest architecture that still protects this app from fake-feeling behavior is one stateful shell and a pure core. I would keep a single `@MainActor @Observable` app engine as the only mutable UI-facing object, and push all simulation, offline math, save migration, and entitlement reduction into plain Swift types and functions that can be tested without SwiftUI or StoreKit live state. I would split build work into four lanes that can move in parallel without inventing incompatible types. `data_domain` owns value types and pure simulation: relays, demand zones, regions, tuning, health state, derived metrics, offline resolution, progression goals, save schema. `services_utilities` owns snapshot storage, schema migration, clock abstraction, StoreKit entitlement resolution, and history retention. `primary_ui` owns the `@Observable` engine, enum-driven presentation state, SwiftUI screens, the single `Canvas` render surface, and the bottom-sheet interactions. `polish_resilience` owns recovery UX, accessibility action wiring, test fixtures, `.storekit` config, and corruption/offline edge-case verification. **Claude (Frontend Engineer · User Advocate) — Round 1**

Coming at this as the person who has to make the map feel instant and legible on a real phone in someone's hand during a coffee break, not just architecturally sound on paper — and as the person who'll reject anything that makes the first 30 seconds feel like a spreadsheet.

The architecture I want locked is a two-boundary system, enforced by the compiler, not by convention. Boundary one: a local Swift Package, call it `LongwaveSimulation`, that depends on nothing but `Foundation`. This is where `RelayType`, `Coordinate`, `RegionTopology`, `DemandZone`, `HealthLevel`, `SimulationCore`, the offline formula, and `GameSnapshot`'s schema types live. The reason this has to be a real package target and not just "a folder we agree not to import SwiftUI into" is that "agree not to" is exactly the kind of rule that survives until someone's in a hurry on a Tuesday and needs a UIColor for a debug print. A package boundary makes the violation a compile error, not a code-review miss. Boundary two is the app target itself, where `NetworkEngine` (the single `@MainActor @Observable` object) lives, along with persistence, commerce, and all views. `NetworkEngine` imports `LongwaveSimulation`, calls its pure functions with an injected clock, and is the only thing views ever touch.

One concrete gap I want to close that the design phase left slightly underspecified: `presentedSurface` was drafted with four cases (`.inspector`, `.paywall`, `.settings`, `.buildConfirm`), but the design also says the build sheet has three *modes* on one continuous surface (build/inspect/confirm) and that progression is a fifth reachable surface. If I build `presentedSurface` exactly as drafted, I either end up with progression as a stray boolean (the boolean-soup bug we're explicitly trying to avoid) or I awkwardly split "confirm" as a sibling of "inspector" when they're really the same sheet in a different mode. I want to restructure it as: `presentedSurface: PresentedSurface?` where `PresentedSurface` is `.buildSheet(BuildSheetMode)`, `.progression`, `.paywall`, `.settings` — and `BuildSheetMode` is its own enum (`.build(Coordinate)`, `.inspect(RelayID)`, `.confirm(PendingAction)`) nested inside. That's one property, one switch, and it matches "the sheet plus map are one continuous decision surface" literally in code, not just in spirit. Separately, the offline-return card and the recovery banner are *not* modal — they're non-blocking overlays on the map itself, so they need their own small piece of state, `mapOverlay: MapOverlay?`, distinct from `presentedSurface`. Conflating those two would mean a recovery banner could get dismissed by opening the build sheet, which is exactly the kind of accidental interaction a real user would experience as "the app randomly hid an important message."

For the coordinate system: I want to lock now, explicitly, that placement is on a discrete tile grid, not continuous floating-point coordinates. This isn't just an implementation preference — the design phase already commits to "move north/south/east/west in grid steps" for VoiceOver, and if the visual drag path uses continuous coordinates while the accessibility path uses discrete steps, we've recreated the exact "two input pipelines drifting apart" risk the architecture review spent a whole round trying to kill. One `Coordinate { var x: Int; var y: Int }`, one `RegionBounds { width, height }`, both paths convert screen-space taps/drags to the nearest tile and call the same `attemptPlace`/`attemptMove`. This also makes hit-testing trivial against the Canvas-rendered map — no scroll or zoom transform stack to invert, since there's a fixed viewport per region (no pan/zoom in MVP; I'm calling that a locked decision here since nobody's specified a camera system and one region fits on one screen). Interference itself stays continuous even though placement is discrete: `pairwiseInterference(distance:) = overlapFraction²` where `overlapFraction = max(0, (2×relayRadius − distance) / (2×relayRadius))`, with distance floored at a minimum before this formula runs so two relays can never divide by zero — but they also simply cannot occupy the same tile, so the true zero-distance case is prevented at the placement-validation layer, not papered over in the math.

On performance, I'm holding the line from the design phase hard: the Canvas/TimelineView layer takes plain structs (`[Relay]`, `[DemandZone]`, current health) as input and redraws itself every animation tick; it never owns SwiftUI view identity per relay. But there's a real interaction risk nobody's addressed yet — relays need to be tappable, draggable, and VoiceOver-addressable, and none of those work on a `Canvas` by itself. My answer: an invisible layer of ordinary SwiftUI views (one per relay, one per demand zone), positioned by the same coordinate-to-screen-space math the Canvas uses, purely for hit-testing, gesture recognition, and `accessibilityElement`/custom actions — zero visual content of their own. This keeps rendering cheap (one Canvas) while keeping interaction and accessibility using real, idiomatic SwiftUI view plumbing instead of hand-rolled hit-testing inside `Canvas`'s tap handler, which is the kind of thing that technically works and is a nightmare to keep accessible.

For persistence, the file layout should mirror the two-boundary split: `LongwaveSimulation` owns the `GameSnapshot` schema (version int, region dictionary, currency, milestone state, capped history ring buffer) as pure Codable structs with no I/O logic in them at all. The app target owns `SnapshotStore`, which does the actual `FileManager` work — temp-file-then-rename, corrupted-file preservation with a timestamped suffix, fallback to last-known-good. I want `SnapshotStore.load()` to return a `LoadResult` (`snapshot` plus an optional `recoveryNotice` enum case) rather than throwing past a recovery attempt — that way the caller can't accidentally skip presenting the recovery banner, because the notice is sitting right there in the return value forcing a decision. Health, throughput, and other derived numbers never enter this schema; they're recomputed from topology on load, every time, so there is structurally no way for a stale persisted health value to disagree with what `SimulationCore` would compute fresh.

Testing strategy: because we have zero third-party dependencies, we don't get snapshot-testing libraries for the visual states, so I want fixture-driven SwiftUI `#Preview`s treated as a real deliverable, not a nice-to-have — a `NetworkEngine` initializer that takes an injected fixture snapshot and a fixed clock, with one preview per named state from the design phase (empty, healthy, strained, choking, placement-preview, offline-return, recovery-banner, console-waking). That's how "every screen has a real empty/loading/success/error state" gets verified by a human looking at it, rather than asserted only in XCTest. On top of that, `LongwaveSimulation` gets pure XCTest coverage with zero simulator dependency (offline formula at 8h/400d, distance-floor/interference continuity, identical-coordinate rejection), and the app target gets a synchronous rapid-fire test against `attemptPlace`/`attemptUpgrade` asserting exactly one deduction, plus the `.storekit`-config-driven entitlement state tests already agreed in the architecture phase.

My own biggest risk/objection to my own proposal: splitting the simulation core into a real local Swift Package adds a bit of Xcode project-generation complexity versus a single flat app target with disciplined folders, and if this MVP is being scaffolded headlessly, package resolution and target wiring is one more thing that can go subtly wrong in a way a flat target can't. I still think it's worth it because it turns "please don't import SwiftUI here" from a code-review nitpick into a compiler error, but I want that tradeoff on the table rather than assumed away.

The one thing I most want preserved out of this phase regardless of how the debate goes: `presentedSurface` and `mapOverlay` as two separate, enum-driven, mutually-independent pieces of state (not four-plus booleans, and not one enum trying to cover both modal and non-modal presentation), and the discrete-tile coordinate system shared identically by the drag path and the VoiceOver path. Both are cheap to write down now and expensive to untangle after two build lanes have each written their own half of it.

**Coordinator (Claude) — decision after round 1**

Here's where round 1 landed. Both Codex and Claude clearly agree on the big shape of the architecture — a hard split between a pure, framework-free simulation/domain layer and the app-facing engine/UI layer, defensive file-based persistence with quarantine-on-corruption, derived values (health, throughput) never persisted, entitlement reduced to a small enum the game layer consumes, and a `presentedSurface`-style enum instead of boolean soup. Nobody's contesting any of that.

But this round actually surfaced a real, concrete disagreement rather than resolving into one contract, and it's exactly the kind of thing this phase's acceptance check exists to catch: **Codex and Claude each produced their own `interfaces-json` block, and the two disagree on real substance, not just cosmetics.** Specifically:

1. **Coordinate system.** Codex's `Relay`/`DemandZone` use a continuous `CGPointCodable` position. Claude explicitly argues for a discrete tile-grid `Coordinate{x,y}` instead, reasoning that continuous drag-coordinates plus discrete VoiceOver grid-steps would recreate the "two input pipelines can drift apart" bug the architecture phase spent a whole round eliminating. This wasn't responded to by Codex — it's a live, unaddressed proposal, not an agreement.
2. **Engine type.** Codex's `GameEngine` is declared as a `struct` with `mutating` async methods carrying `@Observable`. Claude's `NetworkEngine` is a `final class`. `@Observable` is meant for reference types acting as one shared source of truth across views — a mutating-struct engine doesn't actually give the "one object every screen reads" guarantee the design phase locked in. This needs an explicit correction, not a silent pick.
3. **`PresentedSurface` shape.** Codex's draft only has 4 cases and no slot for the progression screen or the sheet's three internal modes (build/inspect/confirm) as one continuous surface. Claude restructured it into `PresentedSurface{buildSheet(BuildSheetMode), progression, paywall, settings}` plus a separate non-modal `mapOverlay` for the offline-return card and recovery banner. This is a genuine fix to a gap in Codex's draft, not just a style choice, and hasn't been confirmed by Codex yet.
4. **Naming duplication**: `RegionState` vs `RegionTopology`, `NetworkDerivedState` vs `NetworkHealthSnapshot`, `SimulationEngine` (protocol) vs `SimulationCore`/`NetworkActioning` (static funcs + protocol), `Clock` vs `ClockProviding`, `SaveRecoveryBannerModel` vs `RecoveryMessage`. ### Round 2


**Codex (QA & Risk · Pragmatist) — Round 2**

Claude’s tile-grid push is the strongest correction on the table and I don’t think Codex should fight it. The whole app has been bending toward “one simulation, two thin callers” for three phases now, and continuous coordinates quietly break that promise. If drag uses arbitrary `CGPoint` values while VoiceOver moves in discrete N/S/E/W steps, you’ve created two different games that merely look similar. Then all the scary bugs are soft failures: a player drags a relay to a sub-tile sweet spot the VoiceOver path can never reach, offline recomputation rounds differently than preview, or a “too close” rejection depends on floating-point jitter at the placement boundary. None of those crash. They just make the system feel dishonest. For this MVP, “network topology” does not need analog precision. Discrete grid cells are a feature, not a compromise: they make previews stable, accessibility real, persistence simpler, corrupted-save diffing legible, and test fixtures human-readable. If later we want a smoother visual feel, we can animate *rendered positions* between tile centers without changing authored state. That’s the correct line: discrete authored coordinates, continuous presentation.

I also think Claude’s split between `presentedSurface` and `mapOverlay` is not optional cleanup; it fixes a real state-model bug before it ships. Offline-return and recovery are non-modal facts about the world. They should survive opening the sheet, and they should not compete with paywall/settings for the same enum slot. Trying to shove both modal surfaces and passive overlays into one state machine is how you get “recovery banner vanished because I tapped a relay” and nobody notices until QA. Same story with the engine type: it has to be a `final class`. A mutating `@Observable` struct here is category error. One shared state owner across five surfaces, StoreKit refreshes, scene-phase writes, and synchronous attempt* mutations wants reference semantics. On the simulation contract, I’d simplify the provisional merge slightly: keep a protocol-shaped simulation service, but keep the derived model broad enough to include steady-state credits/sec so offline math, map meter, progression, and VoiceOver preview all read one computed payload. The one place I’d tighten Claude’s version is persistence: don’t put “throwing past recovery” anywhere near the main load path. `load()` returning `LoadResult` with an optional recovery notice is exactly right because it forces the UI to decide how to surface recovery. That is a risk-control move, not just API taste.

The other pragmatic thing I want locked now is the boundary between preview and commit. We keep saying “every action previews consequence before commit,” but if preview and commit run through different formulas, we’ve failed even if the type names look clean. So the contract should make preview and commit siblings on the same simulation service, both operating on the same `RegionTopology` and producing the same `PlacementOutcome` shape. Then tests can assert “preview.after == committed derived state” for legal actions. That’s the kind of boring invariant that actually keeps a game trustworthy. I’d rather ship three relay types, one region, and a hard tile grid with perfect preview/commit honesty than chase an illusion of analog placement and spend the rest of the build arguing with float math.

**Claude (Frontend Engineer · User Advocate) — Round 2**

Codex's round 1 lines up with mine almost everywhere it matters — same hard split between a pure domain/simulation layer and a stateful engine, same "derived values never persisted," same capped Pro history as summaries not raw frames, same "lapse means frozen, not destroyed." Good, no fight there. But the coordinator's right that the two `interfaces-json` blocks don't actually match, and two of those mismatches aren't stylistic, they're the kind of thing that breaks a build if we don't settle it now.

On coordinates: Codex, you went with `CGPointCodable` for `Relay.position`, and I want to actually steelman why someone would want that before I re-argue discrete — a continuous position lets relays sit at organic, non-gridded spots, which fits the "warm analog console, not a spreadsheet" aesthetic better than everything snapping to a lattice, and it makes coverage-falloff circles overlap in visually richer ways than uniform grid spacing would. That's a real point. But it doesn't actually require the *authoritative* position to be continuous — it only requires the *rendered* position to look organic. So here's the synthesis I want to lock: `Coordinate` stays the discrete, authoritative grid position everywhere in `GameSnapshot`/`RegionTopology`/interference math/VoiceOver stepping — that's non-negotiable, because the whole reason we're doing this is so the drag path and the accessibility path are provably the same math, not two approximations that happen to agree today. But the *screen rendering* of a relay doesn't have to snap instantly between grid cells — during an in-flight drag gesture, the view interpolates a continuous on-screen position purely as `@GestureState`/`@State` local to the view, never written into `GameEngine` or `RegionTopology`, and snaps to the nearest valid `Coordinate` on release, calling `attemptMove` exactly once at that point. That gives you the organic, alive feel of relays gliding around during interaction, keeps the "one entry point, two thin callers" guarantee completely intact, and means nobody's ever tempted to add a `CGFloat` field to `Relay` "just for smoothness." I think this is strictly better than either of our round-1 positions and I'd push back hard if someone tried to put continuous state back into the persisted model — that's exactly the seam where the map and the accessibility path could start disagreeing about where a relay actually is.

On the engine being a class: this one I'm not going to soften on, because I think Codex's `struct GameEngine` with `mutating func`s is a real bug, not a style choice. The entire reason every other lane's spec — mine and Codex's both — keeps repeating "the map, progression screen, and paywall all read the same derived state" is that `@Observable` on a reference type gives you exactly one instance that every view in the hierarchy can hold via `@Environment(GameEngine.self)` and see the same mutations land. If `GameEngine` is a value type, the moment it crosses a view boundary — map to sheet to paywall — you either need Binding threading through every single initializer by hand, or you get a copy, and a stale copy of "am I Pro" or "what's the current health" showing in a sheet that's one mutation behind the map is precisely the "quietly wrong, not obviously broken" failure mode Codex spent a whole paragraph warning about. `@Environment` injection for cross-screen shared state basically requires a class here. I don't think this is a real disagreement so much as an oversight — I'd bet Codex agrees once it's phrased as "how does the paywall sheet see the same `entitlement` value the map just read" rather than in the abstract.

`PresentedSurface`/`BuildSheetMode`/`MapOverlay` — glad the coordinator flagged this as a real gap rather than a style pick, because Codex's original 4-case enum genuinely couldn't express "progression is a fifth surface" or "confirm is a mode of the same sheet, not a sibling of inspect" without either a stray boolean or an awkward re-split. I'll defend the merged shape as-is. One addition though: I want a named, shared pure function — `func screenPosition(for coordinate: Coordinate, bounds: RegionBounds, viewportSize: CGSize) -> CGPoint` — that both the `Canvas` renderer and the invisible per-relay hit-testing/accessibility overlay call to go from grid space to screen space. If those two things each do their own coordinate math instead of sharing one function, you get the exact bug where the visible relay dot and its tap target are a few points apart on some screen size, which is invisible in the simulator on one device and a real "why can't I tap my own relay" complaint on another. Small function, but it's the same "one source of truth" discipline applied to geometry instead of game state, and it's cheap to write down now.

On naming, I'll ratify the coordinator's merge picks — `RegionTopology` over `RegionState` (topology is the more accurate word for what it actually holds), the broader `NetworkDerivedState` over my narrower `NetworkHealthSnapshot` since Codex is right that offline math genuinely needs `steadyStateCreditsPerSecond` sitting next to health rather than as a separate calculation someone has to remember to keep in sync, `ClockProviding` over bare `Clock` (reads better as a capability name), and `RecoveryMessage` over `SaveRecoveryBannerModel` since it's shorter and used in two places (map banner, settings), not just a banner. On `SimulationEngine` taking `inout RegionTopology`/`inout Decimal` as two separate inout params for the commit functions — I'll live with it, but I'd flag for whoever implements this that it's slightly easier to accidentally call `commitPlace` with a topology and a currency value that came from two different snapshots if they're threaded as separate inouts instead of one inout `GameSnapshot`-scoped struct. Not worth blocking on, just worth a code-review note later.

### Round 3


**Codex (QA & Risk · Pragmatist) — Round 3**

Claude’s merged contract is basically the right one, and I don’t think we should waste another round pretending there’s still a conceptual disagreement here. The leftover mismatches are naming and completeness issues, not architectural forks. I’m with Claude on `NetworkEngine` over `GameEngine` because that object is not “the game” in any useful bounded sense; it’s the main-actor orchestration shell around simulation, persistence, entitlement, and presentation. Calling it `GameEngine` invites the exact god-object creep we’ve been trying to fence off since the architecture review. I’m also firmly on the side of preview coverage for all four action kinds. If Codex says “every important action previews consequence before commitment,” then `retune` and `upgrade` cannot be second-class citizens that only reveal their effect after spend. That would be a real product regression, not a type-naming nit. Same thing with `resolveOffline`: keep one public entry point. Internally we can split elapsed-resolution and earnings math into helpers, but the contract the rest of the app tests against should stay “given topology, lastActive, now, policy, tell me the offline result.” That’s the smallest surface that still proves the locked promise.

The one extra thing I want nailed down, because it’s where these specs usually lie by omission, is failure ownership. `bootstrap()` and `handleScenePhaseChange(_:)` are `async`, but the app-facing mutate paths are synchronous; that split is correct only if the async side never mutates currency/topology halfway through a save or entitlement refresh. In other words: StoreKit refresh, snapshot load/save, and recovery messaging can be async, but the committed game-state transitions that spend credits or move relays must remain single-turn main-actor mutations bracketed around pure simulation calls. If we let “scene went to background” race a relay upgrade, we’ll create the exact kind of once-in-200 sessions corruption bug that survives casual testing and shows up in production reviews as “my credits disappeared.” The merged contract already points the right way, so I’d stop debating and lock it as-is, with one practical interpretation attached: `NetworkEngine` owns sequencing, `SimulationEngine` owns truth, `SnapshotStoring` owns atomic durability, and nobody else gets to be clever.

**Claude (Frontend Engineer · User Advocate) — Round 3**

All four of the coordinator's rubber-stamp items are right and I'm not going to relitigate any of them — `NetworkEngine` as the name, full four-way preview coverage on `SimulationEngine`, one `resolveOffline` entry point, and `MapInteractionNode` riding on `screenPosition`. Codex's concession on coordinates last round was the correct call and I want to say clearly: "discrete authored coordinates, continuous presentation" is a better one-line summary of that decision than anything I wrote, and I'm adopting that phrasing going forward.

But sitting with the merged contract one more time, as the person who has to actually wire up the retune controls in the bottom sheet, I found a real mismatch nobody's caught yet: `Relay.tuning` is typed as `RelayTuning { powerBias: Double; channelOffset: Double }` — two independent continuous knobs — but `attemptRetune`/`previewRetune`/`commitRetune` all take a single `delta: Int`, and the design phase locked "retune up/down" as *one* accessibility action, not two separate sliders for power vs. channel. Those two facts can't both be true. If `RelayTuning` really has two independent dimensions, then "retune up/down" is meaningfully ambiguous — up/down *which* axis? — and a sighted player dragging some hypothetical 2D tuning control would have access to a state space the VoiceOver path structurally cannot reach with a single up/down stepper, which is exactly the two-input-pipelines-diverge bug we just spent two rounds killing for placement. I think this crept in because Codex's round-1 draft introduced the two-Double struct for "more analog, realistic tuning feel," which is the same instinct as the continuous-coordinate proposal, and deserves the same fix: collapse it to one authoritative discrete lever. Concretely, `Relay.tuning` should just be `var tuningLevel: Int` (bounded, say `0...tuningLevelMax`), matching what I actually drafted back in round 1 before Codex's richer struct replaced it in the merge. If we want power/channel-style flavor text in the inspector ("retuned toward stability" vs "toward throughput"), that can be cosmetic copy derived from `tuningLevel` inside `SimulationCore`/`SimulationEngine`, not a second persisted dimension. This is a small change but I want it locked before build starts, because "how many independent things can a player tune per relay" is a real game-design fact, not an implementation footnote, and right now the type system and the interaction spec disagree about the answer.

Separately, since we're finalizing: nobody has written down the actual function that turns `EntitlementState` (global) plus `hasLocalRegionTwoData()` (per-device fact) into `RegionAccessState` (per-region, what the Pro screen and region-two map actually render). Both pieces of raw material exist in `EntitlementProviding`, but if the reduction logic itself isn't a named, shared thing, I'd bet money two different lanes end up hand-rolling slightly different if/else trees for "what do I show when restored-but-empty vs. lapsed-frozen," and one of them gets the four-state matrix subtly wrong — this is precisely the kind of state that's been called out three phases running as the one most likely to ship untested. I want one small pure function for this, owned by services since it's the thing already touching both StoreKit and local save presence.

Everything else in the merged block I'm signing off on as-is — the `PresentedSurface`/`BuildSheetMode`/`MapOverlay` split, `NetworkEngine` as a `final class`, the full preview surface, `screenPosition` plus `MapInteractionNode`, single `resolveOffline`. I think we're done except for the tuning-shape fix and the access-resolution function, and I'd call this ready to lock once those land.

**Codex (QA & Risk · Pragmatist) — Round 4**

**Claude (Frontend Engineer · User Advocate) — Round 4**

Good news is my two round-3 catches didn't get any pushback — Codex's prose didn't explicitly say "yes" to `tuningLevel: Int` or `resolveRegionAccess`, but nothing in the round-3 message contradicts them either, and the reasoning hasn't changed, so I'm treating both as settled and moving on rather than re-arguing something nobody's actually objected to.

Where I do want to push back is Codex's `NetworkEngine` protocol / `LiveNetworkEngine` concrete-class split. I get the instinct — it matches the shape of `SimulationEngine`, `EntitlementProviding`, and `SnapshotStoring`, which are all protocols with swappable implementations, so making the top-level orchestrator follow the same pattern feels like consistency. But look at what those three protocols actually abstract: `EntitlementProviding` wraps StoreKit, a real external system you need a fake for in tests. `SnapshotStoring` wraps `FileManager`, a real external system you need an in-memory fake for in tests. `SimulationEngine` is pure logic but still gets a protocol because in principle you might want a stub that returns canned `NetworkDerivedState` without running real interference math. All three protocols exist because there's something concrete on the other side worth swapping out. `NetworkEngine` doesn't have that property — it's not wrapping an external system, it's the orchestrator sitting *on top of* things that are already wrapped. If I want a fake network state for a `#Preview` or a unit test, I don't need a fake `NetworkEngine` — I need a real `NetworkEngine` constructed with fixture data and fake dependencies underneath it, which I can already do by giving it a proper designated initializer that takes `snapshot`, `derived`, `entitlement`, `regionAccess` directly, plus injected `SimulationEngine`/`SnapshotStoring`/`EntitlementProviding`/`ClockProviding`. That gets every `#Preview` I described back in round 1 — empty, healthy, strained, choking, placement-preview, offline-return, recovery-banner, console-waking — without a second type name to keep straight and without the risk Codex's own note had to spell out in so many words: "every SwiftUI view injects the CONCRETE `LiveNetworkEngine`... never the protocol existential." If the safety of an abstraction requires a guardrail sentence explaining how not to misuse it, that's usually a sign the abstraction is buying less than it costs. I'd rather not have the two-name confusion we already burned two rounds cleaning up (`GameEngine` vs `NetworkEngine`) recur one level down as `NetworkEngine` vs `LiveNetworkEngine`.

And for the one case where you actually do want the real running app to behave differently — XCUITest driving the black-box app for the four-entitlement-state or recovery-banner UI flows — that doesn't go through a protocol swap anyway, since XCUITest can't inject a conforming fake into a process it only talks to from outside. That's done with launch arguments that tell the real `NetworkEngine`'s init to load fixture data instead of the real snapshot store, which is exactly the same designated-initializer mechanism I just described for Previews. So I don't think the protocol layer earns its keep anywhere — not in Previews, not in XCTest, not in XCUITest. Codex's underlying goal (testable, preview-friendly, swappable state) is completely right, I just think the fixture-initializer route gets there with one type instead of two and without the `@Environment` footgun. I'm reverting to a single `final class NetworkEngine: NetworkActioning` with an explicit fixture-capable initializer, and everything else — the tuning fix, `resolveRegionAccess`, the four-way preview coverage, single `resolveOffline`, and Codex's good sequencing point about async bootstrap/scenePhase/restore never interleaving with the synchronous spend/move/retune/upgrade path — stays exactly as landed.

## Final Output

**Architecture overview:** Two hard boundaries. (1) A Foundation-only local Swift Package, `LongwaveSimulation`, holding every pure domain type and the `SimulationEngine` protocol — no SwiftUI/UIKit imports, no direct `Date()` calls, clock always injected via `ClockProviding`. This is a compiler-enforced boundary, not a folder convention. (2) The app target, where a single `@MainActor @Observable final class NetworkEngine` owns all user-visible state (topology, currency, entitlement, presentation) and is the sole router for mutation. There is exactly one synchronous `attemptPlace`/`attemptMove`/`attemptRetune`/`attemptUpgrade`/`preview` entry point (the `NetworkActioning` protocol) — drag gestures, the confirm sheet, and VoiceOver custom actions are all thin callers into the same `NetworkEngine` instance, injected via `@Environment(NetworkEngine.self)` everywhere. Async work (`bootstrap()`, `handleScenePhaseChange()`, `restorePurchases()`) is strictly separated from the synchronous spend/move/retune/upgrade transitions — the two never interleave on the same state. Coordinates are discrete tile-grid positions everywhere authoritative (persisted state, interference math, VoiceOver stepping); any organic on-screen motion during a live drag is view-local interpolation that snaps to a tile on release. Relay tuning is a single bounded `Int` lever, matching the one "retune up/down" action — not a multi-axis continuous space no accessibility path could reach. The live signal/coverage/interference layer renders as one `Canvas` + `TimelineView(.animation)` fed by plain structs; an invisible `MapInteractionNode` overlay (using the same `screenPosition` conversion the renderer uses) provides hit-testing and accessibility so the visible dot and its tap target can never drift apart.

**File/module layout:**
- `LongwaveSimulation` package: `Sources/LongwaveSimulation/{Coordinate,RegionBounds,RegionID,RelayType,Relay,DemandZone,RegionTopology,HealthLevel,RelayMetric,NetworkDerivedState,ClockProviding,ElapsedResolution,OfflinePolicy,OfflineSummary,PlacementError,PlacementOutcome,SimulationEngine,MilestoneState,HistoryEntry,ProHistoryBuffer,EntitlementState,RegionAccessState,GameSnapshot,ScreenPosition}.swift`; `Tests/LongwaveSimulationTests/*` (pure, zero-simulator XCTest).
- App target: `App/LongwaveApp.swift`; `Engine/NetworkEngine.swift`, `Engine/NetworkActioning.swift`, `Engine/PresentedSurface.swift` (+ `BuildSheetMode`, `PendingAction`, `MapOverlay`); `Services/Persistence/SnapshotStore.swift` (conforms `SnapshotStoring`); `Services/Commerce/StoreKitEntitlementService.swift` (conforms `EntitlementProviding`) + `Services/Commerce/ResolveRegionAccess.swift`; `Services/HistoryRetentionService.swift`; `UI/Screens/{MapScreen,BuildInspectSheet,ProgressionScreen,PaywallScreen,SettingsScreen}.swift`; `UI/Components/*`; `UI/Rendering/NetworkCanvas.swift`, `UI/Rendering/MapInteractionNode.swift`; `Accessibility/AccessibilityRelayActionFactory.swift`, `Accessibility/VoiceOverDelta.swift`; `Tests/ServiceTests/*`; `UITests/*`.

**Data models:** `GameSnapshot` is the sole persisted document (schema version, save sequence, wall-clock + last-active timestamps, per-region `RegionTopology`, currency, `MilestoneState`, `ProHistoryBuffer`, last `OfflineSummary`). Derived values — `NetworkDerivedState` (health, throughput, steady-state rate, per-relay metrics, bottlenecks) — are never persisted, always recomputed from topology, and are the one contract every surface (map meter, progression screen, VoiceOver deltas, Pro history) reads. `RegionAccessState` (per-region UI state) is produced only by `resolveRegionAccess(entitlement:hasLocalData:)`, never hand-rolled per screen.

**Persistence strategy:** Hand-rolled `Codable` snapshot store, atomic temp-file-then-rename writes, `current.json`/`lastKnownGood.json` rotation, corrupted files preserved under a timestamped quarantine name before rollback. `load()` returns a `LoadResult` carrying an optional `RecoveryNotice` so recovery UI can't be silently skipped. Offline earnings resolve once on foreground resume via the single `resolveOffline` closed-form entry point; last-active timestamp is written on `scenePhase → .background`, not only on clean termination. No background execution modes.

**Testing strategy:** Pure `LongwaveSimulationTests` cover interference continuity, identical-coordinate rejection, 8-hour/400-day offline gaps, clock-drift floor/cap thresholds — all deterministic via injected `ClockProviding`, no simulator needed. App-target tests cover a synchronous ten-rapid-calls assertion against `attemptPlace`/`attemptUpgrade` (exactly one deduction), corrupted-save-on-disk recovery, and a restore-state matrix test asserting `resolveRegionAccess` returns the correct one of four states for every entitlement × local-data combination. A local `.storekit` config exercises all four entitlement states in-simulator. VoiceOver custom actions are asserted to exist and route through the same `NetworkActioning` entry point via XCUITest; actual spoken-experience quality still needs one real manual VoiceOver pass. Fixture-driven `#Preview`s (via `NetworkEngine`'s fixture initializer) cover every named screen state from the design phase.

## Final Output

**Architecture overview:** Two hard boundaries. (1) A Foundation-only local Swift Package, `LongwaveSimulation`, holding every pure domain type and the `SimulationEngine` protocol — no SwiftUI/UIKit imports, no direct `Date()` calls, clock always injected via `ClockProviding`. This is a compiler-enforced boundary, not a folder convention. (2) The app target, where a single `@MainActor @Observable final class NetworkEngine` owns all user-visible state (topology, currency, entitlement, presentation) and is the sole router for mutation. There is exactly one synchronous `attemptPlace`/`attemptMove`/`attemptRetune`/`attemptUpgrade`/`preview` entry point (the `NetworkActioning` protocol) — drag gestures, the confirm sheet, and VoiceOver custom actions are all thin callers into the same `NetworkEngine` instance, injected via `@Environment(NetworkEngine.self)` everywhere. Async work (`bootstrap()`, `handleScenePhaseChange()`, `restorePurchases()`) is strictly separated from the synchronous spend/move/retune/upgrade transitions — the two never interleave on the same state. Coordinates are discrete tile-grid positions everywhere authoritative (persisted state, interference math, VoiceOver stepping); any organic on-screen motion during a live drag is view-local interpolation that snaps to a tile on release. Relay tuning is a single bounded `Int` lever, matching the one "retune up/down" action — not a multi-axis continuous space no accessibility path could reach. The live signal/coverage/interference layer renders as one `Canvas` + `TimelineView(.animation)` fed by plain structs; an invisible `MapInteractionNode` overlay (using the same `screenPosition` conversion the renderer uses) provides hit-testing and accessibility so the visible dot and its tap target can never drift apart.

No background execution modes.

A local `.storekit` config exercises all four entitlement states in-simulator. VoiceOver custom actions are asserted to exist and route through the same `NetworkActioning` entry point via XCUITest; actual spoken-experience quality still needs one real manual VoiceOver pass.

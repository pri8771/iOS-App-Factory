## Final Output

**Screen inventory (5 screens, map is the product):**
1. **Map** — full-screen, persistent home surface, no tab bar. Translucent floating toolbar overlaid on top holds: currency counter, the health readout (analog-needle-style with a text/icon label, never color-only), and two icon buttons (progression screen, settings).
2. **Build/inspect panel** — a bottom sheet (Apple Maps POI-card pattern) that opens when a relay or valid empty tile is tapped. Partial-height by default, draggable to full height for retune controls, dismiss by swipe-down or tapping the map. Three modes: build (nothing selected), inspect (relay selected), confirm (about to spend currency). This sheet plus the map are one continuous decision surface — nothing here counts as "leaving" the map.
3. **Progression/offline-summary screen** — a compact, four-block dashboard only: currency total, current network health (same source as the map's readout, never a second calculation), last-resume offline-earnings summary, and the near-term goals checklist. Pro history/heatmap appears here as an expandable section, gated to entitled users, collapsing to one flat span when no configuration changed offline.
4. **Pro/paywall screen** — modal sheet, reachable from a persistent small toolbar affordance and from upsell entry points inside the progression screen. Shows the locked second region's different topology in concrete gameplay terms, the bigger offline window, and a heatmap/history preview. Explicit entitlement states: never-purchased, active, restored-but-locally-empty, and lapsed-frozen-read-only.
5. **Settings/recovery screen** — near-stateless: restore purchases (with its own loading/success/error), accessibility note, about/version, and recovery messaging only surfaces here when relevant.

**Primary flow:** Cold launch drops directly onto the map with a designed empty state — one demand zone pulsing gently, starter currency, one glowing "place relay" affordance that doubles as the entire onboarding (no modal tutorial). First tap places the cheap/broad relay; signal-flow animation and currency ticker start immediately (under the 30-second requirement). Placing a second relay near the first live-previews growing coverage overlap and thickening interference texture, with the health needle visibly dipping in preview — teaching the core tradeoff without narration. Committing anyway, watching health sag, then dragging relays apart or dropping in the stabilizer type to recover, is the full "success → self-inflicted failure → smarter fix" arc, entirely on the map. Ongoing loop: open app, read health at a glance, diagnose the bottleneck, preview a fix in the sheet, commit, watch the map change, close app, return later to an offline-summary card and the next fix.

**State model per screen:**
- *Map:* empty (zero relays, one affordance), healthy/strained/choking (three visually and texturally distinct sub-states tied to the health readout), placement-preview (ghost relay, live before/after delta, invalid-placement reason if blocked), selected-relay (sheet open), offline-return (lightweight summary card overlaid on the map, non-blocking), error/recovery (calm banner above the map, e.g. "we recovered your network from a backup").
- *Build/inspect sheet:* closed, inspect, placement-preview, insufficient-funds (button disabled with a stated reason, never a dead tap).
- *Progression screen:* first-launch-empty (no offline earnings yet, goals unchecked) and normal resume (offline summary, health snapshot, checklist).
- *Paywall:* default, purchase-in-progress, success, restore-in-progress, purchase-failed (the one screen where a live network call is unavoidable — copy must make clear a failure here is a store issue, not a break in local game state).
- *Settings:* effectively stateless aside from restore-purchases' own loading/success/error.

**Visual direction:** Warm analog broadcast-console as the physical shell — charcoal, oxidized brass, cream typography, relay housings and map terrain that read as tactile, inspectable hardware (a screenshot should look like "a solved or unsolved machine," not a generic sci-fi HUD). Layered on top, the live simulation is a glowing RF/signal system: pulsing signal-flow lines from each relay to its demand zone (speed/steadiness reflects real throughput), true gradient coverage falloff (never hard-edged), and crosshatch/moiré texture that thickens with interference — rendered with real contrast against the muted housing, since this layer carries all the causality information. Reference touchstone: a vintage receiver's wood-and-brass housing with a glowing green-phosphor scope face set into it — warm and physical below, cool and alive on the display. Restrained, precise motion (metering, not magic) — no heavy bloom or arcade glow.

**Accessibility notes:** Non-color redundant signaling everywhere (icon + text label for health/interference, never hue alone) is baseline. Beyond that, the core spatial mechanic itself gets a real non-visual path, not just a label sweep: relays and demand zones are exposed as VoiceOver accessibility elements with adjustable traits and custom actions (move north/south/east/west in grid steps, retune up/down, upgrade, commit/cancel), each speaking the projected health/interference delta before commit — same underlying placement-validation and interference math as the visual drag path, just a different input modality, built into the existing map/inspector rather than as a separate screen or duplicated data flow. Dynamic Type must not clip or break the inspector sheet at largest accessibility sizes.

### State and interaction / architecture notes (binding constraints, not suggestions for the builder to reinvent)
- The live layer (coverage falloff, interference density, signal pulses) is rendered as **one continuous simulation surface** — a single `Canvas` driven by a single `TimelineView(.animation)`, fed by plain data structs, not a `ForEach` of individually animated SwiftUI shapes. ## Ios Architecture Review

Codex and Claude converged pretty tightly this round, and interestingly there's not really a contested point to resolve — they arrived at almost the same architecture independently, and where Claude added specifics (synchronous atomicity, injected clock, `.xcprivacy` manifest, `.storekit` test config), Codex's framing already implied or was consistent with them. Let me lay out where they landed.

Both agree: one `@MainActor @Observable` engine object owns all game state, with a UI-agnostic, pure-function simulation core underneath (relays, interference, offline formula) that takes date/time as injected parameters rather than calling `Date()` directly — this is what makes the 8-hour/400-day/save-cycle acceptance tests actually runnable as fast unit tests instead of manual QA. Both explicitly reject SwiftData/Core Data in favor of hand-rolled Codable snapshots with temp-file-then-rename atomic writes, schema versioning, and timestamped corrupted-file preservation — reasoning being this app is one document-shaped save, not a relational dataset, and Core Data's autosave/change-tracking semantics fight the atomicity guarantees already locked. Both land on zero third-party dependencies, first-party frameworks only (SwiftUI, Observation, StoreKit 2, Canvas+TimelineView, Foundation/Swift concurrency). Both reject ML/LLM/AR as unjustified for this MVP — no gimmick advisor, no theme-leakage AR. Both agree offline earnings are computed on foreground resume only, no BGTaskScheduler/background execution modes at all.

Claude added a few concrete sharpenings Codex didn't contradict: the validate-and-mutate entry point must be synchronous with no `await` between validating and spending currency (otherwise "one entry point" doesn't actually prevent double-spend races); StoreKit's Pro check must treat *either* the lifetime product ID or the subscription as satisfying entitlement (not hardcode one); the last-active timestamp must be written on `scenePhase → .background`, not just on clean termination, since iOS commonly suspends-then-kills without a termination callback; a `PrivacyInfo.xcprivacy` manifest is needed even for a no-collection app because of required-reason API usage (UserDefaults, file timestamps); and a debug-only `.storekit` test configuration is needed to actually exercise all four entitlement states before ship, since "restored on new device, no local data" is the state most likely to never get manually tested otherwise.

None of this contradicts anything Codex said — Codex was already gesturing at the same shape ("entitlement resolution is its own small service with one output contract... not for gameplay"), Claude just made it concrete and testable. There's no live disagreement on the table.

## Final Output

**SwiftUI architecture:** Single-scene app. One `@MainActor @Observable` engine object owns all user-visible game state (relays, demand zones, currency, entitlement cache) and is the sole router for actions; no per-screen view models duplicating derived state. Underneath it, a plain-Swift, UI-framework-free simulation module holds pure functions for interference/throughput/health and the closed-form offline formula — it must never import SwiftUI/UIKit and must take any clock/date as an injected parameter, never call `Date()` internally, so 8-hour/400-day offline scenarios and save-cycle stress tests run as deterministic millisecond unit tests, not manual/simulator-timed checks. There is exactly one `attemptPlace`/`attemptMove`/`attemptRetune` entry point, and it must be fully synchronous end-to-end (no `await` between validating currency/state and mutating it) — structural sharing alone doesn't prevent a double-spend race, atomicity does. The live signal/interference/coverage layer renders as one `Canvas` driven by one `TimelineView(.animation)`, fed by plain data structs, consistent with the design-phase performance constraint.

**Apple framework choices:** SwiftUI + Observation for UI/state; StoreKit 2 (`Product`, `Transaction`, `Transaction.currentEntitlements`) for purchases, checked against both the lifetime and subscription product IDs so either satisfies "Pro" — never hardcoded to just one; `Canvas`/`TimelineView` for the simulation render layer; Foundation/Swift concurrency for async work; `FileManager`/`Codable` for persistence. No SwiftData, no Core Data — this app's save is one document-shaped snapshot with atomicity and recovery requirements that a hand-rolled store satisfies more directly than an object-graph framework's autosave/faulting semantics would.

**Persistence and local data plan:** Hand-rolled Codable snapshot store in the Documents/App Support directory (not user-visible in Files app — no `UIFileSharingEnabled`), written via temp-file-then-rename, never write-in-place. Schema includes a version int, monotonic + wall-clock save timestamps, region topology, relay placements/tuning, currency/progression, milestone/goal state, offline-cap inputs, and a capped history/event ring buffer for Pro diagnostics. Derived values (health score, throughput totals) are never persisted — always recomputed from canonical topology on load, so there's one source of truth. Corrupted saves are preserved under a timestamped filename before falling back to last-known-good or a clean network. Offline earnings resolve once on foreground resume from a stored last-active timestamp; that timestamp is written on `scenePhase` transitioning to `.background`, not only on clean termination, since suspend-then-jetsam-kill is the common path and would otherwise corrupt elapsed-time math. No background execution modes (no `BGTaskScheduler`) — consistent with "topology is frozen while closed."

**Permissions/privacy plan:** Zero runtime permission requests — no Photos, location, notifications, camera, microphone. Screenshot sharing via system share sheet (`ShareLink`) only. StoreKit is the only network-shaped touchpoint; entitlement reads come from StoreKit's local transaction cache at launch/foreground, never a blocking live call, so the app is fully playable offline including Pro-state checks. Copy should say "no gameplay data leaves your device" rather than an absolute "nothing leaves device" claim, since purchase/restore flows inherently touch Apple's infrastructure. A `PrivacyInfo.xcprivacy` manifest is required even though there's no data collection, because ordinary UserDefaults use and file-timestamp reads during save/recovery trip Apple's required-reason API categories — skipping this is a real App Store rejection risk, not a formality.

**Dependency/license decision:** Zero third-party packages. Every capability needed (rendering, storage, commerce, sharing, accessibility, testing) is covered by first-party frameworks, which also keeps the local-first/privacy story clean. Large-number display should first be attempted with a custom bounded decimal/formatting type rather than reaching for a big-number library; only reconsider a permissive-license dependency if a concrete balancing failure actually demands arbitrary precision.

**Testing implications:** The engine's UI-framework independence is what makes the already-locked acceptance criteria enforceable as automated tests rather than eyeballed QA: injected-clock tests for 8-hour/400-day offline gaps and the clock-drift floor/cap thresholds; a synchronous ten-rapid-calls test against `attemptPlace`/upgrade asserting exactly one deduction; a corrupted-save-on-disk test asserting recovery to last-known-good with the broken file preserved and timestamped; identical/near-identical coordinate placement rejection tests. For StoreKit, a local `.storekit` test configuration file should let all four entitlement states (never-purchased, active, restored-but-locally-empty, lapsed-frozen) be exercised in-simulator without real sandbox transactions — flagged specifically because "restored on new device with no local region data" is the state most likely to ship untested otherwise. VoiceOver custom actions can be partially automated (asserting they exist and route through the same validate-and-mutate entry point via XCUITest), but actual spoken-experience quality still needs a real manual VoiceOver pass — that can't be claimed as covered by automation alone.

## Final Output

**Architecture overview:** Two hard boundaries. (1) A Foundation-only local Swift Package, `LongwaveSimulation`, holding every pure domain type and the `SimulationEngine` protocol — no SwiftUI/UIKit imports, no direct `Date()` calls, clock always injected via `ClockProviding`. This is a compiler-enforced boundary, not a folder convention. (2) The app target, where a single `@MainActor @Observable final class NetworkEngine` owns all user-visible state (topology, currency, entitlement, presentation) and is the sole router for mutation. There is exactly one synchronous `attemptPlace`/`attemptMove`/`attemptRetune`/`attemptUpgrade`/`preview` entry point (the `NetworkActioning` protocol) — drag gestures, the confirm sheet, and VoiceOver custom actions are all thin callers into the same `NetworkEngine` instance, injected via `@Environment(NetworkEngine.self)` everywhere. Async work (`bootstrap()`, `handleScenePhaseChange()`, `restorePurchases()`) is strictly separated from the synchronous spend/move/retune/upgrade transitions — the two never interleave on the same state. Coordinates are discrete tile-grid positions everywhere authoritative (persisted state, interference math, VoiceOver stepping); any organic on-screen motion during a live drag is view-local interpolation that snaps to a tile on release. Relay tuning is a single bounded `Int` lever, matching the one "retune up/down" action — not a multi-axis continuous space no accessibility path could reach. The live signal/coverage/interference layer renders as one `Canvas` + `TimelineView(.animation)` fed by plain structs; an invisible `MapInteractionNode` overlay (using the same `screenPosition` conversion the renderer uses) provides hit-testing and accessibility so the visible dot and its tap target can never drift apart.

**File/module layout:**
- `LongwaveSimulation` package: `Sources/LongwaveSimulation/{Coordinate,RegionBounds,RegionID,RelayType,Relay,DemandZone,RegionTopology,HealthLevel,RelayMetric,NetworkDerivedState,ClockProviding,ElapsedResolution,OfflinePolicy,OfflineSummary,PlacementError,PlacementOutcome,SimulationEngine,MilestoneState,HistoryEntry,ProHistoryBuffer,EntitlementState,RegionAccessState,GameSnapshot,ScreenPosition}.swift`; `Tests/LongwaveSimulationTests/*` (pure, zero-simulator XCTest).
- App target: `App/LongwaveApp.swift`; `Engine/NetworkEngine.swift`, `Engine/NetworkActioning.swift`, `Engine/PresentedSurface.swift` (+ `BuildSheetMode`, `PendingAction`, `MapOverlay`); `Services/Persistence/SnapshotStore.swift` (conforms `SnapshotStoring`); `Services/Commerce/StoreKitEntitlementService.swift` (conforms `EntitlementProviding`) + `Services/Commerce/ResolveRegionAccess.swift`; `Services/HistoryRetentionService.swift`; `UI/Screens/{MapScreen,BuildInspectSheet,ProgressionScreen,PaywallScreen,SettingsScreen}.swift`; `UI/Components/*`; `UI/Rendering/NetworkCanvas.swift`, `UI/Rendering/MapInteractionNode.swift`; `Accessibility/AccessibilityRelayActionFactory.swift`, `Accessibility/VoiceOverDelta.swift`; `Tests/ServiceTests/*`; `UITests/*`.

**Data models:** `GameSnapshot` is the sole persisted document (schema version, save sequence, wall-clock + last-active timestamps, per-region `RegionTopology`, currency, `MilestoneState`, `ProHistoryBuffer`, last `OfflineSummary`). Derived values — `NetworkDerivedState` (health, throughput, steady-state rate, per-relay metrics, bottlenecks) — are never persisted, always recomputed from topology, and are the one contract every surface (map meter, progression screen, VoiceOver deltas, Pro history) reads. `RegionAccessState` (per-region UI state) is produced only by `resolveRegionAccess(entitlement:hasLocalData:)`, never hand-rolled per screen.

**Persistence strategy:** Hand-rolled `Codable` snapshot store, atomic temp-file-then-rename writes, `current.json`/`lastKnownGood.json` rotation, corrupted files preserved under a timestamped quarantine name before rollback. `load()` returns a `LoadResult` carrying an optional `RecoveryNotice` so recovery UI can't be silently skipped. Offline earnings resolve once on foreground resume via the single `resolveOffline` closed-form entry point; last-active timestamp is written on `scenePhase → .background`, not only on clean termination. No background execution modes.

**Testing strategy:** Pure `LongwaveSimulationTests` cover interference continuity, identical-coordinate rejection, 8-hour/400-day offline gaps, clock-drift floor/cap thresholds — all deterministic via injected `ClockProviding`, no simulator needed. App-target tests cover a synchronous ten-rapid-calls assertion against `attemptPlace`/`attemptUpgrade` (exactly one deduction), corrupted-save-on-disk recovery, and a restore-state matrix test asserting `resolveRegionAccess` returns the correct one of four states for every entitlement × local-data combination. A local `.storekit` config exercises all four entitlement states in-simulator.

Coming at this from "what breaks when a SwiftUI developer actually tries to build this," because the design_discussion phase left two things genuinely unresolved (navigation container, Export/Share state triple) and glossed over a few more that will bite hard once real PhotoKit/Vision/AVFoundation timing enters the picture.

First, navigation structure, since nobody committed to one yet. I'm against a TabView. This app's whole design thesis is "one calm hub screen, two cinematic drill-down moments," and a persistent tab bar is exactly the kind of chrome that fights full-bleed Day Detail and Month Reel playback — you either have to hide/show the tab bar per screen (which SwiftUI makes fiddly and prone to layout jumps when it reappears) or you accept a bar sitting under your ceremony animation forever. I'd build this as a single NavigationStack rooted at Home Calendar, with Archive and Settings reached via two toolbar buttons (a small shelf/history icon top-leading, a gear top-trailing), and Day Detail / Month Reel pushed as full-screen destinations that own their own chrome entirely. This also matches "the calendar is the actual hub" language from the design phase — if Archive and Settings were peer tabs, they'd visually claim equal importance to Home, which contradicts the stated hierarchy.

Second, the state model needs to stop being described as "visual states" and become an actual typed enum computed from data, because otherwise every screen risks drifting into ad hoc boolean flags that quietly collide — which is precisely the bug the last four phases spent so much effort forbidding (limited-access looking like no-video). I'd define a single `DayCellState` enum — `.hasPick(thumbnail)`, `.empty`, `.scanning`, `.waitingForICloud(progress)`, `.deferred`, `.limitedLibraryExcluded` (yes, a sixth state — the design phase's five don't actually cover the case where limited-library access means a day was never even eligible to be scanned, which is different from "scanned and found nothing," and I don't think those two can share a look either, or we've just moved the original bug one level down) — computed by a pure function of the day's PHAsset visibility state and DayRecord, never inferred by the view from multiple booleans. Every screen (calendar cell, day detail, archive tile) should render off one of these enums, not reconstruct state locally, or divergence is inevitable the first time someone touches one screen and forgets the other.

Third, I want to push back a little on how swap is architecturally described. The design phase says swap "invalidates only that day's slot in the render cache" — fine — but if swap synchronously kicks off a background re-render of the month reel every time it's tapped, someone who swaps three days in a row while reviewing their week generates three wasted AVFoundation exports. Swap should be cheap and instantaneous from the user's perspective (it's just switching which already-scored PHAsset plays in Day Detail — no new Vision work, no new export), and the month's cached reel should just be marked dirty. The actual re-render should be lazy, triggered the next time the user opens Month Reel and sees the "re-rendering after swap" state — not eagerly the moment the swap happens. That's a meaningfully different background job model than "swap invalidates cache" implies, and it needs to be explicit in the handoff so nobody builds an eager re-render pipeline.

Fourth, I'm skeptical of the reel-reveal-ceremony-shown-exactly-once framing without a concrete persistence rule. What happens if the app is killed mid-ceremony animation — does "shown once" get recorded at the start of the animation or at its completion? If it's recorded at start and the app dies before the user actually sees it, that ceremony is gone forever and the user's very first reel just silently downgrades to the plain cached view with no explanation. I'd record "ceremony seen" only on completion (including a completion path for users who skip/tap through it), so a killed app in the middle just means they see the ceremony again next launch — a minor repeat is a much smaller failure than permanently losing the one moment this whole app was designed around.

Fifth, on the calendar grid itself: nobody has stress-tested this against Dynamic Type yet, and it's the single screen most likely to fall apart under it. A month grid with serif date numerals and five-to-six distinct state glyphs per cell will not survive accessibility-sized text — cells will either truncate the state indicator or overlap neighboring cells. I'd mandate that the calendar switches from a grid to a vertical list of day rows once the user's Dynamic Type category crosses into the accessibility range, same data, same states, just laid out top-to-bottom with room for full-size text and glyphs — this needs to be a named requirement in the handoff, not something a builder discovers three weeks in when QA files it as a bug.

Sixth, Export/Share needs the same loading/success/error triple as Month Reel, not just the watermark-preview state called out earlier — an AVAssetExportSession for the export step can fail (storage pressure, permission revoked mid-export, backgrounding) exactly like the monthly render can, and if that path is undocumented a builder will likely just reuse a spinner with no error recovery, which contradicts the global rule that every reachable state needs explicit handling.

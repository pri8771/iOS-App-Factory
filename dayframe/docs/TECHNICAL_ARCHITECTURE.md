## Final Output

**Screen inventory:**
1. Permission primer + three distinct post-decision states (full access / limited access / denied) — visually distinct, not one shell with swapped copy
2. Home Calendar (the hub) — month grid, swipeable, always legible over cinematic
3. Day Detail — full-bleed autoplay of that day's pick, persistent swap control
4. Month Reel — the one screen with real ceremony (first-reveal assembly animation)
5. Reel Archive/History — shelf of month covers, free-tier older months shown frosted/desaturated (visible loss, not just a lock icon)
6. Export/Share — includes a preview state showing watermark-vs-clean before committing
7. Paywall — contextual variants triggered by the specific friction point (swap limit hit / locked archive month / clean export attempt), not one generic sheet
8. Settings — deliberately plain: subscription management, processing-cap/skip-reason disclosure, notification toggle, privacy statement

**Primary user flow:** Grant permission → land on current month's calendar (thin non-blocking backfill progress, never a full-screen spinner) → tap any scored day → plays immediately, no wait on month completion → swap in-place if wrong (same screen, snaps to next candidate) → back to calendar, watch it fill in → calendar itself signals "reel ready" (not just a notification) → open Month Reel, watch the one-time assembly ceremony, then straight playback on return visits → export/share with watermark preview → hitting a wall (swap limit, locked month, clean export) triggers the matching contextual paywall.

**State model per screen (the parts that actually gate correctness):**
- Calendar cell: has-a-pick / honestly-empty (permanent, never reads as pending or error) / scanning-in-progress / waiting-for-iCloud / deferred-low-signal — five states, must never visually collapse into each other.
- Day Detail: playing / no-next-candidate (swap control disabled, not erroring) / source-deleted ("no longer available," graceful) / iCloud-waiting-with-retry.
- Month Reel: not-ready-yet (concrete progress, "18 of ~22 days," not a bare percentage) / first-reveal ceremony (shown exactly once per month) / cached-return-view (instant, no re-render, no ceremony replay) / re-rendering-after-swap (visually distinct from first-reveal) / render-failed-with-retry (old cached reel stays playable throughout).
- Archive: unlocked / locked-frosted / current-month-in-progress.
- Paywall: renders differently per trigger context rather than one generic sheet.

**Visual direction:** A film/darkroom keepsake identity — near-black/charcoal stage so the video is the brightest thing on screen, one warm tungsten-amber accent (not iOS system blue), humanist/serif display type for dates and month names, thumbnails treated as frames (thin border, faint sprocket suggestion) rather than square Photos-grid tiles. Cinematic spend is deliberately concentrated on two moments only — Day Detail entry and the Month Reel reveal — with the Home calendar kept restrained and fast-scanning by design, since daily usability would lose to demo-polish if the frame metaphor bled into the grid itself. Reduce Motion gets a real fallback (simple cross-fade into cover art), not just "disable the animation."

**Accessibility notes:** Every calendar state needs an explicit VoiceOver label ("July 3rd, moment available" / "no video" / "waiting for iCloud" / "April, locked — subscribe to unlock") since the whole honest-empty-state design is meaningless if it's encoded visually only. Swap needs a real accessible button alongside any gesture — swipe-only would undermine the correctability wedge the entire product is built on. Dynamic Type must hold up specifically in reel-ready progress copy and the paywall, the two screens most likely to break under large text. The amber-on-charcoal palette needs an actual contrast check before final visual polish locks in, so the swap button reads as a control, not decoration.

## Final Output

**Navigation architecture:** Single `NavigationStack` rooted at Home Calendar — no `TabView`. Archive and Settings are reached via toolbar buttons (shelf icon top-leading, gear top-trailing); Day Detail and Month Reel push as full-screen destinations owning their own chrome. This preserves the "calendar is the one true hub" hierarchy from design_discussion — peer tabs would visually claim equal importance for screens that are deliberately secondary.

**State model, made concrete and typed:**
- `DayCellState`: `.hasPick(thumbnail)`, `.empty`, `.scanning`, `.waitingForICloud(progress)`, `.deferred`, `.limitedLibraryExcluded` — six states now, not five. The sixth exists because "day was never eligible to scan (limited-library permission)" is a genuinely different situation from "day was scanned and found nothing," and collapsing those two would reintroduce the exact bug the earlier phases banned. Every screen (calendar cell, day detail, archive tile) renders off this single enum, computed as a pure function of PHAsset visibility + DayRecord — never reconstructed locally per view.
- Day Detail: `.playing`, `.noNextCandidate` (swap disabled, not erroring), `.sourceDeleted` ("no longer available"), `.iCloudWaitingWithRetry`.
- Month Reel: `.notReadyYet(dayCount)` (concrete "18 of ~22 days," never a bare percentage), `.firstRevealCeremony` (recorded as seen only on completion/skip, not on start — so a kill mid-animation just replays it next launch instead of permanently losing the one moment the app is built around), `.cachedReturnView` (instant, no re-render), `.reRenderingAfterSwap` (visually distinct from first reveal), `.renderFailedWithRetry` (old cached reel stays playable throughout).
- Export/Share now gets the same loading/success/error triple as Month Reel (not just the watermark-preview state) — an `AVAssetExportSession` here fails for the same reasons (storage pressure, permission revoked mid-export, backgrounding).
- Archive: `.unlocked`, `.lockedFrosted`, `.currentMonthInProgress`.
- Paywall: renders per trigger context (swap limit / locked month / clean export), never one generic sheet.

**Swap/render sequencing (corrected from design_discussion's phrasing):** Swap is cheap and synchronous from the user's perspective — it only changes which already-scored PHAsset plays in Day Detail, no new Vision work. It marks the month's cached reel dirty but does **not** eagerly kick off a re-render; the actual re-render is lazy, triggered only when the user next opens Month Reel and sees `.reRenderingAfterSwap`. This avoids three wasted AVFoundation exports if someone swaps three days in a row.

**Accessibility, beyond what design_discussion named:** Calendar grid switches from a 7-column month view to a vertical list of day rows once Dynamic Type crosses into the accessibility size range — same states, same data, laid out top-to-bottom so text and state glyphs have room. Every `DayCellState` gets an explicit VoiceOver string. Swap is a real accessible button (not swipe-only). Contrast on the amber accent must be checked at ≥4.5:1 for text/control use, not just decorative use.

**Design tokens:** Charcoal stage `#121212`–`#151515`; single warm accent `#D98E4A` (tungsten amber); New York (serif) for month names/date numerals, San Francisco for functional UI text; 8pt spacing scale; thumbnails as frames (1–2pt border, 4–6pt inset, never edge-to-edge). Motion: ceremony assembly ~1.2–1.8s eased spring; Reduce-Motion fallback is a flat 300ms cross-fade into cover art (not a bare cut); swap transition under 200ms so correction feels immediate.

**Component inventory:** `DayCell` (six-state), `FilmFrameThumbnail`, `DayDetailPlayerView`, `SwapControl` (real button + distinct disabled/VoiceOver state), `ReelProgressIndicator` (count-based, not percentage), `ReelRevealFilmstrip`, `ReelPlayerView`, `ArchiveShelfCard`, `ExportPreviewSheet` (with its own loading/success/error), `ContextualPaywallSheet` (parameterized by trigger), `PermissionPrimerCard` (three distinct variants), `SettingsRow`.

**Architecture note for the next phase:** Repository layer (`PhotoKitGateway`, `VisionScorer`, `ReelRenderer` as separate mockable protocols) behind a `DayFrameStore`, so view models never touch PhotoKit/Vision directly — this is what makes the features phase's acceptance tests (kill mid-backfill, kill mid-export, corrupted clip) actually testable in CI. Persistence via SwiftData, with `DayRecord` including the unused `remoteURL`/`syncState` fields now. Named risk: background scanning writes and main-context UI reads on the same SwiftData store are a likely source of actor-isolation warnings unless scanning explicitly goes through a background `ModelActor` — this must be stated explicitly in the architecture handoff, not left as "just use SwiftData."

**External design decision: no external Claude Design detour needed.** The visual system above is specific enough (real hex values, real type/spacing/motion numbers, a full state and component inventory) for a SwiftUI builder to implement directly, and the two screens carrying real design risk — the reveal ceremony and film-frame thumbnails — are better resolved by iterating in SwiftUI than in static mockups that can't show motion. If the user still wants an external visual gut-check first, the ready-to-use prompt is:

*"Design a premium iOS 'monthly video memory' app called Dayframe in a film/darkroom keepsake style — near-black charcoal stage, one warm tungsten-amber accent, serif display numerals for dates, video thumbnails treated as film frames with thin borders rather than square grid tiles. Produce: a month-calendar hub screen showing a 7-column day grid with distinct visual treatments for 'has a moment,' 'no video that day,' 'processing,' 'waiting on cloud sync,' and 'locked/subscriber-only,' plus a full-bleed video detail screen with a persistent 'swap' control, and a 'reel ready' reveal screen where daily thumbnails assemble into a filmstrip before playback. Keep the calendar screen restrained and highly scannable; concentrate ornamentation only on the detail and reveal screens."*

## Final Output

**SwiftUI architecture:** Plain MV, not MVVM ceremony — `@Observable`, `@MainActor` view models per destination (`CalendarViewModel`, `DayDetailViewModel`, `MonthReelViewModel`), each reading through the main-context `DayFrameStore` facade over the three repository protocols locked in design_handoff (`PhotoKitGateway`, `VisionScorer`, `ReelRenderer`). The actor plan is promoted from "named risk" to a required architecture element: PhotoKit/Vision work runs off-main by nature; all SwiftData writes from the backfill scan go through a dedicated background `ModelActor`; every view model is `@MainActor` reading the main context. This boundary is what keeps the six-state `DayCellState` model from drifting into the exact silent-collision bug flagged twice already.

**Apple framework choices:** PhotoKit (`PHPhotoLibrary`, `PHAsset`, `PHCachingImageManager`) for enumeration/thumbnailing; Vision broken out by actual request type rather than treated as one primitive — `VNGenerateAttentionBasedSaliencyImageRequest` for saliency, `VNDetectFaceRectanglesRequest` for the bonus face signal, and sharpness computed manually (Laplacian-variance-style) against `CVPixelBuffer` via Accelerate/Core Image since Vision has no built-in sharpness request; AVFoundation (`AVAssetImageGenerator` for sampling, `AVMutableComposition` + `AVAssetExportSession` for reel assembly); StoreKit 2 (`Transaction.currentEntitlements`) for fully local, offline-capable subscription entitlement with no server component; `BackgroundTasks` (`BGProcessingTask`) for opportunistic backfill top-up only, explicitly not the primary backfill mechanism; `UserNotifications` for the local reel-ready ping.

**Named limitation carried into tech_specs:** `BGProcessingTask` is best-effort, not a guarantee — iOS can delay, refuse, or skip scheduling it based on battery and app-usage trust signals. The real backfill work happens while foregrounded or within the ~30-second background execution window after backgrounding (with a proper expiration handler that checkpoints rather than gets killed mid-write). "Backfill finishes overnight unattended" is not a safe assumption on stock iOS and must be stated as a limitation, not solved by an API name.

**Persistence plan:** SwiftData for `DayRecord`/`MonthRecord` metadata (including the already-agreed unused `remoteURL`/`syncState` fields), scoped per the actor plan above. Rendered reel video files live in the file system under Application Support (not Documents — app-managed derived data, not user documents), explicitly marked `isExcludedFromBackupKey` since they're fully re-derivable and shouldn't silently bloat iCloud backup quota.

**Permissions/privacy plan:** `.readOnly` PhotoKit authorization unless export-to-Photos requires `.readWrite` — least-privilege matters given the app's core trust pitch. Limited-library access is already a first-class state; this phase adds that a user changing their limited selection later must trigger a delta rescan, not a silent no-op. Info.plist usage strings need to be specific and reviewer-legible ("Dayframe scans your videos on-device to build a daily highlight and monthly reel — nothing leaves your phone"), and a `PrivacyInfo.xcprivacy` manifest is required (not optional) since the app touches "required reason" API categories — file timestamps and disk-space checks for the temp-file cleanup and pre-render storage checks already locked into earlier phases.

**Dependency/license decision:** Zero third-party dependencies. First-party frameworks already listed cover the entire locked feature set. Explicitly rejecting any IAP wrapper (e.g., RevenueCat) — StoreKit 2's local transaction/entitlement APIs are sufficient for a backend-less app, and adding a subscription SaaS SDK here would solve a problem this app doesn't have.

**Testing implications:** The repository-protocol boundary is the only practical path to unit testability, since PhotoKit and a populated video library aren't available in a clean CI simulator — mock `PhotoKitGateway`/`VisionScorer`/`ReelRenderer` and test scoring tie-breaks, swap invalidation, render-cache hash-key derivation, and paywall gating against fakes. Given how many requirements in this thread are "state X must never look like state Y," the `DayCellState` enum and Month Reel state machine need enumerated state-matrix/snapshot tests, not just spot checks. For the parts that can't be faked (real Vision scoring, real `AVAssetExportSession` behavior), bundle one or two small fixture video clips in the test target and run a narrow but real end-to-end pipeline test on-device/simulator rather than assuming full-library-scale testing is achievable in CI.

Coming at this as the person who has to make sure a parent scrolling through her camera roll at 11pm never sees a spinner, a crash, or a screen she doesn't understand — so I'm going to be concrete about the actual Swift-level shape of this thing rather than re-describing what's already locked.

The architecture is a thin MV stack with a hard seam underneath it. Every screen gets one `@Observable @MainActor` view model (`CalendarViewModel`, `DayDetailViewModel`, `MonthReelViewModel`, `ArchiveViewModel`, `PaywallViewModel`, `SettingsViewModel`) and every one of them talks only to `DayFrameStore`, never to PhotoKit, Vision, or AVFoundation directly. `DayFrameStore` is itself a thin `@MainActor` facade that reads from the main SwiftData context and delegates all mutation to a background `ScanCoordinator` actor, which owns a `@ModelActor` for writes. This isn't just tidy layering — it's the only way the acceptance tests from app_features (kill mid-backfill, corrupted clip mid-day, kill mid-export) are runnable at all without a real Photos library in CI, and it's the boundary that stops a background scan write and a foreground calendar read from racing on the same SwiftData store, which was flagged as the top concurrency risk last phase.

Navigation is a single `NavigationStack` with a typed `Route` enum driving a `NavigationPath`, matching the no-TabView decision from design_handoff — Day Detail and Month Reel push as destinations, Archive and Settings are toolbar-triggered pushes, not tabs. I want to be specific about one thing that matters for real usage: the calendar view must never block scrolling on background scan or thumbnail generation. I'd use `PHCachingImageManager.startCachingImages` keyed to the visible month's date range, updated on scroll via `.onScrollGeometryChange` or simple visible-range tracking, and cancel/rebuild the cache request set every time the visible month changes — no Vision work happens on-demand from the calendar, ever; the calendar only ever reads already-computed `DayCellState` off SwiftData. If a build accidentally wires "user opens month view" to "trigger scoring for that month," we've broken the entire quiet-background-work premise and a user will feel it as jank the first time she flicks past a busy month.

On data models: `DayRecord` is keyed by a `dayID: String` in `yyyy-MM-dd` form derived from the asset's own capture-timezone metadata (per detailed_discussion's resolved requirement — this must be computed once at scan time and stored, not recomputed from device timezone at render time, or a user who travels will watch their calendar silently reshuffle). I'm deciding it's the latter — a single rolling allowance that resets at local midnight, tracked in one small persisted `SwapAllowanceState` (lastSwapDate + count), independent of which video-day you're correcting. That's simpler to reason about, it's the harsher and more monetization-relevant reading, and "you get to fix one thing a day" is a cleaner mental model for a non-technical user than a per-day-record counter she'd have to discover the boundaries of by trial and error.
